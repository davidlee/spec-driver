# AUD-007 — DE-086 Conformance Audit

## Scope

Post-implementation conformance audit for DE-086 (memory verification primitive:
SHA-stamped attestation for staleness tracking). Covers all acceptance criteria
from the delta, all IP-086 verification entries, and skill integration.

## Evidence

### Test suite

- `just` passed: 3773 tests, 0 failed, ruff clean, pylint 9.72/10
- No regressions from DE-086 work across three phases

### IP-086 verification coverage (all verified)

- **VT-memory-verified-sha**: SHA not stamped on create; stamped on `--verify`; not on plain edit
- **VT-frontmatter-writer**: `update_frontmatter_fields` tested including insertion and return type
- **VT-memory-staleness**: 17 tests; batched git; paths + globs; scoped/unscoped; tiering
- **VT-cli-edit-verify**: 5 tests; stamps SHA+dates; refuses without git; mutex with `--status`
- **VT-cli-list-stale**: 2 CLI tests + 7 formatter tests; tiered output; correct sort
- **VT-confidence-required**: Creation requires confidence; default is medium
- **VA-skill-integration**: Skills updated per DR §5.9
- **VA-backward-compat**: Existing memories without `verified_sha`/`confidence` load and display correctly

### Smoke tests (agent-verified)

- `edit memory --verify` stamps SHA correctly (`61e8069f`)
- `edit memory --verify --status active` correctly refused (mutex)
- `list memories --stale` shows three-tier output with correct ordering
- `list memories` (no `--stale`) works normally with existing memories

## Findings

### F-001: All acceptance criteria met

- **Outcome**: `create memory` stamps `verified` date and `confidence` but NOT `verified_sha` ✓
- **Outcome**: `edit memory --verify` updates `verified` date, `verified_sha`, and `updated` ✓
- **Outcome**: `edit memory --verify` refuses when git unavailable ✓
- **Outcome**: `edit memory --verify` and `--status` are mutually exclusive ✓
- **Outcome**: `list memories --stale` ranks in three tiers ✓
- **Outcome**: Globs converted to git pathspecs ✓
- **Outcome**: Backward compatibility maintained ✓
- **Disposition**: `aligned`

### F-002: Owning specs are stubs

- **Observation**: SPEC-132 (memory) is `stub` with no requirements. SPEC-116 (frontmatter metadata) and SPEC-110 (CLI) are `draft` but do not have DE-086-specific requirements.
- **Outcome**: No spec-level requirements to verify against. The delta's own acceptance criteria and DR-086 served as the authoritative design surface.
- **Disposition**: `tolerated_drift` — owning specs are under-specified. This is pre-existing and not caused by DE-086. SPEC-132 should eventually be populated with memory subsystem requirements including verification and staleness.

### F-003: Skill integration complete

- **Observation**: All three memory skills updated per DR §5.9.
  - `/capturing-memory`: confidence calibration (low/medium/high definitions)
  - `/retrieving-memory`: qualitative staleness surfacing
  - `/reviewing-memory`: `--stale` as default review queue
- **Disposition**: `aligned`

### F-004: list.py size deferred

- **Observation**: `list.py` is 2750+ lines with 95 pre-existing pylint warnings. DE-086 added ~50 lines for `--stale` support, delegating to domain + formatters per architecture guidelines.
- **Disposition**: `follow_up_backlog` — ISSUE-048 created for factoring review. Not a DE-086 regression.

## Recommendations

- Populate SPEC-132 with memory subsystem requirements when next working on that spec.
- ISSUE-048 (list.py factoring) should be addressed before further list.py feature work.
