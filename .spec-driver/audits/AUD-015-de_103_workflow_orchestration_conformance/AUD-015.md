## 4. Findings

### F-001: Placeholder renderers still registered

- **Severity:** low
- **Outcome:** All 7 workflow schemas have placeholder renderers (`_placeholder_renderer`) that return empty strings. Real renderers were not needed — commands write YAML directly via I/O modules, not through the block rendering system.
- **Disposition:** `tolerated_drift` — placeholder renderers are harmless no-ops. If `spec-driver show schema` or TUI rendering ever needs to display workflow blocks, real renderers can be added then. Not blocking.

### F-002: `operations.py` skinny-CLI pattern deferred

- **Severity:** low
- **Outcome:** The skinny-CLI pattern (domain logic in `operations.py`, CLI as thin wrapper) was started in Phase 02 but not adopted. CLI commands delegate directly to I/O modules and state machine. The unused `operations.py` was removed in Phase 06.
- **Disposition:** `aligned` — the current pattern (CLI → I/O modules → state machine) is consistent across all commands and well-tested. A future refactor to a formal operations layer is optional.

### F-003: Domain map assembly uses bundle structure, not git diff

- **Severity:** low
- **Outcome:** `review prime` builds `domain_map` from delta bundle files (DE, IP, phases, workflow). Git-diff-based code-area discovery was deferred. The staleness evaluator does use `get_changed_files` for cache invalidation.
- **Disposition:** `follow_up_backlog` — git-diff-based domain map construction would improve reviewer bootstrap quality for code-heavy deltas. Not blocking for current use.

### F-004: IP-103 progress checkboxes not updated

- **Severity:** low
- **Outcome:** IP-103 §9 progress tracking checkboxes still show `[ ]` for all phases.
- **Disposition:** `spec_patch` — update now.

### F-005: Verification coverage block in IP-103 shows all VTs as `planned`

- **Severity:** low
- **Outcome:** VT-103-001 through VT-103-008 are all `planned` in the IP. Most are now verified by tests.
- **Disposition:** `spec_patch` — update verification statuses.
