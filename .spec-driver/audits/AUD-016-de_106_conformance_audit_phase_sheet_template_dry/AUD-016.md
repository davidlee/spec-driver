bookkeeping across frontmatter, blocks, and markdown.

## Findings

### F-001: FR-001 — Phase creation emits canonical frontmatter

**Outcome**: pass
**Disposition**: aligned

`create_phase()` populates `plan`, `delta`, `objective`, `entrance_criteria`, `exit_criteria` in frontmatter. No embedded YAML blocks emitted. Tests assert frontmatter fields and block absence.

### F-002: FR-003 — Frontmatter-first reading with legacy fallback

**Outcome**: pass
**Disposition**: aligned

`artifacts.py` uses `PhaseSheet.has_canonical_fields()` to detect new-format phases. Falls back to `extract_phase_overview()` for legacy. Never merges. Compatibility verified with mixed workspaces.

### F-003: FR-004 — Delta display shows phase summaries

**Outcome**: pass
**Disposition**: aligned

`show delta DE-106` renders all 3 phases with objectives from frontmatter (P02, P03) and blocks (P01). `_enrich_phase_data()` regex fallback provides task stats for block-free phases.

### F-004: FR-005 — Phase validation via Pydantic model

**Outcome**: pass
**Disposition**: aligned

Validator uses `PhaseSheet` for new-format phases (canonical frontmatter present). Legacy phases checked for `phase.overview` block. 12 phase validator tests cover both paths.

### F-005: NF-002 — Workspace validation produces no false warnings

**Outcome**: pass
**Disposition**: aligned

`spec-driver validate` does not warn about missing overview blocks for new-format phases. Warning suppression gated on `plan`+`delta` presence in frontmatter.
