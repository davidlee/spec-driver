## Findings

### F-001: PROD-004.FR-001 — metadata-driven validation for requirements

- **Observed**: Block schema `supekku:spec.requirements@v1` with `BlockMetadata` + `MetadataValidator` validates all requirement fields. `WorkspaceValidator._validate_spec_requirements_blocks()` wires schema validation into workspace validation pipeline. Spec creation emits empty block. Template includes placeholder.
- **Expected**: FR-001 specifies "System MUST validate frontmatter for all registered kinds using metadata definitions."
- **Outcome**: Requirements blocks are now validated via the same metadata-driven pattern as other block schemas. Block-first parser reads structured data; regex fallback preserved for unmigrated specs.
- **Disposition**: `aligned`

### F-002: PROD-004.FR-002 — strict-mode enforcement for requirements

- **Observed**: Strict mode in parser (no regex fallback, zero records on missing block), validator (missing-block error), and operational guard (blocks flip until all specs migrated). Separate config path `[validation.strict_requirements]` because `get_strict_map()` filters to artifact kinds only.
- **Expected**: FR-002 specifies "System MUST enforce kind-specific required fields and constraints."
- **Outcome**: Strict enforcement follows DE-139 pattern. Guard correctly blocks on 70 unmigrated specs. Config wiring to registry deferred — explicit opt-in only.
- **Disposition**: `aligned`

### F-003: SPEC-114 requirements status — all pending

- **Observed**: SPEC-114 (blocks/metadata) has 9 requirements all at `pending` status. DE-140 exercises FR-001 (Schema Definition API), FR-003 (Runtime Validation), FR-004 (Validation Errors) via the new `spec_requirements_metadata.py` module.
- **Expected**: Requirements touched by DE-140 implementation should reflect current status.
- **Outcome**: SPEC-114 requirement statuses are stale but predate DE-140 — they were `pending` before this delta and reflect the broader DE-136 program's incomplete coverage sweep. Not a DE-140 regression.
- **Disposition**: `follow_up_delta`
- **Ref**: DE-136 umbrella should reconcile SPEC-114 requirement statuses during corpus-wide sweep

### F-004: Drift ledger YAML quoting bug

- **Observed**: DL-049 drift entries have unquoted `detail:` values containing colons (e.g., `detail: NF-001: description is empty placeholder`), causing YAML parse warnings during `validate workspace`.
- **Expected**: Drift ledger entries should produce valid YAML.
- **Outcome**: Bug in `write_drift_ledger()` — values with colons need quoting. Does not affect migration correctness.
- **Disposition**: `follow_up_backlog`

### F-005: Config wiring not yet connected

- **Observed**: `records_from_spec()` accepts `strict: bool` kwarg but registry does not read `[validation.strict_requirements]` config to pass it. The flip command writes config but nothing reads it yet.
- **Expected**: By design (DEC-140 notes §9) — "the flip is connecting config → registry → parser after migration completes."
- **Outcome**: Intentional deferral. Full corpus migration must complete before wiring makes sense.
- **Disposition**: `aligned`
