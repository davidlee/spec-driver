## Observations

### Implementation conformance to DR-135

All three design decisions are faithfully implemented:

- **DEC-DR135-001** (wrap at source): `load_markdown_file` is the single point of YAML-error translation. No other module imports `yaml.YAMLError`.
- **DEC-DR135-002** (ValueError subclass): `MarkdownLoadError(ValueError)` — existing `except (ValueError, OSError)` clauses at `artifacts.py:181`, `registry.py:85`, `drift/registry.py:77`, `requirements/parser.py:132` become correct with zero call-site edits.
- **DEC-DR135-003** (compact message format): `f"invalid YAML frontmatter in {path}{where}: {exc.__class__.__name__}"` — original exception chained as `__cause__`.

### Deviation from original DR-135

DR-135 §3 originally claimed "no call-site edits". Implementation revealed that `changes/registry.py:85-96` only handles top-level artifact files (delta, revision, audit), not phase files inside a delta bundle's `phases/` directory. The ISSUE-054 repro path goes through `artifacts.py:181` which silently `continue`d with no diagnostic. A scoped 2-line edit was added at `artifacts.py:261-264`. DR-135 §3 / §4 were updated to reflect this. Decision was approved by user.
