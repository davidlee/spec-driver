# Audit template out of sync with audit findings schema

## Problem

The template generated by `spec-driver create audit` is out of date with the
evolved `supekku:audit.findings@v1` schema. Two concrete gaps observed during
AUD-030 creation:

### 1. Missing `outcome` field

The generated findings array template is `findings: []` — no field guidance.
The schema expects each finding to have an `outcome` field constrained to
enumerated values: `aligned`, `drift`, `risk`. Without this field or guidance,
agents naturally write descriptive sentences that fail validation as warnings.

### 2. Disposition format changed from flat string to mapping

The schema now requires `disposition` as a mapping with `status` and `kind`
keys (e.g. `{status: reconciled, kind: aligned}`), not a flat string like
`aligned` or `follow_up_delta`. The template gives zero indication of this
structure, and there's no schema example available via
`spec-driver show schema audit.findings -f yaml-example` (returns an error).

### Concrete failure

Both AUD-029 and AUD-030 hit these issues. AUD-029 has flat-string dispositions
that are now errors. AUD-030 originally had flat-string dispositions before
being corrected by reading AUD-025 for the correct format.

## Impact

Every new audit hits the same friction. The template is the first thing agents
see, and it actively misleads about the correct format.

## Desired fix

1. `create audit` template should include a commented-out example finding
   showing correct `outcome`, `disposition`, `evidence` structure.
2. `spec-driver show schema audit.findings -f yaml-example` should work
   (currently errors: `string indices must be integers`).

## Evidence

- AUD-030: had to reverse-engineer correct format from AUD-025
- AUD-029: still has invalid flat-string dispositions (not yet patched)
- `spec-driver show schema audit.findings -f yaml-example` → error
- `spec-driver show schema audit.findings -f json-schema` → "not yet available"
