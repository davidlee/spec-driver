## 1. Summary & Context

- **Product Spec(s)**: [PROD-014](../../specify/product/PROD-014/PROD-014.md) – Contract Mirror Tree Index
- **Implementation Plan**: [IP-027](./IP-027.md) – draft
- **Design Revision**: [DR-027](./DR-027.md) – draft
- **Change Drivers**: Contract discovery ergonomics; agents and developers need predictable source-path-based navigation to contracts without memorising SPEC-_/contracts/_ layouts.

## 2. Motivation

- Current contract discovery requires knowledge of internal `specify/tech/SPEC-*/contracts/` paths and `by-slug/by-package/by-language` index layouts.
- Developers and agents think in source paths (`supekku/scripts/lib/sync/engine.py`), not spec IDs (`SPEC-110`).
- A repo-root `.contracts/` tree mirroring source paths bridges that gap: open `.contracts/public/supekku/scripts/lib/sync/engine.py.md` and you have the public API contract.
- Enables `rg` across `.contracts/**` as a source-structured search corpus.

## 3. Scope & Objectives

- **Primary Outcomes**:
  - `.contracts/` directory at repo root with symlinks mirroring source paths (FR-001, FR-002)
  - Per-language path mapping: Zig file-based, Go package-based, TS file-based, Python module-based (FR-003)
  - Idiom aliases: `.contracts/api` → `public`, `.contracts/implementation` → `all` (FR-004)
  - Clean rebuild: stale entries removed, writes confined to `.contracts/` (FR-005)
  - Registry-driven: no code parsing or toolchain required (FR-006)
  - Graceful handling of missing/conflicting variants (FR-007)
- **Operational Constraints**: Must not break existing `by-*` indices or contract generation. `.contracts/` should be safe to `.gitignore`.
- **Dependencies**: None — uses existing `registry_v2.json` and contract artefacts already in place.
- **Language scope**: Python + Zig in phase 1. Go/TS architecture only (no contracts to test against).

## 4. Out of Scope

- Relocating underlying contract artefacts (they stay under `SPEC-*/contracts/`)
- Removing existing `specify/tech/by-*` symlink indices
- Aspirational/manual contracts in `.contracts/`
- Per-language CLI surfaces for mirror management

## 5. Approach Overview

- **System Touchpoints**: New module `supekku/scripts/lib/contracts/mirror.py`, integration into `supekku/cli/sync.py`
- **Key Changes**:
  - `ContractMirrorTreeBuilder` class: reads `registry_v2.json` + scans `SPEC-*/contracts/` to build `.contracts/<view>/...` symlink tree
  - Per-language path mapping functions (parallel to `SpecIndexBuilder` pattern)
  - Hook into post-sync index rebuild in `supekku/cli/sync.py` (after `rebuild_indices()`)
  - Alias symlink creation for `api` → `public` and `implementation` → `all`
- **Migration / Rollout Notes**: Add `.contracts/` to `.gitignore`. Run `sync` to populate.

- **Requirements Coverage**:

  - FR-003: Unit tests per language for path mapping (Zig, Go, TS, Python)
  - FR-004: Integration test asserts alias symlinks exist and resolve correctly
  - FR-006: Builder only reads registry JSON + contract directories (verified by test design — no mocking of toolchains)

- **Planned Artefacts**:

## 7. Risks & Mitigations

- **Risk**: Symlink depth makes relative paths fragile – _Likelihood_: low – _Impact_: medium – _Mitigation_: Use `os.path.relpath` consistently; test deep paths.
- **Risk**: Conflicting mirror destinations from overlapping registry entries – _Likelihood_: low – _Impact_: low – _Mitigation_: Deterministic precedence (lexicographically smallest SPEC wins) + actionable warning.
- **Risk**: Platform differences in symlink behaviour (Windows) – _Likelihood_: low – _Impact_: low – _Mitigation_: Document Unix-only for v1; same constraint as existing indices.

## 10. Closing Notes

### Python dedup for multi-identifier SPECs

The registry maps multiple identifiers to the same SPEC (e.g. `supekku/scripts/lib/specs` → `SPEC-123`), but the Python mapper scans the entire contracts directory regardless of which identifier triggered it. The builder deduplicates by SPEC ID per language to avoid producing duplicate entries that would surface as self-conflicts.

### Coupling: contract filename conventions

The mirror builder has per-language knowledge of contract filenames baked into its mappers (`interfaces.md`, `internals.md`, `api.md`, etc.). If an adapter changes its output naming, the mirror builder silently produces nothing for that language. Consider a future validation check that warns when a SPEC has a contracts directory but the mirror produced zero entries for it.

### Cleanup strategy: nuke-and-rebuild

`SpecIndexBuilder` does careful per-symlink removal and empty-dir pruning. The mirror builder uses `shutil.rmtree` instead, since `.contracts/` is entirely derived. Simpler, but every rebuild is a full rewrite. Fine at current scale; if NF-001 (<2s for ~10k artefacts) ever becomes tight, selective cleanup would be the optimisation path.

### DR-027 open question resolved

The concern about Python adapter-level variant names (`api`/`implementation`) vs filesystem names (`public`/`all`) is moot — the builder discovers from the filesystem, not from adapter metadata. DR-027 §8 item can be closed.
