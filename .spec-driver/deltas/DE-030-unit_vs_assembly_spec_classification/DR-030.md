# DR-030 – unit_vs_assembly_spec_classification

## 1. Executive Summary
- **Delta**: [DE-030](./DE-030.md)
- **Status**: draft (update when approved)
- **Owners / Team**: <names or teams accountable>
- **Last Updated**: 2026-02-21
- **Synopsis**: Introduce a metadata-first taxonomy (`unit` vs `assembly`) and navigation support (views + CLI filters) so assembly specs remain usable in legacy repos.

## 2. Problem & Constraints
- **Current Behaviour**: Tech specs are treated as a single bucket; sync-created unit specs can drown out human-authored assembly specs, making cross-cutting subsystem documentation feel unworkably messy.
- **Drivers / Inputs**:
  - `specify/decisions/ADR-003-separate_unit_and_assembly_specs.md` (taxonomy + truth model)
  - `specify/product/PROD-015/PROD-015.md` (UX and workflow intent)
- **Constraints / Guardrails**:
  - Non-breaking: no filesystem/prefix migration in this delta.
  - Warn-only validation in v1; do not block adoption.
  - Keep listing/indexing fast enough for interactive use (<2s target).
- **Out of Scope**:
  - Claims/ownership schema for overlapping interface constraints.
  - Migrating to `UNIT-*` / `ASM-*` IDs and new roots.

## 3. Architecture Intent
- **Target Outcomes**:
  - Make spec intent machine-readable: `category` + `c4_level` used consistently. (PROD-015.FR-001)
  - Make assembly specs easy to find via views and CLI filters. (PROD-015.FR-003/FR-004)
  - Ensure sync-created unit specs are clearly labeled. (PROD-015.FR-002)
- **Guiding Principles**:
  - Metadata-first; no migration required.
  - Views are derived; safe to delete/rebuild.
  - Contracts/code remain observed truth; taxonomy is about spec intent and ergonomics.

## 4. Code Impact Summary
| Path | Current State | Target State |
| --- | --- | --- |
| `supekku/scripts/lib/core/frontmatter_metadata/spec.py` | `category` exists but is described as requirement categorization | `category` documented as spec taxonomy; reserved `unit|assembly` |
