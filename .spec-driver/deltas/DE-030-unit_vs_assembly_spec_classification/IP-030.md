## 1. Summary

- **Delta**: DE-030
- **Specs**: PROD-015, ADR-003
- **Desired Outcome**: Explicit `unit`/`assembly` taxonomy on tech specs via metadata; navigation views + CLI filters; warn-only validation; default listing hides unit specs.

## 2. Context & Constraints

- **Current Behaviour**: All tech specs are a single bucket. Taxonomy fields (`category`, `c4_level`) are not consistently present/used. `list specs` shows everything.
- **Target Behaviour**: Taxonomy fields in frontmatter; `list specs` defaults to hiding unit tech specs; by-category/by-c4-level symlink views; validator warns on unclassified tech specs.
- **Dependencies**: DE-028 (sync defaults / opt-in spec creation) already landed.
- **Constraints**: Non-breaking. No filesystem/prefix migration. Warn-only validation. Keep interactive performance acceptable.

## 5. Phase Detail Snapshot

### Phase 1 — Model, creation defaults, CLI filters

**Scope**: Make `category` and `c4_level` first-class on the `Spec` model; set defaults on creation paths; add CLI filter flags; change `list specs` default to hide unit specs.

**Tasks**:

1.1. **Add `category` and `c4_level` properties to `Spec` model** (`specs/models.py`)

- Simple frontmatter reads, same pattern as `kind`/`status`.
- Include in `to_dict()` output when present.

  1.2. **Update `build_frontmatter` in `specs/creation.py`**

- Accept optional `category` and `c4_level` params.
- Default `category` to `"assembly"` for human-created specs (DEC-030-04).

  1.3. **Set taxonomy defaults in sync stub creation** (`sync_specs.py:_create_spec_directory_and_file`)

- Add `"category": "unit"` and `"c4_level": "code"` to the frontmatter dict.

  1.4. **Update frontmatter schema description** (`core/frontmatter_metadata/spec.py`)

- Revise `category` description to mention reserved values `unit`/`assembly`.

  1.5. **Add `--category` and `--c4-level` filters to `list specs`** (`cli/list.py`)

- **Scope**: `--category` and `--c4-level` apply to tech specs only (`SPEC-*` / `kind: spec`). Product specs (`PROD-*`) do not participate in unit/assembly taxonomy and MUST NOT be filtered by these flags.
- `--category` accepts `all` or comma-separated values. **Default** (when flag omitted): `assembly,unknown` (i.e. hides `unit` tech specs by default).
  - `--category all` disables category filtering for tech specs (shows `unit` as well).
  - Missing/empty `category` is treated as `unknown` for filtering.
- `--c4-level` accepts `all` or comma-separated values. Default (when flag omitted): `all`.
  - Missing/empty `c4_level` is treated as `unknown` for filtering.
- Interaction with `--kind`:
  - `--kind` continues to control whether tech/product/both are shown.
  - If `--kind all` (default), products are shown and tech specs are shown with the default “hide unit” category filtering applied.

  1.6. **Tests**: VT-030-001 (model roundtrip), VT-030-002 (creation defaults), VT-030-003 (CLI filtering).

### Phase 2 — Index views + validation

**Scope**: Extend `SpecIndexBuilder` with by-category and by-c4-level symlink views; add warn-only taxonomy validation.

**Tasks**:

2.1. **Add `by-category` views to `SpecIndexBuilder`** (`specs/index.py`)

- Read `category` from frontmatter during rebuild.
- Create `by-category/{unit,assembly,unknown}/SPEC-XXX → ../../SPEC-XXX` symlinks.
- Only tech specs (`SPEC-*`); PROD specs excluded from taxonomy views.

  2.2. **Add `by-c4-level` views to `SpecIndexBuilder`**

- Same pattern. Buckets: `system`, `container`, `component`, `code`, `interaction`, `unknown`.

  2.3. **Add taxonomy warnings to `WorkspaceValidator`**

- New method `_validate_spec_taxonomy`.
- Warn when a tech spec (`SPEC-*`) is missing `category` or `c4_level`.
- Warn on inconsistent combos (`category: unit` + `c4_level` not `code`).
- Scoped to tech specs only (DEC-030-05).

  2.4. **Tests**: VT-030-004 (index views), VT-030-005 (validation warnings), VT-030-006 (non-breaking).

### Phase 3 — Alias symlinks + show spec taxonomy

**Scope**: Add ergonomic directory aliases for taxonomy views; display taxonomy in `show spec` output.

**Tasks**:

3.1. **Add alias directory symlinks to `SpecIndexBuilder.rebuild()`** (`specs/index.py`)

- `assemblies -> by-category/assembly`, `units -> by-category/unit`, `c4 -> by-c4-level`
- Idempotent: cleaned and recreated on each rebuild.

  3.2. **Display taxonomy in `show spec` detail output** (`formatters/spec_formatters.py`)

- Add `_format_taxonomy` section showing `Category` and `C4 Level` when present.
- Already included in `to_dict` (JSON output) from phase 1.

  3.3. **Tests**: Alias symlinks resolve through to specs; formatter displays taxonomy fields.

## 7. Risks & Mitigations

| Risk                                                         | Mitigation                                                             | Owner |
| ------------------------------------------------------------ | ---------------------------------------------------------------------- | ----- |
| Default filtering surprises users expecting to see all specs | `--category all` escape hatch; document in help text                   | Dev   |
| Existing specs have no `category` → all show as `unknown`    | `unknown` included in default filter; validator warns to guide cleanup | Dev   |
| Index rebuild slower with new views                          | Reuse existing frontmatter read; measure against NF-001 target         | Dev   |
