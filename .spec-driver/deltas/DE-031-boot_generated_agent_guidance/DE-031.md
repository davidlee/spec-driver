## 1. Summary & Context

- **Product Spec(s)**: `specify/product/PROD-016/PROD-016.md` – “Agentic SDLC Configurator” (config + installer + bootstrap).
- **Implementation Plan**: `change/deltas/DE-031-boot_generated_agent_guidance/IP-031.md` – single-phase implementation.
- **Change Drivers**: Skills should be static and portable; project-specific conventions and enabled primitives should be rendered into small, token-cheap markdown “modules” installed under `.spec-driver/agents/**`.

## 2. Motivation

Today the installer copies a static `supekku/agents/` directory into `.spec-driver/agents/`, but those files are not
config-tailored. The `/boot` skill needs to:

- teach “how to run spec-driver” using the project’s preferred execution strategy
- explain enabled primitives and paths (kanban lanes, doc roots, contracts root)
- load a glossary slice for enabled primitives only (token conservation)

This delta makes `.spec-driver/agents/**` a stable home for **installer-rendered** markdown modules generated from
templates using `.spec-driver/workflow.toml` (and optionally `.spec-driver/doctrine.md` as an escape hatch).

## 3. Scope & Objectives

**Primary Outcomes**

- Generate and maintain the following project-local modules under `.spec-driver/agents/`:
  - `exec.md` (how to run spec-driver; `uv run` vs `uvx` vs etc.)
  - `workflow.md` (ceremony + enabled primitives + canonical paths)
  - `glossary.md` (enabled primitives only; include “term translation” such as card/plan)
  - `policy.md` (policy layer guidance: ADRs/policies/standards based on config)
- Render these from templates using `.spec-driver/workflow.toml` as the primary source.

**Operational Constraints**

- Installer-owned outputs: `.spec-driver/agents/**` may be overwritten on install.
- Keep modules short; the goal is token-cheap agent onboarding.

**Dependencies**

- `PROD-016.FR-008` defines `.spec-driver/agents/` as the canonical home for generated agent-facing markdown.

## 4. Out of Scope

- OpenSkills allowlist exposure / `AGENTS.md` skills table sync (handled in DE-032).
- Writing the actual `/boot` skill implementation (lives in `davidlee/spec-driver-skills`).

## 5. Approach Overview

**System Touchpoints**

- Installer: `supekku/scripts/install.py` (`initialize_workspace`)
- Template engine: `supekku/scripts/lib/core/templates.py` (Jinja2)
- Spec-driver dir constants: `supekku/scripts/lib/core/paths.py` (`SPEC_DRIVER_DIR`)

**Key Changes**

- Add agent-doc templates to the template environment (package templates).
- Teach the installer to:
  1. read `.spec-driver/workflow.toml` (or use defaults if missing)
  2. render the four agent modules into `.spec-driver/agents/`
  3. overwrite files in-place (installer-owned)

**Migration / Rollout Notes**

- Keep the existing “copy package_root/agents → .spec-driver/agents” behavior as a fallback only if templates are missing.

## 6. Verification Strategy

**Acceptance Criteria**

- Running `spec-driver install` produces:
  - `.spec-driver/agents/exec.md`
  - `.spec-driver/agents/workflow.md`
  - `.spec-driver/agents/glossary.md`
  - `.spec-driver/agents/policy.md`
- The generated docs reflect configured paths (kanban lanes, doc roots) and hide disabled primitives.
- Re-running install is safe and produces minimal diffs when config is unchanged.

## 7. Risks & Mitigations

- **Risk**: Template/config mismatch produces confusing agent guidance – _Likelihood_: medium – _Impact_: medium – _Mitigation_: keep templates small; add tests for key substitutions.
- **Risk**: Config file missing or invalid TOML – _Likelihood_: low – _Impact_: medium – _Mitigation_: fall back to safe defaults and emit a warning in generated `workflow.md`.
