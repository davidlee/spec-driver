## 1. Summary & Context

- **Product Spec(s)**: `specify/product/PROD-016/PROD-016.md` – installer must install only the correct skills/guidance for the chosen config.
- **Change Drivers**: OpenSkills `--universal` installs skills into `.agent/skills` but does not provide non-interactive selective `sync` into `AGENTS.md`. We need deterministic, config-driven exposure via an allowlist.

## 2. Motivation

OpenSkills is a good portable skills distribution mechanism, but by default either:

- skills are installed into agent-visible locations (too noisy / not conditional), or
- `openskills sync` is interactive / “all skills” and cannot be driven from config.

We want:

- skills installed into `.agent/skills/**` (portable cache)
- _only_ a selected subset exposed to agents via the `AGENTS.md` `<available_skills>` block
- selection controlled by `.spec-driver/skills.allowlist` (user-editable, produced by `spec-driver configure` later)

## 3. Scope & Objectives

**Primary Outcomes**

- Introduce `.spec-driver/skills.allowlist` (one skill name per line; `#` comments allowed).
- Add a CLI command: `spec-driver skills sync` that:
  - reads allowlist
  - reads installed project skills from `.agent/skills/<name>/SKILL.md`
  - writes only those skills into the `AGENTS.md` `SKILLS_TABLE_START/END` region, preserving any existing global skills
  - is idempotent (safe to re-run; minimal diffs)

**Dependencies**

- Assumes skills are installed via `npx openskills install <repo> --universal -y`.

## 4. Out of Scope

- Interview-driven config recommendation (handled by configurator work; not required here).
- Managing non-OpenSkills provider skill systems.

## 5. Approach Overview

**System Touchpoints**

- New CLI group: `supekku/cli/skills.py` (or similar), wired into `supekku/cli/main.py`
- Allowlist file: `.spec-driver/skills.allowlist`
- Skill cache: `.agent/skills/**` (OpenSkills universal install)
- AGENTS file: `AGENTS.md` in repo root

**Key Changes**

- Implement a small renderer for the OpenSkills `AGENTS.md` XML block:
  - `<skills_system priority="1">` + `<usage>` + `<available_skills>` + `<skill>` entries
  - only rewrite content between `<!-- SKILLS_TABLE_START -->` and `<!-- SKILLS_TABLE_END -->`
  - preserve existing `<skill>` entries with `<location>global</location>` inside that region
- Parse skill description from YAML frontmatter in `.agent/skills/<name>/SKILL.md`.

**Migration / Rollout Notes**

- If `AGENTS.md` does not contain a skills block, create one.
- If an allowlisted skill is missing from `.agent/skills/`, emit a warning and skip it.

## 6. Verification Strategy

**Acceptance Criteria**

- `spec-driver skills sync` produces a stable skills table with only allowlisted project skills.
- Re-running the command produces no changes.
- Global skills are preserved.

## 7. Risks & Mitigations

- **Risk**: AGENTS.md format drift upstream – _Likelihood_: medium – _Impact_: medium – _Mitigation_: keep formatting isolated; add a unit test pinned to the known format.

## 8. Follow-ups & Tracking

- **Follow-up**: `spec-driver configure` should generate `.spec-driver/skills.allowlist` from ceremony/toggles (future).
