# DR-033 – memory records schema and command surface

## 1. Executive Summary

- **Delta**: [DE-033](./DE-033.md)
- **Status**: draft
- **Owners / Team**: TBD
- **Last Updated**: 2026-03-02
- **Synopsis**: Convert JAMMS v0.1 into a spec-driver-native architecture by normalizing metadata contracts and command shape while preserving deterministic retrieval goals.

## 2. Problem & Constraints

- **Current Behaviour**:
  - The proposal in `change/revisions/RE-017-memory_schema_and_cli_taxonomy_alignment/mem.brief.md` uses frontmatter fields that do not directly align with current base frontmatter conventions.
  - Proposed commands are noun-first (`mem list`, `mem show`, etc.), which conflicts with current CLI architecture and discoverability model.
- **Drivers / Inputs**:
  - `change/revisions/RE-017-memory_schema_and_cli_taxonomy_alignment/mem.brief.md`
  - RE-017
  - `supekku/cli/main.py` command-group structure
- **Constraints / Guardrails**:
  - Follow verb-noun command structure used across existing artifact operations.
  - Keep CLI modules thin and delegate domain logic/formatting to library packages.
  - Avoid creating competing truth sources; memory should point to canonical specs/ADRs/docs.
- **Out of Scope**:
  - Semantic/vector retrieval.
  - Automatic rewriting of canonical specs from memory.

## 3. Architecture Intent

- **Target Outcomes**:
  - First-class memory artifacts can be created/listed/shown/found with consistent CLI semantics.
  - Memory metadata is validated using existing metadata-driven schema machinery.
  - Retrieval is deterministic and auditable by tests.
- **Guiding Principles**:
  - Favor schema compatibility over raw field-name preservation from the source brief.
  - Keep domain logic in dedicated package(s), formatting in formatter modules, and CLI orchestration thin.
  - Keep memory concise and relational, not canonical for implementation detail.
- **State Transitions / Lifecycle Impact**:
  - Memory records support explicit lifecycle statuses aligned with project expectations and documented filtering rules.

## 7. Design Decisions & Trade-offs

- **DEC-033-001**: Use verb-noun commands, not `mem <verb>`.
  - Rationale: fits established CLI structure and improves predictability.
  - Trade-off: brief examples must be translated, not adopted verbatim.
- **DEC-033-002**: Keep base frontmatter canonical; map memory-specific semantics through explicit schema extensions.
  - Rationale: avoids bypassing existing validation architecture.
  - Trade-off: some JAMMS names change (`title`/`type`/`time.*` mappings).
- **DEC-033-003**: Memory records remain pointers/constraints with provenance; they do not replace canonical docs/specs.
  - Rationale: avoids competing truths and preserves architecture integrity.
  - Trade-off: users must maintain references to canonical artifacts.
- **DEC-033-004**: Defer global inverse relation traversal to a dedicated graph index increment (v1.1+).
  - Rationale: current registries are domain-local and use mixed link surfaces (`relations` plus typed fields), so full inverse queries would be brittle without normalization.
  - Trade-off: v1 keeps memory links advisory and avoids over-building.

## 11. v1.1 Extension Path (Deferred)

- **Goal**: reliable forward and inverse link traversal across memory + existing artifact families.
- **Proposed Data Shape**:
  - Build a derived graph registry under `.spec-driver/registry/graph.yaml`.
  - Nodes: `{id, kind, path}` for spec/prod/adr/policy/standard/delta/revision/audit/memory.
  - Edges: `{source, relation, target, provenance}` where `provenance` is `frontmatter.relations`, typed field alias, or memory-native relation.
- **Normalization Rules**:
  - Canonical ID normalization (`ADR-001`, `SPEC-123`, etc) with strict prefix and padding handling.
  - Alias typed fields to canonical relations (`policies[] -> relates_to`, etc) during graph build.
  - Path references in memory remain best-effort unless resolvable to canonical IDs.
- **Query Surface (candidate)**:
  - `list memories --related <ID>`
  - `list memories --requires-reading <ID-or-path>`
  - `list <artifact-plural> --related-by <MEM-ID>`
- **Validation Policy (candidate)**:
  - v1.1 default: warnings for unresolved targets.
  - strict mode: treat unresolved canonical IDs as validation errors.
- **Tracking**:
  - See ISSUE-031 for concise unresolved gaps inventory and acceptance criteria.

## 12. v1 Requirements + ACS

### MEM-FR-001: Memory Artifact Kind + Frontmatter Contract

Requirement:

- The system MUST support memory as a first-class artifact kind (`kind: memory`) with required base frontmatter fields and memory-specific fields defined in a dedicated metadata profile.

ACS:

1. `spec-driver schema show frontmatter.memory --format json-schema` returns a schema.
2. A minimal valid memory artifact passes validation and CLI parse/load paths.
3. Invalid memory-specific fields fail with deterministic diagnostics.

Verification:

- VT-MEM-SCHEMA-001

### MEM-FR-002: Verb-Noun CLI Surface for Memory

Requirement:

- Memory operations MUST be exposed using existing verb groups (for example `create memory`, `list memories`, `show memory`, `find memory`) and MUST NOT introduce noun-first command roots.

ACS:

1. `spec-driver --help` and subgroup help text show memory under verb groups.
2. No top-level `mem` command root is required for v1.
3. `list memories` and `show memory` support both human-readable and JSON output patterns aligned with existing artifacts.

Verification:

- VT-MEM-CLI-001

### MEM-FR-003: Deterministic Selection + Filtering

Requirement:

- Memory listing/retrieval MUST support deterministic filtering and ordering for v1 context surfacing.

ACS:

1. Filters exist for memory type/tag/status/path/command scope.
2. Ordering is deterministic and documented (severity/weight/specificity/recency/id tie-break).
3. Same input set and filter options produces stable output ordering across runs.

Verification:

- VT-MEM-SELECTION-001

### MEM-FR-004: Advisory Cross-Artifact Links (No Registry Coupling)

Requirement:

- Memory references to ADR/policy/standard/spec/delta/etc MUST be parseable and filterable, but MUST NOT mutate non-memory registries or backlinks in v1.

ACS:

1. Memory link references are visible in memory show/list outputs where requested.
2. Workspace sync of decisions/policies/standards/changes behaves unchanged when memory files are present.
3. Missing referenced targets produce warnings/diagnostics (not forced lifecycle coupling) in v1.

Verification:

- VT-MEM-LINKS-001

### MEM-NFR-001: No Memory Block Metadata in v1

Requirement:

- v1 memory artifacts MUST rely on frontmatter only; no dedicated memory YAML fenced block is required.

ACS:

1. Valid memory artifacts contain no required `supekku:*` memory block.
2. Memory schema discoverability is provided via `frontmatter.memory`, not a new block marker.
3. Documentation states block metadata is deferred.

Verification:

- VT-MEM-SCHEMA-001
- VT-MEM-CLI-001

### MEM-NFR-002: YAGNI Deferral of Global Inverse Traversal

Requirement:

- Full inverse traversal and graph indexing MUST remain deferred to a follow-up increment and backlog tracking item.

ACS:

1. DR includes a deferred design path for graph index/inverse traversal.
2. Backlog issue exists and captures unresolved gaps and acceptance criteria.
3. v1 implementation scope excludes mandatory global graph registry generation.

Verification:

- ISSUE-031 tracking + scope conformance review in DE-033/IP-033.

### MEM-FR-005: External Memory Skills Cohesion Contract

Requirement:

- v1 memory support MUST define and document the companion skills in the `spec-driver-skills` repo so memory workflows are cohesive end-to-end, even though skill artifacts are maintained externally.

ACS:

1. Required skills are identified with invocation reasons:
   - `capturing-memory`
   - `retrieving-memory`
   - `maintaining-memory`
2. Optional skill is identified and scoped:
   - `reviewing-memory` (structured review queue/cadence workflows).
3. DR/DE docs in this repo include explicit references that these skills are external deliverables and part of v1 acceptance readiness.
4. Implementation completion checklist includes “skills authored + pushed” as a release-readiness gate for memory feature cohesion.

Verification:

- VT-MEM-SKILLS-001
