## 1. Summary & Context

- **Product Spec(s)**:
  - `specify/product/PROD-012/PROD-012.md` (contracts/sync workflow behavior)
  - `specify/product/PROD-015/PROD-015.md` (taxonomy and agent workflow ergonomics)
- **Technical Spec(s)**:
  - DE-033 implementation outputs (`memory` schema, registry, selection, resolve pipeline)
- **Implementation Plan**: [IP-036](./IP-036.md) – draft
- **Change Drivers**:
  - Memory frontmatter is currently token-heavy and regresses after `resolve links`.
  - `resolve links` persists full resolved payloads (`id`, `path`, `kind`, `label`) for every link.
  - The same anti-pattern appears beyond memories: frontmatter/blocks carry defaults and derived fields that can be reconstructed.

## 2. Motivation

- We need tooling-level controls so metadata stays compact by default, rather than relying on manual cleanup discipline.
- Current behavior creates a loop where authoring stays lean but automation re-inflates files.
- Target state:
  - machine-value metadata is preserved;
  - redundant/default/derived metadata is omitted by default;
  - persistence of verbose metadata is explicit and opt-in.

## 3. Scope & Objectives

- **Primary Outcomes**:
  - Add write controls and persistence modes for memory link resolution so `resolve` and `sync --memory-links` do not auto-bloat records.
  - Introduce canonical metadata compaction rules (drop defaults/derived values) for memory frontmatter.
  - Evaluate and design a **general-purpose** frontmatter/YAML-block compaction approach for specs, deltas, revisions, plans, and related artifacts.
  - Pilot general-purpose compaction in at least one non-memory artifact family with compatibility safeguards.
- **Operational Constraints**:
  - No regressions in retrieval/ranking behavior for memories.
  - Backward compatibility for previously written metadata formats.
  - Deterministic serialization and stable diffs.
- **Dependencies**:
  - Builds on DE-033 memory command surface and schema baseline.
  - Must remain compatible with current block parsers/validators.

## 4. Out of Scope

- Replacing markdown body links (`[[...]]`) with a new link syntax.
- Hard migration of every historical artifact in one pass.
- Redesigning requirement/spec taxonomy or lifecycle semantics.
- Introducing embeddings/vector retrieval.

## 5. Approach Overview

- **System Touchpoints**:
  - `supekku/cli/resolve.py`
  - `supekku/cli/sync.py`
  - `supekku/scripts/lib/memory/links.py`
  - `supekku/scripts/lib/core/spec_utils.py`
  - `supekku/scripts/lib/core/frontmatter_metadata/*`
  - `supekku/scripts/lib/blocks/*`
  - memory and artifact formatter/registry tests
- **Key Changes**:
  - Resolve/sync write controls:
    - default mode analyzes/reports link resolution but does not persist bulky derived payloads.
    - explicit write mode persists chosen link detail level.
  - Link persistence modes:
    - `none` (do not store resolved `links.out`)
    - `missing` (store unresolved targets only)
    - `compact` (store minimal resolved references)
    - `full` (current behavior, opt-in)
  - Canonical compaction profile framework:
    - artifact-specific rules for omitting defaults and derived metadata.
    - shared implementation so memory/spec/delta families use one mechanism.
  - General-purpose design gate:
    - define which fields are canonical vs derived in frontmatter and YAML blocks.
    - decide where derivation should occur (read-time vs sync-time vs view-time).
- **Migration / Rollout Notes**:
  - Phase rollout:
    1. memory-specific controls;
    2. general compaction framework;
    3. non-memory pilot;
    4. optional repository-wide cleanup.
  - Preserve read compatibility for existing files before enabling aggressive pruning.

## 6. Verification Strategy

- **Requirements Coverage**:
  - command-level behavior tests for resolve/sync write modes and defaults.
  - serialization tests for compact/full/none link persistence.
  - regression tests ensuring selection/ranking outputs are unchanged for equivalent records.
  - pilot-family tests showing general compaction does not break validators/parsers.
- **Planned Artefacts**:
  - VT-036-001: resolve/sync mode semantics and no-bloat defaults.
  - VT-036-002: memory compaction rules and compatibility.
  - VT-036-003: generic compaction framework + pilot artifact family.
  - VA-036-001: before/after metadata size analysis across memory + pilot families.
- **Acceptance Criteria**:
  - Running link resolution in default workflow does not expand memory frontmatter.
  - Explicit full persistence remains available for workflows that need it.
  - At least one non-memory artifact family successfully uses the shared compaction approach.
  - Documented policy for canonical vs derived metadata is committed and referenced by skills/docs.

## 7. Risks & Mitigations

- **Risk**: Hidden dependency on persisted `links.path/kind` in external tooling – _Likelihood_: medium – _Impact_: medium – _Mitigation_: retain `full` mode and backward-compatible readers; gate default changes behind explicit tests.
- **Risk**: Over-pruning removes semantically meaningful metadata – _Likelihood_: medium – _Impact_: high – _Mitigation_: define canonical/derived matrix per field and require fixture-based round-trip tests.
- **Risk**: General framework adds premature abstraction – _Likelihood_: medium – _Impact_: medium – _Mitigation_: memory-first implementation and one pilot family before broad rollout.
