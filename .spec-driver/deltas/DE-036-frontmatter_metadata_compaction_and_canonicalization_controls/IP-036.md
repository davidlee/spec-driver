## 1. Summary

- **Delta**: DE-036 - frontmatter metadata compaction and canonicalization controls
- **Specs Impacted**:
  - `specify/product/PROD-012/PROD-012.md`
  - `specify/product/PROD-015/PROD-015.md`
  - memory/frontmatter/block schemas under `supekku/scripts/lib/`
- **Problems / Issues**:
  - metadata inflation from derived/default fields;
  - `resolve links` persistence causing recurring frontmatter growth.
- **Desired Outcome**: metadata remains compact by default across memory + at least one additional artifact family, with explicit opt-in for verbose persistence.

## 2. Context & Constraints

- **Current Behaviour**:
  - memory authoring can be lean, but `resolve links` writes full `links.out` payloads.
  - defaults and generated metadata are often persisted even when reconstructible.
- **Target Behaviour**:
  - write paths preserve canonical semantics with minimal persisted metadata.
  - verbose/generated metadata is opt-in and mode-controlled.
- **Dependencies**:
  - DE-033 memory command/schema baseline.
  - Existing block parser/validator compatibility.
- **Constraints**:
  - no retrieval/ranking regressions;
  - backward-compatible parsing of existing files;
  - deterministic ordering for stable diffs.
