## 1. Summary
- **Delta**: DE-037 - install and refresh seed memories during workspace install
- **Specs Impacted**:
  - install workflow and memory onboarding conventions
- **Problems / Issues**:
  - no memory pack bootstrap in install flow;
  - current `seed` tag conflated user starter memories with spec-driver-owned memories.
- **Desired Outcome**: installer enforces two-bucket ownership semantics (`seed` install-or-leave; `spec-driver` replace/sync). Bucket identity determined by ID namespace (`mem.*.spec-driver.*` vs `mem.*.project.*`), not tags.

## 2. Context & Constraints
- **Current Behaviour**:
  - install creates directories/registries/templates/about/agents/skills;
  - no seed memory install/refresh step exists.
- **Target Behaviour**:
  - installer manages two buckets:
    - `seed`: add missing starter memories only, never overwrite local edits.
    - `spec-driver`: refresh by ID from package source on reinstall.
  - unmanaged memories remain unchanged.
- **Dependencies**:
  - package includes discoverable seed memory source path.
  - existing prompt/dry-run category framework.
- **Constraints**:
  - preserve existing install semantics and tests;
  - idempotent and deterministic behavior.
