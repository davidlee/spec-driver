## 1. Summary & Context

- Canonical Sequence to Anchor On:
  - delta -> DR -> IP -> phase sheet(s) -> implement -> audit -> revision (from audit findings) -> patch specs -> close
  - Treat “revision-first” as a non-canonical concession path, not the primary narrative.
- This delta defines the documentation and memory surface area required to make that sequence lockable via a future `workflow.toml` switch, but does not implement enforcement.
- **Research**: [workflow-research.md](./workflow-research.md) — delta-first canonical model research
- **Change Drivers**: Research findings (section E) identifying documentation/memory drift that would confuse agents and block mass adoption

## 2. Motivation

Spec-driver's documentation, memories, and skills present multiple workflow narratives that compete with each other.
The code is permissive by default, with a primary close-out coverage gate on `complete delta` plus other hard failure paths, but:

- The core-loop memory leads with `revision intent` as step 2, positioning revision-first as the primary narrative
- `lifecycle.md` documents status enums that don't exist in code (`implemented`, `verified`)
- `README.md` describes an "Archive" step that has no implementation
- `RUN.md` documents the primary close-out gate (coverage blocks) but is unreferenced from canonical docs
- Ceremony mode is described as if it has enforcement, but it's advisory-only

An unsuspecting agent or user reading these sources will form an incorrect mental model, hit walls at delta completion, and waste time on non-canonical paths.

## 3. Scope & Objectives

- **Primary Outcomes**:
  1. Memories promote delta-first as the canonical narrative; revision-first framed as Town Planner concession
  2. Stale status enums and ghost statuses corrected in all memories and docs
  3. Coverage block requirement is discoverable from the canonical onboarding path
  4. Ceremony mode described accurately as advisory guidance, not enforcement
  5. Proposed `strict_mode` workflow setting contract for optional canonical lock-in is documented
  6. Enforcement touchpoints, hard-fail flag policy for non-canonical paths, and shared status-policy enforcement expectations are mapped for future implementation
  7. Recommended v1 coverage aggregation precedence and mixed-status warning policy are documented for follow-up implementation
  8. Critical drift/gaps to mass adoption identified and documented for future work
  9. Workflow-critical memories have retrieval-precision scope metadata so command/path queries surface operational records ahead of broad conceptual ones
- **Operational Constraints**: Documentation/memory changes only — no code changes in this delta
- **Dependencies**: None

## 4. Out of Scope

- Code changes (adding `verified` to requirement lifecycle, `create audit` command, `complete revision` command, ceremony enforcement)
- Implementing any new `workflow.toml` switch or runtime branching logic
- Rewriting `lifecycle.md`, `README.md`, `RUN.md`, or `processes.md` (those are doc changes; this delta focuses on memories and skills that agents actually consume)
- Updating the `doc/artefacts/workflow-research.md` content that was the input to this delta

## 5. Approach Overview

- **System Touchpoints**: Memory records, skill definitions, agent workflow docs
- **Key Changes**:
  - Reframe `mem.pattern.spec-driver.core-loop` to lead with delta-first
  - Update `mem.concept.spec-driver.revision` to frame as concession for Town Planner
  - Fix `mem.pattern.spec-driver.delta-completion` to reference correct status enums and coverage block requirement
  - Update ceremony memories to accurately describe advisory-only enforcement
  - Document a proposed `strict_mode` switch contract and non-goal boundary (documentation-only)
  - Define baseline `strict_mode` semantics: non-canonical paths hard-fail, no exception knobs in this contract
  - Add a path compatibility matrix that includes relevant flags and alternate modes
  - Add a future enforcement touchpoint map for implementation follow-up
  - Define that follow-up status-policy enforcement should use a shared code path for both read-time and write-time checks
  - Document recommended v1 coverage precedence (`failed/blocked` dominant, mixed => `in-progress`) and warning-level validation on mixed status sources
  - Add a new memory documenting the coverage block prerequisite (sharp edge)
  - Add an explicit memory scope-tuning pass and associated guidance to reduce retrieval noise in command/path queries
  - Produce a "gaps to mass adoption" document in the delta bundle
