## 1. Summary & Context

- **Product Spec(s)**: [PROD-016](../../../specify/product/PROD-016/PROD-016.md) – workflow switchboard/configurator and command-surface expectations.
- **Implementation Plan**: [IP-039](./IP-039.md) – drafted and phase-scaffolded.
- **Change Drivers**:
  - `DE-038` gaps register carry-forward queue (`strict_mode`, `create audit`, `complete revision`, command UX cleanup).
  - Operator/agent pain during headless `complete delta` flows.

## 2. Motivation

- Workflow-critical command surfaces remain incomplete relative to intended ceremony/workflow model.
- Remaining gaps force ad-hoc manual steps and brittle prompt choreography, reducing reliability for agent and CI use.
- The target state is a deterministic, automation-safe command set with explicit strict/permissive workflow behavior.

## 3. Scope & Objectives

- **Primary Outcomes**:
  - Implement `strict_mode` runtime branching contract in workflow config and enforcement touchpoints (`PROD-016.FR-002`).
  - Ensure completion behavior is automation-safe and deterministic in headless execution (`PROD-016.FR-009`).
  - Add first-class lifecycle command coverage for audit/revision workflows (`PROD-016.FR-010`).
- **Operational Constraints**:
  - Preserve default permissive behavior when strict mode is disabled.
  - Keep CLI files thin; delegate behavior to script/lib modules.
- **Dependencies**:
  - `DE-038` design intent for strict-mode semantics and enforcement map.

## 4. Out of Scope

- Full ceremony-policy redesign beyond the `strict_mode` contract.
- Broad docs rewrite outside command/workflow references required by this delta.
- New lifecycle statuses or taxonomy changes not required by command implementation.

## 5. Approach Overview

- **System Touchpoints**:
  - `supekku/scripts/lib/core/config.py`
  - `supekku/scripts/complete_delta.py`
  - `supekku/cli/create.py`
  - `supekku/cli/complete.py`
  - `supekku/scripts/lib/changes/coverage_check.py`
  - workflow docs/memory records tied to completion behavior
- **Key Changes**:
  - Add `strict_mode` config support and wire enforcement points for non-canonical command paths.
  - Introduce `create audit` and `complete revision` first-class CLI flows.
  - Finalize completion UX ergonomics (non-interactive prompt defaults + guidance/error messaging alignment).
- **Migration / Rollout Notes**:
  - `strict_mode` defaults to disabled to preserve current workflows.
  - Document bypass/override behavior explicitly for operator predictability.

## 6. Verification Strategy

- **Requirements Coverage**:
  - `PROD-016.FR-002`: config+runtime tests for strict/permissive behavior.
  - `PROD-016.FR-009`: regression tests for non-interactive completion and prompt defaults.
  - `PROD-016.FR-010`: CLI tests for `create audit` and `complete revision`.
- **Planned Artefacts**:
  - VT-039-001: strict-mode enforcement path tests.
  - VT-039-002: headless completion behavior tests.
  - VT-039-003: audit/revision command lifecycle tests.
- **Acceptance Criteria**:
  - Commands operate without manual prompt choreography in non-interactive execution.
  - Revision and audit lifecycle flows are available as first-class CLI verbs.
  - Strict-mode toggle is implemented and documented with deterministic behavior.

## 7. Risks & Mitigations

- **Risk**: Strict-mode enforcement may break tolerated legacy flows – _Likelihood_: medium – _Impact_: high – _Mitigation_: default `strict_mode=false`, targeted gating, explicit migration notes.
- **Risk**: New command surfaces diverge from lifecycle semantics – _Likelihood_: medium – _Impact_: medium – _Mitigation_: share lifecycle constants/validation and add end-to-end command tests.
- **Risk**: UX improvements drift from memories/docs – _Likelihood_: low – _Impact_: medium – _Mitigation_: update workflow-critical memories/docs in same change-set.

## 8. Follow-ups & Tracking

- **Future Phases / Deltas**:
  - Optional policy split of strict-mode into finer controls (if single switch proves too coarse).
- **Decisions**:
  - `2026-03-04`: `strict_mode=true` universally blocks force-style bypass paths (including `--force` and equivalent non-canonical escape flags). If this proves too restrictive in practice, split-policy controls will be considered in a later delta.

## 9. Implementation Notes

- Canonical execution command: `uv run spec-driver`.
- Verification baseline: `just check` plus focused CLI/script tests for new command paths.
