## 1. Summary & Context

- **Product Specs**: PROD-010 (CLI UX), PROD-013 (CLI Artifact File Access)
- **Implementation Plan**: [IP-041](./IP-041.md)
- **Change Drivers**: DE-038 gaps-to-adoption.md (§3 Missing Automation), direct observation of workflow friction

## 2. Motivation

Several artifact types can be created via CLI but lack basic addressability commands
(show/view/edit/find). This forces users and agents to manually locate files or
guess paths, breaking the "consistent CLI for all artifacts" contract expressed
in PROD-010 and PROD-013.

The most impactful gap: implementation plans (IP) have zero CLI presence — they
cannot be created independently, listed, shown, or found. Agents reference IPs
constantly by ID but have no programmatic way to discover or inspect them.

## 3. Scope & Objectives

**Primary Outcomes**:

### Tier 1 — Missing subcommands

Current coverage matrix (`Y` = exists, **gap** = missing):

| Artifact     | create  | list    | show    | view    | edit    | find    |
| ------------ | ------- | ------- | ------- | ------- | ------- | ------- |
| plan (IP)    | **gap** | **gap** | **gap** | **gap** | **gap** | **gap** |
| audit        | Y       | Y       | **gap** | **gap** | **gap** | **gap** |
| backlog (×4) | Y       | Y       | **gap** | **gap** | **gap** | **gap** |
| memory       | Y       | Y       | Y       | **gap** | **gap** | Y       |
| requirement  | Y       | Y       | Y       | Y       | Y       | **gap** |

Commands to add:

| Gap                                        | Commands                                                                                       |
| ------------------------------------------ | ---------------------------------------------------------------------------------------------- |
| Plans have zero CLI presence               | `create plan --delta DE-XXX`, `list plans`, `show plan`, `view plan`, `edit plan`, `find plan` |
| Audits not individually addressable        | `show audit`, `view audit`, `edit audit`, `find audit`                                         |
| Backlog items not individually addressable | `show {issue,problem,improvement,risk}` + `view`/`edit`/`find` variants                        |
| Requirements missing find                  | `find requirement`                                                                             |
| Memories missing view/edit                 | `view memory`, `edit memory`                                                                   |

### Tier 2 — List flag inconsistencies

Baseline flags for `list` subcommands: `--json --format --filter --regexp -i --status --truncate`

| Command          | Missing vs baseline                                                |
| ---------------- | ------------------------------------------------------------------ |
| `list deltas`    | `--filter`                                                         |
| `list adrs`      | `--filter`                                                         |
| `list policies`  | `--filter`                                                         |
| `list standards` | `--filter`                                                         |
| `list cards`     | `--filter`, `--truncate` (note: cards use `--lane` not `--status`) |
| `list memories`  | `--filter`                                                         |

**Operational Constraints**: Each subcommand follows the established skinny-CLI
pattern (~20 lines). Domain registries and formatters already exist for most types.
`discover_backlog_items()` serves as the backlog registry. `ChangeRegistry(kind="audit")`
already loads audits.

**Dependencies**: None. All required registries exist.

## 4. Out of Scope

- **DR commands** — DRs live in delta bundle directories; `find delta` suffices
- **Phase list/show** — phases are in predictable paths under delta dirs
- **Contract CLI** — contracts have their own filesystem tree; separate delta if needed
- **ADR/policy/standard `complete`** — lifecycle transitions via `edit` is adequate
- **`strict_mode` runtime** — deferred per DE-038 gaps-to-adoption

## 5. Approach Overview

- **System Touchpoints**: `supekku/cli/{show,view,edit,find,create,list}.py`, `supekku/scripts/lib/formatters/`, `supekku/scripts/lib/changes/creation.py`
- **Key Changes**:
  1. Extract IP scaffolding from `create_delta()` into reusable `create_plan()` domain function; wire `create plan` CLI command
  2. Add `format_audit_details()` formatter; wire show/view/edit/find for audits
  3. Wire show/view/edit/find for backlog items using existing `format_backlog_details()` and `discover_backlog_items()`
  4. Wire find for requirements, view/edit for memories
  5. Backfill missing `--filter` flag on `list {deltas,adrs,policies,standards,memories}`; add `--filter`, `--status`, `--truncate` to `list cards`
- **Pattern**: Copy established command patterns (normalize ID → registry → lookup → format/open). No new abstractions needed — this is mechanical gap-filling.

## 6. Verification Strategy

- **VT**: Tests for each new CLI command (invoke via typer test client, verify output)
- **VT**: Tests for `create_plan()` domain function
- **VT**: Tests for `format_audit_details()` formatter
- **VA**: `just lint` + `just pylint` pass
- **VT**: Tests for `--filter` flag additions on list commands
- **Acceptance**: Every artifact type that has `create` also has at least `show`; all `list` subcommands support `--filter --status --truncate`; `uv run spec-driver <verb> --help` lists the new subcommands

## 7. Risks & Mitigations

- **Risk**: IP creation logic tightly coupled to delta creation – _Likelihood_: low – _Impact_: medium – _Mitigation_: Extract cleanly; existing code already has clear boundaries
- **Risk**: Duplication across CLI boilerplate — existing commands already repeat normalize→registry→lookup→format pattern ~20 times with no shared abstraction. Adding 25+ more commands compounds this. – _Likelihood_: high – _Impact_: medium – _Mitigation_: Investigate shared abstractions during phase planning; extract common helpers where warranted before bulk-adding commands. Write DRY, modular code — don't just copy-paste.
- **Risk**: Test duplication mirrors CLI duplication — each command getting near-identical integration tests. – _Likelihood_: high – _Impact_: medium – _Mitigation_: Unit-test shared capabilities at the domain/formatter layer; integration-test assembled CLI commands for wiring correctness only, not business logic.
