# DR-041 – CLI completeness — fill obvious command gaps

## 1. Executive Summary

- **Delta**: [DE-041](./DE-041.md)
- **Status**: draft
- **Synopsis**: Extract shared CLI helpers to eliminate boilerplate, then use them to fill all missing show/view/edit/find subcommands and list flag gaps.

## 2. Problem & Constraints

### Current Behaviour

The CLI has significant coverage gaps — several artifact types that can be
created lack basic addressability (show/view/edit/find). Implementation plans
have zero CLI presence despite being referenced constantly.

The existing commands work correctly but repeat the same patterns verbatim:

- **show.py**: 10 commands, each repeating mutual-exclusivity check + registry
  lookup + output-mode dispatch (~30 lines each)
- **view.py**: 8 commands, each repeating registry lookup + pager open (~20 lines each)
- **edit.py**: 8 commands, identical to view but with editor (~20 lines each)
- **find.py**: 8 commands, each repeating registry iterate + fnmatch (~15 lines each)

Adding 25+ new commands by copy-pasting this pattern would roughly double the
duplication.

### Drivers / Inputs

- DE-038 gaps-to-adoption.md §3 (Missing Automation)
- PROD-010 (CLI UX: consistent patterns across all artifacts)
- PROD-013 (CLI Artifact File Access: show/view/edit for all types)

### Constraints

- Must preserve existing command signatures — no breaking changes
- Skinny CLI pattern: commands must remain thin orchestration
- No premature abstraction: shared helpers must earn their keep by eliminating
  real, proven duplication (which exists ~30x over)

## 3. Architecture Intent

### Target Outcomes

1. Every artifact type with `create` also has `show` (with `--json --path --raw`)
2. Every artifact type with `show` also has `view` and `edit`
3. All `list` subcommands support `--filter` and `--truncate`; all except `list cards` support `--status` (cards use `--lane` for lane-based filtering)
4. Implementation plans can be created independently via `create plan --delta DE-XXX`
5. New commands built on shared helpers, not copy-pasted boilerplate
6. One existing command per verb group migrated to validate the helpers

### Guiding Principles

- **DRY the proven pattern**: The resolve→lookup→not-found-error→dispatch pattern
  is repeated ~30 times identically. Extract it once.
- **Explicit commands, shared internals**: Keep individual `@app.command` functions
  (no magic factories). Each command remains a readable 3-5 line function.
- **Unit test capabilities, integration test assemblies**: Test the shared helpers
  thoroughly; test commands only for wiring correctness.

## 4. Code Impact Summary

### 4.1 Shared helpers (`supekku/cli/common.py`)

#### ArtifactNotFoundError

Custom exception raised by `resolve_artifact` when lookup fails. Normalizes
error handling across all verb groups — commands catch one exception type
instead of `(FileNotFoundError, ValueError, KeyError)`.

```python
class ArtifactNotFoundError(Exception):
  """Raised when an artifact cannot be found by ID."""
  def __init__(self, artifact_type: str, artifact_id: str):
    self.artifact_type = artifact_type
    self.artifact_id = artifact_id
    super().__init__(f"{artifact_type} not found: {artifact_id}")
```

#### AmbiguousArtifactError

Raised when a single-target lookup (show/view/edit) matches multiple artifacts.
Includes all matching paths so the user can disambiguate.

```python
class AmbiguousArtifactError(Exception):
  """Raised when multiple artifacts match a single-target lookup."""
  def __init__(self, artifact_type: str, artifact_id: str, paths: list[Path]):
    self.artifact_type = artifact_type
    self.artifact_id = artifact_id
    self.paths = paths
    path_list = "\n  ".join(str(p) for p in paths)
    super().__init__(
      f"Ambiguous {artifact_type} ID {artifact_id}, matches:\n  {path_list}"
    )
```

#### ArtifactRef

Lightweight result type from `resolve_artifact`. The `record` field uses a
`Union` of all artifact model types for type safety.

```python
@dataclass(frozen=True)
class ArtifactRef:
  id: str
  path: Path
  record: ArtifactRecord  # Union[Spec, ChangeArtifact, Decision, ...]
```

#### resolve_artifact(artifact_type, raw_id, root) -> ArtifactRef

Handles the normalize→registry→lookup→not-found-error pattern for all artifact
types. Uses a dispatch table (not if/elif chain) for clean extensibility:

```python
ARTIFACT_RESOLVERS: dict[str, Callable] = {
  "spec":        lambda root, id: _resolve_spec(root, id),
  "delta":       lambda root, id: _resolve_change(root, id, "delta"),
  "revision":    lambda root, id: _resolve_change(root, id, "revision"),
  "audit":       lambda root, id: _resolve_change(root, id, "audit"),
  "adr":         lambda root, id: _resolve_decision(root, id),
  "policy":      lambda root, id: _resolve_policy(root, id),
  "standard":    lambda root, id: _resolve_standard(root, id),
  "requirement": lambda root, id: _resolve_requirement(root, id),
  "card":        lambda root, id: _resolve_card(root, id),
  "memory":      lambda root, id: _resolve_memory(root, id),
  "issue":       lambda root, id: _resolve_backlog(root, id, "issue"),
  "problem":     lambda root, id: _resolve_backlog(root, id, "problem"),
  "improvement": lambda root, id: _resolve_backlog(root, id, "improvement"),
  "risk":        lambda root, id: _resolve_backlog(root, id, "risk"),
  "plan":        lambda root, id: _resolve_plan(root, id),
}
```

Each resolver function is small and type-specific. Raises `ArtifactNotFoundError`
if lookup fails.

#### find_artifacts(artifact_type, pattern, root) -> Iterable[ArtifactRef]

Companion to `resolve_artifact` for the `find` verb group. Iterates all
artifacts of a type and yields `ArtifactRef` for those matching the pattern.

```python
def find_artifacts(artifact_type: str, pattern: str, root: Path) -> Iterable[ArtifactRef]:
  """Yield ArtifactRef for all artifacts matching fnmatch pattern."""
  ...
```

Uses a similar dispatch table to `resolve_artifact` but returns iterables
from registry `.collect()` / `.all_specs()` / `discover_backlog_items()`.

#### emit_artifact(ref, \*, json_output, path_only, raw_output, format_fn, json_fn)

Handles the output-mode dispatch pattern for show commands. Includes
mutual-exclusivity check.

**JSON serialization is always explicit** — `json_fn` is **required**, not
optional. Different artifact types have incompatible `to_dict` signatures
(`to_dict()` vs `to_dict(root)`) so a generic `hasattr`-based fallback is
unsafe and already crashes today (e.g. `show revision --json`).

Each show command must pass a `json_fn` that knows the correct serialization
for its artifact type. This is typically a thin lambda or a dedicated formatter.

```python
def emit_artifact(
  ref: ArtifactRef,
  *,
  json_output: bool = False,
  path_only: bool = False,
  raw_output: bool = False,
  format_fn: Callable[[Any], str],
  json_fn: Callable[[Any], str],
) -> None:
  if sum([json_output, path_only, raw_output]) > 1:
    typer.echo("Error: --json, --path, and --raw are mutually exclusive", err=True)
    raise typer.Exit(EXIT_FAILURE)

  if path_only:
    typer.echo(ref.path)
  elif raw_output:
    typer.echo(ref.path.read_text())
  elif json_output:
    typer.echo(json_fn(ref.record))
  else:
    typer.echo(format_fn(ref.record))
  raise typer.Exit(EXIT_SUCCESS)
```

Example usage at call site:

```python
# Types with to_dict(root):
emit_artifact(ref, ...,
  format_fn=format_audit_details,
  json_fn=lambda r: json.dumps(r.to_dict(root), indent=2))

# Types with to_dict() (no args):
emit_artifact(ref, ...,
  format_fn=format_card_details,
  json_fn=lambda r: json.dumps(r.to_dict(), indent=2))

# Types with dedicated JSON formatters:
emit_artifact(ref, ...,
  format_fn=lambda r: format_delta_details(r, root=root),
  json_fn=lambda r: format_delta_details_json(r, root=root))
```

### 4.2 New show commands (`supekku/cli/show.py`)

Each new command becomes ~5 lines:

```python
@app.command("audit")
def show_audit(
  audit_id: Annotated[str, typer.Argument(help="Audit ID (e.g., AUD-001)")],
  json_output: JsonOption = False,
  path_only: PathOption = False,
  raw_output: RawOption = False,
  root: RootOption = None,

  """Show detailed information about an audit."""
  ref = resolve_artifact("audit", audit_id, root)
```

### 4.6 Plan resolution and listing

Plans live at predictable paths: `change/deltas/DE-XXX-*/IP-XXX.md`.

**`_resolve_plan(root, raw_id)`**: Normalizes the ID (e.g. `41` → `IP-041`),
scans `change/deltas/*/IP-{id}.md`, loads frontmatter, returns `ArtifactRef`.

**`discover_plans(root) -> list[PlanSummary]`**: Domain function in
`changes/registry.py`. Scans `change/deltas/*/IP-*.md`, parses frontmatter
and the `plan.overview` YAML block (via `extract_plan_overview()`), returns
`PlanSummary` dataclasses. `PlanSummary` is a frozen dataclass with: `id`,
`status`, `name`, `slug`, `path`, `updated`, `delta_id`, `phases` (list of
phase overview dicts). This is distinct from `ChangeArtifact` because plans
need overview-block data (delta ref, phase list) that doesn't map to the
change artifact schema. Follows `discover_backlog_items()` pattern.

**`list plans`**: Calls `discover_plans()`, applies filters, passes results to
`format_plan_list_table()`. Standard flags: `--json --format --filter --status
--truncate`.

**`show plan`**: Uses `resolve_artifact("plan", ...)` + `emit_artifact` with
`format_plan_details` formatter. Plans are markdown files with frontmatter, so
`--path` and `--raw` work the same as other artifacts.

## 7. Design Decisions & Trade-offs

### DEC-041-01: Shared helpers over command factories

**Decision**: Extract `resolve_artifact()` + `emit_artifact()` + `find_artifacts()`
helpers. Keep explicit `@app.command` functions that call them.

**Rejected alternative**: Command factory/registration pattern
(`register_show_command(app, "audit", ...)`) — more magical, harder to debug,
obscures the command surface from IDE navigation and grep.

**Rationale**: The duplication is in the _implementation body_, not in the
_command declaration_. Helpers eliminate body duplication while keeping
declarations explicit and greppable.

### DEC-041-02: Targeted backlog lookup with duplicate-aware semantics

**Decision**: Add `find_backlog_items_by_id(item_id, root)` using predictable path
patterns (`backlog/{subdir}/{ID}-*/{ID}.md`) for targeted lookup.
`discover_backlog_items()` remains for list/find operations.

**Duplicate ID handling**: The repo has duplicate backlog IDs (e.g. IMPR-003 x2,
ISSUE-019 x2). Behavior varies by context:

- **Single-target commands** (`show`, `edit`, `view`): If lookup returns >1
  match, **error with all matching paths** so the user can disambiguate. Do not
  silently pick one.
- **Multi-target commands** (`list`, `find`): Return all matches. No error.
- **`validate`**: Warn on duplicate backlog IDs (not a hard error, but flagged).

This treats duplicates as a data quality issue surfaced by the system, not
prevented by it at the command layer.

**Rejected alternative**: Silently return first match — lossy, hides data issues.

**Rationale**: O(n) filesystem scan for `show issue ISSUE-001` is wasteful.
The directory naming convention makes targeted lookup straightforward. Erroring
on ambiguity for single-target commands is the safest contract.

### DEC-041-03: ArtifactNotFoundError normalizes error handling

**Decision**: `resolve_artifact` raises a single custom `ArtifactNotFoundError`.
Commands catch this one type and present consistent error messages.

**Rejected alternative**: Let each command handle `(FileNotFoundError, ValueError,
KeyError)` individually (current state).

**Rationale**: The resolver's contract should be: returns `ArtifactRef` or
raises `ArtifactNotFoundError`. Internal exceptions from registries are
implementation details that shouldn't leak to commands.

### DEC-041-04: Proof-of-concept migration of existing commands

**Decision**: Migrate the `revision` commands (show/view/edit/find) to use the
new helpers within this delta. Remaining existing commands are a follow-up.

**Rejected alternative A**: Migrate all existing commands (too large a scope).
**Rejected alternative B**: Migrate none (doesn't validate the design).

**Rationale**: Migrating one representative artifact type proves the helpers
handle real edge cases. `revision` is ideal — it uses `ChangeRegistry` +
`normalize_id`, which is the most common pattern.

### DEC-041-06: PlanSummary dataclass for plan discovery

**Decision**: `discover_plans()` returns `list[PlanSummary]`, a dedicated frozen
dataclass, rather than reusing `ChangeArtifact`.

**Rationale**: The `list plans` table needs columns from the `plan.overview`
YAML block (delta ID, phase list) — data that `ChangeArtifact` doesn't model.
Plans are structurally different from deltas/revisions/audits: they embed
structured overview blocks rather than relying solely on frontmatter +
relations. Forcing them into `ChangeArtifact` would require overloading the
`.plan` field (which on deltas holds the _child_ plan's overview — using it
on the plan itself would be circular).

**Alternative rejected**: Reuse `ChangeArtifact` with overview data stuffed
into a dict field — semantically confusing and fragile.

### DEC-041-05: Canonical requirement ID format and accepted aliases

**Decision**: The canonical requirement ID format is **dot-separated**:
`SPEC-009.FR-001`. The CLI normalizes colon-separated input (`SPEC-009:FR-001`)
to dot form before lookup. The `normalize_id` function (or a
requirement-specific variant) handles this.

**Evidence**:

- The actual requirements registry (`.spec-driver/registry/requirements.yaml`)
  uses dot form exclusively as dictionary keys
- CLI help text already shows dot form (`SPEC-009.FR-001`)
- Colon form appears only in one test fixture (`list_test.py`), which is a
  test data inconsistency, not an intentional alternate format

**Action during implementation**:

- Add requirement ID normalization: `SPEC-XXX:FR-YYY` → `SPEC-XXX.FR-YYY`
- Fix the `list_test.py` fixture to use dot form (test data bug)
- `resolve_artifact("requirement", ...)` and `find_artifacts("requirement", ...)`
  normalize before lookup
