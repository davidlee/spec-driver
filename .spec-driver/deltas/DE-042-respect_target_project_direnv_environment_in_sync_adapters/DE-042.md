## 1. Summary & Context

- **Product Spec(s)**: PROD-012 – Code Contract Documentation
- **Implementation Plan**: [IP-042](./IP-042.md)
- **Change Drivers**: `sync --contracts` produces empty contract directories (no files) for Go projects, even when toolchain is available

## 2. Motivation

Two distinct problems prevent Go contract generation from working:

### A. gomarkdoc invocation is broken (primary)

The Go adapter passes fully-qualified import paths (`im/cmd/im`) to gomarkdoc, which expects `./`-prefixed relative paths (`./cmd/im`). Additionally, subprocess calls omit `cwd=repo_root`, so gomarkdoc can't resolve packages. The `CalledProcessError` is silently caught and reported as `status="unchanged"`, hiding the failure completely.

**Evidence**: `gomarkdoc --format github im/cmd/im` → "invalid package at import path". `gomarkdoc --format github ./cmd/im` (from repo root) → success.

Affected code:

- `go.py:216` — `module_pkg = f"{module}/{unit.identifier}"` (wrong format)
- `go.py:277,298` — `subprocess.run(cmd, ...)` missing `cwd=repo_root`
- `go.py:319` — bare `except CalledProcessError` returns "unchanged" (silent failure)

### B. PATH not forwarded from calling shell (secondary)

When spec-driver is invoked from a directory without the target project's toolchain in PATH (e.g. running from spec-driver's own repo against a nix-direnv Go project), `shutil.which("go")` and subprocess calls can't find the tools. This is a real but separate concern from the invocation bug.

## 3. Scope & Objectives

- **Primary Outcomes (Phase 1 — gomarkdoc fix)**:
  - gomarkdoc receives `./`-prefixed relative paths
  - subprocess calls run with `cwd=repo_root`
  - gomarkdoc errors are surfaced, not swallowed
  - Go contracts are actually generated
- **Primary Outcomes (Phase 2 — PATH forwarding)**:
  - Subprocess calls and `which()` checks can use a caller-provided environment
  - Adapter base class supports optional `env` parameter
  - Engine resolves and passes through calling shell PATH
- **Operational Constraints**: No new dependencies; graceful degradation
- **Dependencies**: None

## 4. Out of Scope

- direnv-specific detection/integration (`direnv export json`)
- CLI flags for explicit tool paths (`--go-bin`)
- Python adapter changes (AST-based, no external toolchain)

## 5. Approach Overview

- **System Touchpoints**: `sync/adapters/go.py`, `core/go_utils.py`, `sync/adapters/base.py`, `sync/engine.py`
- **Phase 1 — Fix gomarkdoc invocation**:
  1. Change `_generate_variant` to use `./`-prefixed relative path instead of fully-qualified import path
  2. Add `cwd=self.repo_root` to all gomarkdoc subprocess calls
  3. Stop swallowing `CalledProcessError` — log/propagate the error
  4. Review whether `generate()` should use `unit.identifier` directly (already relative) instead of reconstructing via module name
- **Phase 2 — PATH forwarding**:
  1. Add optional `env` to `LanguageAdapter.__init__`
  2. Pass `env` through to subprocess calls and use for `which()` checks
  3. `SpecSyncEngine` captures and forwards calling environment
  4. Update `go_utils.py` functions to accept optional `env`

## 6. Verification Strategy

- **VT-042.1**: Unit test — gomarkdoc called with `./`-prefixed path and `cwd=repo_root`
- **VT-042.2**: Unit test — `CalledProcessError` from gomarkdoc is not silently swallowed
- **VT-042.3**: Unit test — adapter uses provided env in subprocess calls
- **VT-042.4**: Unit test — graceful fallback when env not provided
- **VH-042.1**: Manual — `sync --contracts` in `../test-driver` produces Go contract files
- **VH-042.2**: Manual — existing projects without direnv continue to work
- **Acceptance Criteria**:
  - `.contracts/` contains actual markdown files after sync, not just empty dirs
  - gomarkdoc failures are visible in sync output
  - No regression for standard installations

## 7. Risks & Mitigations

- **Risk**: Some gomarkdoc versions may accept import paths — _Likelihood_: low — _Impact_: low — _Mitigation_: `./` prefix is documented as the standard invocation; import path support was never reliable
- **Risk**: Surfacing previously-swallowed errors causes noisy output — _Likelihood_: medium — _Impact_: low — _Mitigation_: Use warning-level logging, don't abort the entire sync
