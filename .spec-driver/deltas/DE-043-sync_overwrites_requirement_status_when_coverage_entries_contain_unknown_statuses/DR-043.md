# DR-043 – Verification coverage validation and lifecycle guidance

## 1. Executive Summary

- **Delta**: [DE-043](./DE-043.md)
- **Status**: draft
- **Last Updated**: 2026-03-05
- **Synopsis**: Add validation at the sync boundary so unknown coverage statuses
  cannot corrupt derived requirement state, and create discoverable lifecycle
  guidance so agents don't make the same class of mistake again.

## 2. Problem & Constraints

### Current Behaviour

Coverage entries flow from `load_coverage_blocks()` into
`_apply_coverage_blocks()` without validation. Any entry — regardless of
whether its `status` field contains a value from the coverage vocabulary — is
passed to `_compute_status_from_coverage()`, which uses hardcoded string
comparisons. An unknown status like `deferred` triggers the mixed-status
fallback (`len(statuses) > 1` → `STATUS_IN_PROGRESS`), silently overwriting
the requirement's derived status on every sync.

```
load_coverage_blocks()          ← parse only, no validation
  → _apply_coverage_blocks()    ← no filtering, no warning
    → _compute_status_from_coverage()  ← hardcoded strings, no canonical ref
      → record.status = computed       ← silent overwrite
```

Meanwhile, `VerificationCoverageValidator` exists and correctly rejects unknown
statuses — but is only used in test code and manual validation paths, never
during sync.

### Three Competing `VALID_STATUSES` Sets

| Module                      | Set Name         | Values                                           | Used For               |
| --------------------------- | ---------------- | ------------------------------------------------ | ---------------------- |
| `requirements/lifecycle.py` | `VALID_STATUSES` | pending, in-progress, active, retired            | Requirement lifecycle  |
| `blocks/verification.py`    | `VALID_STATUSES` | planned, in-progress, verified, failed, blocked  | Coverage entry status  |
| `changes/lifecycle.py`      | `VALID_STATUSES` | draft, pending, in-progress, completed, deferred | Change artifact status |

All three are exported as `VALID_STATUSES`. The identical name across different
semantic domains is a direct contributor to the bug — agents and users naturally
conflate them.

### Drivers

- ISSUE-033: external bug report from `test-driver` project
- `mem.fact.spec-driver.status-enums`: already flagged the need for strict-mode
  validation across write/read paths
- PROD-009 guiding principle: "Later evidence overlays earlier declarations but
  never silently discards them"

### Constraints

- Minimal code change surface — this is a bug fix with guidance, not an
  architecture rework
- No schema changes, no new CLI commands
- Must not break existing valid coverage blocks

### Out of Scope

- Shared status-policy validator across all paths (future work)
- Adding `deferred` to coverage vocabulary
- Provenance/timestamp overlay (PROD-009 future feature)
- Tech spec for `blocks/verification.py`

## 3. Architecture Intent

### Target Outcomes

1. **Invalid coverage statuses cannot influence derived requirement state**
   (PROD-008.FR-001, PROD-009.FR-003). The sync boundary validates and
   filters entries before they reach status derivation.

2. **Status derivation references canonical vocabulary, not hardcoded strings**
   (defensive design). If the vocabulary evolves, derivation automatically
   follows.

3. **Agents can discover and apply correct lifecycle guidance** for common
   corner cases (PROD-009.FR-001). The guidance must be findable via
   `spec-driver list memories` with natural scope queries.

4. **Existing memory drift is corrected** — `mem.concept.spec-driver.verification`
   says `passed` where the canonical vocabulary says `verified`.

### Guiding Principles

- **Fail loud, fail early**: invalid data must produce a visible warning at
  the point of ingestion, not a silent downstream corruption.
- **Single vocabulary reference**: derivation logic should import and reference
  the canonical set, not duplicate it as string literals.
- **Guidance over enforcement**: the lifecycle guide informs agents; it does
  not add hard gates beyond what already exists in the validator.

### State Transitions / Lifecycle Impact

No changes to the lifecycle state machine itself. The coverage status vocabulary
(`planned → in-progress → verified | failed | blocked`) remains unchanged.
The change is about _enforcement_: unknown values are now rejected at ingestion
rather than silently processed.

### Design Detail: Entry Filtering in `_apply_coverage_blocks()`

The filtering happens at the point where entries are appended to `coverage_map`,
within each source-type loop. This is the natural boundary — entries are
already being checked for `req_id` presence; adding a status check is
consistent.

```python
# In each extraction loop (specs, deltas, plans, audits):
status = entry.get("status")
if status and status not in VALID_COVERAGE_STATUSES:
  print(
    f"WARNING: Coverage entry for {req_id} has unknown status "
    f"{status!r} (source: {source_file.name}); entry excluded "
    f"from status derivation",
    file=sys.stderr,
  )
  # Still store the entry in coverage_entries for visibility,
  # but mark it so derivation skips it
```

**Design choice**: invalid entries are still stored on `record.coverage_entries`
(for transparency — the user can see what was in the block) but are excluded
from the `coverage_map` that feeds `_compute_status_from_coverage()`. This
preserves the "record what was found" principle while preventing derivation
corruption.

### Design Detail: `_compute_status_from_coverage()` Refactor

Replace string literals with references to the canonical set. The logic
structure stays identical — this is a mechanical substitution that prevents
drift if the vocabulary ever changes.

```python
from supekku.scripts.lib.blocks.verification import (
  VALID_COVERAGE_STATUSES,
)

# In _compute_status_from_coverage:
statuses = {
  e.get("status") for e in entries
  if e.get("status") in VALID_COVERAGE_STATUSES
}
```

This is the defensive second layer: even if a caller bypasses
`_apply_coverage_blocks()`, derivation still ignores unknown statuses.

## 7. Design Decisions & Trade-offs

### DEC-043-01: Validate and filter at the boundary, not in derivation

**Decision**: Primary validation happens in `_apply_coverage_blocks()` (the
ingestion boundary). `_compute_status_from_coverage()` gets a secondary
defensive filter.

**Rationale**: Boundary validation is the "correct" location — it's where data
enters the system. The derivation filter is defence-in-depth. This follows the
existing pattern: `_apply_coverage_blocks()` already checks for missing
`req_id`.

**Alternative considered**: Only filter in derivation. Rejected because: (a)
invalid entries would still be stored in `coverage_entries` without any
warning, and (b) the ingestion boundary is the natural place to tell the user
"your data is wrong."

### DEC-043-02: Add `VALID_COVERAGE_STATUSES` alias in `verification.py`

**Decision**: Add `VALID_COVERAGE_STATUSES` as a module-level alias for
`VALID_STATUSES` in `verification.py`. Preserve the original name for backward
compatibility.

**Rationale**: The three-way name collision (`VALID_STATUSES` in three modules)
is a direct contributor to this class of bug. A disambiguated name makes imports
self-documenting: `from ...verification import VALID_COVERAGE_STATUSES` is
unambiguous at the call site.

**Alternative considered**: Rename `VALID_STATUSES` everywhere. Rejected as
too large a change for this delta; the alias achieves discoverability without
breaking existing imports.

### DEC-043-03: Lifecycle guidance as a memory record

**Decision**: Create a `mem.guide.spec-driver.requirement-lifecycle` memory
record answering the common corner cases, rather than a new spec, ADR, or
doc artifact.

**Rationale**: Memory records are the discovery mechanism agents actually use
(`spec-driver list memories`). A doc file or spec section would not surface
in the retrieval workflow. The memory can link to PROD-008, PROD-009, and
the lifecycle.py code as provenance.

**Content must cover**:

1. The three status vocabularies — what they are, where they apply, how they
   differ
2. "I want to change an established requirement" — requirements are
   spec-owned; changes flow through spec revisions (RE-XXX) or delta-driven
   spec updates, never direct registry edits
3. "A requirement starts in backlog and wants to move into a spec" — backlog
   issues/problems/improvements capture pain; when scoped into a delta, the
   delta's spec updates introduce the formal requirement; the requirement
   doesn't "transit" — it's authored fresh in the spec with traceability
   back to the backlog item
4. "When are lifecycle transitions expected?" — capture → deliver → observe
   cycle from PROD-008 §2

## 8. Open Questions

- [x] Should `_apply_coverage_blocks` run full `VerificationCoverageValidator`
      or just check statuses? — **Resolved**: just check statuses. Full validation
      would add overhead, require refactoring the per-entry extraction loops, and
      risk surfacing irrelevant structural warnings during sync. Status checking
      is the minimum necessary to prevent the specific bug. Full validation can be
      added in the future shared-validator work.
