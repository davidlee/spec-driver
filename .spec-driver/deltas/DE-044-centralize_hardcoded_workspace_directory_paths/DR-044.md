# DR-044 – Centralize hardcoded workspace directory paths

## 1. Executive Summary

- **Delta**: [DE-044](./DE-044.md)
- **Status**: draft
- **Last Updated**: 2026-03-05
- **Synopsis**: Extract the four workspace root directory names (`specify`, `change`, `backlog`, `memory`) from ~120 scattered string literals into named constants and helper functions in `core/paths.py`.

## 2. Problem & Constraints

### Current Behaviour

`paths.py` centralizes `.spec-driver` as `SPEC_DRIVER_DIR` with corresponding
`get_*` helpers. But the four workspace directory names are raw strings everywhere:

```python
# SpecRegistry.__init__
self.tech_dir = self.root / "specify" / "tech"

# ChangeRegistry.__init__
self.directory = self.root / "change" / _KIND_TO_DIR[kind]

# BacklogRegistry.create_entry
entry_dir = self.root / "backlog" / template.subdir / ...

# MemoryRegistry.__init__
self.directory = directory or (self.root / "memory")
```

Each registry, CLI command, and test file independently constructs these paths
from string literals.

### Drivers

- PROB-003: magic path strings throughout codebase
- POL-002: avoid magic strings and numbers

### Constraints

- Pure refactor — zero behaviour change
- Must not break any existing test
- Must not introduce a config mechanism yet (out of scope per DE-044)

### Out of Scope

- `workflow.toml` configurability (follow-up delta)
- Renaming directories
- Restructuring nesting (e.g. `specify/tech` → `specs/tech`)

## 3. Architecture Intent

### Target Outcomes

1. Every workspace directory name is defined exactly once as a constant
2. All production code references those constants (or helpers built on them)
3. Test fixtures use the same constants for path construction
4. Future rename = change one constant, run tests

### Guiding Principles

- Follow the existing `SPEC_DRIVER_DIR` / `get_spec_driver_root()` pattern
- Constants are plain strings (directory names), helpers return `Path` objects
- Subdirectory names within a workspace dir (e.g. `tech`, `product`, `deltas`,
  `issues`) remain local to the owning registry — they are domain knowledge,
  not infrastructure

### 4.3 `Workspace` — use helpers

`sync_requirements()` and property constructors replace inline path
construction with the same helpers the registries use.

### 4.4 CLI commands — use helpers or constants

~15 CLI files construct paths like `root / "change" / "deltas"` or
`root / "specify" / "tech"`. Each switches to the corresponding helper.

### 4.5 `install.py` — derive from constants

The `directories` list in `setup_workspace()` switches from string literals to
expressions built on the constants:

```python
directories = [
    f"{CHANGES_DIR}/{AUDITS_SUBDIR}",
    f"{CHANGES_DIR}/{DELTAS_SUBDIR}",
    f"{CHANGES_DIR}/{REVISIONS_SUBDIR}",
    f"{SPECS_DIR}/{DECISIONS_SUBDIR}",
    ...
]
```

### 4.6 Test fixtures — use constants

Test files constructing `root / "specify" / "tech"` etc. switch to the same
helpers or constants. This is the largest change by file count but is
mechanical.

## 6. Design Decisions & Trade-offs

### DD-044-01: Constants + helpers, not a config/registry object

**Decision**: Add plain string constants and `get_*()` functions to `paths.py`.
Do not introduce a `WorkspaceLayout` class or config-driven resolution.

**Rationale**: Follows the existing `SPEC_DRIVER_DIR` / `get_spec_driver_root()`
pattern. A config object would be premature — the follow-up delta can wrap
these constants in config resolution without changing call sites (helpers
would read config internally).

### DD-044-02: Subdirectory constants live in `paths.py` too

**Decision**: Subdirectory names like `tech`, `decisions`, `deltas`, `issues`
are also defined as constants in `paths.py` (e.g. `TECH_SPECS_SUBDIR`),
not left as local knowledge in each registry.

**Rationale**: These names are referenced across registries, CLI, install, and
tests. Leaving them scattered would defeat the purpose. The helpers
(`get_tech_specs_dir`, `get_deltas_dir`) compose root + subdir, keeping
call sites clean.

### DD-044-03: Helpers take `repo_root: Path | None` consistently

**Decision**: All new helpers follow the existing `get_spec_driver_root()`
signature — `repo_root=None` triggers auto-discovery via `find_repo_root()`.

**Rationale**: Consistency with existing API. Callers with a known root pass it
explicitly; CLI/scripts that need discovery pass `None`.
