## 1. Summary & Context

- **Change Drivers**: ISSUE-035 (Memory CLI UX improvements)
- **Governance**: ADR-002 (Do not store backlinks in frontmatter) — backlinks must be computed at runtime from forward references

Agents navigate the memory graph to build context during onboarding and task execution. Currently the graph is forward-only: a memory declares outgoing `[[links]]`, but there is no way to discover what links _to_ a given memory, nor to expand a memory's link neighbourhood without sequential manual calls.

## 2. Motivation

- An agent finds `mem.pattern.spec-driver.core-loop` and wants to know what depends on it. 12 memories link to it — currently undiscoverable without grep.
- An agent hits a signpost and needs the neighbourhood. It must chain N sequential `show memory` calls, parse links from each, repeat. Token-wasteful and slow.
- Default `show memory` gives metadata without body, and `--raw` gives the full file — but there's no way to get just the body without frontmatter. Agents that already have metadata from `list` or `show` must parse `--raw` output to extract content.
- These gaps make the memory graph effectively flat for agents: a list with forward-only links rather than a navigable graph.

## 3. Scope & Objectives

- **Primary Outcomes**:
  1. `list memories --links-to <id>`: runtime-computed backlinks (reverse edges) for any artifact, respecting ADR-002 (no frontmatter storage)
  2. `show memory --links-depth N`: expand outgoing link graph N levels, returning a subgraph summary in a single call
  3. `show memory --body-only/-b`: body text only, no frontmatter — for agents that already have metadata and want content without parsing overhead

## 4. Out of Scope

- Stored/cached backlink indices (ADR-002 prohibits frontmatter storage; registry caching is acceptable but not required for v1)
- `resolve links` improvements (verbose, scoped, backlog item resolution) — remain on ISSUE-035 for a separate delta
- Generic `--content-type` across all artifact types (ISSUE-036)
- Full graph visualization or export

## 5. Approach Overview

- **System Touchpoints**:
  - `supekku/scripts/lib/memory/links.py` — link parsing/resolution (pure functions)
  - `supekku/cli/resolve.py` — `_build_artifact_index()` for runtime backlink computation
  - `supekku/cli/list.py` — `list memories` command
  - `supekku/cli/show.py` — `show memory` command
  - `supekku/scripts/lib/formatters/memory_formatters.py` — display formatting

- **Key Decisions**:
  - **Backlink data source**: Parse all memory bodies at query time (not from resolved frontmatter links). Pure function, no prerequisites. Registry caching is the upgrade path if scale demands it — keep the computation isolated so the cache layer slots in without touching callers.
  - **`--links-depth` output format**: Compact table (id, name, depth) by default. `--tree` flag for indented tree view. `--json` for structured output.

- **Key Changes**:
  1. **Backlinks**: Compute reverse edges by parsing all memory bodies at query time. Add `--links-to` filter to `list memories`. Per ADR-002, this is a runtime query — no frontmatter mutation.
  2. **Link depth expansion**: `show memory --links-depth N` resolves outgoing links recursively to depth N. Default output is compact table (id + name + depth); `--tree` for indented tree. Reuses existing `resolve_all_links` and `parse_links` machinery. Capped depth with visited-set cycle protection.
  3. **Body-only mode**: `show memory --body-only/-b` outputs just the markdown body, stripping frontmatter. Complements existing `--raw` (full file) and `--json` (structured metadata).

## 6. Verification Strategy

- **VT**: Unit tests for backlink computation, depth expansion, summary formatting
- **VT**: CLI integration tests for each new flag
- **VA**: Agent walkthrough — boot, find a hub memory, use `--links-to` and `--links-depth` to navigate graph
- **Acceptance Criteria**:
  - `list memories --links-to mem.pattern.spec-driver.core-loop` returns ≥10 results
  - `show memory mem.signpost.spec-driver.overview --links-depth 1` returns all directly linked memories
  - `show memory <id> --body-only` output contains no frontmatter delimiters and starts with body content
  - Zero lint warnings, all existing tests pass

## 7. Risks & Mitigations

- **Risk**: Backlink computation is O(N×L) where N=memories, L=links per memory — _Likelihood_: low (corpus is ~35 records) – _Impact_: low – _Mitigation_: acceptable for current scale; if corpus grows, add registry-cached backlink index (ADR-002 permits registry caching)
- **Risk**: `--links-depth` unbounded recursion on cyclic graphs – _Likelihood_: medium – _Impact_: low – _Mitigation_: cap depth, track visited set
