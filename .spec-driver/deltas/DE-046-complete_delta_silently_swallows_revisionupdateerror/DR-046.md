# DR-046 – Fix silent failure in complete delta

## 1. Executive Summary

- **Delta**: [DE-046](./DE-046.md)
- **Status**: draft
- **Last Updated**: 2026-03-05
- **Synopsis**: Eliminate silent failure modes in `complete delta` by printing
  errors and tolerating pre-existing validation issues during targeted updates.

## 2. Problem & Constraints

### Current Behaviour

`complete_delta.py` line 307-308:

```python
except RevisionUpdateError:
    return False
```

Catches `RevisionUpdateError` without printing the error. The user gets exit
code 1 with zero output.

`updater.py` line 138-144:

```python
validator = RevisionBlockValidator()
validation_errors = validator.validate(data)
if validation_errors:
    raise RevisionUpdateError(...)
```

Validates the _entire_ revision block after modifying one field. Pre-existing
issues (e.g. PROD-_ in `spec_id` fields that expect SPEC-_) cause the update
to be rejected even though the actual change is valid.

### Drivers

- DE-043 closure blocked by this bug
- User experience: silent exit code 1 is unacceptable

### Constraints

- Minimal change surface — two files
- Must not weaken validation for genuinely new errors

### Out of Scope

- Fixing pre-existing data in revision files
- Overhauling `RevisionBlockValidator` schema rules

## 3. Architecture Intent

### Target Outcomes

1. Every error path in `complete delta` produces descriptive stderr output
2. Targeted lifecycle status updates succeed when pre-existing validation
   issues exist in unrelated fields

### Guiding Principles

- **No silent failures**: if a command exits non-zero, it must say why
- **Targeted validation**: changing field X should not fail because field Y
  was already invalid

## 4. Code Impact Summary

| Path                                              | Current State                                             | Target State                                                             |
| ------------------------------------------------- | --------------------------------------------------------- | ------------------------------------------------------------------------ |
| `supekku/scripts/complete_delta.py` L307-308      | `except RevisionUpdateError: return False` (silent)       | Print error to stderr, then return False                                 |
| `supekku/scripts/lib/changes/updater.py` L137-144 | Validates entire block post-update; rejects if any errors | Validate before AND after; only reject if update introduced _new_ errors |

### Design Detail: Diff-Based Validation (DEC-046-01)

```python
# Before update
pre_errors = set(_render_errors(validator.validate(data)))

# Apply update
lifecycle["status"] = new_status

# After update
post_errors = set(_render_errors(validator.validate(data)))

# Only new errors matter
new_errors = post_errors - pre_errors
if new_errors:
    raise RevisionUpdateError(
        "Update introduced validation errors:\n" + "\n".join(sorted(new_errors))
    )
```

This allows the update to proceed when only pre-existing issues are present.

### Design Detail: Descriptive Errors (DEC-046-02)

Every `except` and `return False` path in `complete_delta.py` must print to
stderr. Audit all error paths, not just the one we found.

## 7. Design Decisions & Trade-offs

### DEC-046-01: Diff-based validation

**Decision**: Validate block before and after update; only reject if the update
introduced new errors.

**Rationale**: The alternative — skip post-update validation entirely — would
miss genuinely invalid changes. The diff approach is conservative: it only
relaxes the gate for errors that were already present.

**Trade-off**: Slightly more expensive (two validation passes). Acceptable for
a non-hot-path CLI command.

### DEC-046-02: All error paths must print to stderr

**Decision**: Audit all early-return paths in `complete_delta.py` and ensure
each one prints a descriptive message to stderr before returning non-zero.

**Rationale**: Silent failures are always bugs in CLI tools. Even if the user
doesn't read stderr, tooling and agents can.
