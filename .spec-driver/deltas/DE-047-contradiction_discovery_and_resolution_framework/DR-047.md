# DR-047 – Contradiction discovery and resolution framework

## 1. Executive Summary

- **Delta**: [DE-047](./DE-047.md)
- **Status**: draft
- **Owners**: david
- **Last Updated**: 2026-03-05
- **Synopsis**: Systematic process to surface, resolve, and codify
  contradictions across the PROD spec corpus using a drift ledger, before any
  spec can be promoted to `active`.

## 2. Problem & Constraints

- **Current Behaviour**: 16 PROD specs (15 draft, 1 active) contain evolved and
  sometimes contradictory design intent. Terms are used inconsistently. Some
  specs reflect earlier architectural thinking that has since been superseded by
  ADRs or informal decisions. Agents encounter conflicting guidance and cannot
  treat any single document as authoritative.

- **Drivers / Inputs**:
  - ISSUE-039 — umbrella for spec corpus reconciliation
  - ADR-003 (unit vs assembly specs) — established after several PROD specs
    were written
  - Informal evolution of concepts like "ceremony modes", "contracts vs specs",
    "unit vs assembly" that may not be consistently reflected

- **Constraints / Guardrails**:
  - Contradiction resolution requires human design decisions — cannot be fully
    automated
  - Must not change design intent; only surface and resolve conflicts
  - Resolutions must be durable (ADRs or spec revisions), not ad-hoc edits

- **Out of Scope**:
  - Promoting specs to `active` (subsequent delta)
  - TECH spec work (Phase C/D)
  - Code changes
  - DL tooling (CLI, registry) — uses schema manually per IMPR-007

## 3. Architecture Intent

- **Target Outcomes**:
  - A populated **drift ledger** (DL-047) documenting all cross-document
    conflicts, stale claims, missing decisions, and ambiguous intent
  - Each entry **resolved** via ADR, spec revision, or editorial fix
  - All PROD specs internally consistent and aligned with accepted ADRs

- **Guiding Principles**:
  - Surface contradictions systematically — don't rely on stumbling across them
  - Batch related entries by topic so one design decision resolves several
  - Prefer ADRs for decisions that apply across 3+ specs
  - Prefer spec revisions for conflicts local to 1–2 specs

## 4. Drift Ledger

The primary output artifact is `drift/DL-047-spec-corpus-reconciliation.md`,
a drift ledger following the schema defined in
[IMPR-007](../../backlog/improvements/IMPR-007-drift_ledger_primitive/IMPR-007.md).

Entry types expected:

- `contradiction` — two documents assert incompatible things
- `stale_claim` — document asserts something no longer true
- `missing_decision` — no document addresses a question that needs answering
- `ambiguous_intent` — a document's meaning is unclear

Resolution routing:

- **ADR**: affects 3+ specs or redefines a term/concept
- **RE**: local to 1–2 specs
- **editorial**: wording fix, no design decision needed

The agent survey (VA-047-001) must cover every document in this list:

### Topic areas (expected)

- **Taxonomy**: unit vs assembly, file vs package, category/c4_level semantics
- **Contracts**: role, authority (canonical vs derived), storage location
- **Lifecycle**: requirement statuses, spec statuses, promotion criteria
- **Governance**: ceremony modes, when deltas are required, revision-first vs
  delta-first
- **Workflow**: agent autonomy boundaries, what agents should/shouldn't do
- **CLI**: flag conventions, output modes, progressive disclosure

## 5. Process

### Phase 1: Agent survey (VA-047-001)

1. Read every document in the corpus inventory
2. For each, extract key claims: definitions, lifecycle rules, responsibility
   boundaries, process requirements, assumptions
3. Cross-reference: for each concept, compare all documents that mention it;
   flag disagreements, ambiguities, incompatible assumptions
4. Populate `DL-047` with entries at `triaged` status (severity, topic, owner
   assigned)

**Gate**: every corpus document referenced by at least one source entry.

### Phase 2: Human resolution (VH-047-001)

1. Review ledger entries grouped by topic
2. For each batch: make the design call
3. Codify: create ADR or note the spec revision needed
4. Update entry status to `adjudicated`, then `resolved` with resolution_ref
   and affected_artifacts

**Gate**: all `blocking` and `significant` entries resolved. `cosmetic` entries
may be deferred or resolved editorially during Phase B (PROD promotion).

### Phase 3: Application

For each resolved entry:

1. If ADR: create via `spec-driver create adr`
2. If spec revision: create RE-xxx
3. Update affected PROD specs to align
4. Editorial fixes applied directly

### Phase 4: Consistency check (VA-047-002)

Agent re-scans the updated corpus to verify:

- No remaining open `blocking` or `significant` entries in DL-047
- New ADRs reflected in all relevant PROD specs
- No new contradictions introduced by resolution edits

**Gate**: DL-047 has 0 open entries at `blocking` or `significant` severity.

## 7. Design Decisions & Trade-offs

- **Drift ledger over ad-hoc register**: Uses IMPR-007 schema rather than a
  bespoke format. The ledger persists as a durable record of what was found
  and how it was resolved. Useful for future agents.

- **Batch by topic, not by document**: Resolving all "lifecycle semantics"
  entries together is more efficient than going doc-by-doc, because one
  design decision often resolves multiple entries.

- **ADR threshold**: 3+ specs affected, or a genuinely architectural question.
  Everything else is a spec revision or editorial fix.

- **Manual DL usage**: No CLI tooling for this delta. The ledger file is
  created and edited by hand, following the IMPR-007 schema as an operational
  guide. Tooling built later if the primitive proves its value.
