# DR-048 – Configurable workspace directory paths via workflow.toml

## 1. Executive Summary

- **Delta**: [DE-048](./DE-048.md)
- **Status**: draft
- **Owners**: david
- **Last Updated**: 2026-03-06
- **Synopsis**: Add `[dirs]` config section to `workflow.toml` and wire it into
  `paths.py` helpers via `init_paths(config)` / `reset_paths()`, enabling
  workspace directory names to be overridden without code changes.

## 2. Problem & Constraints

- **Current Behaviour**: DE-044 centralized workspace directory names as
  module-level string constants in `core/paths.py`. 44 production files import
  these constants. But the values are hardcoded literals — there is no mechanism
  to override them via configuration.

- **Drivers / Inputs**:
  - PROB-003 — remaining success criterion: "optional: directory names
    configurable via workflow.toml"
  - IMPR-008 — workspace layout consolidation (depends on configurability)
  - ADR-006 — consolidate workspace dirs under `.spec-driver/`
  - POL-002 — avoid magic strings

- **Constraints / Guardrails**:
  - Non-breaking: defaults must match current layout exactly
  - No directory moves, no installer changes, no migration
  - `init_paths()` must be safe to skip — helpers fall back to current defaults

- **Out of Scope**:
  - Changing default directory locations (DE-049)
  - Migration tooling
  - Convenience symlinks
  - Installer changes

## 3. Architecture Intent

- **Target Outcomes**:
  - `workflow.toml` supports `[dirs]` for overriding any workspace directory name
  - `paths.py` helpers resolve from config when initialized, from defaults otherwise
  - Tests can exercise both default and custom layouts cleanly

- **Guiding Principles**:
  - Minimal API surface: two functions (`init_paths`, `reset_paths`)
  - No change to helper signatures — callers are unaware of config
  - Config is read once at startup, not per-call

## 4. Code Impact Summary

| Path                  | Current State                                                                | Target State                                                                                                                              |
| --------------------- | ---------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------- |
| `core/paths.py`       | Module-level string constants; `get_*_dir()` helpers reference them directly | Constants become mutable module state; `init_paths(config)` overrides from `[dirs]`; `reset_paths()` restores defaults; helpers unchanged |
| `core/config.py`      | `DEFAULT_CONFIG` has no `[dirs]` section                                     | Add `dirs` section with all directory name defaults matching current constants                                                            |
| `core/config_test.py` | Tests config loading and merge                                               | Add tests for `[dirs]` section parsing, partial overrides, invalid values                                                                 |
| `core/paths_test.py`  | (new)                                                                        | Tests for `init_paths()`, `reset_paths()`, custom dir names, fallback behavior                                                            |
| `test_base.py`        | `RepoTestCase` creates temp repos                                            | Add `reset_paths()` call in `tearDown` to prevent state leakage                                                                           |
| CLI entry point       | Loads config, no path init                                                   | Add `init_paths(config)` call after config loading                                                                                        |

  DE-049 (layout consolidation — depends on this)

## 7. Design Decisions & Trade-offs

- **Module-level init over per-call config reads**: `init_paths(config)` sets
  module state once at startup. Avoids adding `repo_root` dependency to every
  config lookup (chicken-and-egg with `find_repo_root()`). Avoids changing
  signatures of 19 helper functions. Trade-off: module-level mutable state
  requires `reset_paths()` in test teardown.

- **Fallback to defaults when uninitialized**: If `init_paths()` is never
  called (e.g. in tests that don't care about config), helpers use current
  constant values. This means zero existing test breakage.

- **Config key naming**: Keys in `[dirs]` match the constant names but
  lowercase without suffix: `specs`, `changes`, `backlog`, `memory`,
  `tech_specs`, `decisions`, `deltas`, etc. Direct mapping, no indirection.

## 8. Open Questions

None — spike resolved all design questions.

## 9. Rollout & Operational Notes

- **Migration**: None. Defaults are unchanged. No existing workflow.toml files
  need updating.
- **Recovery**: `reset_paths()` restores defaults. Removing `[dirs]` from
  workflow.toml reverts to default behavior.
