## 1. Summary & Context
- **Change Drivers**: IMPR-008 (workspace layout consolidation), ADR-004 (pending acceptance)
- **Implementation Plan**: [IP-049](./IP-049.md)
- **Predecessor**: DE-048 (configurable paths via `[dirs]`)

## 2. Motivation

The current layout scatters four top-level directories (`specify/`, `change/`,
`backlog/`, `memory/`) across the repo root. This feels invasive for public
adoption. The intermediate grouping dirs (`specify/`, `change/`) add nesting
without adding meaning.

This delta flattens the layout under `.spec-driver/`:

```
.spec-driver/
  config/           # user-serviceable: workflow.toml, doctrine.md, templates/
  tech/             # technical specs
  product/          # product specs
  decisions/        # ADRs
  policies/
  standards/
  deltas/           # change artifacts
  revisions/
  audits/
  backlog/          # issues, problems, improvements, risks
  memory/           # agent memories
  contracts/        # generated API docs
  registry/         # generated YAML indices
  agents/           # generated agent guidance
  hooks/            # lifecycle hooks
```

Backward-compat symlinks at repo root (`specify/ → .spec-driver/tech/` etc.)
keep existing cross-references, agent instructions, and memories working
without mass edits.

## 3. Scope & Objectives
- **Primary Outcomes**:
  - New installs use flattened `.spec-driver/` layout
  - `config/` separates user-serviceable files from managed internals
  - Installer creates optional convenience symlinks at repo root
  - `[symlinks]` section in `workflow.toml` controls which symlinks to create
  - Existing cross-references work via backward-compat symlinks
- **Operational Constraints**: ADR-004 must be accepted first
- **Dependencies**: DE-048 (configurable paths), ADR-004 (layout decision)

## 4. Out of Scope
- Migration tooling for existing workspaces (deferred — YAGNI at current adoption)
- Content cross-reference mass-update (~357 files — symlinks handle this)
- `kanban/` — intentionally stays at repo root

## 5. Approach Overview
- **System Touchpoints**: `paths.py` defaults, `install.py`, `config.py`, symlink builders, agent instructions, templates
- **Key Changes**:
  - Update default constant values in `paths.py` (or default `[dirs]` config)
  - Update `initialize_workspace()` to create dirs under `.spec-driver/`
  - Add `config/` grouping for `workflow.toml`, `doctrine.md`, `templates/`
  - Add symlink creation step to installer, controlled by `[symlinks]` config
  - Update agent instructions (~11 files), templates (~11 files)
  - Rebuild spec index and contract mirror symlinks (auto via builders)
  - Registry paths auto-regenerated by sync

## 6. Verification Strategy
- **VT-049-install**: Fresh install creates correct directory structure
- **VT-049-symlinks**: Backward-compat symlinks resolve to correct targets
- **VT-049-regression**: Full test suite passes with new defaults
- **VA-049-agent**: Agent instructions reference correct paths
- **Acceptance Criteria**: `spec-driver install` in clean dir produces flattened layout; all CLI commands work; `just` green

## 7. Risks & Mitigations
- **Risk**: Symlink builders produce broken links after path changes — *Likelihood*: low — *Impact*: medium — *Mitigation*: builders rebuild from scratch; test with fresh install
- **Risk**: Agent instructions/memories reference old paths — *Likelihood*: high — *Impact*: low — *Mitigation*: backward-compat symlinks; manual update of ~11 instruction files
- **Risk**: Large diff obscures review — *Likelihood*: medium — *Impact*: low — *Mitigation*: phase structural changes separately from content updates

## 8. Follow-ups & Tracking

- **Deferred**: Content cross-reference cleanup (~357 files) — symlinks handle this
