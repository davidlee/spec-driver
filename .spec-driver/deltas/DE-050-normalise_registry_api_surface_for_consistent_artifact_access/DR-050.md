# DR-050 – Normalise registry API surface for consistent artifact access

## 1. Executive Summary

- **Delta**: [DE-050](./DE-050.md)
- **Status**: draft
- **Last Updated**: 2026-03-06
- **Synopsis**: Bring all registries into conformance with ADR-009's standard
  registry API convention via additive changes, enabling generic artifact access
  for CLI consumers and IMPR-009.

## 2. Problem & Constraints

- **Current Behaviour**: Nine registries present 5+ incompatible patterns for ID
  lookup, collection, iteration, and construction. `cli/common.py` contains
  per-registry workarounds: RequirementsRegistry needs a path not root
  (lines 396-397), ChangeRegistry requires `collect()` + dict access for single
  lookups (lines 348-349), CardRegistry needs try/except around a raising
  resolver (lines 409-412), and BacklogRegistry uses bare functions with
  ambiguity handling (lines 461-465).

- **Drivers**: ISSUE-019 (registry API drift), IMPR-009 (TUI needs consistent
  access), ADR-009 (now accepted — defines the target convention).

- **Constraints**:
  - Additive changes only — no breaking removals, no constructor signature changes that break positional callers
  - All existing tests must continue to pass
  - Both linters must pass (`just lint` + `just pylint`)
  - Deprecation aliases for renamed methods
  - Constructor changes must preserve old call patterns alongside new keyword-only `root`

- **Out of Scope**:
  - Class-ifying BacklogRegistry (Tier 3 — separate delta)
  - Formal Protocol/ABC (ADR-009 explicitly defers)
  - Backfilling stub specs SPEC-115, SPEC-117, SPEC-123
  - RequirementsRegistry internal refactoring
  - Persistence naming normalisation (`sync` vs `save`)

## 3. Architecture Intent

- **Target Outcomes**:
  - Every class-based registry exposes `find(id) -> Record | None` per ADR-009
  - Every class-based registry exposes `collect() -> dict[str, Record]`
  - Every class-based registry exposes `iter(status=None)` (CardRegistry: `iter(lane=None)`)
  - Every class-based registry exposes `filter(...) -> list[Record]` with domain-specific params per ADR-009
  - BacklogRegistry gains a `find_item(id, root) -> BacklogItem | None` function (preserves ambiguity warning)
  - `cli/common.py` resolvers simplified to use `find()` uniformly
  - Constructor signatures gain `root` keyword where missing; existing positional signatures preserved

- **Guiding Principles**:
  - Derive convention from Group A (Decision, Policy, Standard, Memory) — don't
    invent new abstractions
  - Additive first; deprecate rather than remove
  - Domain-specific extras (ChangeRegistry `kind`, Memory `directory`) are fine

- **State Transitions**: No lifecycle/status changes. This is a pure API surface
  normalisation.

## 7. Design Decisions & Trade-offs

- **DEC-050-01**: Additive only. No method removals. No constructor signature
  changes that break existing positional callers. `get()` on SpecRegistry gets a
  deprecation `warnings.warn()` pointing to `find()`.

- **DEC-050-02**: BacklogRegistry stays functional. The new `find_item()`
  convenience function returns the first match or `None`, but emits
  `warnings.warn()` if multiple matches are found. This preserves the
  intentional ambiguity semantics of `find_backlog_items_by_id()` (which
  returns a list because duplicates are possible across backlog subdirs).
  The common.py backlog resolver continues using the list-based function
  with `AmbiguousArtifactError` — `find_item()` is for simpler callers
  that don't need strict ambiguity handling.

- **DEC-050-03**: RequirementsRegistry constructor preserves the existing
  positional `registry_path` parameter. A new `root` keyword parameter is
  added alongside: when `root` is provided and `registry_path` is not, the
  registry auto-discovers the path via `get_registry_dir(root)`. This means
  `RequirementsRegistry(path)` continues to work, and
  `RequirementsRegistry(root=root)` becomes the new canonical form. No
  existing caller breaks.

- **DEC-050-04**: SpecRegistry `get()` → `find()` rename uses deprecation alias
  pattern. `get()` calls `find()` internally and emits a `DeprecationWarning`.

- **DEC-050-05**: CardRegistry has `lane`, not `status`. The ADR-009 convention
  specifies `iter(status=None)` as the standard iteration surface, but Card's
  domain concept is lane (backlog/doing/done), not a lifecycle status. The
  CardRegistry therefore exposes `iter(lane=None)` and `filter(lane=None)` as
  a documented semantic concession. This is honest about the taxonomy mismatch
  rather than mapping lane to status and confusing callers.

- **DEC-050-06**: Constructor changes preserve existing call patterns. For
  CardRegistry, `__init__(self, root: Path)` stays positional to avoid breaking
  callers; `root` additionally accepts keyword form. For RequirementsRegistry,
  see DEC-050-03. The goal is that every existing `FooRegistry(x)` call
  continues to work unchanged.

## 8. Open Questions

- [x] Should SpecRegistry switch from eager to lazy loading? — **No.** ADR-009
      allows both strategies. Eager is appropriate for the small spec corpus.
- [x] Should `filter()` be added to all registries in Tier 1? — **Yes.** ADR-009
      requires `filter()` as part of the standard surface. Filter parameters remain
      domain-specific per ADR-009 §5, but every class-based registry must expose the
      method.

## 9. Rollout & Operational Notes

- **Migration**: Purely additive. No data migration, no file moves. No
  constructor signatures are changed in a breaking way — new keyword paths are
  added alongside existing positional ones (see DEC-050-03, DEC-050-06).
- **Deprecation**: `SpecRegistry.get()` emits `DeprecationWarning`. Plan removal
  in a future delta after callers migrate.
- **Recovery**: All changes are additive method additions and constructor
  overloads. Rollback is revert commits — no existing call site is modified
  in a way that can't be undone.
