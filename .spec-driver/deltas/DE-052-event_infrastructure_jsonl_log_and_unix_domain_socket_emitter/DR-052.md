# DR-052 – Event infrastructure: JSONL log and Unix domain socket emitter

## 1. Executive Summary
- **Delta**: [DE-052](./DE-052.md)
- **Status**: draft
- **Last Updated**: 2026-03-07
- **Synopsis**: Add structured event emission at the process boundary of every
  CLI command invocation, writing JSONL to a persistent log and
  fire-and-forgetting a datagram to a Unix domain socket for real-time
  consumers. Emission is gated by a command-invocation flag so that help,
  completion, and no-args paths are excluded.

## 2. Problem & Constraints

- **Current Behaviour**: CLI invocations produce no structured trace. There is
  no way to follow agent activity, audit command history, or feed real-time
  updates to a TUI.
- **Drivers / Inputs**: IMPR-009 (TUI dashboard — track mode needs an event
  stream). Events are independently useful for debugging and audit.
- **Constraints / Guardrails**:
  - Stdlib only — no new runtime dependencies
  - Zero-cost when no TUI is running (socket send fails silently)
  - Event emission must not perceptibly slow CLI commands
  - Non-blocking — never wait for socket delivery
  - Errors in event emission must never cause a CLI command to fail
- **Out of Scope**:
  - TUI socket listener (DE-054)
  - Event replay / log rotation
  - Configurable event scope filtering (defer until "mutation" has a formal
    definition in the command model)

## 3. Architecture Intent

- **Target Outcomes**:
  - Every leaf-command invocation produces a structured JSONL event
  - Real-time consumers can listen on a Unix domain socket
  - Agent sessions are attributable via `SPEC_DRIVER_SESSION` env var
  - Non-zero exits are captured as error events (training/feedback data)
  - Mutation commands that create/modify artifacts report touched IDs
- **Guiding Principles**:
  - Skinny CLI: emission is a cross-cutting concern at the process boundary,
    not scattered across command functions
  - Fail-silent: event infrastructure is advisory, never load-bearing
  - Separation of concerns: `core/events.py` owns emission logic; `main()`
    wires it at the boundary; consumers (TUI) are decoupled
- **Coverage — what is logged**:
  - Any invocation where a leaf command body actually executes (detected via
    the `_command_invoked` flag — see DEC-052-01)
  - Both successful (exit 0) and failed (exit nonzero) commands
- **Coverage exclusions** (intentionally not logged):
  - `--help` / `--version` — Click intercepts as eager parameters, exits
    before any command body runs; flag is never set
  - `no_args_is_help` on groups — Click raises `NoArgsIsHelpError` during
    `parse_args` (before `invoke`); no command body runs; flag is never set.
    This covers bare `spec-driver`, `spec-driver create`, `spec-driver list`,
    and all other groups with `no_args_is_help=True`
  - Shell completion callbacks — exit before `main()` is reached
  - Invalid command / parse errors — `UsageError` raised during parsing,
    before `Command.invoke`; flag is never set
- **Best-effort guarantees**:
  - JSONL append is best-effort local-workspace telemetry, not an audit-grade
    guarantee. Small single-line appends with `O_APPEND` are atomic on common
    local Linux filesystems (ext4, btrfs, tmpfs). Concurrent agents writing to
    the same log may interleave on NFS or unusual filesystems.
  - Socket delivery is fire-and-forget with no acknowledgement.
- **State Transitions / Lifecycle Impact**: None — events are append-only
  observations, not lifecycle state.

## 7. Design Decisions & Trade-offs

### DEC-052-01: Process-boundary wrapper gated by command-invocation flag

**Decision**: Emit events at the process boundary in `main()` (the console
entrypoint at `supekku/cli/main.py:148`), wrapping `app()` to capture all
exit paths. Emission is gated by a `_command_invoked` flag that is only set
when a leaf `Command.invoke()` fires — ensuring help paths, group-level
no-args, parse errors, and completion callbacks never produce events.

**Why not result_callback**: Click's `result_callback` fires inside
`Group.invoke()` on normal return only. Every command in this codebase exits
via `raise typer.Exit()` (306 instances, zero normal returns). `typer.Exit`
(a `click.exceptions.Exit`) propagates out of `invoke()` and is caught in
Click's `main()` at a higher level, completely bypassing `_process_result()`.
The callback would never fire.

**The command-invocation flag**:

A module-level boolean `_command_invoked` in `core/events.py`, initially
`False`. It is set to `True` when a leaf `click.Command.invoke()` executes —
meaning a command's callback body actually ran. This is the definitive
signal that a real command executed, not a help/parse/completion path.

The flag is set by intercepting `click.Command.invoke()`. The exact mechanism
(custom `Command` subclass via typer's `cls` parameter, or a one-time
monkey-patch of `Command.invoke`) is an implementation detail. The contract
is:

- **Flag is set** iff `click.Command.invoke()` is entered (i.e., a leaf
  command body begins execution)
- **Flag is NOT set** for:
  - `--help` / `--version` (Click eager parameters, handled before invoke)
  - `no_args_is_help` on any group — Click raises `NoArgsIsHelpError` during
    `Group.parse_args()` (core.py:1803), before `Group.invoke()` dispatches
    to a subcommand, so `Command.invoke()` is never reached
  - `UsageError` / `BadParameter` during argument parsing
  - Shell completion (exits before Click's `main()`)

This correctly handles the nested group case: `spec-driver create` (no
subcommand) triggers `no_args_is_help` on the `create` group, which raises
`NoArgsIsHelpError` before any leaf `Command.invoke()` fires. The flag
remains `False`. But `spec-driver create delta "test"` dispatches through the
`create` group to the `delta` leaf command, which fires `Command.invoke()`,
setting the flag.

**Boundary contract — exact behavior by exit path**:

Click's `BaseCommand.main()` (standalone_mode=True) converts all outcomes
to `sys.exit()` calls. The wrapper catches `SystemExit` and `BaseException`
to read the exit code. It only emits if `_command_invoked is True`.

| Origin | Click handling | Exit code | Flag set? | Event emitted? |
| --- | --- | --- | --- | --- |
| `raise typer.Exit(0)` | `sys.exit(0)` | 0 | Yes | Yes, `status: "ok"` |
| `raise typer.Exit(1)` | `sys.exit(1)` | 1 | Yes | Yes, `status: "error"` |
| Normal return from command | `ctx.exit()` → `sys.exit(0)` | 0 | Yes | Yes, `status: "ok"` |
| `ClickException` (UsageError, BadParameter) after invoke | `e.show()` → `sys.exit(e.exit_code)` | 1–2 | Yes | Yes, `status: "error"` |
| `KeyboardInterrupt` / `EOFError` during command | → `Abort()` → `sys.exit(1)` | 1 | Yes | Yes, `status: "error"` |
| Unhandled exception during command | traceback → `sys.exit(1)` | 1 | Yes | Yes, `status: "error"` |
| `--help` / `--version` | `sys.exit(0)` | 0 | No | **No** |
| `no_args_is_help` (any group) | `NoArgsIsHelpError` → `sys.exit(2)` | 2 | No | **No** |
| Parse error (no invoke reached) | `UsageError` → `sys.exit(2)` | 2 | No | **No** |
| Shell completion | exits before wrapper | n/a | No | **No** |
| `BaseException` bypassing Click | propagates | 1 | Maybe | Only if flag was set |

Note on `KeyboardInterrupt`: Click catches `KeyboardInterrupt` at
core.py:1394 and converts it to `Abort()` → `sys.exit(1)`. By the time the
wrapper sees the `SystemExit`, interrupt is indistinguishable from any other
error. The status is `"error"`, not `"interrupted"` — there is no
`"interrupted"` status in the v1 schema because the wrapper cannot reliably
detect it.

**Implementation shape**:

```python
from supekku.scripts.lib.core.events import (
    command_was_invoked,
    emit_event,
    mark_command_invoked,
)

# Intercept leaf Command.invoke to set the flag
_original_command_invoke = click.Command.invoke

def _tracking_invoke(self, ctx):
    mark_command_invoked()
    return _original_command_invoke(self, ctx)

click.Command.invoke = _tracking_invoke


def main() -> None:
    try:
        app()
    except SystemExit as exc:
        if command_was_invoked():
            _emit(sys.argv, exc.code)
        raise
    except BaseException:
        if command_was_invoked():
            _emit(sys.argv, 1)
        raise


def _emit(argv: list[str], exit_code: int | None) -> None:
    code = exit_code if isinstance(exit_code, int) else (1 if exit_code else 0)
    status = "ok" if code == 0 else "error"
    emit_event(argv=argv, exit_code=code, status=status)
```

**Why monkey-patch `Command.invoke`**: This is the simplest mechanism that
reliably covers all leaf commands without modifying any command module. The
alternative — passing a custom `cls` to every `typer.Typer()` and
`@app.command()` — requires touching every CLI module and is easy to forget
for new commands. The patch is applied once in `main.py`, covers all
commands automatically, and the surface is minimal (one method, wrapping
with a flag set). If Typer adds a native post-invoke hook in the future, the
patch can be replaced.

**Rationale preserved from original**:
- Skinny CLI — command functions stay focused on their domain
- Automatic coverage — new commands emit events without author action
- Single point of control for emission logic

**Alternatives rejected**:
- `result_callback`: unsound — never fires (see above)
- `sys.argv` heuristic for help detection: fails for nested group
  `no_args_is_help` (e.g. `spec-driver create` has `["create"]` in argv,
  which is not bare and has no `--help`, but is still a help path)
- Explicit `emit_event()` per command: scattered, violates skinny CLI
- `atexit` handler: fires too late, after stdout/stderr are closed; no
  access to exit code; unreliable ordering with Click's own cleanup

### DEC-052-02: SOCK_DGRAM for socket protocol

**Decision**: Use `AF_UNIX` + `SOCK_DGRAM` (datagram) for the socket.

**Rationale**:
- Fire-and-forget: no connection state, no blocking, no handshake
- Events are small (<1KB) — well within kernel buffer limits
- Sender doesn't need to know if anyone is listening
- Stdlib only: `socket.AF_UNIX`, `socket.SOCK_DGRAM`

**Socket path length**: `AF_UNIX` pathname addresses are limited to 108 bytes
(107 usable + null terminator) on Linux. The path
`.spec-driver/run/tui.sock` is short (28 chars relative), but the absolute
path depends on the repo checkout depth. **Runtime check**: before attempting
socket send, verify `len(absolute_socket_path) <= 104`. If exceeded, skip
socket emission and log a one-time warning to stderr. The JSONL log is
unaffected.

**Alternatives rejected**:
- `SOCK_STREAM`: requires connection management, blocking on connect
- Named pipe (FIFO): blocking writer semantics when no reader
- Abstract namespace sockets (`\0` prefix): Linux-only, not portable to macOS

### DEC-052-03: Session attribution via SPEC_DRIVER_SESSION env var

**Decision**: Read `SPEC_DRIVER_SESSION` from the environment. Populate it
from Claude Code's `session_id` via the existing `startup.sh` hook using the
`CLAUDE_ENV_FILE` mechanism.

**How it works**:
1. Claude Code fires `SessionStart` hook, passes JSON with `session_id` on
   stdin
2. `startup.sh` reads stdin once (it must, since the boot prompt JSON is
   also output from this hook), extracts `session_id` using a Python
   one-liner, and exports it via `CLAUDE_ENV_FILE`
3. All subsequent Bash tool invocations (including `uv run spec-driver ...`)
   inherit the env var

**Extraction mechanism** — complete hook flow (no `jq` dependency; Python is
guaranteed available since spec-driver requires it):

```bash
#!/bin/sh
# Read the full hook input from stdin (Claude Code passes JSON)
INPUT=$(cat)

# Extract session_id using Python (always available in spec-driver projects)
SESSION_ID=$(echo "$INPUT" | python3 -c \
  "import sys,json; print(json.load(sys.stdin).get('session_id',''))" 2>/dev/null)

# Persist session ID for all subsequent Bash tool invocations
if [ -n "$CLAUDE_ENV_FILE" ] && [ -n "$SESSION_ID" ]; then
  echo "export SPEC_DRIVER_SESSION=$SESSION_ID" >> "$CLAUDE_ENV_FILE"
fi

# Output the boot prompt (existing behavior)
echo '{"systemMessage":"Welcome to spec-driver","hookSpecificOutput":{"hookEventName":"SessionStart","additionalContext":"you MUST use the /boot skill IMMEDIATELY after receiving the next prompt."}}'
```

Note: stdin must be consumed into `$INPUT` first because it can only be read
once. The boot prompt JSON is written to stdout (not stdin), so it is
unaffected.

**Fallback detection** (when `SPEC_DRIVER_SESSION` is unset):
- `CLAUDECODE=1` → tag session as `"claude"` (agent type, not session ID).
  This env var is set in Claude Code's Bash tool execution environment.
- Default: `null` — human or unknown caller

**Consequences**:
- Works today for Claude Code without protocol changes
- Codex/other agents: set `SPEC_DRIVER_SESSION` via their own boot mechanism,
  or accept type-level detection only
- No dependency on external tools — Python is the only requirement

### DEC-052-04: Log all commands unconditionally

**Decision**: Log every CLI invocation where a leaf command body executes. No
configurable scope filter in v1.

**Rationale**:
- Discovery commands (list, show, find) are valuable for understanding agent
  navigation patterns
- Error events on non-zero exits provide training/feedback data regardless of
  command type
- "Mutations" has no formal definition in the command model; adding a filter
  before defining the concept would be premature abstraction
- A future delta can add `[events] scope` once mutation semantics are defined

**Config shape** (v1):
```toml
[events]
enabled = true
```

Single knob. `enabled = false` disables all event emission (log + socket).

### DEC-052-05: Version field in event schema

**Decision**: Include `v: 1` in every event record.

**Rationale**: Costs one field. Prevents pain if schema evolves. Consumers can
filter/migrate by version.

**Event schema** (v1):
```jsonl
{"v":1,"ts":"2026-03-07T14:23:01Z","session":"abc123","cmd":"create delta","argv":["create","delta","--from-backlog","IMPR-009"],"artifacts":["DE-052"],"exit_code":0,"status":"ok"}
```

Fields:
- `v`: schema version (integer, always `1`)
- `ts`: ISO 8601 UTC timestamp
- `session`: `SPEC_DRIVER_SESSION` value, `"claude"` (type fallback), or `null`
- `cmd`: resolved command path (e.g. `"create delta"`, `"sync"`, `"list specs"`)
- `argv`: raw argument list from `sys.argv[1:]` — includes flags, preserves
  full invocation for audit/debugging (e.g. `--force`, `--json`, `--raw`)
- `artifacts`: list of artifact IDs touched (populated by `record_artifact()`;
  empty list when no command enriches it)
- `exit_code`: integer exit code (0 = success)
- `status`: `"ok"` or `"error"` (two values only — Click converts interrupts
  to `sys.exit(1)` before the wrapper sees them, making interrupt
  indistinguishable from error at this level)

### DEC-052-06: Artifact collector for opt-in enrichment

**Decision**: Provide a `record_artifact(artifact_id: str)` function in
`core/events.py` that commands can call to register artifacts they
create/modify. The process-boundary wrapper reads and clears this list when
emitting the event.

**Implementation**: Module-level list in `core/events.py`, not a
`contextvars.ContextVar` (CLI is single-threaded, no async; a plain list is
simpler and sufficient).

```python
_touched_artifacts: list[str] = []

def record_artifact(artifact_id: str) -> None:
    _touched_artifacts.append(artifact_id)

def _drain_artifacts() -> list[str]:
    arts = _touched_artifacts.copy()
    _touched_artifacts.clear()
    return arts
```

**Coverage**: Initially add `record_artifact()` calls to the creation
functions in `changes/creation.py`, `decisions/creation.py`,
`memory/creation.py`, `specs/creation.py`, `policies/creation.py`, and the
completion path in `complete_delta.py`. These are domain-layer functions, not
CLI commands, so the calls live at the point where the artifact ID is
assigned — consistent with the skinny CLI pattern.

**Consequence**: Commands that don't call `record_artifact()` emit events
with `artifacts: []`. This is acceptable — the field is enrichment, not a
contract. The TUI track view (IMPR-009) can still show command history;
artifact attribution is additive.

## 9. Rollout & Operational Notes

- **Migration**: None. New capability, no existing state to migrate.
- **Install integration**: `startup.sh` gains session extraction logic.
  Existing installs update on next `spec-driver install` (hook is
  installer-owned, overwritten on install).
- **Gitignore**: `.spec-driver/run/` added — runtime directory, never
  committed.
- **Log growth**: JSONL file grows unbounded at current scale. Defer
  rotation; file is append-only and trivially truncatable. Revisit if usage
  grows.
- **Recovery**: Delete `.spec-driver/run/events.jsonl` to reset. No other
  state depends on it.
- **Path ownership**: The `.spec-driver/run/` directory is internal
  infrastructure — hardcoded in `paths.py`, not user-configurable. Both the
  event log and socket live here. This is consistent with `.spec-driver/run/`
  being runtime state (like a PID file), not workspace content.
