## 2. Motivation

Navigating 13+ artifact types via CLI is friction-heavy for humans who want to
browse, explore relationships, or get an overview of workspace state. lazyspec
demonstrates the value of a terminal dashboard for this domain.

This delta delivers the core TUI: a browsable artifact dashboard with search,
filtering, preview, and editor integration.

## 3. Scope & Objectives

- **Primary Outcomes**:
  - `textual>=8.0,<9.0` + `watchfiles>=1.0,<2.0` added as `[tui]` optional
    extra in pyproject.toml
  - `spec-driver tui` entry point launches the dashboard
  - 3-panel layout: artifact type selector (colour-grouped), filtered list,
    markdown preview
  - Reusable artifact projection layer (`core/artifact_view.py`) normalising
    all registries to a common `ArtifactEntry` view model with error isolation
  - Shared column metadata (`formatters/column_defs.py`) consumed by both CLI
    formatters and TUI list views
  - Public style resolution API (`resolve_style()`, `styled_text()`) added to
    `formatters/theme.py`
  - Type/status filtering using existing registry `filter()` / `iter()` APIs
  - Fuzzy search across all artifact types
  - Editor integration: `$EDITOR` on selected artifact via `app.suspend()`
  - File watching: live reload via snapshot caching with per-registry
    watch-triggered invalidation
- **Operational Constraints**:
  - TUI is presentation layer only — no business logic, no state mutation
  - No business-logic changes to registries. Shared formatter/theme
    infrastructure will be extended (additive API, column metadata extraction)
  - Use `paths.py` getters for all directory resolution
  - Graceful error if `textual` not installed; helpful error if `$EDITOR` unset
  - `.tcss` uses only Textual-native tokens for layout chrome — no colour
    hex/rgb literals (POL-002, mechanically enforced by VT)
- **Dependencies**: DE-050 (normalised registry API — completed)
- **Exit Criteria**: Follow-up delta scoped for BacklogRegistry class
  normalisation (currently function-based, shimmed in adapter)

## 5. Approach Overview

- **System Touchpoints**:
  - `supekku/scripts/lib/formatters/theme.py` — additive: `resolve_style()`,
    `styled_text()` public API
  - `supekku/scripts/lib/formatters/column_defs.py` — new: shared column
    metadata per artifact type
  - `supekku/scripts/lib/formatters/*_formatters.py` — refactor to consume
    `column_defs.py`
  - `supekku/scripts/lib/core/artifact_view.py` — new: reusable artifact
    projection layer (no TUI imports)
  - `supekku/tui/` — new TUI package
  - `pyproject.toml` — `[tui]` optional extra
  - `supekku/cli/main.py` — `tui` command entry point
- **Key Changes**:
  - Shared infrastructure (non-TUI-specific):
    - `theme.py`: `resolve_style(name) → Style`, `styled_text(value, name) → Text`
    - `column_defs.py`: per-artifact-type column metadata (fields, labels,
      width hints) consumed by CLI formatters and TUI
    - `core/artifact_view.py`: `ArtifactEntry` view model, per-registry
      adapters (incl. BacklogRegistry shim), error isolation boundary,
      snapshot caching with watch-triggered invalidation
  - New `supekku/tui/` package:
    - `app.py` — Textual `App` subclass, screen composition, keybindings
    - `browser.py` — 3-panel browser screen
    - `widgets/` — type selector, artifact list, preview panel
    - `theme.tcss` — Textual CSS for layout chrome only (Textual-native tokens,
      no colour literals)
  - `pyproject.toml`: `[project.optional-dependencies] tui = ["textual>=8.0,<9.0", "watchfiles>=1.0,<2.0"]`
  - `supekku/cli/main.py`: `tui` command with import guard

Type selector: flat `OptionList`, ordered by group (governance → change →
domain → operational). Group identity conveyed by colour only via
`styled_text()` — no headings or indentation.

### Artifact projection layer

Reusable layer in `core/artifact_view.py` normalising all registries to a
common `ArtifactEntry` view model (id, title, status, path, artifact_type).
Uses DE-050's `collect/find/iter/filter` API. BacklogRegistry (function-based)
has a thin shim. The adapter is an error-isolation boundary — per-registry
failures produce degraded entries, not crashes. Results are snapshot-cached;
watch events trigger per-registry invalidation.

### Styling

Data cell styling uses pre-resolved Rich `Text` objects via `styled_text()`
in `theme.py` — the same style names used by CLI formatters. Layout chrome
(`.tcss`) uses only Textual-native tokens; no colour palette values appear
in the CSS file.

  terminal rendering
- **Acceptance Criteria**:
  - `spec-driver tui` launches without error when `[tui]` extra installed
  - `spec-driver tui` gives helpful error when extra not installed
  - All artifact types appear in type selector with correct counts
  - Selecting an artifact shows markdown preview
  - `e` opens `$EDITOR`, returns to TUI
  - File changes on disk trigger per-registry refresh within ~1s
  - `just` passes

## 7. Risks & Mitigations

- **Risk**: Rich markup rendering — _Status_: **resolved by spike**. Pre-resolved
  `Text` objects are the only viable approach; markup strings and
  `Text.from_markup()` with theme names do not work in Textual DataTable.
- **Risk**: Registry errors corrupt TUI — _Likelihood_: medium — _Impact_: high —
  _Mitigation_: Adapter is error-isolation boundary; per-registry failures
  surface as degraded UI state, stderr redirected during registry calls
- **Risk**: Startup/refresh performance — _Likelihood_: medium — _Impact_: medium —
  _Mitigation_: Snapshot-on-load, watch-triggered per-registry invalidation,
  no per-interaction filesystem rescans
- **Risk**: Column logic drift between CLI and TUI — _Likelihood_: medium —
  _Impact_: medium — _Mitigation_: Shared `column_defs.py` consumed by both
- **Risk**: Textual major version breakage — _Likelihood_: medium — _Impact_: low —
  _Mitigation_: Bounded range `>=8.0,<9.0`; TUI is optional extra
- **Risk**: `watchfiles` conflicts with Textual event loop — _Likelihood_: low —
  _Impact_: medium — _Mitigation_: `watchfiles` has async API compatible with
  asyncio
