## 2. Motivation

Navigating 13+ artifact types via CLI is friction-heavy for humans who want to
browse, explore relationships, or get an overview of workspace state. lazyspec
demonstrates the value of a terminal dashboard for this domain.

This delta delivers the core TUI: a browsable artifact dashboard with search,
filtering, preview, and editor integration.

## 3. Scope & Objectives

- **Primary Outcomes**:
  - `textual` + `watchfiles` added as `[tui]` optional extra in pyproject.toml
  - `spec-driver tui` entry point launches the dashboard
  - 3-panel layout: artifact type selector, filtered list, markdown preview
  - Type/status filtering using existing registry `filter()` / `iter()` APIs
  - Fuzzy search across all artifact types (Textual built-in `Matcher`)
  - Editor integration: `$EDITOR` on selected artifact via `app.suspend()`
  - File watching: live reload when artifacts change on disk
  - Theme CSS derived from `formatters/theme.py` colour palette
- **Operational Constraints**:
  - TUI is presentation layer only — no business logic
  - Reuse registries and formatters; do not duplicate domain logic
  - Use `paths.py` getters for all directory resolution
  - Graceful error if `textual` not installed
- **Dependencies**: DE-050 (normalised registry API — consistent `find()`,
  `collect()`, `iter()` across all registries)

## 5. Approach Overview

- **System Touchpoints**: `supekku/tui/` (new package), `pyproject.toml`,
  `supekku/cli/main.py` (entry point), `formatters/theme.py` (read, not modified)
- **Key Changes**:
  - New `supekku/tui/` package:
    - `app.py` — Textual `App` subclass, screen composition
    - `browser.py` — 3-panel browser screen
    - `widgets/` — type selector, artifact list, preview panel
    - `theme.tcss` — Textual CSS derived from `theme.py` palette

  - `pyproject.toml`: `[project.optional-dependencies] tui = ["textual>=3.0", "watchfiles>=1.0"]`
  - `supekku/cli/main.py`: `tui` command with import guard

### Registry adapter

Thin layer normalising registry access for the TUI. Uses DE-050's consistent
`collect()` / `find()` / `iter()` API. Maps each registry to a common
`ArtifactEntry` view model (id, title, status, path, artifact_type).

### Theme mapping

Read `SPEC_DRIVER_THEME` style definitions from `theme.py`. Generate Textual
CSS rules mapping style names to colours. Single source of truth for palette.

## 6. Verification Strategy

- **VT**: Unit tests for registry adapter (normalisation, view model mapping)
- **VT**: Theme CSS generation produces valid Textual CSS
- **VT**: Textual `App` can be pilot-tested headlessly
- **VH**: Manual smoke test — launch `spec-driver tui`, browse, search, edit
- **Acceptance Criteria**:
  - `spec-driver tui` launches without error when `[tui]` extra installed
  - `spec-driver tui` gives helpful error when extra not installed
  - All artifact types appear in type selector with correct counts
  - Selecting an artifact shows markdown preview
  - `e` opens `$EDITOR`, returns to TUI
  - File changes on disk update the list within ~1s
  - `just` passes

## 7. Risks & Mitigations

- **Risk**: Rich markup in formatters doesn't render cleanly in Textual —
  *Likelihood*: medium — *Impact*: medium — *Mitigation*: Spike early; fall
  back to returning Rich Table objects (Option B from research)
- **Risk**: 13 artifact types make type selector unwieldy — *Likelihood*: low —
  *Impact*: low — *Mitigation*: Group by category; most-used types first
- **Risk**: Textual version churn — *Likelihood*: medium — *Impact*: low —
  *Mitigation*: Pin to stable minor; TUI is optional extra
- **Risk**: `watchfiles` conflicts with Textual event loop — *Likelihood*: low —
  *Impact*: medium — *Mitigation*: `watchfiles` has async API compatible with
  asyncio
