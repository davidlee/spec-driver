# DR-053 ‚Äì TUI artifact browser MVP with Textual

## 1. Executive Summary

- **Delta**: [DE-053](./DE-053.md)
- **Status**: draft
- **Last Updated**: 2026-03-07
- **Synopsis**: Add an optional Textual-based TUI for browsing spec-driver
  artifacts, with 3-panel layout, search, and editor integration.

## 2. Problem & Constraints

- **Current Behaviour**: All artifact navigation is CLI-only ‚Äî `list`, `show`,
  `find` commands one at a time. Humans wanting to browse, explore
  relationships, or get a workspace overview face high friction across 13+
  artifact types.
- **Drivers / Inputs**: IMPR-009 (TUI dashboard for human navigation); IMPR-009
  research doc validated approach and mapped registry/formatter surface.
- **Constraints / Guardrails**:
  - TUI is **presentation layer only** ‚Äî no business logic, no state mutation
  - Reuse registries via DE-050's normalised `collect/find/iter/filter` API;
    no business-logic changes to registries
  - Shared formatter/theme infrastructure will be extended: additive API in
    `theme.py`, column metadata extraction into `column_defs.py`, formatter
    refactor to consume shared defs
  - Use `paths.py` getters for all directory resolution
  - TUI deps (`textual`, `watchfiles`) are optional extras ‚Äî graceful error
    when not installed
- **Out of Scope**: Track mode / event stream (DE-054), relation graph view,
  inline artifact creation, verification dashboard

## 3. Architecture Intent

- **Target Outcomes**:
  - `spec-driver tui` launches a browsable 3-panel dashboard (type selector,
    artifact list, markdown preview)
  - All artifact types accessible with status filtering and fuzzy search
  - Editor integration via `$EDITOR` + `app.suspend()`
  - Live reload on file changes via `watchfiles`
- **Guiding Principles**:
  - **Thin presentation layer**: TUI reads from registries, never writes
  - **Reusable projection layer**: `core/artifact_view.py` normalises all
    registries (including BacklogRegistry shim) to a common `ArtifactEntry`
    view model. No TUI imports ‚Äî usable by CLI search/export/future consumers.
  - **Single styling source**: `theme.py` gains `resolve_style()` and
    `styled_text()` as public API. Both CLI formatters (via markup) and TUI
    (via `styled_text()`) consume the same style names. No parallel palette.
  - **Shared column metadata**: Column definitions (fields, labels, width
    hints) extracted from formatters into `column_defs.py`, consumed by both
    CLI table formatters and TUI list views. Single source for "what to show".
  - **Error isolation**: The adapter catches per-registry failures and surfaces
    them as UI state. No stderr writes or uncaught exceptions leak into the
    terminal UI.
  - **Snapshot caching**: Registries scanned once at startup, results cached.
    Watch events invalidate per-registry, triggering targeted re-collect.
  - **Graceful degradation**: Missing optional deps ‚Üí helpful error, not crash
- **State Transitions / Lifecycle Impact**: None. TUI is read-only.

**Testing approach rationale**: Agents cannot interact with a TUI. The headless
`Pilot` API (`app.run_test(size=...)`) allows programmatic widget state
inspection, simulated key/click input, and screenshot export ‚Äî sufficient for
structural correctness. Visual polish and interactive features (editor suspend,
file watch responsiveness, real terminal rendering) require VH gates.

**Error handling coverage**: The adapter is an error-isolation boundary. VT
tests verify that per-registry failures (YAML parse errors, malformed
frontmatter, missing directories) produce degraded UI state ("‚öÝ load error")
rather than crashes or stderr corruption.

## 7. Design Decisions & Trade-offs

### DEC-053-01: Pre-resolved Text objects via theme.py public API (decided)

**Context**: Textual's `DataTable` needs styled cell content. Three approaches
tested in spike.

**Decision**: Add `resolve_style(name)` and `styled_text(value, style_name)` to
`formatters/theme.py` as public API. TUI calls `styled_text()` to build `Text`
objects with resolved styles from `SPEC_DRIVER_THEME`.

**Rationale**: Rich markup strings are stored as literal text by DataTable.
`Text.from_markup()` with theme style names doesn't resolve in Textual (no
Rich Theme context). Pre-resolved `Text` objects are the only approach that
renders colours correctly. Making `resolve_style()` a public function (rather
than reaching into `SPEC_DRIVER_THEME.styles` dict) gives a stable API surface
for both CLI and TUI consumers.

**Consequence**: The `get_*_status_style()` helpers return style names ‚Üí
`styled_text()` resolves them. Single resolution path, single palette.

### DEC-053-02: Extract shared column metadata; raw markdown for preview (decided)

**Context**: Existing formatters own nontrivial column/presentation logic
(columns, display-name cleanup, width rules, status styling, prefix stripping).
TUI list views need the same column definitions.

**Decision**: Extract column metadata (field names, display labels, width hints)
into `formatters/column_defs.py`. Both CLI formatter `_prepare_row` functions
and TUI row builders consume the same definitions. Preview panel uses `Markdown`
widget with `Path.read_text()`.

**Rationale**: Avoids parallel column-selection logic that would drift
(POL-001). The data flow is different (formatters ‚Üí Rich markup strings, TUI ‚Üí
`Text` objects), but the _what to show_ question should have one answer.
`Markdown` widget handles raw markdown natively.

**Trade-off**: Requires a small refactor of existing formatters to consume
`column_defs.py` instead of inline column definitions. Pays for itself by
eliminating a guaranteed drift vector.

### DEC-053-03: Hand-written .tcss for chrome; Textual-native tokens only (decided)

**Context**: 163 CSS rules extractable from `SPEC_DRIVER_THEME`. Generate
`.tcss` at runtime? Bridge palette into CSS?

**Decision**: No generation, no runtime bridge. The `.tcss` file handles layout
chrome only (borders, padding, sizing, focus indicators) using exclusively
Textual-native tokens and defaults. It is intentionally **not** palette-matched
to `SPEC_DRIVER_THEME`. All palette-derived styling flows through
`styled_text()` on data content, not through CSS.

The `.tcss` must not contain colour hex/rgb literals ‚Äî enforced by
VT-053-tcss-lint (POL-002).

**Rationale**: Two clean, separated styling surfaces with no coupling between
them: (1) data content styled via `styled_text()` / `resolve_style()` from
`theme.py`, (2) layout chrome styled via Textual-native CSS. No hidden bridge,
no magic values, no drift vector.

### DEC-053-04: Artifact projection layer in core/artifact_view.py (decided)

**Context**: The adapter normalises all registries to a common `ArtifactEntry`
view model. TUI is the first consumer, but CLI search/export/future dashboards
will need exactly the same projection.

**Decision**: `core/artifact_view.py`. No TUI imports ‚Äî pure data in, pure
data out. Contains per-registry adapters, BacklogRegistry shim, and error
isolation. Reusable by any consumer that needs a uniform artifact view.

**Rationale**: The risk of parallel implementation (building a second projection
layer when the next consumer appears) outweighs the risk of premature
generalisation here. The interface is simple and registry-shaped ‚Äî it's a
natural part of `core/`.

### DEC-053-05: Expanded VT coverage + VH for interactive-only (decided)

**Context**: Agents cannot interact with a running TUI.

**Decision**: Three-tier VT + VH:

- **VT (unit)**: Adapter normalisation, column defs, styled_text, error
  isolation (malformed artifacts, registry failures), edge cases (empty
  registries, `$EDITOR` unset, import guard)
- **VT (integration)**: Textual pilot tests ‚Äî widget composition, DataTable
  content, keybinding dispatch, Markdown rendering
- **VH**: Manual gates focused on what genuinely requires a human terminal ‚Äî
  editor suspend/resume, file watch responsiveness, real terminal rendering

**Rationale**: VT covers structural correctness and failure modes. VH covers
only what cannot be automated headlessly.

### DEC-053-06: Adapter as error-isolation boundary (decided)

**Context**: Registries have side effects that break TUI rendering.
`ChangeRegistry.collect()` prints to stderr on malformed artifacts.
`SpecRegistry` propagates parsing failures. Backlog functions raise YAML errors.

**Decision**: The artifact view layer catches per-registry failures. Each
registry load is wrapped ‚Äî failures produce a degraded `ArtifactEntry` list
with an error indicator, not crashes or stderr corruption. Stderr is redirected
during registry calls to prevent screen corruption.

### DEC-053-07: Snapshot-on-load with watch-triggered invalidation (decided)

**Context**: `collect()` rescans the filesystem on every call. With 13
registries and watch-triggered reloads, naive calling creates performance
problems.

**Decision**: Each registry `collect()` is called once at startup. Results are
cached as a snapshot (`dict[str, ArtifactEntry]`). Normal TUI interaction reads
from the snapshot, never the underlying registry. `watchfiles` change events
invalidate the affected registry's cache and trigger a targeted re-collect for
that type only.

### DEC-053-08: BacklogRegistry shim + fast-follow normalisation delta (decided)

**Context**: BacklogRegistry is function-based (no class), diverging from the
`collect/find/iter/filter` convention established by DE-050.

**Decision**: Build a thin shim in `core/artifact_view.py` wrapping
`discover_backlog_items()` / `find_item()` into the `ArtifactEntry` adapter
pattern. Create a follow-up delta (exit criteria for DE-053) to normalise
BacklogRegistry into a proper class.

**Rationale**: Unblocks TUI without widening scope. The shim is disposable ‚Äî
once the normalisation delta lands, it simplifies to the same pattern as other
registries.

### DEC-053-09: Bounded dependency ranges (decided)

**Context**: DE-053 originally specified `textual>=3.0`, `watchfiles>=1.0` ‚Äî
open-ended ranges for a UI framework and file watcher.

**Decision**: `textual>=8.0,<9.0` and `watchfiles>=1.0,<2.0`. Bounded to
current major version.

**Rationale**: UI frameworks have breaking API changes across majors. Bounded
ranges contain exposure. CI/test ratchet catches breakage within the range.

### DEC-053-10: Type selector ‚Äî flat list, colour-grouped (decided)

**Context**: 11 artifact types in the type selector. Flat list vs grouped with
headers/indentation vs something in between.

**Decision**: Flat `OptionList`, no headers or indentation. Items ordered by
group (governance ‚Üí change ‚Üí domain ‚Üí operational). Group identity conveyed by
colour only, using per-group theme styles via `styled_text()`.

**Rationale**: 11 items doesn't need structural grouping. Colour carries the
category signal without consuming vertical space or adding interaction
complexity. The ordering makes related types adjacent. Non-breaking to add
headers/separators later if needed.

### DEC-053-11: ArtifactTypeMeta ‚Äî shared display/classification metadata (decided)

**Context**: TUI type selector needs display names (plural + count) and group
colours. CLI has a parallel `_PLURAL_TO_SINGULAR` dict in `list.py`. No
canonical reusable mapping exists.

**Decision**: Add `ArtifactGroup` enum and `ArtifactTypeMeta` dataclass to
`core/artifact_view.py` alongside `ArtifactType`. Metadata provides `singular`,
`plural`, and `group` per type. Convenience properties on `ArtifactType`
(`art_type.singular`, `.plural`, `.group`) for ergonomic access. No style keys
in the metadata ‚Äî presentation derived by convention
(`f"artifact.group.{group.value}"`), keeping domain model presentation-free.

**Rationale**: Single source of truth for type display names and classification.
CLI can derive from it (follow-up, not Phase 2 scope). Style derivation by
convention avoids coupling domain to presentation. "Backlog Item(s)" as display
name for now ‚Äî backlog sub-kinds expected to evolve into distinct artifact types
in near term (related: ISSUE-016, -024, -026, -034, -043, -045, IMPR-010).

### DEC-053-12: Fuzzy search scoped to current type only (decided)

**Context**: DE-053 scope mentions "fuzzy search across all artifact types."
Two interpretations: filter within current type, or cross-type search showing
mixed results.

**Decision**: Phase 2 implements fuzzy filter on current type's artifact list
only. Cross-type search is a distinct feature, captured as follow-up.

**Rationale**: Current-type filtering is simpler, immediately useful, and
doesn't require mixed-type result rendering. Cross-type search is a separate
UX with different display needs.

### DEC-053-13: Status filter via Select widget (decided)

**Context**: Mockup shows `[status: all ‚ñæ]` in the artifact list column. Three
options: keybinding-cycled label, `Select` dropdown, or filter `Input` with
autocomplete.

**Decision**: Textual `Select` widget, populated with statuses for the current
type. Rebuilds options when type selection changes. Sits in a Vertical container
above the DataTable.

**Rationale**: Natural fit for a small bounded set of values (~3-6 statuses)
that varies per type. Discoverable without documentation. One row of vertical
space ‚Äî acceptable. Keybinding cycling has poor discovery; autocomplete Input is
overkill for this cardinality.

### DEC-053-14: BrowserScreen as Screen subclass (decided)

**Context**: Should the 3-panel browser be a `Screen` subclass (enabling screen
stack) or direct `App.compose()` content?

**Decision**: `Screen` subclass. App pushes `BrowserScreen` as default.

**Rationale**: Textual's idiomatic composition unit. Enables future views
(detail, help overlay) via screen stack without refactoring. Trivial cost ‚Äî
saves a predictable refactor.

### DEC-053-15: Built-in Footer widget (decided)

**Context**: Custom footer vs Textual's built-in `Footer` which auto-discovers
declared bindings.

**Decision**: Use Textual's built-in `Footer`. Replace with custom only if a
requirement emerges that `Footer` can't serve.

**Rationale**: The mockup's keybinding bar is exactly what `Footer` produces
from declared bindings. One line to mount, zero maintenance. Custom footer
adds manual sync burden with no current benefit.

### DEC-053-16: Textual Messages for inter-widget communication (decided)

**Context**: Type selection ‚Üí list update ‚Üí preview update. Options: Textual
Messages, reactive attributes, or hybrid.

**Decision**: Textual Messages. Custom message classes (`TypeSelected`,
`ArtifactSelected`). Screen handles messages in `on_*` methods, updates
downstream widgets.

**Rationale**: Explicit over magic. Two message classes, two handlers ‚Äî trivial
boilerplate for three widgets and two interaction points. Messages are
debuggable, follow Textual conventions, and scale clearly if interactions grow.
Reactive chains add indirection without payoff at this scale.

## 8. Open Questions

None remaining ‚Äî all resolved.

## 9. Known Limitations (MVP)

- **Accessibility**: MVP does not target screen reader compatibility.
  Keybinding collisions with common terminal emulators (ctrl+c, ctrl+z) will
  be tested during VH smoke but no formal accessibility audit is planned.
  Explicit non-goal for MVP; revisit if adoption grows.

## 10. Rollout & Operational Notes

- **Installation**: `uv pip install -e ".[tui]"` or `uv sync --extra tui`
- **Entry point**: `spec-driver tui` ‚Äî import guard checks for `textual`,
  prints install instructions if missing
- **No migration**: New feature, no existing state affected
- **Rollback**: Remove `[tui]` extra and `tui` command. `theme.py` API
  additions and `column_defs.py` are additive and remain useful independently.
  `core/artifact_view.py` is reusable and stays.
- **Exit criteria**: Create follow-up delta for BacklogRegistry class
  normalisation (see DEC-053-08).
