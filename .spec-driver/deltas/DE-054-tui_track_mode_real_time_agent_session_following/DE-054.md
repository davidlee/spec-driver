## 1. Summary & Context

- **Implementation Plan**: [IP-054](./IP-054.md)
- **Change Drivers**: IMPR-009 (TUI dashboard — agent session tracking)

## 2. Motivation

Humans watching an agent work have no real-time visibility into what spec-driver
commands are being executed, which artifacts are being created or modified, or
how far through a workflow the agent has progressed. The CLI output scrolls past
in the agent's terminal — the human needs their own view.

Track mode gives humans a live dashboard of spec-driver activity, filterable by
agent session or showing all local activity.

## 3. Scope & Objectives

- **Primary Outcomes**:
  - Track panel/view in the TUI showing live event stream
  - Unix domain socket listener in Textual async loop
  - Event log replay on startup (catch up on recent history)
  - Session filtering: follow one session or all activity
  - Click-to-navigate: selecting an event navigates to that artifact in the
    browser panel
  - Session list: show active/recent sessions with colour coding
- **Operational Constraints**:
  - Must not block or slow the browser panel
  - Socket listener must handle malformed events gracefully
  - Replay must be bounded (last N events or time window)
- **Dependencies**:
  - DE-052 (event infrastructure — emitter, log, socket)
  - DE-053 (TUI browser — shell to add track view to)

## 4. Out of Scope

- Event filtering beyond session ID (e.g. by artifact type, command)
- Alerting or notifications
- Multi-machine / networked event streaming
- Historical analysis or aggregation

## 5. Approach Overview

- **System Touchpoints**: `supekku/tui/` (extend), `core/events.py` (import
  event schema)
- **Key Changes**:
  - New `supekku/tui/track.py`:
    - `TrackPanel` widget — `RichLog` showing event stream
    - `SessionSelector` — list of active/recent sessions
    - `EventListener` — async socket receiver (`asyncio.DatagramProtocol`)
  - Extend `app.py`:
    - Add track view/panel toggle (keybinding)
    - Wire event-to-artifact navigation
  - Event log replay:
    - On startup, read last N lines of `.spec-driver/run/events.jsonl`
    - Populate track panel with recent history before live events arrive

### Socket listener

```python
class EventListener(asyncio.DatagramProtocol):
    def datagram_received(self, data, addr):
        event = json.loads(data)
        self.app.post_message(TrackEvent(event))
```

Textual's `post_message()` bridges async socket events into the widget
message system. The `TrackPanel` handles `TrackEvent` messages and appends
to its `RichLog`.

## 6. Verification Strategy

- **VT**: Unit tests for event log replay (parsing, bounding)
- **VT**: Socket listener handles malformed JSON gracefully
- **VT**: Session filtering produces correct subset
- **VH**: Manual smoke test — run agent in one terminal, TUI in another,
  verify events appear in real time
- **Acceptance Criteria**:
  - Events from CLI commands appear in track panel within ~100ms
  - Session filter shows only events for selected session
  - Clicking an event navigates to the artifact in browser
  - Malformed socket data logged and ignored, no crash
  - `just` passes

## 7. Risks & Mitigations

- **Risk**: High event volume overwhelms RichLog — *Likelihood*: low (spec-driver
  commands are infrequent) — *Impact*: low — *Mitigation*: Bound display to
  last N events, older events scroll off
- **Risk**: Socket listener interferes with Textual event loop — *Likelihood*:
  low — *Impact*: medium — *Mitigation*: Use `asyncio.DatagramProtocol`
  which integrates cleanly with Textual's loop
- **Risk**: Race between log replay and live socket events — *Likelihood*:
  medium — *Impact*: low — *Mitigation*: Replay first, then start listener;
  dedup by timestamp+cmd if needed
