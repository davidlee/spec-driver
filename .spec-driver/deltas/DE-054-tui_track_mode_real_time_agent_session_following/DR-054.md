# DR-054 – TUI track mode: real-time agent session following

## 1. Executive Summary
- **Delta**: [DE-054](./DE-054.md)
- **Status**: draft
- **Last Updated**: 2026-03-07
- **Synopsis**: Add a Track view to the TUI that receives real-time CLI events
  via Unix domain socket (or log-tail fallback), displays them as a scrolling
  stream, and lets users filter by agent session and navigate to touched
  artifacts in the browser.

## 2. Problem & Constraints

- **Current Behaviour**: DE-052 emits structured JSONL events and
  fire-and-forgets datagrams to `.spec-driver/run/tui.sock`. DE-053 provides
  a browsable TUI. Nothing listens on the socket; events are write-only.
- **Drivers / Inputs**: IMPR-009 track mode design. DE-052 event schema (v1).
  DE-053 TUI shell (screen stack, widget patterns, theme infrastructure).
- **Constraints / Guardrails**:
  - Must not block or slow the browser panel
  - Socket listener must handle malformed events gracefully
  - Must not scuttle a running TUI's socket if a second instance starts
  - Replay must be bounded (last N events)
  - TUI is presentation-only — no mutation of event log or workspace state
  - Reuse existing Textual patterns: Messages for inter-widget communication,
    `styled_text()` for data styling, Screen subclass for view composition
- **Out of Scope**:
  - Event filtering beyond session ID (e.g. by artifact type, command)
  - Alerting or notifications
  - Multi-machine / networked event streaming
  - Historical analysis or aggregation
  - Log rotation / compaction

## 3. Architecture Intent

- **Target Outcomes**:
  - `t` keybinding toggles between BrowserScreen and TrackScreen
  - TrackScreen shows a 2-panel layout: session list + event stream
  - Events from CLI commands appear within ~100ms (socket) or ~1s (log-tail)
  - Session filter shows only events for selected session
  - Selecting an event with artifacts navigates to that artifact in browser
  - Second TUI instance degrades gracefully to log-tail mode
- **Guiding Principles**:
  - Consistent with DE-053 patterns: Screen subclass, Textual Messages,
    `styled_text()`, no business logic in TUI
  - `EventListener` is a reusable async abstraction, not coupled to Textual
    widgets — it yields parsed event dicts
  - Socket lifecycle is defensive: probe before scuttling, clean up on exit
  - Fail-silent on all I/O: malformed events logged and skipped, socket
    errors never crash the TUI
- **State Transitions / Lifecycle Impact**: None — Track is read-only
  observation of the event stream.

## 7. Design Decisions & Trade-offs

### DEC-054-01: TrackScreen as persistent installed Screen, toggled via switch_screen

**Decision**: Track mode is a `TrackScreen(Screen)` subclass. Both
`BrowserScreen` and `TrackScreen` are installed via `install_screen()` at
mount time. The `t` keybinding calls `switch_screen()` to toggle between
them. Both screens remain mounted at all times — widget state is preserved
across toggles.

**Rationale**: Consistent with DEC-053-14 (BrowserScreen as Screen subclass).
Clean separation of concerns — browser and track have different layouts,
different widgets, different interaction models. Using `install_screen` +
`switch_screen` (rather than push/pop) preserves TrackScreen's widget state
(event buffer, scroll position, session selection) across toggles. Push/pop
would destroy the TrackScreen widget tree on each pop, requiring rebuild
from the app's buffer on every re-entry.

**Consequence**: Cross-screen navigation (track event → browser artifact)
requires an app-level action that switches to BrowserScreen and tells it
to navigate. The app owns both screens and can reference them by name.

**Lifecycle**: `SpecDriverApp.on_mount()` installs both screens
(`install_screen(browser, "browser")`, `install_screen(track, "track")`),
then `switch_screen("browser")` to start on the browser. The `t` binding
checks `self.screen` type and switches to the other.

**Alternatives rejected**:
- Push/pop screen stack: destroys TrackScreen on pop, losing scroll
  position, session selection, and requiring full replay from buffer
- Togglable panel within BrowserScreen: couples two distinct UX modes,
  complicates layout, violates SRP for BrowserScreen
- Replacing preview panel: loses preview capability while tracking

### DEC-054-02: Socket listener with probe-before-bind and log-tail fallback

**Decision**: On startup, the `EventListener` attempts to bind
`.spec-driver/run/tui.sock`. Before binding, it checks for an existing socket:

1. Socket file does not exist → bind normally (**socket mode**)
2. Socket file exists → probe with a test `sendto`:
   - `sendto` succeeds → live TUI is listening → **log-tail mode**
   - `ECONNREFUSED` → stale socket from crash → unlink, bind (**socket mode**)
   - Other error → **log-tail mode** (defensive)

**Log-tail mode**: Watch the **containing directory** (`.spec-driver/run/`)
via `watchfiles.awatch`, filtered to `events.jsonl` changes only — watching
the file directly is unreliable (`awatch` on a single file may not yield on
append). Track file position via `seek`/`tell`, read new lines from
last-known offset. Latency ~220-300ms (observed on Linux). Functionally
equivalent to socket mode.

**Platform scope for socket probe**: The probe's errno behaviour
(`ECONNREFUSED` for stale sockets, `FileNotFoundError` for absent) is
verified on Linux. macOS `AF_UNIX` `SOCK_DGRAM` semantics are documented
as compatible but the specific errno for stale-socket `sendto` is not
independently verified. If macOS behaves differently, the probe's `except
OSError` fallback defaults to log-tail mode (safe, not broken — just
conservative).

**Socket cleanup**: The TUI that owns the socket removes it in a `finally`
block on shutdown (both normal exit and cancellation). If cleanup fails
(crash, SIGKILL), the next TUI's probe detects the stale socket via
`ECONNREFUSED`.

**Probe implementation**:
```python
def _probe_socket(sock_path: str) -> bool:
  """Return True if a live listener exists at sock_path."""
  try:
    probe = socket.socket(socket.AF_UNIX, socket.SOCK_DGRAM)
    try:
      probe.sendto(b'{"v":0}', sock_path)
      return True  # someone is listening
    finally:
      probe.close()
  except (ConnectionRefusedError, FileNotFoundError):
    return False
  except OSError:
    return True  # assume live on unexpected errors (defensive)
```

The probe sends `{"v":0}` — a valid JSON object with schema version 0, which
the listener can recognise and discard as a probe (real events have `v: 1`).

**Rationale**: Protects a running TUI from being scuttled by a second
instance. The JSONL log is always written (DE-052 design), so log-tail is
a reliable fallback with no data loss. No lock files or coordination
protocols needed.

**Alternatives rejected**:
- Unconditional unlink before bind: scuttles first instance
- Lock file alongside socket: extra state to manage, cleanup is equally
  fragile on crash
- Per-instance sockets: requires emitter to broadcast to multiple paths,
  complicates DE-052's fire-and-forget design

### DEC-054-03: Replay last N events (200) from JSONL log on startup

**Decision**: On startup, read the last 200 lines of `events.jsonl`. Parse
each as JSON, skip malformed lines. Feed into TrackScreen as initial state
before live events begin. The replay count is a module constant, not
user-configurable.

**Implementation**: Read the file, split by newlines, take the last N. No
need for efficient tail-seeking at this scale — event logs are small (each
line ~200 bytes, 200 lines ≈ 40KB). If the file doesn't exist or is empty,
replay returns an empty list.

**Rationale**: Simple, bounded, predictable. Time-based replay ("last 30
minutes") requires parsing timestamps before filtering, and the useful
window varies. Line-count is O(1) to reason about and guarantees bounded
memory.

**Bootstrap sequence** (closes the replay-to-live gap):

1. `replay_events()` reads last N lines, records the **file offset** at EOF
2. `start()` begins listening (socket bind or log-tail watch)
3. Immediately after listener is active, perform a **drain pass**: read from
   the recorded offset to current EOF, parse any new lines emitted between
   steps 1 and 2
4. Normal live event flow takes over

This ensures no events are missed in the window between replay and listener
activation. The drain pass may re-emit events that the socket already
delivered (if socket mode), but at spec-driver's command frequency this is
negligible. No dedup in v1 — accept occasional duplicate rendering rather
than adding timestamp-tracking complexity.

### DEC-054-04: Session discovery from event stream with deterministic colour hashing

**Decision**: Sessions are discovered dynamically from the `session` field in
replayed and live events. The `SessionList` widget maintains a
`dict[str | None, SessionInfo]` mapping session IDs to metadata (last event
timestamp, event count). The list is ordered by most-recent-event-first.

**Session states**:
- **Active**: has events within the last 10 minutes (constant, not configurable)
- **Recent**: has events, but older than 10 minutes
- `null` session events are grouped under an "unknown" label

**Colour assignment**: Stable digest of session ID to a small palette of 8
distinct colours. Uses `int.from_bytes(hashlib.md5(session_id.encode()).digest()[:2]) % 8`
— stable across Python processes (unlike `hash()`, which is randomised by
`PYTHONHASHSEED`). Same session always gets the same colour across restarts.
Colours are defined as theme style keys (`track.session.0` through
`track.session.7`) so they flow through `styled_text()` per DEC-053-01.

**"All" mode**: Default. Shows events from all sessions. Session column in
the event stream uses per-session colour. Selecting a specific session in
the session list filters the stream.

**Rationale**: No external state needed — sessions are emergent from the
event data. Deterministic colours avoid confusion when the same session
appears across TUI restarts. 8 colours is sufficient for the expected
concurrency (1-3 simultaneous agents).

### DEC-054-05: Uniform EventListener abstraction

**Decision**: `EventListener` is a class with a uniform interface regardless
of transport:

```python
class EventListener:
  """Async event listener — socket or log-tail mode."""

  async def start(self, app: App) -> None:
    """Begin listening. Posts TrackEvent messages to the app."""

  async def stop(self) -> None:
    """Stop listening and clean up resources (including socket unlink)."""

  def replay_events(self) -> list[dict]:
    """Return replayed events from log (call before start)."""
```

Internally:
- **Socket mode**: Pre-bind a `socket.AF_UNIX` / `socket.SOCK_DGRAM` socket
  in Python, then register it with asyncio via
  `asyncio.get_running_loop().create_datagram_endpoint(protocol, sock=sock)`.
  The `local_addr` parameter does not support `AF_UNIX` paths — passing a
  pre-bound `sock=` is the only working path. On `datagram_received`, parse
  JSON, discard probes (`v: 0`), and call
  `app.post_message(TrackEvent(event))`.
- **Log-tail mode**: `watchfiles.awatch` on the **containing directory**
  (`.spec-driver/run/`), filtered to `events.jsonl` changes only (not the
  file directly — `awatch` on a single file is unreliable). Track file
  position via `seek`/`tell`, read new lines from last-known offset, parse,
  post messages.

Both modes post `TrackEvent` messages to the app. TrackScreen handles them
identically.

**Socket binding detail**:
```python
sock = socket.socket(socket.AF_UNIX, socket.SOCK_DGRAM)
sock.bind(sock_path)
sock.setblocking(False)
loop = asyncio.get_running_loop()
transport, protocol = await loop.create_datagram_endpoint(
  lambda: _EventProtocol(app),
  sock=sock,
)
```

`asyncio.get_running_loop()` is available inside `App.on_mount()` because
Textual runs its own asyncio event loop (confirmed: `SpecDriverApp` already
uses `asyncio.create_task` in its watcher setup).

**Lifecycle**: `SpecDriverApp.on_mount()` installs both screens, creates
the listener, calls `get_replay_events()` to seed TrackScreen's buffer,
then starts the listener as an `asyncio.Task`. `on_unmount()` stops it
(closes transport, unlinks socket if owned). TrackScreen stays mounted
across toggles — no replay needed on re-entry.

**Rationale**: Widgets don't know or care about the transport. Testing can
inject events directly via `TrackEvent` messages without socket/file
infrastructure.

### DEC-054-06: Cross-screen navigation via app action

**Decision**: When a user selects an event row that has artifacts, the
`DataTable.RowSelected` message triggers
`action_navigate_artifact(artifact_id)` on the app. This:

1. Switches to BrowserScreen via `switch_screen("browser")`
2. Calls `browser_screen.navigate_to_artifact(artifact_id)`

`BrowserScreen.navigate_to_artifact(artifact_id)` resolves the artifact's
type from the snapshot, switches the type selector, and highlights the
artifact row in the list.

**ID resolution**: `ArtifactSnapshot` currently has no cross-type lookup.
Add `ArtifactSnapshot.find_entry(artifact_id: str) -> ArtifactEntry | None`
that scans `entries` across all `ArtifactType` values and returns the first
match. This is O(n) over total artifacts but acceptable — the snapshot is
small (hundreds of entries) and navigation is user-initiated, not hot-path.
Once found, the entry's `artifact_type` determines which type to select in
the browser, then the entry's `id` selects the row in the DataTable.

**If the artifact ID doesn't resolve** (not in snapshot, or type unknown):
notify with a warning message. Don't crash, don't switch screens.

**Rationale**: Minimal coupling — app mediates between screens. Browser
gains one method. Snapshot gains one lookup method (reusable by future
consumers). Track doesn't import or know about browser internals.

### DEC-054-07: TrackPanel uses DataTable for event stream with row selection

**Decision**: `TrackPanel` wraps a `DataTable` with `cursor_type="row"`.
Events are added as rows via `add_row()` with styled `Text` cells. The
table has 5 columns: timestamp, session, command, artifact, status.

**Event format**:
```
14:23:01  [session]  create delta           DE-052  ok
14:23:03  [session]  draft-design-revision   DR-001  ok
14:23:45  [session]  complete delta          DE-052  error
```

Fields: timestamp (HH:MM:SS from ISO 8601 `ts`), session (truncated to 8
chars, coloured per DEC-054-04), command (`cmd` field), first artifact
(from `artifacts` list, or blank), status (coloured: green for ok, red for
error).

**Display bound**: Capped at 500 rows. When `row_count` exceeds the cap,
the oldest row is removed via `remove_row()`. This is display-only — the
full history remains in `events.jsonl`.

**Row selection**: `DataTable.RowSelected` message fires on Enter or click.
The row key stores the first artifact ID (or empty string). TrackScreen
handles the message and, if an artifact ID is present, triggers
`action_navigate_artifact(artifact_id)` on the app (DEC-054-06).

**Styling**: Cell content uses `styled_text()` per cell. Style keys:
`track.timestamp`, `track.session.N` (0-7), `track.status.ok`,
`track.status.error`, `track.cmd`, `track.artifact`.

**Rationale**: DataTable provides row cursor, keyboard navigation, and
click-to-select for free. Events are tabular (5 fixed columns). RichLog
lacks per-line selection, which would require a custom cursor
implementation to support click-to-navigate (DEC-054-06). The cost
difference is minimal: manual row pruning via `remove_row()` replaces
RichLog's `max_lines`, everything else is comparable or simpler.

**Consequence**: When switching session filter, the table is cleared via
`clear()` and replayed from the in-memory event buffer (filtered). The
buffer holds all events (replay + live), bounded by the same 500-event cap.

## 9. Rollout & Operational Notes

- **Migration**: None. New view in existing TUI.
- **Dependencies**: No new package dependencies. `watchfiles` (already
  required by DE-053) is used for log-tail mode. `socket` is stdlib.
- **Theme additions**: New style keys (`track.timestamp`, `track.session.0-7`,
  `track.status.ok`, `track.status.error`, `track.cmd`, `track.artifact`)
  must be added to `SPEC_DRIVER_THEME` in `theme.py`.
- **Socket path**: `.spec-driver/run/tui.sock` — same directory as
  `events.jsonl`, already in `.gitignore` (DE-052).
- **Recovery**: If socket gets into a bad state, delete
  `.spec-driver/run/tui.sock`. Next TUI launch creates a fresh one.
