## 2. Motivation

- Skill invocation is still easy to bypass during early task handling, especially before an agent has committed to the governing workflow path.
- The existing `spec-driver` skill is effective at forcing CLI-first interaction with spec-driver entities, but it is not yet part of an explicit boot-time routing discipline.
- The target state is stricter, repo-native skill routing that:
  - loads core routing guidance from boot
  - keeps procedural ownership in skills rather than prose
  - preserves spec-driver's install model: uniform packaged skills with project-specific customisation via `.spec-driver/hooks/*`
- `/tmp/superpowers/skills/using-superpowers/SKILL.md` provides a useful example of routing strictness, but its global ceremony must be filtered through repo doctrine rather than copied directly.

## 3. Scope & Objectives

- **Primary Outcomes**:
  - Explicitly integrate stronger skill-routing guidance into the boot path without creating a competing doctrine layer.
  - Decide whether routing should remain separate from `spec-driver` or expand that skill's scope.
  - Preserve the design principle that installed skills remain uniform across installs and defer project-specific behaviour to hook stubs and generated guidance.
  - Define how the audit step in the canonical loop should force implementation findings back into spec updates before delta closure.
  - Turn audits from optional retrospective notes into an explicit reconciliation checkpoint for evergreen product and technical specs.
  - Capture optional, composable patterns worth future skillisation, especially brainstorming for authoring/design work and adversarial review for DR/spec evaluation.
  - Make DR authoring more concrete by biasing toward code-adjacent examples when they reduce ambiguity.
  - Ensure the DR loop performs an internal adversarial pass, can hand off to an external adversarial reviewer, and reconciles the owning delta before planning begins.
  - Ensure relevant ADRs, policies, and standards are consulted before drafting and during critical review of a DR.
  - Nudge delta close-out to revisit any originating backlog entries so they do not drift stale after completion.
- **Operational Constraints**:
  - Do not regress the successful narrow role of the `spec-driver` skill.
  - Do not import `superpowers` assumptions that conflict with ADR-004 or posture-aware workflow guidance.
  - Prefer additive routing guidance over broad new runtime enforcement.
  - Preserve ADR-008's rule that evidence prompts explicit reconciliation rather than silently overriding normative spec truth.
  - Preserve enough flexibility for per-repo defaults, especially around commit policy.
- **Dependencies**:
  - Likely touches installer/skill sync and generated boot guidance surfaces.
  - `ISSUE-009` blocks clean backlog-skill workflow design until backlog item statuses have a defined canonical vocabulary.
  - `DE-079` now owns implementation of the canonical audit reconciliation contract; `DE-055` retains the workflow/doctrine design that motivated that follow-up.
  - `evidence-based-skill-development.md` has been added to this delta bundle as pending research input; consume the compressed relevant extract when it is available rather than reading the full doc directly.
  - May require product/spec clarification if the customisation principle is not already explicit enough in governance docs.

## 4. Out of Scope

- Replacing the existing skill library with `superpowers` semantics.
- Imposing mandatory brainstorming/design approval on all changes.
- Reworking broader workflow doctrine beyond skill-routing and boot guidance.
- Full spec corpus reconciliation for the skills subsystem.

## 5. Approach Overview

- **System Touchpoints**:
  - `supekku/skills/boot/`
  - `supekku/skills/spec-driver/`
  - `supekku/scripts/lib/skills/sync.py`
  - `supekku/scripts/install.py`
  - `supekku/templates/agents/boot.md`
  - `.spec-driver/hooks/doctrine.md`
- **Key Changes**:
  - Research and document a repo-native routing model inspired by `using-superpowers` but constrained by ADR-004, ADR-005, and `PROD-016`.
  - Make boot explicitly reinforce CLI-first and skill-first routing for spec-driver entities.
  - Evaluate a separate routing skill versus widening `spec-driver`; current leaning is separate routing skill plus explicit boot linkage to `spec-driver`.
  - Make routing explicitly distinguish execution-ready delta work from pre-execution planning so agents do not skip DR/IP/phase creation.
  - Split generic `/notes` journaling from structured delta execution doc maintenance through a delta-specific skill.
  - Make `/preflight` and handoff skills surface assumptions, unresolved questions, and design tensions before declaring implementation readiness.
  - Separate optional brainstorming imports into a generic question-resolution loop and artifact-specific authoring guidance rather than importing one large universal skill.
  - Land the first artifact-specific authoring enhancement in `draft-design-revision` so DR work uses explicit design-question triage and section-by-section validation.
  - Extend `draft-design-revision` so DR work stays progressive, uses short code samples when useful, runs an internal adversarial review, optionally offers an external adversarial-review prompt, and reconciles `DE-XXX.md` before offering `/plan-phases`.
  - Make `/doctrine` an explicit step inside `draft-design-revision` so governance constraints shape both drafting and adversarial review.
  - Make non-trivial work treat a missing or stale DR as a stop condition before `/plan-phases`; IP and phase creation are downstream of current design, not a substitute for it.
  - Extend `close-change` and delta-completion guidance so closure checks for linked originating backlog entries and updates them or raises follow-up ambiguity explicitly.
  - Define the workflow/doctrine boundary for a stronger audit contract so drift found during audit cannot disappear without disposition, then hand concrete schema/runtime implementation to `DE-079`.
  - Distinguish discovery audits from conformance audits at the doctrine level while keeping both aligned with the canonical `implement -> audit/contracts -> revision/spec reconcile -> close` loop.
- **Migration / Rollout Notes**:
  - Installer/sync changes must preserve the generated-vs-user-owned split.
  - Any customisation seam should live in hook docs or generated guidance, not bespoke per-project skill copies.

## 7. Risks & Mitigations

- **Risk**: Expanding `spec-driver` into a meta-skill dilutes its effective CLI-first contract. - _Likelihood_: medium - _Impact_: medium - _Mitigation_: keep routing as a separate responsibility unless research shows the split is harmful.
- **Risk**: Boot-time strictness accidentally imports non-canonical `superpowers` workflow constraints. - _Likelihood_: medium - _Impact_: high - _Mitigation_: pin design decisions to ADR-004 and posture guidance.
- **Risk**: Customisation support drifts away from the generated-guidance plus hook model. - _Likelihood_: medium - _Impact_: high - _Mitigation_: treat `PROD-016` hook semantics as a hard guardrail.
