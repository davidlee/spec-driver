## 2. Motivation
- Skill invocation is still easy to bypass during early task handling, especially before an agent has committed to the governing workflow path.
- The existing `spec-driver` skill is effective at forcing CLI-first interaction with spec-driver entities, but it is not yet part of an explicit boot-time routing discipline.
- The target state is stricter, repo-native skill routing that:
  - loads core routing guidance from boot
  - keeps procedural ownership in skills rather than prose
  - preserves spec-driver's install model: uniform packaged skills with project-specific customisation via `.spec-driver/hooks/*`
- `/tmp/superpowers/skills/using-superpowers/SKILL.md` provides a useful example of routing strictness, but its global ceremony must be filtered through repo doctrine rather than copied directly.

## 3. Scope & Objectives
- **Primary Outcomes**:
  - Explicitly integrate stronger skill-routing guidance into the boot path without creating a competing doctrine layer.
  - Decide whether routing should remain separate from `spec-driver` or expand that skill's scope.
  - Preserve the design principle that installed skills remain uniform across installs and defer project-specific behaviour to hook stubs and generated guidance.
  - Capture optional, composable patterns worth future skillisation, especially brainstorming for authoring/design work and adversarial review for DR/spec evaluation.
- **Operational Constraints**:
  - Do not regress the successful narrow role of the `spec-driver` skill.
  - Do not import `superpowers` assumptions that conflict with ADR-004 or posture-aware workflow guidance.
  - Prefer additive routing guidance over broad new runtime enforcement.
  - Preserve enough flexibility for per-repo defaults, especially around commit policy.
- **Dependencies**:
  - Likely touches installer/skill sync and generated boot guidance surfaces.
  - May require product/spec clarification if the customisation principle is not already explicit enough in governance docs.

## 4. Out of Scope
- Replacing the existing skill library with `superpowers` semantics.
- Imposing mandatory brainstorming/design approval on all changes.
- Reworking broader workflow doctrine beyond skill-routing and boot guidance.
- Full spec corpus reconciliation for the skills subsystem.

## 5. Approach Overview
- **System Touchpoints**:
  - `supekku/skills/boot/`
  - `supekku/skills/spec-driver/`
  - `supekku/scripts/lib/skills/sync.py`
  - `supekku/scripts/install.py`
  - `supekku/templates/agents/boot.md`
  - `.spec-driver/hooks/doctrine.md`
- **Key Changes**:
  - Research and document a repo-native routing model inspired by `using-superpowers` but constrained by ADR-004, ADR-005, and `PROD-016`.
  - Make boot explicitly reinforce CLI-first and skill-first routing for spec-driver entities.
  - Evaluate a separate routing skill versus widening `spec-driver`; current leaning is separate routing skill plus explicit boot linkage to `spec-driver`.
- **Migration / Rollout Notes**:
  - Installer/sync changes must preserve the generated-vs-user-owned split.
  - Any customisation seam should live in hook docs or generated guidance, not bespoke per-project skill copies.

## 7. Risks & Mitigations
- **Risk**: Expanding `spec-driver` into a meta-skill dilutes its effective CLI-first contract. - *Likelihood*: medium - *Impact*: medium - *Mitigation*: keep routing as a separate responsibility unless research shows the split is harmful.
- **Risk**: Boot-time strictness accidentally imports non-canonical `superpowers` workflow constraints. - *Likelihood*: medium - *Impact*: high - *Mitigation*: pin design decisions to ADR-004 and posture guidance.
- **Risk**: Customisation support drifts away from the generated-guidance plus hook model. - *Likelihood*: medium - *Impact*: high - *Mitigation*: treat `PROD-016` hook semantics as a hard guardrail.
