# DR-055 – Tighten skill routing and boot-time workflow guidance

## 1. Executive Summary
- **Delta**: [DE-055](./DE-055.md)
- **Status**: draft
- **Owners / Team**: david
- **Last Updated**: 2026-03-09
- **Synopsis**: Tighten skill invocation by making routing an explicit boot-time concern, strengthen DR authoring with doctrine consultation and adversarial review, and define the audit-loop doctrine now handed to `DE-079`, while porting remaining audit-to-spec/spec-revision skill work to `DE-083`.

## 2. Problem & Constraints
- **Current Behaviour**:
  - `spec-driver` is effective when the task already points at spec-driver entities or commands.
  - Boot establishes onboarding and policy context, but it does not yet seem to enforce a stronger routing discipline comparable to the best parts of `using-superpowers`.
  - Agents can still drift into ad hoc exploration before deciding which skill governs the task.
  - The canonical loop says `implement -> audit/contracts -> revision/spec reconcile -> close`, but the operational workflow still treats audits as optional and underspecified.
  - `audit-change` tells agents to reconcile findings back into specs, yet it does not currently require creation or completion of a structured audit artefact.
  - The audit entity exists, but its rich frontmatter fields (`findings`, `patch_level`, `next_actions`) are not part of any enforced close-out contract.
  - `complete delta` currently gates on parent-spec coverage only; it does not confirm that an audit exists, that drift has been dispositioned, or that required spec patches or revisions were made.
  - Requirements sync records `verified_by` audit IDs, but it does not surface unresolved audit drift as a first-class workflow outcome.
  - Several live workflow failures remain outside pure routing:
    - deltas can be completed while still `draft`
    - agents may skip DR, IP, or phase sheets and rush to implementation
    - `/notes` is easy to omit after task completion
    - commit expectations are underspecified
    - ordering between DR, IP, phase creation, and execution is not stated strongly enough
    - capture/backlog work lacks a dedicated skill and end-to-end connective tissue into and out of delta execution
- **Drivers / Inputs**:
  - `ADR-004` requires a single canonical workflow doctrine.
  - `ADR-005` makes memories and skills the canonical guidance layers.
  - `ADR-008` requires observed evidence to trigger explicit reconciliation instead of silently overwriting normative truth.
  - `PROD-016` says configuration should stay small and project customisation should come through generated guidance and hooks rather than process-key explosion.
  - `/tmp/superpowers` demonstrates useful strictness about choosing the governing skill before acting.
  - The DE-053 handover case shows that entrance-criteria checks alone are not enough; readiness claims must survive a critical read of DR/IP/phase artefacts.
  - `ISSUE-009` shows backlog-item lifecycles still lack canonical status enums, which blocks precise workflow wording for backlog-oriented skills.
- **Constraints / Guardrails**:
  - Do not collapse posture-aware workflow into universal mandatory brainstorming or planning.
  - Do not let routing guidance compete with memories, skills, ADRs, or specs for doctrinal ownership.
  - Do not replace user-owned customisation with per-project skill forks.
  - Do not let audit evidence become a silent competing truth source; specs remain authoritative through explicit reconciliation.
  - Improvements should distinguish between globally stronger defaults and per-repo policy/configuration.
- **Out of Scope**:
  - Full workflow redesign.
  - Replacing `spec-driver` with a general-purpose workflow skill.
  - New runtime command gates beyond what existing commands already enforce.

## 3. Architecture Intent
- **Target Outcomes**:
  - Agents choose the governing skill path earlier and more consistently.
  - The `spec-driver` skill remains the canonical route for CLI/entity interactions with spec-driver artifacts.
  - Booted guidance reinforces routing without erasing project posture or local customisation seams.
  - DR authoring uses progressive disclosure plus code-adjacent examples to sharpen design intent before planning.
  - Adversarial review stays inside the DR loop as a required internal pass, with optional external challenge after integration rather than as a mandatory global ceremony.
  - Relevant ADRs, policies, and standards are consulted before drafting and revisited during critical review so governance is not an afterthought.
  - Audit work becomes the concrete bridge from observed implementation truth back to authoritative specs, especially for tech-spec maintenance.
  - Closure can distinguish "verified and reconciled" from "implemented but still drifting", rather than relying on coverage blocks alone.
  - Audit-loop implementation ownership is split cleanly: `DE-055` defines workflow doctrine and skill expectations; `DE-079` implements the artifact/schema/runtime contract.
- **Guiding Principles**:
  - Keep conceptual guidance in memories and procedural guidance in skills.
  - Prefer one narrow skill per responsibility over a monolithic workflow meta-skill.
  - Uniform install, local adaptation: package-managed skills stay reusable across installs; hooks and generated docs absorb project-specific variation.
  - Make reconciliation explicit and auditable; if audits discover drift, the workflow must force a disposition rather than trusting narrative prose.
- **State Transitions / Lifecycle Impact**:
  - This delta has landed its workflow-guidance changes; remaining runtime audit work and further authorship-skill tuning now live in follow-up deltas.
  - No normative lifecycle states change, but the landed guidance aligns workflow expectations with ADR-004 and ADR-008 so observed evidence prompts explicit reconciliation before closure.

## 7. Design Decisions & Trade-offs
- `DEC-055-001` accepted: keep `spec-driver` focused on spec-driver CLI/entity handling.
- `DEC-055-002` accepted: introduce stricter routing as a separate concern, not as a widening of `spec-driver`.
- Optional future pattern: brainstorming can be valuable for authoring/design work, but should enter as an explicit opt-in or targeted routing branch, not as universal workflow doctrine.
- Brainstorming appears to contain two separable values:
  - a generic tight loop for resolving one open question at a time with options, tradeoffs, recommendation, and write-back
  - a stronger authoring choreography where designs are presented section by section before final write-out
- The generic loop is plausibly reusable across deltas, capture, DRs, and specs.
- The section-by-section authoring pattern is more artifact-specific and should likely land first in `draft-design-revision`, `shape-revision`, and similar skills rather than in one monolithic imported skill.
- The highest-leverage shared step is explicit triage of open questions, risks, underspecified areas, assumptions, and critical design decisions before section drafting begins; downstream quality depends on this step being strong.
- First implementation target: `draft-design-revision`, because DRs benefit most from early correction of foundational assumptions and from concrete design detail before planning.
- DR authoring should bias toward small code-adjacent examples when a paragraph would otherwise stay vague about module boundaries, APIs, or tricky control flow.
- The planning agent should challenge its own DR before declaring it coherent, attacking vague language, missing samples, weak verification, and stale delta framing.
- Relevant ADRs, policies, and standards should be gathered up front and then treated as part of the adversarial checklist, not as optional reading that can be skipped once drafting starts.
- Once that internal review is integrated, the skill may offer a prompt for an external adversarial reviewer, but the optional external pass should not block local coherence work.
- The owning delta must be updated after DR feedback lands so scope, risks, acceptance criteria, and next-step guidance do not lag behind the design revision.
- For non-trivial work, missing or stale DR design should be a routing/planning stop condition. Agents should return to `draft-design-revision` before creating or refining IP/phase work.
- Delta closure should include backlog follow-through when the work originated from backlog, but the skill must not invent unsupported backlog status semantics.
- Additional enforcement targets now identified from live use: lifecycle status discipline, stronger artefact ordering, task-level note capture, and explicit commit policy.
- Handover quality matters in multi-phase execution; `/continuation`, `/next`, and `/preflight` should preserve unresolved questions and tensions instead of collapsing them into a generic "ready" posture.
- Ordering guidance should live first in the routing layer so agents do not misread "active delta" as permission to skip DR/IP/phase creation.
- Backlog-oriented workflow skills need a stable lifecycle vocabulary before they can prescribe status-aware transitions with confidence; `ISSUE-009` is the current blocker.
- Generic `/notes` should remain card-oriented; structured DE/IP/phase/DR maintenance is a separate responsibility and should be modelled as such.
- Trade-off: stronger routing reduces drift, but too much global ceremony would violate posture-aware doctrine.

### Current Audit Mechanism Assessment

- This section now serves as the doctrinal/design source for `DE-079`, not as the implementation home for audit schema/runtime details.

- The doctrinal story is coherent:
  - `ADR-004` and `mem.pattern.spec-driver.core-loop` define `implement -> audit/contracts -> revision/spec reconcile -> close`.
  - `ADR-008` says observed evidence must trigger explicit reconciliation instead of silently replacing spec truth.
  - `mem.concept.spec-driver.audit` frames audits as the verify step that feeds post-audit spec reconciliation.
- The runtime story is not coherent enough yet:
  - `audit-change` instructs spec reconciliation but does not require a structured AUD artefact or a per-finding disposition.
  - `create audit` scaffolds only a light shell, with no owning-delta linkage and no canonical reconciliation block.
  - Audit metadata supports `findings`, `patch_level`, and `next_actions`, but the closure path does not consume them.
  - `complete delta` only checks parent-spec coverage completeness; it can close work that never produced an audit or never reconciled drift found during audit.
  - Registry sync records `verified_by` audit IDs and coverage evidence, but unresolved audit drift is not propagated as a first-class warning or error that closure must address.
  - Audit lifecycle language is inconsistent today (`complete`, `completed`, `approved`, `in-review`), which weakens both authoring and validation.

### Proposed Audit Contract

- Qualifying work should pass through an explicit audit checkpoint after implementation and before closure.
- A delta qualifies for audit gating when it changes code or generated contracts in a way that should reconcile against specs, requirements, or other declared normative artifacts.
- In practice, "qualifying" means one or more of:
  - the delta implements, updates, or verifies named requirements
  - the delta changes spec-governed behaviour, interfaces, contracts, or lifecycle claims
  - the delta changes technical reality that would leave a TECH spec misleading if audit and reconciliation were skipped
- Non-qualifying deltas may still produce audit-style evidence, but they should not be closure-gated by the stronger audit contract. Examples include purely editorial workflow docs, wording-only guidance edits, or mechanical refactors that intentionally preserve observed behaviour and do not change declared contracts.
- A qualifying audit should capture:
  - **mode**: `conformance` or `discovery`
  - **owner**: `delta_ref` when the audit belongs to a delta-driven change
  - **scope**: audited specs, requirements, code paths, and evidence window
  - **findings**: each finding tagged with outcome plus machine-checkable disposition
  - **reconciliation**: for each divergent or risky finding, link one of:
    - spec already aligned; no patch required
    - spec patched directly
    - revision created or updated
    - follow-up delta or backlog item created
    - explicitly tolerated unresolved drift with rationale when posture allows
- The important shift is not "add more prose"; it is "every finding must end with a deterministic disposition".
- Concrete schema choices, lifecycle enum alignment, and `complete delta` gate mechanics now live in `DE-079`; this DR keeps the workflow/doctrine contract that those details must satisfy.

### Qualifying Delta Rule

- The closure gate should key off audit relevance, not "any delta whatsoever".
- The recommended rule is:
  - gate conformance audits for deltas that alter requirement-bearing or spec-governed behaviour
  - do not gate documentation-only or behaviour-preserving maintenance deltas that leave normative truth unchanged
  - allow projects to run optional audits for non-qualifying work without promoting them into closure blockers
- This keeps the doctrine aligned with `ADR-004` and `ADR-008` without imposing audit ceremony on work that cannot create meaningful behavioural drift.

### Standalone AUD Artifact Expectation

- A qualifying gated audit should produce a standalone `AUD-*` artifact.
- Reasoning:
  - closure needs a first-class artifact to reference and validate
  - per-finding disposition must survive handoff and fresh-agent review
  - revisions, coverage updates, and follow-up work need a stable reconciliation hub
- Embedded notes or coverage-only evidence may remain acceptable for:
  - non-qualifying deltas
  - exploratory discovery work that is not yet part of a closure gate
  - low-ceremony local checks that do not claim to satisfy the canonical audit checkpoint
- This is the compromise that preserves low-ceremony flexibility without weakening the audit contract for deltas that actually need reconciliation.

### Conformance vs Discovery Severity

- Conformance and discovery audits should not have identical closure semantics.
- Recommended posture:
  - **conformance audit**: closure-blocking for qualifying delta work until every drift or risk finding has a disposition and any required reconciliation artifacts exist
  - **discovery/backfill audit**: warning-first by default, because its job is to surface legacy truth and create follow-up work rather than pretending the repo was already in perfect conformance
- Discovery warnings should still escalate into blockers when:
  - the audit is explicitly attached to a delta that claims closure-ready conformance
  - the finding shows the owning spec is materially misleading for active implementation work
  - the finding lacks any owning follow-up artifact or rationale
- This preserves town-planner doctrine while avoiding brittle legacy adoption.

### Finding Disposition Shape

- The current `findings` plus `patch_level` plus `next_actions` split is too loose for closure gating.
- `DE-055` requires the workflow/doctrine outcome:
  - every finding ends with a machine-checkable disposition
  - the disposition records whether routing is still pending, has been reconciled, or has been accepted into owned follow-up work
  - the disposition records what kind of reconciliation path was used, such as alignment, spec patch, revision, follow-up delta/backlog, or tolerated drift
  - the disposition carries structured references and rationale so closure and fresh-agent review do not depend on prose-only interpretation
- `DE-079` now owns the concrete schema decision for this doctrine. At the time of handoff, that means:
  - an inline per-finding `disposition` object rather than a separate reconciliation block
  - derived closure/gating effect rather than a stored `closure_effect` field
  - optional structured drift-ledger references inside the disposition
  - `tolerated_drift` rather than `accepted_drift` as the explicit warning/blocking posture for unresolved but consciously routed drift

### Audit and Drift Ledger Compatibility

- The audit disposition model should be compatible with DE-065's drift-ledger primitive, but the two artifacts should not be collapsed into one.
- Their roles are different:
  - the `AUD-*` artifact is the closure-grade record for a specific implementation audit
  - the `DL-*` ledger is a durable cross-work tracker for drift that may outlive the originating audit or delta
- Recommended compatibility contract:
  - each audit finding may optionally link to one or more drift-ledger entries when the finding represents durable, cross-cutting, or discovery-heavy drift
  - drift entries created from audits should record that origin via `discovered_by.kind: audit` and a reference back to the audit or finding
  - the audit disposition should still state the actual reconciliation outcome: `spec_patch`, `revision`, `follow_up_delta`, `follow_up_backlog`, `tolerated_drift`, or equivalent
- A drift-ledger reference should therefore be an optional linked tracking surface, not the disposition kind itself.
- This avoids two failure modes:
  - treating the drift ledger as a competing truth source instead of a drift-management tool
  - forcing every audit to create ledger entries even when the finding is local, already reconciled, or closure-bounded
- Practical guidance:
  - **conformance audits** should usually close findings directly through audit disposition; create drift entries only when the issue clearly needs durable cross-work tracking
  - **discovery/backfill audits** are the stronger fit for optional drift-ledger linkage, because they often surface pre-existing drift that needs long-lived adjudication rather than immediate delta closure
- Closure should never consider "linked to a drift ledger" sufficient on its own for a blocking conformance finding. The finding still needs an owned disposition and any required follow-up artifact.

### Workflow Adaptation

1. **Execution handoff**
   - `execute-phase` still hands off to `audit-change` when implementation evidence is ready.
2. **Audit pass**
   - `audit-change` must create or update the relevant AUD artefact for qualifying work.
   - The skill must gather verification evidence, update coverage blocks, and record per-finding dispositions.
3. **Reconciliation**
   - If findings indicate drift, the skill must patch specs and/or create the needed revision or follow-up work before closure.
   - Product specs may tolerate forward intent if lifecycle and coverage state remain explicit.
   - Tech specs should usually be reconciled to observed truth before closure because coherence is architectural, not merely per-requirement.
4. **Closure**
   - `close-change` and `complete delta` should treat missing required audits or undispositioned conformance drift as incomplete close-out, not as optional hygiene.
   - Discovery audits should surface unresolved findings as explicit warnings unless the audit has been promoted into a closure-critical conformance checkpoint.

### Recommended Tooling Shape

- `audit-change` becomes the canonical procedural owner of the reconciliation loop.
- `DE-079` should implement the concrete contract across:
  - audit metadata/template shape
  - audit scaffolding
  - delta close-out gating
  - validation and derived reporting
- `DE-055` remains responsible for the workflow posture those implementations must preserve:
  - conformance audits as closure-grade reconciliation checkpoints
  - discovery audits as warning-first by default
  - drift-ledger linkage as optional durable tracking, not closure truth

### Trade-offs

- Stronger audit gating raises ceremony for some deltas.
- That trade is acceptable for requirement-bearing or spec-shaping work because the whole point is to stop "implemented now, fix docs later" drift.
- Requiring a standalone `AUD-*` for qualifying gated work raises formality, but it also gives closure, validation, and handoff a stable reconciliation artifact.
- Discovery or backfill work should remain warning-first unless it is being used to justify closure of a conformance-bound delta; otherwise legacy adoption becomes needlessly brittle.
- Keeping `spec-driver` narrow still matters: audit governance belongs in `audit-change`, `close-change`, and completion rules, not in a generalized meta-skill.
