# DR-055 – Tighten skill routing and boot-time workflow guidance

## 1. Executive Summary
- **Delta**: [DE-055](./DE-055.md)
- **Status**: draft
- **Owners / Team**: david
- **Last Updated**: 2026-03-07
- **Synopsis**: Tighten skill invocation by making routing an explicit boot-time concern while preserving spec-driver's split between canonical packaged skills and project-owned hook customisation.

## 2. Problem & Constraints
- **Current Behaviour**:
  - `spec-driver` is effective when the task already points at spec-driver entities or commands.
  - Boot establishes onboarding and policy context, but it does not yet seem to enforce a stronger routing discipline comparable to the best parts of `using-superpowers`.
  - Agents can still drift into ad hoc exploration before deciding which skill governs the task.
  - Several live workflow failures remain outside pure routing:
    - deltas can be completed while still `draft`
    - agents may skip DR, IP, or phase sheets and rush to implementation
    - `/notes` is easy to omit after task completion
    - commit expectations are underspecified
    - ordering between DR, IP, phase creation, and execution is not stated strongly enough
    - capture/backlog work lacks a dedicated skill and end-to-end connective tissue into and out of delta execution
- **Drivers / Inputs**:
  - `ADR-004` requires a single canonical workflow doctrine.
  - `ADR-005` makes memories and skills the canonical guidance layers.
  - `PROD-016` says configuration should stay small and project customisation should come through generated guidance and hooks rather than process-key explosion.
  - `/tmp/superpowers` demonstrates useful strictness about choosing the governing skill before acting.
  - `ISSUE-009` shows backlog-item lifecycles still lack canonical status enums, which blocks precise workflow wording for backlog-oriented skills.
- **Constraints / Guardrails**:
  - Do not collapse posture-aware workflow into universal mandatory brainstorming or planning.
  - Do not let routing guidance compete with memories, skills, ADRs, or specs for doctrinal ownership.
  - Do not replace user-owned customisation with per-project skill forks.
  - Improvements should distinguish between globally stronger defaults and per-repo policy/configuration.
- **Out of Scope**:
  - Full workflow redesign.
  - Replacing `spec-driver` with a general-purpose workflow skill.
  - New runtime command gates beyond what existing commands already enforce.

## 3. Architecture Intent
- **Target Outcomes**:
  - Agents choose the governing skill path earlier and more consistently.
  - The `spec-driver` skill remains the canonical route for CLI/entity interactions with spec-driver artifacts.
  - Booted guidance reinforces routing without erasing project posture or local customisation seams.
  - Optional high-value patterns such as brainstorming and adversarial review can be added later as composable aids rather than mandatory gates.
- **Guiding Principles**:
  - Keep conceptual guidance in memories and procedural guidance in skills.
  - Prefer one narrow skill per responsibility over a monolithic workflow meta-skill.
  - Uniform install, local adaptation: package-managed skills stay reusable across installs; hooks and generated docs absorb project-specific variation.
- **State Transitions / Lifecycle Impact**:
  - This delta is in research/planning.
  - No direct lifecycle model changes are expected; this is guidance and install-surface behaviour.

## 6. Supporting Context
- **Research**:
  - `using-superpowers` is strongest where it forbids workflow drift before selecting a governing skill.
  - Its universal brainstorming and execution ceremony is not compatible with this repo's posture-aware model.
- **Hypotheses**:
  - Explicit boot-chain linkage to `spec-driver` will further reduce ad hoc file-search and CLI-guessing failures.
  - A separate routing skill will preserve clearer responsibilities than widening `spec-driver`.
- **Related Deltas / Specs**:
  - `DE-032` and `DE-034` for skill installation/sync history
  - `SPEC-151` for the skills implementation surface
  - `PROD-011` and `PROD-002` for workflow guidance surfaces

## 7. Design Decisions & Trade-offs
- `DEC-055-001` proposed: keep `spec-driver` focused on spec-driver CLI/entity handling.
- `DEC-055-002` proposed: introduce stricter routing as a separate concern, likely a separate skill or explicit boot guidance, not as a widening of `spec-driver`.
- Optional future pattern: brainstorming can be valuable for authoring/design work, but should enter as an explicit opt-in or targeted routing branch, not as universal workflow doctrine.
- Optional future pattern: adversarial review can be valuable as a fresh-agent challenge pass over a DR or spec, but it should remain distinct from implementation-phase audit semantics.
- Additional enforcement targets now identified from live use: lifecycle status discipline, stronger artefact ordering, task-level note capture, and explicit commit policy.
- Backlog-oriented workflow skills need a stable lifecycle vocabulary before they can prescribe status-aware transitions with confidence; `ISSUE-009` is the current blocker.
- Trade-off: stronger routing reduces drift, but too much global ceremony would violate posture-aware doctrine.
