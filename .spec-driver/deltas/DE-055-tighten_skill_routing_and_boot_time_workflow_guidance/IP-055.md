## 1. Summary

- **Delta**: DE-055 - tighten skill routing and boot-time workflow guidance
- **Specs Impacted**: `SPEC-151`, `PROD-011`, `PROD-002`
- **Problems / Issues**: workflow drift before skill selection; CLI/entity tasks not always routed through `spec-driver`; unclear explicit statement of packaged-skill plus hook-stub customisation principle; weak enforcement around delta status, artefact ordering, task notes, and commit policy; DR authoring still needs stronger code-level specificity, governance consultation, adversarial review, and explicit DE reconciliation before planning; agents still too often skip DR and try to jump straight into IP/phase planning; delta closure can leave originating backlog items stale
- **Desired Outcome**: a concrete design for stricter, repo-native skill routing and DR authoring that strengthens boot-time discipline, improves the DR feedback loop, keeps governance in the loop, preserves the generated-guidance plus user-hook model, blocks non-trivial DR skipping before IP handoff, and nudges backlog follow-through during delta closure

## 2. Context & Constraints

- **Current Behaviour**: boot and `spec-driver` each do part of the job, but there is no explicit always-first routing layer that chooses the governing skill path before substantive work.
- **Target Behaviour**: booted guidance should strongly steer skill selection; `spec-driver` should remain the mandatory CLI-first path when interacting with spec-driver entities; project-specific variation should continue to flow through hooks and generated docs.
- **Dependencies**: installer and skill-sync surfaces may need updates if a new routing skill is introduced.
- **Constraints**:
  - Respect ADR-004 workflow canon and posture variance.
  - Respect ADR-005 guidance-layer ownership.
  - Respect `PROD-016` generated vs user-owned file semantics.
