## 1. Summary
- **Delta**: DE-055 - tighten skill routing and boot-time workflow guidance
- **Specs Impacted**: `SPEC-151`, `PROD-011`, `PROD-002`
- **Problems / Issues**: workflow drift before skill selection; CLI/entity tasks not always routed through `spec-driver`; unclear explicit statement of packaged-skill plus hook-stub customisation principle
- **Problems / Issues**: workflow drift before skill selection; CLI/entity tasks not always routed through `spec-driver`; unclear explicit statement of packaged-skill plus hook-stub customisation principle; weak enforcement around delta status, artefact ordering, task notes, and commit policy
- **Desired Outcome**: a concrete design for stricter, repo-native skill routing that strengthens boot-time discipline without violating doctrine or the generated-guidance plus user-hook model

## 2. Context & Constraints
- **Current Behaviour**: boot and `spec-driver` each do part of the job, but there is no explicit always-first routing layer that chooses the governing skill path before substantive work.
- **Target Behaviour**: booted guidance should strongly steer skill selection; `spec-driver` should remain the mandatory CLI-first path when interacting with spec-driver entities; project-specific variation should continue to flow through hooks and generated docs.
- **Dependencies**: installer and skill-sync surfaces may need updates if a new routing skill is introduced.
- **Constraints**:
  - Respect ADR-004 workflow canon and posture variance.
  - Respect ADR-005 guidance-layer ownership.
  - Respect `PROD-016` generated vs user-owned file semantics.
