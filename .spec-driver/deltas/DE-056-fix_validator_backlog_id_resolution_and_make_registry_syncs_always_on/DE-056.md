# DE-056 – Fix validator backlog ID resolution and make registry syncs always-on

## 1. Summary & Context
- **Implementation Plan**: [IP-056](./IP-056.md)
- **Change Drivers**: ISSUE-016 (backlog/requirements relation traversal gaps),
  validation errors on DE-049/052/053/054 (`implements -> IMPR-xxx` rejected)

## 2. Motivation

`spec-driver validate` rejected valid `implements -> IMPR-xxx` relations on four
deltas because the validator only accepted FR/NF requirement IDs as `implements`
targets. Backlog item IDs (IMPR, ISSUE, etc.) were already accepted for
`applies_to` but not for `implements`.

Root cause had two layers:
1. Validator used `load_backlog_registry()` (reads `backlog.yaml` ordering file)
   instead of `discover_backlog_items()` (filesystem walk). The ordering file was
   stale — missing IMPR-006 through IMPR-009.
2. Even with correct IDs, backlog items weren't in the valid target set for
   `implements` relations.

While investigating, we found that both `--backlog` and `--adr` registry syncs
were opt-in flags on `spec-driver sync`, meaning registries would silently drift
unless users remembered to pass these flags.

## 3. Scope & Objectives
- **Primary Outcomes**:
  - Validator accepts backlog item IDs as valid `implements` targets
  - Validator uses filesystem discovery instead of stale ordering file
  - ADR and backlog registry syncs run unconditionally on `spec-driver sync`

## 4. Out of Scope
- Full backlog-to-requirements integration (ISSUE-016 phase 1+)
- Making `--contracts` or `--specs` opt-in (potential future work)

## 5. Approach Overview
- **System Touchpoints**: `validator.py`, `sync.py`
- **Key Changes**:
  - `validator.py`: Switch from `load_backlog_registry` to `discover_backlog_items`;
    add `backlog_ids` to valid `implements` target set
  - `sync.py`: Remove `--backlog` and `--adr` opt-in flags; both sync unconditionally

## 6. Verification Strategy
- **Acceptance Criteria**:
  - `spec-driver validate` passes (was failing with 4 errors)
  - `spec-driver sync` (no flags) shows ADR + backlog sync output
  - All existing tests pass

## 7. Risks & Mitigations
- **Risk**: Removing CLI flags is a breaking change for scripts using `--backlog`/`--adr`
  – *Likelihood*: low – *Impact*: low – *Mitigation*: flags were additive opt-ins,
  removal just means the behaviour is now default. No scripts would break from
  *more* syncing happening.
