## 1. Summary & Context
- **Change Drivers**: DE-053 exit criteria, DE-050 (normalised registry API),
  IMPR-010 ("backlog undercooked")
- **Related Issues**: ISSUE-016, ISSUE-024, ISSUE-026, ISSUE-034, ISSUE-043,
  ISSUE-045

## 2. Motivation

BacklogRegistry is the only registry that uses a function-based API
(`discover_backlog_items()`, `find_item()`) rather than a class with the
`collect/find/iter/filter` interface established by DE-050. This forces
every consumer (TUI artifact view, CLI list, future search/export) to
maintain a shim adapter.

Normalising to a class-based registry eliminates the shim in
`core/artifact_view.py`, simplifies the `_REGISTRY_FACTORIES` map, and
unblocks backlog subsystem improvements.

## 3. Scope & Objectives
- **Primary Outcomes**:
  - `BacklogRegistry` class with `collect/find/iter/filter` API matching
    DE-050 convention
  - Remove `_collect_backlog()` shim from `core/artifact_view.py`
  - Add `BacklogRegistry` to `_REGISTRY_FACTORIES` like all other registries
- **Dependencies**: None (DE-050, DE-053 completed)

## 4. Out of Scope
- Backlog sub-kind type splitting (issues/problems/improvements as distinct
  ArtifactTypes) — potential future delta
- Full triage of ISSUE-016, -024, -026, -034, -043, -045 — evaluate as
  potential riders during DR/IP

## 5. Approach Overview
- **System Touchpoints**: `supekku/scripts/lib/backlog/registry.py`,
  `supekku/scripts/lib/core/artifact_view.py`
- **Key Changes**:
  - New `BacklogRegistry` class wrapping existing `discover_backlog_items()`
  - Update `artifact_view.py` to use standard factory pattern
  - Update CLI consumers if any call functions directly

## 6. Verification Strategy
- Unit tests for `BacklogRegistry.collect/find/iter/filter`
- Existing TUI + CLI tests continue to pass
- `just` passes

## 7. Risks & Mitigations
- **Risk**: Breaking existing backlog CLI commands — *Likelihood*: low —
  *Impact*: medium — *Mitigation*: Preserve function API as thin wrappers
  around class methods during transition
