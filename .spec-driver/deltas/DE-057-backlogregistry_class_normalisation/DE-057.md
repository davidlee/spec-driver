## 2. Motivation

BacklogRegistry is the last function-based registry in the codebase. Every
consumer — CLI (`common.py`, `list.py`, `sync.py`), TUI (`artifact_view.py`),
and future search/export — must work around the absence of the standard
`collect/find/iter/filter` API. The TUI carries a dedicated `_collect_backlog()`
shim with duplicated error-isolation logic and a special-case branch in
`ArtifactSnapshot._load_type()`.

Beyond the registry gap, the backlog subsystem has accumulated operational
rough edges: undefined status enums, broken dry-run threading, missing link
resolution support, and a greedy CLI flag. These are individually minor but
collectively make backlog a second-class citizen.

This delta normalises the registry to ADR-009, defines per-kind status enums,
and fixes the operational gaps — bringing backlog to parity with other
first-class artifact types.

## 3. Scope & Objectives

- **Primary Outcomes**:
  1. **BacklogRegistry class** conforming to ADR-009 convention:
     - `find(id) -> BacklogItem | None`
     - `collect() -> dict[str, BacklogItem]`
     - `iter(status=None, kind=None) -> Iterator[BacklogItem]`
     - `filter(kind=None, status=None, ...) -> list[BacklogItem]`
     - Constructor: `(*, root: Path | None = None)`
     - `kind` as a first-class filter parameter (not 4 separate registries)
  2. **Remove artifact_view.py shim**: delete `_collect_backlog()`, remove
     special-case in `_load_type()`, add `BacklogRegistry` to
     `_REGISTRY_FACTORIES`
  3. **Per-kind status sets** (ISSUE-009 partial): define accepted lifecycle
     status values per backlog item kind with permissive validation.
     Theme/rendering alignment remains as ISSUE-009 follow-up
  4. **Fix ISSUE-026**: thread `dry_run` parameter through `_sync_backlog()`
     in `cli/sync.py`
  5. **Fix ISSUE-034**: extend link resolver to support backlog item wikilinks
     (`[[ISSUE-016]]`, `[[PROB-002]]`, etc.)
  6. **Mitigate ISSUE-043**: add ID-format validation callback to
     `--from-backlog` in `cli/create.py` (Click root cause unfixable
     without CLI shape change)
  7. **Update CLI consumers**: `cli/common.py` resolvers, `cli/list.py`,
     `cli/sync.py` to use `BacklogRegistry` class
- **Operational Constraints**:
  - Existing module-level functions preserved as thin wrappers during
    transition (avoid breaking callers)
  - All existing tests must continue to pass
  - Both linters must pass (`just lint` + `just pylint`)
  - ADR-002 remains in force: no backlinks in frontmatter
- **Dependencies**: None (DE-050, DE-053 completed)

## 5. Approach Overview

### Design decision: unified registry, not speciation

Backlog items share a single `BacklogItem` model with kind-specific fields.
The priority-ordering mechanism operates across types. The primary interaction
surface is the unified, cross-type, prioritised backlog view. Four separate
registries would quadruple boilerplate for marginal gain. Instead,
`BacklogRegistry` supports kind-scoped `collect(kind=...)`, `iter(kind=...)`,
and `filter(kind=...)` — the `list issues` / `list improvements` CLI commands
become `filter(kind="issue")` underneath.

### System touchpoints

- `supekku/scripts/lib/backlog/registry.py` — new `BacklogRegistry` class
- `supekku/scripts/lib/backlog/models.py` — per-kind status enums
- `supekku/scripts/lib/core/artifact_view.py` — remove shim, add factory
- `supekku/cli/common.py` — simplify backlog resolvers
- `supekku/cli/list.py` — use `BacklogRegistry` for list backlog/issues/etc.
- `supekku/cli/sync.py` — thread `dry_run` through `_sync_backlog()`
- `supekku/cli/create.py` — fix `--from-backlog` flag parsing
- Link resolver (location TBD in DR) — add backlog item support

### Phased delivery

- **Phase 1**: BacklogRegistry class + status enums + tests. Core modelling.
- **Phase 2**: Remove artifact_view.py shim, update CLI consumers, fix
  operational gaps (ISSUE-026, -034, -043). Integration.
- **Phase 3**: Verification, cleanup, close backlog items.

## 7. Risks & Mitigations

- **Risk**: Breaking existing backlog CLI commands during registry transition
  — _Likelihood_: low — _Impact_: medium — _Mitigation_: Preserve module-level
  functions as thin wrappers; grep all callers before removing
- **Risk**: Status enum definitions contentious or incomplete — _Likelihood_:
  medium — _Impact_: low — _Mitigation_: Derive from existing TEMPLATES dict
  defaults and observed frontmatter values; provisionally in scope, can split
  to separate delta during DR if it balloons
- **Risk**: `--from-backlog` fix may require Typer/Click workaround —
  _Likelihood_: medium — _Impact_: low — _Mitigation_: Research Click eager
  option patterns during DR; fallback to positional argument
