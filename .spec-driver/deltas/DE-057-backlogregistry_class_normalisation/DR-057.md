# DR-057 – BacklogRegistry class normalisation and backlog subsystem improvements

## 1. Executive Summary

- **Delta**: [DE-057](./DE-057.md)
- **Status**: draft
- **Last Updated**: 2026-03-07
- **Synopsis**: Normalise the backlog subsystem from function-based API to a
  class-based `BacklogRegistry` conforming to ADR-009, define per-kind status
  enums, and fix operational gaps (dry-run, link resolution, greedy flag).

## 2. Problem & Constraints

- **Current Behaviour**: Backlog uses module-level functions
  (`discover_backlog_items()`, `find_item()`, `find_backlog_items_by_id()`)
  instead of a class with `collect/find/iter/filter`. This forces every
  consumer to adapt: `artifact_view.py` has a 30-line `_collect_backlog()`
  shim duplicating error isolation, `_load_type()` has a backlog special-case,
  `cli/common.py` has bespoke resolver functions, and `cli/list.py` calls
  discovery functions directly. Status values are undefined — 14 distinct
  values observed in the wild with no validation. Sync doesn't respect
  dry-run. Link resolution ignores backlog items. `--from-backlog` swallows
  subsequent flags.
- **Drivers / Inputs**: ADR-009 (standard registry API), DE-050 Tier 3
  (deferred), DE-053 exit criteria (DEC-053-08), ISSUE-009, -026, -034, -043
- **Constraints / Guardrails**:
  - ADR-009 governs the API surface (find/collect/iter/filter, keyword-only
    root constructor)
  - ADR-002: no backlinks in frontmatter
  - Existing module-level functions must keep working during transition
  - Priority ordering (`priority.py`) remains a separate concern
  - `just` must pass (tests + both linters)
- **Out of Scope**: Backlog speciation into separate ArtifactTypes,
  requirements sync (ISSUE-016), prioritize UX (IMPR-010), Registry
  Protocol/ABC

## 3. Architecture Intent

- **Target Outcomes**:
  - `BacklogRegistry` is a first-class ADR-009-conformant registry
  - `artifact_view.py` treats backlog identically to all other types — no
    shim, no special-case
  - Per-kind status values are defined and validated
  - All 4 operational gaps fixed (ISSUE-009, -026, -034, -043)
- **Guiding Principles**:
  - Mirror the DecisionRegistry pattern — same constructor shape, same
    method signatures, domain-specific params where ADR-009 allows
  - Keep the class thin — wrap existing discovery/lookup functions, don't
    reimplement
  - Status validation is permissive (warn on unknown) rather than strict
    (reject), to avoid breaking existing items with ad-hoc statuses
- **State Transitions / Lifecycle Impact**: Status enum definitions codify
  the per-kind lifecycle but do not change any existing item's status value

## 7. Design Decisions & Trade-offs

### DEC-057-01: Single BacklogRegistry class with kind as domain-specific filter (decided)

**Context**: 4 backlog item types (issue, problem, improvement, risk) share
a single `BacklogItem` model. Options: (A) one registry with `kind` filter,
(B) 4 separate registries, (C) hybrid with sub-type awareness in ArtifactType.

**Decision**: Option A. Single `BacklogRegistry` with `kind` as a
domain-specific filter parameter on `iter()` and `filter()`.

**Rationale**: Items share a unified model. Priority ordering operates across
types. The primary interaction surface is the cross-type prioritised backlog.
4 registries = 4x boilerplate, 4 ArtifactType values, 4 factory entries —
for marginal gain. `list issues` = `filter(kind="issue")`.

**Consequence**: `ArtifactType.BACKLOG` remains a single enum value. TUI
shows "Backlog (57)". Per-kind views are a filter, not a type distinction.

### DEC-057-02: Per-kind status sets as frozenset constants (decided, revised after adversarial review)

**Context**: 14 distinct status values observed in the wild. Need validation
without breaking existing items. Adversarial review identified that the
original `TERMINAL_STATUSES` set conflated three concerns: allowed values,
terminal semantics, and default list visibility.

**Decision**: Define three distinct concepts in `backlog/models.py`:

1. **Per-kind accepted lifecycle values** — what statuses are intentional:

```python
ISSUE_STATUSES: frozenset[str] = frozenset({
  "open", "triaged", "in-progress", "resolved", "done",
  "implemented", "verified",
})
PROBLEM_STATUSES: frozenset[str] = frozenset({
  "captured", "investigating", "mitigated", "resolved",
})
IMPROVEMENT_STATUSES: frozenset[str] = frozenset({
  "idea", "planned", "in-progress", "implemented",
})
RISK_STATUSES: frozenset[str] = frozenset({
  "suspected", "confirmed", "mitigated", "accepted", "expired",
})

BACKLOG_STATUSES: dict[str, frozenset[str]] = {
  "issue": ISSUE_STATUSES,
  "problem": PROBLEM_STATUSES,
  "improvement": IMPROVEMENT_STATUSES,
  "risk": RISK_STATUSES,
}
```

2. **Default-hidden statuses** — matching current `list_backlog()` behaviour
   exactly (no behaviour change):

```python
# Matches current hardcoded exclusion in list_backlog()
DEFAULT_HIDDEN_STATUSES: frozenset[str] = frozenset({
  "resolved", "implemented",
})
```

3. **Validation** — permissive, via `is_valid_status()` helper that warns
   on unknown values (DEC-057-08).

**Rationale**: `frozenset` over `Enum` — simpler, no import ceremony, easy
to check membership. Separating "accepted lifecycle values" from
"default-hidden in list views" avoids the user-visible behaviour change the
adversarial review flagged: today only `resolved` and `implemented` are
hidden; broadening to `done`, `mitigated`, `accepted`, `expired` would
silently change list output.

**Consequence**: `list_backlog()` uses `DEFAULT_HIDDEN_STATUSES` (preserving
current behaviour). Per-kind status sets are available for validation,
IMPR-010 (status-aware checkboxes), and theme alignment. Theme/rendering
alignment for status styling is out of scope — ISSUE-009 cannot be fully
closed by this delta alone; this delta delivers the validation/modelling
half.

### DEC-057-03: Priority ordering stays external to registry class (decided)

**Context**: `priority.py` (360 lines) handles ordering, editor integration,
head-tail partitioning, and `backlog.yaml` persistence. Should this move
into `BacklogRegistry`?

**Decision**: No. Priority ordering remains in `priority.py` as a separate
concern. `BacklogRegistry` handles discovery and lookup. `priority.py`
handles ordering and persistence.

**Rationale**: The registry pattern (ADR-009) is about artifact
discovery/lookup, not presentation ordering. Mixing them would make
`BacklogRegistry` the only registry with ordering semantics — a design
outlier. The two modules compose cleanly: `BacklogRegistry.filter()` returns
items, `sort_by_priority()` orders them.

### DEC-057-04: Module-level functions preserved as thin wrappers (decided)

**Context**: Existing callers use `discover_backlog_items()`, `find_item()`,
etc. directly.

**Decision**: Keep module-level functions as thin wrappers delegating to
`BacklogRegistry` methods. No deprecation warnings in this delta — evaluate
removal in a future cleanup.

```python
def discover_backlog_items(*, root=None, kind="all"):
  """Wrapper for backwards compatibility."""
  registry = BacklogRegistry(root=root)
  kind_filter = kind if kind != "all" else None
  return list(registry.iter(kind=kind_filter))
```

**Rationale**: Minimises blast radius. Callers migrate at their own pace.
The wrappers are trivial and disposable. Breaking all callers simultaneously
is unnecessary risk for zero user-facing benefit.

### DEC-057-05: Eager loading with cache; collect() returns full dict (decided, revised after adversarial review)

**Context**: ADR-009 specifies `collect() -> dict[str, Record]` and allows
either lazy or eager loading. DE-057 wants kind-scoped collection. Adversarial
review flagged that naive layering of `find()`/`iter()` over `collect()` would
regress kind-scoped flows from targeted filesystem search to repeated
full-corpus scans.

**Decision**: Eager loading with cache:
- Constructor scans all items once and caches internally
  (`self._items: dict[str, BacklogItem]`)
- `collect()` returns the cached dict (no re-scan)
- `find(id)` reads from cache (O(1) lookup)
- `iter(kind=None, status=None)` filters the cache
- `filter(kind=None, status=None, ...)` filters the cache

This mirrors DecisionRegistry's pattern where `collect()` scans on each call
but the corpus is small. Backlog corpus is similarly small (~60 items) so
eager loading in the constructor is appropriate.

**Rationale**: ADR-009's `collect()` contract is "return complete dict." Adding
parameters diverges from the convention. Eager caching avoids the performance
regression the review identified — `find()` is O(1) from cache rather than
re-discovering all items. The backlog corpus size (~60 items) makes eager
loading cheap.

**Consequence**: `iter()` and `filter()` gain `kind` as a domain-specific
parameter — consistent with ADR-009's allowance for domain-specific filter
params. Short-lived registry instances (e.g. in CLI commands) pay the
one-time scan cost; long-lived instances (e.g. TUI snapshot) benefit from
the cache.

### DEC-057-06: Link resolver uses BacklogRegistry to populate artifact index (decided)

**Context**: `resolve.py:_build_artifact_index()` builds an
`ArtifactIndex = dict[str, tuple[Path, str]]` by calling per-type collection
functions. Backlog is missing (ISSUE-034).

**Decision**: Add `_collect_backlog_items()` to `resolve.py` that instantiates
`BacklogRegistry` and populates the index with entries keyed by ID (ISSUE-016,
PROB-002, etc.) mapped to `(item.path, item.kind)`.

```python
def _collect_backlog_items(root: Path, index: ArtifactIndex) -> None:
  from supekku.scripts.lib.backlog.registry import BacklogRegistry
  registry = BacklogRegistry(root=root)
  for item_id, item in registry.collect().items():
    index[item_id] = (item.path.relative_to(root), item.kind)
```

**Rationale**: Follows the exact pattern of `_collect_decisions()`,
`_collect_specs()`, etc. already in `resolve.py`. Simple, consistent.

### DEC-057-07: --from-backlog mitigation via ID validation (decided, revised after adversarial review)

**Context**: ISSUE-043 — `--from-backlog --help` consumes `--help` as the
backlog item ID because Click/Typer treats option values greedily. The
adversarial review confirmed this is a Click-level limitation: the parser
consumes the next token as the value regardless.

**Decision**: Mitigation, not root cause fix:
1. Add a callback that validates the value looks like a backlog item ID
   and raises `typer.BadParameter` with a clear message if not
2. Replace `discover_backlog_items()` with `BacklogRegistry.find()`

```python
def _validate_backlog_id(value: str | None) -> str | None:
  if value is None:
    return None
  if not re.match(r"^(ISSUE|PROB|IMPR|RISK)-\d+$", value):
    raise typer.BadParameter(
      f"'{value}' does not look like a backlog item ID "
      "(expected ISSUE-NNN, PROB-NNN, IMPR-NNN, or RISK-NNN)"
    )
  return value
```

**What this fixes**: `--from-backlog --spec SPEC-001` now gives a clear
error ("'--spec' does not look like a backlog item ID") instead of silently
consuming `--spec` and later failing with "backlog item '--spec' not found".

**What this does NOT fix**: `--from-backlog --help` still won't show help
for the `create delta` command — Click still consumes `--help` as the option
value. This is a Click architectural limitation. A full fix would require
changing the CLI shape (e.g. making `--from-backlog` a boolean flag with the
ID as a separate argument), which is out of scope for this delta.

**Rationale**: The mitigation is worth doing because it changes a confusing
silent failure into a clear error for the common case (accidentally omitting
the ID or mistyping flags). ISSUE-043 can be partially closed with a note
about the remaining `--help` edge case.

### DEC-057-08: Status frozensets represent accepted lifecycle values (decided)

**Context**: Adversarial review asked whether the frozensets represent
"observed current values", "allowed future lifecycle values", or
"UI/rendering vocabulary". The current corpus uses one set, theme styles
another, and the DR proposes a third.

**Decision**: The frozensets represent **accepted lifecycle values** — the
intentional superset of statuses for each kind. They are not an exhaustive
observation of the wild, nor a rendering vocabulary.

- Values derived from: TEMPLATES defaults + observed frontmatter + intentional
  lifecycle progression
- Unknown values (e.g. `draft`, `stub`, `adjudicated`) get a
  `logger.warning()` but are not rejected
- Theme/status-rendering alignment is a separate concern (ISSUE-009
  remainder) — this delta delivers the validation/modelling half only

**Rationale**: The alternative — deriving sets purely from observation —
would include accidental values and miss intentional ones. Defining accepted
values gives IMPR-010 and future theme work a stable vocabulary to build on.

### DEC-057-09: Shim removal changes error behaviour — test explicitly (decided)

**Context**: Adversarial review noted that `_collect_safe()` in
`artifact_view.py` emits per-record error placeholders (ArtifactEntry with
`error` field), while the current `_collect_backlog()` shim mostly logs
and skips adaptation failures.

**Decision**: After shim removal, backlog items flow through `_collect_safe()`
which produces per-record error placeholders on adaptation failure. This is
a behaviour change — previously failed backlog items were silently dropped;
now they appear as error entries in the TUI.

This is the **correct** behaviour (matches all other registries) but must be:
1. Called out as an intentional change, not treated as invisible refactoring
2. Tested explicitly in VT-057-artifact-view: verify that a malformed backlog
   item produces a degraded ArtifactEntry with an error message rather than
   being silently skipped

**Rationale**: Consistent error handling across all registries is better than
special-case silence for backlog. The TUI already renders error entries as
degraded state.
