## 1. Summary & Context

- **Product Spec(s)**: PROD-003 – policy/standard management; this delta uses
  that capability to add a repo governance standard for linting.
- **Implementation Plan**: [IP-058](./IP-058.md) – in progress
- **Change Drivers**:
  - Repository pylint output is dominated by `missing-function-docstring`, which
    reduces signal for more important warning families.
  - The current `pylint-per-file-ignores` glob only matches repo-root test
    files, so nested tests still emit warning noise.
  - The repo needs an explicit standard that discourages "pre-existing debt"
    rationalization when new warnings are introduced.
  - Agents need a faster, more compact pylint workflow than repeatedly reading
    the raw full-repo stream.

## 2. Motivation

- Establish a default standard for how this repo interprets pylint output and
  prioritizes cleanup work.
- Restore signal by fixing the broken recursive test-file ignore instead of
  weakening pylint globally.
- Make repo-level lint gates authoritative even when file-local runs are useful
  for diagnosis.

## 3. Scope & Objectives

- **Primary Outcomes**:
  - Publish a default Standard describing lint warning prioritization and
    anti-rationalization expectations.
  - Fix `pyproject.toml` so test-file ignores actually apply to nested tests.
  - Add a compact pylint reporting workflow for touched files and full-repo
    runs, with full json persisted to disk.
  - Keep production-code lint scrutiny intact.
- **Operational Constraints**:
  - Do not relax high-signal pylint checks globally.
  - Do not attempt broad warning cleanup in this delta.
- **Dependencies**: None beyond existing lint tooling.

## 4. Out of Scope

- Repo-wide cleanup of historical pylint debt.
- Revising fail-under scoring strategy beyond documenting how scores should be
  interpreted.
- Adding new lint tooling or replacing pylint.

## 5. Approach Overview

- **System Touchpoints**:
  - `pyproject.toml`
  - `Justfile`
  - `supekku/scripts/pylint_report.py`
  - `supekku/scripts/lib/core/pylint_report.py`
  - `.spec-driver/standards/STD-002-*.md`
  - DE/DR/IP/phase artefacts for DE-058
- **Key Changes**:
  - Replace non-recursive test-file ignore globs with recursive ones.
  - Author a default Standard that prioritizes high-signal warning families and
    rejects "existing debt makes new debt acceptable" reasoning.
  - Add `just pylint-report` and `just pylint-files ...` targets backed by a
    compact json2 summarizer.

## 6. Verification Strategy

- **Planned Artefacts**:
  - VA-DE-058-001: repo lint configuration review and before/after reasoning
  - VA-DE-058-002: `just pylint` after config/doc changes
  - VA-DE-058-003: `just pylint-report` / `just pylint-files ...` workflow validation
- **Acceptance Criteria**:
  - STD-002 exists with concrete guidance and traceability to DE-058.
  - Recursive test-file ignores are configured in `pyproject.toml`.
  - `just pylint` runs with reduced test-docstring noise and no new
    configuration errors.
  - Compact summary targets persist full json and surface message counts/files
    without requiring raw full-stream inspection.

## 7. Risks & Mitigations

- **Risk**: Recursive ignores hide warnings outside tests.
  - _Likelihood_: low
  - _Impact_: medium
  - _Mitigation_: Limit patterns to test naming conventions only.
- **Risk**: Standard is interpreted as permission to ignore all low-signal
  warnings forever.
  - _Likelihood_: medium
  - _Impact_: medium
  - _Mitigation_: State that lower priority is not blanket exemption and keep
    repo-level lint gates in force.

## 8. Follow-ups & Tracking

- **Future Deltas**:
  - Revisit pylint score gating separately once warning-family signal is
    improved and current debt is better categorized.
- **Open Decisions / Questions**:
  - Whether repo-level `fail-under` should become a meaningful ratchet rather
    than a permissive floor is intentionally deferred.

## 9. Implementation Notes

- Local file-scoped pylint runs may be used to diagnose warning families, but
  they do not replace repo-level verification.
