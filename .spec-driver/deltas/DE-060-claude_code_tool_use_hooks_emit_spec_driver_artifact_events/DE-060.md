## 1. Summary & Context

- **Implementation Plan**: [IP-060](./IP-060.md)
- **Change Drivers**: DE-052 (CLI event emission), DE-054 (TUI track mode),
  DE-059 (auto-follow). The TUI currently only sees spec-driver CLI invocations.
  Agent file reads/writes of spec-driver artifacts are invisible.

## 2. Motivation

When an agent reads or edits a spec-driver artifact (delta, spec, ADR, etc.),
the TUI track view has no visibility — it only sees `spec-driver` CLI command
events. Claude Code hooks (`pre`/`post` on `Read`/`Edit`/`Write` tools)
provide a natural interception point. A small hook script can pattern-match
the file path, classify the artifact type, and emit a synthetic event through
the existing JSONL log + Unix socket pipeline.

This gives the TUI real-time awareness of _which artifacts the agent is
touching_, not just which spec-driver commands it runs.

## 3. Scope & Objectives

- **Primary Outcomes**:
  - Claude Code `post` hooks on `Read`, `Edit`, `Write` tools detect
    spec-driver artifact paths and emit structured events
  - Events use the existing `events.py` JSONL + socket emission pipeline
  - TUI track view displays artifact-touch events alongside CLI events
  - Session ID is propagated from the Claude Code environment
- **Dependencies**: DE-052 (event infrastructure), DE-054 (TUI track mode)

## 4. Out of Scope

- Modifying the event schema version (v1 is sufficient)
- Custom TUI rendering for artifact events (standard TrackEvent display)
- Hook configuration UI or per-project hook customisation
- Watching non-spec-driver files

## 5. Approach Overview

- **System Touchpoints**:
  - `supekku/claude.hooks/` — new hook script (co-located with existing
    `startup.sh`)
  - `.claude/settings.json` — register new hook entries
  - `supekku/scripts/lib/core/events.py` — reuse existing emission API
  - `supekku/tui/widgets/track_panel.py` — may need minor formatting for
    artifact-touch event type
- **Key Changes**:
  1. Hook script that receives tool-use JSON from Claude Code, extracts
     the file path, matches against spec-driver artifact path patterns
     (`.spec-driver/deltas/`, `.spec-driver/specs/`, `.spec-driver/decisions/`,
     etc.), classifies artifact type and ID, emits event
  2. Event shape: `{v: 1, ts, session, cmd: "artifact.read|write",
artifacts: [artifact_id], tool: "Read|Edit|Write", path: "..."}`
  3. Session detection: `SPEC_DRIVER_SESSION` env var already propagated
     by existing `startup.sh` hook (via `CLAUDE_ENV_FILE`)

## 6. Verification Strategy

- **VT-060-01**: Hook script correctly classifies artifact paths and emits
  events (unit test with mock paths)
- **VT-060-02**: Non-artifact paths are silently ignored (no event emitted)
- **VT-060-03**: Events appear in JSONL log with correct schema
- **VH-060-01**: Manual smoke — agent edits a delta file, TUI track view
  shows the artifact-touch event

## 7. Risks & Mitigations

- **Risk**: Hook latency slows tool execution — _Likelihood_: low —
  _Impact_: medium — _Mitigation_: hook is fire-and-forget, fail-silent,
  same pattern as `events.py`
- **Risk**: Claude Code hook API shape changes — _Likelihood_: low —
  _Impact_: medium — _Mitigation_: thin adapter, easy to update
- **Risk**: Session ID not available in hook env — _Likelihood_: medium —
  _Impact_: low — _Mitigation_: fall back to "claude" session label
  (existing `_detect_session()` pattern)
