# DR-060 – Claude Code tool-use hooks emit spec-driver artifact events

## 1. Executive Summary

- **Delta**: [DE-060](./DE-060.md)
- **Status**: draft
- **Synopsis**: Add a Claude Code PostToolUse hook that detects when an agent
  reads/edits/writes a spec-driver artifact and emits a structured event to
  the existing JSONL + socket pipeline, making artifact activity visible in
  the TUI track view.

## 2. Problem & Constraints

- **Current Behaviour**: The TUI track view only sees `spec-driver` CLI command
  events. When an agent reads a delta file, edits an IP, or writes a phase
  sheet, these touches are invisible.
- **Drivers**: DE-052 (event infrastructure), DE-054 (track view), DE-059
  (auto-follow) — this is the natural next step in agent observability.
- **Constraints**:
  - Hook must be async (non-blocking) — zero latency impact on tool execution
  - Fail-silent — hook errors must never disrupt the agent
  - Must work with the existing event schema (v1)
- **Out of Scope**: Custom TUI rendering for artifact events, fswatch-based
  approaches, hook configuration UI.

## 3. Architecture Intent

- **Target Outcomes**:
  - Artifact reads/writes by the agent appear as events in the TUI track view
  - Events include artifact type, artifact ID, file path, tool name, and session
  - No changes to the core event schema or listener infrastructure
- **Guiding Principles**: Reuse existing emission pipeline. Keep the hook
  script minimal and testable. Fail-silent everywhere.

## 7. Design Decisions & Trade-offs

### DEC-060-01: PostToolUse async hook on Read|Edit|Write

**Decision**: Register an async `PostToolUse` hook with matcher `Read|Edit|Write`.
The hook fires after each successful file tool call, receives the full JSON
context on stdin including `session_id`, `tool_name`, and `tool_input.file_path`.

**Rationale**: PostToolUse fires after the tool succeeds (confirmed action).
Async mode (`"async": true`) means zero blocking — the hook runs in the
background and Claude continues immediately. This matches the fail-silent
design of the existing event infrastructure.

**Alternative considered**: PreToolUse — rejected because it fires before
the tool runs and adds pre-execution latency. Also, for Write/Edit, we want
to emit only on success.

### DEC-060-02: Self-contained Python script — no supekku imports

**Decision**: The hook script is a self-contained Python script that:

1. Reads JSON from stdin (`json.load(sys.stdin)`)
2. Extracts `tool_input.file_path`
3. Matches against spec-driver artifact path patterns
4. If match: constructs event dict, appends JSONL line to log file,
   fire-and-forget sends to Unix socket
5. If no match: exits silently

**Rationale**: The hook runs as `python3 script.py`, not via `uv run`,
so `supekku` is not importable. The emission logic is trivial (~15 lines
for JSONL append + socket send) and duplicating it is a lesser coupling
cost than `sys.path` hacking or requiring `uv run` overhead on every
tool call. The script is a boundary adapter — self-containment is a
feature, not a compromise.

**Path classification patterns**:

```
.spec-driver/deltas/DE-*    → delta (+ DR/IP/phase/notes by filename)
.spec-driver/tech/SPEC-*    → spec
.spec-driver/decisions/     → adr
.spec-driver/revisions/     → revision
.spec-driver/audits/        → audit
.spec-driver/backlog/       → backlog
.spec-driver/policies/      → policy
.spec-driver/standards/     → standard
.spec-driver/product/       → product_spec
```

### DEC-060-03: Session ID from hook JSON, not env var

**Decision**: Use `session_id` from the PostToolUse JSON input rather than
`SPEC_DRIVER_SESSION` from the environment.

**Rationale**: The hook JSON always includes `session_id` — it's authoritative
and requires no env-var propagation. The env var is a fallback for CLI
commands that don't receive hook JSON.

## 8. Event Shape

The emitted event reuses the v1 schema with a synthetic `cmd` field:

```json
{
  "v": 1,
  "ts": "2026-03-08T12:34:56.789+00:00",
  "session": "abc123",
  "cmd": "artifact.read",
  "argv": ["artifact.read", ".spec-driver/deltas/DE-060/DE-060.md"],
  "artifacts": ["DE-060"],
  "exit_code": 0,
  "status": "ok"
}
```

- `cmd`: `artifact.read` | `artifact.edit` | `artifact.write`
  (derived from `tool_name`)
- `artifacts`: extracted artifact ID(s) from the path
- `argv[1]`: relative path to the artifact file

## 9. Hook Registration

In `.claude/settings.json`:

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Read|Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "python3 \"$CLAUDE_PROJECT_DIR/supekku/claude.hooks/artifact_event.py\"",
            "async": true
          }
        ]
      }
    ]
  }
}
```
