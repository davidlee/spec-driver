## 1. Summary & Context

- **Implementation Plan**: [IP-064](./IP-064.md)
- **Change Drivers**: ISSUE-022, ISSUE-015

## 2. Motivation

There is no single command to verify that a spec-driver workspace is
healthy. Users and agents discover problems piecemeal — a missing
dependency here, a broken cross-reference there, a stale registry
elsewhere. Debugging is ad hoc.

`spec-driver doctor` would provide a structured diagnostic report:
dependency checks, configuration validation, registry integrity,
cross-reference consistency, and actionable fix suggestions.

This is especially valuable for:

- New project onboarding (is the install complete?)
- Post-upgrade verification (did the migration work?)
- Agent preflight (is the workspace in a good state to work in?)
- Debugging obscure failures (why did `complete delta` reject this?)

## 3. Scope & Objectives

- **Primary Outcomes**:
  - `spec-driver doctor` command with structured diagnostic output
  - Check categories (each independently runnable):
    - **deps**: Python version, required packages (uv, watchfiles, etc.)
    - **config**: workflow.toml validity, CLAUDE.md references, skills
      allowlist
    - **structure**: expected directories exist, no orphaned bundles
    - **registries**: all registries load without errors, ID uniqueness
    - **refs**: cross-references resolve (relations, applies_to, etc.)
    - **lifecycle**: no stuck in-progress deltas older than N days, no
      draft artifacts with implementation work
  - Per-check: pass/warn/fail with actionable message
  - Summary: total pass/warn/fail counts, exit code reflects worst status
  - `--check <category>` to run a single category
  - `--json` for machine-readable output
- **Dependencies**: None

## 4. Out of Scope

- Auto-fix / repair mode (future follow-up — doctor diagnoses, doesn't
  treat)
- Performance profiling or benchmarks
- Remote/network checks (GitHub, npm registry, etc.)

## 5. Approach Overview

- **System Touchpoints**: new `supekku/cli/doctor.py`, new
  `supekku/scripts/lib/diagnostics/` package for check implementations
- **Key Changes**:
  1. Diagnostic check framework: registry of checks, each returns
     pass/warn/fail with message
  2. Individual check modules per category
  3. CLI command with table output and JSON mode
  4. Exit code: 0 = all pass, 1 = warnings only, 2 = failures

## 6. Verification Strategy

- **VT**: Unit tests for each check category (pass and fail cases)
- **VT**: CLI integration test for output format and exit codes
- **Acceptance Criteria**:
  - `spec-driver doctor` runs in this workspace and reports coherent results
  - Each check category produces actionable output on failure
  - `--json` output is parseable and complete
  - Exit code reflects worst diagnostic status

## 7. Risks & Mitigations

- **Risk**: Check scope creep — too many checks dilute value –
  _Likelihood_: medium – _Impact_: low – _Mitigation_: start with the 6
  categories above, add more via backlog items
- **Risk**: Slow checks (registry loading) on large workspaces –
  _Likelihood_: low – _Impact_: medium – _Mitigation_: lazy-load
  registries only for checks that need them; `--check` flag for targeted runs
