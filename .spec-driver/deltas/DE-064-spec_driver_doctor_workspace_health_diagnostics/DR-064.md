# DR-064 – spec-driver doctor: workspace health diagnostics

## 1. Executive Summary

- **Delta**: [DE-064](./DE-064.md)
- **Status**: draft
- **Last Updated**: 2026-03-08
- **Synopsis**: Add `spec-driver workspace doctor` command with structured
  diagnostic checks across 6 categories, delegating to existing validation
  infrastructure where possible.

## 2. Problem & Constraints

- **Current Behaviour**: No single command to verify workspace health. Users
  discover problems piecemeal. `workspace validate` checks cross-references
  but not dependencies, config validity, directory structure, or lifecycle
  staleness.
- **Drivers / Inputs**: ISSUE-022, ISSUE-015
- **Constraints / Guardrails**:
  - Must not duplicate `WorkspaceValidator` logic — delegate for refs
  - Pure functions and thin CLI pattern per project conventions
  - POL-001: maximise code reuse
  - POL-002: no magic strings/numbers (staleness threshold configurable)
- **Out of Scope**: auto-fix / repair mode, performance profiling, remote
  checks, `--fix` flag

## 3. Architecture Intent

### Target Outcomes

- `spec-driver workspace doctor` produces a structured diagnostic report
- Each of 6 categories independently runnable via `--check <category>`
- Per-result: pass/warn/fail with actionable message and optional suggestion
- Summary with counts and appropriate exit code
- JSON output for automation

### Guiding Principles

- **Separation**: diagnostics package owns the check model and runner;
  individual checks call into existing domain modules (validation, registries,
  core/paths, core/npm_utils) — never re-implement what exists
- **Pure functions**: each check category is a function
  `(Workspace) -> list[DiagnosticResult]`
- **No state**: runner collects results, formatter renders them

### Check Categories

| Category     | What it checks                                                     | Severity             | Key dependencies                  |
| ------------ | ------------------------------------------------------------------ | -------------------- | --------------------------------- |
| `deps`       | Python version, ts-doc-extract, Node.js, git/jj                    | warn (optional deps) | `core/npm_utils`, `shutil.which`  |
| `config`     | workflow.toml validity, CLAUDE.md refs, skills allowlist           | fail/warn            | `core/config`, filesystem         |
| `structure`  | Expected directories exist, no orphaned bundles                    | fail/warn            | `core/paths`                      |
| `registries` | All registries load without errors, ID uniqueness                  | fail                 | `Workspace` registry accessors    |
| `refs`       | Cross-references resolve                                           | fail/warn            | `WorkspaceValidator` (delegation) |
| `lifecycle`  | Stuck in-progress deltas (>N days), draft artifacts with impl work | warn                 | `ChangeRegistry`, datetime        |

### Exit Codes

- `0` — all checks pass
- `1` — warnings present, no failures
- `2` — at least one failure

## 5. Core Data Model

```python
@dataclass(frozen=True)
class DiagnosticResult:
    """Single diagnostic finding within a category."""
    category: str           # "deps", "config", "structure", "registries", "refs", "lifecycle"
    name: str               # specific check, e.g. "ts-doc-extract", "workflow-toml"
    status: str             # "pass", "warn", "fail"
    message: str            # human-readable description
    suggestion: str | None  # actionable fix hint, or None for pass results

@dataclass(frozen=True)
class CategorySummary:
    """Aggregated results for one check category."""
    category: str
    status: str             # worst status: pass < warn < fail
    results: tuple[DiagnosticResult, ...]
```

### Severity Aggregation

A category's status is the worst status across its results:

- All pass → pass
- Any warn, no fail → warn
- Any fail → fail

Overall exit code derives from the worst `CategorySummary.status`.

### Validator Delegation Bridge

The `refs` check calls `validate_workspace(ws)` and translates:

- `ValidationIssue(level="error")` → `DiagnosticResult(status="fail")`
- `ValidationIssue(level="warning")` → `DiagnosticResult(status="warn")`
- `ValidationIssue(level="info")` → `DiagnosticResult(status="pass")`

This keeps cross-reference logic in one place.

## 6. Check Implementations

### `deps` — dependency checks

- Python version ≥ 3.11 (from pyproject.toml)
- `git` or `jj` available via `shutil.which`
- `node` available (warn if missing — optional for TS sync)
- `ts-doc-extract` available (warn if missing — optional for TS sync)
  Uses existing `core/npm_utils.is_package_installed_globally()`

### `config` — configuration validity

- `workflow.toml` exists and parses without error
- `CLAUDE.md` exists at repo root
- Skills allowlist (`.spec-driver/skills.allowlist`) exists and is non-empty
- Agent docs directory (`.spec-driver/agents/`) exists

### `structure` — directory structure

- `.spec-driver/` root exists
- Required subdirectories: `specs/`, `deltas/`, `decisions/`, `backlog/`,
  `memory/`, `registry/`, `templates/`
- No orphaned bundle directories (directory exists but no primary .md file)

### `registries` — registry integrity

- Each registry type loads without exceptions
- No duplicate IDs across loaded artifacts
- Uses `Workspace` accessors: `.specs`, `.delta_registry`, `.revision_registry`,
  `.audit_registry`, `.decisions`

### `refs` — cross-reference consistency

- Delegates entirely to `validate_workspace(ws)`
- Translates `ValidationIssue` → `DiagnosticResult`

### `lifecycle` — lifecycle hygiene

- In-progress deltas older than N days (default: 5, configurable)
- Draft artifacts that reference implementation work
- Staleness threshold: read from `workflow.toml` `[doctor]` section if present,
  else default 5

## 7. CLI Interface

```
spec-driver workspace doctor [OPTIONS]

Options:
  --check CATEGORY   Run single category only (deps|config|structure|registries|refs|lifecycle)
  --json             Machine-readable JSON output
  --verbose, -v      Include pass results in text output (normally suppressed)
  --root PATH        Workspace root override
```

### Text Output Format

```
spec-driver doctor
==================

deps
  ✓ python (3.12.1)
  ✓ git
  ⚠ node — not found (optional: needed for TypeScript sync)
  ⚠ ts-doc-extract — not found
    → Install with: npm install -g ts-doc-extract

config
  ✓ workflow.toml
  ✓ CLAUDE.md
  ✓ skills allowlist

...

Summary: 12 pass, 2 warn, 0 fail
```

Default: suppress pass results (show only warn/fail). `--verbose` shows all.

### JSON Output Shape

```json
{
  "categories": [
    {
      "category": "deps",
      "status": "warn",
      "results": [
        {
          "name": "python",
          "status": "pass",
          "message": "Python 3.12.1",
          "suggestion": null
        },
        {
          "name": "ts-doc-extract",
          "status": "warn",
          "message": "not found",
          "suggestion": "npm install -g ts-doc-extract"
        }
      ]
    }
  ],
  "summary": { "pass": 12, "warn": 2, "fail": 0 },
  "exit_code": 1
}
```

## 8. Package Structure

```
supekku/scripts/lib/diagnostics/
├── __init__.py          # exports: DiagnosticResult, CategorySummary, run_checks
├── models.py            # DiagnosticResult, CategorySummary dataclasses
├── runner.py            # run_checks(ws, categories=None) -> list[CategorySummary]
└── checks/
    ├── __init__.py      # CHECK_REGISTRY: dict[str, CheckFn]
    ├── deps.py          # check_deps(ws) -> list[DiagnosticResult]
    ├── config.py        # check_config(ws) -> list[DiagnosticResult]
    ├── structure.py     # check_structure(ws) -> list[DiagnosticResult]
    ├── registries.py    # check_registries(ws) -> list[DiagnosticResult]
    ├── refs.py          # check_refs(ws) -> list[DiagnosticResult]
    └── lifecycle.py     # check_lifecycle(ws) -> list[DiagnosticResult]

supekku/scripts/lib/formatters/
└── diagnostic_formatters.py  # format_doctor_text(), format_doctor_json()
```

Type alias: `CheckFn = Callable[[Workspace], list[DiagnosticResult]]`

`CHECK_REGISTRY` maps category name → check function. Runner iterates it,
optionally filtered by `--check`.

## 9. Design Decisions & Trade-offs

### DEC-064-01: `workspace doctor` not top-level

Doctor is a workspace health command. Grouping under `workspace` alongside
`install` and `validate` is cohesive. Avoids top-level command proliferation.

### DEC-064-02: Delegate to WorkspaceValidator for refs

Cross-reference validation already exists and is tested. Doctor's `refs` check
translates results rather than reimplementing. Satisfies POL-001.

### DEC-064-03: New `diagnostics/` package

`validation/` owns semantic cross-reference validation. `diagnostics/` owns
operational health checks (deps, config, structure, lifecycle). Different
concerns, different packages. Doctor bridges them.

### DEC-064-04: No `--fix` in v1

Diagnosis only. Repair mode is a follow-up (per DE-064 out-of-scope).

### DEC-064-05: Staleness threshold from config

Default 5 days. Overridable via `workflow.toml`:

```toml
[doctor]
staleness_days = 5
```

Avoids magic number (POL-002).
