## 1. Summary & Context
- **Product Spec(s)**: none yet — drift ledger is a new primitive
- **Technical Spec(s)**: none yet — to be created as part of this delta
- **Implementation Plan**: [IP-065](./IP-065.md) – draft
- **Change Drivers**: IMPR-007, ADR-008 (normative vs observed truth reconciliation)

## 2. Motivation
- Current workflows improvise reconciliation tracking. Audits discover drift
  but there is no first-class artifact to track resolution from discovery
  through adjudication to closure.
- ADR-008 establishes that specs own normative truth and evidence overlays
  record observed truth, but there is no structured mechanism for managing
  the gap between them.
- The DE-047 pilot demonstrated the need: 30+ drift entries accumulated during
  spec corpus reconciliation with no formal home.

## 3. Scope & Objectives
- **Primary Outcomes**:
  - Drift ledger file format: frontmatter schema + entry schema (per IMPR-007 D1–D14)
  - Entry lifecycle: open → triaged → adjudicated → resolved/deferred/dismissed
  - CLI: `create drift`, `list drift`, `show drift`
  - Schema introspection: `schema show drift.entry`
  - Storage: `drift/DL-NNN-slug.md` with ledger-local entry IDs
- **Operational Constraints**: MVP — no registry, no automated adjudication, no lifecycle enforcement
- **Dependencies**: none blocking; ADR-008 is accepted and provides the truth model

## 4. Out of Scope
- `drift.yaml` registry (deferred per D2 until post-pilot)
- Automated adjudication or mandatory enforcement in runtime gates
- `--create-ledger` flag on `create delta` / `create spec` (follow-up)
- TUI views for drift entries
- Cross-ledger queries or aggregation

## 5. Approach Overview
- **System Touchpoints**:
  - New domain package: `supekku/scripts/lib/drift/` (models, registry-stub, lifecycle)
  - Schema definitions for frontmatter + entry block
  - CLI commands: `supekku/scripts/drift.py` (thin orchestration)
  - Formatters: `supekku/scripts/lib/formatters/drift_formatters.py`
  - Storage directory: `drift/`
- **Key Changes**:
  - Define drift ledger frontmatter schema and entry block schema
  - Implement `DriftLedger` model and parser (markdown with structured entries)
  - Add CLI create/list/show commands
  - Add schema introspection support for `drift.entry`

## 6. Verification Strategy
- **Planned Artefacts**:
  - VT: unit tests for model, parser, lifecycle transitions, formatters
  - VT: CLI integration tests for create/list/show
  - VA: schema validation against IMPR-007 decisions D1–D14
- **Acceptance Criteria**:
  - `spec-driver create drift "test"` produces a valid ledger file
  - `spec-driver list drift` shows existing ledgers
  - `spec-driver show drift DL-NNN` renders ledger with entries
  - Entry schema matches IMPR-007 draft (progressive strictness per D13)
  - All tests pass, both linters clean

## 7. Risks & Mitigations
- **Risk**: Entry parsing complexity (freeform markdown sections with structured fields) – *Likelihood*: medium – *Impact*: medium – *Mitigation*: follow the parity ledger pattern; keep parser simple, accept YAML-in-markdown for entry fields
- **Risk**: Schema drift from IMPR-007 decisions during implementation – *Likelihood*: low – *Impact*: low – *Mitigation*: treat D1–D14 as binding; deviations require explicit decision update

## 8. Follow-ups & Tracking
