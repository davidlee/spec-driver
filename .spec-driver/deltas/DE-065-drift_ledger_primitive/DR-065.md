# DR-065 – Drift Ledger primitive

## 1. Executive Summary
- **Delta**: [DE-065](./DE-065.md)
- **Status**: draft
- **Last Updated**: 2026-03-08
- **Synopsis**: Introduce drift ledger as a first-class primitive — models, parser, registry, formatters, and CLI.

## 2. Problem & Constraints

- **Current Behaviour**: Drift tracking is ad-hoc. The DE-047 pilot created
  `drift/DL-047-spec-corpus-reconciliation.md` manually. No domain model,
  registry, CLI support, or schema validation exists. Agents and users cannot
  programmatically discover, filter, or query drift entries.

- **Drivers / Inputs**:
  - IMPR-007: schema design with 14 decisions (D1–D14) — treated as
    authoritative starting point, but subject to reconciliation where
    pilot usage reveals gaps (see `backlog` resolution path addition)
  - ADR-008: normative vs observed truth reconciliation model
  - DL-047 pilot (spec-driver): 21 entries across a spec corpus survey —
    validates the format and workflow under real use
  - deck_of_dwarf parity ledger: independent validation at a different scale
    (large design implementation tracking) — the pattern IMPR-007 D1's
    ledger-as-file model was lifted from

- **Constraints / Guardrails**:
  - Follow existing domain package patterns (backlog, decisions, changes)
  - DE-049: all managed content under `.spec-driver/`
  - No registry YAML file in this delta (deferred per D2)
  - No lifecycle enforcement in this delta (advisory only)
  - Entry schema follows IMPR-007 D1–D14 as the authoritative starting
    point; deviations are documented in this DR and require reconciliation
    back to IMPR-007

- **Out of Scope**: registry sync, automated adjudication, `--create-ledger`
  flag, TUI views, cross-ledger aggregation, schema introspection
  (`schema show drift.entry` deferred until format is proven post-migration)

## 3. Architecture Intent

- **Target Outcomes**:
  - Drift ledgers are discoverable via `DriftLedgerRegistry`
  - Entries within a ledger are parsed into structured `DriftEntry` models
  - CLI provides `create drift`, `list drift`, `show drift`
  - Formatters render ledger lists and entry details

- **Guiding Principles**:
  - Follow the backlog package as the structural template for registry and
    discovery patterns; the model is materially more complex (nested typed
    substructures) due to the domain
  - Entry data lives in fenced YAML blocks — parsed by `yaml.safe_load()`,
    not custom regex
  - Progressive strictness: the parser accepts incomplete entries (per D13)
  - Pure formatter functions, thin CLI, domain logic in the package

- **Fractal design intent**: The same format must work for a 3-entry ledger
  attached to a single spec revision and a 50-entry ledger tracking a
  cross-cutting program. The heading-per-entry structure scales for human
  scanning; the YAML block per entry scales for programmatic querying. The
  stress point at large scale is discoverability — addressed by registry
  filtering, not format changes.

- **State Transitions / Lifecycle Impact**:
  - Ledger statuses: `open`, `closed` (advisory, not enforced)
  - Entry statuses: `open` → `triaged` → `adjudicated` → `resolved` |
    `deferred` | `dismissed` | `superseded`
  - This delta does NOT enforce transition rules — status is a string field
    validated against an accepted set, with warnings for unknown values
    (matching the backlog permissive validation pattern per DEC-057-08)

## 5. Design Decisions

### DEC-065-01: Storage under `.spec-driver/drift/` — durable records

The pilot placed ledgers at `drift/` (repo top-level). All other managed
content lives under `.spec-driver/` per DE-049 / DEC-049-02.

**Decision**: Drift ledgers live at `.spec-driver/drift/DL-NNN-slug.md`.

**Rationale**: Resolved drift entries have durable value as precedent for
future work. A ledger created for DE-047 is still useful when DE-072 touches
the same specs — it records what was found, how it was assessed, and how it
was resolved. This makes ledgers cross-work reference material, not
initiative-local working documents.

If ledgers were initiative-local, they would belong inside the delta bundle
(like `notes.md`). But burying them in a completed delta's directory makes
them undiscoverable for the use case that matters most: "has this drift been
seen before?"

**Alternative considered**: Store inside delta bundles (`DE-047/DL-047.md`).
Rejected because it conflates the ledger's lifecycle with the delta's lifecycle
and makes cross-delta discovery require scanning all delta directories.

### DEC-065-02: Entry data in fenced YAML blocks

**Revised** — supersedes original design (pilot YAML-in-markdown list items).

The pilot format used `- key: value` markdown list items for entry data. This
is pleasant to write and scan but ambiguous to parse: multi-line analysis
fields collide with list syntax, nested structures require custom indentation
rules, and unknown fields are indistinguishable from malformed known fields.

**Decision**: Entry structured data lives in a fenced YAML code block.
Freeform content (analysis, narrative) lives outside the fence as markdown.

**Target format** (indented to avoid nested fence ambiguity):

    ### DL-047.001: Phase-01 auto-creation — PROD-002 vs PROD-011

    ```yaml
    status: resolved
    entry_type: contradiction
    severity: blocking
    topic: workflow
    owner: david
    sources:
      - kind: prod
        ref: PROD-002
        note: "FR-002: default behavior creates complete delta bundle"
    claims:
      - kind: assertion
        label: A
        text: "Default behavior creates plan + first phase alongside delta"
    assessment: confirmed
    resolution_path: RE
    affected_artifacts:
      - PROD-002
    evidence:
      - 2026-03-05 discovered during PROD spec cross-reference survey
    ```

    PROD-011 was written specifically to reverse PROD-002's phase-01
    auto-creation. PROD-002 was never revised. An agent reading PROD-002
    in isolation will implement the wrong behavior.

**Rationale**:
- `yaml.safe_load()` handles all parsing — nesting, quoting, multiline
  strings, edge cases. No custom parser for data fields.
- Fenced blocks render reliably in TUI viewers, GitHub, and editors
- Freeform analysis/narrative lives outside the fence as natural markdown —
  no collision between prose and data syntax
- LLMs write either format equally well; the difference is entirely in
  parse reliability and validation
- Schema validation of the YAML dict is a solved problem

**Trade-off**: Slightly more ceremony per entry (fence markers). Acceptable
given that entries are authored by LLMs in-flow, not typed by hand under
time pressure.

### DEC-065-03: Heading-based section split with YAML block extraction

**Revised** — supersedes original regex-based per-field extraction.

**Decision**: Parse entries by:
1. Separate freeform body content (between frontmatter and first `### `
   heading) from entry sections
2. Split remaining body on `### ` headings (entry boundaries)
3. Extract entry ID and title from heading text (`### DL-NNN.MMM: title`)
4. Within each section, extract the fenced YAML block and parse via
   `yaml.safe_load()`
5. Capture remaining markdown outside the fence as `analysis` (freeform)

**Design detail**:
- `parse_ledger_body(body: str) -> tuple[str, list[DriftEntry]]` — returns
  (freeform_body, entries)
- Returns partially-populated entries for incomplete YAML (progressive
  strictness per D13)
- Unknown YAML keys are preserved in `extra: dict`
- Parse warnings (not errors) for: missing YAML block, malformed YAML,
  unrecognised entry ID format

**Parser contract — edge case behaviour**:

| Case | Behaviour |
| --- | --- |
| Malformed YAML block (syntax error) | Warning; entry created with `extra: {"_parse_error": msg}`, no structured fields |
| Valid YAML, missing required nested keys (e.g. `Source` without `kind`) | Warning; malformed record skipped, others preserved |
| Duplicate entry IDs in one ledger | Warning on second occurrence; both entries preserved, consumer decides |
| Entry section with no YAML block | Warning; entry created with heading-derived `id` and `title` only |
| Multiple fenced YAML blocks in one entry section | First block parsed; subsequent blocks treated as part of analysis text |

These cases must be pinned by tests in VT-065-parser.

**Fence/heading precedence rule**: The parser processes fences before
headings. Content inside a fenced code block (any language, including
` ```yaml ` and ` ``` `) is opaque — `### ` lines inside a fence are not
treated as entry boundaries. This means: find fence boundaries first, then
split on `### ` headings only in unfenced regions. This case must be covered
by VT-065-parser.

**Rationale**: Heading boundaries are unambiguous in unfenced content. YAML
parsing is delegated to a battle-tested library. The only custom parsing is
fence detection, section splitting, and fence extraction.

### DEC-065-04: Registry is read-only discovery

**Decision**: `DriftLedgerRegistry` discovers and loads ledger files. It does
NOT write a `drift.yaml` registry file.

**Interface**:
```python
class DriftLedgerRegistry:
    def __init__(self, root: Path | None = None): ...
    def collect(self) -> dict[str, DriftLedger]: ...
    def find(self, ledger_id: str) -> DriftLedger | None: ...
    def iter(self, *, status: str | None = None) -> Iterator[DriftLedger]: ...
```

**Scope**: This interface operates at the ledger level. Entry-level querying
(find entry across ledgers, filter entries by status/topic) is explicitly
deferred. For this delta, entry access is through the parent ledger.

**Rationale**: D2 explicitly defers the registry YAML until post-pilot. The
discovery layer is still needed for `list` and `show` commands. Adding sync
later is additive — the interface doesn't change.

### DEC-065-05: Flat directory (durable constraint)

**Decision**: `.spec-driver/drift/DL-NNN-slug.md` — flat directory, no
subdirectories per ledger. This is a durable layout, not a temporary
simplification.

**Rationale**: Ledgers are single files (D1). There are no companion artifacts
(unlike deltas which have DR/IP/phases). A subdirectory-per-ledger structure
would add directory overhead for a single file. If a future need arises for
companion artifacts, that is a design decision for a future delta — not
something to pre-accommodate now.

### DEC-065-06: Typed substructures for nested fields

**Decision**: `Source`, `Claim`, and `DiscoveredBy` are typed dataclasses,
not `dict`.

```python
@dataclass(frozen=True)
class Source:
    kind: str       # spec, prod, adr, policy, doc, impl, contract, ...
    ref: str        # artifact ID or path
    note: str = ""  # optional context

@dataclass(frozen=True)
class Claim:
    kind: str       # assertion, observation, gap, question
    text: str
    label: str = "" # optional: expected, observed, A, B, freeform

@dataclass(frozen=True)
class DiscoveredBy:
    kind: str       # audit, survey, agent, human
    ref: str = ""   # optional: VA-047-001, etc.
```

**Rationale**: `list[dict]` with implicit shape contracts means every consumer
does defensive key checking. Typed dataclasses make the shape explicit, enable
IDE support, and catch invalid structures at parse time rather than downstream.

**Validation**: The parser constructs these from YAML dicts. Missing required
fields (`kind` for Source, `kind` + `text` for Claim) produce parse warnings
and skip the malformed record. This is the progressive strictness contract:
the *model* is strict, the *parser* is lenient.

### DEC-065-07: Entry IDs — sequential, never reused, author-allocated

**Decision**: Entry IDs are sequential within the ledger (`DL-047.001`,
`DL-047.002`, ...). IDs are never reused after deletion. The author (human or
LLM) is responsible for allocating the next sequential ID. Duplicate IDs
within a ledger are invalid state — the parser emits a warning and preserves
both entries (see parser contract table in DEC-065-03), but CLI display
should flag duplicates visibly.

**Rationale**: Entries are append-only working records. Reusing IDs after
deletion would break external references. Auto-allocation would require
parsing the ledger to find the max ID — acceptable for a CLI command but
unnecessary overhead for in-flow authoring where the author can see the
existing entries.

**Entry-level find**: Not supported in this delta. Entry access is through
the parent ledger (`registry.find("DL-047").entries`). Entry-level querying
across ledgers is a future concern.

### DEC-065-08: Freeform body content preserved

**Decision**: Content between the frontmatter and the first `### ` entry
heading is preserved as `body: str` on the `DriftLedger` model. This
includes any narrative, tables, or context the author placed there.

**Rationale**: The pilot contains a substantive corpus coverage table before
the entries section. This is useful context. The parser must not discard it.
`show drift` renders the body content before entry listings.

## 7. Model Design

### DriftEntry

```python
@dataclass
class DriftEntry:
    id: str                              # "DL-047.001"
    title: str                           # from heading after ID
    status: str                          # open, triaged, adjudicated, ...
    entry_type: str                      # contradiction, implementation_drift, ...
    severity: str = ""                   # blocking, significant, cosmetic
    topic: str = ""                      # lifecycle, taxonomy, contracts, ...
    owner: str = ""
    sources: list[Source] = field(...)
    claims: list[Claim] = field(...)
    assessment: str = ""                 # confirmed, disputed, not_drift, deferred
    resolution_path: str = ""            # ADR, RE, DE, editorial, no_change, backlog
    resolution_ref: str = ""             # ADR-008, DE-050, etc.
    affected_artifacts: list[str] = field(...)
    discovered_by: DiscoveredBy | None = None
    analysis: str = ""                   # freeform markdown outside YAML fence
    evidence: list[str] = field(...)     # append-only timeline entries
    extra: dict = field(...)             # unknown YAML keys preserved
```

Note: `resolution_path` includes `backlog` — present in the pilot
(`DL-047-spec-corpus-reconciliation.md`) but missing from the IMPR-007 schema
draft. Added to match observed usage. **IMPR-007 reconciliation required**:
the schema draft's `RESOLUTION_PATHS` section should be updated to include
`backlog` with rationale (follow-up tracked as backlog item, not an immediate
fix). This is evidence that IMPR-007 is not yet fully stable as a binding
source — the implementation may surface further divergences.

### DriftLedger

```python
@dataclass
class DriftLedger:
    id: str                              # "DL-047"
    name: str
    status: str                          # open, closed
    path: Path
    created: str = ""
    updated: str = ""
    delta_ref: str = ""                  # owning delta, if any
    body: str = ""                       # freeform content before entries
    frontmatter: dict = field(...)       # raw frontmatter
    entries: list[DriftEntry] = field(...)
```

### Lifecycle constants

```python
LEDGER_STATUSES = frozenset({"open", "closed"})

ENTRY_STATUSES = frozenset({
    "open", "triaged", "adjudicated",
    "resolved", "deferred", "dismissed", "superseded",
})

ENTRY_TYPES = frozenset({
    "contradiction", "implementation_drift", "stale_claim",
    "missing_decision", "ambiguous_intent",
})

SEVERITIES = frozenset({"blocking", "significant", "cosmetic"})

ASSESSMENTS = frozenset({"confirmed", "disputed", "not_drift", "deferred"})

RESOLUTION_PATHS = frozenset({
    "ADR", "RE", "DE", "editorial", "no_change", "backlog",
})
```

Lifecycle constants are defined in models (co-located with the types they
validate), not in a separate `lifecycle.py`. This follows the backlog pattern
where status sets live in `models.py`. A separate lifecycle module is not
justified until transition enforcement is added.

## 10. Authoring & Creation

**`create drift` output**: A ledger file with frontmatter and an `## Entries`
heading. No template entry — entries are added by the author (human or LLM)
by appending `### ` sections with fenced YAML blocks.

**Entry authoring path**: Manual markdown editing (or LLM-assisted). There is
no `create drift-entry` command in this delta. Entries are authored in-flow
during focused work (audits, surveys, implementation). A CLI entry-creation
command is a potential follow-up if the manual path proves insufficient.

**Creation template**:
```markdown
---
id: DL-NNN
name: <ledger name>
created: 'YYYY-MM-DD'
updated: 'YYYY-MM-DD'
status: open
kind: drift_ledger
delta_ref: <DE-NNN or empty>
---

# DL-NNN — <ledger name>

<optional freeform body: context, scope, coverage notes>

## Entries

<!-- append entries as ### sections with fenced YAML blocks -->
```
