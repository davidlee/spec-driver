## 2. Motivation

Spec-driver artifacts have no mechanism to reference external systems (JIRA,
Linear, GitHub Issues). Users managing projects across tools cannot track
correspondence between spec-driver entities and external tickets/discussions.
Adding optional `ext_id` and `ext_url` fields to artifact frontmatter closes
this gap with minimal schema surface.

## 3. Scope & Objectives

- **Primary Outcomes**:
  1. Frontmatter schemas accept optional `ext_id` (string) and `ext_url` (string)
     fields across all artifact types (specs, requirements, backlog items,
     policies, standards, deltas)
  2. Domain models expose `ext_id` and `ext_url` attributes
  3. `list` commands support `--external` / `-e` flag to show ext_id column
  4. `show` commands display external references when present
  5. JSON output includes ext_id/ext_url fields
- **Operational Constraints**: Purely additive â€” no migration needed, fields are
  optional with no default
- **Dependencies**: None

## 4. Out of Scope

- Validation of ext_url format (URL parsing/verification)
- Bidirectional sync with external systems
- ext_id uniqueness enforcement
- Registry-level ext_id indexing (can be a follow-up if needed)
