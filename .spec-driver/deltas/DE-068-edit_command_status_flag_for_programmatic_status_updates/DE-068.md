## 1. Summary & Context

- **Product Spec**: PROD-013 – CLI Artifact File Access (FR-004: edit command)
- **Implementation Plan**: [IP-068](./IP-068.md)
- **Change Drivers**: ISSUE-014 (need registry commands for value changes without manual edits), ISSUE-009 (status fields lack enums)

## 2. Motivation

Currently `spec-driver edit <entity> <id>` only opens `$EDITOR`. To change a single field like `status`, users must open the file, find the frontmatter field, edit it, save, and close. Agents face the same friction — they must read/parse/edit files directly.

A `--status <value>` flag on edit subcommands would enable quick, validated status transitions from the CLI without opening an editor. This is the most common single-field edit operation across all entity types.

## 3. Scope & Objectives

- **Primary Outcomes**:
  - `spec-driver edit <entity> <id> --status <value>` updates frontmatter status in-place (no editor launch)
  - Status values validated against existing domain-level enums: changes (`lifecycle.py`), requirements (`lifecycle.py`), backlog items (`models.py`), verification (`verification.py`), drift ledgers (`drift/models.py`). Permissive with warning only for specs, policies, standards, and ADRs (which lack enums — tracked by ISSUE-009)
  - Works for all entity types that have a `status` frontmatter field
  - Adds new `edit drift` subcommand (ledger-level) — drift ledgers currently lack an edit command entirely
- **Dependencies**: None — builds on existing frontmatter editing patterns (`changes/updater.py`, block infrastructure)

## 4. Out of Scope

- Generic `--field key=value` editing (future delta if warranted)
- Adding status enums to entity types that still lack them — specs, policies, standards, ADRs (ISSUE-009 scope)
- Lifecycle transition validation (e.g. "draft can only go to active") — future concern

## 5. Approach Overview

- **System Touchpoints**: `supekku/cli/edit.py`, `supekku/scripts/lib/core/frontmatter_schema.py`, entity registries
- **Key Changes**:
  - Add `--status` option to each edit subcommand in `edit.py`
  - Add new `edit drift` subcommand for drift ledgers (currently missing from edit surface)
  - When `--status` is provided: resolve artifact path, update frontmatter `status` field in-place, skip editor launch
  - Validate against domain-level enums where available (changes, requirements, backlog, verification, drift ledgers via `ENUM_REGISTRY` / `core/enums.py`); warn-and-proceed for entity types without enums (specs, policies, standards, ADRs)
  - Reuse existing frontmatter read/write infrastructure

## 6. Verification Strategy

- **VT**: Unit tests for status update on representative entity types (delta, spec, issue)
- **VT**: Validation rejection test for invalid backlog status values
- **VA**: CLI integration smoke test across all entity types
- **Acceptance Criteria**: `spec-driver edit delta DE-068 --status in-progress` updates frontmatter without opening editor

## 7. Risks & Mitigations

- **Risk**: Frontmatter corruption on write – _Likelihood_: low – _Impact_: high – _Mitigation_: Use existing safe YAML write patterns; round-trip test
- **Risk**: Inconsistent status values across entity types – _Likelihood_: medium – _Impact_: low – _Mitigation_: Validate where enums exist, warn elsewhere; ISSUE-009 tracks full enum coverage

## 8. Follow-ups & Tracking

- **Backlog Items**: ISSUE-009 (status enums for non-backlog entities), ISSUE-014 (broader registry edit commands)
- **Future**: Generic `--field` editing if `--status` pattern proves useful
