# DR-068 – edit command --status flag for programmatic status updates

## 1. Executive Summary

- **Delta**: [DE-068](./DE-068.md)
- **Status**: draft
- **Last Updated**: 2026-03-08
- **Synopsis**: Add `--status` flag to `edit` subcommands for non-interactive frontmatter status updates with per-kind validation.

## 2. Problem & Constraints

- **Current Behaviour**: `spec-driver edit <entity> <id>` always opens `$EDITOR`. Changing a single frontmatter field (status) requires opening the file, locating the field, editing, saving, closing. Agents must parse/edit YAML directly.
- **Drivers**: ISSUE-014 (CLI commands for value changes), ISSUE-009 (status enum coverage)
- **Constraints**:
  - Must not corrupt frontmatter — line-level replacement only, no full YAML round-trip
  - Must validate where enums exist, reject invalid values (strict)
  - Must reject empty/whitespace-only status values
  - Must update `updated:` date alongside `status:`
- **Out of Scope**: Generic `--field` editing, adding new status enums, lifecycle transition guards

## 3. Architecture Intent

- **Target Outcomes**:
  - `spec-driver edit <entity> <id> --status <value>` updates frontmatter in-place, prints confirmation, exits
  - Status validated per entity kind using existing enum infrastructure
  - Existing callers (`complete delta`, `complete revision`) consolidated onto shared primitive

- **Guiding Principles**:
  - Reuse over re-implementation — extract, don't duplicate
  - Skinny CLI — edit.py dispatches to shared primitive, no business logic inline
  - Validation is per-kind, not one-size-fits-all

## 5. Design Detail

### 5.1 Shared primitive: `update_frontmatter_status()`

Extract from the existing `_update_artifact_frontmatter_status()` pattern in `completion.py`:

```python
# supekku/scripts/lib/core/frontmatter_writer.py

def update_frontmatter_status(path: Path, status: str) -> bool:
    """Update status and updated date in artifact frontmatter (line-level).

    Returns True if status was found and updated, False if no status field found.
    Raises FileNotFoundError if path doesn't exist.
    """
```

Behaviour:

- Rejects empty/whitespace-only status values (raises `ValueError`)
- Iterates lines, replaces `status: <old>` → `status: <new>` and `updated: <old>` → `updated: '<today>'`
- Writes status values unquoted (`status: draft`, not `status: 'draft'`) — matches 100% of existing convention across `.spec-driver/`
- Preserves all other content byte-for-byte (no YAML parse/dump)
- Returns False if no `status:` line found in frontmatter

This is deliberately simple — no validation, no enum awareness. Validation belongs in the caller (CLI layer or domain logic).

### 5.2 Validation dispatch

The CLI layer validates before calling the primitive. Dispatch logic:

```python
# In edit.py or a thin helper

def validate_status_for_entity(entity_type: str, status: str) -> None:
    """Validate status value for an entity type.

    Raises ValueError if:
    - status is empty/whitespace-only
    - enum exists for entity_type and status is not in it

    Accepts silently if no enum is registered for the entity type.
    """
    if not status or not status.strip():
        raise ValueError("Status must not be empty")

    enum_path = f"{entity_type}.status"
    valid_values = get_enum_values(enum_path)

    if valid_values is None:
        return  # No enum — accept silently

    if status in valid_values:
        return  # Valid

    raise ValueError(
        f"Invalid status '{status}' for {entity_type}. "
        f"Valid values: {', '.join(valid_values)}"
    )
```

Entity-to-enum mapping requires adding backlog entries to `ENUM_REGISTRY`:

```python
# enums.py additions
from supekku.scripts.lib.backlog.models import (
    ISSUE_STATUSES, PROBLEM_STATUSES, IMPROVEMENT_STATUSES, RISK_STATUSES,
)

from supekku.scripts.lib.drift.models import LEDGER_STATUSES

ENUM_REGISTRY |= {
    "issue.status": lambda: sorted(ISSUE_STATUSES),
    "problem.status": lambda: sorted(PROBLEM_STATUSES),
    "improvement.status": lambda: sorted(IMPROVEMENT_STATUSES),
    "risk.status": lambda: sorted(RISK_STATUSES),
    "drift.status": lambda: sorted(LEDGER_STATUSES),
    # Aliases — revision/audit share change lifecycle
    "revision.status": lambda: sorted(s for s in CHANGE_STATUSES if s != "complete"),
    "audit.status": lambda: sorted(s for s in CHANGE_STATUSES if s != "complete"),
}
```

**Coverage by entity type after this change:**

| Entity type                          | Enum source                                     | Enforcement             |
| ------------------------------------ | ----------------------------------------------- | ----------------------- |
| delta                                | `CHANGE_STATUSES` via `delta.status`            | Strict (reject)         |
| revision                             | `CHANGE_STATUSES` via `delta.status`            | Strict — same lifecycle |
| audit                                | `CHANGE_STATUSES` via `delta.status`            | Strict — same lifecycle |
| requirement                          | `REQUIREMENT_STATUSES` via `requirement.status` | Strict                  |
| issue                                | `ISSUE_STATUSES` via `issue.status`             | Strict                  |
| problem                              | `PROBLEM_STATUSES` via `problem.status`         | Strict                  |
| improvement                          | `IMPROVEMENT_STATUSES` via `improvement.status` | Strict                  |
| risk                                 | `RISK_STATUSES` via `risk.status`               | Strict                  |
| drift                                | `LEDGER_STATUSES` via `drift.status`            | Strict                  |
| spec, policy, standard, memory, card | None                                            | Accept (no enum)        |

Notes:

- Revision/audit share change lifecycle statuses — handled via alias entries in `ENUM_REGISTRY`.
- Drift ledger resolver exists as `"drift_ledger"` in `_ARTIFACT_RESOLVERS` (`common.py:497`). The `edit drift` subcommand must use this key, not `"drift"`.
- The `edit drift` subcommand uses `"drift"` as the entity type for enum lookup (`drift.status`), but `"drift_ledger"` for artifact resolution. The mapping is confined to the CLI layer.

### 5.3 CLI edit subcommand pattern

Each edit subcommand gains an optional `--status` parameter:

```python
@app.command("delta")
def edit_delta(
    delta_id: Annotated[str, typer.Argument(help="Delta ID")],
    status: Annotated[str | None, typer.Option("--status", help="Set status")] = None,
    root: RootOption = None,
):
    # ... resolve artifact (existing logic) ...

    if status:
        validate_status_for_entity("delta", status)  # raises ValueError on invalid
        if update_frontmatter_status(artifact.path, status):
            typer.echo(f"Updated {delta_id} status to '{status}'")
        else:
            typer.echo(f"Error: no status field found in {artifact.path}", err=True)
            raise typer.Exit(EXIT_FAILURE)
        return

    open_in_editor(artifact.path)
```

DEC-068-02: `--status` and editor are mutually exclusive. When `--status` is provided, the command updates and exits — no editor launch. This keeps the operation scriptable and predictable.

### 5.4 Consolidation of existing callers

After extracting `update_frontmatter_status()`:

- `completion.py:_update_artifact_frontmatter_status()` → thin wrapper or direct call to `update_frontmatter_status()`
- `complete_delta.py:update_delta_frontmatter()` → `update_frontmatter_status(path, "completed")`

Both existing callers already do exactly what the new primitive does. The consolidation removes ~40 lines of near-duplicate code.

## 7. Design Decisions & Trade-offs

- **DEC-068-01**: Extract shared `update_frontmatter_status()` to `core/frontmatter_writer.py` rather than leaving duplicates in completion.py / complete*delta.py. \_Rationale*: POL-001 (maximise reuse), two existing callers + new edit caller = three consumers. _Consequence_: consolidation touchpoint in completion code.

- **DEC-068-02**: `--status` is mutually exclusive with editor launch. _Rationale_: predictable scripting — `--status` is a non-interactive operation that should succeed or fail cleanly. Opening the editor after a status update would be confusing. _Consequence_: if users want to set status AND edit content, they run two commands.

- **DEC-068-03**: Validation uses `ENUM_REGISTRY` with strict rejection where enums exist, silent acceptance where they don't. _Rationale_: a direct edit tool should refuse known-invalid states — unlike bulk import/sync (where DE-057's permissive posture made sense), a user explicitly setting `--status` has clear intent and should get immediate feedback on invalid values. Entity types without enums (specs, policies, standards) accept any value — adding enums is ISSUE-009 scope. _Consequence_: adding enums to ISSUE-009 entity types automatically enables validation here with no code change.

## 8. Risks & Mitigations

- **Risk**: Line-level replacement corrupts frontmatter with unusual formatting. _Likelihood_: low. _Impact_: high. _Mitigation_: Verified that 100% of status fields in `.spec-driver/` use unquoted format (`status: value`). The existing `_update_artifact_frontmatter_status()` has been running in production via `complete delta` without issues. The `startswith("status:")` pattern is battle-tested. Round-trip tests will cover edge cases.

- **Risk**: `ENUM_REGISTRY` import chain pulls in heavy modules at CLI startup. _Likelihood_: medium. _Impact_: low. _Mitigation_: Backlog model imports are lightweight (frozensets). Validation helper can lazy-import if needed.

## 9. Follow-ups

- ISSUE-009: Add status enums for specs, policies, standards, ADRs — would automatically enable validation in this command
- ISSUE-014: This establishes the pattern for broader `--field` editing if warranted
- Consolidation: consider whether `update_frontmatter_status` should grow into a more general `update_frontmatter_field()` — defer until second use case emerges
