# DR-069 – Complete formatter migration to format_list_table generic helper

## 1. Executive Summary
- **Delta**: [DE-069](./DE-069.md)
- **Status**: draft
- **Last Updated**: 2026-03-08
- **Synopsis**: Migrate the 4 remaining formatters to `format_list_table`, extract shared cell helpers for CLI + TUI reuse, and eliminate the `no_truncate` naming inconsistency.

## 2. Problem & Constraints

**Current behaviour**: 4 of 10 list-table formatters (spec, change, backlog, requirement) still use the old inline pattern — manual `create_table()`, `calculate_column_widths()`, per-row `add_row_with_truncation()` loops. This duplicates ~60 lines of boilerplate per formatter. Additionally:

- Tag cell formatting (`", ".join(tags)` + `[#d79921]` markup) is repeated 8x across all formatters
- Date cell formatting (`.strftime("%Y-%m-%d")` + fallback) is repeated 10x
- Three already-migrated formatters (decision, policy, standard) share byte-identical `_calculate_column_widths` implementations
- `spec_formatters` and `change_formatters` use `no_truncate` (inverted boolean) while everything else uses `truncate`
- `add_row_with_truncation` in `table_utils.py` also uses `no_truncate`

**Drivers**: IMPR-006; POL-001 (maximise code reuse, minimise sprawl)

**Constraints**: Pure refactoring — no functional changes. All existing tests must continue passing. Cell helpers should be Rich-compatible pure functions to support reuse across CLI tables and future TUI views.

**Out of scope**: Changing `format_list_table` signature. TUI integration itself. Formatter tests beyond what currently exists (though new helpers get their own tests).

## 3. Architecture Intent

**Target outcomes**:
1. All 10 list-table formatters use `format_list_table` — one pattern, one dispatch path
2. Cell-level formatting is reusable across CLI tables and future TUI views
3. `no_truncate` is eliminated — `truncate` is the sole parameter name throughout
4. Identical column width calculators are consolidated

**Guiding principles**:
- Cell helpers are pure functions: `(value, options) → str`. No Rich Table dependency, no side effects. They return Rich markup strings — deliberately Rich-coupled, reusable across any Rich-compatible rendering context (CLI table, Textual widget, panel).
- `format_list_table` signature is stable. Dynamic column needs (spec_formatters' `show_external`/`include_packages`) are handled by the caller constructing the column list and closures, not by adding parameters to the generic helper.
- Migration is mechanical: each formatter follows the same extract-helpers-then-rewire pattern already proven by 6 predecessors.

## 4. Code Impact Summary

### 4.1 New module: `formatters/cell_helpers.py`

Rich-compatible pure functions for reusable cell formatting:

```python
def format_tags_cell(tags: Sequence[str], style: str = "#d79921") -> str:
  """Format tags for display with Rich markup. Returns empty string for empty/None tags."""

def format_date_cell(
  date: date | datetime | None,
  missing: str = "—",
  fmt: str = "%Y-%m-%d",
) -> str:
  """Format a date for display. Returns `missing` sentinel when None."""
```

Design notes:
- `format_tags_cell` takes an explicit `style` parameter (defaulting to current gold) so TUI themes can override without forking
- `format_date_cell` has configurable `missing` sentinel — table rows use `"—"`, TSV uses `"N/A"`, JSON uses `None` (handled separately)
- Both are pure: same input → same output, no imports beyond stdlib + typing
- Both return Rich markup strings — reusable in any Rich-compatible context (CLI tables, Textual widgets). If a future rendering context needs structured data instead of pre-formatted strings, that's a separate concern; these helpers still eliminate 8–10x duplication regardless

### 4.2 Shared column widths: `table_utils.py`

Extract the identical 5-column width calculator shared by decision, policy, and standard formatters:

```python
def governance_5col_widths(terminal_width: int) -> dict[int, int]:
  """Column widths for the common ID/Title/Tags/Status/Updated layout."""
```

This becomes the default `column_widths` callback for any formatter using that column shape. Individual formatters can still supply their own when the layout differs (e.g. backlog has 6 columns, memory has 2).

### 4.3 Eliminate `no_truncate`

| Location | Change |
|---|---|
| `table_utils.py:add_row_with_truncation` | **Remove** `no_truncate` param entirely (public API surface reduction — see note) |
| `table_utils_test.py` | Update 2 call sites that pass `no_truncate=True` → pass `max_widths=None` instead |
| `spec_formatters.py:format_spec_list_table` | Rename param `no_truncate` → `truncate` (invert semantics) |
| `change_formatters.py:format_change_list_table` | Rename param `no_truncate` → `truncate` (invert semantics) |
| `cli/list.py` | Remove `not` inversion at 3 call sites (lines 342, 498, 706) — pass `truncate=truncate` directly |

**API surface reduction note**: `add_row_with_truncation` is a public function (no underscore prefix). Its `no_truncate` param is redundant — when `max_widths is None`, truncation is already skipped. After migration, `format_list_table` is the sole production caller and never passes `no_truncate`. Removing the param is safe; test call sites are the only other consumers and are updated in the same change.

### 4.4 Formatter migrations

Each follows the proven pattern:

1. Extract `_prepare_{artifact}_row(item) → list[str]` using `format_tags_cell`, `format_date_cell`
2. Extract `_prepare_{artifact}_tsv_row(item) → list[str]` using `format_date_cell(missing="N/A")`
3. Extract or reuse `_calculate_column_widths` (or reference `governance_5col_widths`)
4. Replace inline body with `return format_list_table(...)`

**spec_formatters** — special handling:
- `show_external` and `include_packages` are dynamic column flags
- Build column list conditionally before the `format_list_table` call; `show_external` uses existing `EXT_ID_COLUMN`, `include_packages` uses new `PACKAGES_COLUMN` (added to `column_defs.py` — currently a raw string `"Packages"`)
- `_prepare_spec_row` and `_prepare_spec_tsv_row` are closures (or partials) capturing those flags
- No change to `format_list_table` signature

**change_formatters** — straightforward migration, `format_plan_list_table` is already migrated in the same file.

**backlog_formatters** — has a custom 6-column width calculator; extract as local `_calculate_column_widths`, don't force into `governance_5col_widths`.

**requirement_formatters** — similar to backlog; extract row helpers and local width calculator.

### 4.5 Already-migrated formatters: adopt helpers

Decision, policy, standard, memory formatters replace inline tag/date formatting with `format_tags_cell`/`format_date_cell` calls. Decision, policy, standard replace their local `_calculate_column_widths` with `governance_5col_widths`.

## 6. Design Decisions

**DEC-069-01**: Cell helpers in dedicated `cell_helpers.py` module.
- *Rationale*: Separates cell content formatting (tags, dates) from table structure (widths, truncation, dispatch). Both CLI tables and future TUI views need cell formatting; neither should depend on the other's infrastructure.

**DEC-069-02**: Dynamic columns via closure, not `format_list_table` signature change.
- *Rationale*: `format_list_table` is stable and generic. Spec formatter's `show_external`/`include_packages` are caller concerns — the caller builds the column list and captures flags in `prepare_row`/`prepare_tsv_row` closures. This keeps the generic helper simple and avoids feature-specific parameters leaking into shared infrastructure.

**DEC-069-03**: Eliminate `no_truncate`; remove redundant param from `add_row_with_truncation` (public API reduction).
- *Rationale*: Inverted booleans are a readability footgun. The `no_truncate` param on `add_row_with_truncation` is redundant — `max_widths=None` already skips truncation. This is a deliberate public API surface reduction: the param is removed, not renamed. After migration `format_list_table` is the sole production caller. Test call sites are updated in the same change.

**DEC-069-04**: Consolidate identical column width calculators to `governance_5col_widths` in `table_utils.py`.
- *Rationale*: Three byte-identical implementations violate POL-001. The 5-column ID/Title/Tags/Status/Updated layout is the most common shape; giving it a name promotes discovery and reuse.

**DEC-069-05**: Design cell helpers for TUI reuse (Rich-compatible, not presentation-agnostic).
- *Rationale*: TUI artifact browser (DE-066) will need the same cell-level formatting. Pure functions with configurable style/sentinel params produce Rich markup strings — reusable in any Rich-compatible context (CLI tables, Textual widgets). This is deliberately Rich-coupled, not framework-neutral: if a future rendering context needs structured data instead of pre-formatted strings, that's a separate concern with a clear solution path. The immediate value is eliminating 8–10x duplication.

## 7. Rollout & Operational Notes

Pure internal refactoring. No CLI interface changes (except `no_truncate` → `truncate` at the formatter API level, which is internal — CLI callers use `--truncate` flag unchanged). No migration, no rollback concerns.
