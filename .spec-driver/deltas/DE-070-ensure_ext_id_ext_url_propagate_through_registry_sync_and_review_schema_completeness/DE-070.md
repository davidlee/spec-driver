## 1. Summary & Context

- **Technical Spec(s)**: SPEC-116 – `supekku/scripts/lib/core/frontmatter_metadata`
- **Implementation Plan**: [IP-070](./IP-070.md)
- **Change Drivers**: DE-067 follow-up — ISSUE-024 acceptance criteria left
  unchecked (registry YAML capture, documentation/schema)

## 2. Motivation

DE-067 added `ext_id` and `ext_url` to all domain models, formatters, and CLI.
Two gaps remain:

1. **Registry sync**: derived registry YAML files (specs, requirements, backlog,
   policies, standards) do not yet include `ext_id`/`ext_url` when present in
   frontmatter. This means `sync` output loses external references.

2. **Schema completeness**: `schema show` / validation schemas may not reflect
   the new fields. A broader review of schema completeness is warranted — not
   just for ext_id/ext_url but for any fields that models support but schemas
   don't describe.

## 3. Scope & Objectives

- **Primary Outcomes**:
  1. Registry sync propagates ext_id/ext_url into derived YAML for all artifact types
  2. `schema show` output reflects ext_id/ext_url for all applicable schemas
  3. Audit of schema vs model field parity across all artifact types — document
     and fix any gaps found
- **Operational Constraints**: Purely additive; no breaking changes
- **Dependencies**: DE-067 (complete)

## 4. Out of Scope

- ext_id uniqueness enforcement or indexing
- ext_url format validation
- New schema validation rules beyond field presence

## 5. Approach Overview

- **System Touchpoints**:
  - Registry sync adapters (each artifact type's registry → YAML serialisation)
  - Schema definitions / validation (wherever schemas are defined)
  - `schema show` command output
- **Key Changes**:
  1. Add ext_id/ext_url to registry YAML serialisation for each artifact type
  2. Add ext_id/ext_url to schema definitions
  3. Audit all artifact types for model ↔ schema field parity; fix gaps

## 6. Verification Strategy

- **VT**: Roundtrip test — artifact with ext_id/ext_url → sync → registry YAML
  contains the fields
- **VT**: `schema show` output includes ext_id/ext_url for applicable types
- **VA**: Agent audit of model vs schema field parity across all types
- **Acceptance Criteria**:
  1. `sync` preserves ext_id/ext_url in registry YAML
  2. Schema definitions are complete w.r.t. model fields
  3. `just check` passes

## 7. Risks & Mitigations

- **Risk**: Schema changes affect validation strictness –
  _Likelihood_: low – _Impact_: low –
  _Mitigation_: Fields are optional; adding them to schema is additive
