## 1. Summary & Context
- **Technical Spec(s)**: SPEC-129 – installer component responsible for bootstrapping the repo workspace layout.
- **Implementation Plan**: [IP-072](./IP-072.md) – draft execution plan for removing installer-created compatibility symlinks.
- **Change Drivers**: ADR-006 introduced compatibility symlinks during the workspace consolidation. The current request is to stop creating those derived views during install.

## 2. Motivation
- The installer currently creates root-level compatibility symlinks such as `specify/`, `change/`, `backlog/`, and `memory/`.
- Those compatibility views are no longer desired in newly installed workspaces; the canonical install should leave `.spec-driver/` as the only workspace root created by the installer.
- This change should be narrow: remove installer symlink creation and align tests and user-facing install messaging without widening into migration tooling or broader path rewrites.

## 3. Scope & Objectives
- **Primary Outcomes**:
  - Stop the installer from creating backward-compatibility workspace symlinks.
  - Update installer verification so tests assert the canonical `.spec-driver/` layout without root-level compatibility links.
  - Adjust installer output/docs that currently claim those symlinks are created.
- **Operational Constraints**: Preserve canonical directory creation under `.spec-driver/`; do not break unrelated installer behaviors.
- **Dependencies**: None identified beyond the existing installer code and tests.

## 4. Out of Scope
- Adding migration tooling for existing repositories that already contain compatibility symlinks.
- Rewriting unrelated documentation and code paths that still reference historical locations.

## 5. Approach Overview
- **System Touchpoints**: `supekku/scripts/install.py` and the installer test suite.
- **Key Changes**:
  - Remove or bypass the installer helper that creates compatibility symlinks after canonical directory creation.
  - Update tests to verify that install leaves the canonical directories in place and omits root-level symlink creation.
  - Update user-facing install text so it does not advertise removed compatibility paths.
- **Migration / Rollout Notes**: Existing repos are untouched; this only changes behavior for installer runs after the change lands.

## 6. Verification Strategy
- **Requirements Coverage**: Use installer-focused automated tests to verify canonical directories are still created and compatibility symlinks are not.
- **Planned Artefacts**: Targeted pytest coverage for installer behavior, plus local lint on touched files.
- **Acceptance Criteria**: Installer tests pass, touched-file lint passes, and the installer no longer creates the compatibility symlink paths in a fresh workspace.

## 7. Risks & Mitigations
- **Risk**: Installer tests may encode the old symlink behavior. – *Likelihood*: medium – *Impact*: medium – *Mitigation*: update tests together with code and verify the new expected layout explicitly.
- **Risk**: User-facing install messages may drift from actual behavior. – *Likelihood*: medium – *Impact*: low – *Mitigation*: update any installer output that enumerates created paths.

## 8. Follow-ups & Tracking
- **Future Phases / Deltas**: Reconcile ADR-006 so the documented installer/default-layout behavior matches the new canonical-only install behavior delivered by this delta.
- **Open Decisions / Questions**: None at scoping time; implementation will treat installer-only removal as the bounded target.

## 9. Implementation Notes
- Run `uv run spec-driver` commands for workflow artefacts and use targeted pytest/lint verification while implementing.
- Reference context: ADR-006 for the current compatibility-path rationale and `supekku/scripts/install.py` for the active installer implementation.
