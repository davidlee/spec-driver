# DR-073 – CLI output selectors, view UX, and resolve-links improvements

## 1. Executive Summary

- **Delta**: [DE-073](./DE-073.md)
- **Status**: draft
- **Last Updated**: 2026-03-09
- **Synopsis**: Unify output selection across `show`, fix `view` pager default,
  add `read` alias, and improve `resolve links` diagnostics.

## 2. Problem & Constraints

### Current Behaviour

- **`show`**: Each subcommand has ad-hoc output flags (`--raw`, `--json`,
  `--body-only`, `--path`). No unified content-type selector.
- **`view`**: Always pages via `open_in_pager()`. No way to get plain stdout
  output. Pager cascade is `$PAGER` → `ov` → `less` → `more` — no rich rendering.
- **`read`**: Does not exist. Users who can't remember `show` vs `view`
  have no fallback synonym.
- **`resolve links`**: Whole-corpus only. No verbose mode showing which files
  contain missing targets. No scoped resolution.

### Drivers

- ISSUE-035 (memory CLI UX), ISSUE-036 (standardize output selectors)
- User feedback: pager-by-default on `view` is disruptive for quick lookups.

### Constraints

- No breaking changes to existing `--raw`/`--json`/`--path` flags.
- `-c` is unused on `show` and `view` subcommands (used on some `list`
  commands for `--category` — no conflict).
- `view` behavioural change (no pager by default) is intentional and desired.

## 3. Architecture Intent

### Target Outcomes

1. `show <artifact> -c markdown|frontmatter|yaml` as unified content selector.
2. `view` renders markdown to stdout by default (glow → rich → raw); `--pager/-p` adds paging.
3. `read` works identically to `view`.
4. `resolve links --verbose` reports missing targets with containing files.
5. `resolve links --path <file>` / `--id <mem-id>` scopes resolution.

### Guiding Principles

- Additive changes only; existing flags remain functional.
- Thin CLI — new flags compose with existing output helpers, not new formatters.
- **Reduce duplication**: `view.py` has ~16 near-identical subcommands
  (load registry → get artifact → `open_in_pager(path)`). The `--pager` flag
  addition is an opportunity to collapse these into a shared helper or
  decorator, not to copy-paste the flag into each one.

## 6. Design Decisions & Trade-offs

### DEC-073-01: Two cascades — render-only vs render+page

**Render-only** (default `view`): `glow` → `rich` → raw stdout.
Purpose: pretty output without blocking the terminal.

**Render+page** (`view --pager`): `$PAGER` → `glow -p` → `rich --pager` → `ov` → `less` → `more`.
Purpose: full-screen reading of longer documents.

**Rationale**: `$PAGER` is an explicit user choice and should override automatic
tool detection in paged mode. In non-paged mode, the goal is rendering — plain
pagers like `less` are not useful, so they don't appear in the cascade.

### DEC-073-02: `--content-type` coexists with existing flags

`--content-type/-c` is a unified alternative to `--raw`/`--json`/`--body-only`.
Existing flags remain as shortcuts. If both are specified, `--content-type` wins
with a warning.

**Rationale**: No breaking changes. Existing scripts and muscle memory keep
working. New flag provides a discoverable, consistent interface.

**Mapping**:

- `-c markdown` ≈ `--raw` (full file content)
- `-c frontmatter` ≈ default `show` (metadata only, structured)
- `-c yaml` → raw YAML frontmatter block only

### DEC-073-03: `read` as second `add_typer` registration

```python
# main.py
app.add_typer(view.app, name="view", help="View artifacts (rendered markdown)")
app.add_typer(view.app, name="read", help="Read artifacts (alias for view)")
```

**Rationale**: Simplest possible approach — zero code duplication, same app
instance, same `InferringGroup` dispatch. Both names appear in `--help`.

**Risk**: Typer may not support registering the same app instance twice. If not,
fallback is a thin wrapper module that re-exports view.app commands.

### DEC-073-04: `view` renders to stdout by default, `--pager` adds paging

Current `view` always pages via `$PAGER`/`less`/`more`. Target: two modes.

**Without `--pager`** (default) — rendered stdout, no paging:

1. `glow <file>` (if available) — renders markdown to terminal
2. `rich <file>` / `python -m rich.markdown` (if available)
3. Raw markdown to stdout (fallback)

**With `--pager/-p`** — rendered + paged:

1. `$PAGER` (if set — user's explicit choice wins)
2. `glow -p <file>` (built-in pager mode)
3. `rich --pager <file>`
4. `ov` → `less` → `more`

This is a **behavioural change** but the current default is the pain point.
No deprecation period — the old behaviour is what we're fixing.

**Implementation**: New helper functions in `common.py`:

```python
def render_file(path: Path) -> None:
    """Render markdown file to stdout (glow > rich > raw)."""
    ...

def render_file_paged(path: Path) -> None:
    """Render markdown file in a pager ($PAGER > glow -p > rich --pager > less)."""
    ...
```

Replace `open_in_pager(path)` calls in view.py with:

```python
if pager:
    render_file_paged(path)
else:
    render_file(path)
```

## 7. Resolve Links Design Detail

### `--verbose` flag

When set, `resolve links` additionally prints each missing target with the
file(s) that reference it:

```
MISSING: mem.concept.foo
  referenced by: .spec-driver/memories/mem.pattern.bar.md:14
  referenced by: .spec-driver/memories/mem.signpost.baz.md:7
```

### `--path <file>` / `--id <mem-id>` scoping

- `--path <file>`: resolve only links found in the given file.
- `--id <mem-id>`: resolve only links found in the memory with the given ID
  (convenience wrapper — resolves ID to path, then delegates to `--path`).
- Composable with `--verbose` and `--dry-run`.
- `--link-mode` still applies to scoped resolution.
