## 1. Summary & Context

- **Implementation Plan**: [IP-074](./IP-074.md)
- **Change Drivers**: User request — no way to filter backlog by p1/p2/p3

## 2. Motivation

`list issues/problems/improvements/risks` can sort by severity (`--order severity`)
but cannot filter to only show items at a given priority level. The registry
layer already supports `filter(severity=...)` — this is pure CLI plumbing.

## 3. Scope & Objectives

- **Primary Outcome**: `--severity` flag on `list backlog` and all 4 kind-specific
  wrappers (`list issues`, `list problems`, `list improvements`, `list risks`).
- **Dependencies**: None.

## 4. Out of Scope

- Severity enum validation (accept any string, match against item.severity).
- Multi-value severity filter (e.g. `--severity p1,p2`).

## 5. Approach Overview

- **System Touchpoints**: `supekku/cli/list.py`
- **Key Changes**:
  - Add `severity` param to `list_backlog()` + filter logic
  - Pass through from 4 kind-specific wrappers
- **Migration**: None — additive flag.

## 6. Verification Strategy

- VT: CLI test confirming `--severity p1` filters correctly.
- **Acceptance Criteria**: `list issues --severity p1` shows only p1 items.

## 7. Risks & Mitigations

None — trivial additive change, registry support exists.
