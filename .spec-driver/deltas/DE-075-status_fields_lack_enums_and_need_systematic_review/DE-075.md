## 1. Summary & Context

- **Product Spec(s)**: PROD-004 – Frontmatter Metadata Validation; PROD-008 – Requirements Lifecycle Coherence

- **Change Drivers**: ISSUE-009 – status fields lack enums and need systematic review

## 2. Motivation

Status fields for specs, ADRs, policies, standards, and memories are free-form strings with no enum constraints. The `ENUM_REGISTRY` in `core/enums.py` already covers changes, requirements, backlog, verification, and drift — but governance artifacts are missing.

Adding these enums is the remaining work to close ISSUE-009 and will immediately enable CLI validation via `edit --status` (DE-068) with no further code changes.

Additionally, `theme.py` colour mappings diverge from defined enums in several places (e.g. risk `suspected` unstyled, backlog statuses styled but undefined).

## 3. Scope & Objectives

- **Primary Outcomes**:
  1. Define status constant sets for: specs, ADRs/decisions, policies, standards, memories
  2. Register all five in `ENUM_REGISTRY`
  3. Align `theme.py` status colour mappings with defined enums (fix gaps, remove phantoms)
- **Operational Constraints**: Pure additive — no breaking changes to existing status values on disk
- **Dependencies**: None (DE-068 consumes `ENUM_REGISTRY` independently)

## 4. Out of Scope

- Frontmatter schema enforcement (replacing `".+"` patterns with enum constraints) — separate delta
- Migrating existing documents to normalised status values
- Converting frozensets to Python `Enum` types (ISSUE-009 acceptance criterion, future delta)
- Status lifecycle documentation / semantics (PROD-009)

## 5. Approach Overview

- **System Touchpoints**: `specs/`, `decisions/`, `policies/`, `standards/`, `memory/`, `core/enums.py`, `formatters/theme.py`
- **Key Changes**:
  1. Add `STATUS_*` constants (frozensets) to appropriate domain modules — follow the pattern in `backlog/models.py` and `changes/lifecycle.py`
  2. Register new paths in `ENUM_REGISTRY` (`spec.status`, `adr.status`, `policy.status`, `standard.status`, `memory.status`)
  3. Audit `theme.py` — ensure every defined status has a colour and remove styled-but-undefined entries
- **Migration / Rollout Notes**: No migration needed. All status values currently on disk must be members of the new enums (survey confirms they are).

## 6. Verification Strategy

- **VT**: Unit tests for each new constant set (membership, non-empty, frozen)
- **VT**: `ENUM_REGISTRY` returns correct values for new paths
- **VA**: `theme.py` audit — every status in every enum has a colour mapping
- **Acceptance Criteria**:
  - `get_enum_values("spec.status")` returns correct list
  - `get_enum_values("adr.status")` returns correct list
  - `get_enum_values("policy.status")` returns correct list
  - `get_enum_values("standard.status")` returns correct list
  - `get_enum_values("memory.status")` returns correct list
  - All existing on-disk status values are members of their respective enums
  - No unstyled status values in any enum; no styled values outside any enum

## 7. Risks & Mitigations

- **Risk**: On-disk status value not in proposed enum – _Likelihood_: low – _Impact_: medium – _Mitigation_: Survey already done; add any stragglers found during implementation
- **Risk**: Theme colour removal breaks display – _Likelihood_: low – _Impact_: low – _Mitigation_: Only remove truly phantom entries (styled but never valid)
