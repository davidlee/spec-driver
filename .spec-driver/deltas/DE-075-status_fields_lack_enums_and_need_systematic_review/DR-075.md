# DR-075 – Status enum values for governance artifacts

## 1. Executive Summary

- **Delta**: [DE-075](./DE-075.md)
- **Status**: draft
- **Last Updated**: 2026-03-09
- **Synopsis**: Define canonical status enums for governance artifacts (specs, ADRs, policies, standards, memories) and unify backlog item statuses into a shared lifecycle, resolving value ambiguity and enabling ENUM_REGISTRY validation.

## 2. Problem & Constraints

- **Current Behaviour**: Status fields for specs, ADRs, policies, standards, and memories are free-form strings. Theme.py defines colour mappings that diverge from actual usage. No validation exists.
- **Drivers**: ISSUE-009; DE-068 (`edit --status`) consumes ENUM_REGISTRY — adding entries here enables validation automatically.
- **Constraints**: All values currently on disk must be valid members of the new enums or migrated. Governance artifacts are pure additive; backlog unification requires a small migration.
- **Out of Scope**: Frontmatter schema enforcement, Python Enum types, status lifecycle semantics (PROD-009).

## 3. Architecture Intent

- **Target Outcomes**:
  - Every governance artifact type has a defined frozenset of valid statuses
  - All five registered in `ENUM_REGISTRY`
  - Theme mappings 1:1 with defined enums (no phantoms, no gaps)
- **Guiding Principles**: Follow existing pattern (frozensets in domain modules, registered in `core/enums.py`). Minimal set — only statuses that represent distinct lifecycle states.

## 4. Design Decisions

### DEC-075-01: Spec statuses — drop `live`

`live` is an archaic synonym of `active` with no semantic distinction. Remove from theme, do not include in enum.

**Proposed enum:**

```
SPEC_STATUSES = frozenset({"stub", "draft", "active", "deprecated", "archived"})
```

Lifecycle: `stub` → `draft` → `active` → `deprecated` → `archived`

- `stub`: auto-created placeholder, minimal content
- `draft`: under active authoring
- `active`: accepted and governing
- `deprecated`: superseded or no longer recommended
- `archived`: retained for reference, no longer governing

On-disk check: `stub` (42), `draft` (10), `active` (product specs) — all covered.

### DEC-075-02: ADR statuses — keep `superseded`, add theme mapping

Registry.py already infers statuses from directory names. Extract these as a constant.

**Proposed enum:**

```
ADR_STATUSES = frozenset({
    "draft", "proposed", "accepted", "rejected",
    "deprecated", "superseded", "revision-required"
})
```

`superseded` currently has no theme colour — add one. All 7 values are meaningful and distinct in ADR governance.

### DEC-075-03: Policy and standard statuses — enforcement-level, not lifecycle

Policies and standards use `required` and `default` on disk, not `active`. These are enforcement levels that double as lifecycle indicators. Introducing `active` alongside `required` creates confusion about what "active" means when `required` already implies it.

**Proposed policy enum:**

```
POLICY_STATUSES = frozenset({"draft", "required", "deprecated"})
```

**Proposed standard enum:**

```
STANDARD_STATUSES = frozenset({"draft", "required", "default", "deprecated"})
```

Standards get `default` (recommended, opt-out allowed) alongside `required` (must comply). Policies are always required when active, so only `required`.

Theme: drop `policy.status.active`, add `policy.status.required`. Standards already have all four themed.

### DEC-075-04: Memory statuses — trim the tail

7 statuses for memories is excessive. `deprecated`, `superseded`, and `obsolete` are three shades of "no longer current." Memories are lightweight artifacts — a simpler lifecycle suffices.

**Proposed enum:**

```
MEMORY_STATUSES = frozenset({"draft", "active", "review", "superseded", "archived"})
```

- `draft`: being written, not yet reliable
- `active`: current, trustworthy
- `review`: flagged for accuracy check (useful for memory maintenance skills)
- `superseded`: replaced by a specific successor (linkable)
- `archived`: no longer relevant, retained for reference

Dropped: `deprecated` (use `archived`), `obsolete` (use `archived` or `superseded`).

### DEC-075-05: Unify backlog statuses — shared base with risk extensions

Per-kind status sets (DE-057) introduced 4 separate frozensets totalling 20 values across issue/problem/improvement/risk. Many are kind-flavoured synonyms of the same lifecycle phase (`captured`/`open`/`idea`/`suspected` all mean "new"). The kind field already carries domain semantics — status doesn't need to repeat it.

**Proposed unified base:**

```
BACKLOG_STATUSES = frozenset({"open", "triaged", "in-progress", "resolved"})
```

**Risk-specific extensions** (semantically distinct — you don't "resolve" a risk):

```
RISK_EXTRA_STATUSES = frozenset({"accepted", "expired"})
RISK_STATUSES = BACKLOG_STATUSES | RISK_EXTRA_STATUSES
```

Lifecycle: `open` → `triaged` → `in-progress` → `resolved` (+ `accepted`/`expired` for risks)

- `open`: captured, not yet assessed
- `triaged`: assessed, prioritised (valuable for risks and defects; optional for others)
- `in-progress`: actively being worked
- `resolved`: addressed and closed
- `accepted` (risk only): acknowledged, not fixed — conscious decision to live with it
- `expired` (risk only): no longer relevant, overtaken by events

**What changes on disk**: Items using `captured`, `idea`, `suspected` → `open`. Items using `done`, `implemented`, `verified` → `resolved`. Items using `investigating`, `planned` → `in-progress` or `triaged`. `mitigated` (risk/problem) → `resolved`. `confirmed` (risk) → `triaged`.

**Migration**: One-time frontmatter update for affected items. Small set — can be done manually or with a simple script.

**ENUM_REGISTRY impact**: Replace 4 per-kind entries (`issue.status`, `problem.status`, `improvement.status`, `risk.status`) with 2 (`backlog.status`, `risk.status`) — or keep per-kind keys pointing to the shared set for backwards compatibility with existing CLI filters.

### DEC-075-06: Place status constants in `lifecycle.py` per domain package

Resolved OQ-075-01. `changes/lifecycle.py` and `requirements/lifecycle.py` both use dedicated lifecycle modules. `backlog/models.py` is the outlier (justified by per-kind coupling, now simplified by DEC-075-05). New constants go in `lifecycle.py` per package for consistency. Backlog stays in `models.py` since the constants are tightly coupled to the `BacklogItem` model and the file is small.
