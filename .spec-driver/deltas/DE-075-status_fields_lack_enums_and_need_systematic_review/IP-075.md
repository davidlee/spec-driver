## 1. Summary

- **Delta**: DE-075 – Status enum coverage for governance artifacts
- **Design Revision**: DR-075 – DEC-075-01 through DEC-075-06
- **Problems / Issues**: ISSUE-009
- **Desired Outcome**: Every entity type has a defined, registered, and themed status enum. Backlog statuses unified into shared lifecycle.

## 2. Context & Constraints

- **Current Behaviour**: 5 governance types (specs, ADRs, policies, standards, memories) lack status enums. Backlog has 4 speciated sets (20 values). Theme diverges from code.
- **Target Behaviour**: Per DR-075 — lifecycle.py per domain, ENUM_REGISTRY populated, theme aligned, backlog unified to 6 values.
- **Dependencies**: None (DE-068 consumes ENUM_REGISTRY independently).
- **Constraints**: All on-disk values must remain valid or be migrated. Backlog migration is manual (this repo only).

## 5. Testing & Verification Plan

- **Unit tests**: Each new lifecycle.py gets a test file; backlog/models_test.py updated for unified constants
- **ENUM_REGISTRY tests**: core/enums_test.py (or new) — all 5 new paths return expected values
- **Theme VA**: Programmatic or manual audit comparing theme keys to enum members
- **Disk VA**: Scan frontmatter post-migration to confirm no orphaned status values

## 6. Risks & Mitigations

| Risk | Mitigation | Owner |
|---|---|---|
| Undiscovered on-disk status value not in enum | Survey done in DR-075; re-scan in phase 2 | Dev |
| Backlog migration breaks existing filters/views | Unified set is a subset of ALL_VALID_STATUSES; update DEFAULT_HIDDEN_STATUSES | Dev |
| Theme colour removal breaks display for items with legacy status | Migrate items first (phase 2), then clean theme | Dev |
