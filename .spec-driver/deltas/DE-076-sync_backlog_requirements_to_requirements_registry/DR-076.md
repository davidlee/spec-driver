# DR-076 – Sync backlog requirements to requirements registry

## 1. Executive Summary
- **Delta**: [DE-076](./DE-076.md)
- **Status**: draft
- **Last Updated**: 2026-03-09
- **Synopsis**: Extend the requirements registry sync pipeline to discover requirements defined in backlog artifacts (issues, problems, improvements), adding source provenance metadata and CLI filtering.

## 2. Problem & Constraints

- **Current Behaviour**: `RequirementsRegistry.sync_from_specs()` only scans spec directories (`specify/product/`, `specify/tech/`) and change artifacts. Requirements defined in backlog items (e.g. ISSUE-013.FR-013.001) are structured identically but invisible to `spec-driver list requirements`, coverage tracking, and filtering.

- **Drivers / Inputs**:
  - ISSUE-016 — comprehensive problem statement with five identified gaps
  - PROD-008 — Requirements Lifecycle Coherence: registry should be a complete derived projection
  - ADR-008 — truth model: registry is derived, not authoritative

- **Constraints / Guardrails**:
  - Pure additive — no breaking changes to existing RequirementRecord serialization
  - Existing `requirements.yaml` format must remain backward-compatible (new fields optional)
  - Must not conflict with DE-075 (in-progress): DE-075 adds status enums to `ENUM_REGISTRY` for governance artifacts. DE-076 adds new fields to RequirementRecord. No overlap — DE-075 touches `core/enums.py` and domain status constants; DE-076 touches `registry.py` sync pipeline and CLI filters. If DE-075 adds a `requirement.source_kind` enum path, DE-076 should consume it rather than defining its own.

- **Out of Scope**:
  - Requirement migration tooling (`migrate requirement` command — ISSUE-016.FR-016.003)
  - Automatic promotion rules, maturity lifecycle dashboards
  - Schema enforcement for backlog requirement format (graceful skip on parse failure)

## 3. Architecture Intent

- **Target Outcomes**:
  - `sync_from_specs()` discovers requirements in `backlog/issues/`, `backlog/problems/`, `backlog/improvements/` (FR-016.001)
  - `list requirements --source-kind issue` filters by artifact source (FR-016.002)
  - Coverage tracking treats backlog-sourced requirements identically to spec-sourced ones (FR-016.004)
  - All existing workflows unchanged (NF-016.001)

- **Guiding Principles**:
  - Extend, don't fork: add backlog scanning as another source in the existing sync pipeline
  - Two regex patterns, one pipeline: specs use bullet format (`- **FR-001**: Title`), backlog uses heading + dotted format (`### FR-016.001: Title`). Both feed `_records_from_content`.
  - Source provenance is metadata, not a separate registry — unified view per PROD-008

- **State Transitions / Lifecycle Impact**:
  - New RequirementRecord fields: `source_kind` (spec|issue|problem|improvement) and `source_type` (prod|tech|backlog)
  - Deserialization default: `""` (empty). Sync sets values at extraction boundary (DEC-076-05). CLI treats `""` as matching any filter.
  - No changes to requirement status lifecycle (pending→in-progress→active→retired)

## 7. Design Decisions & Trade-offs

- **DEC-076-01**: Extend existing sync pipeline, not parallel discovery.
  *Rationale*: `sync_from_specs` already handles multi-source scanning (specs, deltas, revisions, audits). Adding backlog as another source follows the established pattern. A separate "sync_from_backlog" would create a competing sync path.
  *Consequence*: `backlog_dirs` becomes another optional parameter. The method name `sync_from_specs` is now actively misleading — it syncs from specs, deltas, revisions, audits, plans, and backlog. Rename to `sync()` in the cleanup phase. The old name becomes a deprecated alias (one-line wrapper) for backward compat.

- **DEC-076-02**: Source provenance as optional fields with backward-compatible empty defaults.
  *Rationale*: Existing serialized records lack `source_kind`/`source_type`. Using `""` (empty) as the deserialization default means old records load without error. Sync backfills correct values at next run (DEC-076-05).
  *Serialization*: `to_dict()` includes both fields. `from_dict()` defaults missing fields to `""`. `merge()` takes the incoming (new) value for `source_kind`/`source_type` — sync is authoritative for provenance, so the freshly-extracted value wins over a stale or empty stored value.
  *Consequence*: First sync after DE-076 will add `source_kind`/`source_type` to all existing records in `requirements.yaml`. This is a benign diff.

- **DEC-076-03**: Extend requirement parsing to handle backlog formats.
  *Rationale*: Specs use `- **FR-001**: Title` (bullet, bold, 3-digit sequential). Backlog items use `### FR-016.001: Title` (heading, dotted `{artifact-num}.{seq}`). These are structurally different — the existing `_REQUIREMENT_LINE` regex does not match headings and cannot capture dotted numbers. A second regex is needed for the heading format, and the number capture group must handle the dotted scheme.
  *Design*: Add `_REQUIREMENT_HEADING` regex for `### FR-NNN.MMM: Title`. Both regexes feed the same `_records_from_content` pipeline. Label construction for dotted format: `FR-016.001` (full dotted label), UID: `ISSUE-016.FR-016.001`. `_records_from_content` remains source-agnostic — the caller provides the artifact ID, the regex determines the label format.
  *Consequence*: The heading regex also picks up spec requirements authored as headings (if any). This is acceptable — it widens extraction, doesn't narrow it.

- **DEC-076-04**: Pre-implementation cleanup before adding the backlog source.
  *Rationale*: The merge-or-create upsert loop is already duplicated twice (spec_registry path and spec_dirs path). Adding backlog scanning without cleanup would create a third copy. `_iter_plan_files` is a redundant specialization of `_iter_change_files`. Cleaning these up first keeps the feature diff minimal and reviewable.
  *Consequence*: Phase 1 is a pure refactor with no behavioral change — all existing tests must pass identically.

- **DEC-076-06**: `seen` set must track backlog-sourced requirements; cleanup logic must not purge them.
  *Rationale*: Lines 392–408 of `sync_from_specs` iterate all records and "clean" any not in `seen`. Currently `seen` is only populated by the spec extraction loops. If backlog requirements are upserted but not added to `seen`, they will be silently dropped on the next sync.
  *Design*: The `_upsert_record` helper (DEC-076-04) unconditionally adds every upserted record to `seen`. This guarantees that any source — specs, backlog, or future additions — is protected from cleanup. No source-specific `seen` logic needed.

- **DEC-076-05**: Set source provenance at the extraction boundary; sync always backfills.
  *Rationale*: Using `source_kind="spec"` as a default would mean records that were never explicitly classified claim to be spec-sourced. Instead, `_upsert_record` accepts `source_kind` and `source_type` parameters, set once per source loop. This makes provenance explicit and truthful.
  *Backfill strategy*: Sync sets `source_kind` based on the source loop that produced the record — spec loops set `"spec"`, backlog loops set `"issue"`/`"problem"`/`"improvement"`. On deserialization, missing `source_kind` defaults to `""`. The CLI filter does **not** infer from UID prefix — it treats `""` as matching any filter (i.e. unclassified records always pass). After one sync cycle, all records will have explicit `source_kind` values. This avoids two competing inference strategies.
  *Consequence*: First sync after DE-076 backfills all records. The YAML diff is benign.

- **DEC-076-07**: Use `BacklogRegistry` for iteration; split frontmatter from body via `load_markdown_file`.
  *Rationale*: `BacklogRegistry` already discovers all backlog items, provides frontmatter and `path`. Reimplementing file discovery in `RequirementsRegistry` would duplicate `BacklogRegistry._load()`. The missing piece is the markdown body — `BacklogRegistry` only stores frontmatter, not body text.
  *Design*: The sync loop iterates `BacklogRegistry.iter()`, calls `load_markdown_file(item.path)` to get `(frontmatter, body)`, then calls `_records_from_content(item.id, frontmatter, body, item.path, repo_root)`. **Must use `load_markdown_file`** — not `item.path.read_text()` — to strip YAML frontmatter before regex scanning. This matches the existing `spec_dirs` path (line 313). The `backlog_dirs` parameter on `sync()` is replaced by an optional `backlog_registry: BacklogRegistry | None` parameter — consistent with how `spec_registry: SpecRegistry | None` is already passed.
  *Consequence*: Requires importing `BacklogRegistry` in requirements registry. This is a new dependency direction (requirements → backlog) but follows the same pattern as the existing requirements → specs dependency.

- **DEC-076-08**: Heading-format requirements are extracted but do not create false positives in specs.
  *Rationale*: Adding a heading regex (`### FR-NNN.MMM:`) widens extraction. If specs also use headings with requirement-like text (e.g. `### FR-001: Title` as a section header above the actual bullet-format requirement), this could create duplicate records.
  *Design*: The heading regex only matches the dotted format (`FR-NNN.MMM`) — not the 3-digit format used by specs. This prevents false positives: spec headings like `### FR-001: Title` won't match the dotted regex, and backlog headings like `### FR-016.001: Title` won't match the bullet regex. The two regexes are format-specific, not additive.

## 8. Open Questions

All resolved during design:
- Source provenance representation: decided on flat fields, not nested object
- Backlog vs spec requirement format: structurally different (dotted vs sequential); two regexes needed
- DE-075 coordination: no conflict, different code surfaces
- Pre-cleanup scope: extract `_upsert_record`, eliminate `_iter_plan_files`, rename `sync_from_specs` → `sync()`
- `seen` set: `_upsert_record` unconditionally adds to `seen`
- `source_kind` backfill: sync sets it; CLI doesn't infer; `""` passes all filters until first sync
- Iteration mechanism: `BacklogRegistry` for discovery, read body from file path
- Heading regex scope: dotted format only, no overlap with spec bullet format

## 9. Rollout & Operational Notes

- **Migration / Backfill**: None required. Next `spec-driver sync requirements` will discover backlog requirements and add `source_kind`/`source_type` to all records. Idempotent and safe.
- **Recovery / Rollback**: Registry is derived — delete `requirements.yaml` and re-sync to revert to any state.
- **Known limitation (pre-existing)**: Requirement regexes match inside markdown code fences (` ```markdown ` blocks). This can produce phantom records from example/documentation text. Not introduced by DE-076 — affects spec sync today. Out of scope; note for future improvement.
