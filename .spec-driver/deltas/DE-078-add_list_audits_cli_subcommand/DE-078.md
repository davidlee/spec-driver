## 1. Summary & Context

- **Product Spec(s)**: PROD-016 – Agentic SDLC Configurator (FR-010: first-class command coverage for change primitives)
- **Technical Spec(s)**: SPEC-110 – CLI Architecture (FR-002: delegate to registries/formatters; FR-005/FR-006: consistent flag patterns and output formats)
- **Implementation Plan**: [IP-078](./IP-078.md)

## 2. Motivation

Every other first-class primitive has a dedicated `list` subcommand (`list deltas`, `list revisions`, `list plans`, etc.). Audits are the sole exception — users must use `list changes -k audit`, which is non-discoverable and inconsistent. All infrastructure exists (ChangeRegistry, formatters); only the CLI wiring is missing.

## 3. Scope & Objectives

- **Primary Outcomes**:
  - `spec-driver list audits` (+ `list audit` alias) with standard flags (--status, --filter, --regexp, --format, --json, --truncate, --external)
  - Follows identical pattern to `list revisions`
  - Delegates to `ChangeRegistry(kind="audit")` and `format_change_list_table()`

## 4. Out of Scope

- `complete audit` — audits are not lifecycle-completed artifacts like deltas/revisions
- Audit-specific formatters or columns beyond what `format_change_list_table` provides
- Any changes to the audit domain model or registry

## 5. Approach

- **File**: `supekku/cli/list.py`
- **Pattern**: Clone `list_revisions` (lines ~1347–1446), change kind to `"audit"`
- **No new modules, no new formatters** — pure CLI wiring

## 6. Verification Strategy

- **VT-078-001**: Unit test that `list audits` returns expected output for test fixtures
- **VT-078-002**: Flag parity test (--status, --filter, --regexp, --format, --json behave identically to `list revisions`)
- **Acceptance**: `uv run spec-driver list audits --help` shows standard flags; `just` passes clean

## 7. Risks & Mitigations

- **Risk**: None significant — trivial wiring change with established pattern
