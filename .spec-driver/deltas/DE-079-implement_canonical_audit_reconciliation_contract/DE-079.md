## 2. Motivation

- The canonical loop says audits are the reconciliation checkpoint between implementation and closure, but the current runtime treats them as optional and underspecified.
- `audit-change` is advisory only — it does not require a structured AUD artefact or per-finding disposition.
- `create audit` scaffolds a nearly empty shell with no owning-delta linkage or reconciliation structure.
- `complete delta` gates only on parent-spec coverage; it does not check whether an audit exists, whether drift was dispositioned, or whether required spec patches were made.
- Requirements sync records `verified_by` audit IDs but does not surface unresolved audit drift as a first-class workflow outcome.
- Audit lifecycle language is inconsistent (`complete`, `completed`, `approved`, `in-review`).
- The audit template and metadata schema are richer than the runtime semantics that consume them.
- DE-055 designed the contract; this delta implements it.

## 3. Scope & Objectives

- **Primary Outcomes**:
  - Audit template and schema carry a canonical reconciliation contract: mode, delta_ref, per-finding structured disposition, optional drift-ledger linkage
  - `audit-change` skill becomes the canonical procedural runsheet for creating/completing audits with per-finding disposition
  - `complete delta` adds audit-reconciliation pre-checks for qualifying deltas
  - Validation surfaces missing audits and unresolved findings as closure-grade warnings/errors
  - Audit lifecycle uses canonical change statuses consistently
  - Drift-ledger compatibility: audit findings can optionally link to DL entries without collapsing the two primitives
- **Operational Constraints**:
  - Remove deprecated fields cleanly (only 2 existing audits; 1 needs manual edit)
  - Do not make every delta require an audit — only qualifying deltas (requirement-bearing or spec-governed behaviour changes)
  - Preserve the conformance vs discovery distinction in closure severity
- **Dependencies**:
  - DE-055 (design source — DR-055 audit sections)
  - DE-065 (drift ledger primitive — completed; compatibility target)

## 5. Approach Overview

- **System Touchpoints**:
  - `supekku/templates/audit.md` — template with reconciliation contract
  - `supekku/scripts/lib/core/frontmatter_metadata/audit.py` — audit metadata: disposition sub-schema, mode, delta_ref
  - `supekku/scripts/lib/core/frontmatter_metadata/delta.py` — delta metadata: audit_gate, audit_gate_rationale
  - `supekku/scripts/lib/changes/creation.py` — audit scaffolding with mode, delta_ref, disposition placeholders
  - `supekku/skills/audit-change/SKILL.md` — canonical reconciliation runsheet
  - `supekku/scripts/complete_delta.py` — audit-gating pre-checks (resolve audit_gate, locate audit, derive closure effects)
  - `supekku/scripts/lib/validation/validator.py` — missing audit / undispositioned finding / invalid status×kind warnings
- **Key Changes**:
  1. Standardise audit metadata on shared change lifecycle (DEC-079-002). Drop `in-review`/`complete`.
  2. Add per-finding inline disposition with structured refs and drift_refs (DEC-079-001)
  3. Remove `patch_level` and `next_actions` entirely; manually edit the one existing audit that uses them (DEC-079-004)
  4. Add `mode` and `delta_ref` to audit frontmatter; add `audit_gate` and `audit_gate_rationale` to delta frontmatter (DEC-079-003)
  5. Make `create audit` scaffold useful audits with owning-delta linkage and disposition structure
  6. Rewrite `audit-change` skill as the explicit reconciliation runsheet requiring AUD artefact and per-finding disposition
  7. Add audit-gating pre-check to `complete delta` for qualifying deltas (DEC-079-003, DEC-079-005)
  8. Surface audit gaps in validation output (missing audit, undispositioned findings, invalid combinations)
  9. Derive `closure_effect` at check time; never store it (DEC-079-005, DEC-079-006)
  10. Optional drift-ledger linkage from findings to DL entries via disposition.drift_refs

## 6. Verification Strategy

- **Requirements Coverage**: to be mapped once DR-079 is shaped
- **Planned Artefacts**:
  - VA-079-001: Audit template/schema review — confirm per-finding disposition is machine-checkable
  - VA-079-002: Delta completion gate test — confirm qualifying deltas cannot close without reconciled audit
  - VA-079-003: Validation output review — confirm missing/unresolved audits surface as warnings/errors
  - VT-079-\*: Unit tests for schema, creation, completion, and validation changes
- **Acceptance Criteria**:
  - Every conformance audit finding must end with a deterministic disposition before the owning delta can close

  - `complete delta` rejects qualifying deltas with missing or unreconciled audits (unless `--force`)
  - Existing audits are migrated (1 manual edit); deprecated fields removed from schema

## 7. Risks & Mitigations

- **Risk**: Over-ceremony — requiring audits for trivial deltas. _Likelihood_: medium. _Impact_: medium. _Mitigation_: `audit_gate: auto` resolves to non-gating when `applies_to.requirements` is empty; `exempt` with rationale for genuinely non-qualifying work.
- **Risk**: Audit schema changes break existing audit artefacts. _Likelihood_: low. _Impact_: high. _Mitigation_: additive fields only; existing audits validate without new fields; deprecated fields remain parseable.
- **Risk**: `tolerated_drift` becomes an easy escape hatch. _Likelihood_: medium. _Impact_: high. _Mitigation_: conformance + tolerated_drift defaults to block (DEC-079-006); relaxation requires explicit `closure_override` with rationale.
