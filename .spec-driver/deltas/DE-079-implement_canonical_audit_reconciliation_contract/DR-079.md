# DR-079 — Implement canonical audit reconciliation contract

## 1. Executive Summary

- **Delta**: [DE-079](./DE-079.md)
- **Status**: draft
- **Owners / Team**: david
- **Last Updated**: 2026-03-09
- **Synopsis**: Close the gap between doctrinal audit expectations (ADR-004 §10, ADR-008 §3–4) and current runtime behaviour by implementing a machine-checkable per-finding disposition contract, standardising audit metadata on shared change lifecycle statuses, and adding audit-gating pre-checks to delta completion.

## 2. Problem & Constraints

### Current Behaviour

- `audit-change` is advisory: it instructs spec reconciliation but does not require a structured AUD artefact or per-finding disposition.
- `create audit` scaffolds a nearly empty shell — no owning-delta linkage, no mode, no reconciliation structure.
- The audit template uses non-canonical statuses (`in-review`, `complete`) that diverge from the shared change lifecycle in `lifecycle.py`.
- `findings` exist in the schema but carry only `outcome` (drift/aligned/risk) with no structured disposition.
- `patch_level` and `next_actions` are separate top-level arrays disconnected from individual findings — reconciliation intent is scattered, not per-finding.
- `complete delta` gates only on parent-spec coverage; it does not check whether an audit exists, whether drift was dispositioned, or whether required reconciliation artefacts were created.
- Requirements sync records `verified_by` audit IDs but does not surface unresolved audit drift as a closure-grade outcome.

### Drivers / Inputs

- **ADR-004 §10**: Closure requires owning-record reconciliation. "Implemented now, fix the specs later" is drift, not closure.
- **ADR-008 §3**: Drift is resolved explicitly, never by timestamp precedence.
- **ADR-008 §4**: New evidence triggers reconciliation; it does not rewrite authority.
- **DR-055**: Designed the audit reconciliation contract (DEC-055-011 through DEC-055-015); this delta implements it.
- **DE-065**: Drift ledger primitive is completed; audit findings may optionally link to DL entries.

### Constraints / Guardrails

- Additive schema changes only — existing audits must remain valid without the new fields.
- Not every delta requires an audit — only qualifying deltas (determined by `audit_gate` resolution).
- Conformance and discovery audits have different closure severity by design.
- Drift-ledger linkage is always optional. A DL entry never satisfies a blocking conformance finding by itself.
- `closure_effect` is derived, never stored — no competing truths.
- Reuse shared change lifecycle vocabulary (POL-001, POL-002).

### Out of Scope

- Skill routing and boot-time guidance (DE-055).
- DR authoring improvements (DE-055).
- Drift-ledger creation/management CLI (DE-065, completed).
- Backlog status vocabulary (blocked on ISSUE-009).
- Full spec corpus reconciliation.

## 3. Architecture Intent

### Target Outcomes

- Every conformance audit finding ends with a deterministic, machine-checkable disposition before the owning delta can close.
- Discovery audits surface findings as warnings without blocking closure by default.
- `complete delta` rejects qualifying deltas with missing or unreconciled audits (unless `--force`).
- Audit metadata uses the canonical change lifecycle from `lifecycle.py`, eliminating `in-review` and `complete`.
- `create audit` scaffolds useful audits with mode, delta_ref, and disposition structure.
- `audit-change` becomes the canonical procedural runsheet for producing closure-grade audits.
- Existing audits without the new disposition fields remain valid (additive, not breaking).

### Guiding Principles

- **Single closure truth per finding**: disposition is inline, not a separate block.
- **Status routes, kind classifies**: `status` says whether the audit has finished routing the finding; `kind` says what the routing path is.
- **Derive, don't declare closure effect**: the gate result is a function of (mode, outcome, status, kind), computed at check time.
- **Explicit is better than automatic**: qualifying deltas are identified by an auditable `audit_gate` field, not invisible inference.

### State Transitions / Lifecycle Impact

**Audit lifecycle** (canonical path):

```
draft → in-progress → completed
```

`pending` and `deferred` remain valid as shared change statuses but are exceptional, not the normal audit loop.

**Finding disposition lifecycle**:

```
pending → reconciled   (for aligned, spec_patch, revision)
pending → accepted     (for follow_up_delta, follow_up_backlog, tolerated_drift)
```

**Delta completion with audit gate**:

```
complete delta DE-XXX
  → resolve audit_gate (auto → required|non-gating, required, exempt)
  → if required:
      → collect all AUD-* with delta_ref: DE-XXX and mode: conformance and status: completed
      → union all findings across collected audits
      → warn if finding IDs collide across audits
      → derive closure_effect for each finding
      → block if any finding derives to block
  → existing coverage check (unchanged)
  → proceed to completion
```

**Multi-audit resolution**: when multiple completed conformance audits exist for one delta, all contribute findings by union. This handles split-scope audits (one per spec area) and reruns. Finding ID collisions across audits are a validation warning — the author should deduplicate or use distinct IDs. Supersession (later audit replaces earlier for overlapping scope) is deferred until proven necessary.

### Delta frontmatter addition

```yaml
# In delta frontmatter (DE-XXX.md)
audit_gate: auto # auto | required | exempt
audit_gate_rationale: "" # required when audit_gate: exempt
```

### Finding disposition schema (detailed)

```yaml
findings:
  - id: FIND-001
    description: Content reconciler skips schema enforcement.
    outcome: drift # drift | aligned | risk
    disposition:
      status: reconciled # reconciled | accepted | pending
      kind: spec_patch # aligned | spec_patch | revision | follow_up_delta | follow_up_backlog | tolerated_drift
      refs:
        - kind: spec
          ref: SPEC-101
      drift_refs: # optional, plural, inside disposition
        - kind: drift_entry
          ref: DL-047.003
      rationale: "Patched SPEC-101 to remove schema enforcement requirement"
      closure_override: # optional — only for non-default gate relaxation
        effect: warn # must be less strict than derived default
        rationale: "..."
```

### Valid status × kind pairs

| kind                | pending | accepted | reconciled |
| ------------------- | ------- | -------- | ---------- |
| `aligned`           | —       | —        | yes        |
| `spec_patch`        | yes     | —        | yes        |
| `revision`          | yes     | —        | yes        |
| `follow_up_delta`   | yes     | yes      | —          |
| `follow_up_backlog` | yes     | yes      | —          |
| `tolerated_drift`   | —       | yes      | —          |

### Valid outcome × kind pairs

| outcome   | valid kinds                                                                         |
| --------- | ----------------------------------------------------------------------------------- |
| `aligned` | `aligned` only                                                                      |
| `drift`   | `spec_patch`, `revision`, `follow_up_delta`, `follow_up_backlog`, `tolerated_drift` |
| `risk`    | `spec_patch`, `revision`, `follow_up_delta`, `follow_up_backlog`, `tolerated_drift` |

Rules:

- `aligned` outcome confirms no action needed — only `aligned` kind is valid.
- `drift` and `risk` outcomes require an action-bearing kind — you cannot disposition drift as "aligned."
- Invalid outcome × kind combinations are schema errors, rejected by validation.

Semantic rules:

- `status` describes whether this audit has finished routing the finding.
- `kind` describes what that routing path is.
- `accepted` for `follow_up_delta`/`follow_up_backlog` means "the follow-up artefact exists and has an owner" — not that the underlying drift is resolved.
- `reconciled` is not valid for follow-up kinds because the actual reconciliation happens in the follow-up, not in this audit.

### Closure-effect derivation rules

```
if audit.mode == conformance:
  if disposition.status == pending:                                    → block
  if disposition.status == reconciled:                                 → none
  if disposition.kind == tolerated_drift and status == accepted:       → block (default)
  if disposition.status == accepted and kind in (follow_up_delta, follow_up_backlog):
    if refs contain a valid owned artefact:                            → warn
    else:                                                              → block
else:  # discovery
  if disposition.status == pending:                                    → warn
  if disposition.status in (reconciled, accepted):                     → none
```

`closure_override` can relax a derived `block` to `warn` or `none`, but requires a rationale and is surfaced in validation output. It cannot escalate (that would be a schema error — escalation should come from the disposition itself, not an override).

### Audit template new fields

```yaml
# New top-level audit fields
mode: conformance # conformance | discovery
delta_ref: DE-079 # owning delta, optional for standalone audits
```

### Implementation notes

- Missing `audit_gate` field on existing deltas defaults to `auto`. No migration needed.
- `complete delta` discovers audits via the existing `ChangeRegistry` by matching `delta_ref` on audit artefacts.
- Disposition `kind` enum values (`aligned`, `spec_patch`, `revision`, `follow_up_delta`, `follow_up_backlog`, `tolerated_drift`) must be defined as constants in the audit metadata module, not inline strings (POL-002).
- Disposition `status` enum values (`reconciled`, `accepted`, `pending`) are audit-specific and distinct from change lifecycle statuses. They should also be defined as constants.

### Removed fields

- `patch_level` and `next_actions` are removed from the schema, template, and metadata module.
- The one existing audit that uses a deprecated field is manually edited as part of this delta.
- No backward compatibility shim — two existing audits do not justify carrying dead schema.
- Audits without the new `disposition` fields remain structurally valid but cannot satisfy a `required` audit gate.

## 7. Design Decisions & Trade-offs

- **DEC-079-001** accepted: Per-finding disposition is inline within each finding. `drift_refs` are inside disposition, plural, and structured `{kind, ref}`. A separate reconciliation block was rejected because it creates exactly the finding/disposition drift that ADR-008 warns against.

- **DEC-079-002** accepted: Audits use the shared change lifecycle statuses from `lifecycle.py`. Canonical audit path is `draft → in-progress → completed`. `in-review` and `complete` are dropped. `pending` and `deferred` remain valid as shared statuses but are exceptional.

- **DEC-079-003** accepted: Delta audit gating uses `audit_gate: auto | required | exempt`. `auto` resolves to `required` when `applies_to.requirements` is non-empty, non-gating otherwise. `exempt` requires `audit_gate_rationale`. Validator warns if `exempt` appears inconsistent with qualifying signals. `complete delta` enforces the resolved gate.

- **DEC-079-004** accepted: `patch_level` and `next_actions` are removed entirely, not deprecated. Per-finding disposition subsumes their role. The one existing audit using a deprecated field is manually edited. No backward compatibility shim.

- **DEC-079-005** accepted: `closure_effect` is derived from `(audit.mode, finding.outcome, disposition.status, disposition.kind)`, never stored. Non-default gate relaxation requires `closure_override` with rationale (not `--force`).

- **DEC-079-006** accepted: Conformance + `tolerated_drift` defaults to `block`, not `warn`. This prevents it from becoming an easy escape hatch in the exact place the contract is trying to be stricter.

- **DEC-079-007** accepted: `status` describes whether this audit has finished routing the finding. `kind` describes what that routing path is. Invalid status×kind combinations are enforced by schema validation.

- **DEC-079-008** accepted: Multiple conformance audits per delta are resolved by union — all completed conformance audits with matching `delta_ref` contribute findings. Finding ID collisions across audits trigger a validation warning. Supersession (later audit replaces earlier for overlapping scope) is deferred until split-scope reruns prove necessary in practice.

- **DEC-079-009** accepted: outcome × disposition.kind validity is enforced. `aligned` outcome requires `aligned` kind only. `drift` and `risk` outcomes require action-bearing kinds. Nonsensical combinations like `outcome: drift + kind: aligned` are schema errors.

- **DEC-079-010** accepted: `audit_gate` is a top-level frontmatter field on deltas, not inside `delta.relationships`. It is a lifecycle/workflow property, not a traceability relationship.

- **DEC-079-011** accepted: Audit-gating pre-check lives in a separate module (`changes/audit_check.py`), following the existing `coverage_check.py` pattern. `complete_delta.py` imports and calls it.

## 9. Rollout & Operational Notes
