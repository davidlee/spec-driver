## 1. Summary & Context

- **Implementation Plan**: [IP-080](./IP-080.md)
- **Change Drivers**: User-reported config naming confusion; `enabled = false` hides card term explanation causing agent disorientation

## 2. Motivation

- `[cards]` config section describes kanban functionality but uses a generic name that collides with the overloaded "card" term in skills/doctrine
- `[docs]` (artefacts_root, plans_root) is kanban-specific state but lives as a peer section, obscuring the relationship
- When `cards.enabled = false`, the "Card means..." glossary/workflow explanation is suppressed — but skills and doctrine still reference the term, leaving agents without context

## 3. Scope & Objectives

- **Primary Outcomes**:
  - Rename `[cards]` → `[kanban]` in DEFAULT_CONFIG, TOML generation, templates, and tests
  - Nest `[docs]` keys under `[kanban]` (i.e. `kanban.artefacts_root`, `kanban.plans_root`)
  - Always render the "Card means..." explanation in agent docs regardless of `kanban.enabled`
  - Backward-compat: accept `[cards]` and top-level `[docs]` in existing workflow.toml files, migrating silently

## 4. Out of Scope

- Renaming the `supekku/scripts/lib/cards/` package (separate delta if desired)
- Changes to card registry or card formatter code

## 5. Approach

- **Touchpoints**: `config.py`, `config_test.py`, `install_test.py`, Jinja2 templates (`glossary.md`, `workflow.md`), memory reference doc, project `workflow.toml`
- **Key Changes**:
  - Config defaults: rename key, merge docs into kanban subsection
  - Config loading: add compat shim that promotes `[cards]` → `[kanban]` and `[docs]` → `[kanban]`
  - Templates: update variable paths, move card explanation outside `enabled` conditional
  - Tests: update all assertions to new key names

## 6. Verification Strategy

- All existing tests pass after key rename (VT)
- New test: loading legacy `[cards]`/`[docs]` TOML produces correct merged config (VT)
- Agent docs render card explanation with `kanban.enabled = false` (VT)
- `just check` green (VT)

## 7. Risks & Mitigations

- **Risk**: Existing installs have `[cards]` in workflow.toml – _Likelihood_: certain – _Impact_: low – _Mitigation_: compat shim in config loader
