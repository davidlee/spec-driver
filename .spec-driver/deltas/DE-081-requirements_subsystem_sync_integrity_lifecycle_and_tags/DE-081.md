## 1. Summary & Context

## 2. Motivation

The requirements subsystem has four related backlog items. One is already fixed; three need work.

- **Stale coverage accumulates** (ISSUE-032, p2): `_apply_coverage_blocks()` unions coverage artifacts but never removes invalidated ones. Registry drifts from spec truth, violating NF-002 (idempotency) and ADR-008 §5 (registry is derived).
- **Unknown coverage statuses** (ISSUE-033, p2): **Already fixed by DE-043.** Current code filters unknown statuses at ingestion and derivation boundaries. Close as resolved.
- **No deprecation path** (ISSUE-029, p3): No lifecycle mechanism to deprecate or supersede a requirement. `blocked` coverage status used as workaround.
- **Inline tags unpopulated** (ISSUE-018, p3): Registry schema has `tags` field but inline requirements have no mechanism to populate it.

## 3. Scope & Objectives

- **O1**: Coverage evidence derived fresh each sync — stale artifacts removed (ISSUE-032 → SPEC-122.NF-002)
- **O2**: Requirement lifecycle includes explicit deprecation/supersession (ISSUE-029 → SPEC-122.FR-002)
- **O3**: Inline requirements support tags (ISSUE-018)

**Dependencies**: None — all work is internal to the requirements subsystem.

## 4. Out of Scope

- Automated enforcement of lifecycle transitions (future policy work)
- `superseded_by` cross-linking (requires relationship model work)
- Registry-level `drift.yaml` integration
- SPEC-150 (new requirements spec — stub, not yet shaped)

## 5. Approach Overview

- **System Touchpoints**: `supekku/scripts/lib/requirements/registry.py`, `supekku/scripts/lib/requirements/lifecycle.py`
- **Key Changes**:
  - Clear and rebuild coverage fields each sync pass (replacement, not accumulation)
  - Guard terminal-status requirements against coverage-derived status overwrite
  - Add `deprecated` and `superseded` status constants
  - Extend `_REQUIREMENT_LINE` regex for `[tag1, tag2]` bracket syntax
  - Add `tag` parameter to `filter()` method
- **Design**: See [DR-081](./DR-081.md) for detailed design decisions and code impact

## 6. Verification Strategy

- **VT**: Unit tests for coverage replacement, idempotency, terminal status guard, tag extraction, tag filtering
- **VA**: Agent verification that `spec-driver validate` passes cleanly after changes
- **Acceptance Criteria**:
  - Re-running sync produces identical registry (NF-002 idempotency restored)
  - Removing coverage block entries causes corresponding evidence to disappear on next sync
  - `deprecated`/`superseded` statuses accepted; not overwritten by coverage derivation
  - Tags extracted from inline `[tag1, tag2]` syntax appear in registry
  - `filter(tag="x")` returns tagged requirements

## 7. Risks & Mitigations

- **Risk**: Clearing all coverage*evidence before rebuild may surprise users who manually edited the registry – \_Likelihood*: low – _Impact_: low – _Mitigation_: Correct per ADR-008 §5; registry is derived, not authoritative.
- **Risk**: Adding statuses to FR-002 may require SPEC-122 revision – _Likelihood_: high – _Impact_: low – _Mitigation_: Bundle spec update in the same delta.
