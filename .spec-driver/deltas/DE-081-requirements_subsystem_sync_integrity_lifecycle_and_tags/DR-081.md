# DR-081 – Requirements subsystem: sync integrity, lifecycle, and tags

## 1. Executive Summary

- **Delta**: [DE-081](./DE-081.md)
- **Status**: draft
- **Last Updated**: 2026-03-09
- **Synopsis**: Fix stale coverage accumulation, add requirement deprecation lifecycle, and enable inline requirement tags.

## 2. Problem & Constraints

### Current Behaviour

**Coverage accumulation (ISSUE-032)**: `_apply_coverage_blocks()` at line 813 of
`registry.py` unions `coverage_evidence` with each sync pass. Once an artefact ID
enters the set, only manual registry edits can remove it. If a coverage block
drops VT-001, the registry still carries it. This violates NF-002 (sync
idempotency) and ADR-008 §5 (registry is derived, not authoritative).

**Unknown coverage statuses (ISSUE-033)**: Already fixed by DE-043. The current
code at lines 717-718 filters to `VALID_COVERAGE_STATUSES` before derivation, and
`_extract_coverage_entries()` warns on unknown statuses. This issue should be
closed as resolved.

**No deprecation path (ISSUE-029)**: `lifecycle.py` defines four statuses:
`pending`, `in-progress`, `active`, `retired`. There is no way to express that a
requirement is deprecated (still documented but no longer enforced) or superseded
(replaced by another requirement). Users work around this with `blocked` coverage
status and notes.

**Tags unpopulated (ISSUE-018)**: `RequirementRecord` has a `tags` field
(line ~78) that is always empty. The extraction regex `_REQUIREMENT_LINE` parses
label, category, and title but has no tag syntax. Standalone requirement files
support tags via frontmatter, but inline requirements cannot specify them.

### Drivers / Inputs

- ISSUE-032 (p2), ISSUE-033 (p2, already fixed), ISSUE-029 (p3), ISSUE-018 (p3)
- ADR-008: registry is derived; coverage_evidence must reflect current source truth
- PROD-008: specs own normative lifecycle; registry projects it
- PROD-009: lifecycle semantics must be explicit with provenance

### Constraints / Guardrails

- ADR-008 §5: registry is derived projection — replacement semantics align
- ADR-009: new filtering (by tag, by deprecated) must follow `filter(...)` convention
- POL-002: new statuses must be named constants, not magic strings
- SPEC-122.NF-002: sync idempotency must be restored

### Out of Scope

- Automated lifecycle transition enforcement
- Coverage provenance tracking (normative vs observed split per ADR-008 §2-3)
- `superseded_by` cross-linking in the registry (future: requires relationship model)

## 3. Architecture Intent

### Target Outcomes

- **O1** (ISSUE-032 → NF-002): Coverage evidence derived fresh each sync from current coverage blocks. Re-running sync produces identical registry.
- **O2** (ISSUE-029 → FR-002): `deprecated` and `superseded` are valid requirement lifecycle statuses with clear semantics.
- **O3** (ISSUE-018): Inline requirements support tags via bracket syntax; tags populate the registry.

### State Transitions / Lifecycle Impact

Current requirement statuses: `pending → in-progress → active → retired`

After DE-081:

```
pending → in-progress → active → retired
                    ↘            ↘
                     deprecated   superseded
```

- `deprecated`: requirement is documented but no longer enforced. May still have passing tests. Distinct from `retired` (which implies clean removal). A requirement can be deprecated from `in-progress` (never finished, no longer needed) or from `active` (was live, now soft-removed).
- `superseded`: requirement replaced by another. Should carry a note or reference identifying the successor. Terminal — a superseded requirement does not return to active.

Both are terminal states. Neither participates in coverage-driven status derivation — they are normative lifecycle claims owned by the spec, not derived from verification evidence (per ADR-008 §1).

### DEC-081-01: Replace coverage_evidence per sync, not per source

**Current**: `coverage_evidence = sorted(set(record.coverage_evidence) | artefacts)`
**Target**: Clear all coverage fields before rebuilding; then set from current sources.

The `_apply_coverage_blocks()` method scans all coverage sources (specs, deltas,
plans, audits) in a single pass and builds `coverage_map` from scratch. For
requirements that appear in `coverage_map`, replacement is straightforward.

However, there is a subtlety: if a requirement was previously covered but its
coverage block is removed entirely, that requirement won't appear in
`coverage_map` at all. Under the current union-only approach, stale evidence
persists silently.

**Solution**: At the start of `_apply_coverage_blocks()`, clear `coverage_evidence`
and `coverage_entries` for all records. Then rebuild from `coverage_map`. This
ensures the registry is a pure derivation of current sources.

```python
# Clear all coverage fields — will be rebuilt from current sources
for record in self.records.values():
  record.coverage_evidence = []
  record.coverage_entries = []

# ... then rebuild from coverage_map as before, but with assignment not union:
record.coverage_evidence = sorted(artefacts)
```

Status derivation for requirements that lose all coverage: `_compute_status_from_coverage`
returns `None` for empty entries, so the requirement's status is untouched.
Terminal-status requirements (deprecated/superseded) are also unaffected (guarded
separately). A requirement that was `active` via coverage and then loses all
coverage will remain `active` — this is correct, as the loss of evidence is drift
to surface, not an automatic status downgrade (per ADR-008 §3).

**Risk**: Manual registry edits to coverage fields are overwritten. Correct per
ADR-008 §5 (registry is derived). If coverage_evidence needs data not in any
coverage block, the right fix is adding the coverage block, not editing the
registry.

### DEC-081-02: deprecated/superseded are requirement lifecycle statuses

These are normative claims about a requirement's standing, not observations from
verification. Per ADR-008 §1, specs own normative lifecycle truth. A deprecated
requirement might still have `verified` coverage — the deprecation is a governance
decision, not a test result.

Implementation:

- Add constants to `lifecycle.py`
- Add `TERMINAL_STATUSES = {STATUS_RETIRED, STATUS_DEPRECATED, STATUS_SUPERSEDED}`
- Guard in `_apply_coverage_blocks()`: skip status derivation when current status is terminal
- `VerificationCoverageValidator` unchanged — these are not coverage statuses

### DEC-081-03: Inline tag syntax using square brackets

**Chosen syntax**: `- **FR-001**(category)[tag1, tag2]: Title`

Square brackets after the optional category parentheses. This:

- Does not conflict with existing markdown syntax in this context
- Extends the existing pattern naturally (label → category → tags → title)
- Is low-friction for authors
- Parses unambiguously (brackets cannot appear in category or title delimiter)

Updated regex captures an optional `[...]` group. Tags are split on comma,
stripped, and stored as a list.

Example:

```markdown
- **FR-010**(api)[security, auth]: Validate all tokens before dispatch
- **NF-005**[performance]: Response time under 200ms
- **FR-011**: Plain requirement with no tags
```

### DEC-081-04: ISSUE-033 already fixed — close as resolved

DE-043 (commit `62cfcd5`) added the `VALID_COVERAGE_STATUSES` filter at the
derivation boundary and warnings at the ingestion boundary. The current code
is correct. ISSUE-033 should be closed with a reference to DE-043.

## 7. Design Decisions & Trade-offs

- **DEC-081-01**: Replace coverage_evidence per sync. Simpler, correct per ADR-008. Trade-off: manual registry edits overwritten — acceptable because registry is derived.
- **DEC-081-02**: Lifecycle statuses, not coverage statuses. Follows ADR-008 §1 normative ownership. Trade-off: `deprecated` requirements with passing tests won't auto-derive status — correct, deprecation is a governance choice.
- **DEC-081-03**: Square bracket tag syntax. Low friction, unambiguous. Trade-off: another regex extension — but the pattern is already complex and well-tested.
- **DEC-081-04**: Close ISSUE-033. No code change needed. Avoids re-fixing solved problem.
