## 1. Summary & Context

- **Product Spec(s)**:
  - `PROD-011` - workflow guidance surface for audit, revision, and closure sequencing
  - `PROD-002` - delta bundle expectations and handoff surfaces for follow-up design/planning work
- **Technical Spec(s)**:
  - `SPEC-151` - skills library implementation surface; still too thin for the remaining authorship-skill work
- **Implementation Plan**: [IP-083](./IP-083.md) - phase planning now starts at executable work because `DR-083` has already settled the design boundary
- **Change Drivers**:
  - `DE-055` closure handoff: routing and DR-loop guidance landed, but deeper audit-to-spec and authorship-skill work was intentionally split out
  - `DE-079` runtime audit reconciliation contract: introduces the machine-checkable audit flow that authorship skills must feed cleanly
  - `ADR-004`, `ADR-005`, and `ADR-008`
  - user priority: make the audit loop strongly biased toward flowing back into authoritative specifications, via existing specs/revisions where possible and new specs when needed

## 2. Motivation

- `DE-055` improved routing, DR authoring, and closure guidance, but it deliberately stopped short of tuning the skill surfaces that turn audit findings into authoritative spec outcomes.
- `DE-079` implements the runtime audit contract, but it does not yet solve the authorship problem: when audit findings reveal drift or missing authority, agents still need clearer guidance for whether to patch an existing spec, create/update a revision, or introduce a new spec.
- `shape-revision` is still minimal, and there is no equally strong spec-authoring loop for post-audit reconciliation work.
- `PROD-001` already owns greenfield spec creation, so the remaining gap is not "how to create a spec at all" but how audit findings decide between patching current authority, escalating into revision, or using revision work to justify a new spec boundary.
- The target state is that audit findings default toward authoritative spec reconciliation rather than lingering as prose, loose follow-up notes, or ambiguous “someone should update docs later” outcomes.

## 3. Scope & Objectives

- **Primary Outcomes**:
  - Define the skill-level decision path from audit findings to authoritative spec outcomes:
    existing spec patch, revision, or revision-led new spec creation.
  - Strengthen revision/spec authorship skills so post-audit reconciliation is concrete, doctrine-aware, and section-by-section rather than vague.
  - Make audit guidance and authorship guidance reinforce each other so conformance work naturally flows back into specs before closure.
- **Operational Constraints**:
  - Do not duplicate `DE-079` runtime schema, validation, or `complete delta` gate work.
  - Do not reopen the routing/boot doctrine already settled in `DE-055` unless authorship work reveals a genuine conflict.
  - Preserve `ADR-008`: observed evidence triggers explicit reconciliation; it does not silently replace spec truth.
- **Dependencies**:
  - `DE-055` for settled routing/DR-loop doctrine
  - `DE-079` for the runtime audit contract that this skill work must complement
  - Possible follow-up spec/revision governance clarification if current PROD/SPEC surfaces remain too thin after design review

## 4. Out of Scope

- Audit metadata/schema fields, validation rules, and `complete delta` gating mechanics in `DE-079`
- Boot-time routing and `using-spec-driver` doctrine from `DE-055`
- Backlog lifecycle semantics blocked on `ISSUE-009`
- Full spec corpus reconciliation beyond the specific authorship/workflow surfaces needed for this change

## 5. Approach Overview

- **System Touchpoints**:
  - `supekku/skills/audit-change/SKILL.md`
  - `supekku/skills/shape-revision/SKILL.md`
  - `supekku/skills/draft-design-revision/SKILL.md`
  - relevant spec/revision creation or authoring surfaces identified during `DR-083`
  - `PROD-011`, `PROD-002`, and `SPEC-151`
- **Key Changes**:
  - Clarify the preferred post-audit reconciliation path: existing spec first, revision when authority must move/change, and new spec only as a revision-led fallback when current spec topology is insufficient.
  - Strengthen revision/spec authorship skills with the same kind of triage, doctrine pass, and concrete section-by-section shaping now present in DR authoring.
  - Reuse the existing `spec-driver` plus `PROD-001` create-spec path for the rare justified new-spec case instead of adding a dedicated spec-authoring skill in this delta.
- **Migration / Rollout Notes**:
  - Design first; no implementation should start here until `DR-083` settles the authorship-skill boundary.
  - Expect skill wording and possibly generated guidance changes, but keep packaged-skill uniformity intact.

## 7. Risks & Mitigations

- **Risk**: This delta duplicates `DE-079` audit-runtime work. – _Likelihood_: medium – _Impact_: high – _Mitigation_: keep `DE-083` scoped to authorship/workflow guidance and reference `DE-079` for contract mechanics.
- **Risk**: Skill guidance over-corrects into mandatory new-spec creation where existing specs or revisions would suffice. – _Likelihood_: medium – _Impact_: medium – _Mitigation_: bias toward reconciling into existing authority first and make “new spec” an explicit, justified branch.
- **Risk**: Spec/revision authorship guidance becomes another competing doctrine layer. – _Likelihood_: low – _Impact_: high – _Mitigation_: keep governance in ADRs/specs, concepts in memories, and procedures in skills per `ADR-005`.
