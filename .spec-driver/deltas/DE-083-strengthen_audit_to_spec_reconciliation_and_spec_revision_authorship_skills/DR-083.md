# DR-083 – Strengthen audit-to-spec reconciliation and spec/revision authorship skills

## 1. Executive Summary
- **Delta**: [DE-083](./DE-083.md)
- **Status**: draft
- **Owners / Team**: david
- **Last Updated**: 2026-03-09
- **Synopsis**: Strengthen the skill-level path from audit findings back into authoritative specs so reconciliation defaults to existing spec patches or revisions, introduces new specs only when justified, and gives revision/spec authorship the same concrete guidance quality already applied to DR authoring.

## 2. Problem & Constraints
- **Current Behaviour**:
  - `DE-079` is implementing the runtime audit contract, but it does not by itself make post-audit authorship decisions legible to agents.
  - `audit-change` can now disposition findings toward `spec_patch`, `revision`, or follow-up work, but it does not yet teach a strong preference order for how authoritative specs should be reconciled.
  - `shape-revision` is still a minimal runsheet and does not yet provide the richer triage, doctrine pass, and section-by-section guidance now present in `draft-design-revision`.
  - There is no dedicated spec-authoring skill, so the branch from "audit found missing or misplaced authority" to "patch an existing spec, revise ownership, or create a new spec" remains underspecified.
- **Drivers / Inputs**:
  - `ADR-004`: implementation should flow through audit/contracts into revision/spec reconciliation before closure.
  - `ADR-005`: memories own concepts; skills own procedures.
  - `ADR-008`: observed evidence prompts explicit reconciliation rather than silently replacing spec truth.
  - `DE-055` and `DR-055`: settled the workflow doctrine, improved DR authoring, and intentionally split remaining authorship work into this delta.
  - `DE-079` and `DR-079`: settle the runtime audit contract that this authorship guidance must complement.
- **Constraints / Guardrails**:
  - Do not duplicate `DE-079` schema, validation, or `complete delta` gate work.
  - Do not reopen `DE-055` routing/boot questions unless this work finds a direct conflict.
  - Bias toward existing authority first; new-spec creation must be explicit and justified.
  - Keep governance in ADRs/specs, concepts in memories, and procedures in skills.
- **Out of Scope**:
  - Audit metadata/runtime gate implementation in `DE-079`
  - Broader workflow-routing work from `DE-055`
  - Backlog lifecycle semantics blocked on `ISSUE-009`
  - Full spec corpus reconciliation unrelated to the authorship path being designed here

## 3. Architecture Intent
- **Target Outcomes**:
  - Agents handling audit findings have a clear default path back into authoritative specs.
  - Existing specs are patched or revised by default when they remain the correct authority surface.
  - New spec creation is reserved for cases where current authority boundaries are genuinely insufficient.
  - Revision/spec authorship skills become concrete enough to turn audit findings into durable spec work instead of narrative follow-up.
- **Guiding Principles**:
  - Existing authority first.
  - Revision when authority or requirement ownership changes.
  - New spec only when current spec topology cannot carry the reconciled truth cleanly.
  - Reuse the strongest existing authorship patterns instead of introducing a new skill surface prematurely.
- **State Transitions / Lifecycle Impact**:
  - This delta should not introduce new lifecycle enums.
  - It should clarify the post-audit decision path:
    `audit finding -> spec patch | revision | justified new spec -> updated authoritative record`

## 6. Supporting Context
- **Research**:
  - `mem.concept.spec-driver.spec` says specs are the durable normative record and become canonical after explicit reconciliation.
  - `mem.concept.spec-driver.revision` says revisions are typically the post-audit/post-implementation reconciliation artefact in canonical delta-first flow.
  - `DR-055` already established that audit findings should end in explicit reconciliation rather than prose-only closure narratives.
- **Hypotheses**:
  - Much of the remaining ambiguity can be removed by strengthening existing skill loops rather than inventing a large new meta-skill.
  - The strongest missing behavior is not runtime gating; it is authorship guidance for where the reconciled truth should live.
- **Related Deltas / Specs**:
  - `DE-055` / `DR-055` for settled workflow doctrine
  - `DE-079` / `DR-079` for runtime audit contract implementation
  - `PROD-011`, `PROD-002`, and `SPEC-151`

## 7. Design Decisions & Trade-offs
- `DEC-083-001` accepted: bias post-audit reconciliation toward existing authoritative specs first; use revisions when authority changes; treat new-spec creation as an explicit fallback.
- `DEC-083-002` accepted: keep runtime audit contract work in `DE-079` and use `DE-083` for the skill/workflow layer that teaches authors how to reconcile from audit output.
- `DEC-083-003` proposed: strengthen existing authorship skills first before introducing a dedicated spec-authoring skill.
- Trade-off: a dedicated spec-authoring skill may ultimately be clearer, but starting there risks premature abstraction and another overlapping guidance surface.
