# DR-083 – Strengthen audit-to-spec reconciliation and spec/revision authorship skills

## 1. Executive Summary
- **Delta**: [DE-083](./DE-083.md)
- **Status**: draft
- **Owners / Team**: david
- **Last Updated**: 2026-03-09
- **Synopsis**: Strengthen the skill-level path from audit findings back into authoritative specs so reconciliation defaults to existing spec patches, escalates to revision when authority changes, and treats new-spec creation as a rare revision-led outcome rather than a parallel default path.

## 2. Problem & Constraints
- **Current Behaviour**:
  - `DE-079` is implementing the runtime audit contract, but it does not by itself make post-audit authorship decisions legible to agents.
  - `audit-change` can now disposition findings toward `spec_patch`, `revision`, or follow-up work, but it does not yet teach a strong preference order for how authoritative specs should be reconciled.
  - `shape-revision` is still a minimal runsheet and does not yet provide the richer triage, doctrine pass, and section-by-section guidance now present in `draft-design-revision`.
  - There is no dedicated spec-authoring skill, but `PROD-001` and `spec-driver create spec` already cover greenfield spec creation. The real gap is deciding when audit reconciliation should stay inside existing authority, when it must move into a revision, and when revision work genuinely needs a new spec boundary.
- **Drivers / Inputs**:
  - `ADR-004`: implementation should flow through audit/contracts into revision/spec reconciliation before closure.
  - `ADR-005`: memories own concepts; skills own procedures.
  - `ADR-008`: observed evidence prompts explicit reconciliation rather than silently replacing spec truth.
  - `DE-055` and `DR-055`: settled the workflow doctrine, improved DR authoring, and intentionally split remaining authorship work into this delta.
  - `DE-079` and `DR-079`: settle the runtime audit contract that this authorship guidance must complement.
- **Constraints / Guardrails**:
  - Do not duplicate `DE-079` schema, validation, or `complete delta` gate work.
  - Do not reopen `DE-055` routing/boot questions unless this work finds a direct conflict.
  - Bias toward existing authority first; new-spec creation must be explicit and justified.
  - Keep governance in ADRs/specs, concepts in memories, and procedures in skills.
  - Respect `DE-079` disposition kinds as fixed design input for this delta; DE-083 should teach authors how to use them rather than inventing new audit-runtime categories.
- **Out of Scope**:
  - Audit metadata/runtime gate implementation in `DE-079`
  - Broader workflow-routing work from `DE-055`
  - Backlog lifecycle semantics blocked on `ISSUE-009`
  - Full spec corpus reconciliation unrelated to the authorship path being designed here

## 3. Foundational Triage
- **Questions closed before planning**:
  - DE-083 can stay inside strengthened existing skills; a separate spec-authoring skill would duplicate `spec-driver` and `PROD-001` before proving a repeatable gap.
  - The top-level authorship choice must align with `DE-079` disposition kinds, which means "new spec" cannot become a new peer audit disposition in this delta.
  - Governance follow-through should focus first on the workflow surface (`PROD-011`) and the skills subsystem boundary (`SPEC-151`), with bundle-handoff and create-spec specs revised only if implementation proves they need explicit new wording.
- **Key risks**:
  - Treating new-spec creation as a peer branch would reopen `DE-079` and weaken the machine-checkable audit contract.
  - Adding a dedicated spec-authoring skill now would violate `POL-001` by creating a competing authoring surface before exhausting reuse of `shape-revision`, `spec-driver`, and `PROD-001`.
  - Leaving `SPEC-151` as a stub would make the authorship-skill boundary invisible in the technical corpus even if the skills themselves improve.
- **Assumptions carried**:
  - `audit-change` remains the closure-grade audit procedure.
  - `shape-revision` remains the right home for authority/requirement movement and lineage capture.
  - `spec-driver create spec` remains the canonical creation path when a new spec boundary is justified.
- **Authorities constraining this design**:
  - `ADR-004` and `mem.pattern.spec-driver.core-loop` for audit -> revision/spec reconciliation -> close
  - `ADR-005` for skill-vs-memory-vs-governance ownership
  - `ADR-008` for explicit reconciliation rather than silent overwrite
  - `ADR-003` for avoiding competing truths and making spec-boundary changes explicit

## 4. Architecture Intent
- **Target Outcomes**:
  - Agents handling audit findings have a clear default path back into authoritative specs.
  - Existing specs are patched or revised by default when they remain the correct authority surface.
  - New spec creation is reserved for cases where revision work proves the current authority boundaries are genuinely insufficient.
  - Revision/spec authorship skills become concrete enough to turn audit findings into durable spec work instead of narrative follow-up.
- **Guiding Principles**:
  - Existing authority first.
  - Revision when authority or requirement ownership changes.
  - New spec only when current spec topology cannot carry the reconciled truth cleanly, and treat that as a revision-led topology change rather than a separate audit disposition.
  - Reuse the strongest existing authorship patterns instead of introducing a new skill surface prematurely.
- **State Transitions / Lifecycle Impact**:
  - This delta should not introduce new lifecycle enums.
  - It should clarify the post-audit decision path:
    `audit finding -> existing spec patch | revision -> optional justified new spec -> updated authoritative record`

### Branch Criteria

| Branch | Use when | Required output |
| --- | --- | --- |
| Existing spec patch | The current owning spec is still the right authority surface; the finding is a corrective update to wording, coverage, examples, or constraints inside that same scope; no requirement/responsibility ownership moves. | Patch the existing spec now, keep audit disposition on `spec_patch`, and reconcile coverage before closure. |
| Revision | Authority, requirement ownership, or cross-spec lineage changes; multiple specs must move together; or the finding cannot be expressed as a simple patch without obscuring why the truth moved. | Create or update a revision, capture source/destination ownership and rationale, then apply the corresponding spec updates. |
| Revision-led new spec | Revision analysis shows no existing spec can own the reconciled truth cleanly without overlap, competing truths, or topology distortion; a distinct product, assembly, or unit boundary has emerged. | Keep the audit on the `revision` path, create the new spec through `spec-driver`, and update the affected legacy specs/revision links so authority is explicit. |

## 7. Supporting Context
- **Research**:
  - `mem.concept.spec-driver.spec` says specs are the durable normative record and become canonical after explicit reconciliation.
  - `mem.concept.spec-driver.revision` says revisions are typically the post-audit/post-implementation reconciliation artefact in canonical delta-first flow.
  - `mem.concept.spec-driver.audit` and `DR-079` together make `spec_patch` and `revision` the audit-facing authorship dispositions; DE-083 should refine the author behavior behind those kinds rather than add new ones.
  - `PROD-001` already owns greenfield spec creation workflow, while `PROD-001` itself records a separate gap for refining existing specs; that supports strengthening revision/spec-patch guidance before inventing another create-spec skill.
  - `DR-055` already established that audit findings should end in explicit reconciliation rather than prose-only closure narratives.
- **Hypotheses**:
  - Much of the remaining ambiguity can be removed by strengthening existing skill loops rather than inventing a large new meta-skill.
  - The strongest missing behavior is not runtime gating; it is authorship guidance for where the reconciled truth should live and when topology changes require revision-led new authority.
- **Related Deltas / Specs**:
  - `DE-055` / `DR-055` for settled workflow doctrine
  - `DE-079` / `DR-079` for runtime audit contract implementation
  - `PROD-011`, `PROD-002`, `PROD-001`, and `SPEC-151`

## 8. Design Decisions & Trade-offs
- `DEC-083-001` accepted: bias post-audit reconciliation toward existing authoritative specs first; use revisions when authority changes; treat new-spec creation as an explicit fallback.
- `DEC-083-002` accepted: keep runtime audit contract work in `DE-079` and use `DE-083` for the skill/workflow layer that teaches authors how to reconcile from audit output.
- `DEC-083-003` accepted: strengthen existing authorship skills first and reuse `spec-driver` plus `PROD-001` for the exceptional new-spec case.
- `DEC-083-004` accepted: new-spec creation is nested inside the revision branch because authority-boundary changes need lineage and should not reopen the `DE-079` audit contract.
- `DEC-083-005` accepted: revise workflow doctrine and subsystem-boundary specs directly, and treat delta-bundle/create-spec specs as collaborator surfaces unless implementation proves a direct wording gap.
- Trade-off: a dedicated spec-authoring skill may ultimately be clearer, but starting there would duplicate existing creation guidance before proving repeated failure of the narrower, more coherent routing.

## 9. Readiness Outcome
- `DR-083` is now shaped enough to start phase planning because the three blocking design questions are closed:
  - No dedicated spec-authoring skill is required for this delta.
  - Branch criteria are explicit and aligned with `DE-079`.
  - Direct vs collaborator governance surfaces are identified.
- Residual implementation work is now tactical:
  - strengthen `audit-change`
  - strengthen `shape-revision`
  - revise `PROD-011` and `SPEC-151`
  - decide during implementation whether `PROD-002` and `PROD-001` need direct text changes
