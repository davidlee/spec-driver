## 1. Summary & Context

- **Product Specs**: PROD-010 (CLI UX), PROD-002 (Delta Creation Workflow), PROD-004 (Frontmatter Metadata Validation)
- **Implementation Plan**: [IP-085](./IP-085.md) ŌĆō not started
- **Change Drivers**: IMPR-006 (formatter migration), IMPR-011 (TUI relational display)

## 2. Motivation

Deltas already support `context_inputs` and `relations` in frontmatter, plus a structured `delta.relationships` block. But the CLI doesn't expose these well:

- `list deltas` has `--implements` for requirements but no general relation/backlog filtering
- `list backlog` has no reverse query ("which deltas reference this issue?")
- `show delta` renders relations but there's no `--relations` column in list output
- Skills that guide delta creation from backlog items don't consistently instruct agents to populate `context_inputs` and `relations`
- No generic relation query pattern ŌĆö each command reinvents filtering

Target state: relation metadata is first-class in CLI discovery, filtering, and display, using a generic pattern reusable across artifact types and in the TUI.

## 3. Scope & Objectives

- **Primary Outcomes**:
  1. Pure generic query functions in `relations/query.py` with `RelationQueryable` Protocol ŌĆö `collect_references()`, `find_related_to()`, `find_by_relation()`
  2. CLI `list` commands gain `--related-to`, `--relation TYPE:TARGET`, and `--refs` flags (deltas + specs initially)
  3. CLI `list` output optionally includes refs column (count in table, `type:target` in TSV)
  4. `Spec` model gains `.relations` property for generic query participation
  5. `create delta --from-backlog` auto-populates `context_inputs` and `relations`
  6. Skills updated to guide consistent metadata population when creating deltas from backlog items
  7. Named constants for relation types and reference sources (POL-002)
- **Operational Constraints**: `--implements` keeps its current narrow semantics (applies_to.requirements only). `--informed-by` stays as-is. No breaking changes.
- **Supported artifact types**: ChangeArtifact (full), Spec (after `.relations` property). DecisionRecord and BacklogItem are out of scope (incompatible models).
- **Dependencies**: None blocking

## 4. Out of Scope

- Full IMPR-006 formatter migration (use generic table for new columns only where already migrated)
- TUI implementation (design for reuse, but TUI integration is IMPR-011's scope)
- TUI cross-registry scored search orchestrator (designed for, built under IMPR-011)
- New frontmatter fields or schema changes
- Relation graph visualisation
- Normalising `spec.informed_by` into standard `relations` (future delta)
- `StrEnum` upgrade for relation types (future)
- DecisionRecord or BacklogItem model normalisation (incompatible shapes)

## 5. Approach Overview

- **System Touchpoints**: registries (changes, specs, backlog), CLI list/show/find, formatters, skills
- **Key Changes**:
  1. **Query layer**: Pure generic functions in `relations/query.py` with `RelationQueryable` Protocol ŌĆö operating on any artifact with `.relations`. Registries call them; they don't own them.
  2. **Constants**: `RELATION_TYPES` and `REFERENCE_SOURCES` in `core/relation_types.py` (POL-002)
  3. **Spec model**: Add `.relations` property for generic query participation
  4. **CLI list commands**: `--related-to ID`, `--relation TYPE:TARGET`, `--refs` column flag
  5. **CLI create**: `--from-backlog` auto-populates `context_inputs` and `relations`
  6. **Skills**: Update `scope-delta` to guide consistent metadata from backlog provenance

## 6. Verification Strategy

- **VT**: Automated tests for generic relation query methods, CLI flag parsing, output format
- **VA**: Agent verification that skills produce correct metadata when creating deltas from backlog items
- **Acceptance Criteria**:
  - `list deltas --related-to IMPR-006` returns deltas with IMPR-006 in relations or context_inputs
  - `list specs --related-to ADR-001` finds specs via both `.relations` and `.informed_by`
  - TSV/table output includes refs column when requested via `--refs`
  - Existing `--implements` / `--informed-by` behaviour unchanged
  - `create delta --from-backlog ISSUE-XXX` produces delta with `context_inputs` and `relations` referencing the source item

## 9. Implementation Notes

- Design revision: [DR-085](./DR-085.md) ŌĆö includes full adversarial review disposition
- `RelationQueryable` Protocol documents the contract; `ReferenceHit` with `.detail` field carries slot-specific provenance
- `collect_references()` is the foundational primitive; TUI scorer will call it directly
- `--implements` keeps narrow semantics (applies_to.requirements only) ŌĆö not reimplemented via generic query
- `collect_references()` also searches `.informed_by` on Spec, resolving the two-relation-system UX gap
- `--relation TYPE:TARGET` splits on first colon; unrecognised type emits warning
- `--refs` (not `--relations`) for column flag, since display spans all reference slots
- Existing precedent: `ChangeRegistry.find_by_implements()`, `SpecRegistry.find_by_informed_by()`
- `list_changes.py` already does hybrid `applies_to` + `relations` search ŌĆö generalise this pattern
- `format_list_table()` in `table_utils.py` is the target generic formatter
