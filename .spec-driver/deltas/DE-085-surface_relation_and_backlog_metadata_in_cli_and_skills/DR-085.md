# DR-085 – Surface relation and backlog metadata in CLI and skills

## 1. Executive Summary

- **Delta**: [DE-085](./DE-085.md)
- **Status**: draft
- **Last Updated**: 2026-03-09
- **Synopsis**: Add a generic relation query layer and surface relation/backlog metadata uniformly across CLI list/show/find commands, designed for reuse in TUI cross-registry search.

## 2. Problem & Constraints

### Current Behaviour

Relation-based discovery is fragmented:

- `list deltas --implements REQ-ID` searches only `applies_to.requirements` via `ChangeRegistry.find_by_implements()`
- `list specs --informed-by ADR-ID` searches `spec.informed_by` via `SpecRegistry.find_by_informed_by()`
- `list changes --applies-to ID` (legacy script) does a hybrid search of `applies_to.requirements` + `relations` targets — the most complete, but not exposed in the main CLI
- No reverse query from backlog items ("which deltas reference IMPR-006?")
- No way to filter by relation type ("show me all `depends_on` edges")
- `context_inputs` are never searched despite being a meaningful forward reference
- Relation types are inline string literals in metadata definition (POL-002 concern)
- `Spec` model has no `.relations` property despite specs having relations in frontmatter
- `create delta --from-backlog` populates title and requirements but not `context_inputs` or `relations` back to the source backlog item

### Drivers / Inputs

- IMPR-006: formatter migration to `format_list_table` — new relation columns should use the generic pattern
- IMPR-011: TUI relational display — needs reusable scored search across registries

### Constraints / Guardrails

| Source  | Constraint                                                                                                                            |
| ------- | ------------------------------------------------------------------------------------------------------------------------------------- |
| ADR-002 | No backlinks in frontmatter. All reverse queries computed at runtime from forward references.                                         |
| ADR-009 | Standard registry API: `find()`, `collect()`, `iter()`, `filter()`. Relation query is a domain-specific extension, not a replacement. |
| POL-001 | One generic implementation, not per-registry reimplementations.                                                                       |
| POL-002 | Relation types and reference sources must be named constants, not inline literals.                                                    |

### Out of Scope

- Full IMPR-006 formatter migration
- TUI implementation (design for reuse only)
- Cross-registry scored search orchestrator (designed for, built under IMPR-011)
- New frontmatter fields or schema changes
- Relation graph visualisation
- Normalising `spec.informed_by` into standard `relations` (future delta)
- `StrEnum` upgrade for relation types (future)
- DecisionRecord model normalisation (incompatible shape — see §5.8)

## 3. Architecture Intent

### Target Outcomes

1. A single set of pure query functions that operate on any artifact conforming to `RelationQueryable`
2. CLI `list` commands gain `--related-to ID` and `--relation TYPE:TARGET` flags
3. Existing `--implements` preserved with its current narrow semantics; `--informed-by` unchanged
4. Reference metadata visible in list output: count in table, `type:target` pairs in TSV
5. Skills guide agents to populate `context_inputs` and `relations` when creating deltas from backlog items
6. `create delta --from-backlog` auto-populates `context_inputs` and `relations`

### Guiding Principles

- **Pure functions over registry methods**: The query logic lives in `relations/query.py` as pure functions. Registries call them; they don't own them.
- **Two-layer design**: Low-level boolean/filter functions serve CLI. High-level scored search (future) serves TUI. Both share the same matching primitives.
- **Union of forward references**: `--related-to` searches all reference slots on an artifact. The match result carries provenance (which slot matched) so callers can distinguish.
- **Honest scope**: The query layer is generic in interface but richest for `ChangeArtifact` (which has `.relations`, `.applies_to`, `.context_inputs`). Other artifact types participate to the extent they expose `.relations`. This asymmetry is documented, not hidden.

### Supported Artifact Types

| Type                                    |   `.relations`   | `.applies_to` | `.context_inputs` |            `.informed_by`            | Participation                             |
| --------------------------------------- | :--------------: | :-----------: | :---------------: | :----------------------------------: | ----------------------------------------- |
| `ChangeArtifact` (delta/revision/audit) |        ✓         |       ✓       |         ✓         |                  —                   | Full                                      |
| `Spec`                                  | ✓ (new property) |       —       |         —         | ✓ (searched by `collect_references`) | Good                                      |
| `BacklogItem`                           |        —         |       —       |         —         |                  —                   | Not participating (no relations in model) |
| `DecisionRecord`                        |        —         |       —       |         —         |                  —                   | Not participating (incompatible shape)    |

`BacklogItem` and `DecisionRecord` model normalisation is out of scope. They can be brought in by future deltas that add `.relations` to their models.

### Layered Architecture

```
┌─────────────────────────────────────┐
│  TUI cross-registry search (future) │  ← scored, multi-registry
├─────────────────────────────────────┤
│  CLI --related-to / --relation      │  ← single-registry, boolean
├─────────────────────────────────────┤
│  relations/query.py                 │  ← pure functions
│  - collect_references(artifact)     │     operates on RelationQueryable
│  - matches_related_to(artifact, id) │
│  - matches_relation(artifact, ...)  │
├─────────────────────────────────────┤
│  Artifact models (.relations,       │
│    .applies_to, .context_inputs,    │
│    .informed_by)                    │  ← data layer
└─────────────────────────────────────┘
```

## 5. Design Detail

### 5.1 Named Constants (POL-002)

```python
# supekku/scripts/lib/core/relation_types.py

RELATION_TYPES: frozenset[str] = frozenset({
    "implements",
    "verifies",
    "depends_on",
    "collaborates_with",
    "provides_for",
    "supersedes",
    "superseded_by",
    "relates_to",
    "blocks",
    "blocked_by",
    "decomposes",
    "tracked_by",
})

REFERENCE_SOURCES: frozenset[str] = frozenset({
    "relation",
    "applies_to",
    "context_input",
    "informed_by",
})
```

`base.py` imports `RELATION_TYPES` instead of the inline list.

### 5.2 Query Functions

```python
# supekku/scripts/lib/relations/query.py

from typing import Any, Protocol, runtime_checkable

@runtime_checkable
class RelationQueryable(Protocol):
    """Artifact that exposes relations as list[dict] with type/target keys.

    Additional reference slots (.applies_to, .context_inputs, .informed_by)
    are searched if present but not required by the protocol.
    """
    relations: list[dict[str, Any]]


@dataclass(frozen=True)
class ReferenceHit:
    """A matched reference with provenance.

    Attributes:
        target: The referenced artifact ID.
        source: Which slot the reference came from (see REFERENCE_SOURCES).
        detail: Slot-specific qualifier. Meaning varies by source:
            - source="relation": the relation type (e.g. "implements", "depends_on")
            - source="applies_to": the applies_to key (e.g. "requirement", "spec", "prod")
            - source="context_input": the context input type (e.g. "issue", "research")
            - source="informed_by": always "adr"
    """
    target: str
    source: str    # from REFERENCE_SOURCES
    detail: str    # slot-specific qualifier


def collect_references(artifact: Any) -> list[ReferenceHit]:
    """Collect all forward references from an artifact.

    Searches (when present on the artifact):
    - .relations (list[dict] with type/target keys)
    - .applies_to (dict with specs/requirements/prod keys)
    - .context_inputs (list[dict] with type/id keys)
    - .informed_by (list[str] — spec-specific)

    Returns ReferenceHit with provenance for each reference found.
    """

def matches_related_to(artifact: Any, target_id: str) -> bool:
    """True if artifact references target_id in any slot.

    Case-insensitive on target IDs.
    """

def matches_relation(
    artifact: Any,
    *,
    relation_type: str | None = None,
    target: str | None = None,
) -> bool:
    """True if artifact has a .relations entry matching type and/or target.

    Searches only .relations, not other reference slots.
    """

def find_related_to(
    artifacts: Iterable[Any],
    target_id: str,
) -> list[Any]:
    """Filter artifacts that reference target_id in any slot."""

def find_by_relation(
    artifacts: Iterable[Any],
    *,
    relation_type: str | None = None,
    target: str | None = None,
) -> list[Any]:
    """Filter artifacts by .relations type and/or target."""
```

Key design choices:

- **`RelationQueryable` Protocol** (S1 accepted): A minimal `runtime_checkable` Protocol documenting the contract. Not an ABC — ADR-009 defers that for registries, and this is for query function inputs.
- **`collect_references()` is the primitive**: `matches_related_to()` and `matches_relation()` are convenience wrappers. The TUI scorer will call `collect_references()` directly.
- **`detail` not `relation_type`** (W3 accepted): The field carries slot-specific meaning with a documented mapping, avoiding the overloaded semantics of a field named `relation_type`.
- **Spec `.informed_by` searched** (W2 accepted): `collect_references()` treats `.informed_by` as a reference slot, so `--related-to ADR-001` on specs finds both `.relations` entries and `.informed_by` entries. This resolves the UX gap where `--related-to` and `--informed-by` would give different results.
- **Case-insensitive matching** on target IDs (consistent with existing `list_changes.py` pattern).

### 5.3 Spec Model Update (B1)

Add `.relations` property to `Spec`:

```python
# supekku/scripts/lib/specs/models.py

@property
def relations(self) -> list[dict[str, Any]]:
    """Return list of relation dicts from frontmatter."""
    relations = self.frontmatter.data.get("relations", [])
    if isinstance(relations, list):
        return [
            dict(r) for r in relations
            if isinstance(r, Mapping) and "type" in r and "target" in r
        ]
    return []
```

This uses the same pattern as other Spec properties — reads from `frontmatter.data`, validates shape, returns clean data.

### 5.4 Registry Integration

**`ChangeRegistry.find_by_implements()`** — keeps its current narrow semantics (B2 accepted). It searches only `applies_to.requirements`, not the broader `find_related_to`:

```python
def find_by_implements(self, requirement_id: str | None) -> list[ChangeArtifact]:
    if not requirement_id:
        return []
    req_lower = requirement_id.lower()
    return [
        a for a in self.collect().values()
        if a.applies_to
        and req_lower in [r.lower() for r in a.applies_to.get("requirements", [])]
    ]
```

This is functionally identical to the current implementation. The refactor opportunity is purely internal cleanup (extracting the search logic), not a semantic change.

**`SpecRegistry.find_by_informed_by()`** — unchanged.

The new `--related-to` flag on CLI uses `find_related_to()` from the query module, which is the broader search. This gives users both paths:

- `--implements` — narrow, requirements only
- `--related-to` — broad, all reference slots

### 5.5 CLI Flags

New flags on `list deltas` and `list specs`:

```
--related-to ID      Show artifacts referencing ID in any reference slot
--relation TYPE:ID   Show artifacts with a specific relation in .relations (e.g. implements:PROD-010)
--refs               Include refs column in output (count in table, type:target in TSV)
```

Parsing: `--relation TYPE:TARGET` splits on the first colon only. `--relation` with no colon is an error. Unrecognised TYPE values emit a warning on stderr (W6 accepted) but still filter (the type may be valid in frontmatter even if not in `RELATION_TYPES`).

Existing flags unchanged:

- `--implements REQ-ID` — same narrow semantics as today
- `--informed-by ADR-ID` — same spec-specific search as today

Filter application order (consistent with existing pattern):

1. `--related-to` / `--relation` / `--implements` (pre-filter)
2. `--status` (status filter)
3. `--filter` / `--regexp` (text filter)

### 5.6 Reference Column Rendering

```python
# supekku/scripts/lib/formatters/relation_formatters.py

def format_refs_count(artifact: Any) -> str:
    """Format total reference count for table column.

    Uses collect_references() to count all slots, not just .relations.
    Returns e.g. '3 refs' or ''.
    """

def format_refs_tsv(artifact: Any) -> str:
    """Format all references as source.detail:target pairs for TSV.

    Returns e.g. 'relation.implements:PROD-010,context_input.issue:IMPR-006'.
    """
```

The column uses `collect_references()` so it aligns with `--related-to` search scope (W4/S2 accepted). The flag is `--refs` not `--relations` (S4 accepted).

Table output with `--refs`:

```
ID       Status       Name                              Refs
DE-085   draft        Surface relation and backlog...   4 refs
DE-083   in-progress  Strengthen audit to spec...       7 refs
```

TSV output with `--refs`:

```
DE-085  draft  Surface relation...  relation.relates_to:IMPR-006,relation.relates_to:IMPR-011,context_input.issue:IMPR-006,context_input.issue:IMPR-011
```

### 5.7 `create delta --from-backlog` Enhancement (W7)

When `--from-backlog ITEM-ID` is used, auto-populate:

```python
# In the --from-backlog branch of create.py
context_inputs = [{"type": "issue", "id": item.id}]
relations = [{"type": "relates_to", "target": item.id}]
```

These are written into the delta frontmatter alongside the existing title and requirements population. This ensures deltas created via CLI from backlog items are immediately discoverable via `--related-to`.

### 5.8 Unsupported Artifact Types (B3)

`DecisionRecord` has a fundamentally different shape — named lists (`supersedes`, `related_decisions`, `specs`, `policies`, etc.) rather than a unified `.relations` list. `BacklogItem` has no relations at all.

These types do **not** participate in the generic query layer. Adding `--related-to` to `list decisions` or `list backlog` is out of scope for this delta. Future deltas can normalise these models by adding a `.relations` property that synthesises from the named lists, similar to how Spec will expose `.informed_by` through `collect_references()`.

The query functions gracefully handle artifacts without `.relations` — they're simply not `RelationQueryable` and `collect_references()` returns `[]` for them.

### 5.9 Skill Updates

`scope-delta` skill updated to instruct agents:

> When creating a delta from a backlog item (or motivated by one), populate:
>
> - `context_inputs` with `type: issue` and the backlog item ID
> - `relations` with `type: relates_to` (or more specific type) and the backlog item ID as target
> - `applies_to.requirements` with any requirement IDs from the backlog item

This makes the metadata consistently discoverable via `--related-to`.

### 5.10 Future TUI Integration Point

The `collect_references()` function returns `list[ReferenceHit]` with provenance. A future cross-registry search orchestrator (IMPR-011) would:

1. Fuzzy-match the query against artifact IDs/names across all registries (high score)
2. For each direct hit, call `collect_references()` to get its forward references
3. Resolve those references across registries (lower score, tagged with match reason)
4. Return `list[SearchResult(artifact, score, match_reason)]`

Note: `collect_references()` is richest for deltas (3 slots) and adequate for specs (2 slots). The TUI orchestrator may need artifact-type-aware logic for DecisionRecord's named lists until those models are normalised. This is an IMPR-011 concern, not DE-085.

## 7. Design Decisions & Trade-offs

### DEC-085-001: Generic query functions with `RelationQueryable` Protocol

Pure functions in `relations/query.py` operating on any artifact conforming to `RelationQueryable` (minimal `runtime_checkable` Protocol requiring `.relations: list[dict]`). Additional slots searched via `getattr` if present.

- **Pro**: POL-001 compliance, no duplication, reusable in TUI
- **Pro**: Protocol documents the contract without forcing inheritance (ADR-009 compatible)
- **Pro**: Pure functions preferred (CLAUDE.md architecture guide)
- **Con**: Protocol only covers `.relations`; other slots are still duck-typed
- **Accepted**: The Protocol covers the required minimum. Optional slots degrade gracefully.

### DEC-085-002: `--implements` keeps original narrow semantics

`--implements` continues to search only `applies_to.requirements`. It is **not** reimplemented as a call to `find_related_to()`, which would expand its semantics to include `.relations` and `.context_inputs` (B2 fix).

- **Pro**: No semantic change, no subtle regression
- **Pro**: Clear distinction: `--implements` = "which deltas claim to implement this requirement"; `--related-to` = "which deltas mention this ID anywhere"
- **Accepted**

### DEC-085-003: `--related-to` searches all reference slots including `informed_by`

Searches `.relations`, `.applies_to`, `.context_inputs`, and `.informed_by` (when present). This means `list specs --related-to ADR-001` finds specs with `informed_by: [ADR-001]` even if they have no `relations` entry for it — eliminating the confusing gap where `--related-to` misses things `--informed-by` finds (W2 fix).

- **Pro**: "Find everything linked to X" matches user intent
- **Pro**: No UX gap between `--related-to` and `--informed-by` results
- **Con**: `collect_references()` has artifact-type-aware knowledge (knows about `informed_by`)
- **Accepted**: This is a small, documented adaptation — not a violation of genericity. The function searches slots by presence, with `informed_by` as one more optional slot.

### DEC-085-004: `--refs` flag, count in table, `type:target` in TSV

Column uses `collect_references()` to count all reference slots, aligning display with search semantics. Flag name is `--refs` not `--relations` since it spans more than `.relations` (S4 accepted).

- **Pro**: Progressive disclosure (PROD-010 principle)
- **Pro**: Column semantics match `--related-to` search semantics — no confusion
- **Accepted**

### DEC-085-005: `ReferenceHit.detail` with documented mapping

Field renamed from `relation_type` to `detail` to avoid implying it always carries a relation type. The mapping is documented in the dataclass docstring (W3 fix).

- **Pro**: Honest naming — `detail` doesn't promise more than it delivers
- **Pro**: Documented mapping prevents caller confusion
- **Accepted**
