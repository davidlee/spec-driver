## 1. Summary & Context

- **Technical Spec(s)**: SPEC-132 (memory subsystem), SPEC-116 (frontmatter metadata)
- **Implementation Plan**: [IP-086](./IP-086.md)
- **Change Drivers**: ISSUE-047 (child of ISSUE-020 — general SHA tracking)

## 2. Motivation

- Spec-driver memories describe code, patterns, and conventions that evolve. There is no mechanical way to assess whether a memory is still accurate.
- Memories with `scope.paths` describe specific codebase regions, but nothing records which codebase state they were validated against.
- A git SHA stamped at verification time, combined with scope paths, enables mechanical staleness detection: count commits affecting scoped paths since verified SHA.
- This is a pilot for potential broader SHA tracking across all artifact types (ISSUE-020).

## 3. Scope & Objectives

- **Primary Outcomes**:
  - `verified_sha` field (full 40-char SHA) in memory frontmatter schema, tool-stamped (not agent-supplied)
  - `spec-driver edit memory <id> --verify` for attestation-only (content replacement deferred)
  - `spec-driver create memory` stamps `verified` date and `confidence` but NOT `verified_sha` (creation is authoring, not attestation)
  - `confidence` made required on memory schema (default: `medium`; skill-calibrated)
  - `spec-driver list memories --stale` with scope-aware staleness ranking (batched git, paths + globs, three-tier sort)
  - Skill updates to `/retrieving-memory`, `/reviewing-memory`, and `/capturing-memory`
- **Operational Constraints**: Minimal storage (one SHA, one date). Git provides history.
- **Dependencies**: None — builds on existing memory and frontmatter infrastructure.

## 4. Out of Scope

- General-purpose SHA tracking for non-memory artifacts (ISSUE-020 horizon)
- Vector/semantic similarity for memory retrieval
- Backfilling SHAs on existing memories (they'll get stamped on next verification)
- Tool-enforced scope calibration (skill-governed by design)
- Content replacement alongside `--verify` (two-step workflow for V1)

## 5. Approach Overview

- **System Touchpoints**:
  - `supekku/scripts/lib/core/git.py` — new module: SHA utility + constant
  - `supekku/scripts/lib/core/frontmatter_metadata/memory.py` — schema (verified_sha + confidence required)
  - `supekku/scripts/lib/memory/creation.py` — creation path (verified date, confidence, no SHA)
  - `supekku/scripts/lib/core/frontmatter_writer.py` — generalized multi-field updater
  - `supekku/cli/edit.py` — `--verify` flag (mutex with `--status`, refuses without git)
  - `supekku/cli/list.py` — `--stale` flag (batched git, three-tier sort)
  - `supekku/scripts/lib/memory/staleness.py` — new module: batched staleness computation
  - `.spec-driver/skills/` — retrieving-memory, reviewing-memory, capturing-memory
- **Key Changes**:
  - Add `verified_sha` (full 40-char SHA) to memory frontmatter metadata schema
  - Make `confidence` required with default `medium`
  - Add git HEAD resolution utility + `SHA_HEX_PATTERN` constant to core
  - Add `update_frontmatter_fields()` to frontmatter writer (generalizes existing pattern)
  - Wire SHA stamping into edit-with-verify path (not creation)
  - Implement batched staleness computation using single git invocation (paths + globs)
  - Update memory list command with three-tier staleness ranking
  - Update skills: confidence calibration, staleness-aware retrieval and review

## 6. Verification Strategy

- **Acceptance Criteria**:
  - `create memory` stamps `verified` date and `confidence` but NOT `verified_sha`
  - `edit memory --verify` updates `verified` date, `verified_sha`, and `updated`
  - `edit memory --verify` refuses when git is unavailable (non-zero exit)
  - `edit memory --verify` and `--status` are mutually exclusive
  - `list memories --stale` ranks memories in three tiers: scoped+attested, scoped+unattested, unscoped
  - Globs converted to git pathspecs for staleness computation
  - Existing memories without `verified_sha` or `confidence` continue to work (backward compat)

## 7. Risks & Mitigations

- **Risk**: Scope calibration quality — _Likelihood_: medium – _Impact_: medium – _Mitigation_: skill guidance; design acknowledges this is skill-governed
- **Risk**: Complex glob patterns don't translate to git pathspecs — _Likelihood_: low – _Impact_: low – _Mitigation_: current corpus uses simple directory prefixes; warn on untranslatable patterns
- **Risk**: Agent reflexively stamps `confidence: high` — _Likelihood_: medium – _Impact_: medium – _Mitigation_: skill calibration guidance; default is `medium`, `high` requires justification
