# DR-086 – Memory verification primitive: SHA-stamped attestation for staleness tracking

## 1. Executive Summary

- **Delta**: [DE-086](./DE-086.md)
- **Status**: draft
- **Last Updated**: 2026-03-09
- **Synopsis**: Add tool-enforced git SHA attestation to memory artifacts, enabling mechanical staleness detection via scope-aware commit counting. Make confidence a required field to complement the trust model.

## 2. Problem & Constraints

### Current Behaviour

- Memories have `verified` (date) and `scope.paths`/`scope.globs` fields, but no record of which codebase state they were validated against.
- Agents trust memories blindly or re-read and reason about accuracy every time.
- No ranking mechanism for memory review priority beyond human judgement.
- `confidence` exists but is optional and rarely populated — it carries almost no signal.

### Drivers / Inputs

- ISSUE-047 (memory verification primitive)
- ISSUE-020 (general SHA tracking — this is the pilot)
- ADR-005 (memories are canonical guidance layer — their trustworthiness matters)

### Constraints / Guardrails

- ADR-002: no backlinks in frontmatter — `verified_sha` is a forward/self property, not a backlink. Compliant.
- ADR-009: registry API convention — any new registry methods must follow `find`/`collect`/`iter`/`filter` patterns.
- POL-001: reuse existing infrastructure. Extend `frontmatter_writer`, don't duplicate.
- POL-002: no magic strings. SHA pattern, field names, and git commands must use named constants.
- STD-002: lint compliance required.

### Out of Scope

- SHA tracking for non-memory artifact types
- Vector/semantic similarity
- Backfilling SHAs on existing memories
- Tool-enforced scope calibration
- Content replacement alongside `--verify` (deferred — V1 is attestation-only)

## 3. Architecture Intent

### Target Outcomes

1. `edit memory --verify` records an explicit attestation checkpoint (SHA + date)
2. New memories get `verified` date but NOT `verified_sha` — first explicit attestation establishes the SHA
3. `confidence` is required at creation, calibrated by skill guidance
4. `list memories --stale` ranks scoped memories by commits-since-verification
5. Skills consume staleness and confidence signals to guide agent behaviour

### Guiding Principles

- **Tool-enforced, not agent-supplied**: SHA comes from `git rev-parse HEAD`, not from agent input
- **Flat storage**: one `verified_sha` string field — git provides history
- **Full SHA in storage, short in display**: avoid ambiguity as repos grow
- **Query-time computation**: staleness is computed on demand, not stored (ADR-002 spirit: derived data stays derived)
- **Graceful degradation**: no git → `--verify` refuses; no scope → no commit-based staleness; no `verified_sha` → memory works, shown as "never attested"
- **Confidence + freshness = trust**: confidence is the quality axis, SHA is the freshness axis. Together they give a trust picture neither provides alone.

### Trust Model

`verified_sha` records that _an agent explicitly attested this memory was accurate at commit X_. It is not stamped at creation — creation is authoring, not verification. The distinction matters: a newly created memory with `confidence: medium` and no `verified_sha` says "someone wrote this with reasonable confidence, but it hasn't been checked yet." An attested memory with `confidence: high` and a recent SHA says "someone verified this against the code at this commit."

## 5. Detailed Design

### 5.1 Git SHA Utility

New module: `supekku/scripts/lib/core/git.py`

```python
"""Git integration utilities for spec-driver."""

import subprocess
from pathlib import Path

SHA_HEX_PATTERN = r"^[0-9a-f]{40}$"

def get_head_sha(root: Path | None = None) -> str | None:
  """Return full 40-char SHA of HEAD, or None if not in a git repo."""
  try:
    result = subprocess.run(
      ["git", "rev-parse", "HEAD"],
      capture_output=True, text=True, timeout=5,
      cwd=root,
    )
    return result.stdout.strip() if result.returncode == 0 else None
  except (FileNotFoundError, subprocess.TimeoutExpired):
    return None

def short_sha(sha: str, length: int = 8) -> str:
  """Truncate a full SHA for display."""
  return sha[:length]
```

**DEC-086-01**: Store full 40-char SHA in frontmatter. Display short SHA in CLI output. Full SHA avoids ambiguity as the repo grows — short SHAs can become ambiguous when new objects collide. The frontmatter is tool-managed, not human-edited; readability there is not worth the correctness risk.

### 5.4 Memory Creation

**DEC-086-04**: New memories get `verified` date (today) but NOT `verified_sha`. Creation is authoring, not attestation. The `verified` date provides age-based staleness for unscoped memories without falsely claiming the content has been checked.

```python
def build_memory_frontmatter(memory_id, options):
  today = date.today().isoformat()
  return {
    "id": memory_id,
    "name": options.name,
    "kind": "memory",
    "status": options.status,
    "memory_type": options.memory_type,
    "created": today,
    "updated": today,
    "verified": today,
    "confidence": options.confidence or "medium",
    "tags": options.tags,
    "summary": options.summary,
  }
```

**DEC-086-05**: `confidence` is required at creation. Default is `medium`. `MemoryCreationOptions` gains a `confidence: str = "medium"` field. Skill guidance calibrates:

- `low` = provisional/inferred, needs validation
- `medium` = derived from reasonable context, probably right
- `high` = verified against code, specs, or direct observation

### 5.5 Frontmatter Writer Extension

**DEC-086-02**: Extend the line-level writer rather than introducing a full YAML round-trip.

```python
@dataclass
class FieldUpdateResult:
  """Result of a frontmatter field update operation."""
  updated: set[str]   # fields that existed and were replaced
  inserted: set[str]  # fields that were missing and added

def update_frontmatter_fields(
  path: Path,
  updates: dict[str, str],
) -> FieldUpdateResult:
  """Update multiple frontmatter fields via line-level replacement.

  For each key in updates, finds `key: ...` in frontmatter and replaces
  the line. Only operates on simple scalar fields (not nested objects).

  Fields not found in existing frontmatter are inserted before the
  closing --- marker, in dict iteration order.

  Args:
    path: Path to the artifact markdown file (must exist).
    updates: Mapping of field name to new value string.

  Returns:
    FieldUpdateResult with sets of updated and inserted field names.
  """
```

Insertion order: dict iteration order (Python 3.7+ guarantees insertion order). Callers control ordering via the dict literal they pass. The return type distinguishes updated (found and replaced) from inserted (added new), so callers can detect first-time field additions.

### 5.6 Edit Memory --verify

```python
@app.command("memory")
def edit_memory(
  memory_id: ...,
  status: StatusOption = None,
  verify: bool = typer.Option(
    False, "--verify",
    help="Attest memory accuracy at current HEAD",
  ),
  root: RootOption = None,
) -> None:
```

Validation rules:

1. **`--verify` and `--status` are mutually exclusive.** If both passed, print error and exit non-zero.
2. **`--verify` requires git.** If `get_head_sha()` returns `None`, refuse with: `"Error: cannot verify — git not available or not in a repository"`. Do not stamp a null SHA — that would clear a previously valid SHA, which is data loss, not degradation.
3. **No status gating.** Verification is allowed regardless of memory status. A deprecated memory might be worth verifying to confirm it _should_ be deprecated.

When `--verify` passes validation:

1. Resolve the memory artifact
2. Get HEAD SHA via `get_head_sha()`
3. Call `update_frontmatter_fields(path, {"verified": today, "verified_sha": sha, "updated": today})`
4. Print confirmation: `Verified: {memory_id} at {short_sha(sha)}`

### 5.7 Staleness Computation

New module: `supekku/scripts/lib/memory/staleness.py`

```python
@dataclass
class StalenessInfo:
  memory_id: str
  verified_sha: str | None
  verified_date: date | None
  scope_paths: list[str]
  commits_since: int | None  # None if unscoped or no SHA
  days_since: int | None     # fallback for unscoped or unattested memories
  has_scope: bool            # distinguishes unscoped from scoped-but-no-sha

def compute_batch_staleness(
  records: list[MemoryRecord],
  root: Path,
) -> list[StalenessInfo]:
  """Compute staleness for multiple memories using a single git invocation."""
```

**DEC-086-06**: Batch git invocation. Instead of one subprocess per memory, use a single `git log` pass:

1. Find the oldest `verified_sha` across all scoped memories
2. Run `git log --oneline <oldest_sha>..HEAD --name-only -- <all_scope_paths>` once
3. Parse output: for each commit, record which paths were affected
4. For each memory, count commits that touched its scope paths since its `verified_sha`

This is simpler to test (mock one subprocess, not N), faster (one git call), and more composable (the commit data can be reused for different filtering).

**DEC-086-07**: Include `scope.globs` in V1. Empirical data from the current corpus shows 12 memories use globs, 7 are globs-only. The glob patterns are consistently simple directory prefixes (`supekku/tui/**`, `supekku/cli/**`). These translate directly to git pathspecs. Convert by:

- Stripping trailing `/**` → use as directory pathspec
- Passing other simple patterns through to git (git handles `*` and `**` in pathspecs)

Memories that are truly unscoped (no `paths` and no `globs`) fall back to `days_since` from `verified` date (or `updated` if never verified).

### 5.8 List Memories --stale

Add `--stale` flag to the memory list command.

**Tiered output** (explicit sort semantics):

1. **Tier 1: Scoped, attested memories** — sorted by `commits_since` descending (most stale first)
2. **Tier 2: Scoped, unattested memories** — sorted by `days_since` descending (have scope but no SHA yet)
3. **Tier 3: Unscoped memories** — sorted by `days_since` descending

Each tier is visually distinguishable. Commits and days are incommensurable; tiering avoids false comparisons.

Example output:

```
ID                                Confidence  Verified    Stale     Scope
── scoped, attested ──
mem.pattern.cli.skinny            high        2026-02-01  47↑       supekku/cli/
mem.signpost.spec-driver.file     medium      2026-03-08  3↑        .spec-driver/
── scoped, unattested ──
mem.fact.tui.widget-selection     medium      2026-03-01  —         supekku/tui/ (8d, no sha)
── unscoped ──
mem.concept.spec-driver.delta     medium      2026-01-15  —         (53d)
```

### 5.9 Skill Integration

**`/capturing-memory`**: Add confidence calibration guidance:

- `low`: "I'm inferring or generalizing — this should be validated"
- `medium`: "I derived this from reasonable context — probably right"
- `high`: "I verified this against code, specs, or direct observation"
- Default to `medium`. Require justification for `high`.
- Reinforce: new memories are authored, not verified. The creating agent should not claim `high` confidence unless it truly verified the content against the source.

**`/retrieving-memory`**: When presenting a memory, surface staleness qualitatively. Use language like "this memory has not been attested" (no SHA), "many commits have affected its scope since attestation" (high staleness), or "recently attested, scope is quiet" (low staleness). Do not use a numeric threshold — qualitative guidance avoids POL-002 issues and is more robust to different repo cadences.

**`/reviewing-memory`**: Use `list memories --stale` output as the default review queue. Prioritize tier 1 (scoped, attested, high commit count) as the highest-value review targets. Flag unscoped memories separately for manual review based on age.

## 7. Design Decisions & Trade-offs

### DEC-086-01: Full 40-char SHA in storage, short SHA in display only

- **Chosen**: `git rev-parse HEAD` (full SHA) in frontmatter; `short_sha()` for CLI display
- **Rationale**: short SHAs become ambiguous as repos grow. `git rev-list --count <sha>..HEAD` can fail if a short SHA resolves to multiple objects. Frontmatter is tool-managed, not human-edited — readability is not worth the correctness risk.
- **Trade-off**: slightly longer frontmatter lines. Negligible.

### DEC-086-02: Extend line-level writer, not replace with full YAML round-trip

- **Chosen**: add `update_frontmatter_fields(path, fields)` alongside existing `update_frontmatter_status`
- **Rationale**: full YAML round-trip risks reordering fields, changing quoting, losing comments. Line-level replacement is proven safe. The new function generalizes the pattern.
- **Trade-off**: can't handle nested fields. Not needed here; separate concern if needed later.
- **Return type**: `FieldUpdateResult` with `updated` and `inserted` sets — callers can distinguish replacement from first-time addition.
- **POL-001 compliance**: reuses and generalizes existing infrastructure.

### DEC-086-03: Staleness is query-time, not stored

- **Chosen**: compute staleness on `list --stale`, don't persist it
- **Rationale**: staleness changes with every commit. Storing it would require constant updates (ADR-002 spirit: derived data stays derived).
- **Trade-off**: `list --stale` invokes git. Acceptable — batched approach (DEC-086-06) keeps it to one subprocess.

### DEC-086-04: No `verified_sha` at creation — verified date only

- **Chosen**: new memories get `verified: today` but no `verified_sha`
- **Rationale**: creation is authoring, not attestation. Conflating them poisons the staleness signal — a memory born with wrong scope would appear "fresh" despite being wrong from birth. The `verified` date provides age-based ranking for unscoped memories. The SHA requires an explicit `--verify` act.
- **Trade-off**: new memories appear in `list --stale` as "unattested" rather than "fresh." This is honest — they haven't been checked yet. The two-step cost (create, then verify after use) is low and correctly reflects the trust model.

### DEC-086-05: Confidence required at creation, default `medium`

- **Chosen**: make `confidence` a required field with default `medium`
- **Rationale**: confidence and verified_sha are complementary trust axes. Confidence = quality, SHA = freshness. Without confidence, the trust model is one-dimensional. Making it required ensures every memory carries a quality signal. Default `medium` avoids burdening agents while ensuring the field is populated.
- **Calibration**: skill-governed. `/capturing-memory` provides definitions for each level. Agents should not claim `high` without direct verification.
- **Migration**: existing memories without `confidence` continue to load (model treats absent as `None`). The schema change means new memories must have it. Existing memories get it on next edit.

### DEC-086-06: Batched git invocation for staleness

- **Chosen**: single `git log` pass for all scoped memories, partition results per memory
- **Rationale**: one subprocess per memory is an architectural smell — untestable without mocking N subprocesses, fragile, not composable. A single pass is simpler to test (one mock), faster, and the module's interface stays the same.
- **Trade-off**: slightly more complex parsing logic. Worth it.

### DEC-086-07: Include scope.globs in V1 staleness

- **Chosen**: convert glob patterns to git pathspecs in V1
- **Rationale**: 7 of 12 glob-using memories are globs-only. Excluding them would make staleness ranking incomplete for ~20% of scoped memories with no indication to the user. The glob patterns are simple directory prefixes (`supekku/tui/**`) that translate directly to git pathspecs.
- **Trade-off**: complex glob patterns (if any appear in future) might not translate cleanly. Current corpus has none. Add a warning for untranslatable patterns if needed.

## 8. Rollout & Operational Notes

### Migration / Backfill

- No backfill. Existing memories continue to work without `verified_sha` or `confidence`.
- Memories get SHA-stamped on their next `edit --verify` cycle.
- Confidence gets populated on next edit or creation.

### Backward Compatibility

- `verified_sha` and `confidence` are optional in the model (accept `None`)
- Schema makes `confidence` required for new artifacts only
- All paths handle absent fields gracefully
- No existing functionality changes behaviour

### Git Availability

- `get_head_sha()` returns `None` outside git repos
- `edit --verify` refuses when git is unavailable (error, not silent degradation)
- `list --stale` degrades gracefully: no git = all memories shown as "no staleness data"

## 9. External Adversarial Review Findings

Review performed against DR-086 v1. All findings triaged and integrated.

### Blockers (resolved)

1. **§5.1 Short SHA is a landmine**: Short SHAs become ambiguous as repos grow, causing `git rev-list` failures. → **DEC-086-01**: store full 40-char SHA. Display short.
2. **§5.4 Implicit verification conflates authoring with attestation**: Stamping `verified_sha` at creation poisons the staleness signal for memories born with wrong content/scope. → **DEC-086-04**: no SHA at creation; `verified` date only.

### Significant (resolved)

3. **§5.6 Missing validation**: `--verify` without git would stamp null SHA (data loss); no mutex with `--status`; no consideration of status gating. → Refuse without git; mutex enforced; status gating explicitly declined (deprecated memories may be worth verifying).
4. **§5.7 Subprocess per memory is architectural smell**: Untestable, fragile, not composable. → **DEC-086-06**: batched single git invocation.
5. **§5.7 scope.globs deferral creates coverage gap**: 7 globs-only memories would be silently treated as unscoped. → **DEC-086-07**: include globs in V1.

### Minor (resolved)

6. **§5.5 Writer insertion ordering underspecified**: → Specified as dict iteration order; return type split into updated/inserted sets.
7. **§5.8 Sorting semantics underspecified**: Commits and days are incommensurable. → Explicit three-tier sort.
8. **§5.9 Threshold undefined**: Hardcoded threshold would violate POL-002. → Qualitative skill guidance, no numeric threshold.

### Complementary decision (from brainstorm)

9. **Confidence as required field**: Addresses "just written but wrong" scenario. Confidence (quality) + SHA (freshness) = two-axis trust model. → **DEC-086-05**.
