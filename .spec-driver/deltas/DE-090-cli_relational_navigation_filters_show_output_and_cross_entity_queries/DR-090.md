# DR-090 â€“ CLI relational navigation (P0â€“P4)

## 1. Executive Summary

- **Delta**: [DE-090](./DE-090.md)
- **Status**: draft
- **Last Updated**: 2026-03-13
- **Synopsis**: Fix relation display bugs (P0), add relational filter flags (P1), enrich show output with reverse lookups (P2), add dedicated domain-field collectors and generic reverse-reference filtering (P3/P4), and add neighbourhood graph view (P5).

## 2. Problem & Constraints

### Current Behaviour

- `show delta` renders relations as `- : ISSUE-007` â€” relation type is lost (reads `"kind"` instead of `"type"`)
- `show spec` output is sparse â€” no relations, no requirements summary
- `list plans` crashes when any plan has malformed YAML in its `plan.overview` block
- Common governance queries require multi-command pipelines: "which audits cover this delta?", "which deltas touch this spec?", "which backlog items reference this entity?"

### Drivers / Inputs

- PROD-010 (CLI UX) â€” reverse relationship queries, filtering
- SPEC-110 (CLI specification) â€” consistent flag patterns
- ADR-009 â€” registry API convention

### Constraints / Guardrails

- POL-001: maximise reuse â€” use existing `find_related_to()`, `find_by_relation()` from `relations/query.py`
- POL-002: no magic strings â€” use `ARTIFACT_PREFIXES` and relation type constants
- STD-002: lint signal â€” zero new warnings
- Skinny CLI pattern: filters in CLI, no business logic

### Out of Scope

- TUI relational navigation (DE-087)
- New entity types or frontmatter schema changes
- Backlog frontmatter schema enforcement
- Transitive closure / multi-hop graph traversal

## 3. Architecture Intent

### Target Outcomes

- All relation types visible in `show delta` output (PROD-010.FR-005)
- `show spec` includes relations and requirement counts
- `list plans` degrades gracefully on parse errors
- Single-command answers to: "audits for this delta", "revisions for this delta", "deltas for this spec", "requirements implemented by this delta", "backlog items related to this entity"

### Guiding Principles

- Follow existing filter patterns â€” each new flag mirrors established `--spec`/`--related-to` implementations
- Pure functions for new formatter helpers
- No new registry methods needed â€” filter logic stays in CLI layer using existing query functions and field access
- Exception: `--implemented-by` requires loading a single delta to read its `requirements.implements` â€” this is a targeted lookup, not a registry extension

#### P0-2: Enrich `show spec`

New helpers in `spec_formatters.py`:

**Data access note**: `Spec.frontmatter` is a `FrontmatterValidationResult` (frozen dataclass), not a dict. Relations are accessed via `spec.frontmatter.relations` (tuple of `Relation` objects with `.type`/`.target` attributes). Requirements are NOT in spec frontmatter â€” they are extracted from the spec body by `RequirementsRegistry`. The requirements count must come from the registry, not the model.

```python
def _format_spec_relations(spec: Spec) -> list[str]:
  """Format relations section for spec details."""
  if not spec.frontmatter.relations:
    return []
  lines = ["", "Relations:"]
  for rel in spec.frontmatter.relations:
    lines.append(f"  - {rel.type}: {rel.target}")
  return lines
```

For requirements summary, `format_spec_details()` needs a `requirements_count` parameter (or the caller passes pre-computed counts):

```python
def _format_requirements_summary(
  fr_count: int, nf_count: int, other_count: int = 0,
) -> list[str]:
  """Format requirements count summary for spec details."""
  parts = []
  if fr_count:
    parts.append(f"{fr_count} FR")
  if nf_count:
    parts.append(f"{nf_count} NF")
  if other_count:
    parts.append(f"{other_count} other")
  return ["", f"Requirements: {', '.join(parts)}"] if parts else []
```

The caller (`show_spec` in `show.py`) loads `RequirementsRegistry`, counts requirements for the spec ID, and passes counts to the formatter. This keeps the formatter pure and avoids registry coupling in the formatters layer.

Add both to `format_spec_details()` sections list, between packages and file path.

JSON output (`show spec --json`): `Spec.to_dict()` currently omits relations. Add `relations` (serialised from `spec.frontmatter.relations`) to `to_dict()`. Requirements count can be included when the caller provides it, or as a separate JSON enrichment in the show command.

#### P3-1: Dedicated domain-field collectors

Domain models store references in named fields (e.g. `DecisionRecord.specs`, `RequirementRecord.implemented_by`) that are **not** authored frontmatter relations. Encoding these as fake `.relations` entries would inject synthetic types (`references`, `belongs_to`, etc.) into `RELATION_TYPES` semantics, poisoning `find_by_relation()` and refs TSV output.

Instead, each model family gets a dedicated collector in `relations/query.py` that yields `ReferenceHit` with `source="domain_field"` and the originating field name as `detail`. This is the same pattern P3-2 uses for BacklogItem â€” one consistent approach for all non-relation domain references.

**New `REFERENCE_SOURCES` entry**: Add `"domain_field"` to the frozenset in `core/relation_types.py`.

**Collector for DecisionRecord** (`relations/query.py`):

```python
_DECISION_REFERENCE_FIELDS: tuple[str, ...] = (
    "specs", "deltas", "requirements", "revisions", "audits",
    "policies", "standards", "related_decisions", "related_policies",
    "supersedes", "superseded_by",
)

def _collect_from_decision_fields(artifact: Any) -> list[ReferenceHit]:
    """Extract references from DecisionRecord domain fields."""
    hits: list[ReferenceHit] = []
    for field_name in _DECISION_REFERENCE_FIELDS:
        for target in getattr(artifact, field_name, None) or []:
            target_str = str(target).strip()
            if target_str:
                hits.append(ReferenceHit(
                    target=target_str, source="domain_field", detail=field_name,
                ))
    return hits
```

**Collector for PolicyRecord / StandardRecord** (`relations/query.py`):

```python
_GOVERNANCE_REFERENCE_FIELDS: tuple[str, ...] = (
    "specs", "requirements", "deltas", "standards", "policies",
    "related_policies", "related_standards",
    "supersedes", "superseded_by",
)

def _collect_from_governance_fields(artifact: Any) -> list[ReferenceHit]:
    """Extract references from Policy/Standard domain fields."""
    hits: list[ReferenceHit] = []
    for field_name in _GOVERNANCE_REFERENCE_FIELDS:
        for target in getattr(artifact, field_name, None) or []:
            target_str = str(target).strip()
            if target_str:
                hits.append(ReferenceHit(
                    target=target_str, source="domain_field", detail=field_name,
                ))
    return hits
```

**Collector for RequirementRecord** (`relations/query.py`):

```python
_REQUIREMENT_REFERENCE_FIELDS: tuple[str, ...] = (
    "specs", "implemented_by", "verified_by", "coverage_evidence",
)

def _collect_from_requirement_fields(artifact: Any) -> list[ReferenceHit]:
    """Extract references from RequirementRecord domain fields."""
    hits: list[ReferenceHit] = []
    # Scalar field
    primary = getattr(artifact, "primary_spec", None)
    if primary:
        hits.append(ReferenceHit(
            target=str(primary).strip(), source="domain_field", detail="primary_spec",
        ))
    # List fields
    for field_name in _REQUIREMENT_REFERENCE_FIELDS:
        for target in getattr(artifact, field_name, None) or []:
            target_str = str(target).strip()
            if target_str:
                hits.append(ReferenceHit(
                    target=target_str, source="domain_field", detail=field_name,
                ))
    return hits
```

**Dispatch in `collect_references()`**: The collectors are model-agnostic (they use `getattr` with defaults), so all five can be chained unconditionally â€” fields that don't exist on a given model simply yield no hits. However, for clarity and minor efficiency, use a single `_collect_from_domain_fields()` dispatcher:

```python
def _collect_from_domain_fields(artifact: Any) -> list[ReferenceHit]:
    """Extract references from domain-specific fields (non-relation)."""
    return (
        _collect_from_decision_fields(artifact)
        + _collect_from_governance_fields(artifact)
        + _collect_from_requirement_fields(artifact)
        + _collect_from_backlog_fields(artifact)
    )
```

Updated `collect_references()`:

```python
def collect_references(artifact: Any) -> list[ReferenceHit]:
    return (
        _collect_from_relations(artifact)
        + _collect_from_applies_to(artifact)
        + _collect_from_context_inputs(artifact)
        + _collect_from_informed_by(artifact)
        + _collect_from_domain_fields(artifact)
    )
```

**Key invariant**: Authored frontmatter relations flow through `_collect_from_relations()` with `source="relation"` and the relation type as `detail`. Domain-specific reference fields flow through dedicated collectors with `source="domain_field"` and the field name as `detail`. The two paths never cross â€” `RELATION_TYPES` stays clean.

#### P3-2: BacklogItem reference collection

`BacklogItem` references live in raw frontmatter, not structured fields. This follows the same dedicated-collector pattern as P3-1 â€” `source="backlog_field"` provenance, no invented relation types.

**Collector in `relations/query.py`** (called by `_collect_from_domain_fields()`):

```python
def _collect_from_backlog_fields(artifact: Any) -> list[ReferenceHit]:
    """Extract references from backlog-specific frontmatter fields."""
    hits: list[ReferenceHit] = []
    frontmatter = getattr(artifact, "frontmatter", None)
    if not isinstance(frontmatter, dict):
        return hits
    for delta_id in frontmatter.get("linked_deltas", []):
        target = str(delta_id).strip()
        if target:
            hits.append(ReferenceHit(target=target, source="backlog_field", detail="linked_delta"))
    for req_id in frontmatter.get("related_requirements", []):
        target = str(req_id).strip()
        if target:
            hits.append(ReferenceHit(target=target, source="backlog_field", detail="related_requirement"))
    return hits
```

This preserves distinct provenance (`source="backlog_field"`) without inventing relation types. BacklogItem does NOT get a `.relations` property â€” its references are collected through the query layer's domain-field dispatcher (see P3-1).

Also update P1-8 (`list backlog --related-to`) to use `matches_related_to()` instead of raw frontmatter access, so it picks up `linked_deltas`/`related_requirements` via the new collector.

### P4 â€” Generic Reverse Reference Filtering

#### P4-1: Query infrastructure

**New functions in `relations/query.py`**:

```python
def collect_reverse_reference_targets(
    referrers: Iterable[Any],
) -> set[str]:
    """Collect all target IDs referenced by a set of artifacts."""
    targets: set[str] = set()
    for referrer in referrers:
        for hit in collect_references(referrer):
            targets.add(hit.target.upper())
    return targets


def partition_by_reverse_references(
    candidates: Sequence[T],
    referrers: Iterable[Any],
    candidate_id_fn: Callable[[T], str] = lambda c: c.id,
) -> tuple[list[T], list[T]]:
    """Partition candidates into (referenced, unreferenced) by referrers.

    Returns a tuple of two lists:
      - candidates whose ID appears as a target in any referrer's references
      - candidates whose ID does not
    """
    targets = collect_reverse_reference_targets(referrers)
    referenced, unreferenced = [], []
    for c in candidates:
        bucket = referenced if candidate_id_fn(c).upper() in targets else unreferenced
        bucket.append(c)
    return referenced, unreferenced
```

def load_all_artifacts(root: Path, artifact_type: str) -> list[Any]:
    """Load all artifacts of a given type for cross-registry queries."""
    if artifact_type in ("delta", "revision", "audit"):
        return list(ChangeRegistry(root=root, kind=artifact_type).iter())
    if artifact_type == "spec":
        return list(SpecRegistry(root=root).all_specs())
    if artifact_type in ("issue", "problem", "improvement", "risk"):
        return [i for i in BacklogRegistry(root=root).iter() if i.kind == artifact_type]
    if artifact_type == "backlog":
        return list(BacklogRegistry(root=root).iter())
    if artifact_type == "adr":
        return list(DecisionRegistry(root=root).iter())
    if artifact_type == "policy":
        return list(PolicyRegistry(root=root).iter())
    if artifact_type == "standard":
        return list(StandardRegistry(root=root).iter())
    if artifact_type == "requirement":
        return list(RequirementsRegistry(root=root).iter())
    raise typer.BadParameter(f"Unknown artifact type: {artifact_type}")
```

### P5 â€” Neighbourhood View (`--related`)

#### P5-1: `show <kind> <id> --related`

Adds a `--related` flag to `show spec`, `show delta`, `show requirement`, `show issue`.

**Scope justification** (external review finding #4): These are the four primary navigation anchors â€” specs define intent, deltas track change, requirements trace obligations, issues capture problems. `show adr --related`, `show revision --related`, etc. are useful but lower-traffic; deferred to a follow-up.

When `--related` is passed, the "Related:" counts section (P2) is replaced by the full itemised "Referenced by" section â€” not shown alongside it (internal review finding #3).

**Output** (appended after normal show output):

```

    DE-009  completed  CLI JSON consistency fixes
    DE-011  completed  CLI enhanced filtering and self-documentation

    ...

**Implementation**: Composition of P2 + P3 + P4 infrastructure:

1. `show` command checks `--related` flag
2. Loads registries relevant to the target kind (not all registries unconditionally â€” external review finding #6)
3. Forward references: `collect_references(artifact)` â€” what does this entity reference?
4. Reverse references: for each loaded registry, `find_related_to(all_artifacts, entity_id)` â€” what references this entity?
5. Groups results by kind

| spec        | ChangeRegistry, DecisionRegistry, RequirementsRegistry, PolicyRegistry, StandardRegistry |

| requirement | ChangeRegistry (deltas), DecisionRegistry, PolicyRegistry, StandardRegistry              |
| issue       | ChangeRegistry (deltas)                                                                  |

```python
def format_related_section(
    reverse_refs_by_kind: dict[str, list[tuple[str, str]]],  # kind -> [(id, name), ...]
) -> list[str]:
    """Format the 'Referenced by' section for --related output."""
    lines = []
    if reverse_refs_by_kind:
        lines.append("")
        lines.append("Referenced by:")
        for kind, refs in sorted(reverse_refs_by_kind.items()):
            kind_label = kind.replace("_", " ").title() + "s"
            lines.append(f"  {kind_label} ({len(refs)}):")
            for ref_id, ref_name in refs:
                lines.append(f"    {ref_id}  {ref_name}")
    return lines
```

## 7. Design Decisions & Trade-offs

- **DEC-090-01**: Display relation types raw (`relates_to`, `implements`) not humanized. Rationale: matches frontmatter, agents consume it, avoids mapping table.
- **DEC-090-02**: `show spec` requirements: count by default, `--requirements` flag for full list. Rationale: progressive disclosure; `view spec` shows full body.
- **DEC-090-03**: Plan parse errors warn to stderr and skip. Rationale: simpler than collecting; stderr doesn't pollute piped/JSON output.
- **DEC-090-04**: `--delta` filter on audits/revisions checks `relations` targets only. Audits/revisions reference deltas via relations, not a dedicated field.
- **DEC-090-05**: `--implemented-by` does single-delta lookup, not full reverse index. User specifies one delta; loading all deltas is unnecessary overhead.
- **DEC-090-06**: Reverse lookups use existing generic query functions (`find_related_to`) called from CLI layer; no new registry methods. `--related` graph walk composes the same functions across multiple registries.
- **DEC-090-07**: Auditâ†”delta linkage via `relations` with `type: documents`, not a dedicated frontmatter field. Consistent with how revisions already link to deltas. ADR-002 compatible (forward reference, not backlink).
- **DEC-090-08**: `show spec` shows reverse lookup counts in default view; itemised listing deferred to `--related`. Keeps default fast and focused.
- **DEC-090-09**: Domain-specific reference fields (e.g. `DecisionRecord.specs`, `RequirementRecord.implemented_by`) are surfaced through dedicated collectors with `source="domain_field"` provenance, not through synthetic `.relations` properties. Authored frontmatter relations flow through `_collect_from_relations()` with `source="relation"`; domain fields flow through model-family collectors. The two paths never cross â€” `RELATION_TYPES` stays clean. BacklogItem uses a parallel `source="backlog_field"` collector for its raw frontmatter references.
- **DEC-090-10**: Governance queries (`--unaudited`, `--unimplemented`) are convenience aliases for the generic `--not-referenced-by` mechanism. Same engine, friendlier surface.
- **DEC-090-11**: Cross-registry composition for `--referenced-by` / `--not-referenced-by` uses a shared `load_all_artifacts(root, kind)` helper in `cli/common.py`, avoiding a parallel switchboard. CLI is the composition point; avoids coupling registries to each other.
- **DEC-090-12**: `--related` shows one-hop references only (direct), not transitive closure. Keeps output predictable and avoids graph explosion.
- **DEC-090-13**: `--related` scoped to `show spec`, `show delta`, `show requirement`, `show issue` (primary navigation anchors). Other kinds deferred to follow-up.
- **DEC-090-14**: P2-3 (backlog JSON defaults) is an additive contract change â€” documented explicitly, not hidden under "no schema changes".
- **DEC-090-15**: `--related` loads only registries relevant to the target kind, not all registries unconditionally. Keeps latency proportional to neighbourhood, not corpus.
