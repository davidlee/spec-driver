# DR-090 – CLI relational navigation (P0–P4)

## 1. Executive Summary
- **Delta**: [DE-090](./DE-090.md)
- **Status**: draft
- **Last Updated**: 2026-03-13
- **Synopsis**: Fix relation display bugs (P0), add relational filter flags (P1), enrich show output with reverse lookups (P2), add generic reverse-reference filtering (P3), and add neighbourhood graph view (P4).

## 2. Problem & Constraints

### Current Behaviour
- `show delta` renders relations as `- : ISSUE-007` — relation type is lost (reads `"kind"` instead of `"type"`)
- `show spec` output is sparse — no relations, no requirements summary
- `list plans` crashes when any plan has malformed YAML in its `plan.overview` block
- Common governance queries require multi-command pipelines: "which audits cover this delta?", "which deltas touch this spec?", "which backlog items reference this entity?"

### Drivers / Inputs
- PROD-010 (CLI UX) — reverse relationship queries, filtering
- SPEC-110 (CLI specification) — consistent flag patterns
- ADR-009 — registry API convention

### Constraints / Guardrails
- POL-001: maximise reuse — use existing `find_related_to()`, `find_by_relation()` from `relations/query.py`
- POL-002: no magic strings — use `ARTIFACT_PREFIXES` and relation type constants
- STD-002: lint signal — zero new warnings
- Skinny CLI pattern: filters in CLI, no business logic

### Out of Scope
- TUI relational navigation (DE-087)
- New entity types or frontmatter schema changes
- Backlog frontmatter schema enforcement
- Transitive closure / multi-hop graph traversal

## 3. Architecture Intent

### Target Outcomes
- All relation types visible in `show delta` output (PROD-010.FR-005)
- `show spec` includes relations and requirement counts
- `list plans` degrades gracefully on parse errors
- Single-command answers to: "audits for this delta", "revisions for this delta", "deltas for this spec", "requirements implemented by this delta", "backlog items related to this entity"

### Guiding Principles
- Follow existing filter patterns — each new flag mirrors established `--spec`/`--related-to` implementations
- Pure functions for new formatter helpers
- No new registry methods needed — filter logic stays in CLI layer using existing query functions and field access
- Exception: `--implemented-by` requires loading a single delta to read its `requirements.implements` — this is a targeted lookup, not a registry extension

#### P0-2: Enrich `show spec`

New helpers in `spec_formatters.py`:

**Data access note**: `Spec.frontmatter` is a `FrontmatterValidationResult` (frozen dataclass), not a dict. Relations are accessed via `spec.frontmatter.relations` (tuple of `Relation` objects with `.type`/`.target` attributes). Requirements are NOT in spec frontmatter — they are extracted from the spec body by `RequirementsRegistry`. The requirements count must come from the registry, not the model.

```python
def _format_spec_relations(spec: Spec) -> list[str]:
  """Format relations section for spec details."""
  if not spec.frontmatter.relations:
    return []
  lines = ["", "Relations:"]
  for rel in spec.frontmatter.relations:
    lines.append(f"  - {rel.type}: {rel.target}")
  return lines
```

For requirements summary, `format_spec_details()` needs a `requirements_count` parameter (or the caller passes pre-computed counts):

```python
def _format_requirements_summary(
  fr_count: int, nf_count: int, other_count: int = 0,
) -> list[str]:
  """Format requirements count summary for spec details."""
  parts = []
  if fr_count:
    parts.append(f"{fr_count} FR")
  if nf_count:
    parts.append(f"{nf_count} NF")
  if other_count:
    parts.append(f"{other_count} other")
  return ["", f"Requirements: {', '.join(parts)}"] if parts else []
```

The caller (`show_spec` in `show.py`) loads `RequirementsRegistry`, counts requirements for the spec ID, and passes counts to the formatter. This keeps the formatter pure and avoids registry coupling in the formatters layer.

Add both to `format_spec_details()` sections list, between packages and file path.

JSON output (`show spec --json`): `Spec.to_dict()` currently omits relations. Add `relations` (serialised from `spec.frontmatter.relations`) to `to_dict()`. Requirements count can be included when the caller provides it, or as a separate JSON enrichment in the show command.

#### P3-1: Per-model `.relations` property

Each uncovered model gets a `@property` that yields its reference fields as `list[dict[str, Any]]`, conforming to `RelationQueryable`. This follows the pattern `Spec` already uses.

**DecisionRecord** (`decisions/registry.py`):
```python
@property
def relations(self) -> list[dict[str, Any]]:
    """Yield domain-specific reference fields as relation dicts."""
    rels: list[dict[str, Any]] = []
    _field_map = {
        "specs": "references", "deltas": "references",
        "requirements": "references", "revisions": "references",
        "audits": "references", "policies": "references",
        "standards": "references",
        "related_decisions": "relates_to",
        "related_policies": "relates_to",
        "supersedes": "supersedes",
        "superseded_by": "superseded_by",
    }
    for field, rel_type in _field_map.items():
        for target in getattr(self, field, []) or []:
            rels.append({"type": rel_type, "target": str(target)})
    return rels
```

**PolicyRecord** and **StandardRecord**: Same pattern, adapted to their field names.

**RequirementRecord** (`requirements/registry.py`):
```python
@property
def relations(self) -> list[dict[str, Any]]:
    rels: list[dict[str, Any]] = []
    for spec_id in self.specs or []:
        rels.append({"type": "belongs_to", "target": spec_id})
    if self.primary_spec:
        rels.append({"type": "primary_spec", "target": self.primary_spec})
    for delta_id in self.implemented_by or []:
        rels.append({"type": "implemented_by", "target": delta_id})
    for artifact_id in self.verified_by or []:
        rels.append({"type": "verified_by", "target": artifact_id})
    for artifact_id in self.coverage_evidence or []:
        rels.append({"type": "coverage_evidence", "target": artifact_id})
    return rels
```

#### P3-2: BacklogItem reference collection

`BacklogItem` references live in raw frontmatter, not structured fields. Rather than synthesising a `.relations` property with invented types (external review finding #2), extend `collect_references()` with a new collector:

**New collector in `relations/query.py`**:
```python
def _collect_from_backlog_fields(artifact: Any) -> list[ReferenceHit]:
    """Extract references from backlog-specific frontmatter fields."""
    hits: list[ReferenceHit] = []
    frontmatter = getattr(artifact, "frontmatter", None)
    if not isinstance(frontmatter, dict):
        return hits
    for delta_id in frontmatter.get("linked_deltas", []):
        target = str(delta_id).strip()
        if target:
            hits.append(ReferenceHit(target=target, source="backlog_field", detail="linked_delta"))
    for req_id in frontmatter.get("related_requirements", []):
        target = str(req_id).strip()
        if target:
            hits.append(ReferenceHit(target=target, source="backlog_field", detail="related_requirement"))
    return hits
```

Add to `collect_references()`:
```python
def collect_references(artifact: Any) -> list[ReferenceHit]:
    return (
        _collect_from_relations(artifact)
        + _collect_from_applies_to(artifact)
        + _collect_from_context_inputs(artifact)
        + _collect_from_informed_by(artifact)
        + _collect_from_backlog_fields(artifact)
    )
```

This preserves distinct provenance (`source="backlog_field"`) without inventing relation types. No magic strings. BacklogItem does NOT get a `.relations` property — its references are collected through the query layer's backlog-aware collector.

Also update P1-8 (`list backlog --related-to`) to use `matches_related_to()` instead of raw frontmatter access, so it picks up `linked_deltas`/`related_requirements` via the new collector.

### P4 — Generic Reverse Reference Filtering

#### P4-1: Query infrastructure

**New functions in `relations/query.py`**:

```python
def collect_reverse_reference_targets(
    referrers: Iterable[Any],
) -> set[str]:
    """Collect all target IDs referenced by a set of artifacts."""
    targets: set[str] = set()
    for referrer in referrers:
        for hit in collect_references(referrer):
            targets.add(hit.target.upper())
    return targets


def partition_by_reverse_references(
    candidates: Sequence[T],
    referrers: Iterable[Any],
    candidate_id_fn: Callable[[T], str] = lambda c: c.id,
) -> tuple[list[T], list[T]]:
    """Partition candidates into (referenced, unreferenced) by referrers.

    Returns a tuple of two lists:
      - candidates whose ID appears as a target in any referrer's references
      - candidates whose ID does not
    """
    targets = collect_reverse_reference_targets(referrers)
    referenced, unreferenced = [], []
    for c in candidates:
        bucket = referenced if candidate_id_fn(c).upper() in targets else unreferenced
        bucket.append(c)
    return referenced, unreferenced
```

**Registry resolution**: CLI resolves `<type>` string to a registry via a shared `load_all_artifacts(root, kind)` helper in `cli/common.py` (external review finding #3 — avoid a new CLI-side switchboard). This helper sits alongside the existing `_ARTIFACT_RESOLVERS` dispatch table and reuses the same registry knowledge:

```python
def load_all_artifacts(root: Path, artifact_type: str) -> list[Any]:
    """Load all artifacts of a given type for cross-registry queries."""
    if artifact_type in ("delta", "revision", "audit"):
        return list(ChangeRegistry(root=root, kind=artifact_type).iter())
    if artifact_type == "spec":
        return list(SpecRegistry(root=root).all_specs())
    if artifact_type in ("issue", "problem", "improvement", "risk"):
        return [i for i in BacklogRegistry(root=root).iter() if i.kind == artifact_type]
    if artifact_type == "backlog":
        return list(BacklogRegistry(root=root).iter())
    if artifact_type == "adr":
        return list(DecisionRegistry(root=root).iter())
    if artifact_type == "policy":
        return list(PolicyRegistry(root=root).iter())
    if artifact_type == "standard":
        return list(StandardRegistry(root=root).iter())
    if artifact_type == "requirement":
        return list(RequirementsRegistry(root=root).all())
    raise typer.BadParameter(f"Unknown artifact type: {artifact_type}")
```

**Filter application** (same pattern in every list command):
```python
if referenced_by or not_referenced_by:
    ref_type = referenced_by or not_referenced_by
    referrers = load_all_artifacts(repo_root, ref_type)
    referenced, unreferenced = partition_by_reverse_references(items, referrers)
    items = referenced if referenced_by else unreferenced
```

### P5 — Neighbourhood View (`--related`)

#### P5-1: `show <kind> <id> --related`

Adds a `--related` flag to `show spec`, `show delta`, `show requirement`, `show issue`.

**Scope justification** (external review finding #4): These are the four primary navigation anchors — specs define intent, deltas track change, requirements trace obligations, issues capture problems. `show adr --related`, `show revision --related`, etc. are useful but lower-traffic; deferred to a follow-up.

When `--related` is passed, the "Related:" counts section (P2) is replaced by the full itemised "Referenced by" section — not shown alongside it (internal review finding #3).

**Output** (appended after normal show output):
```
Referenced by:
  Deltas (9):
    DE-009  completed  CLI JSON consistency fixes
    DE-011  completed  CLI enhanced filtering and self-documentation
    ...
  Revisions (3):
    RE-024  completed  Delta DE-011 completion
    ...
  Audits (2):
    AUD-004  completed  DE-078 conformance: list audits subcommand
    ...
```

**Implementation**: Composition of P2 + P3 + P4 infrastructure:
1. `show` command checks `--related` flag
2. Loads registries relevant to the target kind (not all registries unconditionally — external review finding #6)
3. Forward references: `collect_references(artifact)` — what does this entity reference?
4. Reverse references: for each loaded registry, `find_related_to(all_artifacts, entity_id)` — what references this entity?
5. Groups results by kind

**Per-kind registry loading** (finding #6 — load only relevant registries):

| Target kind | Registries loaded for reverse lookup |
|-------------|-------------------------------------|
| spec | ChangeRegistry, DecisionRegistry, RequirementsRegistry |
| delta | ChangeRegistry (audits, revisions), BacklogRegistry |
| requirement | ChangeRegistry (deltas) |
| issue | ChangeRegistry (deltas) |

**Scope**: One hop only (DEC-090-12). Direct references, not transitive closure.

**Performance**: Acceptable for workspaces under ~500 artifacts (current corpus ~150). If latency exceeds 500ms, optimise by adding registry result caching.

**Formatter**: New `format_related_section()` in `relation_formatters.py`:

```python
def format_related_section(
    reverse_refs_by_kind: dict[str, list[tuple[str, str]]],  # kind -> [(id, name), ...]
) -> list[str]:
    """Format the 'Referenced by' section for --related output."""
    lines = []
    if reverse_refs_by_kind:
        lines.append("")
        lines.append("Referenced by:")
        for kind, refs in sorted(reverse_refs_by_kind.items()):
            kind_label = kind.replace("_", " ").title() + "s"
            lines.append(f"  {kind_label} ({len(refs)}):")
            for ref_id, ref_name in refs:
                lines.append(f"    {ref_id}  {ref_name}")
    return lines
```

## 7. Design Decisions & Trade-offs

- **DEC-090-01**: Display relation types raw (`relates_to`, `implements`) not humanized. Rationale: matches frontmatter, agents consume it, avoids mapping table.
- **DEC-090-02**: `show spec` requirements: count by default, `--requirements` flag for full list. Rationale: progressive disclosure; `view spec` shows full body.
- **DEC-090-03**: Plan parse errors warn to stderr and skip. Rationale: simpler than collecting; stderr doesn't pollute piped/JSON output.
- **DEC-090-04**: `--delta` filter on audits/revisions checks `relations` targets only. Audits/revisions reference deltas via relations, not a dedicated field.
- **DEC-090-05**: `--implemented-by` does single-delta lookup, not full reverse index. User specifies one delta; loading all deltas is unnecessary overhead.
- **DEC-090-06**: Reverse lookups use existing generic query functions (`find_related_to`) called from CLI layer; no new registry methods. `--related` graph walk composes the same functions across multiple registries.
- **DEC-090-07**: Audit↔delta linkage via `relations` with `type: documents`, not a dedicated frontmatter field. Consistent with how revisions already link to deltas. ADR-002 compatible (forward reference, not backlink).
- **DEC-090-08**: `show spec` shows reverse lookup counts in default view; itemised listing deferred to `--related`. Keeps default fast and focused.
- **DEC-090-09**: Domain models get `.relations` properties to expose domain-specific reference fields to the generic query layer. BacklogItem uses a dedicated collector (`_collect_from_backlog_fields`) instead, since its references live in raw frontmatter, not structured fields. No invented relation types — proper provenance via `source`/`detail` on `ReferenceHit`.
- **DEC-090-10**: Governance queries (`--unaudited`, `--unimplemented`) are convenience aliases for the generic `--not-referenced-by` mechanism. Same engine, friendlier surface.
- **DEC-090-11**: Cross-registry composition for `--referenced-by` / `--not-referenced-by` uses a shared `load_all_artifacts(root, kind)` helper in `cli/common.py`, avoiding a parallel switchboard. CLI is the composition point; avoids coupling registries to each other.
- **DEC-090-12**: `--related` shows one-hop references only (direct), not transitive closure. Keeps output predictable and avoids graph explosion.
- **DEC-090-13**: `--related` scoped to `show spec`, `show delta`, `show requirement`, `show issue` (primary navigation anchors). Other kinds deferred to follow-up.
- **DEC-090-14**: P2-3 (backlog JSON defaults) is an additive contract change — documented explicitly, not hidden under "no schema changes".
- **DEC-090-15**: `--related` loads only registries relevant to the target kind, not all registries unconditionally. Keeps latency proportional to neighbourhood, not corpus.
