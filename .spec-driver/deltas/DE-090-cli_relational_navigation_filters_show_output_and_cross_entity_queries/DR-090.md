# DR-090 – CLI relational navigation (P0–P1)

## 1. Executive Summary
- **Delta**: [DE-090](./DE-090.md)
- **Status**: draft
- **Last Updated**: 2026-03-13
- **Synopsis**: Fix relation display bugs, enrich `show spec` output, make `list plans` resilient to parse errors, and add 5 relational filter flags to existing list commands.

## 2. Problem & Constraints

### Current Behaviour
- `show delta` renders relations as `- : ISSUE-007` — relation type is lost (reads `"kind"` instead of `"type"`)
- `show spec` output is sparse — no relations, no requirements summary
- `list plans` crashes when any plan has malformed YAML in its `plan.overview` block
- Common governance queries require multi-command pipelines: "which audits cover this delta?", "which deltas touch this spec?", "which backlog items reference this entity?"

### Drivers / Inputs
- PROD-010 (CLI UX) — reverse relationship queries, filtering
- SPEC-110 (CLI specification) — consistent flag patterns
- ADR-009 — registry API convention

### Constraints / Guardrails
- POL-001: maximise reuse — use existing `find_related_to()`, `find_by_relation()` from `relations/query.py`
- POL-002: no magic strings — use `ARTIFACT_PREFIXES` and relation type constants
- STD-002: lint signal — zero new warnings
- Skinny CLI pattern: filters in CLI, no business logic

### Out of Scope (this DR revision)
- P2: richer `show` output (reverse lookups in show delta/spec)
- P3: governance queries (`--unaudited`, `--unimplemented`)
- P4: graph/neighbourhood view (`--related`)
- These will be covered in a subsequent DR revision before their implementation phases.

## 3. Architecture Intent

### Target Outcomes
- All relation types visible in `show delta` output (PROD-010.FR-005)
- `show spec` includes relations and requirement counts
- `list plans` degrades gracefully on parse errors
- Single-command answers to: "audits for this delta", "revisions for this delta", "deltas for this spec", "requirements implemented by this delta", "backlog items related to this entity"

### Guiding Principles
- Follow existing filter patterns — each new flag mirrors established `--spec`/`--related-to` implementations
- Pure functions for new formatter helpers
- No new registry methods needed — filter logic stays in CLI layer using existing query functions and field access
- Exception: `--implemented-by` requires loading a single delta to read its `requirements.implements` — this is a targeted lookup, not a registry extension

#### P0-2: Enrich `show spec`

New helpers in `spec_formatters.py`:

```python
def _format_spec_relations(spec: Spec) -> list[str]:
  """Format relations section for spec details."""
  relations = spec.frontmatter.get("relations", [])
  if not relations:
    return []
  lines = ["", "Relations:"]
  for rel in relations:
    rel_type = rel.get("type", "")
    target = rel.get("target", "")
    lines.append(f"  - {rel_type}: {target}")
  return lines


def _format_requirements_summary(spec: Spec) -> list[str]:
  """Format requirements count summary for spec details."""
  reqs = spec.frontmatter.get("requirements", [])
  if not reqs:
    return []
  fr_count = sum(1 for r in reqs if str(r.get("id", "")).startswith("FR-")
                 or ".FR-" in str(r.get("id", "")))
  nf_count = sum(1 for r in reqs if str(r.get("id", "")).startswith("NF-")
                 or ".NF-" in str(r.get("id", "")))
  parts = []
  if fr_count:
    parts.append(f"{fr_count} FR")
  if nf_count:
    parts.append(f"{nf_count} NF")
  other = len(reqs) - fr_count - nf_count
  if other:
    parts.append(f"{other} other")
  return ["", f"Requirements: {', '.join(parts)}"] if parts else []
```

Add both to `format_spec_details()` sections list, between packages and file path.

JSON output (`show spec --json`): include `relations` and `requirements` fields from `spec.frontmatter` in the dict. Verify `spec.to_dict()` already includes them; if not, add.

#### P1-4/5: `list audits --delta <ID>` / `list revisions --delta <ID>`

Add parameter (mirrors existing `--spec` pattern):

```python
delta: Annotated[
  str | None,
  typer.Option("--delta", "-d", help="Filter by delta reference"),
] = None,
```

Filter logic (after existing `--spec` filter):

```python
if delta:
  delta_upper = delta.upper()
  # Normalize bare numeric: "90" → "DE-090"
  delta_normalized = normalize_id("delta", delta_upper)
  items = [
    a for a in items
    if (
      a.relations
      and any(delta_normalized in str(rel.get("target", "")).upper()
              for rel in a.relations)
    )
    or str(getattr(a, "delta_ref", "")).upper() == delta_normalized
  ]
```

Note: `ChangeArtifact` may not expose `delta_ref` directly. Check whether it's in `frontmatter` or a model field. If frontmatter-only, access via `a.frontmatter.get("delta_ref", "")`.

Filter logic:

```python
if implemented_by:
  from supekku.scripts.lib.changes.registry import ChangeRegistry
  from supekku.scripts.lib.blocks.plan import extract_delta_relationships
  from supekku.cli.common import normalize_id

  delta_id = normalize_id("delta", implemented_by)
  delta_registry = ChangeRegistry(root=repo_root, kind="delta")
  delta_art = delta_registry.find(delta_id)
  if delta_art is None:
    typer.echo(f"Error: delta not found: {delta_id}", err=True)
    raise typer.Exit(EXIT_FAILURE)

  # Read implements list from delta.relationships block
  _, body = load_markdown_file(delta_art.path)
  rels_block = extract_delta_relationships(body, source_path=delta_art.path)
  impl_ids = set()
  if rels_block:
    impl_ids = {r.upper() for r in rels_block.data.get("requirements", {}).get("implements", [])}

  requirements = [r for r in requirements if r.id.upper() in impl_ids]
```

## 7. Design Decisions & Trade-offs

- **DEC-090-01**: Display relation types raw (`relates_to`, `implements`) not humanized. Rationale: matches frontmatter, agents consume it, avoids mapping table.
- **DEC-090-02**: `show spec` requirements count-only in rendered output; full list in JSON. Rationale: `view spec` shows full body; count is sufficient for discovery.
- **DEC-090-03**: Plan parse errors warn to stderr and skip. Rationale: simpler than collecting; stderr doesn't pollute piped/JSON output.
- **DEC-090-04**: `--delta` filter checks both `relations` targets and `delta_ref` frontmatter field. Rationale: DRs/IPs reference delta via `delta_ref` without always having a separate relation entry.
- **DEC-090-05**: `--implemented-by` does single-delta lookup, not full reverse index. Rationale: user specifies one delta; loading all deltas is unnecessary overhead.
