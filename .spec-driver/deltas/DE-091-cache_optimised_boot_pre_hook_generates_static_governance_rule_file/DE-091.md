## 1. Summary & Context

- **Technical Spec(s)**: SPEC-129 – sync installer-component
- **Implementation Plan**: [IP-091](./IP-091.md)
- **Change Drivers**: Boot sequence requires ~6 Read tool calls per session, busting token cache on every turn

## 2. Motivation

- The current boot skill triggers multiple Read tool calls to load governance files (dogma, exec, glossary, workflow, policy, memory, doctrine). Each call happens in conversation turns, outside the cacheable system prompt prefix.
- Claude Code's prompt caching only covers content loaded at session start: CLAUDE.md, AGENTS.md (with inlined `@` references), and unconditional `.claude/rules/` files.
- By generating a static rule file at pre-session time, all governance context moves into the cache-friendly prefix — eliminating per-session tool call overhead and improving token efficiency.
- See: `mem.fact.claude-code.context-loading`

## 3. Scope & Objectives

- **Primary Outcomes**:
  - `startup.sh` hook calls `spec-driver admin preboot` to generate `.agents/spec-driver-boot.md`
  - Installer creates symlink `.claude/rules/spec-driver-boot.md` → `../../.agents/spec-driver-boot.md`
  - Claude Code auto-loads the file into its cacheable prefix — zero tool calls for boot content
  - Boot skill becomes a lightweight validator: checks file exists (warns if missing), prints sigil, enforces routing
- **Operational Constraints**: Must be idempotent; must not break non-Claude-Code agent targets; user can disable by removing symlink
- **Dependencies**: None — this is additive to existing boot flow

## 4. Out of Scope

- Changing the governance file sources themselves (dogma, exec, glossary, etc.)
- Removing the boot skill entirely (it retains a validation role)
- Inlining full governance doc content (only listings)
- Modifying prompt caching behaviour in Claude Code (we work within its existing model)
- Fallback to old read-heavy boot path (if preboot file missing, boot warns and continues)

## 5. Approach Overview

- **System Touchpoints**: `supekku/scripts/lib/core/preboot.py`, `supekku/cli/admin.py`, `supekku/claude.hooks/startup.sh`, `supekku/scripts/install.py`, `supekku/skills/boot/SKILL.md`
- **Key Changes**:
  - New `spec-driver admin preboot` command: reads 7 boot-sequence files, shells out to CLI (respecting workflow.toml exec) for governance listings as TSV, concatenates into `.agents/spec-driver-boot.md`
  - Installer creates `.agents/` dir and `.claude/rules/spec-driver-boot.md` symlink
  - `startup.sh` calls preboot before emitting session JSON (errors swallowed)
  - Boot skill reduced to lightweight validator

## 6. Verification Strategy

- **Acceptance Criteria**:
  - VT: Generated file contains all 7 source files + 3 governance listings as TSV
  - VT: Running preboot twice produces identical output
  - VA: Fresh Claude Code session has boot context in prefix without Read tool calls
  - VA: Boot skill validates file and prints sigil without file reads
  - VA: Missing preboot file → boot skill warns and continues

## 7. Risks & Mitigations

- **Risk**: Generated file drifts from source governance files – _Likelihood_: medium – _Impact_: medium – _Mitigation_: Regenerated every session start via hook; also regenerated on `spec-driver install`
- **Risk**: Preboot fails silently – _Likelihood_: low – _Impact_: low – _Mitigation_: Boot skill validator warns if file missing
- **Risk**: Large concatenated file – _Likelihood_: low – _Impact_: medium – _Mitigation_: Governance files are compact by design; only listings, not full docs

## 8. Follow-ups & Tracking

- **Future Phases / Deltas**: Consider whether boot skill can be fully retired once cache-optimised path is proven stable
