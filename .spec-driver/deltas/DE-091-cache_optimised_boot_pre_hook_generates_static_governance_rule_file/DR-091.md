# DR-091 – Cache-optimised boot: pre-hook generates static governance rule file

## 1. Executive Summary
- **Delta**: [DE-091](./DE-091.md)
- **Status**: draft
- **Owners**: david
- **Last Updated**: 2026-03-12
- **Synopsis**: Replace per-session tool-call boot with a pre-generated static file loaded into Claude Code's cacheable prefix.

## 2. Problem & Constraints

### Current Behaviour
- `boot.md` agent template triggers `/boot` skill on every session start
- Boot skill contains `@` references to 7 governance files — these are **not** inlined by Claude Code (skill-level refs appear as literal text)
- Agent makes ~6 Read tool calls to load them; each tool call has a unique ID that busts token cache
- `policy.md` instructs agent to run 3 `spec-driver list` commands for governance artifact listings — more cache-busting tool calls
- Every turn after boot pays full re-encoding cost for this content

### Drivers
- `mem.fact.claude-code.context-loading` — documents the caching model
- `mem.pattern.installer.boot-architecture` — documents current boot flow
- ADR-005 — memories and skills are canonical guidance (boot content is guidance)

### Constraints
- Must be idempotent
- Must not break non-Claude-Code agent targets
- Must degrade gracefully if preboot fails (boot skill still works)
- Symlink managed by installer so user can disable by removing it

### Out of Scope
- Inlining full governance doc content (ADR/policy/standard bodies)
- Removing the boot skill entirely
- Modifying Claude Code's prompt caching behaviour

## 3. Architecture Intent

### Target Outcomes
- Zero tool calls during boot for governance content loading
- All boot-sequence content in Claude Code's cacheable prefix (stable across turns)
- Single generated file serves both Claude Code (via `.claude/rules/` symlink) and other agents (via `.agents/` + `@` ref in AGENTS.md)

### Guiding Principles
- Generation runs outside the agent session (shell hook, not tool call)
- Installer owns symlink lifecycle (deliberate opt-in, user can disable)
- Boot skill retains validation role (existence check + sigil + routing gate) — cheap, no file reads

### Data Flow

```
startup.sh (session start hook)
  └─→ uv run spec-driver admin preboot $PWD
        ├─→ reads .spec-driver/about/dogma.md
        ├─→ reads .spec-driver/agents/{exec,glossary,workflow,policy,memory}.md
        ├─→ reads .spec-driver/hooks/doctrine.md
        ├─→ runs spec-driver list adrs -s accepted --format=tsv
        ├─→ runs spec-driver list policies -s required --format=tsv
        ├─→ runs spec-driver list standards -s required --format=tsv
        └─→ writes .agents/spec-driver-boot.md

.claude/rules/spec-driver-boot.md  (symlink → ../../.agents/spec-driver-boot.md)
  └─→ Claude Code auto-loads into cacheable prefix at session start

Agent session starts:
  └─→ /boot skill checks preboot file exists
        ├─→ present: prints Δ ∴ ⊤, enforces routing
        └─→ missing: warns, prints Δ ∴ ⊤, enforces routing
```

## 7. Design Decisions & Trade-offs

- **DEC-091-1**: Output to `.agents/spec-driver-boot.md` with symlink from `.claude/rules/`. Rationale: single source file, consistent with skills symlink pattern, Claude Code loads via `.claude/rules/` auto-load, other agents via `@` ref.
- **DEC-091-2**: Installer manages the symlink, not preboot. Rationale: user can remove symlink to disable; preboot silently recreating it would be surprising.
- **DEC-091-3**: Boot skill becomes lightweight validator, not removed. No fallback to old read-heavy path — if preboot file is missing, warn and move on. Rationale: sigil confirmation and routing gate are cheap and useful as a visible handshake; maintaining two boot paths adds complexity for marginal benefit.
- **DEC-091-4**: Governance listings as TSV, not full doc content. Rationale: full docs are read on-demand based on relevance; listings give agents enough to know what exists and what to read.
- **DEC-091-5**: `startup.sh` calls preboot inline, errors swallowed with `2>/dev/null`. Rationale: failed preboot must not break session start; boot skill warns if file missing.
- **DEC-091-6**: Preboot shells out to CLI for governance listings (using exec command from workflow.toml) rather than importing registry APIs. Rationale: clean boundary, avoids crooked import paths.
- **DEC-091-7**: Output file at `.agents/spec-driver-boot.md` (repo root `.agents/` dir). ADR-006 concern is visible top-level dir pollution; `.agents/` is hidden and an established convention already used as a skills target.

## 8. Generated File Format

```markdown
<!-- Generated by spec-driver admin preboot — do not edit -->

# Spec-Driver Boot Context

## Dogma
<contents of .spec-driver/about/dogma.md>

## Execution Environment
<contents of .spec-driver/agents/exec.md>

## Glossary
<contents of .spec-driver/agents/glossary.md>

## Workflow
<contents of .spec-driver/agents/workflow.md>

## Policy & Governance
<contents of .spec-driver/agents/policy.md>

## Accepted ADRs
<TSV from spec-driver list adrs -s accepted --format=tsv>

## Required Policies
<TSV from spec-driver list policies -s required --format=tsv>

## Required Standards
<TSV from spec-driver list standards -s required --format=tsv>

## Memory
<contents of .spec-driver/agents/memory.md>

## Project Doctrine
<contents of .spec-driver/hooks/doctrine.md>
```

## 9. Rollout & Operational Notes
- **Migration**: Existing installs get the symlink on next `spec-driver install`. Preboot starts running on next session if `startup.sh` is updated (package hook source is installer-managed).
- **Recovery**: Remove the `.claude/rules/` symlink to revert to tool-call boot. Boot skill's fallback path handles this automatically.
