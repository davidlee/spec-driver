## 1. Summary & Context
- **Product Spec(s)**: PROD-016 – agentic sdlc configurator (agent harness is a core enabler)
- **Implementation Plan**: [IP-092](./IP-092.md)
- **Change Drivers**: spec-driver currently has no integrated coding agent — users must run pi (or other agents) separately and rely on skills/AGENTS.md for context injection. Embedding a jailed agent subprocess gives spec-driver direct control over agent lifecycle, context injection, and sandboxed execution.

## 2. Motivation
- spec-driver manages the full SDLC workflow (deltas, specs, audits, phases) but has no way to directly drive a coding agent
- Running agents separately means context injection is passive (AGENTS.md, skills) rather than active (dynamic delta state, governance gating)
- Untrusted LLM tool execution (bash, write) should be sandboxed; bubblewrap provides this on Linux with zero overhead
- pi already has a well-designed RPC mode (JSON lines on stdin/stdout) purpose-built for embedding
- The jailed-agents.nix infrastructure already exists and works

## 3. Scope & Objectives
- **Primary Outcomes**:
  - `spec-driver agent` command that launches a jailed pi subprocess in RPC mode
  - Python RPC client that speaks pi's JSONL protocol
  - Bubblewrap sandbox construction for non-Nix Linux systems
  - Sandbox discovery (jailed-pi on PATH → bwrap fallback → unsandboxed with warning)
  - Textual TUI widget for streaming agent interaction
  - pi extension (TypeScript) for spec-driver context injection
- **Operational Constraints**: macOS has no bubblewrap; agent runs unsandboxed there with a warning
- **Dependencies**: pi >= 0.57.1 (RPC mode), bubblewrap (Linux only)

## 4. Out of Scope
- Porting pi's agent loop or tools to Python
- Custom model registry or API key management (delegated to pi)
- Docker/container-based sandboxing as an alternative to bubblewrap
- macOS sandbox-exec integration (deprecated, undocumented)
- Tool authorisation gating based on delta state (future delta)

## 5. Approach Overview
- **System Touchpoints**: new `supekku/agent/` package, `.pi/extensions/spec-driver.ts`, Nix flake updates
- **Key Changes**:
  - Python RPC client managing pi subprocess lifecycle and JSONL protocol
  - Bubblewrap command builder for non-Nix Linux (replicating jail.nix essentials)
  - Agent binary discovery with tiered sandbox preference
  - Textual widget rendering streaming agent events
  - TypeScript extension auto-installed into projects for context injection
- **Architecture**:
  - spec-driver TUI (host) ↔ JSONL stdin/stdout ↔ jailed pi (sandbox)
  - Only the LLM agent runs in the sandbox; spec-driver TUI has full terminal access
  - API keys forwarded via env vars into the sandbox

## 6. Verification Strategy
- **Planned Artefacts**:
  - VT: RPC client protocol tests (mock subprocess)
  - VT: bwrap command builder output tests
  - VT: sandbox discovery tests (mock PATH scenarios)
  - VA: end-to-end agent session with jailed-pi
- **Acceptance Criteria**:
  - `spec-driver agent` launches pi in RPC mode and renders streaming output
  - On Linux with bwrap: agent runs sandboxed (no host filesystem access outside project dir)
  - On macOS or without bwrap: agent runs with clear warning
  - spec-driver extension injects active delta context into agent system prompt

## 7. Risks & Mitigations
- **Risk**: pi RPC protocol changes between versions – *Likelihood*: low – *Impact*: medium – *Mitigation*: pin minimum pi version, test against protocol types
- **Risk**: bwrap unavailable or restricted (e.g. unprivileged user namespaces disabled) – *Likelihood*: medium – *Impact*: low – *Mitigation*: graceful fallback to unsandboxed with warning
- **Risk**: Textual rendering of streaming markdown is complex – *Likelihood*: high – *Impact*: medium – *Mitigation*: start with plain text streaming, iterate on rendering
- **Risk**: Two-language boundary (Python↔TS extension) complicates debugging – *Likelihood*: medium – *Impact*: low – *Mitigation*: extension is minimal (~200 LOC), context injection testable independently

## 8. Follow-ups & Tracking
