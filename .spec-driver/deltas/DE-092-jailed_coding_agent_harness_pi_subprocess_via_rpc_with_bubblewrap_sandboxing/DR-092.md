# DR-092 – Jailed coding agent harness: pi subprocess via RPC with bubblewrap sandboxing

## 1. Executive Summary
- **Delta**: [DE-092](./DE-092.md)
- **Status**: draft
- **Last Updated**: 2026-03-14
- **Synopsis**: Embed pi as a sandboxed subprocess speaking JSONL, driven by a Python RPC client and rendered in Textual, with bubblewrap isolation on Linux.

## 2. Problem & Constraints

### Current Behaviour
spec-driver has no integrated coding agent. Users run pi (or other agents) independently. Context flows one-way via AGENTS.md and skills — spec-driver cannot actively inject delta state, gate tool use, or observe agent actions.

### Drivers
- Active context injection: system prompt should reflect current delta, phase, governance state
- Sandboxed execution: LLM-driven tool calls (bash, write) execute untrusted code
- Unified workflow: `spec-driver agent` as a single entry point for governed coding

### Constraints
- Must work on NixOS (with jail.nix) and non-Nix Linux (with raw bubblewrap)
- Must degrade gracefully on macOS (no bubblewrap available)
- Must not require porting pi's agent loop, tools, or model management to Python
- pi's RPC protocol is the integration surface — no internal API coupling

### Out of Scope
- Docker/Lima sandbox fallback
- macOS sandbox-exec
- Tool authorisation gating (future delta)
- Custom model registry

## 3. Architecture Intent

### System Topology

```
┌──────────────────────────────────────────────────┐
│  HOST                                            │
│                                                  │
│  spec-driver agent [--model X] [--continue]      │
│    │                                             │
│    ├─ discovery.py: find pi, detect sandbox      │
│    ├─ sandbox.py: build bwrap command            │
│    ├─ rpc_client.py: manage subprocess + JSONL   │
│    └─ agent_view.py: Textual widget              │
│         │                                        │
│         │ stdin/stdout (JSON lines)              │
│         ▼                                        │
│  ┌────────────────────────────────────────────┐  │
│  │  SANDBOX (bubblewrap)                      │  │
│  │                                            │  │
│  │  pi --mode rpc [--continue] [--session-dir]│  │
│  │    ├─ agent loop (tool calls)              │  │
│  │    ├─ reads .pi/extensions/spec-driver.ts  │  │
│  │    ├─ spec-driver CLI available on PATH    │  │
│  │    └─ project dir bind-mounted rw          │  │
│  └────────────────────────────────────────────┘  │
└──────────────────────────────────────────────────┘
```

### Target Outcomes
- `spec-driver agent` launches and manages a jailed pi subprocess
- Streaming agent output rendered in Textual TUI
- Active delta/phase context injected into agent system prompt
- spec-driver CLI callable by the agent as a tool inside the sandbox
- Sandboxed on Linux; warned but functional on macOS

### Guiding Principles
- **Subprocess, not library**: pi is a black box communicating via JSONL. No internal API coupling.
- **Host controls, sandbox executes**: TUI and orchestration on host; only LLM tool execution in sandbox.
- **Graceful degradation**: sandbox unavailability is a warning, not a blocker.
- **Minimal extension surface**: the pi extension is small (~200 LOC), injecting context and registering one tool.

## 4. Component Design

### 4.1 RPC Client (`supekku/agent/rpc_client.py`, ~300 LOC)

Manages pi subprocess lifecycle and speaks the JSONL protocol.

**Responsibilities:**
- Spawn pi subprocess with correct arguments
- Send `RpcCommand` objects as JSON lines to stdin
- Read `RpcResponse` and `AgentSessionEvent` objects from stdout
- Handle `extension_ui_request` events (proxy to Textual dialogs)
- Lifecycle: start, prompt, steer, abort, get_state, compact, shutdown

**Key types (mirroring pi's RPC protocol):**

```python
@dataclass
class RpcCommand:
    type: str           # prompt, steer, abort, get_state, set_model, ...
    id: str | None      # correlation ID
    # ... type-specific fields

@dataclass  
class RpcEvent:
    type: str           # message_update, tool_execution_start, ...
    # ... event-specific fields

class PiRpcClient:
    proc: subprocess.Popen
    
    async def send(self, cmd: RpcCommand) -> None
    async def events(self) -> AsyncIterator[dict]
    async def prompt(self, text: str) -> None
    async def steer(self, text: str) -> None
    async def abort(self) -> None
    async def get_state(self) -> dict
    async def set_model(self, provider: str, model_id: str) -> None
    async def compact(self, instructions: str | None = None) -> dict
    async def shutdown(self) -> None
```

**Protocol details:**
- Commands: JSON objects with `type` field, optional `id` for correlation
- Responses: `{"type": "response", "command": "...", "success": true/false, ...}`
- Events: streamed as they occur, same types as `AgentSessionEvent`
- Extension UI requests: `{"type": "extension_ui_request", "id": "...", "method": "select|confirm|input|notify", ...}`

### 4.2 Sandbox Builder (`supekku/agent/sandbox.py`, ~150 LOC)

Constructs bubblewrap invocations for non-Nix Linux. Replicates the essential jail.nix combinators in pure Python.

**Sandbox properties (matching `specDev` profile defaults):**
- Namespaces: unshare user, ipc, pid, uts, cgroup; share net (API calls need network)
- Filesystem: project dir bind-mounted rw at `/workspace/<name>`; system paths ro-bound
- Environment: cleared, then selective forwarding (API keys, TERM, PATH)
- Git identity: overridden to `clanker@local` (specDev profile; host identity in research/offline)
- Git push: SSH disabled via `GIT_SSH_COMMAND` override (all profiles)
- Process: `--die-with-parent`, `--new-session`
- Persisted home: `~/.local/share/jail.nix/home/agent` (specDev); profile-specific names for others

Note: jailed-agents.nix now supports **profiles** (`specDev`, `research`, `offline`) that control persistence, networking, and git policy. The `specDev` profile is the default and matches spec-driver's use case. The `sandbox.py` fallback for non-Nix Linux should replicate the `specDev` profile behaviour.

```python
class BwrapSandbox:
    def __init__(self, project_dir: str, agent_cmd: str = "pi"):
        ...
    
    def build_command(self, agent_args: list[str]) -> list[str]:
        """Build the full bwrap + pi command line."""
        ...
    
    @staticmethod
    def is_available() -> bool:
        """Check if bwrap is installed and functional."""
        ...
    
    @staticmethod
    def check_user_namespaces() -> bool:
        """Check if unprivileged user namespaces are enabled."""
        ...
```

**Nix vs non-Nix differences:**
| Aspect | Nix (jail.nix) | Non-Nix (sandbox.py) |
|---|---|---|
| Store paths | Closure-only `/nix/store` mounts | N/A |
| System binaries | Via `add-pkg-deps` closure | `--ro-bind /usr /usr` (coarser) |
| TLS certs | From nix store | `--ro-bind /etc/ssl /etc/ssl` |
| DNS | From nix store | `--ro-bind /etc/resolv.conf` |
| Persist home | `~/.local/share/jail.nix/home/<profile>` | `~/.local/share/spec-driver/agent-home/` |
| Profiles | `specDev`/`research`/`offline` via `profile` arg | single profile (matches `specDev`) |
| Network | profile-dependent (specDev/research: yes, offline: no) | always enabled |
| Git identity | profile-dependent (specDev: sandboxed, others: host) | sandboxed (`clanker`) |

### 4.3 Discovery (`supekku/agent/discovery.py`, ~100 LOC)

Tiered discovery of pi binary and sandbox capability.

**Discovery order:**
1. `jailed-pi` on PATH → Nix-managed, already sandboxed with profile (skip bwrap wrapping)
2. `pi` on PATH + `bwrap` available → wrap with `BwrapSandbox` (specDev-equivalent)
3. `pi` on PATH, no `bwrap` → run unsandboxed, emit warning
4. Nothing found → error with install instructions

```python
@dataclass
class AgentDiscovery:
    agent_cmd: list[str]    # full command to execute
    sandboxed: bool         # whether the agent is sandboxed
    sandbox_method: str     # "jail.nix" | "bwrap" | "none"
    profile: str | None     # jail.nix profile if detected (specDev, research, offline)
    warning: str | None     # warning message if unsandboxed

def discover_agent(project_dir: str) -> AgentDiscovery:
    ...
```

### 4.4 Textual Agent View (`supekku/tui/agent_view.py`, ~500 LOC)

Textual widget that renders streaming agent output and accepts user input.

**Renders:**
- Streaming assistant message text (markdown)
- Tool call start/end with name + arguments
- Tool result summaries (collapsed by default)
- Model/thinking level status
- Input editor for prompts

**Event mapping:**

| pi RPC Event | Widget Action |
|---|---|
| `message_update` | Append text to streaming message area |
| `message_end` | Finalise message, enable input |
| `tool_execution_start` | Show tool name + args in collapsible block |
| `tool_execution_end` | Show result summary, collapse |
| `auto_compaction_start` | Show "compacting..." indicator |
| `auto_retry_start` | Show retry countdown |
| `extension_ui_request` | Proxy to Textual dialog (select/confirm/input) |

### 4.5 CLI Subcommand (`supekku/cli/agent.py`, ~100 LOC)

```
spec-driver agent [--model PROVIDER/MODEL] [--continue] [--no-sandbox] [--thinking LEVEL]
```

- `--model`: passed through to pi as `--model`
- `--continue`: resume last session
- `--no-sandbox`: force unsandboxed execution
- `--thinking`: set thinking level (low/medium/high)

### 4.6 pi Extension (`.pi/extensions/spec-driver.ts`, ~200 LOC)

Auto-loaded by pi when running inside the project. Provides spec-driver-aware context injection.

**Hooks:**
- `before_agent_start`: inject active delta context, phase state, and governance rules into system prompt
- `tool_result`: observe file changes for delta auditing (future: gate writes)
- `session_start`: load and display active delta summary

**Registered tool:**
- `spec_driver`: wraps `spec-driver` CLI, allowing the LLM to create/show/edit spec-driver entities

## 5. Design Decisions

### DEC-092-01: pi as subprocess, not Python port
**Rationale**: pi has ~15k LOC of well-tested agent infrastructure (tools, session management, compaction, model registry, extension system). Porting to Python would be months of work and permanent drift from upstream. The RPC protocol is stable and purpose-built for embedding.
**Consequence**: Two-language boundary. Extension must be TypeScript. Debugging crosses process/language lines.

### DEC-092-02: Only the LLM agent is sandboxed
**Rationale**: The TUI needs full terminal access (raw mode, signals, clipboard). Jailing it adds complexity for no security benefit. The threat model is untrusted LLM-generated tool calls (bash commands, file writes), not the spec-driver orchestrator.
**Consequence**: API keys must be forwarded into the sandbox via environment variables.

### DEC-092-03: bubblewrap on Linux, unsandboxed on macOS
**Rationale**: bubblewrap uses Linux namespaces — no macOS equivalent exists that's practical. macOS sandbox-exec is deprecated and undocumented. Docker/Lima is too heavy for a dev tool subprocess.
**Consequence**: macOS users get a warning. This matches the threat model: dev machines with single-user access are lower risk than shared/CI environments.

### DEC-092-04: Tiered sandbox discovery
**Rationale**: Nix users already have `jailed-pi` with proper closure isolation. Non-Nix Linux users can use raw bubblewrap. The fallback to unsandboxed keeps the tool functional everywhere.
**Consequence**: Three code paths to test, but each is simple. The `AgentDiscovery` result type makes the active path explicit.

## 6. Open Questions

- [ ] **Session storage location**: should pi sessions live in `.spec-driver/sessions/` (in-project, version-controllable) or `~/.pi/agent/sessions/` (pi default, persisted home in sandbox)? In-project is simpler but adds noise to the repo. — Owner: @davidlee
- [ ] **Extension auto-generation**: should `spec-driver init` create `.pi/extensions/spec-driver.ts`, or should it be shipped as a project template? Auto-generation risks staleness; template requires manual setup. — Owner: @davidlee
- [ ] **Minimum pi version policy**: pin exact version or semver range? The RPC protocol has no formal stability guarantee. — Owner: @davidlee

## 7. Rollout & Operational Notes

### Phase 1: Foundation
- RPC client, sandbox builder, discovery
- Minimal TUI (plain text streaming, no markdown rendering)
- Extension with context injection

### Phase 2: Polish
- Markdown rendering in Textual
- Tool call/result display
- Model/thinking level switching from TUI
- Session management (continue, new, compact)

### Phase 3: Integration
- Nix flake `makeJailedPiForSpecDriver` export (using `specDev` profile by default)
- Profile passthrough: `spec-driver agent --profile research` → selects correct jailed-pi variant
- `spec-driver init` extension scaffolding
- Documentation
