# DR-093 – pi extension: preboot hook for fresh governance context

## 1. Executive Summary

- **Delta**: [DE-093](./DE-093.md)
- **Status**: draft
- **Synopsis**: Ensure `spec-driver admin preboot` runs before pi reads AGENTS.md, both at launch and on `/reload`, using a Nix wrapper and a pi extension.

## 2. Problem & Constraints

### Current Behaviour

`spec-driver admin preboot` generates `.agents/spec-driver-boot.md` which is inlined into AGENTS.md. This runs manually or via Claude Code's startup hook. When using pi, there is no equivalent hook — the preboot content may be stale.

### Constraints

- System prompt must be stable across turns for Anthropic token cache efficiency
- Preboot must complete _before_ pi reads AGENTS.md (both at launch and on `/reload`)
- Extension events available: `session_start` (too late — fires after resource load), `session_shutdown` (fires before resource reload — correct timing)
- Must work inside bubblewrap jail (spec-driver on PATH via extraPkgs)

## 3. Design

### 3.1 Belt: Nix Wrapper

Runs on the host, before the jail starts:

```nix
# In flake.nix
jailed-pi-preboot = pkgs.writeShellScriptBin "jailed-pi" ''
  ${spec-driver}/bin/spec-driver admin preboot "$PWD" >/dev/null 2>&1
  exec ${jailPkgs.jailed-pi}/bin/jailed-pi "$@"
'';
```

- Runs preboot against the host project directory
- Regenerates `.agents/spec-driver-boot.md`
- Then launches jailed-pi which reads the now-fresh AGENTS.md
- Errors swallowed (preboot failure shouldn't block the agent)

### 3.2 Suspenders: pi Extension

Runs inside the jail on `/reload`:

```typescript
// .pi/extensions/spec-driver-preboot.ts
import type { ExtensionFactory } from "@mariozechner/pi-coding-agent";

const extension: ExtensionFactory = (pi) => {
  pi.on("session_shutdown", async (_event, ctx) => {
    await pi.exec("spec-driver", ["admin", "preboot", ctx.cwd]);
  });
};

export default extension;
```

**Why `session_shutdown`:**

pi's `/reload` sequence:

1. `await extensionRunner.emit({ type: "session_shutdown" })` — **extension runs preboot here**
2. `settingsManager.reload()`
3. `await resourceLoader.reload()` — **re-reads AGENTS.md here (now fresh)**
4. `_buildRuntime()` — rebuilds system prompt
5. `await extensionRunner.emit({ type: "session_start" })`

The `session_shutdown` handler is awaited before the resource reload, so preboot completes before AGENTS.md is re-read.

**On actual exit:** `session_shutdown` also fires on process exit. Preboot runs (~100ms), is idempotent, and leaves the file fresh for the next launch. Harmless.

**Failure handling:** if `spec-driver` is not on PATH or preboot fails, `pi.exec()` returns a non-zero exit code. The extension silently ignores it — the Nix wrapper (belt) already ran preboot at launch, so the content is at worst stale rather than missing.

### 3.3 Cache Safety

Preboot output is deterministic given the same project state:

- Same ADRs, policies, standards, memories → byte-identical output
- If nothing changed between `/reload` calls, AGENTS.md is unchanged, system prompt is unchanged, cache prefix holds
- Only when project governance state actually changes does the output differ, which is exactly when a cache bust is _correct_

## 4. Design Decisions

### DEC-093-01: Belt and suspenders

**Rationale**: Neither mechanism alone covers both cases. The Nix wrapper can't help on `/reload` (pi is already running). The extension can't help at startup (extensions load after AGENTS.md). Together they cover both.

### DEC-093-02: Hook session_shutdown, not session_start

**Rationale**: `session_start` fires _after_ `resourceLoader.reload()` — too late. `session_shutdown` fires _before_ it, and is awaited, giving preboot time to complete.

### DEC-093-03: No per-turn injection

**Rationale**: Using `before_agent_start` to inject/replace the system prompt every turn would bust the token cache on every prompt. The system prompt must be stable. Preboot runs only at launch and on explicit `/reload`.
