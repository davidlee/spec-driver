## 1. Summary & Context
- **Prior art**: DE-060 — Claude Code `PostToolUse` hook (`artifact_event.py`)
  that classifies file paths against spec-driver artifact patterns and emits
  events to JSONL log + Unix socket for TUI real-time tracking.
- **Infrastructure**: DE-093 — pi extension install pipeline (`_install_pi_config`),
  `pi.extensions/` package source, preboot extension.
- **Implementation Plan**: [IP-094](./IP-094.md)

## 2. Motivation
- The spec-driver TUI tracks artifact touches in real time (which deltas, specs,
  phases the agent is working on). This works for Claude Code via `artifact_event.py`.
- pi has no equivalent — pi sessions are invisible to the TUI.
- pi's extension API provides `tool_result` events with typed tool input (including
  file paths for read/edit/write), making in-process observation possible without
  shell-out overhead.

## 3. Scope & Objectives
- **Primary Outcomes**:
  - pi extension that observes `tool_result` events for read/edit/write tools
  - Classifies file paths against spec-driver artifact patterns (same as DE-060)
  - Emits v1 events to `.spec-driver/run/events.jsonl` + `tui.sock` Unix datagram
  - Installed automatically via `spec-driver install` (DE-093 pipeline)
- **Operational Constraints**: Must be fail-silent — never interfere with pi operation

## 4. Out of Scope
- Bash tool observation (no file path in input)
- Session ID tracking (pi doesn't expose session IDs in the same way)
- Changes to the TUI consumer — it already reads the event format

## 5. Approach Overview
- **Pure TypeScript** implementation in `supekku/pi.extensions/`
- Reimplement `classify_path` regex patterns and `build_event` schema in TS
- Use Node.js `fs.appendFileSync` for JSONL, `dgram` for Unix socket
- Single extension file, ~80-100 LOC
- **Key design decisions**:
  - In-process (no Python shell-out) — zero per-tool-call overhead
  - Same event schema v1 as `artifact_event.py` — TUI consumer is format-agnostic
  - Same artifact pattern list — kept in sync manually (patterns are stable)

## 6. Verification Strategy
- Unit tests for path classification (port from `artifact_event_test.py`)
- Unit tests for event schema
- Integration: verify JSONL output from mocked tool_result events

## 7. Risks & Mitigations
- **Risk**: Pattern drift between Python and TS copies – *Likelihood*: low
  (patterns change rarely) – *Impact*: low (missed classification, not breakage)
  – *Mitigation*: comment cross-reference to `artifact_event.py`
- **Risk**: Unix socket not available on all platforms – *Likelihood*: low
  (TUI is Unix-only) – *Impact*: low – *Mitigation*: fail-silent, log-only fallback

## 8. Follow-ups & Tracking
- Session ID propagation (if pi exposes it in future API)
- Pattern deduplication (shared pattern file) if drift becomes a problem
