## 1. Summary & Context

The base frontmatter schema (`BASE_FRONTMATTER_METADATA`) already defines `tags`, `ext_id`, and `ext_url` as optional fields. The requirement frontmatter metadata inherits them and even shows them in examples. However:

1. **Creation gap**: `create_requirement_breakout()` doesn't emit these fields in generated frontmatter, and the CLI `create requirement` command doesn't accept `--tags`, `--ext-id`, or `--ext-url` flags.
2. **Sync gap**: `RequirementsRegistry._records_from_content()` extracts requirements from body regex only. When breakout requirement files exist with frontmatter containing `tags`/`ext_id`/`ext_url`, those values are ignored during sync. Tags are only captured from inline `[tag1, tag2]` syntax in spec body bullet points.

The `RequirementRecord` dataclass already has `tags`, `ext_id`, `ext_url` fields and handles their serialisation correctly.

- **Product Spec(s)**: PROD-008 – Requirements Lifecycle Coherence, PROD-009 – Requirement Lifecycle Semantics
- **Implementation Plan**: [IP-095](./IP-095.md)

## 2. Motivation

Users cannot set `tags`, `ext_id`, or `ext_url` on requirements through the CLI creation flow, and even if they add them manually to frontmatter, the sync pipeline silently drops them. This makes it impossible to:

- Tag requirements for filtering/categorisation via the registry
- Link requirements to external systems (Jira, Linear, GitHub Issues)
- Use `--show-external` in list output for requirements

Other artifact types (specs, deltas, policies, standards, backlog items) all correctly read `ext_id`/`ext_url` from frontmatter during sync.

## 3. Scope & Objectives

- **Primary Outcomes**:
  1. `create requirement` CLI accepts `--tags`, `--ext-id`, `--ext-url` and emits them in frontmatter
  2. Sync reads `tags`, `ext_id`, `ext_url` from requirement breakout file frontmatter
  3. Inline `[tag]` syntax in spec body continues to work and merges with frontmatter tags
- **Operational Constraints**: Must not break existing requirement extraction from spec body bullets

## 4. Out of Scope

- Adding frontmatter-level tags/ext to non-requirement artifacts (already works)
- Registry YAML schema changes (already supports these fields)
- Formatter changes (already support ext_id/ext_url display)

## 5. Approach Overview

- **System Touchpoints**:
  - `supekku/scripts/lib/changes/creation.py` – `create_requirement_breakout()`
  - `supekku/scripts/create_requirement.py` – CLI arg parser
  - `supekku/cli/create.py` – CLI create subcommand
  - `supekku/scripts/lib/requirements/registry.py` – `_records_from_content()` / `_records_from_frontmatter()`
- **Key Changes**:
  1. Add `--tags`, `--ext-id`, `--ext-url` params to creation pipeline
  2. Read `tags`/`ext_id`/`ext_url` from frontmatter when processing breakout requirement files and merge into extracted `RequirementRecord`

## 6. Verification Strategy

- Unit tests for creation with new params
- Unit tests for sync reading frontmatter fields on breakout files
- Merge test: inline tags + frontmatter tags = union
- Existing tests remain green

## 7. Risks & Mitigations

- **Risk**: Frontmatter tags conflict with inline tags – _Likelihood_: low – _Impact_: low – _Mitigation_: Merge strategy (union of both sources, sorted)
- **Risk**: Breakout files without frontmatter tags break – _Likelihood_: low – _Impact_: low – _Mitigation_: All fields are optional, defaulting to empty
