# DR-095 – Requirement frontmatter: populate tags, ext_id, ext_url in creation and sync

## 1. Executive Summary

- **Delta**: [DE-095](./DE-095.md)
- **Status**: draft
- **Last Updated**: 2026-03-14
- **Synopsis**: Wire `tags`, `ext_id`, `ext_url` through requirement creation CLI and ensure sync reads them from breakout file frontmatter.

## 2. Problem & Constraints

- **Current Behaviour**: `create requirement` emits frontmatter without `tags`/`ext_id`/`ext_url`. The sync pipeline extracts requirements from body regex only, ignoring these frontmatter fields on breakout files. The `RequirementRecord` model and formatters already support these fields.
- **Constraints**: Must not break inline `[tag1, tag2]` extraction from spec body bullets. All three fields remain optional.

## 3. Architecture Intent

- **Target Outcomes**:
  1. `create requirement --tags a,b --ext-id JIRA-123 --ext-url https://...` emits populated frontmatter
  2. Sync merges frontmatter `tags` with inline tags (union, sorted)
  3. Sync reads `ext_id`/`ext_url` from breakout file frontmatter into `RequirementRecord`
- **Guiding Principles**: Follow the existing pattern used by policies, standards, and backlog items which all read `ext_id`/`ext_url` from frontmatter.

## 4. Code Impact Summary

| Path                                           | Current State                                                                                                                  | Target State                                                                                                                                           |
| ---------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `supekku/scripts/lib/changes/creation.py`      | `create_requirement_breakout()` accepts `spec_id, requirement_id, title, kind`                                                 | Add `tags`, `ext_id`, `ext_url` params; emit in frontmatter dict when non-empty                                                                        |
| `supekku/scripts/create_requirement.py`        | Parser has `spec, requirement, title, --kind`                                                                                  | Add `--tags`, `--ext-id`, `--ext-url`                                                                                                                  |
| `supekku/cli/create.py`                        | Passes `spec, requirement, title, kind`                                                                                        | Pass new args through                                                                                                                                  |
| `supekku/scripts/lib/requirements/registry.py` | `_records_from_content()` builds records from regex only; `_records_from_frontmatter()` delegates without reading these fields | Read `tags`/`ext_id`/`ext_url` from frontmatter mapping, pass to `RequirementRecord` constructor. For tags: union of frontmatter tags and inline tags. |

## 5. Design Decisions

- **DEC-001 – Tag merge strategy**: Frontmatter `tags` and inline `[tag1, tag2]` are merged via sorted set union. This is consistent with how `RequirementRecord.merge()` already handles tags when combining records from multiple sources.
