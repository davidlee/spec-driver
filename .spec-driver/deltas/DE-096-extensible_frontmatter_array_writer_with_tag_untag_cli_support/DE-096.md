## 1. Summary & Context

The frontmatter writer (`supekku/scripts/lib/core/frontmatter_writer.py`) currently handles only scalar field updates via line-level replacement. There is no support for array operations ŌĆö add item, remove item, or set array. This blocks CLI `--tag`/`--untag` on edit commands and will block future array-field operations (owners, relations, etc).

- **Product Spec**: PROD-010 ŌĆō CLI UX
- **Prior work**: DE-095 added `--tag` to `create requirement` but edit commands have no tag support at all
- **Implementation Plan**: [IP-096](./IP-096.md)

## 2. Motivation

Every `edit` subcommand only supports `--status`. Users must open an editor to add or remove tags. A solid array writer primitive enables:
- `--tag`/`--untag` on all edit commands (immediate use)
- Future `--owner`/`--unowner`, `--ext-id`, `--ext-url` on edit commands
- Any frontmatter array manipulation from CLI without full YAML round-trip

## 3. Scope & Objectives

**Phase 1 ŌĆö Writer infrastructure**:
- Line-level array field operations in `frontmatter_writer.py`:
  - `add_frontmatter_list_items(path, field, items)` ŌĆö add items to array (create if absent)
  - `remove_frontmatter_list_items(path, field, items)` ŌĆö remove items from array
- Must handle both YAML flow-style `[a, b, c]` and block-style `- a\n- b` arrays
- Preserves all other content byte-for-byte (existing writer contract)
- Deduplicates and sorts tags (convention: tags are always sorted)

**Phase 2 ŌĆö CLI wiring**:
- Add `--tag`/`--untag` options to all `edit` subcommands
- Common option definitions (DRY ŌĆö single definition, reused across commands)
- Wire through `add_frontmatter_list_items` / `remove_frontmatter_list_items`
- Update `updated:` date on any change

## 4. Out of Scope

- `--tag` on remaining `create` commands (mechanical follow-up)
- Other array fields (`--owner`, `--relation`) ŌĆö future deltas, same writer
- Full YAML round-trip rewrite

## 5. Approach Overview

**Writer design**: Extend the existing line-level approach. Parse frontmatter boundaries, locate the target field, determine if it's flow-style or block-style, apply the operation, emit the result. New arrays default to flow-style `[a, b, c]` matching project convention.

**CLI design**: Define `TagOption` and `UntagOption` as reusable `Annotated` types (like the existing `StatusOption`). Each edit command adds them and calls a shared `_apply_tags()` helper.

## 6. Verification Strategy

- Unit tests for writer: flow-style add/remove, block-style add/remove, create-when-absent, empty-after-remove, dedup, idempotent, body preservation
- Unit tests for CLI: edit with --tag, edit with --untag, combined
- All existing writer and edit tests remain green

## 7. Risks & Mitigations

- **Risk**: Multi-line YAML arrays with comments or complex values ŌĆō *Likelihood*: low (tags are simple strings) ŌĆō *Mitigation*: Handle the two standard forms (flow `[]` and block `- item`); reject/warn on unrecognised formats
- **Risk**: Quoted tag values ŌĆō *Likelihood*: low ŌĆō *Mitigation*: Strip and normalise, emit unquoted in flow style
