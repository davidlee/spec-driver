# DR-096 – Extensible frontmatter array writer with tag/untag CLI

## 1. Executive Summary

- **Delta**: [DE-096](./DE-096.md)
- **Status**: draft
- **Last Updated**: 2026-03-14
- **Synopsis**: Line-level array operations for frontmatter, wired to `--tag`/`--untag` on all edit commands.

## 2. Current State

`frontmatter_writer.py` has two functions:

- `update_frontmatter_status(path, status)` — replaces `status:` line, bumps `updated:`
- `update_frontmatter_fields(path, updates)` — replaces/inserts scalar `key: value` lines

Both are line-level (no YAML parse/dump round-trip), preserving file content.

No array support exists. Tags in frontmatter appear as either:

- Flow: `tags: [auth, security, oauth2]`
- Block: `tags:\n- auth\n- security`

Project convention favours flow-style for short lists like tags.

## 3. Writer API Design

### New public functions

```python
@dataclass
class ListUpdateResult:
  field: str
  added: list[str]
  removed: list[str]
  final: list[str]

def add_frontmatter_list_items(
  path: Path,
  field: str,
  items: list[str],
  *,
  sort: bool = True,
) -> ListUpdateResult:
  """Add items to a frontmatter list field. Creates field if absent.
  Deduplicates. Bumps updated date."""

def remove_frontmatter_list_items(
  path: Path,
  field: str,
  items: list[str],
) -> ListUpdateResult:
  """Remove items from a frontmatter list field. No-op for missing items.
  Bumps updated date."""
```

### Internal approach

1. Read file, find frontmatter boundaries (reuse existing `---` detection)
2. Locate target field line within frontmatter
3. Determine format:
   - **Flow**: single line matching `field: [...]` — parse items, modify, re-emit as `field: [a, b, c]`
   - **Block**: `field:` line followed by `- item` lines — parse items, modify, re-emit in same format
   - **Absent**: insert as flow-style `field: [a, b, c]` before closing `---`
4. Apply operation (add with dedup, or remove)
5. Sort if requested (default for tags)
6. Bump `updated:` date
7. Write back

### Format preservation

- If existing field is flow-style → emit flow-style
- If existing field is block-style → emit block-style
- New fields → flow-style (project convention)
- Empty list after removal → emit `field: []`

## 4. CLI Design

### Shared option types in `edit.py`

```python
TagOption = Annotated[
  list[str] | None,
  typer.Option("--tag", "-t", help="Add tag (repeatable)"),
]
UntagOption = Annotated[
  list[str] | None,
  typer.Option("--untag", help="Remove tag (repeatable)"),
]
```

### Shared helper

```python
def _apply_tags(
  artifact_id: str,
  path: Path,
  tags: list[str] | None,
  untags: list[str] | None,
) -> None:
  if tags:
    result = add_frontmatter_list_items(path, "tags", tags)
    typer.echo(f"Added tags to {artifact_id}: {', '.join(result.added)}")
  if untags:
    result = remove_frontmatter_list_items(path, "tags", untags)
    typer.echo(f"Removed tags from {artifact_id}: {', '.join(result.removed)}")
```

### Edit command changes

Each edit command gains `tag: TagOption = None, untag: UntagOption = None` params. When either is provided, call `_apply_tags()` and skip editor (same pattern as `--status`).
