## 1. Summary & Context

- **Implementation Plan**: [IP-097](./IP-097.md) – draft
- **Change Drivers**: ISSUE-031 (graph index gaps), ISSUE-035 (memory CLI UX — remaining items)
- **Prior Art**: DE-045 (completed: `--links-to`, `--links-depth`, `--body-only`), DE-073 (completed: `--content-type`, `resolve links --verbose/--path/--id`)

## 2. Motivation

Current registries are domain-local. Cross-artifact relation queries require
multi-registry scans. There is no unified graph index, no canonical relation
normalization, and no deterministic diagnostics for unresolved canonical IDs.

DE-045 and DE-073 delivered point solutions (backlinks within memory, scoped
link resolution). This delta addresses the structural gap: a derived graph
registry that normalizes edges across all artifact types and enables generic
forward/inverse traversal queries.

ISSUE-035's remaining undelivered item — `--truncate` rendering artefacts in
`list memories` Status column — is a minor fix bundled here for convenience.

## 3. Scope & Objectives

- **Primary Outcomes**:
  1. Derived graph registry (`.spec-driver/registry/graph.yaml`) generated deterministically from repo state
  2. Canonical relation normalization: unified edge representation across `relations`, typed arrays (`policies`, `standards`, `specs`), and wikilinks
  3. At least one forward and one inverse relation query implemented and tested (e.g., `--related <ID>`, `--related-by <ID>`)
  4. Deterministic diagnostics for unresolved canonical IDs (warning vs error semantics, strict mode)
  5. Identifier/path normalization for padded IDs, aliases, and path refs
  6. Fix `--truncate` rendering in `list memories` Status column
- **Operational Constraints**: Must not break existing `--links-to` / `--links-depth` behaviour (DE-045)
- **Dependencies**: None (DE-045 and DE-073 already completed)

## 4. Out of Scope

- Performance optimisation for large corpora (current scale ~35 memories, ~100 links — not a concern)
- Full graph visualisation / `dot` export (future delta)
- Replacing existing domain-local registries with graph registry (graph is derived/supplemental per ADR-007)

## 5. Approach Overview

- **System Touchpoints**: `supekku/scripts/lib/memory/`, `supekku/scripts/lib/registry/`, `cli/resolve.py`, `cli/list.py`, `cli/show.py`
- **Key Changes**:
  1. Graph builder: traverse all registries, extract normalized edges (source, relation_type, target), write `registry/graph.yaml`
  2. Query API: `graph.query(id, direction=forward|inverse|both)` returning typed edge lists
  3. CLI flags: `--related <ID>` / `--related-by <ID>` on relevant `list`/`show` commands
  4. Validator: unresolved canonical ID detection with configurable severity
  5. ID normalizer: handle zero-padding, aliases, path-to-ID resolution
  6. Fix truncation rendering bug in `list memories`

## 6. Verification Strategy

- **VT-097-graph-build**: Graph registry generation is deterministic and complete
- **VT-097-query**: Forward and inverse queries return correct edges
- **VT-097-normalize**: ID normalization handles padding, aliases, paths
- **VT-097-diagnostics**: Unresolved IDs produce correct warning/error output
- **VT-097-truncate**: `list memories --truncate` renders Status column correctly
- **Acceptance Criteria**:
  - Graph registry can be regenerated from repo state and matches expected output
  - At least one forward and one inverse query work end-to-end via CLI
  - Existing `--links-to` / `--links-depth` behaviour unchanged
  - Zero regressions in existing test suite

## 7. Risks & Mitigations

- **Risk**: Relation surface heterogeneity makes normalization fragile – _Likelihood_: medium – _Impact_: medium – _Mitigation_: Start with explicit mapping table; reject unknown relation types rather than guessing
- **Risk**: Graph registry staleness if not regenerated on changes – _Likelihood_: low – _Impact_: low – _Mitigation_: Generate on demand (sync command), not persisted as source of truth
