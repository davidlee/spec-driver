# DR-097 – Cross-artifact graph registry and relation traversal

## 1. Executive Summary

- **Delta**: [DE-097](./DE-097.md)
- **Status**: draft
- **Last Updated**: 2026-03-15
- **Synopsis**: Build on-demand cross-artifact reference graph from existing registries and `collect_references`, add `show relations <ID>` for cross-type neighbourhood queries, registry-aware ID normalization with diagnostics, and unresolved frontmatter reference validation.

## 2. Problem & Constraints

### Current Behaviour

- Registries are domain-local. No unified view of the reference graph.
- `collect_references` (relations/query.py) extracts forward references from any artifact but results are per-artifact, not aggregated.
- `build_artifact_index` (cli/resolve.py) collects all artifact IDs but not edges.
- `--related-to` on `list` commands answers "which Y-type artifacts reference X" — single-type inverse queries only.
- No cross-type inverse query: "given X, find ALL artifacts that reference X."
- ID normalization exists only for memory IDs. Non-canonical forms (e.g., `ADR-11` for `ADR-011`) silently fail to match.
- Unresolved reference validation exists for requirement lifecycle links and decision references in `WorkspaceValidator`, but not for general frontmatter references across all artifact types.

### Drivers / Inputs

- ISSUE-031: Graph index and inverse relation traversal gaps
- ISSUE-035: Memory CLI UX improvements (remaining items)
- ADR-002: Backlinks computed at runtime, not stored in frontmatter
- ADR-007: Derived artefacts (graph is derived observed corpus)

### Constraints / Guardrails

- Graph must be computed at runtime from forward references (ADR-002)
- Graph is derived, not authoritative (ADR-007)
- Must not break existing `--related-to` or `--links-to` / `--links-depth` behaviour
- Must not modify frontmatter to store backlinks
- ID normalization must not assume a fixed digit width (future 4-digit IDs)

### Out of Scope

- Persisted `graph.yaml` registry file (architecture supports later addition)
- Graph visualisation / DOT export
- Replacing domain-local registries
- Body wikilink validation in non-memory files
- Performance optimisation (current scale: ~hundreds of artifacts)

## 3. Architecture Intent

### Target Outcomes

1. **Unified reference graph** — pure function `build_reference_graph(workspace) → ReferenceGraph` that aggregates all forward references into a queryable structure
2. **Cross-type neighbourhood queries** — `show relations <ID>` answers "what does X reference?" and "what references X?" across all artifact types
3. **ID normalization** — registry-aware padding resolution (`ADR-11` → `ADR-011`) with diagnostic warnings on non-canonical forms
4. **Unresolved reference validation** — frontmatter references that point to nothing produce deterministic diagnostics (warn by default, error with `--strict`)

### Guiding Principles

- **Reuse over new**: build on `collect_references`, `build_artifact_index`, and `Workspace` — no parallel implementations
- **Pure functions**: graph building and querying are pure; I/O at the edges only
- **Single responsibility**: graph module does graph ops; validation module does validation; CLI wires them

## 4. Code Impact Summary

### 4.1 New: `supekku/scripts/lib/relations/graph.py`

On-demand cross-artifact reference graph builder and query API.

**Data types:**

```python
@dataclass(frozen=True)
class GraphEdge:
  """A directed reference edge in the artifact graph."""
  source: str       # source artifact ID
  target: str       # target artifact ID
  source_slot: str  # provenance: "relation", "applies_to", "domain_field", etc.
  detail: str       # slot-specific qualifier (relation type, field name, etc.)


@dataclass
class ReferenceGraph:
  """Cross-artifact reference graph computed from registry state.

  Nodes are artifact IDs. Edges are forward references extracted
  by collect_references.
  """
  nodes: dict[str, str]         # id → kind
  edges: list[GraphEdge]        # all directed edges
  _forward: dict[str, list[GraphEdge]]   # source → edges (lazy)
  _inverse: dict[str, list[GraphEdge]]   # target → edges (lazy)
  diagnostics: list[str]        # normalization warnings, etc.
```

**Key functions:**

```python
def build_reference_graph(workspace: Workspace) -> ReferenceGraph:
  """Build complete reference graph from all registries.

  1. Collect all artifact nodes via workspace registries
  2. For each artifact, call collect_references to get forward edges
  3. Normalize target IDs against the node index
  4. Build forward and inverse lookup indices
  5. Collect diagnostics for non-canonical ID forms
  """

def query_forward(graph: ReferenceGraph, artifact_id: str) -> list[GraphEdge]:
  """Return edges where artifact_id is the source."""

def query_inverse(graph: ReferenceGraph, artifact_id: str) -> list[GraphEdge]:
  """Return edges where artifact_id is the target."""

def query_neighbourhood(
  graph: ReferenceGraph,
  artifact_id: str,
  direction: str = "both",  # "forward" | "inverse" | "both"
) -> dict[str, list[GraphEdge]]:
  """Return grouped edges for an artifact's reference neighbourhood."""

def find_unresolved_references(graph: ReferenceGraph) -> list[GraphEdge]:
  """Return edges whose target is not in graph.nodes."""
```

**Building the graph — artifact iteration:**

The graph builder needs to iterate all artifacts and call `collect_references` on each. The challenge is that different registries return different model types. The approach:

1. Use `Workspace` to access all registries (it already aggregates them)
2. Each registry's `collect()` or iteration returns typed records
3. `collect_references` already handles any object via `getattr` — no protocol changes needed
4. Build node index from all registry IDs + kinds (mirrors `build_artifact_index` but using `Workspace` directly)

```python
def _collect_all_artifacts(workspace: Workspace) -> list[tuple[str, str, Any]]:
  """Collect (id, kind, artifact_obj) triples from all registries.

  Uses Workspace properties where available, constructs registries
  directly for domains not yet exposed on Workspace (memory, backlog,
  drift).
  """
  from supekku.scripts.lib.backlog.registry import BacklogRegistry
  from supekku.scripts.lib.drift.registry import DriftLedgerRegistry
  from supekku.scripts.lib.memory.registry import MemoryRegistry

  artifacts: list[tuple[str, str, Any]] = []
  root = workspace.root

  # Decisions (Workspace property)
  for dec_id, dec in workspace.decisions.collect().items():
    artifacts.append((dec_id, "adr", dec))

  # Specs (Workspace property)
  for spec in workspace.specs.all_specs():
    artifacts.append((spec.id, "spec", spec))

  # Changes — use Workspace properties for delta/revision/audit
  for registry in (workspace.delta_registry, workspace.revision_registry,
                   workspace.audit_registry):
    kind = registry.kind
    for art_id, art in registry.collect().items():
      artifacts.append((art_id, kind, art))

  # Memory (construct directly — no Workspace property)
  for mem_id, mem in MemoryRegistry(root=root).collect().items():
    artifacts.append((mem_id, "memory", mem))

  # Backlog (construct directly — no Workspace property)
  for item_id, item in BacklogRegistry(root=root).collect().items():
    artifacts.append((item_id, item.kind, item))

  # Requirements (Workspace property)
  for req_id, req in workspace.requirements.records.items():
    artifacts.append((req_id, "requirement", req))

  # Policies (Workspace property)
  for pol_id, pol in workspace.policies.collect().items():
    artifacts.append((pol_id, "policy", pol))

  # Standards (Workspace property)
  for std_id, std in workspace.standards.collect().items():
    artifacts.append((std_id, "standard", std))

  # Drift ledgers (construct directly — no Workspace property)
  for dl_id, dl in DriftLedgerRegistry(root=root).collect().items():
    artifacts.append((dl_id, "drift_ledger", dl))

  return artifacts
```

### 4.2 Modify: `supekku/scripts/lib/core/artifact_ids.py`

Add `normalize_artifact_id` — registry-aware ID padding resolution.

```python
# Regex for prefix + numeric suffix (any digit count)
_PREFIX_NUM_RE = re.compile(r"^([A-Z]+-?)(\d+)(.*)$")

@dataclass(frozen=True)
class NormalizationResult:
  """Result of attempting to normalize an artifact ID."""
  canonical: str | None    # resolved canonical form, or None if no match
  original: str            # the input
  diagnostic: str | None   # warning message if non-canonical form resolved


def normalize_artifact_id(
  raw: str,
  known_ids: set[str],
) -> NormalizationResult:
  """Attempt to normalize a potentially non-canonical artifact ID.

  Strategy:
    1. Direct lookup — if raw is in known_ids, it's already canonical
    2. Parse prefix + numeric suffix
    3. Try zero-padded variants (3-digit, 4-digit, ...) against known_ids
    4. Return first match with diagnostic, or None if no match

  Does not assume a fixed digit width — tries increasing padding
  until either a match is found or a reasonable limit is reached.
  """
```

The normalizer does NOT modify `ID_PATTERNS` or `classify_artifact_id` — those remain strict validators for canonical forms. Normalization is a separate lookup-time concern.

## 6. Design Decisions & Trade-offs

### DEC-097-01: On-demand graph, no persisted file

**Decision**: Compute graph at query time from registries. No `registry/graph.yaml`.

**Rationale**: Current scale (~hundreds of artifacts) makes computation trivial. Persisted file adds staleness risk and maintenance burden. Architecture permits trivial addition of a `sync graph` materializer later (the graph builder is a pure function).

**Consequence**: Every `show relations` invocation scans all registries. Acceptable at current scale.

### DEC-097-02: New `show relations` command

**Decision**: Cross-type neighbourhood queries live in `show relations <ID>`, not distributed across per-type `show` commands.

**Rationale**: Cross-type queries are inherently cross-cutting. Per-type `--related-to` on `list` stays for filtered single-type views. These are complementary, not competing.

### DEC-097-03: Registry-aware ID normalization

**Decision**: Normalize by trying padding variants against the known ID index, not by hardcoding a digit width.

**Rationale**: Current convention is 3-digit IDs but 4-digit is foreseeable. Trying `ADR-11` → `ADR-011` → `ADR-0011` against the registry is future-proof. Emit diagnostic warning on non-canonical match so drift is visible.

**Consequence**: Normalization requires the ID index as input. This is always available when building the graph or running validation.

### DEC-097-04: Frontmatter-only unresolved validation

**Decision**: Validate references extracted by `collect_references` from frontmatter. Do not scan non-memory markdown bodies for wikilinks.

**Rationale**: Frontmatter references are the structured truth layer. Body wikilinks in non-memory files are prose — validating them invites false positives from examples, documentation, and code blocks. Memory wikilinks already have their own dedicated resolver.

## 7. Rollout & Operational Notes

- No migration needed — new functionality only.
- No data model changes — graph is ephemeral, computed on demand.
- `show relations` is a new command; no existing command signatures change.
- Validation changes surface through existing `doctor` / diagnostics pipeline.
- ID normalization runs during graph build and validation; no separate command needed.
