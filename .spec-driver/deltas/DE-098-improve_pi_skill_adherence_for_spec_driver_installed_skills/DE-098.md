## 1. Summary & Context

- **Implementation Plan**: [IP-098](./IP-098.md)
- **Change Drivers**: Observed low skill adherence in pi compared to Claude Code. Pi agents read skill files but follow them loosely — skipping steps, glossing verification, cherry-picking rather than executing.

## 2. Motivation

Pi and Claude Code share the same skill loading mechanism: descriptions in system prompt, full SKILL.md loaded on demand via `read`. Despite identical structure, pi agents follow spec-driver skills less reliably. The agent "cosplays" the skill rather than executing it.

Root causes identified:

- **Passive skill listing**: the `.spec-driver/AGENTS.md` XML block describes skills without compelling the agent to read and follow them
- **3-hop routing chain**: boot → using-spec-driver → governing skill → follow it. Each hop is a drift point
- **21 skills = decision fatigue**: too many choices under token/attention pressure
- **No acknowledgement protocol**: nothing forces the agent to confirm it read the full skill before acting
- **No enforcement**: skill steps are guidance, not gated procedures

## 3. Scope & Objectives

- **Primary Outcomes**:
  - Higher fidelity skill execution in pi sessions (measurable by observing routing and step completion)
  - Reduced routing hops from boot to governing skill
  - Fewer skills for the agent to choose between

## 4. Out of Scope

- Changes to pi harness internals (we work within pi's extension/skill/settings model)
- Changes to Claude Code's skill loading
- Building a runtime skill executor or enforcement engine

## 5. Options Under Consideration

### Option A: Inline routing into boot context

Embed the `using-spec-driver` routing table directly into `APPEND_SYSTEM.md` / preboot output. Eliminates one hop. The routing logic becomes ambient context rather than on-demand skill.

**Pros**: Always loaded, no read required, eliminates a drift point.
**Cons**: Increases always-on context size. Routing logic now lives in two places (skill file + boot context).

### Option B: Consolidate skills by composition

Merge related micro-skills into fewer, larger skills:

- `capturing-memory` + `maintaining-memory` + `reviewing-memory` → `memory`
- `scope-delta` + `draft-design-revision` + `plan-phases` + `update-delta-docs` → `design-and-plan`
- `notes` + `continuation` + `next` → `handover`

**Pros**: Fewer choices = higher hit rate. Aligns with how work actually flows.
**Cons**: Larger skill files. May lose granularity in descriptions that help routing.

### Option C: Skill acknowledgement protocol

Add a "contract" header to critical skills requiring the agent to state skill name and step count before proceeding. Forces a full read.

**Pros**: Simple. Catches skim-reading. Creates observable evidence of adherence.
**Cons**: Relies on agent compliance with yet another instruction. Could become cargo-cult.

### Option D: Use `disable-model-invocation` for low-frequency skills

Hide rarely-auto-triggered skills (`reviewing-memory`, `continuation`, `doctrine`) from the system prompt. Only accessible via explicit `/skill:name`.

**Pros**: Reduces decision space. Keeps rare skills available.
**Cons**: Agent can't self-route to hidden skills when they're actually needed.

### Option E: Strengthen description trigger language

Rewrite skill descriptions to use stronger imperative language and more specific trigger conditions, making the model more likely to load the right skill.

**Pros**: Low-effort. Works within existing mechanism.
**Cons**: Marginal improvement. Descriptions are already fairly specific.

### Option F: Use `/skill:name` syntax in routing instructions

Boot context and inter-skill references use `/skill:execute-phase` rather than `/execute-phase`. May trigger pi's command handling which forces a read.

**Pros**: Trivial change. Leverages pi's existing mechanism.
**Cons**: Unclear whether this actually forces loading in pi vs just being a naming convention.

## 6. Verification Strategy

- **Acceptance Criteria**: Observable improvement in pi skill adherence during real sessions. Before/after comparison on routing accuracy and step completion.

## 7. Risks & Mitigations

- **Risk**: Consolidating skills may reduce routing precision – _Likelihood_: medium – _Impact_: medium – _Mitigation_: Test consolidated skills in real sessions before committing
- **Risk**: Inlining routing adds always-on token cost – _Likelihood_: high – _Impact_: low – _Mitigation_: Keep inlined content compact; measure token impact
