# DR-098 – improve pi skill adherence for spec-driver installed skills

## 1. Executive Summary

- **Delta**: [DE-098](./DE-098.md)
- **Status**: draft
- **Synopsis**: Inline the skill routing table from `using-spec-driver` into the preboot boot context so agents always have routing guidance without needing to read a skill first.

## 2. Problem & Constraints

- **Current Behaviour**: Agents must boot → read `using-spec-driver` skill → decide governing skill → read governing skill → follow it. The `using-spec-driver` hop is where pi agents most commonly drift — they skip the read and wing it based on the skill description alone.
- **Drivers**: Observed low skill adherence in pi vs Claude Code for identical skill content.
- **Constraints**:
  - Preboot content is shared across harnesses (Claude Code, pi, `.agents` consumers). Changes must help or be neutral for all.
  - Skills are symlinked from `.spec-driver/skills/` — cannot have harness-specific skill variants.
  - Token budget: preboot is always-on context. Keep additions compact.
- **Out of Scope**: Harness-specific preboot variants (Option F deferred). Skill consolidation (Option B — separate delta).

## 3. Architecture Intent

- **Target Outcome**: The routing table that tells agents which skill governs which task is ambient context, not gated behind a skill read.
- **Guiding Principle**: Simplest thing that could work. One new boot-sequence file, one line in `BOOT_SEQUENCE`.

## 4. Design

### New file: `.spec-driver/agents/routing.md`

Extract the routing decision table from `using-spec-driver/SKILL.md` into a compact boot-sequence file. ~25 lines. The essential routing table, stance, and red flags. NOT the guardrails/failure-modes prose — that stays in the skill.

Proposed content:

```markdown
# Skill Routing

Choose the governing skill before doing substantive work. Do not respond,
explore, inspect files, or start implementation until you have decided.

If you skip routing because the task feels familiar, simple, or urgent — stop.
That is a routing failure.

## Routing Table (priority order)

1. **spec-driver entities or CLI** → `/spec-driver`
2. **"what is the right way here?"**, unfamiliar subsystem, assumption risk → `/retrieving-memory`; add `/doctrine` when governance matters
3. **substantive new work, path unclear** → `/preflight`
4. **code-changing intent, no governing artefact** → `/scope-delta` (or `/shape-revision` first when revision-first governance applies)
5. **delta exists, DR/IP/phase incomplete** → `/draft-design-revision` then `/plan-phases`
6. **delta phase active, planning artefacts exist** → `/execute-phase`
7. **implementation complete, reconciliation needed** → `/audit-change` then `/close-change`

## Default Stance

- If unsure, route to the stricter skill, not the looser one.
- Do not implement without a governing artefact (card, delta, or revision).
- `/using-spec-driver` has extended guardrails and failure-mode guidance.
```

### Change: `supekku/scripts/lib/core/preboot.py`

Add `("Skill Routing", f"{SPEC_DRIVER_DIR}/agents/routing.md")` to `BOOT_SEQUENCE`, after Workflow and before Policy & Governance.

### Change: `using-spec-driver/SKILL.md`

Add a note at the top: "Core routing rules are loaded in boot context. This skill provides extended guardrails and failure-mode guidance."

The skill retains its full content — it becomes the detailed reference rather than the only source of routing truth.

| Path | Current State | Target State |

| `supekku/scripts/lib/core/preboot.py` | 7 boot-sequence entries | 8 entries (routing added) |

## 6. Verification

- `preboot_test.py`: Verify new section appears in generated output
- Manual: Run `admin preboot`, confirm routing section present in both output files
- Qualitative: Observe pi sessions for improved routing behaviour (no automated gate)

## 7. Design Decisions

- **DEC-01**: Inline into shared preboot, not harness-specific. Rationale: the routing table helps all harnesses. Divergence adds complexity for uncertain gain. Revisit if A alone doesn't move the needle.
- **DEC-02**: Extract to a new file (`routing.md`) rather than embedding in an existing boot-sequence file. Keeps files single-responsibility and the routing content independently editable.

## 8. Risks

- **Token cost**: ~30 lines of always-on context. Low impact relative to existing preboot size (~150 lines).
- **Duplication**: Routing rules now live in two places (routing.md + using-spec-driver skill). Mitigated by making the skill explicitly reference the boot context as authoritative for core rules.
