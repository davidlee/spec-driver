## 2. Motivation

Tag filtering on `list` commands is inconsistent:

- **3 commands** have `--tag` for filtering (ADRs, policies, standards)
- **1 command** has both `--tag` (metadata pre-filter) and `--match-tag` (scope match) ÔÇö memories
- **12+ commands** have no tag filtering at all despite entities carrying `tags:` in frontmatter

Additionally, `--tag` is overloaded: it means "set a tag" in `edit` commands and "filter by tag" in `list` commands. This is confusing.

**Convention to establish:**
- `--tag`/`-t` = **mutator** (set/add a tag) ÔÇö `edit` and `create` commands only
- `--match-tag` = **query/filter** (filter results by tag) ÔÇö `list` commands only

## 3. Scope & Objectives

### Rename existing `list` filters
- `list adrs --tag` Ôćĺ `--match-tag`
- `list policies --tag` Ôćĺ `--match-tag`
- `list standards --tag` Ôćĺ `--match-tag`
- `list memories --tag` (metadata pre-filter) Ôćĺ `--match-tag` (merge with existing scope `--match-tag`)

### Add `--match-tag` to list commands missing it

All entity types have `tags:` in frontmatter (via base metadata). These list commands need `--match-tag`:

- `list backlog` (registry already has `tag` param ÔÇö just needs CLI wiring)
- `list specs` (registry already has `tag` param ÔÇö just needs CLI wiring)
- `list deltas`
- `list changes`
- `list requirements`
- `list revisions`
- `list audits`
- `list plans`
- `list cards`
- `list drift`
- `list issues` / `list problems` / `list improvements` / `list risks` (if separate from `list backlog`)

### Registry layer
- Where registries already have `tag=` filter param: wire to CLI
- Where registries lack it: add `tag` param to `filter()` method
- `list memories`: reconcile the two tag filter paths into one `--match-tag`

## 4. Out of Scope
