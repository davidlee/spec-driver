## 2. Motivation

Tag filtering on `list` commands is incomplete:

- **3 commands** have `--tag`/`-t` for filtering (ADRs, policies, standards)
- **1 command** has `--tag` (metadata pre-filter) and `--match-tag` (scope match) ‚Äî memories
- **12+ commands** have no tag filtering at all despite entities carrying `tags:` in frontmatter

The verb context (`list` vs `edit`) already disambiguates `--tag` semantics ‚Äî same as docker, kubectl, gh. No rename needed.

## 3. Scope & Objectives

### Convention

| Flag          | Context                  | Semantics                                                              |
| ------------- | ------------------------ | ---------------------------------------------------------------------- |
| `--tag`/`-t`  | `list` commands          | Query ‚Äî filter by tag (repeatable, OR logic, ANDed with other filters) |
| `--tag`/`-t`  | `edit`/`create` commands | Mutator ‚Äî set/add tag                                                  |
| `--untag`     | `edit` commands          | Mutator ‚Äî remove tag                                                   |
| `--match-tag` | `list memories` only     | Scope match (OR with path/command) ‚Äî pre-existing, unchanged           |

### Add `--tag`/`-t` to list commands missing it

- `list backlog` (+ `list issues/problems/improvements/risks` sub-kind wrappers)
- `list specs`
- `list deltas`
- `list changes`
- `list requirements`
- `list revisions`
- `list audits`
- `list plans`

### Make `--tag` repeatable with OR logic

Existing `--tag` on ADRs/policies/standards is single-value. Change to repeatable (match any).

### Model: add `tags` to `PlanSummary`

`PlanSummary` lacks `tags: list[str]`. Add field; populate from frontmatter in `discover_plans()`.

### Model: ensure all taggable models have explicit `tags` field

No `getattr(i, 'tags', [])` fallback. Every model that can be tag-filtered must have `tags: list[str] = field(default_factory=list)`.

## 4. Out of Scope

- Cards (`list cards`) ‚Äî no frontmatter, lightweight kanban
- Drift ledgers (`list drift`) ‚Äî no `tags:` in schema or model
- `ChangeRegistry.filter(tag=)` ‚Äî no CLI consumer; add when needed
- `--tag` on remaining `create` commands (DE-096 follow-up)

## 5. Risks & Mitigations

- **Risk**: Making `--tag` repeatable changes behaviour for existing single-value users on ADRs/policies/standards ‚Äì _Likelihood_: low ‚Äì _Impact_: low ‚Äì _Mitigation_: OR semantics with single value is identical to current behaviour
