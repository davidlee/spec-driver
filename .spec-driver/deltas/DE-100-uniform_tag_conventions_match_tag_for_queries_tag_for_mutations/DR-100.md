# DR-100 – Uniform --tag filtering across all list commands

## 1. Executive Summary

- **Delta**: [DE-100](./DE-100.md)
- **Status**: approved
- **Synopsis**: Add `--tag`/`-t` filtering to all `list` commands whose entities carry `tags:` in frontmatter. Make the flag repeatable with OR semantics. No renames — verb context disambiguates.

## 2. Problem & Constraints

### Current behaviour

| Flag | Context | Semantics |
|------|---------|-----------|
| `--tag`/`-t` | `edit *` | Mutator — add tag |
| `--untag` | `edit *` | Mutator — remove tag |
| `--tag`/`-t` | `list adrs/policies/standards` | Query — filter by tag (single-value) |
| `--tag` | `list memories` | Query — metadata AND pre-filter |
| `--match-tag` | `list memories` | Query — scope OR match |
| _(nothing)_ | `list backlog/specs/deltas/...` | No tag filtering |

3 of 15+ list commands support tag filtering. 12+ do not.

### Constraints

- **ADR-009**: Registries expose `filter(...)` with domain-specific params.
- **POL-001**: Maximise reuse. The inline filtering pattern is already established.
- **POL-002**: No magic strings. All taggable models must have explicit `tags` field.

## 3. Design Decisions

### DEC-100-01: Keep `--tag` — no rename

The verb context (`list` vs `edit`) disambiguates `--tag` semantics. This is how docker (`--label` on `run` sets, `--filter label=` on `ps` filters), kubectl, and gh handle it. Renaming to `--match-tag` buys disambiguation the command structure already provides and introduces an unfamiliar flag name.

`list memories` keeps both `--tag` (metadata pre-filter) and `--match-tag` (scope match) — these are pre-existing, filter different fields, and are unchanged by this delta.

### DEC-100-02: Cards and drift excluded

Cards are lightweight kanban with no frontmatter. Drift ledgers have no `tags:` in their schema. Users who try `list cards --tag foo` get a typer error (unknown option), not silent no-results.

### DEC-100-03: Repeatable with OR logic

`--tag` becomes repeatable: `list backlog --tag cli --tag dx` matches items with *either* tag. OR is the natural "show me items in any of these categories" query. ANDed with all other filters (`--status`, `--filter`, `--regexp`, `--severity`, etc.), consistent with existing filter interaction on all commands.

Single-value use is unchanged — OR with one value = exact match.

### DEC-100-04: No `ChangeRegistry.filter(tag=)`

`list deltas/revisions/audits` all do inline filtering. Adding `tag=` to `ChangeRegistry.filter()` with no CLI consumer is speculative generality. Add when a consumer exists.

### DEC-100-05: Explicit `tags` on all taggable models

Every model that can be tag-filtered must have `tags: list[str] = field(default_factory=list)`. No `getattr(i, 'tags', [])` fallback — that papers over schema drift.

## 4. Code Impact Summary

### CLI (`supekku/cli/list.py`)

**Keep as-is (no rename):**
- `list adrs --tag`/`-t` — make repeatable
- `list policies --tag`/`-t` — make repeatable
- `list standards --tag`/`-t` — make repeatable
- `list memories --tag` / `--match-tag` — unchanged

**Add `--tag`/`-t` (repeatable):**
- `list backlog`
- `list issues` / `list problems` / `list improvements` / `list risks` (thread through to `list_backlog`)
- `list specs`
- `list deltas`
- `list changes`
- `list requirements`
- `list revisions`
- `list audits`
- `list plans`

**Implementation pattern** — inline filter, consistent with how each command already works:

```python
tag: Annotated[
  list[str] | None,
  typer.Option("--tag", "-t", help="Filter by tag (repeatable, OR logic)"),
] = None,

# In filter section (ANDed with other filters):
if tag:
  items = [i for i in items if any(t in i.tags for t in tag)]
```

For commands that use `registry.filter()` (ADRs, policies, standards): change `tag=tag` (single str) to handle list — either pass first match or filter inline after `registry.filter()`. Simplest: move tag filtering inline for these too, keeping `registry.filter()` for other params.

### Models

| Model | Has `tags`? | Change |
|-------|------------|--------|
| `ChangeArtifact` | ✅ | None |
| `BacklogItem` | ✅ | None |
| `Spec` | ✅ | None |
| `DecisionRecord` | ✅ | None |
| `PolicyRecord` | ✅ | None |
| `StandardRecord` | ✅ | None |
| `RequirementRecord` | ✅ | None |
| `PlanSummary` | ❌ | Add `tags: list[str] = field(default_factory=list)`; populate in `discover_plans()` |

### Registries

No registry changes. Tag filtering is inline in CLI layer, consistent with how `list deltas/revisions/audits/plans/backlog` already handle all their filtering.

For ADRs/policies/standards, the existing `registry.filter(tag=...)` path works but only accepts single string. Options:
1. Refactor `registry.filter(tag=)` to accept `list[str]` — clean but touches 3 registries
2. Do tag filtering inline in CLI, skip passing to registry — consistent with other commands

Recommend option 2 for consistency: all tag filtering is inline CLI logic with the same pattern.

## 5. Verification

- Existing tests for `list adrs/policies/standards --tag` still pass (flag unchanged)
- New CLI tests for `--tag` on all added commands (single-value, multi-value OR, no-match)
- Verify `--tag` ANDs with `--status`, `--filter`, `--regexp`
- `PlanSummary.tags` populated correctly from frontmatter
- `just check` green

## 6. Risks

- **Making `--tag` repeatable**: OR with single value = current behaviour. No breaking change.
- **Moving tag filtering inline for ADRs/policies/standards**: Behaviour identical. `registry.filter(tag=)` still exists for programmatic use.
