# DR-100 – Uniform tag conventions

## 1. Executive Summary

- **Delta**: [DE-100](./DE-100.md)
- **Status**: draft
- **Synopsis**: Establish uniform CLI flag conventions for tag operations — `--tag` for mutations, `--match-tag` for queries — and wire `--match-tag` to all list commands whose entities carry `tags:` in frontmatter.

## 2. Problem & Constraints

### Current behaviour

| Flag | Context | Semantics |
|------|---------|-----------|
| `--tag`/`-t` | `edit *` | Mutator — add tag |
| `--untag` | `edit *` | Mutator — remove tag |
| `--tag`/`-t` | `list adrs/policies/standards` | Query — filter by tag |
| `--tag` | `list memories` | Query — metadata AND pre-filter |
| `--match-tag` | `list memories` | Query — scope OR match |
| _(nothing)_ | `list backlog/specs/deltas/...` | No tag filtering |

`--tag` means "set" in edit and "filter" in list. 12+ list commands have no tag filtering despite entities carrying `tags:` in frontmatter.

### Constraints

- **ADR-009**: Registries expose `filter(...)` with domain-specific params. Tag filtering fits naturally.
- **POL-001**: Maximise reuse. The pattern already exists in 3 registries + memory selection.
- **POL-002**: No magic strings. Tag param names should be consistent.

## 3. Target Convention

| Flag | Semantics | Commands |
|------|-----------|----------|
| `--tag`/`-t` | Mutator — set/add | `edit *`, `create *` |
| `--untag` | Mutator — remove | `edit *` |
| `--match-tag` | Query — filter by tag | All `list` commands (where entity has tags) |
| `--has-tag` | Query — metadata AND pre-filter | `list memories` only¹ |

¹ Memories have two distinct tag fields: `tags:` (metadata, what the memory *is*) and `scope.tags` (what the memory *applies to*). `--has-tag` filters the former, `--match-tag` the latter. Other entities have one `tags:` field, so `--match-tag` suffices.

### Out of scope (DEC-100-02)

| Entity | Reason |
|--------|--------|
| Cards (`list cards`) | No frontmatter — lightweight kanban. Adding tags requires frontmatter support, a bigger change. |
| Drift ledgers (`list drift`) | No `tags:` in frontmatter schema or model. Low volume, rarely filtered. |

### CLI wiring pattern

For commands that already call `registry.filter()` (ADRs, policies, standards): rename the parameter.

For commands that do inline filtering (backlog, specs, deltas, etc.): add inline tag check in the same style as existing filters:

```python
match_tag: Annotated[
  str | None,
  typer.Option("--match-tag", help="Filter by tag"),
] = None,

# ... in filter section:
if match_tag:
  items = [i for i in items if match_tag in getattr(i, 'tags', [])]
```

For commands that use `registry.filter()`: pass through as `tag=match_tag`.

### `list memories` changes

Rename `--tag` parameter to `--has-tag`. The existing `--match-tag` (scope match) stays unchanged. The `--has-tag` value continues to be passed as the `tag=` metadata pre-filter in the memory selection API — only the CLI flag name changes.

```python
# Before
tag: ... typer.Option("--tag", help="Filter by tag (metadata pre-filter)") ...
# After  
has_tag: ... typer.Option("--has-tag", help="Filter by tag (metadata pre-filter, AND logic)") ...
```

## 6. Risks & Mitigations

- **Breaking CLI change** (rename `--tag` → `--match-tag` on 3 list commands, `--tag` → `--has-tag` on memories): No known external consumers. Clean rename, no deprecation shim.
- **PlanSummary model change**: Adding a field with `field(default_factory=list)` — backwards compatible.

## 7. Design Decisions

### DEC-100-01: `--match-tag` everywhere, `--has-tag` only on memories

Memories have two distinct tag fields (`tags:` metadata vs `scope.tags`). All other entities have one. One flag suffices for single-field entities. `--has-tag` is the pattern to extend if AND-logic filtering becomes needed elsewhere.

### DEC-100-02: Cards and drift ledgers excluded

Cards are deliberately lightweight (no frontmatter). Drift ledgers don't carry tags. Adding tag support to either is a separate concern.
