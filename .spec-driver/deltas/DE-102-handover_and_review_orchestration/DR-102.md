# DR-102: Handover and Review Orchestration

This design revision defines the workflow control plane for spec-driver: a set
of machine-readable YAML artifacts, a CLI command surface, and lifecycle rules
that make phase transitions deterministic, handoffs structured, and reviewer
bootstrap amortizable — without replacing existing markdown workflow artifacts.

## 1. Design Scope and Boundaries

### In scope (layers 1–2)

- Workflow semantics: states, transitions, roles, handoff, review rounds
- File-based control plane: YAML schemas, directory layout, bridge blocks
- CLI command surface for orchestration operations
- Continuation skill refit to emit structured handoff
- Lifecycle and invalidation rules for reviewer state

### Out of scope (layers 3–5, tracked in IMPR-020)

- Process supervision, daemons, spawn/detach/observe
- IPC, mailboxes, file-watch event loops
- Sandbox/multiplexer/harness management
- Agent adapter configuration (runner profiles, installer hooks)
- Policy engine (model assignment, budget, escalation)
- Harness-specific write paths or sandbox escape hatches

### Authority model

Unambiguous and one-directional:

1. **Markdown artifacts are authority.** `DE-*.md` owns intent. `IP-*.md` owns
   planning and verification. `phases/*.md` owns execution. `notes.md` owns
   human-readable narrative.
2. **YAML workflow files are derived projections.** `workflow/*.yaml` is written
   by CLI commands and reflects orchestration state. It does not override
   markdown authority.
3. **CLI commands are the mutation surface.** All workflow YAML mutations go
   through the spec-driver CLI. Skills, agents, and operators invoke CLI
   commands to mutate state — they do not write workflow YAML directly.
4. **No reconciliation hammer.** If YAML and markdown disagree, the YAML is
   wrong. The fix is to re-run the relevant CLI command, not a sync sweep.

There is no daemon. There is no watcher. There is no bidirectional
reconciliation. There is no direct-write bypass. The control plane is a
projection that the CLI keeps current through explicit command execution.

## 2. Artifact Structure

### Directory layout

```
.spec-driver/deltas/DE-xxx-.../
  DE-xxx.md                        # existing — design/intent authority
  IP-xxx.md                        # existing — plan/verification authority
  notes.md                         # existing — human narrative
  phases/                          # existing — per-phase execution
  workflow/                        # NEW — machine-readable control plane
    state.yaml                     #   orchestration status + pointers
    handoff.current.yaml           #   durable transition payload
    review-index.yaml              #   reviewer bootstrap cache
    review-findings.yaml           #   stable issue ledger
    sessions.yaml                  #   runtime session map (optional)
    review-bootstrap.md            #   generated reviewer briefing
```

Additive only. No existing file formats change.

### Ownership model

| File | Owner | Authored vs derived |
|------|-------|---------------------|
| `DE-*.md` | human / design agent | authored |
| `IP-*.md` | human / planning agent | authored |
| `phases/*.md` | executing agent | authored |
| `notes.md` | executing agent + continuation skill | authored |
| `workflow/state.yaml` | CLI (`handoff`, `review`, `phase` commands) | derived |
| `workflow/handoff.current.yaml` | CLI (`create handoff`) | derived |
| `workflow/review-index.yaml` | CLI (`review prime`) | derived |
| `workflow/review-findings.yaml` | CLI (`review complete`) | derived |
| `workflow/sessions.yaml` | CLI (session commands) | derived, optional |
| `workflow/review-bootstrap.md` | CLI (`review prime`) | generated |

## 3. Artifact Schemas

All nested record types below specify required and optional fields. A field is
required unless explicitly marked `# optional`.

### 3.1 Workflow State — `supekku:workflow.state@v1`

**File:** `workflow/state.yaml`

**Purpose:** Current orchestration status and pointers. 100% derived from CLI
transitions.

```yaml
schema: supekku.workflow.state
version: 1

artifact:
  id: DE-090                       # required — string, entity ID
  kind: delta                      # required — delta | plan | revision | audit | task | other
  path: .spec-driver/deltas/DE-090-slug  # optional — string, relative path
  notes_path: .spec-driver/deltas/DE-090-slug/notes.md  # optional — string

plan:                              # optional — absent if no IP exists
  id: IP-090                       # required within block — string
  path: .spec-driver/deltas/DE-090-slug/IP-090.md  # optional — string

phase:
  id: IP-090.PHASE-05              # required — string, current phase
  path: .spec-driver/deltas/DE-090-slug/phases/phase-05.md  # optional — string
  status: complete                 # required — not_started | in_progress | blocked | complete | skipped

workflow:
  status: awaiting_handoff         # required — see §4 state machine
  active_role: implementer         # required — architect | implementer | reviewer | operator | other
  next_role: reviewer              # optional — string, same enum as active_role
  handoff_boundary: phase          # optional — phase | task | manual
  claimed_by: null                 # optional — string, agent/operator identity; see §4

pointers:                          # optional — all fields within are optional strings
  current_handoff: workflow/handoff.current.yaml
  review_index: workflow/review-index.yaml
  review_findings: workflow/review-findings.yaml
  sessions: workflow/sessions.yaml
  review_bootstrap: workflow/review-bootstrap.md

timestamps:
  created: 2026-03-21T10:00:00Z   # required — ISO 8601
  updated: 2026-03-21T10:30:00Z   # required — ISO 8601
```

**Required fields:** `schema`, `version`, `artifact.id`, `artifact.kind`,
`workflow.status`, `workflow.active_role`, `phase.id`, `phase.status`,
`timestamps.created`, `timestamps.updated`.

**Validation:** Unknown top-level keys rejected unless `x-` prefixed. Path
fields must be relative to artifact root or repo root. IDs must match known
artifact ID patterns.

### 3.2 Handoff — `supekku:workflow.handoff@v1`

**File:** `workflow/handoff.current.yaml`

**Purpose:** Durable phase-boundary transition payload. Structured equivalent of
what the continuation skill currently asks agents to write in prose. A handoff
can target any role — it is not review-specific.

```yaml
schema: supekku.workflow.handoff
version: 1

artifact:
  id: DE-090                       # required — string
  kind: delta                      # required — same enum as state.artifact.kind

transition:
  from_role: implementer           # required — same enum as workflow.active_role
  to_role: reviewer                # required — same enum as workflow.active_role
  boundary: phase                  # optional — phase | task | manual
  status: pending                  # required — pending | accepted | superseded

phase:
  id: IP-090.PHASE-05              # required — string
  status: complete                 # optional — same enum as state.phase.status

required_reading:                  # required — list of strings, minimum 1 entry
  - .spec-driver/deltas/DE-090-slug/DE-090.md
  - .spec-driver/deltas/DE-090-slug/IP-090.md
  - .spec-driver/deltas/DE-090-slug/phases/phase-05.md
  - .spec-driver/deltas/DE-090-slug/notes.md

related_documents: []              # optional — list of strings
key_files: []                      # optional — list of strings

verification:                      # optional block
  commands: []                     # optional — list of strings
  summary: ""                      # optional — string
  status: pass                     # required within block — pass | fail | partial | not_run | unknown

git:                               # optional block
  head: e31eca9                    # required within block — string
  branch: null                     # optional — string | null
  worktree:                        # optional block
    has_uncommitted_changes: true   # required within block — boolean
    has_staged_changes: false       # required within block — boolean

open_items:                        # optional — list of open_item records
  - id: OI-001                     # required — string, unique within handoff
    kind: next_step                # required — next_step | blocker | question | risk
    summary: Implement Phase 06    # required — string
    blocking: false                # required — boolean

design_tensions: []                # optional — list of strings
unresolved_assumptions: []         # optional — list of strings
decisions_to_preserve: []          # optional — list of strings

next_activity:
  kind: review                     # required — implementation | review | architecture |
                                   #            verification | operator_attention
  summary: Assess Phase 05 output  # optional — string

timestamps:
  emitted_at: 2026-03-21T10:30:00Z # required — ISO 8601
```

**Required fields:** `schema`, `version`, `artifact.id`, `artifact.kind`,
`transition.from_role`, `transition.to_role`, `transition.status`, `phase.id`,
`required_reading` (min 1), `next_activity.kind`, `timestamps.emitted_at`.

**Nested record constraints:**
- `open_item`: `id` (string, unique within handoff), `kind` (enum), `summary`
  (string), `blocking` (boolean) — all required.

### 3.3 Review Index — `supekku:workflow.review-index@v1`

**File:** `workflow/review-index.yaml`

**Purpose:** Reviewer bootstrap cache (tier 2 persistence). What the reviewer
has already learned and should not rediscover. Separate from findings.

```yaml
schema: supekku.workflow.review-index
version: 1

artifact:
  id: DE-090                       # required — string
  kind: delta                      # required — same enum as state.artifact.kind

review:
  session_scope: artifact          # optional — artifact | phase | task
  bootstrap_status: warm           # required — cold | warming | warm | stale | invalid | reusable
  last_bootstrapped_at: 2026-03-21T10:25:00Z  # required — ISO 8601
  source_handoff: workflow/handoff.current.yaml  # optional — string, path

domain_map:                        # required — list of domain_entry, minimum 1
  - area: cli                      # required — string, unique within domain_map
    purpose: command routing       # required — string
    files:                         # required — list of strings, minimum 1
      - supekku/cli/common.py
      - supekku/cli/show.py

invariants:                        # optional — list of invariant records
  - id: INV-001                    # required — string, unique within invariants
    summary: JSON output stable    # required — string

risk_areas:                        # optional — list of risk_area records
  - id: RA-001                     # required — string, unique within risk_areas
    summary: Reverse lookup gaps   # required — string
    files: []                      # optional — list of strings

review_focus: []                   # optional — list of strings

known_decisions:                   # optional — list of known_decision records
  - id: KD-001                     # required — string, unique within known_decisions
    summary: use uid not id        # required — string

staleness:
  cache_key:                       # required
    phase_id: IP-090.PHASE-05      # required — string
    head: e31eca9                  # required — string, git commit
  invalidation_triggers:           # optional — list of strings
    - major_scope_change
    - dependency_surface_expanded
    - artifact_rebased
```

**Required fields:** `schema`, `version`, `artifact.id`, `artifact.kind`,
`review.bootstrap_status`, `review.last_bootstrapped_at`, `domain_map` (min 1),
`staleness.cache_key.phase_id`, `staleness.cache_key.head`.

**Nested record constraints:**
- `domain_entry`: `area` (string, unique), `purpose` (string), `files`
  (list of strings, min 1) — all required.
- `invariant`: `id` (string, unique), `summary` (string) — all required.
- `risk_area`: `id` (string, unique), `summary` (string) — required.
  `files` — optional.
- `known_decision`: `id` (string, unique), `summary` (string) — all required.

### 3.4 Review Findings — `supekku:workflow.review-findings@v1`

**File:** `workflow/review-findings.yaml`

**Purpose:** Stable issue ledger across review rounds. Machine-readable input
for an implementer returning from review.

```yaml
schema: supekku.workflow.review-findings
version: 1

artifact:
  id: DE-090                       # required — string
  kind: delta                      # required — same enum as state.artifact.kind

review:
  round: 3                         # required — integer, monotonically increasing
  status: changes_requested        # required — not_started | in_progress |
                                   #            changes_requested | approved | blocked
  reviewer_role: reviewer          # optional — string, same enum as workflow.active_role

blocking:                          # optional — list of finding records
  - id: R3-001                     # required — string, unique across all finding lists
    title: Output mode regression  # required — string
    summary: Field change breaks JSON consumers  # required — string
    status: open                   # required — open | resolved | waived | superseded
    files: []                      # optional — list of strings
    related_invariants: []         # optional — list of strings (invariant IDs)

non_blocking: []                   # optional — list of finding records (same shape)

resolved:                          # optional — list of finding records
  - id: R2-001                     # required — string, unique across all finding lists
    title: Identifier mismatch    # required — string
    summary: Fixed                 # optional — string (optional for resolved/waived)
    status: resolved               # required — open | resolved | waived | superseded
    resolution_summary: Switched to uid  # optional — string

waived: []                         # optional — list of finding records (same shape)

history:                           # optional — list of history_entry records
  - round: 1                       # required — integer
    summary: Initial review        # required — string

timestamps:
  updated: 2026-03-21T10:35:00Z   # required — ISO 8601
```

**Required fields:** `schema`, `version`, `artifact.id`, `artifact.kind`,
`review.round`, `review.status`, `timestamps.updated`.

**Nested record constraints:**
- `finding`: `id` (string, globally unique across blocking/non_blocking/
  resolved/waived within the file), `title` (string), `status` (enum) — all
  required. `summary` required for open/superseded findings, optional for
  resolved/waived. `files`, `related_invariants`, `resolution_summary` —
  optional.
- `history_entry`: `round` (integer), `summary` (string) — all required.

### 3.5 Sessions — `supekku:workflow.sessions@v1`

**File:** `workflow/sessions.yaml`

**Purpose:** Runtime mapping of roles to sessions. Optional — only meaningful
when the operator or a future orchestration layer manages persistent sessions.

```yaml
schema: supekku.workflow.sessions
version: 1

artifact:
  id: DE-090                       # required — string
  kind: delta                      # required — same enum as state.artifact.kind

sessions:                          # required — map of role_name → session_entry
  implementer:
    session_name: sd-DE-090-impl   # required — string | null
    sandbox: pi-mono               # optional — string | null
    status: paused                 # required — active | paused | absent | dead | unknown
    last_seen: 2026-03-21T10:20:00Z  # required — ISO 8601 | null
  reviewer:
    session_name: sd-DE-090-review
    sandbox: gemini-review
    status: active
    last_seen: 2026-03-21T10:34:00Z
```

**Required fields:** `schema`, `version`, `artifact.id`, `artifact.kind`,
`sessions` (at least one entry).

**Nested record constraints:**
- `session_entry`: `session_name` (string | null), `status` (enum),
  `last_seen` (ISO 8601 | null) — all required. `sandbox` — optional.

**Non-authority status:** This file carries no workflow authority. Its absence
does not affect any workflow transition. CLI commands that read workflow state
(`accept handoff`, `workflow status`, etc.) must function identically whether
or not `sessions.yaml` exists.

## 4. State Machine

### States

`planned`, `implementing`, `awaiting_handoff`, `reviewing`,
`changes_requested`, `approved`, `blocked`.

### Transitions

All transitions are triggered by explicit CLI commands. No implicit triggers.

```
planned ──[phase start]──────────────────► implementing
implementing ──[create handoff --to <role>]──► awaiting_handoff
awaiting_handoff ──[accept handoff]──────────► implementing  (if to_role = implementer/architect/operator)
awaiting_handoff ──[accept handoff]──────────► reviewing     (if to_role = reviewer)
reviewing ──[review complete --status changes_requested]──► changes_requested
reviewing ──[review complete --status approved]───────────► approved
changes_requested ──[create handoff --to <role>]──► awaiting_handoff  (cycle)
* ──[block]──────────────────────────────────► blocked
blocked ──[unblock]──────────────────────────► (previous state)
```

### Claim/accept guard

To prevent silent double-acceptance of a handoff:

1. `accept handoff` writes `workflow.claimed_by` in `state.yaml` with the
   accepting agent/operator identity (provided via `--identity <name>` or
   defaulting to `$USER`).
2. If `workflow.claimed_by` is already set and differs from the caller's
   identity, `accept handoff` fails with an error naming the current claimant.
3. If `workflow.claimed_by` matches the caller, the command is idempotent
   (no-op).
4. `create handoff` clears `claimed_by` (new handoff resets the claim).

This is an optimistic lock, not a distributed mutex. It makes double-acceptance
detectable and prevents silent state corruption. Full concurrent orchestration
is an IMPR-020 concern.

**Identity requirements:** `$USER` is a convenience default for single-agent
use. When multiple agent sessions run under the same Unix user, the operator
**must** provide distinct identities via `--identity` (e.g.
`--identity reviewer-gemini`, `--identity impl-claude`). The CLI does not
attempt to infer or generate unique identities — that is an operator
responsibility. Without unique identities, the claim guard cannot distinguish
concurrent agents and the determinism guarantee does not hold.

### Design choices

- **`awaiting_handoff` replaces `awaiting_review`.** Handoff is role-generic.
  The `to_role` field in the handoff payload determines what happens when the
  handoff is accepted, not the state name. `workflow status` must display the
  target role prominently (e.g. "awaiting handoff → reviewer") so the operator
  knows which direction work is flowing.
- **`accept handoff` is explicit.** The receiving agent or operator runs this
  command. There is no daemon to detect and auto-accept.
- **No `create handoff` → auto-transition to `reviewing`.** The handoff sits
  in `awaiting_handoff` until explicitly accepted. This is the clean separation
  between "work is offered" and "work is picked up."
- **Creating a new handoff supersedes any unaccepted prior handoff.**
  `handoff.current.yaml` carries latest-wins semantics. The old payload is
  overwritten — supersession is observable only in git history, not in any
  in-band state field.

## 5. CLI Command Surface

### Core commands

| Command | Effect |
|---------|--------|
| `spec-driver phase start` | `planned` → `implementing`; initialises `workflow/state.yaml` |
| `spec-driver create handoff --to <role>` | writes `handoff.current.yaml`; transitions to `awaiting_handoff` |
| `spec-driver accept handoff [--identity <name>]` | claims + transitions to `implementing` or `reviewing` based on `to_role` |
| `spec-driver review prime` | generates `review-index.yaml` + `review-bootstrap.md` from current state |
| `spec-driver review complete --status <status>` | writes `review-findings.yaml`; transitions to `changes_requested` or `approved` |
| `spec-driver workflow status` | reads `workflow/state.yaml` and prints human-readable summary |
| `spec-driver review teardown` | deletes reviewer state files |
| `spec-driver block` / `spec-driver unblock` | transitions to/from `blocked` |
| `spec-driver phase complete` | marks phase complete; emits handoff if phase bridge has `handoff_ready: true` |

### Command behaviour

- Every command that mutates state writes the relevant YAML file(s)
  atomically (write-to-temp + rename).
- Every command validates its output against the schema before writing.
- Commands fail loudly on schema validation errors. There is no fallback to
  prose parsing, transcript scraping, or heuristic recovery.
- `create handoff` assembles the payload from: current phase sheet, notes.md,
  git state, and verification results. It does not require the operator to
  provide this information manually.
- `review prime` assembles the bootstrap cache from: DE, IP, active phase,
  notes, handoff, changed files, and prior findings (if any). It writes both
  the YAML index and a generated markdown briefing. The diff base for
  determining changed files is: `staleness.cache_key.head` when updating an
  existing cache, or the merge-base with the target branch when creating a
  new cache.

### Write ordering and partial-failure rules

Commands that write multiple files follow a defined write order. If a write
fails partway through, the system is in a known partial state that can be
diagnosed and recovered by re-running the command.

| Command | Write order | Partial-failure semantics |
|---------|-------------|---------------------------|
| `create handoff` | 1. `handoff.current.yaml` 2. `state.yaml` | If (1) succeeds but (2) fails: handoff exists but state is stale. `workflow status` reports the old state. Re-running `create handoff` overwrites (1) and retries (2). No data loss. |
| `accept handoff` | 1. `state.yaml` (with `claimed_by`) | Single file — atomic. |
| `review prime` | 1. `review-index.yaml` 2. `review-bootstrap.md` 3. `state.yaml` (pointer update) | If (1) succeeds but (2)/(3) fail: cache exists but bootstrap doc is missing/stale. Re-run `review prime` rebuilds all three. |
| `review complete` | 1. `review-findings.yaml` 2. `state.yaml` | If (1) succeeds but (2) fails: findings are written but state hasn't transitioned. Re-running `review complete` is idempotent on findings and retries state transition. |
| `phase complete` | 1. `state.yaml` (phase status) 2. `handoff.current.yaml` (if auto-handoff) 3. `state.yaml` (workflow status → `awaiting_handoff`) | If (2) fails: phase is marked complete but handoff not emitted. `create handoff` can be run manually. If (2) succeeds but (3) fails: phase is complete, handoff exists, but workflow status is stale. Re-running `phase complete` detects phase is already complete, skips (1), re-emits handoff (idempotent overwrite), and retries (3). |

**Recovery principle:** Every multi-file command is safe to re-run. Re-running
produces the same end state as a successful first run. No command requires
rollback of a prior write to recover.

## 6. Continuation Skill Refit

The continuation skill currently: updates `notes.md` and prints a prompt.

After this change, it will:

1. Update `notes.md` (unchanged behaviour).
2. Invoke `spec-driver create handoff --to <role>` to write the structured
   handoff payload and transition workflow state.
3. Optionally render a continuation prompt from the structured handoff state
   (controlled by `workflow.toml` policy).

The continuation skill always invokes the CLI. It does not write workflow YAML
directly. This maintains the invariant that all workflow mutations flow through
the CLI's schema validation.

### Failure semantics

- If the CLI command fails (schema validation, IO error), **the continuation
  fails.** The skill does not silently fall back to prose-only mode.
- The operator must fix the issue and retry. This preserves the determinism
  invariant: the CLI can always determine state from the control plane files
  without parsing freeform prose.

### Phase-close integration

`spec-driver phase complete` emits a handoff automatically when either:

- The phase bridge block contains `handoff_ready: true`, **or**
- `workflow.auto_handoff_on_phase_complete = true` in `workflow.toml` (the
  default) and no phase bridge block is present.

When a bridge block is present, it takes precedence: `handoff_ready: false`
suppresses automatic handoff even if the policy is enabled.

The `to_role` for the emitted handoff is determined by: `review_required` in
the bridge block (reviewer if true), falling back to `default_next_role` from
`workflow.toml`.

This makes handoff a natural phase-close outcome for all deltas — both those
with explicit phase bridge markup and those relying on policy defaults. Manual
`create handoff` remains available for non-phase-boundary transitions.

The continuation skill may invoke `phase complete` instead of `create handoff`
when completing a phase, letting the policy or phase bridge drive handoff
emission.

## 7. Bridge Sections

Fenced YAML blocks in existing markdown artifacts. These are pointers and
phase-close signals, not workflow state.

### 7.1 Notes bridge — `supekku:workflow.notes-bridge@v1`

**Location:** `notes.md`, under the "New Agent Instructions" heading.

```yaml
schema: supekku.workflow.notes-bridge
version: 1
artifact: DE-090                   # required — string, entity ID
workflow_state: workflow/state.yaml  # required — string, relative path
current_handoff: workflow/handoff.current.yaml  # optional — string
review_index: workflow/review-index.yaml  # optional — string
review_findings: workflow/review-findings.yaml  # optional — string
review_bootstrap: workflow/review-bootstrap.md  # optional — string
```

**Required fields:** `schema`, `version`, `artifact`, `workflow_state`.

**Semantics:** Pure pointer block. Contains no state of its own. Written by
`create handoff` or `phase complete`. Must not contradict `workflow/state.yaml`
— if it does, the bridge is stale and should be regenerated by re-running the
relevant CLI command.

### 7.2 Phase bridge — `supekku:workflow.phase-bridge@v1`

**Location:** `phases/phase-NN.md`, under a "Workflow" heading.

```yaml
schema: supekku.workflow.phase-bridge
version: 1
phase: IP-090.PHASE-05             # required — string, phase ID
status: complete                   # required — not_started | in_progress | blocked | complete | skipped
handoff_ready: true                # required — boolean
review_required: true              # optional — boolean, defaults to false
current_handoff: ../workflow/handoff.current.yaml  # optional — string, relative path
```

**Required fields:** `schema`, `version`, `phase`, `status`, `handoff_ready`.

**Semantics:** `handoff_ready` and `review_required` are inputs to
`phase complete`. When `handoff_ready` is true, `phase complete` emits a
handoff automatically. `review_required` determines the default `to_role`.
The bridge block is advisory to the CLI — it is not an independent authority
and does not override `workflow/state.yaml`.

## 8. Lifecycle and Invalidation Rules

### Bootstrap status states

| State | Meaning | Entry condition |
|-------|---------|-----------------|
| `cold` | No reviewer cache exists | Initial state; after explicit cache deletion |
| `warming` | Cache is being built | During `review prime` execution |
| `warm` | Cache is current and usable | After successful `review prime`; all cache keys match |
| `stale` | Cache exists but inputs have changed | Any staleness trigger fires (see below) |
| `invalid` | Cache is structurally unusable | Schema version mismatch; corrupt file; referenced files deleted |
| `reusable` | Cache is stale but recoverable | Staleness is bounded (see rules below) |

### Staleness triggers

The review index moves from `warm` to `stale` when any of these conditions is
detected by `review prime`:

1. **Commit drift:** `git.head` in the cache key differs from current HEAD.
2. **Dependency surface expansion:** touched files fall outside the cached
   `domain_map`.
3. **Major scope change:** `DE-*.md` frontmatter `updated` timestamp is newer
   than the cache.
4. **Phase boundary crossing:** `phase.id` in the cache key differs from the
   current active phase.

### Invalidation rules (stale → invalid)

A stale cache becomes `invalid` when:

1. The `review-index.yaml` schema version does not match the current CLI's
   expected version.
2. The file is malformed or fails schema validation.
3. Any file referenced in `domain_map` has been deleted (not just modified).
4. The owning artifact (`DE-*.md`) has been deleted or its ID has changed.

### Reusability rules (stale → reusable)

A stale cache is `reusable` (can be incrementally updated rather than rebuilt
from scratch) when:

1. The staleness is caused only by commit drift or minor scope change.
2. The `domain_map` file list is a subset of the current changed-file set
   (surface contracted, not expanded).
3. No invariants in the cache have been contradicted by the new state.

When reusable, `review prime` performs an incremental update: refreshes the
cache key, updates the domain map delta, and preserves existing invariants and
known decisions. When not reusable, it performs a full rebuild.

### Session death and recovery

These rules apply to **all roles**, not only reviewers.

**When `sessions.yaml` reports any role's session as `dead`:**

- Artifact-persisted state is unaffected. `review-index.yaml`,
  `review-findings.yaml`, `handoff.current.yaml`, and `state.yaml` survive
  session death — they are file-persisted, not session-persisted.
- A new session for that role can be created and bootstrapped from existing
  artifacts without a cold start.
- For reviewers: prior findings are preserved; the new session inherits the
  current round number and does not reset the counter.
- For implementers: the handoff payload and notes remain available for the
  new session to consume via `accept handoff`.

**When `sessions.yaml` is absent:**

- All workflow commands function normally. `sessions.yaml` is informational
  only. `accept handoff`, `workflow status`, `review prime`, and all other
  commands derive their behaviour from `state.yaml`, `handoff.current.yaml`,
  and `review-*.yaml` — never from `sessions.yaml`.

**When `sessions.yaml` references a session that no longer exists:**

- The entry is stale but harmless. The CLI may warn on `workflow status` but
  does not block any transition. The operator or a future orchestration layer
  can update `sessions.yaml` to reflect reality.

**Partially written `sessions.yaml`:**

- If the file exists but fails schema validation, the CLI ignores it with a
  warning. No workflow command depends on `sessions.yaml` for correctness.

### Partial failure: findings vs review index disagreement

If `review-findings.yaml` references a round number higher than what
`review-index.yaml` was bootstrapped for:

- The review index is marked `stale` (not `invalid`).
- `review prime` performs an incremental update, incorporating the findings
  from the newer round into the reviewer's cached understanding.
- No findings are lost. The findings ledger is append-only within a round and
  monotonically versioned across rounds.

### Teardown

Reviewer state (`review-index.yaml`, `review-findings.yaml`,
`review-bootstrap.md`) is torn down (deleted) on:

- `approved` — review is complete (when `review.session_scope = "artifact"`).
- Phase approval (when `review.session_scope = "phase"` — cache is torn down
  at phase boundaries and rebuilt for the next phase).
- Explicit `spec-driver review teardown` command.
- Delta closure (`spec-driver complete delta`).

Reviewer state is **not** torn down on `changes_requested` — it persists across
the implement-review cycle.

The `review.session_scope` policy in `workflow.toml` controls this granularity.
When set to `artifact`, the reviewer cache carries across phases (suitable for
a persistent reviewer who follows the whole delta). When set to `phase`, the
cache is scoped to each phase (suitable when different phases have substantially
different review surfaces).

## 9. Global Policy Configuration

**File:** `.spec-driver/workflow.toml`

```toml
version = 1

[workflow]
state_dir = "workflow"
handoff_boundary = "phase"           # default trigger boundary
auto_handoff_on_phase_complete = true  # emit handoff on phase complete even without bridge block
render_continuation_prompt = true    # print prompt after handoff
update_notes_new_agent_instructions = true
default_next_role = "implementer"

[review]
persistent_session = true            # prefer session reuse
bootstrap_cache = true               # maintain review-index.yaml
session_scope = "artifact"           # artifact | phase — controls teardown granularity
teardown_on = ["approved", "abandoned"]
recreate_on = ["major_scope_change", "cache_invalid", "session_unrecoverable"]

[review.bootstrap]
include_delta = true
include_plan = true
include_active_phase = true
include_notes = true
include_findings = true
include_changed_files = true
max_historical_rounds = 2
```

Policy only. Not per-artifact state. Safe to load once at CLI start.

## 10. Validation Rules

### Hard validation

Applies to all `workflow/*.yaml` files and fenced bridge blocks.

- `schema` and `version` required.
- Unknown top-level keys rejected unless `x-` prefixed.
- Path fields must be relative to artifact root or repo root.
- IDs must match known artifact ID patterns.
- Enum fields must use declared values.
- CLI commands validate before writing; invalid writes fail the command.
- Nested record required fields enforced per §3 constraints.

### Soft validation

Applies to `review-bootstrap.md` and prose sections of `notes.md`.

These are human-readable surfaces. Only the machine-facing bridge blocks within
them are schema-validated.

## 11. Migration and Compatibility

### No breaking changes

- DE frontmatter: unchanged.
- IP overview/verification blocks: unchanged.
- Phase sheet format: unchanged (bridge block is optional).
- `notes.md`: unchanged (bridge block is optional, placed under existing
  "New Agent Instructions" heading).

### Adoption path

1. `workflow/` directory appears only when an orchestration command is first
   run against a delta. Existing deltas without `workflow/` continue to work.
2. The continuation skill detects whether orchestration is active (presence of
   `workflow/state.yaml`) and emits structured handoff only when it is.
3. Bridge blocks are written by CLI commands, not required to be manually added.
4. No migration script needed. No existing artifact needs editing.

## 13. Open Questions

1. **Finding ID format:** Should review finding IDs be globally unique
   (`F-001`) or round-scoped (`R3-001`)? Round-scoped is simpler but risks
   collision across artifacts if findings are ever aggregated.

2. **`workflow.toml` location:** Global (`.spec-driver/workflow.toml`) vs
   per-delta (`workflow/config.toml`)? Global is simpler; per-delta allows
   different review policies per artifact.

## 14. Closed Questions

1. **Bridge block authoring:** The CLI command that creates the handoff writes
   the bridge block in `notes.md`. The continuation skill owns prose content;
   the CLI owns the fenced YAML bridge block. These are different sections of
   the same file with different owners — the bridge block is a pointer to
   machine state, not prose narrative.

2. **Direct YAML write path:** Removed. The continuation skill invokes CLI
   commands exclusively. Sandbox/harness environments that cannot invoke the
   CLI are an IMPR-020 adapter concern, not a control-plane design concern.

3. **Handoff transition status enum:** Changed from review-centric values
   (`awaiting_review`, `changes_requested`, `approved`) to role-generic
   values (`pending`, `accepted`, `superseded`). Review-specific semantics
   live in the state machine and `review-findings.yaml`, not in the handoff
   payload.

4. **Handoff supersession observability:** Supersession is a latest-wins file
   overwrite. The old payload is not preserved in-band — it is observable only
   in git history. No in-file status field is set on the prior handoff because
   the prior file no longer exists after overwrite.

5. **Claim guard identity source:** `$USER` is a convenience default, not a
   correctness guarantee. Concurrent agent sessions under the same Unix user
   must provide unique `--identity` values. The CLI does not auto-generate
   identities.
