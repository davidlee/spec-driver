# DR-102: Handover and Review Orchestration

This design revision defines the workflow control plane for spec-driver: a set
of machine-readable YAML artifacts, a CLI command surface, and lifecycle rules
that make phase transitions deterministic, handoffs structured, and reviewer
bootstrap amortizable — without replacing existing markdown workflow artifacts.

## 1. Design Scope and Boundaries

### In scope (layers 1–2)

- Workflow semantics: states, transitions, roles, handoff, review rounds
- File-based control plane: YAML schemas, directory layout, bridge blocks
- CLI command surface for orchestration operations
- Continuation skill refit to emit structured handoff
- Lifecycle and invalidation rules for reviewer state

### Out of scope (layers 3–5, tracked in IMPR-020)

- Process supervision, daemons, spawn/detach/observe
- IPC, mailboxes, file-watch event loops
- Sandbox/multiplexer/harness management
- Agent adapter configuration (runner profiles, installer hooks)
- Policy engine (model assignment, budget, escalation)

### Authority model

Unambiguous and one-directional:

1. **Markdown artifacts are authority.** `DE-*.md` owns intent. `IP-*.md` owns
   planning and verification. `phases/*.md` owns execution. `notes.md` owns
   human-readable narrative.
2. **YAML workflow files are derived projections.** `workflow/*.yaml` is written
   exclusively by CLI commands. It reflects the current orchestration state but
   does not override markdown authority.
3. **CLI commands are the mutation surface.** State transitions happen through
   explicit CLI invocations, not watch loops, daemons, or implicit file events.
4. **No reconciliation hammer.** If YAML and markdown disagree, the YAML is
   wrong. The fix is to re-run the relevant CLI command, not a sync sweep.

There is no daemon. There is no watcher. There is no bidirectional
reconciliation. The control plane is a projection that the CLI keeps current
through explicit command execution.

## 2. Artifact Structure

### Directory layout

```
.spec-driver/deltas/DE-xxx-.../
  DE-xxx.md                        # existing — design/intent authority
  IP-xxx.md                        # existing — plan/verification authority
  notes.md                         # existing — human narrative
  phases/                          # existing — per-phase execution
  workflow/                        # NEW — machine-readable control plane
    state.yaml                     #   orchestration status + pointers
    handoff.current.yaml           #   durable transition payload
    review-index.yaml              #   reviewer bootstrap cache
    review-findings.yaml           #   stable issue ledger
    sessions.yaml                  #   runtime session map (optional)
    review-bootstrap.md            #   generated reviewer briefing
```

Additive only. No existing file formats change.

### Ownership model

| File | Owner | Authored vs derived |
|------|-------|---------------------|
| `DE-*.md` | human / design agent | authored |
| `IP-*.md` | human / planning agent | authored |
| `phases/*.md` | executing agent | authored |
| `notes.md` | executing agent + continuation skill | authored |
| `workflow/state.yaml` | CLI (`handoff`, `review`, `phase` commands) | derived |
| `workflow/handoff.current.yaml` | CLI (`create handoff`) | derived |
| `workflow/review-index.yaml` | CLI (`review prime`) | derived |
| `workflow/review-findings.yaml` | reviewer agent via CLI (`review complete`) | derived |
| `workflow/sessions.yaml` | CLI (session commands) | derived, optional |
| `workflow/review-bootstrap.md` | CLI (`review prime`) | generated |

## 3. Artifact Schemas

### 3.1 Workflow State — `supekku:workflow.state@v1`

**File:** `workflow/state.yaml`

**Purpose:** Current orchestration status and pointers. 100% derived from CLI
transitions.

```yaml
schema: supekku.workflow.state
version: 1

artifact:
  id: DE-090                       # required — entity ID
  kind: delta                      # required — delta | plan | revision | audit | task | other
  path: .spec-driver/deltas/DE-090-slug
  notes_path: .spec-driver/deltas/DE-090-slug/notes.md

plan:
  id: IP-090
  path: .spec-driver/deltas/DE-090-slug/IP-090.md

phase:
  id: IP-090.PHASE-05              # required — current phase
  path: .spec-driver/deltas/DE-090-slug/phases/phase-05.md
  status: complete                 # required — not_started | in_progress | blocked | complete | skipped

workflow:
  status: awaiting_handoff         # required — see §4 state machine
  active_role: implementer         # required — architect | implementer | reviewer | operator | other
  next_role: reviewer
  handoff_boundary: phase          # phase | task | manual

pointers:
  current_handoff: workflow/handoff.current.yaml
  review_index: workflow/review-index.yaml
  review_findings: workflow/review-findings.yaml
  sessions: workflow/sessions.yaml
  review_bootstrap: workflow/review-bootstrap.md

timestamps:
  created: 2026-03-21T10:00:00Z
  updated: 2026-03-21T10:30:00Z
```

**Required fields:** `schema`, `version`, `artifact.id`, `artifact.kind`,
`workflow.status`, `workflow.active_role`, `phase.id`, `phase.status`.

**Validation:** Unknown top-level keys allowed only if `x-` prefixed. Path
fields must be relative to artifact root or repo root. IDs must match known
artifact ID patterns.

### 3.2 Handoff — `supekku:workflow.handoff@v1`

**File:** `workflow/handoff.current.yaml`

**Purpose:** Durable phase-boundary transition payload. Structured equivalent of
what the continuation skill currently asks agents to write in prose. A handoff
can target any role — it is not review-specific.

```yaml
schema: supekku.workflow.handoff
version: 1

artifact:
  id: DE-090
  kind: delta

transition:
  from_role: implementer           # required
  to_role: reviewer                # required — any valid role
  boundary: phase                  # phase | task | manual
  status: awaiting_review          # awaiting_review | changes_requested |
                                   # ready_for_implementation | approved | blocked

phase:
  id: IP-090.PHASE-05              # required
  status: complete

required_reading:                  # required — minimum context for next agent
  - .spec-driver/deltas/DE-090-slug/DE-090.md
  - .spec-driver/deltas/DE-090-slug/IP-090.md
  - .spec-driver/deltas/DE-090-slug/phases/phase-05.md
  - .spec-driver/deltas/DE-090-slug/notes.md

related_documents: []
key_files:
  - supekku/cli/common.py
  - supekku/core/relation_types.py

verification:
  commands:
    - uv run pytest
    - ruff check .
  summary: "3955 tests passing; ruff clean except pre-existing UP042"
  status: pass                     # pass | fail | partial | not_run | unknown

git:
  head: e31eca9
  branch: null
  worktree:
    has_uncommitted_changes: true
    has_staged_changes: false

open_items:
  - id: OI-001
    kind: next_step
    summary: Implement Phase 06 neighbourhood view
    blocking: false

design_tensions: []
unresolved_assumptions: []
decisions_to_preserve: []

next_activity:
  kind: review                     # required — implementation | review | architecture |
                                   #            verification | operator_attention
  summary: Resume persistent reviewer and assess Phase 05 output

timestamps:
  emitted_at: 2026-03-21T10:30:00Z
```

**Required fields:** `schema`, `version`, `artifact.id`, `transition.from_role`,
`transition.to_role`, `phase.id`, `required_reading`, `next_activity.kind`.

### 3.3 Review Index — `supekku:workflow.review-index@v1`

**File:** `workflow/review-index.yaml`

**Purpose:** Reviewer bootstrap cache (tier 2 persistence). What the reviewer
has already learned and should not rediscover. Separate from findings.

```yaml
schema: supekku.workflow.review-index
version: 1

artifact:
  id: DE-090
  kind: delta

review:
  session_scope: artifact          # artifact | phase | task
  bootstrap_status: warm           # required — cold | warming | warm | stale | invalid | reusable
  last_bootstrapped_at: 2026-03-21T10:25:00Z
  source_handoff: workflow/handoff.current.yaml

domain_map:                        # required — reviewer's cached understanding
  - area: cli
    purpose: command routing and filter orchestration
    files:
      - supekku/cli/common.py
      - supekku/cli/show.py
  - area: relation_collection
    purpose: collecting references and reverse-reference targets
    files:
      - supekku/core/relation_types.py

invariants:
  - id: INV-001
    summary: JSON list/show output should stay contract-consistent for backlog items
  - id: INV-002
    summary: Reverse lookup features must preserve existing skinny-CLI patterns

risk_areas:
  - id: RA-001
    summary: Reverse lookup logic may miss domain-field references
    files:
      - supekku/core/relation_types.py

review_focus:
  - output contract stability
  - correctness of reverse-reference partitioning
  - alias/conflict handling

known_decisions:
  - id: KD-001
    summary: reviewer should treat requirements as using uid not id where relevant

staleness:
  cache_key:                       # required
    phase_id: IP-090.PHASE-05
    head: e31eca9
  invalidation_triggers:
    - major_scope_change
    - dependency_surface_expanded
    - artifact_rebased
```

**Required fields:** `schema`, `version`, `artifact.id`,
`review.bootstrap_status`, `domain_map`, `staleness.cache_key`.

### 3.4 Review Findings — `supekku:workflow.review-findings@v1`

**File:** `workflow/review-findings.yaml`

**Purpose:** Stable issue ledger across review rounds. Machine-readable input
for an implementer returning from review.

```yaml
schema: supekku.workflow.review-findings
version: 1

artifact:
  id: DE-090
  kind: delta

  round: 3                         # required
  status: changes_requested        # required — not_started | in_progress |
                                   #            changes_requested | approved | blocked

blocking:
  - id: R3-001
    title: Output mode regression for downstream scripts
    summary: New reverse-reference output changes a field relied on by existing JSON consumers
    status: open                   # open | resolved | waived | superseded
    files:
      - supekku/cli/output.py
    related_invariants:
      - INV-001

non_blocking:
  - id: R3-002
    title: Add regression test for audit reverse lookups
    summary: Coverage gap; behavior appears correct
    status: open
    files:
      - tests/cli/show_test.py

resolved:
  - id: R2-001
    title: Requirement identifier mismatch
    status: resolved
    resolution_summary: Switched requirements partition logic to use uid

waived: []

history:
  - round: 1
    summary: Initial review of reverse-reference filtering
  - round: 2
    summary: Follow-up after alias conflict fixes

timestamps:
  updated: 2026-03-21T10:35:00Z
```

**Required fields:** `schema`, `version`, `artifact.id`, `review.round`,
`review.status`.

### 3.5 Sessions — `supekku:workflow.sessions@v1`

**File:** `workflow/sessions.yaml`

**Purpose:** Runtime mapping of roles to sessions. Optional — only meaningful
when the operator or a future orchestration layer manages persistent sessions.

```yaml
schema: supekku.workflow.sessions
version: 1

artifact:
  id: DE-090
  kind: delta

sessions:
  implementer:
    session_name: sd-DE-090-impl
    sandbox: pi-mono
    status: paused                 # active | paused | absent | dead | unknown
    last_seen: 2026-03-21T10:20:00Z
  reviewer:
    session_name: sd-DE-090-review
    sandbox: gemini-review
    status: active
    last_seen: 2026-03-21T10:34:00Z
```

**Allowed status values:** `active`, `paused`, `absent`, `dead`, `unknown`.

This file carries no authority. It is informational state for tooling that may
manage sessions. Its absence does not affect any workflow transition.

## 4. State Machine

### States

`planned`, `implementing`, `awaiting_handoff`, `reviewing`,
`changes_requested`, `approved`, `blocked`.

### Transitions

All transitions are triggered by explicit CLI commands. No implicit triggers.

```
planned ──[phase start]──────────────────► implementing
implementing ──[create handoff --to <role>]──► awaiting_handoff
awaiting_handoff ──[accept handoff]──────────► implementing  (if to_role = implementer)
awaiting_handoff ──[accept handoff]──────────► reviewing     (if to_role = reviewer)
reviewing ──[review complete --status changes_requested]──► changes_requested
reviewing ──[review complete --status approved]───────────► approved
changes_requested ──[accept handoff]─────────► implementing
implementing ──[create handoff --to <role>]──► awaiting_handoff  (cycle)
* ──[block]──────────────────────────────────► blocked
blocked ──[unblock]──────────────────────────► (previous state)
```

### Design choices

- **`awaiting_handoff` replaces `awaiting_review`.** Handoff is role-generic.
  The `to_role` field in the handoff payload determines what happens when the
  handoff is accepted, not the state name. `workflow status` must display the
  target role prominently (e.g. "awaiting handoff → reviewer") so the operator
  knows which direction work is flowing.
- **`accept handoff` is explicit and idempotent.** The receiving agent or
  operator runs this command. There is no daemon to detect and auto-accept.
  Re-accepting an already-accepted handoff is a no-op. Concurrent multi-agent
  race conditions are an IMPR-020 concern; this layer assumes single-writer
  by convention.
- **No `create handoff` → auto-transition to `reviewing`.** The handoff sits
  in `awaiting_handoff` until explicitly accepted. This is the clean separation
  between "work is offered" and "work is picked up."
- **Creating a new handoff supersedes any unaccepted prior handoff.**
  `handoff.current.yaml` carries latest-wins semantics. The superseded payload
  is not preserved (recoverable from git history if needed).

## 5. CLI Command Surface

### Core commands

| Command | Effect |
|---------|--------|
| `spec-driver phase start` | `planned` → `implementing`; initialises `workflow/state.yaml` |
| `spec-driver create handoff --to <role>` | writes `handoff.current.yaml`; transitions to `awaiting_handoff` |
| `spec-driver accept handoff` | reads handoff, transitions to `implementing` or `reviewing` based on `to_role` |
| `spec-driver review prime` | generates `review-index.yaml` + `review-bootstrap.md` from current state |
| `spec-driver review complete --status <status>` | writes `review-findings.yaml`; transitions to `changes_requested` or `approved` |
| `spec-driver workflow status` | reads `workflow/state.yaml` and prints human-readable summary |
| `spec-driver block` / `spec-driver unblock` | transitions to/from `blocked` |

### Command behaviour

- Every command that mutates state writes the relevant YAML file(s)
  atomically (write-to-temp + rename).
- Every command validates its output against the schema before writing.
- Commands fail loudly on schema validation errors. There is no fallback to
  prose parsing, transcript scraping, or heuristic recovery.
- `create handoff` assembles the payload from: current phase sheet, notes.md,
  git state, and verification results. It does not require the operator to
  provide this information manually.
- `review prime` assembles the bootstrap cache from: DE, IP, active phase,
  notes, handoff, changed files, and prior findings (if any). It writes both
  the YAML index and a generated markdown briefing. The diff base for
  determining changed files is: `staleness.cache_key.head` when updating an
  existing cache, or the merge-base with the target branch when creating a
  new cache.

## 6. Continuation Skill Refit

The continuation skill currently: updates `notes.md` and prints a prompt.

After this change, it will:

1. Update `notes.md` (unchanged behaviour).
2. Write `workflow/handoff.current.yaml` with the structured handoff payload.
3. Optionally render a continuation prompt from the structured handoff state
   (controlled by `workflow.toml` policy).

### Authorship paths

The handoff payload can be written via two equivalent paths:

- **CLI invocation:** The skill shells out to `spec-driver create handoff`.
  Preferred when the CLI is available on PATH.
- **Direct YAML write:** The skill constructs the payload in-process and writes
  `handoff.current.yaml` directly, applying the same schema validation. This
  path is necessary when the agent runs in a sandbox without CLI access or
  when the skill is embedded in a harness that cannot shell out.

Both paths produce identical, schema-validated output. Neither path involves
prose parsing or transcript scraping.

### Failure semantics

- If the YAML write fails (schema validation, IO error), **the command fails.**
  The continuation skill does not silently fall back to prose-only mode.
- The operator must fix the issue and retry. This preserves the determinism
  invariant: the CLI can always determine state from the control plane files
  without parsing freeform prose.

## 8. Lifecycle and Invalidation Rules

### Bootstrap status states

| State | Meaning | Entry condition |
|-------|---------|-----------------|
| `cold` | No reviewer cache exists | Initial state; after explicit cache deletion |
| `warming` | Cache is being built | During `review prime` execution |
| `warm` | Cache is current and usable | After successful `review prime`; all cache keys match |
| `stale` | Cache exists but inputs have changed | Any staleness trigger fires (see below) |
| `invalid` | Cache is structurally unusable | Schema version mismatch; corrupt file; referenced files deleted |
| `reusable` | Cache is stale but recoverable | Staleness is bounded (see rules below) |

### Staleness triggers

The review index moves from `warm` to `stale` when any of these conditions is
detected by `review prime`:

1. **Commit drift:** `git.head` in the cache key differs from current HEAD.
2. **Dependency surface expansion:** touched files fall outside the cached
   `domain_map`.
3. **Major scope change:** `DE-*.md` frontmatter `updated` timestamp is newer
   than the cache.
4. **Phase boundary crossing:** `phase.id` in the cache key differs from the
   current active phase.

### Invalidation rules (stale → invalid)

A stale cache becomes `invalid` when:

1. The `review-index.yaml` schema version does not match the current CLI's
   expected version.
2. The file is malformed or fails schema validation.
3. Any file referenced in `domain_map` has been deleted (not just modified).
4. The owning artifact (`DE-*.md`) has been deleted or its ID has changed.

### Reusability rules (stale → reusable)

A stale cache is `reusable` (can be incrementally updated rather than rebuilt
from scratch) when:

1. The staleness is caused only by commit drift or minor scope change.
2. The `domain_map` file list is a subset of the current changed-file set
   (surface contracted, not expanded).
3. No invariants in the cache have been contradicted by the new state.

When reusable, `review prime` performs an incremental update: refreshes the
cache key, updates the domain map delta, and preserves existing invariants and
known decisions. When not reusable, it performs a full rebuild.

### Dead session rules

When `sessions.yaml` reports a reviewer session as `dead`:

- The `review-index.yaml` cache remains valid (it is artifact-persisted, not
  session-persisted).
- A new reviewer session can be created and bootstrapped from the existing
  `review-index.yaml` + `review-bootstrap.md` without a cold start.
- Prior findings in `review-findings.yaml` are preserved and available to the
  new session.
- The new session inherits the current review round number; it does not reset
  the round counter.

### Partial failure: findings vs review index disagreement

If `review-findings.yaml` references a round number higher than what
`review-index.yaml` was bootstrapped for:

- The review index is marked `stale` (not `invalid`).
- `review prime` performs an incremental update, incorporating the findings
  from the newer round into the reviewer's cached understanding.
- No findings are lost. The findings ledger is append-only within a round and
  monotonically versioned across rounds.

### Teardown

Reviewer state (`review-index.yaml`, `review-findings.yaml`,
`review-bootstrap.md`) is torn down (deleted) on:

- `approved` — review is complete (when `review.session_scope = "artifact"`).
- Phase approval (when `review.session_scope = "phase"` — cache is torn down
  at phase boundaries and rebuilt for the next phase).
- Explicit `spec-driver review teardown` command.
- Delta closure (`spec-driver complete delta`).

Reviewer state is **not** torn down on `changes_requested` — it persists across
the implement-review cycle.

The `review.session_scope` policy in `workflow.toml` controls this granularity.
When set to `artifact`, the reviewer cache carries across phases (suitable for
a persistent reviewer who follows the whole delta). When set to `phase`, the
cache is scoped to each phase (suitable when different phases have substantially
different review surfaces).

## 9. Global Policy Configuration

**File:** `.spec-driver/workflow.toml`

```toml
version = 1

[workflow]
state_dir = "workflow"
handoff_boundary = "phase"           # default trigger boundary
render_continuation_prompt = true    # print prompt after handoff
update_notes_new_agent_instructions = true
default_next_role = "implementer"

[review]
persistent_session = true            # prefer session reuse
bootstrap_cache = true               # maintain review-index.yaml
session_scope = "artifact"           # artifact | phase — controls teardown granularity
teardown_on = ["approved", "abandoned"]
recreate_on = ["major_scope_change", "cache_invalid", "session_unrecoverable"]

[review.bootstrap]
include_delta = true
include_plan = true
include_active_phase = true
include_notes = true
include_findings = true
include_changed_files = true
max_historical_rounds = 2
```

Policy only. Not per-artifact state. Safe to load once at CLI start.

## 10. Validation Rules

### Hard validation

Applies to all `workflow/*.yaml` files and fenced bridge blocks.

- `schema` and `version` required.
- Unknown top-level keys rejected unless `x-` prefixed.
- Path fields must be relative to artifact root or repo root.
- IDs must match known artifact ID patterns.
- Enum fields must use declared values.
- CLI commands validate before writing; invalid writes fail the command.

### Soft validation

Applies to `review-bootstrap.md` and prose sections of `notes.md`.

These are human-readable surfaces. Only the machine-facing bridge blocks within
them are schema-validated.

## 11. Migration and Compatibility

### No breaking changes

- DE frontmatter: unchanged.
- IP overview/verification blocks: unchanged.
- Phase sheet format: unchanged (bridge block is optional).
- `notes.md`: unchanged (bridge block is optional, placed under existing
  "New Agent Instructions" heading).

### Adoption path

1. `workflow/` directory appears only when an orchestration command is first
   run against a delta. Existing deltas without `workflow/` continue to work.
2. The continuation skill detects whether orchestration is active (presence of
   `workflow/state.yaml`) and emits structured handoff only when it is.
3. Bridge blocks are written by CLI commands, not required to be manually added.
4. No migration script needed. No existing artifact needs editing.

## 12. Evaluation Against Brief Criteria

| Criterion (from IMPR-019) | Satisfied | How |
|---------------------------|-----------|-----|
| Phase completion emits structured handoff without copy-paste | ✓ | `create handoff` assembles payload from phase/notes/git state |
| CLI resumes correct next role deterministically | ✓ | `accept handoff` reads `to_role` from handoff payload |
| Reviewer resumed for round 3 without reconstructing rounds 1–2 | ✓ | `review-index.yaml` caches bootstrap state; `review-findings.yaml` preserves prior rounds |
| Dead reviewer recreated without true cold start | ✓ | Cache and findings survive session death; new session bootstraps from artifacts |
| Status inspectable from predictable files | ✓ | `workflow/state.yaml` is the single status file |
| DE/IP/phase schemas left untouched | ✓ | No changes to existing schemas |
| `notes.md` remains readable and useful | ✓ | Optional bridge block only; prose narrative unchanged |

## 13. Open Questions

1. **Finding ID format:** Should review finding IDs be globally unique
   (`F-001`) or round-scoped (`R3-001`)? Round-scoped is simpler but risks
   collision across artifacts if findings are ever aggregated.

2. **`workflow.toml` location:** Global (`.spec-driver/workflow.toml`) vs
   per-delta (`workflow/config.toml`)? Global is simpler; per-delta allows
   different review policies per artifact.

## 14. Closed Questions

1. **Bridge block authoring:** The CLI command that creates the handoff writes
   the bridge block in `notes.md`. The continuation skill owns prose content;
   the CLI owns the fenced YAML bridge block. These are different sections of
   the same file with different owners — the bridge block is a pointer to
   machine state, not prose narrative.
