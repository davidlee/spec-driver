## 2. Motivation

DE-102 completed the design phase for workflow orchestration. DR-102 defines a
machine-readable control plane (YAML artifacts, CLI commands, state machine,
bridge blocks) that allows spec-driver to:

- Emit structured handoffs at phase boundaries without operator copy-paste
- Resume the correct next role deterministically via `accept handoff`
- Support persistent reviewer sessions across multiple review rounds
- Recover cleanly from agent/session death
- Amortize reviewer bootstrap cost via cached review indexes

This delta implements that design.

## 3. Scope & Objectives

- **Primary Outcomes**:
  - YAML schema definitions and validation for 5 workflow artifacts (state, handoff, review-index, review-findings, sessions) and 2 bridge blocks (notes-bridge, phase-bridge)
  - Schema registration in the block schema registry (`spec-driver list schemas` / `spec-driver show schema`)
  - CLI command surface: `phase start`, `create handoff`, `accept handoff`, `review prime`, `review complete`, `workflow status`, `review teardown`, `block`/`unblock`, `phase complete`
  - State machine implementation (7 states, explicit CLI-driven transitions, claim guard)
  - Continuation skill refit to invoke `create handoff` CLI command
  - Bridge block support in `notes.md` and phase sheets
  - `workflow.toml` extensions (`[workflow]` and `[review]` sections)
  - Memory and documentation updates
- **Operational Constraints**: Additive only — no existing DE/IP/phase/notes schema changes. Existing deltas must continue to work without migration.
- **Dependencies**: None (DR-102 is complete).

## 4. Out of Scope

- Runtime orchestration (process supervision, IPC, sandbox management) — IMPR-020
- Agent adapter configuration (harness-specific write paths) — IMPR-020
- Policy engine (model assignment, budget, escalation) — IMPR-020
- File-watch event loops or daemons — IMPR-020
- Workflow hook installation system (FR-003/FR-004) — separate PROD-011 work
- VCS abstraction (FR-008) — separate PROD-011 work
- Constitution validation tooling (FR-006/FR-007) — separate PROD-011 work

## 5. Approach Overview

- **System Touchpoints**: `supekku/scripts/lib/blocks/` (schema registry), `supekku/cli/` (command routing), continuation skill, `workflow.toml` config loader
- **Key Changes**:
  - New `supekku/scripts/lib/workflow/` package for schema definitions, state machine, and orchestration logic
  - New CLI commands registered under `spec-driver` (see DR-102 §5)
  - Continuation skill updated to invoke CLI for structured handoff (DR-102 §6)
  - Bridge block renderers for notes-bridge and phase-bridge (DR-102 §7)
  - Config loader extended for `[workflow]` and `[review]` sections (DR-102 §9)
- **Migration / Rollout Notes**: No migration needed. `workflow/` directory appears only when an orchestration command is first run against a delta. Continuation skill detects presence of `workflow/state.yaml` and emits structured handoff only when active.

## 6. Verification Strategy

- **Requirements Coverage**: Each CLI command tested for correct state transitions and YAML output. Schema validation tested for required fields, enum constraints, and unknown-key rejection.
- **Planned Artefacts**: VT per phase (see IP-103)
- **Acceptance Criteria**:
  - All 7 DR-102 evaluation criteria satisfied (§12)
  - Schema validation rejects invalid input and accepts valid input
  - State machine transitions match DR-102 §4 exactly
  - Claim guard prevents double-acceptance
  - `workflow status` renders correct human-readable summary
  - Continuation skill emits structured handoff via CLI
  - Existing deltas without `workflow/` continue to work unchanged

## 7. Risks & Mitigations

- **Risk**: Schema design needs refinement during implementation — _Likelihood_: medium — _Impact_: low — _Mitigation_: DR-102 schemas are detailed; adjust within the approved design envelope, surface any deviations in notes.
- **Risk**: CLI command surface is large (9+ commands) — _Likelihood_: medium — _Impact_: medium — _Mitigation_: Phase incrementally; core commands first (state, handoff), review commands second.
- **Risk**: Continuation skill refit breaks existing behaviour — _Likelihood_: low — _Impact_: high — _Mitigation_: Feature-gated by presence of `workflow/state.yaml`; old behaviour preserved when absent.
