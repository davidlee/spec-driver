## 1. Summary

- **Delta**: DE-103 – implement workflow orchestration control plane
- **Design Authority**: DR-102 (approved, from DE-102)
- **Specs Impacted**: PROD-011
- **Backlog**: IMPR-019
- **Desired Outcome**: A working file-based orchestration control plane with CLI commands, schema validation, state machine, and continuation skill integration — per DR-102.

## 2. Context & Constraints

- **Current Behaviour**: Phase transitions are manual. Continuation skill writes prose to notes.md. No machine-readable workflow state. Reviewer bootstrap is cold-start every round.
- **Target Behaviour**: CLI-driven state machine with structured YAML artifacts. Deterministic handoff, reviewer cache persistence, claim-guarded acceptance. Per DR-102.
- **Dependencies**: None (DR-102 is complete).
- **Constraints**: Additive only — no existing artifact schema changes. Feature-gated by presence of `workflow/state.yaml`.

## 6. Testing & Verification Plan

- **Schema validation tests** (Phase 01): Required fields, enum validation, unknown-key rejection, optional field handling for all 7 schemas.
- **State machine tests** (Phase 02): All transitions from DR-102 §4, invalid transitions rejected, `claimed_by` semantics.
- **Command integration tests** (Phases 02–05): Each CLI command tested for correct file output, state transitions, and partial-failure recovery (re-run safety).
- **Continuation skill tests** (Phase 05): Feature detection, CLI invocation, failure propagation (no silent prose fallback).
- **End-to-end tests** (Phase 06): Full workflow cycle (start → implement → handoff → review → changes_requested → handoff → approve) verified against DR-102 §12 criteria.
- **Regression tests** (Phase 06): Existing deltas without `workflow/` continue to work.
