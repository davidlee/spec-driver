## 1. Summary & Context

- **Product Spec(s)**: PROD-006 – Phase Management; PROD-011 – Implementation Execution Workflow
- **Technical Spec(s)**: SPEC-161 – `supekku/scripts/lib/workflow`; SPEC-116 – `supekku/scripts/lib/core/frontmatter_metadata`
- **Implementation Plan**: [IP-104](./IP-104.md)
- **Change Drivers**: Investigation of phase status data quality across 206 existing phases

## 2. Motivation

Phase statuses are unreliable. Investigation across 206 phase sheets found:

| Frontmatter value | Count | Notes |
|---|---|---|
| `draft` | 69 | Many on completed deltas — never transitioned |
| `completed` | 66 | One "correct" terminal value |
| `complete` | 54 | Another variant — no canonical form enforced |
| `done` | 4 | Third variant |
| `in-progress` | 4 | |
| `active` | 2 | |
| `in_progress` | 1 | Underscore variant |

**40 completed deltas** have at least one phase still showing `status: draft`.

Three compounding causes:

1. **No `phase.status` in `ENUM_REGISTRY`** — `validate_status_for_entity("phase", ...)` silently accepts anything. DE-075 added enums for every other artifact type but missed phases.
2. **`phase complete` / `phase start` only write to `workflow/state.yaml`** — the transient control-plane file introduced by DE-103. They never touch the phase sheet's frontmatter, which is what everything reads.
3. **No skill guidance specifying canonical values** — agents improvise freely.

## 3. Scope & Objectives

- **Primary Outcomes**:
  1. Register `phase.status` in `ENUM_REGISTRY` using `changes/lifecycle.py` vocabulary (`draft`, `in-progress`, `completed`, `deferred`)
  2. `phase start` transitions phase frontmatter `draft` → `in-progress`
  3. `phase complete` transitions phase frontmatter to `completed`
  4. Backfill all 206 existing phases to canonical values
  5. Validate phase status on creation and edit paths
  6. Update skill guidance (`/execute-phase`, `/update-delta-docs`) with canonical values
- **Operational Constraints**: Must not break existing completed deltas; backfill is normalisation only
- **Dependencies**: None — builds on existing `changes/lifecycle.py` and `ENUM_REGISTRY` infrastructure

## 4. Out of Scope

- Gating `complete delta` on phase completion status (future improvement)
- Reconciliation between `workflow/state.yaml` phase status and frontmatter (state.yaml is the control plane; frontmatter is the record of truth — they serve different purposes, but frontmatter must be updated)
- Changes to `workflow/state.yaml` schema or semantics

## 5. Approach Overview

- **System Touchpoints**: `supekku/scripts/lib/core/enums.py`, `supekku/cli/workflow.py`, `supekku/scripts/lib/changes/creation.py`, `supekku/scripts/lib/core/frontmatter_writer.py`, skill files
- **Key Changes**:
  1. Add `phase.status` to `ENUM_REGISTRY` — reuse `changes/lifecycle.py` constants (consistent with delta/revision/audit)
  2. In `phase_start()` — after writing `state.yaml`, also update phase sheet frontmatter `status: draft` → `status: in-progress` via `update_frontmatter_status()`
  3. In `phase_complete_command()` — after writing `state.yaml`, also update phase sheet frontmatter to `status: completed` via `update_frontmatter_status()`
  4. Backfill script — normalise existing values using `CANONICAL_STATUS_MAP` from `changes/lifecycle.py`, plus handle `done` → `completed`, `active` → `in-progress`, `in_progress` → `in-progress`
  5. Validate phase status in `create_phase()` and any `edit phase --status` path
  6. Patch skill files with canonical vocabulary and "use CLI, don't hand-edit" guidance
- **Migration / Rollout Notes**: Backfill is a one-time data normalisation. Extended `CANONICAL_STATUS_MAP` handles all observed variants.

## 6. Verification Strategy

- **Requirements Coverage**: Enum registration tested via `enums_test.py`; frontmatter sync tested via `workflow_phase_complete_test.py` and new `workflow_phase_start_test.py` assertions; backfill verified by post-run audit of all phase files
- **Acceptance Criteria**:
  - `uv run spec-driver schema show phase.status` returns canonical values
  - After `phase start DE-xxx`, phase sheet frontmatter says `status: in-progress`
  - After `phase complete DE-xxx`, phase sheet frontmatter says `status: completed`
  - Zero non-canonical phase statuses in the repo after backfill
  - `just check` passes

## 7. Risks & Mitigations

- **Risk**: Backfill touches 206 files in a single commit – _Likelihood_: low – _Impact_: low – _Mitigation_: Normalisation only; values map deterministically; easy to review diff
- **Risk**: `update_frontmatter_status` may not handle all phase file formats – _Likelihood_: low – _Impact_: medium – _Mitigation_: Test against representative sample; the function is already used for deltas/revisions
