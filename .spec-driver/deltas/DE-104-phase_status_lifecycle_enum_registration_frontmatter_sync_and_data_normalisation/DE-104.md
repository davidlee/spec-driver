## 2. Motivation

Phase statuses are unreliable across all surfaces: CLI, validation, skills, memory, and schema.

### Data quality

| Frontmatter value | Count | Notes |
|---|---|---|
| `draft` | 69 | Many on completed deltas â€” never transitioned |

**40 completed deltas** have at least one phase still showing `status: draft`.
**12 phases** are missing `phase.overview` blocks (6 historical/pre-tooling, 6 from agents hand-crafting instead of using CLI).

### Root causes (compounding)

1. **No `phase.status` in `ENUM_REGISTRY`** â€” `validate_status_for_entity("phase", ...)` silently accepts anything. DE-075 added enums for every other artifact type but missed phases.
2. **`phase start`/`phase complete` only write to `workflow/state.yaml`** â€” the transient control-plane file introduced by DE-103. They never touch the phase sheet's frontmatter, which is what everything reads.
3. **`phase complete` writes `"complete"` (non-canonical) to `state.yaml`** and passes `phase_status="complete"` to `build_handoff()`, leaking into handoff payloads.
4. **`validate` has no phase awareness** â€” no status checks, no structural checks, no fix capability.
5. **Skills don't specify canonical phase status values** or instruct agents to use CLI commands for phase lifecycle transitions.
6. **Memory records document incorrect behaviour** â€” `mem.reference.workflow-commands` says "phase status â†’ complete".
7. **Schema example misleads** â€” `frontmatter.phase` example uses `"status": "active"`.
8. **Agents hand-craft phase sheets** despite `/plan-phases` instructing `create phase` CLI usage â€” no validation catches the result.

## 3. Scope & Objectives

### Primary outcomes

1. Register `phase.status` in `ENUM_REGISTRY` using `changes/lifecycle.py` vocabulary
2. Extend `CANONICAL_STATUS_MAP` with all observed variants (`done`, `active`, `in_progress`)
3. `phase start` transitions phase frontmatter `draft` â†’ `in-progress` (frontmatter written before `state.yaml`)
4. `phase complete` transitions phase frontmatter to `completed`; fix `state.yaml` value to `"completed"` and handoff payload; fix idempotency guard for both canonical and legacy values
5. `validate` checks phase frontmatter status against enum
6. `validate --fix` normalises non-canonical phase status values
7. `validate` warns on missing `phase.overview` blocks and non-canonical phase ID formats
8. Fix `frontmatter.phase` schema example to use canonical values
9. Update skills: `/execute-phase`, `/update-delta-docs`, `/continuation`, `/close-change`, `/plan-phases`
10. Update memory records: `mem.reference.workflow-commands`, `mem.fact.spec-driver.status-enums`
11. Backfill all 206 existing phases to canonical status values

### Operational constraints

- Must not break existing completed deltas; backfill is normalisation only
- Must not alter `workflow/state.yaml` semantics beyond fixing the non-canonical `"complete"` value
- `validate --fix` only repairs known-safe normalisations (status values); structural issues (missing blocks) are warnings only

### Dependencies

None â€” builds on existing `changes/lifecycle.py` and `ENUM_REGISTRY` infrastructure
