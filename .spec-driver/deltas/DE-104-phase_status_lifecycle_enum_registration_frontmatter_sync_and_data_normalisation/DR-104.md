# DR-104 – Phase status lifecycle: enum registration, frontmatter sync, and data normalisation

## 1. Executive Summary

- **Delta**: [DE-104](./DE-104.md)
- **Status**: draft
- **Last Updated**: 2026-03-21
- **Synopsis**: Wire phase sheets into the existing status enum and frontmatter update infrastructure so phase lifecycle state is validated, canonical, and updated by CLI commands.

## 2. Problem & Constraints

- **Current Behaviour**: Phase sheet frontmatter `status` is set to `draft` on creation and never programmatically updated. `phase start` and `phase complete` only write to `workflow/state.yaml`. No enum validates phase status values. Result: 6 variant spellings across 206 phases; 69 phases stuck at `draft` on completed deltas.
- **Drivers / Inputs**: Data quality investigation; DE-075 (enum coverage gap); DE-103 (introduced dual-store divergence)
- **Constraints / Guardrails**: Must not alter `workflow/state.yaml` semantics. Backfill must be safe for completed deltas. Must use existing infrastructure (`update_frontmatter_status`, `ENUM_REGISTRY`, `CANONICAL_STATUS_MAP`).
- **Out of Scope**: Gating `complete delta` on phase status. Reconciliation daemon between state.yaml and frontmatter.

## 3. Architecture Intent

- **Target Outcomes**:
  1. `phase.status` registered in `ENUM_REGISTRY` with values: `draft`, `pending`, `in-progress`, `completed`, `deferred`
  2. `phase start` updates phase frontmatter `draft` → `in-progress`
  3. `phase complete` updates phase frontmatter → `completed`
  4. All 206 existing phases normalised to canonical values
  5. Skills reference canonical values and instruct agents to use CLI commands
- **Guiding Principles**: Phases are change artifacts — they share the same lifecycle vocabulary as deltas, revisions, and audits. One source of truth per field (frontmatter for persistent status).
- **State Transitions**: `draft` → `in-progress` (via `phase start`) → `completed` (via `phase complete`). `deferred` available for paused phases.

## 4. Code Impact Summary

| Path | Current State | Target State |
|---|---|---|
| `supekku/scripts/lib/core/enums.py` | No `phase.status` entry | `phase.status` registered using `_change_statuses()` |
| `supekku/cli/workflow.py` `phase_start()` | Writes `state.yaml` only | Also calls `update_frontmatter_status()` on phase sheet |
| `supekku/cli/workflow.py` `phase_complete_command()` | Writes `state.yaml` only | Also calls `update_frontmatter_status()` on phase sheet |
| `supekku/scripts/lib/changes/lifecycle.py` | `CANONICAL_STATUS_MAP` covers `complete` → `completed` | Extended with `done` → `completed`, `active` → `in-progress`, `in_progress` → `in-progress` |
| `supekku/scripts/lib/changes/creation.py` `create_phase()` | No status validation | Validate via `validate_status_for_entity("phase", "draft")` |
| `.agents/skills/execute-phase/SKILL.md` | No canonical phase status guidance | Lists canonical values; instructs use of `phase start`/`phase complete` |
| `.agents/skills/update-delta-docs/SKILL.md` | Mentions "status" without values | Lists canonical values |

## 6. Supporting Context

- **Related Deltas**: DE-075 (enum coverage), DE-103 (workflow control plane), DE-004 (phase management)
- **Related Specs**: SPEC-161 (workflow), SPEC-116 (frontmatter metadata)
- **Existing Infrastructure**: `update_frontmatter_status()` already handles delta/revision/audit frontmatter updates. `CANONICAL_STATUS_MAP` already normalises `complete` → `completed`.

## 7. Design Decisions & Trade-offs

- **DEC-104-01**: Reuse `changes/lifecycle.py` for phase statuses. Phases are change artifacts — they share the same lifecycle as deltas/revisions/audits. No need for a separate `PhaseStatus` enum. The existing `_change_statuses()` helper (which excludes the legacy `complete` alias) provides the right set.
- **DEC-104-02**: Frontmatter is record of truth; `workflow/state.yaml` is transient control plane. Both stores get updated by CLI commands, but frontmatter is what list/show/query reads. If they diverge, frontmatter wins.
- **DEC-104-03**: Backfill via extended `CANONICAL_STATUS_MAP`. All 6 observed variants map deterministically. The backfill script iterates all phase files and rewrites frontmatter status using `normalize_status()`.

## 8. Open Questions

None — uses established patterns.

## 9. Rollout & Operational Notes

- **Migration / Backfill**: One-time script normalises all 206 phase files. Commit as a single atomic change for reviewability.
- **Recovery / Rollback**: Git revert of the backfill commit. No runtime state affected.
