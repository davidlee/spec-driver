# DR-104 – Phase status lifecycle: enum registration, frontmatter sync, and data normalisation

## 1. Executive Summary

- **Delta**: [DE-104](./DE-104.md)
- **Status**: draft
- **Last Updated**: 2026-03-21
- **Synopsis**: Wire phase sheets into existing status enum and frontmatter update infrastructure. Add phase validation to `validate` with `--fix` for normalisation. Fix CLI commands, skills, memory, and schema to produce and enforce canonical phase status values.

## 2. Problem & Constraints

- **Current Behaviour**: Phase frontmatter `status` is set to `draft` on creation and never programmatically updated. `phase start`/`phase complete` only write to `workflow/state.yaml`. No enum validates values. `validate` is unaware of phases. Skills give no canonical vocabulary. The schema example is misleading. Result: 6 variant spellings across 206 phases; 69 stuck at `draft` on completed deltas; 12 missing overview blocks from agents hand-crafting.
- **Drivers**: DE-075 (enum gap), DE-103 (dual-store divergence), data quality investigation
- **Constraints**: Must not alter `state.yaml` semantics beyond value normalisation. `--fix` limited to safe normalisations. Must use existing infrastructure per POL-001. Status values from constants per POL-002.
- **Out of Scope**: PhaseRegistry (IMPR-021), `edit phase`, `show delta` phase status display, `complete delta` gating on phases.

## 3. Architecture Intent

- **Target Outcomes**:
  1. `phase.status` registered in `ENUM_REGISTRY`: `draft`, `pending`, `in-progress`, `completed`, `deferred`
  2. CLI commands update both `state.yaml` and frontmatter with canonical values
  3. `validate` detects non-canonical phase statuses; `--fix` repairs them
  4. `validate` warns on structural issues (missing overview blocks, non-canonical IDs)
  5. Skills instruct agents to use CLI commands, not hand-edit
  6. Memory records document correct behaviour
- **Guiding Principles**: Phases are change artifacts — same lifecycle vocabulary as deltas/revisions/audits (DEC-104-01). Frontmatter is the normative record (ADR-008); state.yaml is transient control plane (DEC-104-02).
- **State Transitions**: `draft` → `in-progress` (via `phase start`) → `completed` (via `phase complete`). `deferred` available for paused phases.

## 4. Code Impact Summary

### 4.1 Enum & lifecycle

| Path | Current | Target |
|---|---|---|
| `supekku/scripts/lib/core/enums.py` | No `phase.status` entry | `"phase.status": _change_statuses` |
| `supekku/scripts/lib/changes/lifecycle.py` | `CANONICAL_STATUS_MAP` covers `complete` → `completed` | Add `done` → `completed`, `active` → `in-progress`, `in_progress` → `in-progress` |

`_change_statuses()` already filters out the legacy `complete` alias and returns `[completed, deferred, draft, in-progress, pending]`. Reusing it for phases means phases and deltas share identical valid values — correct per DEC-104-01.

### 4.2 CLI workflow commands

| Path | Current | Target |
|---|---|---|
| `workflow.py` `phase_start()` | Writes `state.yaml` only | Also calls `update_frontmatter_status(phase_path, "in-progress")` using already-discovered `phase_path` |
| `workflow.py` `phase_complete_command()` | Writes `"complete"` to `state.yaml`; passes `phase_status="complete"` to `build_handoff()` | Writes `"completed"` to `state.yaml`; passes `phase_status="completed"` to `build_handoff()`; calls `update_frontmatter_status(phase_path, "completed")` |

Phase path resolution: `phase_start()` already discovers the phase path and stores it in `state.yaml`. `phase_complete_command()` reads it back from `state_data["phase"]["path"]`. Both have the absolute path available — the frontmatter update uses `repo_root / phase_path`.

Use `STATUS_COMPLETED` and `STATUS_IN_PROGRESS` constants from `changes/lifecycle.py` instead of string literals (POL-002).

### 4.3 Validation

| Path | Current | Target |
|---|---|---|
| `validator.py` | No phase awareness | New `_validate_phase_statuses()` method |
| `workspace.py` (CLI) | No `--fix` flag | Add `--fix` flag, wire through to validator |

**Phase discovery for validation** (DEC-104-04): Walk `deltas/*/phases/phase-*.md`, load frontmatter, check status. No PhaseRegistry needed.

```python
def _validate_phase_statuses(self, *, fix: bool = False) -> None:
    """Validate phase frontmatter statuses across all delta bundles."""
    deltas_dir = get_deltas_dir(self.workspace.root)
    if not deltas_dir.exists():
        return
    valid = get_enum_values("phase.status")
    for delta_dir in sorted(deltas_dir.iterdir()):
        phases_dir = delta_dir / "phases"
        if not phases_dir.is_dir():
            continue
        for phase_file in sorted(phases_dir.glob("phase-*.md")):
            fm, _ = load_markdown_file(phase_file)
            status = fm.get("status", "")
            if status and status not in valid:
                canonical = normalize_status(status)
                if fix and canonical in valid:
                    update_frontmatter_status(phase_file, canonical)
                    self._info(f"Fixed phase status: {status} → {canonical}", str(phase_file))
                else:
                    self._warn(f"Non-canonical phase status: '{status}'", str(phase_file))
            # Structural checks (warnings only)
            if fm.get("kind") != "phase":
                self._warn("Missing or incorrect kind (expected 'phase')", str(phase_file))
            content = phase_file.read_text(encoding="utf-8")
            if "supekku:phase.overview" not in content:
                self._warn("Missing phase.overview block", str(phase_file))
```

**`--fix` semantics** (DEC-104-05): Only normalises status values where `normalize_status()` produces a valid canonical value. Structural issues (missing blocks, wrong kind) are warnings only — they require manual or tooling intervention to fix properly.

## 6. Design Decisions

- **DEC-104-01**: Reuse `changes/lifecycle.py` for phase statuses. Phases are change artifacts — same vocabulary as deltas/revisions/audits. `_change_statuses()` provides the canonical set. POL-001 compliance.
- **DEC-104-02**: Frontmatter is record of truth (ADR-008). `state.yaml` is transient control plane. CLI commands update both stores; if they diverge, frontmatter wins.
- **DEC-104-03**: Extend `CANONICAL_STATUS_MAP` with all observed variants (`done` → `completed`, `active` → `in-progress`, `in_progress` → `in-progress`). The map is a normalisation layer, not a whitelist — adding mappings makes `normalize_status()` robust against recurrence. POL-001 compliance.
- **DEC-104-04**: Walk delta bundles for validation instead of building a PhaseRegistry. ~15 lines of discovery code. PhaseRegistry backlogged as IMPR-021 for when formal phase access is needed.
- **DEC-104-05**: `validate --fix` scoped to status normalisation only. Structural issues (missing overview blocks, wrong kind, non-canonical IDs) are warnings — they need manual intervention or tooling to fix properly. This avoids `--fix` becoming a catch-all mutation flag.

## 7. Rollout & Operational Notes

- **Migration**: Backfill all 206 phases via `validate --fix` after code changes land. Single atomic commit for reviewability.
- **Sequencing**: Enum + lifecycle map first → CLI commands → validation → skills/memory → backfill.
- **Recovery**: Git revert of backfill commit. No runtime state affected.
