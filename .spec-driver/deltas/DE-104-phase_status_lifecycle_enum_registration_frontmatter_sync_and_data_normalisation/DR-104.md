# DR-104 – Phase status lifecycle: enum registration, frontmatter sync, and data normalisation

## 1. Executive Summary

- **Delta**: [DE-104](./DE-104.md)
- **Status**: draft
- **Last Updated**: 2026-03-21
- **Synopsis**: Wire phase sheets into existing status enum and frontmatter update infrastructure. Add phase validation to `validate` with `--fix` for normalisation. Fix CLI commands, skills, memory, and schema to produce and enforce canonical phase status values.

## 2. Problem & Constraints

- **Current Behaviour**: Phase frontmatter `status` is set to `draft` on creation and never programmatically updated. `phase start`/`phase complete` only write to `workflow/state.yaml`. No enum validates values. `validate` is unaware of phases. Skills give no canonical vocabulary. The schema example uses `"active"`. Result: 6 variant spellings across 206 phases; 69 stuck at `draft` on completed deltas; 12 missing overview blocks from agents hand-crafting; 10 missing status fields entirely.
- **Drivers**: DE-075 (enum gap), DE-103 (dual-store divergence), data quality investigation
- **Constraints**: Must not alter `state.yaml` semantics beyond value normalisation. `--fix` limited to safe normalisations. Must use existing infrastructure per POL-001. Status values from constants per POL-002. Lint compliance per STD-002.
- **Out of Scope**: PhaseRegistry (IMPR-021), `edit phase`, `show delta` phase status display, `complete delta` gating on phases, auto-generating missing overview blocks, renaming non-canonical phase IDs.

## 3. Architecture Intent

- **Target Outcomes**:
  1. `phase.status` registered in `ENUM_REGISTRY`: `draft`, `pending`, `in-progress`, `completed`, `deferred`
  2. CLI commands update both `state.yaml` and frontmatter with canonical values
  3. `validate` detects non-canonical phase statuses; `--fix` repairs them
  4. `validate` warns on structural issues (missing overview blocks, missing status, non-canonical IDs)
  5. Skills instruct agents to use CLI commands, not hand-edit
  6. Memory records document correct behaviour
- **Guiding Principles**: Phases are change artifacts — same lifecycle vocabulary as deltas/revisions/audits (DEC-104-01). Frontmatter is the normative record per ADR-008; state.yaml is transient control plane (DEC-104-02).
- **State Transitions**: `draft` → `in-progress` (via `phase start`) → `completed` (via `phase complete`). `deferred` available for paused phases.

## 4. Code Impact Summary

### 4.1 Enum & lifecycle

| Path                                       | Current                                                | Target                                                                            |
| ------------------------------------------ | ------------------------------------------------------ | --------------------------------------------------------------------------------- |
| `supekku/scripts/lib/core/enums.py`        | No `phase.status` entry                                | `"phase.status": _change_statuses`                                                |
| `supekku/scripts/lib/changes/lifecycle.py` | `CANONICAL_STATUS_MAP` covers `complete` → `completed` | Add `done` → `completed`, `active` → `in-progress`, `in_progress` → `in-progress` |

`_change_statuses()` already filters out the legacy `complete` alias and returns `[completed, deferred, draft, in-progress, pending]`. Reusing it for phases means phases and deltas share identical valid values — correct per DEC-104-01.

### 4.2 CLI workflow commands

| Path                                     | Current                                                                                    | Target                                                                                                                                   |
| ---------------------------------------- | ------------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------- |
| `workflow.py` `phase_start()`            | Writes `state.yaml` only                                                                   | Updates frontmatter first, then writes `state.yaml`                                                                                      |
| `workflow.py` `phase_complete_command()` | Writes `"complete"` to `state.yaml`; passes `phase_status="complete"` to `build_handoff()` | Updates frontmatter first, then writes `state.yaml` with `STATUS_COMPLETED`; passes `phase_status=STATUS_COMPLETED` to `build_handoff()` |

**Write ordering** (DEC-104-08): Frontmatter (normative record per ADR-008) is written before `state.yaml` (transient control plane). If the process crashes between writes, frontmatter is correct and `state.yaml` is stale — the safe failure mode, because `state.yaml` is transient and `phase start` can re-initialise it. The reverse ordering would leave `state.yaml` saying "completed" while frontmatter still says "draft", causing the next agent to re-execute the phase.

**`phase_start()` implementation sketch** (R3):

```python
# After _find_plan_and_phase() discovers phase_path:
# 1. Update frontmatter (normative) first
if phase_path:
    abs_phase = repo_root / phase_path
    if abs_phase.exists():
        update_frontmatter_status(abs_phase, STATUS_IN_PROGRESS)

# 2. Then write state.yaml (transient)
written = write_state(delta_dir, data)
```

`phase_start()` has `phase_path` available from `_find_plan_and_phase()`. The frontmatter update is inserted before the existing `write_state()` call and the `raise typer.Exit(EXIT_SUCCESS)`.

**`phase_complete_command()` implementation sketch**:

```python
# 1. Update frontmatter (normative) first
phase_path = phase.get("path")
if phase_path:
    abs_phase = repo_root / phase_path
    if abs_phase.exists():
        update_frontmatter_status(abs_phase, STATUS_COMPLETED)

# 2. Then update state.yaml (transient)
if not already_complete:
    state_data["phase"]["status"] = STATUS_COMPLETED
write_state(delta_dir, state_data)
```

**Idempotency guard fix** (R1): The existing guard at line 1248:

```python
already_complete = phase.get("status") == "complete"  # BUG: stale value
```

Must handle both the new canonical value and in-flight `state.yaml` files that still contain the old `"complete"`:

```python
already_complete = phase.get("status") in (STATUS_COMPLETED, STATUS_COMPLETE)
```

Without this fix, a second `phase complete` on the same delta would not match the guard (`"completed" != "complete"`), re-write `state.yaml`, and re-emit a handoff — a correctness bug.

**Edge case**: `update_frontmatter_status()` returns `False` if no `status` field exists in frontmatter. For the 10 phase files without frontmatter status (historical artifacts), this is a silent no-op — acceptable. The agent is informed by the CLI output that the phase was marked complete; the historical file is left as-is. `validate` will warn about these.

Uses `STATUS_COMPLETED`, `STATUS_IN_PROGRESS`, and `STATUS_COMPLETE` constants from `changes/lifecycle.py` per POL-002.

### 4.3 Validation

| Path                 | Current            | Target                                  |
| -------------------- | ------------------ | --------------------------------------- |
| `validator.py`       | No phase awareness | New `_validate_phase_statuses()` method |
| `workspace.py` (CLI) | No `--fix` flag    | Add `--fix` flag, wire through          |

**Phase discovery** (DEC-104-04, DEC-104-06): Walk `deltas/*/phases/phase-*.md`, matching only the canonical `phase-\d{2}.md` filename pattern. This excludes DE-004's design docs (`phase-05-plan.md` etc.) that were parked in phases directories.

**`--fix` design** (DEC-104-05):

This is the first mutation capability in `validate`. The design is deliberately narrow:

- Thread `fix: bool = False` through `validate_workspace()` → `WorkspaceValidator.__init__()` → `_validate_phase_statuses()`
- When `fix=True` and a phase status is non-canonical but `normalize_status()` produces a valid canonical value: call `update_frontmatter_status()`, emit `info`-level issue with "Fixed: {old} → {new}" message
- When `fix=True` but the value isn't normalisable, or the file lacks a status field: emit `warning` as normal
- `--fix` is orthogonal to `--sync` (sync refreshes registries; fix repairs findings)
- No `--dry-run` needed — run `validate` without `--fix` to preview, then `validate --fix` to repair
- Idempotent: subsequent runs after fix touch zero files

**Structural checks** (warnings only, never fixed):

| Check                          | Condition                                         | Severity                                   |
| ------------------------------ | ------------------------------------------------- | ------------------------------------------ |
| Non-canonical status           | `status` not in `get_enum_values("phase.status")` | warning (error if `--fix` can't normalise) |
| Missing status field           | No `status` key in frontmatter                    | warning                                    |
| Missing `phase.overview` block | `supekku:phase.overview` not in content           | warning                                    |
| Missing `kind: phase`          | `kind` absent or not `"phase"`                    | warning                                    |

```python
def _validate_phase_statuses(self, *, fix: bool = False) -> None:
    """Validate phase frontmatter statuses across all delta bundles."""
    deltas_dir = get_deltas_dir(self.workspace.root)
    if not deltas_dir.exists():
        return
    valid = get_enum_values("phase.status")
    for delta_dir in sorted(deltas_dir.iterdir()):
        if not delta_dir.is_dir():
            continue
        phases_dir = delta_dir / "phases"
        if not phases_dir.is_dir():
            continue
        for phase_file in sorted(phases_dir.glob("phase-[0-9][0-9].md")):
            self._validate_single_phase(phase_file, valid, fix=fix)

def _validate_single_phase(
    self,
    phase_file: Path,
    valid_statuses: list[str],
    *,
    fix: bool = False,
) -> None:
    """Validate a single phase file."""
    try:
        fm, _ = load_markdown_file(phase_file)
    except Exception:
        self._warning(str(phase_file), "Could not parse frontmatter")
        return
    artifact = fm.get("id", str(phase_file.name))
    status = fm.get("status")
    if status is None:
        self._warning(artifact, "Missing status field in frontmatter")
    elif status not in valid_statuses:
        canonical = normalize_status(status)
        if fix and canonical in valid_statuses:
            update_frontmatter_status(phase_file, canonical)
            self._info(artifact, f"Fixed phase status: '{status}' → '{canonical}'")
        else:
            self._warning(artifact, f"Non-canonical phase status: '{status}'")
    if fm.get("kind") != "phase":
        self._warning(artifact, "Missing or incorrect kind (expected 'phase')")
    content = phase_file.read_text(encoding="utf-8")
    if "supekku:phase.overview" not in content:
        self._warning(artifact, "Missing phase.overview block")
```

## 6. Design Decisions

- **DEC-104-01**: Reuse `changes/lifecycle.py` for phase statuses. Phases are change artifacts — same vocabulary as deltas/revisions/audits. `_change_statuses()` provides the canonical set. `pending` is arguably unused for phases in practice, but alignment with sibling types means one less thing to maintain, and it causes no harm. POL-001.
- **DEC-104-02**: Frontmatter is record of truth per ADR-008. `state.yaml` is transient control plane. CLI commands update both; if they diverge, frontmatter wins.
- **DEC-104-03**: Extend `CANONICAL_STATUS_MAP` with all observed variants (`done` → `completed`, `active` → `in-progress`, `in_progress` → `in-progress`). The map is a normalisation layer — adding mappings makes `normalize_status()` robust against recurrence. POL-001.
- **DEC-104-04**: Walk delta bundles for validation instead of PhaseRegistry. ~30 lines of discovery + validation code. PhaseRegistry backlogged as IMPR-021.
- **DEC-104-05**: `validate --fix` scoped to status normalisation only. First mutation capability in `validate` — deliberately narrow. Structural issues (missing overview blocks, missing status field, wrong kind) are warnings only. No `--dry-run` needed — run without `--fix` to preview. Idempotent. If future `--fix` capabilities are added, the pattern may need refactoring into a strategy object — but a single `fix: bool` is proportionate for now.
- **DEC-104-06**: Phase discovery matches `phase-[0-9][0-9].md` only. Excludes DE-004's design docs (`phase-05-plan.md` etc.) and other non-standard files parked in phases directories. The canonical `create_phase()` always produces this pattern.
- **DEC-104-07**: Schema stays permissive (`type: string`); enum enforcement is CLI-only via `validate_status_for_entity()`, consistent with all sibling artifact types. JSON schema constraint deferred to a cross-cutting schema tightening effort if ever warranted.
- **DEC-104-08**: Write ordering: frontmatter (normative per ADR-008) before `state.yaml` (transient). If the process crashes between writes, frontmatter is correct and `state.yaml` is stale — the safe failure mode, because `state.yaml` is transient and `phase start` can re-initialise it.

## 7. Rollout & Operational Notes

- **Sequencing**: Enum + lifecycle map → CLI commands → validation + `--fix` → schema example → skills → memory → backfill via `validate --fix`
- **Backfill**: Run `spec-driver validate --fix` after all code changes land. Expected to normalise ~120 files (those with non-canonical status values). Commit as single atomic change. Idempotent — subsequent runs touch zero files.
- **Recovery**: Git revert of backfill commit. No runtime state affected.
- **10 files without status field**: Left as-is by `--fix` (can't infer correct value). `validate` warns about them. All are historical (DE-002, DE-004, DE-031) — pre-tooling artifacts.
