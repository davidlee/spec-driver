## 1. Summary

- **Delta**: DE-104 — Phase status lifecycle: enum registration, frontmatter sync, and data normalisation
- **Specs Impacted**: SPEC-161, SPEC-116
- **Desired Outcome**: Phase statuses are validated, canonical, and updated by CLI commands. Existing data normalised.

## 2. Context & Constraints

- **Current Behaviour**: 6 variant status spellings across 206 phases; 69 stuck at `draft` on completed deltas; CLI commands only write transient `state.yaml`; no validation.
- **Target Behaviour**: Canonical `phase.status` enum enforced; `phase start`/`phase complete` update frontmatter (normative) before `state.yaml` (transient); `validate` detects and `--fix` repairs drift.
- **Dependencies**: None — builds on existing infrastructure.
- **Constraints**: POL-001 (reuse), POL-002 (constants not literals), STD-002 (lint), ADR-008 (frontmatter is normative truth).

## 4. Phase Overview

| Phase    | Objective                                                    | Entrance           | Exit                                                                              | Sheet                |
| -------- | ------------------------------------------------------------ | ------------------ | --------------------------------------------------------------------------------- | -------------------- |
| Phase 01 | Enum, lifecycle map, CLI frontmatter sync, creation constant | DR approved        | Enum registered; CLI commands update frontmatter; tests pass; lint clean          | `phases/phase-01.md` |
| Phase 02 | Validation + --fix, schema, skills, memory, backfill         | Phase 01 committed | validate works; skills/memory updated; all phases backfilled; `just check` passes | `phases/phase-02.md` |

**Rationale for two phases**: Phase 01 is the core machinery (enum + CLI). Phase 02 is everything that consumes it (validation, guidance, data repair). Phase 02 depends on Phase 01 being committed so `validate --fix` can use the new enum.
