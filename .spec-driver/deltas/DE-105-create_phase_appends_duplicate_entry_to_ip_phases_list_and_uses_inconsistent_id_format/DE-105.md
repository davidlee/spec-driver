## 1. Summary & Context

- **Implementation Plan**: [IP-105](./IP-105.md) – phase planning and execution for the `create phase` fix
- **Change Drivers**: ISSUE-052, ADR-004, ADR-005, POL-001, POL-002

## 2. Motivation

`spec-driver create phase` currently mixes two incompatible notions of phase
identity. New plans are scaffolded with `IP-###-P##` entries in
`supekku:plan.overview@v1`, but `create_phase()` still emits
`IP-###.PHASE-##`, computes the next number from phase filenames only, and
blindly appends the newly generated ID back into the plan block.

That breaks the phase list in two ways:

- manually planned phase entries are not treated as authoritative for numbering
  or metadata lookup
- equivalent phases can be duplicated in the plan overview under different ID
  spellings

The target state is modest: `create phase` must reuse the plan's established
convention, choose the next sequence number from both plan metadata and phase
files, and avoid appending duplicate logical phases.

## 3. Scope & Objectives

- **Primary Outcomes**:
  - Make `create phase` infer numbering from existing plan overview entries and
    already-created phase files.
  - Make `create phase` preserve the existing phase ID convention for the plan
    instead of hardcoding dotted IDs.
  - Prevent duplicate logical phase entries from being appended to
    `supekku:plan.overview@v1`.
- **Operational Constraints**: Keep the fix local to phase creation and its
  tests; do not rewrite the entire dotted-ID ecosystem in one delta.
- **Dependencies**: Existing user worktree changes in `creation.py`,
  `workflow.py`, and lifecycle/status code must be preserved.

## 4. Out of Scope

- Normalizing every existing dotted phase ID in historical deltas.
- Broad workflow/display refactors outside the `create phase` path.
- Audit/closure work for `DE-105` itself.

## 5. Approach Overview

- **System Touchpoints**: `supekku/scripts/lib/changes/creation.py`,
  `supekku/scripts/lib/blocks/plan.py`, `supekku/scripts/lib/changes/creation_test.py`
- **Key Changes**:
  - Add canonical helpers in phase creation to parse phase sequence numbers and
    compare logically equivalent phase IDs.
  - Make phase creation consult plan metadata before choosing the new phase ID
    and before copying phase-specific metadata into the generated sheet.
  - Keep plan mutation idempotent by refusing to append an equivalent phase more
    than once.
- **Migration / Rollout Notes**: No migration. Existing plans keep their current
  convention; newly created phases follow the convention already present in the
  owning plan.

## 6. Verification Strategy

- **Planned Artefacts**: targeted unit coverage in `creation_test.py`
- **Acceptance Criteria**:
  - creating a phase against a plan with `IP-###-P##` entries yields the next
    `-P##` identifier
  - creating a phase against a plan with dotted entries preserves the dotted
    convention
  - equivalent phase IDs are not duplicated in the plan overview
  - metadata lookup still works when the plan uses the established convention

## 7. Risks & Mitigations

- **Risk**: phase-ID normalization helpers accidentally drift from other code
  paths – _Likelihood_: medium – _Impact_: medium – _Mitigation_: keep helpers
  narrowly scoped and reuse them across numbering, lookup, and dedupe inside
  the creation path.
- **Risk**: dirty worktree changes overlap with this surface –
  _Likelihood_: medium – _Impact_: medium – _Mitigation_: patch surgically and
  verify diffs before finalizing.
