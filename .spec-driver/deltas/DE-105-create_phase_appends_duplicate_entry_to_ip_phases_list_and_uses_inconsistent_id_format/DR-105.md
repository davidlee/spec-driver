# DR-105 – create phase appends duplicate entry to IP phases list and uses inconsistent ID format

## 1. Executive Summary

- **Delta**: [DE-105](./DE-105.md)
- **Status**: in-progress
- **Last Updated**: 2026-03-21
- **Synopsis**: make `create phase` derive the next phase from the owning
  plan's existing sequence and ID spelling, then update the plan idempotently

## 2. Problem & Constraints

- **Current Behaviour**: `create_phase()` scans `phases/phase-NN.md` files to
  pick the next number, always generates `IP-###.PHASE-##`, then appends that
  exact string to the plan overview. Plans scaffolded or edited with
  `IP-###-P##` therefore lose metadata alignment and can accumulate equivalent
  duplicate entries.
- **Drivers / Inputs**: ISSUE-052, `render_plan_overview_block()` in
  `supekku/scripts/lib/blocks/plan.py`, ADR-004, ADR-005, POL-001, POL-002
- **Constraints / Guardrails**: preserve existing user edits in adjacent files;
  keep the change local to the creation flow; prefer helper reuse over
  scattered regexes
- **Out of Scope**: wholesale normalization of every consumer that still
  displays or expects dotted phase IDs

## 3. Architecture Intent

- **Target Outcomes**:
  - phase numbering reflects both authored plan metadata and generated phase
    files
  - plan metadata remains the source of truth for per-phase objective and
    criteria
  - plan updates are idempotent across equivalent string spellings
- **Guiding Principles**:
  - preserve the owning plan's convention
  - normalize only enough to compare sequence identity safely
  - keep helpers in the phase-creation module to avoid duplicating regex logic
- **State Transitions / Lifecycle Impact**: no lifecycle model changes

## 4. Code Impact Summary

| Path | Current State | Target State |
| ---- | ------------- | ------------ |
| `supekku/scripts/lib/changes/creation.py` | Numbering comes from filenames only; IDs are hardcoded dotted; plan updates always append raw IDs | Numbering merges plan/file evidence; IDs follow existing plan convention; plan updates skip equivalent duplicates |
| `supekku/scripts/lib/changes/creation_test.py` | Tests assert dotted-only output and placeholder duplication | Tests cover convention reuse, metadata preservation, and dedupe semantics |

## 6. Supporting Context

- `render_plan_overview_block()` already emits `IP-###-P01`
- historical phase files in existing deltas still use dotted IDs
- no retrieved memory defines a stronger canonical phase-ID spelling for this subsystem

## 7. Design Decisions & Trade-offs

- `DEC-105-01` – preserve the plan's established phase-ID convention.
  Rationale: compatibility is lower risk than forcing a migration across mixed
  historical artifacts.
- `DEC-105-02` – compare phase entries by normalized `(plan_id, sequence)`
  rather than raw string equality. Rationale: duplicate prevention and metadata
  lookup must survive equivalent spellings.

None at this stage.

## 9. Rollout & Operational Notes

- **Migration / Backfill**: none; behavior changes only when `create phase` is
  invoked
- **Recovery / Rollback**: revert the localized creation helpers and tests if
  compatibility fallout appears
