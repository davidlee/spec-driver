## 1. Summary & Context

- **Product Spec(s)**: PROD-006 – Phase Management; PROD-011 – Implementation Execution Workflow
- **Technical Spec(s)**: No single current technical spec owns the phase-sheet representation; this delta will likely require a spec revision before implementation
- **Implementation Plan**: [IP-106](./IP-106.md) – scaffolded, not yet planned
- **Change Drivers**: IMPR-022 identified triple-entry bookkeeping across phase frontmatter blocks and markdown; DE-104 showed that this duplication is a direct source of lifecycle drift in real phase sheets

## 2. Motivation

- The current phase template asks agents to keep the same content aligned across three semantic layers: `supekku:phase.overview`, `supekku:phase.tracking`, and markdown body sections. In practice, agents update the body and leave one or both YAML blocks stale.
- The codebase currently uses only a narrow subset of that structured data: `create_phase()` seeds both blocks, `show delta` and `list_changes` read objective/summary fields, and `validate` warns when `phase.overview` is missing. No current runtime uses structured tracking data for enforcement or reporting.
- This means the project pays a permanent authoring and maintenance cost for machine-readable structure that the tooling does not meaningfully exploit. DE-104 can improve agent guidance, but it does not remove the structural incentive to drift.
- The target state is a simpler phase authoring model that preserves useful machine-readable phase data in one canonical structured location while eliminating redundant copies of the same semantics across multiple YAML blocks and prose sections.

## 3. Scope & Objectives

- **Primary Outcomes**:
  - Decide and document the canonical source-of-truth shape for phase sheets, including which phase fields remain machine-readable and where that structured summary should live.
  - Complete the frontmatter-validation path so kind-specific frontmatter metadata is enforced in normal validation/load flows, not only described in schema metadata and tests.
  - Reduce or remove duplicated phase-sheet representations so new phases no longer require triple-entry updates for objectives, criteria, tasks, verification, and risks.
  - Preserve enough structured data for existing UX surfaces such as `show delta` and list summaries without introducing a PhaseRegistry-sized solution.
  - Record a durable doctrine decision for when artifact metadata belongs in frontmatter versus code-fenced YAML versus markdown prose.
  - Define a compatibility and migration path for existing phase sheets and validators so the change can land incrementally.
- **Operational Constraints**: Avoid breaking `show delta`, `list_changes`, phase lifecycle commands, or existing completed deltas; keep the design aligned with ADR-004 workflow posture, ADR-009 registry boundaries, and POL-001 reuse expectations.
- **Dependencies**: Requires follow-on revision/spec work because PROD-006 currently requires creation of a valid `phase.overview` block and sync-time validation of that schema.

## 4. Out of Scope

- Building a full PhaseRegistry or standalone `show/list/edit phase` command surface (IMPR-021 territory)
- Programmatic parsing of markdown checkboxes into authoritative progress state
- Delta-completion policy changes that gate on phase completion
- Reworking implementation-plan structure beyond what is needed to keep phase references coherent
- Converting every historical phase file in the same change unless migration cost proves trivial
- Repo-wide frontmatter refactors unrelated to the new canonical phase representation, except where they are required to complete metadata-driven validation as a coherent platform capability

## 5. Approach Overview

- **System Touchpoints**: `supekku/templates/phase.md`, `supekku/scripts/lib/changes/creation.py`, `supekku/scripts/lib/formatters/change_formatters.py`, `supekku/scripts/list_changes.py`, `supekku/scripts/lib/validation/validator.py`, `supekku/scripts/lib/core/frontmatter_schema.py`, `supekku/scripts/lib/core/frontmatter_metadata/*`, phase-related schema metadata, relevant skills, affected product/spec artefacts, and a new ADR if the design is approved
- **Key Changes**:
  - Recommended direction is a single canonical structured phase summary for machine-readable fields, with markdown body reserved for rich execution detail. The likely fit is enriched phase frontmatter rather than multiple YAML blocks embedded in the body.
  - Promote metadata-driven frontmatter validation from partial/documented capability to the normal enforcement path so the chosen phase summary fields are robustly validated.
  - Retain structured identity, lineage, objective, and other agreed summary fields in that canonical location; remove duplicate `phase.overview` / `phase.tracking` copies where they mirror the same content.
  - Update create/show/list/validate flows to read canonical phase metadata from the chosen single source, with compatibility fallbacks for legacy block-based phases during migration.
  - Revise product/spec language and workflow guidance so agents are no longer instructed to maintain parallel structured and prose copies of the same information.
  - Land an ADR capturing the placement heuristic for frontmatter vs code-fenced YAML vs markdown prose, and create explicit follow-up backlog/delta artefacts for any adjacent improvements that remain out of scope.
- **Migration / Rollout Notes**: Introduce compatibility first, then simplify the template and validator expectations; migrate existing phases opportunistically or with a targeted follow-up script once the new representation is stable.

## 6. Verification Strategy

- **Requirements Coverage**: Update affected PROD-006 requirements and their verification notes so they describe the new canonical phase representation rather than mandatory `phase.overview` blocks.
- **Planned Artefacts**: Regression tests for `create phase`, `show delta`, `list_changes`, and validation; compatibility tests covering both legacy block-based phases and new simplified phases; targeted verification for migration/backfill behaviour if existing phases are rewritten.
- **Acceptance Criteria**:
  - New phases can be created and used without duplicated overview/tracking/body copies of the same summary information.
  - Kind-specific frontmatter validation is enforced through the normal validation/load path for the touched artifact kinds, rather than living only in schema docs or side-channel tests.
  - `show delta` and list/reporting surfaces still expose useful phase summaries.
  - Validation reflects the new canonical representation and no longer warns on intentionally block-free new phases.
  - Design and spec artefacts explicitly record what machine-readable phase data is retained, where it lives, and what is intentionally left to markdown-only detail.
  - An ADR records the placement rule for frontmatter, code-fenced YAML, and markdown prose.
  - Any material follow-up discovered during design or implementation is captured as a backlog item or subsequent delta before DE-106 is closed.

## 7. Risks & Mitigations

- **Risk**: Simplifying the template conflicts with current PROD-006 wording and verification evidence – _Likelihood_: high – _Impact_: high – _Mitigation_: treat this as design/spec work first, with explicit revision updates before implementation
- **Risk**: `show delta` and related formatters lose enough structure to become less useful – _Likelihood_: medium – _Impact_: medium – _Mitigation_: preserve objective/lineage in frontmatter and test formatter output against real phase bundles
- **Risk**: Legacy phases and new phases diverge into two incompatible formats – _Likelihood_: medium – _Impact_: medium – _Mitigation_: make parser/validator compatibility explicit in the design and decide whether migration is mandatory or staged
- **Risk**: Expanding scope into frontmatter-validation hardening turns DE-106 into a platform refactor – _Likelihood_: medium – _Impact_: medium – _Mitigation_: keep the widened scope bounded to completing metadata-driven validation along the phase-authority path, and backlog any broader cleanup discovered
- **Risk**: Future phase-reporting work needs structured criteria/progress that this simplification removes – _Likelihood_: medium – _Impact_: medium – _Mitigation_: keep that capability explicitly deferred and evaluate against IMPR-021 rather than preserving unused duplication pre-emptively

## 8. Follow-ups & Tracking

- **Future Phases / Deltas**: Possible follow-up for bulk migration of historical phases; possible separate work if programmatic phase progress reporting becomes a real requirement; possible follow-up deltas for broader frontmatter-validation adoption beyond the artifact kinds touched here
- **Backlog Items**: IMPR-022 (source backlog item), IMPR-021 (adjacent registry/reporting direction)
- **Open Decisions / Questions**:
  - Should existing block-based phases be migrated eagerly in this delta or supported indefinitely through compatibility reads?
  - Which fields belong in the canonical structured summary: objective only, or also criteria/verification/risk/task-summary data?
  - Does this change require a dedicated spec revision bundle before code work begins, or can the delta carry the necessary spec updates directly under current town-planner posture?
  - How far should DE-106 pull the frontmatter-validation hardening beyond phases before it becomes a separate platform delta?

## 9. Implementation Notes

- Investigation references: IMPR-022, DE-004, DE-104, IMPR-021, ADR-004, ADR-009, PROD-006, PROD-011
- Current code evidence:
  - `create_phase()` and the phase template write both overview and tracking blocks
  - `change_formatters.py` and `list_changes.py` consume objective-like summary fields
  - `validator.py` currently warns on missing `phase.overview` but does not consume most duplicated body content
- DR-106 captures the refined design recommendation: keep structured phase metadata, but keep it once, and use the delta to complete the long-underinvested frontmatter-validation path that should own this class of metadata
