# DR-106 – Phase sheet template DRY: eliminate triple-entry bookkeeping across frontmatter, blocks, and markdown

## 1. Executive Summary

- **Delta**: [DE-106](./DE-106.md)
- **Status**: approved
- **Owners / Team**: spec-driver maintainers
- **Last Updated**: 2026-03-20
- **Synopsis**: Replace phase-sheet triple-entry bookkeeping with a single canonical machine-readable phase summary, harden frontmatter as the long-term home for artifact metadata, and capture the placement rule in an ADR

## 2. Problem & Constraints

- **Current Behaviour**: New phases are created from a template that emits `supekku:phase.overview` and `supekku:phase.tracking` blocks above a markdown body containing overlapping objective, criteria, verification, task, and risk data in richer prose/table form. Agents regularly update only one layer, causing drift.
- **Drivers / Inputs**: IMPR-022 documents the duplication map in detail; DE-104 demonstrates real operational fallout from phase-sheet drift; DE-004 explains why the current structure exists; ADR-004 requires deliberate delta-first design; ADR-009 argues against preserving extra structure merely to speculate about future registry/reporting needs; the current frontmatter stack already contains richer metadata definitions than the runtime validation path enforces, indicating under-finished infrastructure rather than an unsuitable model.
- **Constraints / Guardrails**:
  - Keep `show delta`, list/reporting workflows, and future machine-consumable guidance useful for existing and new phases
  - Do not break lifecycle commands or existing completed deltas
  - Reuse existing frontmatter metadata mechanisms where possible (POL-001)
  - Current PROD-006 wording explicitly requires `phase.overview`, so implementation cannot proceed responsibly without reconciling spec language
  - Prevent scope creep by capturing adjacent findings as backlog items or follow-on deltas instead of quietly widening implementation until closure becomes ambiguous
- **Out of Scope**:
  - Full PhaseRegistry/reporting work
  - Parsing markdown checkboxes into authoritative progress state
  - Any new workflow/status model beyond what frontmatter already carries

## 3. Architecture Intent

- **Target Outcomes**:
  - New phases are authored against one canonical structured summary instead of duplicated block/body mirrors
  - The normal frontmatter validation path enforces the canonical summary fields for touched artifact kinds
  - Existing UX surfaces retain phase identity, status, and objective summaries without needing a dedicated registry
  - Validation distinguishes between legacy compatibility and the new canonical format
- **Guiding Principles**:
  - Keep useful structured data, but keep it once
  - Separate compact machine-readable summary fields from rich execution detail
  - Improve the canonical metadata plane when its current weakness is an implementation gap rather than a design virtue
  - Design compatibility paths explicitly rather than relying on silent drift or hand-maintained duplication
  - Defer future progress-reporting ambitions until a real feature requires them, but do not throw away summary structure that already has clear value
- **State Transitions / Lifecycle Impact**: Phase lifecycle status remains frontmatter-driven (`draft`, `in-progress`, `completed`, `deferred`). This design does not add new state transitions; it narrows where non-status phase content is authored.

## 3a. Field Analysis — Canonical Phase Frontmatter

This section records the per-field placement decision, rationale, and the consumer analysis that supports it. It exists so reviewers can challenge any individual field placement without re-deriving the context.

### Consumer landscape

Three consumer classes inform placement:

1. **Current tooling** — `artifacts.py` (show delta), `change_formatters.py` (display), `validator.py` (warnings)
2. **Agent authoring workflow** — agents read/write phase sheets during execution; drift occurs when the same information must be maintained in multiple locations
3. **Orchestration layer** (autobahn, per IMPR-020) — will consume phase state for gate-checking, context assembly, role transition, and reviewer priming. Treats `spec-driver` artifacts as its durable control plane.

### Design principle: contract vs progress

Phase sheets serve two purposes that should not be conflated in structured data:

- **Planning contract**: what must be true for a phase to start/complete — written during planning, rarely modified
- **Execution progress**: whether those conditions are met yet — updated continuously by agents during implementation

The current `phase.tracking` block conflates these by attaching `completed: true/false` to each criterion item. This forces agents to maintain structured YAML for volatile progress state, which is exactly the duplication pattern that causes drift.

**Decision (DEC-006)**: frontmatter records the contract. Markdown body checkboxes (§3 Entrance Criteria, §4 Exit Criteria) track progress. No structured progress tracking in YAML.

### Compatibility resolution rule

When a phase file contains _both_ frontmatter fields and a legacy `phase.overview` block (e.g., a partially migrated file), the resolution rule is:

- **Frontmatter wins.** If `plan` and `delta` are present in frontmatter, skip block extraction entirely for that phase.
- If either is absent, fall back to block extraction.
- Never merge frontmatter and block data for the same phase.

This keeps the precedence model simple and avoids silent field-level conflicts between two authority surfaces.

### Criteria population in `create_phase()`

`create_phase()` calls `_extract_phase_metadata_from_plan(content, phase_id)` to source `objective`, `entrance_criteria`, and `exit_criteria` from the plan overview block's matching phase entry. This happens _before_ the plan overview is updated with the new phase ID (~line 1051 in `creation.py`).

Two cases:

1. **Plan entry predeclared with metadata**: `_extract_phase_metadata_from_plan` finds the matching entry and returns its `objective`/`entrance_criteria`/`exit_criteria`. These populate frontmatter directly.
2. **Empty plan or ID-only entry**: `_extract_phase_metadata_from_plan` returns `{}`. Frontmatter fields are omitted or emitted as empty lists. Agents populate them during planning.

The new path redirects this same data flow to frontmatter instead of block rendering — `render_phase_overview_block()` and `render_phase_tracking_block()` calls are replaced with frontmatter population. The extraction logic and its empty-plan behavior remain unchanged.

### Regex fallback fidelity

DEC-007 removes the `phase.tracking` block. Two fidelity reductions result:

1. **Task status granularity**: The regex checkbox fallback provides aggregate completion counts (`tasks_completed`/`tasks_total`), not per-task status (`in_progress`, `blocked`). The display table already converts tracking-block data to aggregate form, so the rendered output is equivalent.

2. **Criteria progress**: `_enrich_phase_data()` currently materializes `entrance_criteria` and `exit_criteria` with checkbox-form completion status from the tracking block (lines 336-348 of `change_formatters.py`). However, **no display surface consumes this enriched criteria data** — the phase table renders only phase ID, status (with task counts), and objective. The enrichment path populates the dict but nothing reads it.

Both are **accepted fidelity reductions**. The tracking block data existed as a maintenance burden without a corresponding display consumer. If a future display or orchestration surface needs structured criteria progress, it should be modeled as a deliberate feature request — not recovered by reinstating the tracking block.

### Target phase frontmatter shape

```yaml

  - "DR-106 approved"
  - "Existing phase creation and display tests passing"
exit_criteria:
  - "Frontmatter-based phase data populates show delta correctly"
  - "Legacy block-based phases still display without errors"
  - "Kind-aware frontmatter validation enforced for phase artifacts"
---
```

### What this removes from agent maintenance burden

With this design, creating a new phase requires:

- Frontmatter (auto-populated by `create_phase`) — agents review and may refine `objective` and criteria
- Markdown body sections — agents write execution detail once, in one place

Agents no longer need to maintain:

- `phase.overview` YAML block mirroring frontmatter + body content
- `phase.tracking` YAML block mirroring body checkboxes with structured completion flags
- Parallel representations of the same criteria, tasks, and verification items across three locations

## 6. Supporting Context

- **Research**: IMPR-022 – quantified duplication map and design-direction options
- **Research**: DE-104 – evidence that structural duplication contributes to phase drift in practice
- **Related Deltas / Specs**: DE-004, PROD-006, PROD-011, IMPR-021
- **Orchestration consumer analysis**: autobahn (IMPR-020) will consume spec-driver artifacts as its durable control plane for session lifecycle, gate-checking, and role transitions. Phase frontmatter becomes a direct input to orchestration decisions: `status` for lifecycle gates, `entrance_criteria`/`exit_criteria` for phase readiness/completion checks, `objective` for context assembly and reviewer priming, `plan`/`delta` for session-to-artifact binding. This analysis informed the field set in DEC-005 — fields were evaluated against orchestration value, not just current tooling.
- **Verification level analysis**: Phase-level `verification` (test commands, evidence items) is distinct from the IP-level `verification.coverage` block which provides formal VT/VA/VH → requirement traceability. Phase verification is an ad-hoc execution checklist with no requirement linkage, no status model, and no current consumer. This informed the decision to exclude it from frontmatter (DEC-005).
- **Observation from code**: Current consumers already expect structured summary data, and the repo already has richer frontmatter metadata definitions plus tests that outstrip the runtime validation path. This suggests missing follow-through rather than a fundamental parser mismatch.

## 7. Design Decisions & Trade-offs

- **DEC-001 (recommended)**: Keep machine-readable YAML, but keep it once.
  - Rationale: The problem is redundant structured surfaces, not the existence of structured data. Formatters, CLI flows, skills, and future adherence tooling all benefit from a stable machine-readable summary.
  - Consequence: The design should remove duplication without collapsing important summary metadata into prose-only parsing.
- **DEC-002 (recommended)**: Make phase frontmatter the preferred home for the canonical structured summary if a dedicated phase schema can be introduced cleanly.
  - Rationale: Frontmatter is already the repo’s normal artifact identity/lifecycle surface and avoids embedding extra authoritative schemas inside the markdown body.
  - Consequence: Tightening the normal frontmatter validation path is part of the change, not a separate excuse to keep the old body-embedded authority model.
- **DEC-003 (recommended)**: Keep rich task detail, decisions, findings, and prose guidance in markdown body; keep only summary-level machine-readable fields in the canonical structured surface.
  - Rationale: This preserves agent ergonomics and avoids trying to encode rich implementation context entirely in YAML.
  - Consequence: The exact cut line between summary structure and prose detail is defined in DEC-005.
- **DEC-004 (recommended)**: Land an ADR during DE-106 that defines the default placement rule for frontmatter, code-fenced YAML, and markdown prose.
  - Rationale: The phase cleanup will otherwise remain a local fix rather than a reusable architectural doctrine.
  - Consequence: Any related improvements discovered and deemed out of scope must be explicitly captured as backlog items or subsequent deltas before closure.
- **DEC-005 (recommended)**: Canonical phase frontmatter field set is: `plan`, `delta`, `objective`, `entrance_criteria`, `exit_criteria`.
  - Rationale: See §3a field analysis. These are the fields that (a) have current or imminent machine consumers, (b) are stable after planning time, and (c) serve gate-checking or context-assembly roles for both current tooling and the orchestration layer.
  - Fields excluded with rationale:
    - `verification` (test commands/evidence): No formal requirement linkage — that role belongs to `verification.coverage` on the IP. Phase-level verification is an ad-hoc execution checklist with no current consumer and no status model. If orchestration needs structured test-command data, it should be modeled deliberately, not inherited from a convenience list.
    - `tasks`: Highest-volatility field. Primary source of triple-entry drift. Existing regex checkbox fallback in `_enrich_phase_data()` provides task completion stats without structured YAML maintenance.
    - `risks`: No machine consumer, no gate-checking value. Delta-level risks (DE-xxx §7) are the appropriate structured surface if needed.
  - Consequence: The frontmatter schema in `frontmatter_metadata/plan.py` already defines `objective`, `entrance_criteria`, `exit_criteria`. New fields `plan` and `delta` must be added.
- **DEC-006 (recommended)**: Separate planning contract (frontmatter) from execution progress (markdown checkboxes).
  - Rationale: The `phase.tracking` block conflates "what must be true" with "is it true yet" by attaching `completed: true/false` to each criterion. This forces agents to maintain structured YAML for volatile progress state — the core authoring tax this delta exists to remove. Frontmatter records the contract; markdown body checkboxes in §3/§4 track progress.
  - Consequence: No structured progress-tracking YAML for criteria completion. If orchestration later needs machine-readable progress percentages, that becomes a deliberate feature request.
- **DEC-007 (recommended)**: Drop `phase.tracking` block entirely for new phases; no compatibility reads needed.
  - Rationale: The tracking block's only current consumer (`_enrich_phase_data()` in `change_formatters.py`) already has a regex checkbox fallback that produces equivalent task completion stats. The tracking block adds maintenance burden without unique value. Unlike `phase.overview` (which carries identity/lineage data that must be read during migration), the tracking block carries only progress data that the fallback already handles.
  - Consequence: Existing phases with tracking blocks continue to work via the existing code path (it tries tracking first, falls back to regex). No special compatibility layer needed — the fallback _is_ the compatibility layer.
- **Alternative considered but not recommended**: Keep `phase.overview` and drop only `phase.tracking`.
  - Trade-off: Less disruptive, but it still leaves duplicate criteria/tasks/verification semantics between overview and markdown body.
- **Alternative considered but not recommended**: Make blocks the sole source and generate markdown from them.
  - Trade-off: Maximally structured, but it pushes rich execution detail into awkward YAML and requires generator/rendering machinery the project does not have.
- **Alternative considered but not recommended**: Remove structured phase summaries entirely and rely on markdown parsing.
  - Trade-off: Simplifies authoring, but it weakens formatter/CLI/skill integration and pushes future automation toward brittle prose parsing.
- **DEC-008 (recommended)**: Phase frontmatter validation doubles as the DE-107 Pydantic spike.
  - Rationale: DE-107 needs a single model conversion to evaluate Pydantic feasibility. DE-106 needs kind-aware phase frontmatter validation. The phase model is an ideal spike target: small (5-6 fields beyond base), well-defined field semantics (already in `PLAN_FRONTMATTER_METADATA`), real production consumers, and Pydantic's date/list/string coercion is well-trodden. Using a real model avoids artificial prototyping and produces concrete go/no-go evidence.
  - Dependency: DE-107 is a loose dependency — its spike is executed within DE-106's implementation. DE-107 gets updated to note that its feasibility data comes from DE-106's phase model.
  - Fallback: If the Pydantic spike is negative (import time, round-trip fidelity, or compatibility issues), DE-106 falls back to inline validation in the current `validate_frontmatter()` + `BlockMetadata` system. The fallback is bounded: validate `plan` and `delta` as required strings, `objective` as optional string, `entrance_criteria`/`exit_criteria` as optional string lists.
  - Consequence: Broader kind-aware frontmatter validation for other artifact kinds is explicitly backlogged. DE-107's outcome determines the mechanism.
- **Alternative considered but not recommended**: Include `verification` in frontmatter.
  - Trade-off: Gives orchestration a structured test-command list per phase. However, phase-level verification is an execution convenience checklist, not a formal contract — it has no requirement linkage, no status model, and no current consumer. The IP-level `verification.coverage` block already provides the formal VT/VA/VH traceability surface. Adding verification to phase frontmatter would create a third verification surface (IP coverage block, phase frontmatter, markdown §5) without clear authority boundaries. If orchestration needs structured test commands, it should consume the IP coverage block or define a purpose-built schema.

## 10. Doctrine Outputs

### ADR: metadata placement heuristic

**DEC-004.** An ADR recording when artifact metadata belongs in frontmatter, code-fenced YAML, or markdown prose. Scoped to the heuristic derived from DE-106's phase field analysis:

- **Frontmatter**: identity, lineage, lifecycle status, stable planning-time contract fields (objective, criteria). Machine-readable, low-volatility, consumed by tooling and orchestration.
- **Code-fenced YAML blocks**: cross-artifact relationship graphs, formal verification traceability, and other structured data that exceeds frontmatter's flat key-value model or serves a distinct schema with its own versioning (e.g., `delta.relationships`, `verification.coverage`).
- **Markdown prose**: execution detail, task breakdowns, decisions, findings, progress tracking, and anything that changes frequently during execution or benefits from rich formatting.

The ADR does not mandate retroactive reconciliation of existing artifact kinds. It provides guidance for future artifact design decisions.
