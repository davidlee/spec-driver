# DR-106 – Phase sheet template DRY: eliminate triple-entry bookkeeping across frontmatter, blocks, and markdown

## 1. Executive Summary

- **Delta**: [DE-106](./DE-106.md)
- **Status**: preliminary draft
- **Owners / Team**: spec-driver maintainers
- **Last Updated**: 2026-03-21
- **Synopsis**: Replace phase-sheet triple-entry bookkeeping with a simpler canonical model where frontmatter carries only the machine-readable fields the tooling actually uses, and markdown body remains the human-authored execution surface

## 2. Problem & Constraints

- **Current Behaviour**: New phases are created from a template that emits `supekku:phase.overview` and `supekku:phase.tracking` blocks above a markdown body containing the same objective, criteria, verification, task, and risk data in richer prose/table form. Agents regularly update only one layer, causing drift.
- **Drivers / Inputs**: IMPR-022 documents the duplication map in detail; DE-104 demonstrates real operational fallout from phase-sheet drift; DE-004 explains why the current structure exists; ADR-004 requires deliberate delta-first design; ADR-009 argues against preserving extra structure merely to speculate about future registry/reporting needs.
- **Constraints / Guardrails**:
  - Keep `show delta` and list/reporting workflows useful for existing and new phases
  - Do not break lifecycle commands or existing completed deltas
  - Reuse existing frontmatter metadata mechanisms where possible (POL-001)
  - Current PROD-006 wording explicitly requires `phase.overview`, so implementation cannot proceed responsibly without reconciling spec language
- **Out of Scope**:
  - Full PhaseRegistry/reporting work
  - Parsing markdown checkboxes into authoritative progress state
  - Any new workflow/status model beyond what frontmatter already carries

## 3. Architecture Intent

- **Target Outcomes**:
  - New phases are authored against one canonical representation instead of duplicated block/body mirrors
  - Existing UX surfaces retain phase identity, status, and objective summaries without needing a dedicated registry
  - Validation distinguishes between legacy compatibility and the new canonical format
- **Guiding Principles**:
  - Keep structured data only where current tooling demonstrably consumes it
  - Prefer frontmatter for compact machine-readable summary fields and markdown body for detailed execution content
  - Design compatibility paths explicitly rather than relying on silent drift or hand-maintained duplication
  - Defer future progress-reporting ambitions until a real feature requires them
- **State Transitions / Lifecycle Impact**: Phase lifecycle status remains frontmatter-driven (`draft`, `in-progress`, `completed`, `deferred`). This design does not add new state transitions; it narrows where non-status phase content is authored.

## 4. Code Impact Summary

| Path | Current State | Target State |

## 6. Supporting Context

- **Research**: IMPR-022 – quantified duplication map and design-direction options
- **Research**: DE-104 – evidence that structural duplication contributes to phase drift in practice
- **Related Deltas / Specs**: DE-004, PROD-006, PROD-011, IMPR-021
- **Assumption**: Current tooling only needs machine-readable phase identity, lineage, status, and objective; richer structured progress can remain deferred without blocking current workflows

## 7. Design Decisions & Trade-offs

- **DEC-001 (recommended)**: Canonicalise on `frontmatter + markdown body` for new phases.
  - Rationale: This matches how agents already work, removes the highest-friction duplication, and still preserves a compact machine-readable summary surface.
  - Consequence: PROD-006 and related validation/tests must change; legacy block-only assumptions become compatibility concerns rather than canonical behaviour.
- **DEC-002 (recommended)**: Retain only identity, lineage (`delta`, `plan`), status, and objective as authoritative structured fields.
  - Rationale: `show delta` and list surfaces currently consume summary data, but criteria/tasks/risks/verification are not used programmatically enough to justify redundant YAML.
  - Consequence: Programmatic phase progress reporting remains out of scope until a future change explicitly funds it.
- **Alternative considered but not recommended**: Keep `phase.overview` and drop only `phase.tracking`.
  - Trade-off: Less disruptive than removing both blocks, but it preserves duplicated criteria/tasks/verification content and therefore preserves the main drift vector.
- **Alternative considered but not recommended**: Make blocks the sole source and generate markdown from them.
  - Trade-off: Maximally structured, but it pushes rich execution detail into awkward YAML and requires generator/rendering machinery the project does not have.

## 9. Rollout & Operational Notes

- **Migration / Backfill**: Recommended sequence is compatibility-first: teach readers/validators the new format, then update the creation template, then decide whether to bulk-migrate historical phase files.
- **Observability / Alerts**: No runtime telemetry expected. Validation output and formatter smoke tests are the primary signals that compatibility has been preserved.
- **Recovery / Rollback**: If simplified phases break display or validation, restore legacy block emission in `create_phase()` while retaining any non-breaking reader improvements. Keep migrations, if any, isolated for easy revert.
