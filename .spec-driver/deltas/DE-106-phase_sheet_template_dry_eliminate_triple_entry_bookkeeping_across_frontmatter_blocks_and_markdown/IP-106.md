## 1. Summary

- **Delta**: DE-106 — Phase sheet template DRY: eliminate triple-entry bookkeeping
- **Design Revision**: [DR-106](./DR-106.md) — approved after 3 review passes (14 findings integrated)
- **Specs Impacted**: PROD-006, PROD-011
- **Problems / Issues**: IMPR-022
- **Desired Outcome**: New phases author structured data once (frontmatter) and execution detail once (markdown body). No more parallel block/body maintenance.

## 2. Context & Constraints

- **Current Behaviour**: Phase sheets duplicate content across `phase.overview` block, `phase.tracking` block, and markdown body
- **Target Behaviour**: Frontmatter carries planning contract (`plan`, `delta`, `objective`, `entrance_criteria`, `exit_criteria`). Markdown body carries execution detail and progress. No embedded YAML blocks for new phases.
- **Dependencies**: DE-107 Pydantic spike — Phase 1 serves as the feasibility test. Fallback to inline validation if negative.
- **Constraints**: Maintain compatibility for existing phase bundles. Preserve `show delta` / `list_changes` usefulness. `artifacts.py` is the primary compatibility choke point (E1).

## 4. Key Design Decisions (from DR-106)

- **DEC-005**: Canonical fields: `plan`, `delta`, `objective`, `entrance_criteria`, `exit_criteria`
- **DEC-006**: Contract (frontmatter) vs progress (markdown checkboxes)
- **DEC-007**: Drop `phase.tracking` entirely; regex fallback handles task stats
- **DEC-008**: Phase model is the Pydantic spike for DE-107

## 5. Testing & Verification Plan

- Phase creation tests: verify new frontmatter shape, criteria population from plan entries
- Artifact loading tests: frontmatter-first resolution, block fallback for legacy phases
- Formatter tests: `show delta` phase table with new and legacy formats
- Validator tests: no false warnings for block-free new phases
- Workspace validation: `spec-driver validate` against real historical deltas
- Pydantic spike: round-trip fidelity, import time, YAML date handling
