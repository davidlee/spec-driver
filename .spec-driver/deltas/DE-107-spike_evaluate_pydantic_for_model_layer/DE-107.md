## 1. Summary & Context

- **Technical Specs**: SPEC-132 (memory models), SPEC-159 (drift models), SPEC-116 (frontmatter metadata)
- **Change Drivers**: POL-001 (maximise reuse); DE-106 proved Pydantic feasibility (37ms import, clean parsing, go decision); IMPR-024 requests kind-aware frontmatter validation
- **Predecessor**: DE-106 delivered `PhaseSheet` as the reference Pydantic model. This delta extends that pattern to the remaining model layer.

## 2. Motivation

The codebase has 8 model files (~875 lines) following a repeated pattern:

- `@dataclass` with manual defaults
- `from_frontmatter()` / `from_dict()` classmethod doing field-by-field `fm.get()` with defaults
- Custom `_parse_date()` / type coercion helpers
- Manual `to_dict()` serialization with conditional field inclusion
- Ad-hoc validation functions

DE-106's `PhaseSheet` proved this collapses to type annotations + `ConfigDict(extra="ignore")`. This delta migrates the remaining models to Pydantic, eliminating boilerplate and enabling kind-aware frontmatter validation.

## 3. Scope & Objectives

### Primary Outcomes

- Convert `@dataclass` models to Pydantic `BaseModel` across all model files
- Replace `from_frontmatter()` / `to_dict()` boilerplate with Pydantic construction and `model_dump()`
- Wire kind-aware frontmatter validation in the workspace validator (extending PhaseSheet pattern)
- Update `list schemas` / schema documentation for new frontmatter-first models
- Update `plan.py` frontmatter metadata to include `plan` and `delta` fields (DE-106 gap)

### Model Inventory

| File | Model(s) | Lines | Boilerplate surface | Priority |
|---|---|---|---|---|
| `memory/models.py` | `MemoryRecord` | 161 | `from_frontmatter()`, `to_dict()`, `_parse_date()` | High |
| `backlog/models.py` | `BacklogItem` | 128 | `from_frontmatter()` equivalent | High |
| `specs/models.py` | `SpecEntry` | 140 | `to_dict()` | Medium |
| `drift/models.py` | 4 frozen dataclasses | 170 | Frozenset validation | Medium |
| `cards/models.py` | Kanban card models | 102 | `from_frontmatter()` | Medium |
| `diagnostics/models.py` | Diagnostic models | 47 | Minimal | Low |
| `sync/models.py` | Sync models | 68 | Minimal | Low |
| `docs/python/models.py` | Doc models | 59 | Minimal | Low |

### Documentation Touchpoints

- `list schemas` output: `phase.overview` and `phase.tracking` block schemas remain for legacy; document frontmatter-first approach
- `schema.py` block-to-module mappings: no changes needed (legacy blocks still valid)
- Frontmatter metadata (`plan.py`): add `plan` and `delta` fields to plan/phase kind schema
- Schema examples in `list schemas frontmatter`: will auto-update when models change

### Operational Constraints

- Incremental: one model at a time, behavioural parity gated by existing tests
- No registry API changes — models present the same interface to consumers
- Legacy data compatibility: `extra="ignore"`, optional fields with defaults

### Dependencies

- DE-106 (completed — `PhaseSheet` reference implementation)

## 4. Out of Scope

- CLI command changes
- Block schema removal (legacy blocks remain for backward compatibility)
- Registry architecture changes (registries still construct models the same way)
- Pydantic settings/config management

## 5. Approach Overview

- **Phase 1**: `MemoryRecord` conversion (highest boilerplate, best test coverage)
- **Phase 2**: `BacklogItem`, `SpecEntry`, `cards` models
- **Phase 3**: `drift` models (frozen), remaining small models, frontmatter metadata update, schema docs

Each phase: convert model → update registry/consumer → verify existing tests pass → lint.

## 6. Verification Strategy

- **Behavioural parity**: all existing tests must pass unmodified (or with minimal assertion updates)
- **Corpus validation**: parse real `.spec-driver/` data through new models
- **Import time**: measure and report per-phase

## 7. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Custom `to_dict()` logic not expressible via `model_dump()` | Medium | Medium | Use `@field_serializer` or keep thin wrapper methods where needed |
| Frozen dataclass semantics differ from Pydantic frozen | Low | Low | `ConfigDict(frozen=True)` — test mutation rejection |
| Registry coupling breaks on model signature change | Medium | Medium | Maintain `from_frontmatter()` as thin classmethod wrapper during migration |
| Downstream consumer expects dataclass-specific features | Low | Medium | Grep for `dataclasses.fields()`, `asdict()` usage |
