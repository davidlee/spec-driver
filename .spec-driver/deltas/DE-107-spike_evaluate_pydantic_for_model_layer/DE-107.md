## 1. Summary & Context

- **Technical Specs**: SPEC-132 (memory models), SPEC-159 (drift models), SPEC-116 (frontmatter metadata)
- **Implementation Plan**: [IP-107](./IP-107.md) – spike only
- **Change Drivers**: Recurring pattern of hand-rolled dataclass parsing/validation/serialization across model modules; POL-001 (maximise reuse, minimise sprawl)

## 2. Motivation

The codebase has a recurring pattern across every model module:
- `@dataclass` with manual defaults
- `from_frontmatter()` classmethod doing field-by-field `fm.get()` with defaults
- Custom `_parse_date()` / type coercion helpers
- Manual `to_dict()` serialization with conditional field inclusion
- Ad-hoc validation functions (`is_valid_entry_status()`, frozenset checks, warning logs)

This pattern is duplicated in `memory/models.py`, `drift/models.py`, `docs/python/models.py`, and every registry that parses frontmatter into structured data. Pydantic would collapse validation, coercion, and serialization into type annotations — eliminating boilerplate and making the model layer consistent.

This delta is a **spike**: investigate feasibility, quantify the win, prototype a single model conversion, and produce a go/no-go recommendation with a migration design if positive.

## 3. Scope & Objectives

- **Primary Outcomes**:
  - Inventory of all model/dataclass files and their hand-rolled parsing surface area
  - Prototype conversion of one model (e.g. `MemoryRecord`) to Pydantic BaseModel
  - Measured impact: lines removed, validation coverage gained, behavioural parity confirmed
  - Go/no-go recommendation with migration strategy if positive
  - ADR draft if recommendation is "go"
- **Operational Constraints**: Spike only — no production model changes merged. Time-boxed.
- **Dependencies**: None

## 4. Out of Scope

- Full migration of all models to Pydantic (future delta if spike is positive)
- Changes to CLI, registry, or command layers
- Pydantic settings or config management (not relevant here)

## 5. Approach Overview

- **System Touchpoints**: `supekku/scripts/lib/memory/models.py`, `supekku/scripts/lib/drift/models.py`, `supekku/scripts/lib/docs/python/models.py`, frontmatter parsing paths
- **Key Changes**:
  - Audit: catalogue every `@dataclass` model, `from_frontmatter`/`from_dict`, `to_dict`, custom validators
  - Prototype: convert `MemoryRecord` to `pydantic.BaseModel`, run existing tests
  - Evaluate: dependency footprint, import time impact, compatibility with `python-frontmatter` + YAML round-tripping
- **Migration / Rollout Notes**: Spike branch only. No merge to main without go decision.

## 6. Verification Strategy

- **Acceptance Criteria**:
  - Prototype passes all existing `memory/models_test.py` tests (behavioural parity)
  - Written assessment of: lines saved, validation coverage, dependency cost, migration complexity
  - Clear go/no-go recommendation documented in DR-107

## 7. Risks & Mitigations

- **Risk**: Pydantic's strict validation breaks permissive parsing (e.g. missing fields in legacy files) – _Likelihood_: medium – _Impact_: medium – _Mitigation_: test against real `.spec-driver/` corpus; Pydantic supports `model_config = ConfigDict(extra='ignore')` and optional fields
- **Risk**: Import time regression from Pydantic – _Likelihood_: low – _Impact_: low – _Mitigation_: measure in spike
- **Risk**: YAML date handling mismatch (PyYAML auto-parses dates vs Pydantic coercion) – _Likelihood_: medium – _Impact_: low – _Mitigation_: test round-trip fidelity in prototype

## 8. Follow-ups & Tracking

- **Future Deltas**: Full Pydantic migration delta (if go), covering all model modules and registry consumers
- **Open Decisions**: Go/no-go — resolved by this spike
