# DR-107 – Pydantic model layer migration

## 1. Executive Summary

- **Delta**: [DE-107](./DE-107.md)
- **Status**: approved
- **Synopsis**: Migrate all `@dataclass` models to Pydantic `BaseModel`, following the `PhaseSheet` pattern proven in DE-106. Replace hand-rolled parsing/validation/serialization boilerplate with type annotations and Pydantic configuration.

## 2. Problem & Constraints

### Current state

Every model module follows the same pattern:

```python
@dataclass
class Model:
    field: type = default

    @classmethod
    def from_frontmatter(cls, path, fm):
        return cls(
            field=fm.get("field", default),
            date_field=_parse_date(fm.get("date_field")),
            ...
        )

    def to_dict(self, root):
        data = {}
        if self.field:
            data["field"] = self.field
        ...
        return data
```

This is duplicated across 8 model files. Each new entity type requires reimplementing the same boilerplate.

### Target state

```python
class Model(BaseModel):
    model_config = ConfigDict(extra="ignore")
    field: type = default
```

Construction: `Model(**frontmatter)`. Serialization: `model_dump(exclude_none=True)`.

### Constraints

- **Behavioural parity**: existing tests must pass — they are the specification
- **No registry API changes**: registries construct models the same way
- **Legacy data**: `extra="ignore"` absorbs unknown fields without error
- **Incremental**: one model at a time, merged after tests pass

## 3. Conversion Pattern (from PhaseSheet reference)

### Standard conversion recipe

1. Replace `@dataclass` with `class Model(BaseModel):`
2. Add `model_config = ConfigDict(extra="ignore")`
3. Replace `field(default_factory=list)` with `list[str] | None = None` or `list[str] = Field(default_factory=list)`
4. Remove `from_frontmatter()` — direct construction via `Model(**fm)` works
5. Replace `to_dict()` with `model_dump(exclude_none=True)` or thin wrapper
6. Remove `_parse_date()` helpers — Pydantic handles `date | str | None` coercion
7. Replace `@dataclass(frozen=True)` with `ConfigDict(frozen=True)`

### Special cases

- **`MemoryRecord.to_dict(root)`**: takes a `root` argument for path relativization. Options:
  - Thin wrapper method calling `model_dump()` + path fixup
  - Pydantic serialization context (`model_dump(context={"root": root})` + `@field_serializer`)
  - Decision deferred to implementation — evaluate simplest approach
- **Frozen drift models**: `ConfigDict(frozen=True)` — verify no mutation paths exist
- **`BacklogItem` relations**: stored as `list[dict]` in frontmatter — Pydantic handles natively

## 4. Consumer Impact

### Registries

Each model has a registry that constructs it. Single call site per model — update directly:

```python
# Before
record = MemoryRecord.from_frontmatter(path, fm)

# After
record = MemoryRecord(**fm, path=str(path))
```

No compatibility wrapper needed — `from_frontmatter()` has exactly 1 call site per model (verified by grep).

### Formatters

Formatters typically access model attributes directly — no change needed since Pydantic models expose the same attribute access pattern.

### Validators

Extend the PhaseSheet validation pattern from `validator.py`:
- Each kind gets a Pydantic model
- Validator catches `ValidationError` and reports as warning
- Progressive: add kind-aware validation as models are converted

### Schema documentation

- `list schemas frontmatter` auto-discovers from `frontmatter_metadata/` modules — examples and field descriptions update when the metadata modules do
- `plan.py` frontmatter metadata needs `plan` and `delta` fields added (DE-106 gap)
- Block schemas (`phase.overview`, `phase.tracking`) remain for legacy compatibility — no removal

## 5. Resolved Questions

1. **`to_dict(root)` pattern**: Thin wrapper. `model_dump(exclude_none=True)` + manual path relativization in the wrapper. Same approach as PhaseSheet's `to_phase_entry()`. Pydantic serialization context is overkill for a single field fixup. (DEC-107-003)
2. **`dataclasses.fields()` / `asdict()` usage**: Zero usage in non-test code. Clean migration. (DEC-107-005)
3. **Import time budget**: DE-106 measured 37ms. All models share the same `pydantic` import — incremental cost per model is negligible.
4. **`from_frontmatter()` consumer audit**: `MemoryRecord` has 1 call site (`memory/registry.py:75`). Other models similar. Remove classmethod, update call site to direct construction. (DEC-107-002)

## 6. Phasing

| Phase | Models | Rationale |
|---|---|---|
| 1 | `MemoryRecord` | Highest boilerplate, best test coverage, named as original spike target |
| 2 | `BacklogItem`, `SpecEntry`, card models | Medium complexity, establish registry update pattern |
| 3 | Drift models (frozen), remaining small models, frontmatter metadata update, schema docs | Lower priority, cleanup |
