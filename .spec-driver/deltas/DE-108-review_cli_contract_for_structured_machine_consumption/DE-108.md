## 1. Summary & Context

- **Product Spec(s)**: PROD-010 (CLI UX), PROD-011 (Implementation Execution Workflow)
- **Implementation Plan**: [IP-108](./IP-108.md)
- **Change Drivers**: autobahn DE-001/DR-001 OQ-001 — autobahn needs structured, machine-consumable review CLI commands to orchestrate reviewer sessions programmatically

## 2. Motivation

- DE-102/DE-103 delivered review CLI commands designed for human/agent interactive use (`review prime`, `review complete`, etc.)
- autobahn (runtime orchestration layer) needs these same operations as a **structured machine contract**: JSON output, explicit exit codes, well-defined argument shapes
- Without this contract, autobahn's review vertical slice (slice 2) is untestable end-to-end
- This delta defines the spec-driver side of the boundary — autobahn depends on it, does not author it

## 3. Scope & Objectives

- **Primary Outcomes**:
  - Structured JSON output mode for all review CLI commands (`--output json` or equivalent)
  - Well-defined command signatures: `review prime`, `review status`, `review finding`, `review complete`
  - Documented exit code semantics (success, guard violation, precondition failure)
  - Output schemas documented as contracts autobahn can validate against
- **Operational Constraints**: Must not break existing interactive CLI usage
- **Dependencies**:
  - DE-109 (review state machine formalisation) — informs which states/transitions the CLI exposes, but DE-108 can proceed with current implicit model

## 4. Out of Scope

- Changing review business logic or state machine (that's DE-109)
- Exporting workflow YAML fixtures (that's DE-110)
- autobahn-side implementation — autobahn consumes, does not define
- Non-review CLI commands (workflow status, phase start, etc. — separate future work if needed)

## 5. Approach Overview

- **System Touchpoints**: `supekku/scripts/lib/workflow/review_io.py`, review CLI commands, output formatting
- **Key Changes**:
  - Add `--output json` flag to review commands
  - Define JSON output schemas for each command's response
  - Ensure exit codes are deterministic and documented
  - Add guard-violation responses as structured error payloads (not just stderr text)

## 6. Verification Strategy

- **Acceptance Criteria**:
  - All review commands accept `--output json` and produce valid, parseable JSON
  - Output schemas are documented and stable
  - Guard violations produce structured error responses with specific exit codes
  - Existing interactive usage is unaffected (default output unchanged)

## 7. Risks & Mitigations

- **Risk**: Output schema becomes a cross-project contract — changes break autobahn — _Likelihood_: medium — _Impact_: high — _Mitigation_: Version the output schema; autobahn contract tests catch drift (DE-110)
- **Risk**: Over-specifying command shapes constrains future CLI evolution — _Likelihood_: low — _Impact_: medium — _Mitigation_: Keep contract surface minimal; expose what autobahn needs, not everything

## 8. Follow-ups & Tracking

- **Related Deltas**: DE-109 (state machine), DE-110 (schema export)
- **Open Questions**: Should `--output json` apply to all CLI commands globally, or review-only for now?
