## 1. Summary & Context

- **Product Spec(s)**: PROD-010 (CLI UX), PROD-011 (Implementation Execution Workflow)
- **Implementation Plan**: [IP-108](./IP-108.md)
- **Change Drivers**: autobahn DE-001/DR-001 OQ-001 — autobahn needs structured, machine-consumable review CLI commands to orchestrate reviewer sessions programmatically

## 2. Motivation

- DE-102/DE-103 delivered review CLI commands designed for human/agent interactive use (`review prime`, `review complete`, etc.)
- autobahn (runtime orchestration layer) needs these same operations as a **structured machine contract**: JSON output, explicit exit codes, well-defined argument shapes
- Without this contract, autobahn's review vertical slice (slice 2) is untestable end-to-end
- This delta defines the spec-driver side of the boundary — autobahn depends on it, does not author it

## 3. Scope & Objectives

- **Primary Outcomes**:
  - Structured JSON output mode via `--format json` for all review CLI commands (prime, complete, teardown, status, finding)
  - Consistent JSON envelope with version, command, status, exit_code, data/error
  - New commands: `review status` (query review state), `review finding add/dispose/list` (manage findings programmatically)
  - Granular exit codes: 0 success, 1 unexpected, 2 precondition, 3 guard violation
  - CLI-generic envelope helpers in `common.py` reusable by future commands
  - Finding-level mutation helpers in `review_io.py` (add, dispose)
  - CLI output schema registration deferred to DE-110
- **Operational Constraints**: Must not break existing interactive CLI usage; default output unchanged
- **Dependencies**:
  - DE-109 (review state machine formalisation) — informs which states/transitions the CLI exposes, but DE-108 can proceed with current model
  - DR-109 terminology (bootstrap_status, judgment_status, disposition) — naming alignment required

## 4. Out of Scope

- Changing review business logic or state machine (that's DE-109)
- Exporting workflow YAML fixtures (that's DE-110)
- CLI output schema registration in `.contracts/` (deferred to DE-110, per DEC-108-013)
- autobahn-side implementation — autobahn consumes, does not define
- Non-review CLI commands (workflow status, phase start, etc. — separate future work if needed)

## 5. Approach Overview

- **System Touchpoints**: `supekku/cli/common.py`, `supekku/cli/workflow.py`, `supekku/scripts/lib/workflow/review_io.py`
- **Key Changes**:
  - Add `--format json` to all review commands via `FormatOption` (DEC-108-009)
  - CLI-generic envelope helpers: `cli_json_success`, `cli_json_error`, `emit_json_and_exit` in `common.py` (DEC-108-001)
  - Exit codes `EXIT_PRECONDITION=2`, `EXIT_GUARD_VIOLATION=3` in `common.py` (DEC-108-008)
  - New `review status` command (bootstrap_status, judgment_status, findings_summary, staleness inputs)
  - New `review finding` subgroup: `add` (ID assignment R{round}-{seq}), `dispose` (DR-109 §3.4 guards), `list` (--round N optional)
  - Finding-level mutation helpers in `review_io.py`: `add_finding_to_round()`, `dispose_finding()` (DEC-108-004)
  - Early format check + try/except error routing for stderr suppression in JSON mode (DEC-108-010)
  - Full 40-char SHA in JSON output, short SHA in human output (DEC-108-002)

## 6. Verification Strategy

- **Acceptance Criteria**:
  - All review commands accept `--format json` and produce valid, parseable JSON with versioned envelope
  - Exit codes 0/1/2/3 exercised with correct error kinds
  - Guard violations produce structured error responses (exit 3, kind: guard_violation)
  - Precondition failures produce structured error responses (exit 2, kind: precondition)
  - JSON mode never writes to stderr
  - Finding ID assignment follows R{round}-{seq} scheme
  - Finding cold start (no review-findings.yaml) handled correctly
  - Finding-not-found on dispose yields EXIT_PRECONDITION
  - Existing interactive usage is unaffected (default output unchanged)
  - Full SHA in JSON, short SHA in human output

## 7. Risks & Mitigations

- **Risk**: Output schema becomes a cross-project contract — changes break autobahn — _Likelihood_: medium — _Impact_: high — _Mitigation_: Version the output schema; autobahn contract tests catch drift (DE-110)
- **Risk**: Over-specifying command shapes constrains future CLI evolution — _Likelihood_: low — _Impact_: medium — _Mitigation_: Keep contract surface minimal; expose what autobahn needs, not everything
