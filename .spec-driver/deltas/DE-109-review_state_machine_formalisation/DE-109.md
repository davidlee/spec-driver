## 1. Summary & Context

- **Product Spec(s)**: PROD-011 (Implementation Execution Workflow)
- **Implementation Plan**: [IP-109](./IP-109.md)
- **Change Drivers**: autobahn DE-001/DR-001 OQ-003 — autobahn needs a formal, queryable definition of review states and transitions to make orchestration decisions

## 2. Motivation

- The review lifecycle is currently implicit across DR-102 prose and `review_io.py` implementation
- Two distinct sub-lifecycles exist but aren't formally modelled:
  - **Bootstrap lifecycle**: cold → warming → warm → stale → invalid → reusable
  - **Judgment lifecycle**: not_started → in_progress → approved / changes_requested
- autobahn needs to query current review state and reason about valid transitions without reimplementing spec-driver's logic
- Formalising the state machine also benefits spec-driver internally — guards, validation, and CLI output become derivable from a single source of truth

## 3. Scope & Objectives

- **Primary Outcomes**:
  - Explicit state machine definition for review bootstrap lifecycle (states, transitions, guards)
  - Explicit state machine definition for review judgment lifecycle (states, transitions, guards)
  - Relationship between the two lifecycles documented (interleaving rules)
  - States and transitions registered as enums / constants (not prose-only)
  - Query API: current review state derivable from artifact state (review-index.yaml, review-findings.yaml)
- **Operational Constraints**: Must be consistent with existing DE-102/DE-103 implementation; refactor, don't rewrite
- **Dependencies**: None hard — existing implementation is the source of truth to formalise

## 4. Out of Scope

- Changing the parent workflow state machine (7-state, already formalised in `state_machine.py`)
- CLI output format changes (that's DE-108)
- New review features or transitions not already implicit in current implementation

## 5. Approach Overview

- **System Touchpoints**: `workflow/review_io.py`, `workflow/state_machine.py`, enums/constants
- **Key Changes**:
  - Extract implicit review states from `review_io.py` into explicit state machine (parallel to `state_machine.py` pattern)
  - Define bootstrap and judgment state enums
  - Define valid transitions with guards (e.g., "cannot approve with open blocking findings")
  - Expose current-state derivation as a pure function: `(review_index, review_findings) → ReviewState`

## 6. Verification Strategy

- **Acceptance Criteria**:
  - Review states are defined as enums, not implicit string comparisons
  - State transitions have explicit guard predicates
  - Current state is derivable from artifact files via a single function
  - Existing review CLI behaviour is unchanged (formalisation, not feature change)
  - Tests cover all valid transitions and guard violations

## 7. Risks & Mitigations

- **Risk**: Formalisation reveals inconsistencies in current implementation — _Likelihood_: medium — _Impact_: low — _Mitigation_: Document inconsistencies, fix obvious bugs, defer ambiguities to follow-up
- **Risk**: Over-engineering state machine abstraction for 2 small lifecycles — _Likelihood_: medium — _Impact_: low — _Mitigation_: Use simple enum + transition table, not a generic FSM framework

## 8. Follow-ups & Tracking

- **Related Deltas**: DE-108 (CLI contract — consumes these states), DE-110 (schema export — exports state enums)
- **Open Questions**: Should review state machine reuse the `StateMachine` class from `state_machine.py`, or is a lighter-weight approach appropriate given the smaller state space?
