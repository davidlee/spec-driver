## 1. Summary & Context

- **Product Spec(s)**: PROD-011 (Implementation Execution Workflow)
- **Design Revision**: [DR-109](./DR-109.md)
- **Implementation Plan**: [IP-109](./IP-109.md)
- **Change Drivers**: autobahn DE-001/DR-001 OQ-003 — autobahn needs a formal, queryable review state machine; current implementation is implicit, incomplete, and violates POL-002

## 2. Motivation

- The review lifecycle is implicit across DR-102 prose and scattered implementation
- Two sub-lifecycles (bootstrap, judgment) lack formal state machines, transition tables, or guards
- The "cannot approve with open blocking findings" guard does not exist
- Finding disposition has no structured model — no rationale, no actor, no human-gate for waivers
- Rounds are destructive — each `review complete` overwrites rather than accumulates
- The audit system (DE-079) already solved disposition well; review should align
- autobahn cannot implement its review vertical slice without a formal state machine contract

## 3. Scope & Objectives

- **Primary Outcomes**:
  - `review_state_machine.py` — new module with BootstrapStatus and ReviewStatus StrEnums, transition tables, guard predicates
  - FindingStatus and FindingDispositionAction StrEnums with structured disposition model (action, by, rationale, resolved_at)
  - `can_approve()` guard enforcing: no open blocking findings, agents cannot waive blocking findings
  - Accumulative rounds in review-findings (schema v2), preserving all round history
  - Round-level `session` extension point (opaque dict) for autobahn metadata
  - CLI updates: `review prime` sets judgment via transition, `review complete` enforces guard
  - `workflow_metadata.py` validation lists derived from StrEnums
  - Provisional CLI UX design for finding disposition commands (design only, implementation deferred)
- **Operational Constraints**: Must not break existing interactive CLI usage; schema v1 backward-compatible reads
- **Dependencies**: None hard — existing implementation is the source of truth to formalise

## 4. Out of Scope

- Finding disposition CLI commands (`review finding resolve/defer/waive/supersede`) — designed in DR-109 §4, implementation deferred
- Review query commands (`review status`, `review findings`) — deferred or DE-108
- `--output json` for review commands — DE-108
- Audit-side consumption of review dispositions — follow-up delta
- Review→audit traceability enforcement in validators — follow-up delta
- Changes to parent workflow state machine

## 5. Approach Overview

- **System Touchpoints**: `workflow/review_state_machine.py` (new), `workflow/staleness.py`, `workflow/review_io.py`, `blocks/workflow_metadata.py`, `cli/workflow.py`
- **Key Changes**:
  - Extract BootstrapStatus from staleness.py into review_state_machine.py as StrEnum
  - Create ReviewStatus, FindingStatus, FindingDispositionAction StrEnums
  - Build transition tables and apply functions parallel to state_machine.py pattern
  - Implement can_approve() guard with disposition constraints
  - Refactor review_io.py for accumulative rounds (schema v2)
  - Update CLI to use transition functions and guards
  - Derive workflow_metadata.py lists from enums

## 6. Verification Strategy

- **Requirements Coverage**: All transitions, guards, and disposition constraints tested
- **Planned Artefacts**: VT-109-001 through VT-109-005, VA-109-001 (see DR-109 §5.4)
- **Acceptance Criteria**:
  - All review state transitions go through explicit transition table
  - `review complete --status approved` fails with open blocking findings
  - `review complete --status approved` fails if blocking finding waived by agent
  - Accumulative rounds preserve all prior round data
  - Schema v1 files readable, writes produce v2
  - BootstrapStatus, ReviewStatus, FindingStatus are proper StrEnums (POL-002)
  - Existing interactive CLI output unchanged

## 7. Risks & Mitigations

- **Risk**: Schema v2 breaks existing review-findings — _Likelihood_: medium — _Impact_: low (transient files) — _Mitigation_: v1 read compatibility
- **Risk**: Disposition model over-designed for current use — _Likelihood_: medium — _Impact_: low — _Mitigation_: Model is data-only; CLI deferred. Near-zero carrying cost
- **Risk**: Formalisation reveals implementation inconsistencies — _Likelihood_: medium — _Impact_: low — _Mitigation_: Fix obvious bugs, document edge cases
