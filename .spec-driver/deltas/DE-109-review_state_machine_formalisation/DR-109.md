# DR-109 – Review state machine formalisation

## 1. Executive Summary

- **Delta**: [DE-109](./DE-109.md)
- **Status**: draft
- **Last Updated**: 2026-03-22
- **Synopsis**: Extract, formalise, and complete the implicit review lifecycle into an explicit state machine with proper enums, transition tables, guards, accumulative rounds, and a structured finding disposition model inspired by the audit disposition pattern.

## 2. Problem & Current State

### 2.1 Bootstrap lifecycle (staleness.py + review prime)

- `BootstrapStatus` is a plain class with 6 string constants — not a StrEnum (POL-002 violation)
- `warming` is defined but never used anywhere in the codebase
- Transition logic is split between `evaluate_staleness()` (warm→stale/invalid/reusable) and `review_prime_command()` (cold→warm, invalid→warm)
- No explicit transition table — logic is ad-hoc conditional branches

### 2.2 Judgment lifecycle (review_io.py + review complete)

- `REVIEW_STATUS_VALUES` defines 5 values as a plain list in `workflow_metadata.py`
- Only 2 values are reachable via commands (`changes_requested`, `approved`)
- `not_started` and `in_progress` are never set by any command
- `blocked` exists in the list but is never set
- No transition table, no guards
- The "cannot approve with open blocking findings" guard does not exist

### 2.3 Finding disposition

- Findings have `status` from `FINDING_STATUS_VALUES`: `open`, `resolved`, `waived`, `superseded`
- No structured disposition record — no rationale, no actor, no evidence
- No command to disposition an individual finding
- No constraint preventing agents from silently waiving blocking findings
- The audit system (DE-079) already has a mature disposition model with status×kind matrices and mandatory rationale — review findings should align

### 2.4 Rounds

- `build_findings()` creates a single snapshot, destructively overwriting previous round data
- The `history` field exists but is optional and manually populated
- Round cycling (`changes_requested → in_progress`) has no formal model

### 2.5 Cross-project dependency

autobahn DE-001/DR-001 (OQ-003) requires a formal, queryable review state machine to make orchestration decisions. Without it, autobahn's review vertical slice is unimplementable.

## 3. Target Architecture

### 3.1 Module: `workflow/review_state_machine.py`

New module parallel to `state_machine.py`. Single responsibility: review lifecycle states, transitions, guards, and disposition constraints.

**Dependency direction** (leaf module — no internal deps beyond stdlib/enum):

```
review_state_machine.py  (leaf)
        ↑
   staleness.py          (imports BootstrapStatus)
   review_io.py          (imports ReviewStatus, FindingStatus, FindingDispositionAction)
        ↑
   workflow_metadata.py  (derives validation lists from enums)
        ↑
   cli/workflow.py       (orchestrates via transition functions + guards)
```

### 3.2 Bootstrap lifecycle (derived status with validity invariants)

Unlike the parent workflow and judgment lifecycles, bootstrap status is **derived, not command-driven**. The stored `bootstrap_status: warm` in review-index.yaml is a snapshot — if HEAD moves, the true status is `stale` regardless of what the file says. The only mutation is `review prime`, which rebuilds the cache and writes `warm`.

This is modelled as a **status derivation function** with a **validity matrix** that constrains which transitions are legal, catching logic errors via assertions rather than enforcing transitions via a command table.

**States** — `BootstrapStatus(StrEnum)`, 5 values:

| State | Meaning |
|-------|---------|
| `cold` | No review-index.yaml exists. First-time prime. |
| `warm` | Cache is current. Reviewer can work. |
| `stale` | Commits or phase changes since last prime. Cache behind but structurally intact. |
| `reusable` | Stale with minor drift only. Incremental update sufficient. |
| `invalid` | Cache is broken (schema mismatch, files deleted). Must rebuild. |

`warming` is removed — the prime command is synchronous and atomic; there is no observable intermediate state.

**Validity matrix** (legal status transitions):

```
cold      → warm                          # prime: fresh build
warm      → warm, stale                   # re-prime (idempotent) or drift detected
stale     → warm, invalid, reusable       # re-prime, invalidation, or minor drift
reusable  → warm                          # prime: incremental update
invalid   → warm                          # prime: full rebuild
```

Illegal transitions (e.g. `cold → reusable`, `invalid → reusable`, `warm → cold`) indicate a bug in derivation logic.

**Implementation:**

```python
def derive_bootstrap_status(
  index: dict | None,
  *,
  current_phase_id: str,
  current_head: str,
  changed_files: list[str] | None = None,
  deleted_domain_files: list[str] | None = None,
  previous_status: BootstrapStatus | None = None,
) -> BootstrapStatus:
  """Derive current bootstrap status from observable context.

  Deterministic — no I/O, no mutation. The only code that writes
  bootstrap_status is review prime, which always writes WARM.

  When previous_status is provided, raises AssertionError if the
  derived transition is illegal (indicates a derivation logic bug).
  """
```

`evaluate_staleness()` is refactored to call `derive_bootstrap_status()` internally, passing the stored `bootstrap_status` as `previous_status`. Returns `StalenessResult` with the derived status. The validity matrix assertion lives inside `derive_bootstrap_status()` — it checks `(previous_status, derived_status)` against `VALID_BOOTSTRAP_TRANSITIONS` and raises `AssertionError` on illegal transitions.

### 3.3 Judgment state machine

**States** — `ReviewStatus(StrEnum)`, 4 values:

| State | Meaning |
|-------|---------|
| `not_started` | Review-index may exist but review work has not begun. |
| `in_progress` | Reviewer is actively working. |
| `approved` | Review passed. Terminal for this round. |
| `changes_requested` | Review found issues. Non-terminal — new round can begin. |

`blocked` is removed from judgment lifecycle. Blocking is the parent workflow's concern — the parent state machine already has `block`/`unblock` with `previous_state` restore. If a review is blocked, the parent workflow transitions to `blocked`.

**Transition table:**

```
(not_started,       begin_review)  → in_progress
(in_progress,       approve)       → approved            # guard: can_approve()
(in_progress,       request_changes) → changes_requested
(changes_requested, begin_review)  → in_progress         # new round
```

**Trigger mapping** — which CLI command fires which judgment transition:

| CLI command | Precondition | Judgment transition |
|---|---|---|
| `review prime` | No review-index exists (cold start) | `not_started → in_progress` |
| `review prime` | `judgment_status: changes_requested` | `changes_requested → in_progress` (new round) |
| `review prime` | `judgment_status: in_progress` | No-op (already in progress) |
| `review complete --status approved` | `judgment_status: in_progress` | `in_progress → approved` (guard: `can_approve()`) |
| `review complete --status changes_requested` | `judgment_status: in_progress` | `in_progress → changes_requested` |

`review prime` is explicitly dual-purpose: it brings bootstrap to `warm` AND judgment to `in_progress`. This coupling is intentional — priming is the act that makes a review actionable.

**Judgment status storage**: Judgment status lives in `review-index.yaml` under `review.judgment_status`. review-index is the coordination point for judgment state — both prime and complete write to it.

**File writer matrix:**

| File | Writers | What they write |
|------|---------|-----------------|
| review-index.yaml | `review prime` | bootstrap_status, judgment_status (init/round transition), domain_map, staleness cache |
| | `review complete` | judgment_status transitions (in_progress → approved/changes_requested) |
| review-findings.yaml | `review complete` | round entries (creation, findings content, completed_at) |
| | `review finding *` | disposition updates on existing findings (in-place within originating round) |

`review prime` reads review-index (if exists) to discover current judgment status, applies the transition, and writes the updated status back. `review complete` reads review-index for judgment status (to validate the transition), writes the transition result back to review-index, and writes round content to review-findings.

On cold start, `review prime` creates review-index with `bootstrap_status: warm, judgment_status: in_progress`. No findings file is created until `review complete`.

**Guard scope across rounds**: `can_approve()` evaluates **all blocking findings from all rounds**. Finding IDs are round-scoped (`R{round}-{seq}`) and unique across rounds, so no deduplication is needed. Dispositions are applied in-place within the originating round. This ensures the accumulative model is load-bearing for approval safety — a blocking finding from round 1 that was never dispositioned blocks approval in round 3.

**Guard — `can_approve()`:**

Operates on typed domain objects (Pydantic BaseModel, parsed from YAML by `review_io.py`), not raw dicts:

```python
class ReviewFinding(BaseModel):
  model_config = ConfigDict(extra="ignore")
  id: str
  title: str
  summary: str | None = None
  status: FindingStatus = FindingStatus.OPEN
  disposition: FindingDisposition | None = None

def can_approve(blocking_findings: list[ReviewFinding]) -> tuple[bool, list[str]]:
  """Check whether approval is permitted.

  blocking_findings: all blocking findings collected from all rounds.
  Finding IDs are round-scoped and unique; no deduplication needed.

  Returns (allowed, reasons) where reasons lists blocking violations.
  The guard checks disposition, not status — status is derived from
  disposition and checking both creates short-circuit bugs (ext-review #1).
  """
  reasons = []
  for f in blocking_findings:
    if f.disposition is None:
      reasons.append(f"blocking finding {f.id} has no disposition")
      continue
    if f.disposition.action == FindingDispositionAction.FIX:
      if not f.disposition.resolved_at:
        reasons.append(f"blocking finding {f.id} marked fixed without resolved_at sha")
    elif f.disposition.action == FindingDispositionAction.WAIVE:
      if f.disposition.authority != DispositionAuthority.USER:
        reasons.append(f"blocking finding {f.id} waived without user authority")
      if not f.disposition.rationale:
        reasons.append(f"blocking finding {f.id} waived without rationale")
    elif f.disposition.action == FindingDispositionAction.DEFER:
      if f.disposition.authority != DispositionAuthority.USER:
        reasons.append(f"blocking finding {f.id} deferred without user authority")
      if not f.disposition.backlog_ref:
        reasons.append(f"blocking finding {f.id} deferred without backlog reference")
  return (len(reasons) == 0, reasons)
```

### 3.4 Finding disposition model

Inspired by the audit disposition pattern (DE-079, `core/frontmatter_metadata/audit.py`). Both models share a structural pattern (finding → structured disposition with constrained actions and mandatory rationale for overrides), but solve different domain problems: audit disposition is descriptive/reconciliation-oriented (what happened, how to route it), while review disposition is prescriptive/triage-oriented (what to do about this finding). Consolidation would require a shared abstraction that doesn't yet exist; premature at 2 instances.

**Enums:**

```python
class FindingStatus(StrEnum):
  OPEN = "open"
  RESOLVED = "resolved"
  WAIVED = "waived"
  SUPERSEDED = "superseded"

class FindingDispositionAction(StrEnum):
  FIX = "fix"            # implement the requested change
  DEFER = "defer"        # create backlog item, not blocking now
  WAIVE = "waive"        # accept as-is with rationale
  SUPERSEDE = "supersede" # replaced by different finding

class DispositionAuthority(StrEnum):
  USER = "user"          # human made the decision
  AGENT = "agent"        # agent made the decision
```

**Disposition record** (Pydantic BaseModel per DE-107 migration direction):

```python
class FindingDisposition(BaseModel):
  model_config = ConfigDict(extra="ignore")
  action: FindingDispositionAction
  authority: DispositionAuthority      # who has decision authority
  actor_id: str | None = None         # specific identity when needed
  rationale: str | None = None        # required for waive, defer
  backlog_ref: str | None = None      # e.g. "ISSUE-045", when action=defer
  resolved_at: str | None = None      # git sha, when action=fix
  superseded_by: str | None = None    # finding ID, when action=supersede
  timestamp: str | None = None        # ISO 8601
```

**Status derivation**: Finding `status` is derived from disposition, not independently settable:

| Disposition action | → Finding status | Unblocks blocking finding? |
|---|---|---|
| _(none)_ | `open` | No |
| `fix` (with `resolved_at`) | `resolved` | Yes |
| `fix` (without `resolved_at`) | — | Validation error on blocking |
| `defer` (authority: user + backlog_ref) | `open` (tracked, deprioritised) | Yes (conditional) |
| `defer` (authority: agent or no backlog_ref) | `open` | No |
| `waive` (authority: user + rationale) | `waived` | Yes |
| `waive` (authority: agent or no rationale) | — | Validation error on blocking |
| `supersede` | `superseded` | Yes |

Key: `defer` keeps status `open` because the issue isn't resolved — it's tracked. But it conditionally unblocks if the user has made an explicit, traceable decision (`authority: user`) with a concrete follow-up artifact (`backlog_ref`).

**Derivation enforcement**: `derive_finding_status(disposition: FindingDisposition | None) -> FindingStatus` is a pure function in `review_state_machine.py`. Status is **re-derived on read** (normalization) and **enforced on write** (the stored value is a convenience for grep/filtering, not authority). `review_io.py` calls the derivation function in both paths. v1 files with `status` but no `disposition` are preserved as-is (no disposition to derive from — legacy data).

**Validity constraints:**

| Action | Blocking finding | Non-blocking finding | Required fields |
|--------|------------------|---------------------|-----------------|
| `fix` | `resolved_at` required for blocking | ✓ | `resolved_at` required for blocking, recommended otherwise |
| `defer` | `authority: user` + `backlog_ref` required to unblock | ✓ | `rationale` required, `backlog_ref` required for blocking |
| `waive` | `authority: user` + `rationale` required | ✓ | `rationale` required |
| `supersede` | ✓ | ✓ | `superseded_by` required |

**YAML shape:**

```yaml
blocking:
  - id: R1-001
    title: Missing validation
    summary: Schema check absent for edge case
    status: resolved              # derived from disposition.action=fix
    disposition:
      action: fix
      authority: agent
      resolved_at: "abc1234"
      timestamp: "2026-03-20T12:00:00+00:00"
  - id: R1-003
    title: Error handling gap
    summary: Missing handler for timeout case
    status: open                  # derived: defer keeps status open
    disposition:
      action: defer
      authority: user
      rationale: "Out of scope for this delta; timeout handling is a separate concern"
      backlog_ref: "ISSUE-047"
      timestamp: "2026-03-20T14:00:00+00:00"
```

### 3.5 Accumulative rounds model

**Schema version**: Bumps review-findings from v1 to v2.

**Target structure:**

```yaml
schema: supekku.workflow.review-findings
version: 2
artifact:
  id: DE-103
  kind: delta
review:
  current_round: 2
  # judgment_status lives in review-index.yaml, not here
rounds:
  - round: 1
    status: changes_requested
    reviewer_role: reviewer
    completed_at: "2026-03-20T10:00:00+00:00"
    session: {}                # opaque, autobahn-owned (see §3.6)
    blocking:
      - id: R1-001
        title: Missing validation
        status: resolved
        disposition:
          action: fix
          authority: agent
          resolved_at: "abc1234"
    non_blocking:
      - id: R1-002
        title: Style nit
        status: waived
        disposition:
          action: waive
          authority: user
          rationale: "Intentional deviation for readability"
  - round: 2
    status: in_progress
    reviewer_role: reviewer
    blocking: []
    non_blocking: []
```

**Backward compatibility — v1→v2 read migration:**

v1 files are not auto-migrated. Review-findings are transient (per-delta, per-review-cycle), there are no dependent external consumers yet, and the v1 structure doesn't carry enough information for lossless migration (e.g., `resolved` and `waived` lists don't preserve original severity — blocking vs non-blocking). The migration path is: `review teardown` + `review prime` to start fresh with v2.

On read: if `version == 1`, raise a clear error directing the user to teardown and re-prime. Writes always produce v2.

**Finding ID stability**: Finding IDs (e.g. `R1-001`) are prefixed with round number. Findings dispositioned in round N appear in round N's data with updated status. Cross-round references use the original finding ID.

### 3.6 Round session metadata

The `session` key on a round record is a freeform metadata extension point:

- **Optional** — omitted if review is not orchestrated by autobahn
- **Dict type required** — validated as dict if present, contents not validated by spec-driver
- **Opaque to spec-driver** — preserved faithfully on read/write, never interpreted
- **Owned by autobahn** — autobahn defines its own schema for this block (e.g. `agent_id`, `harness`, `session_id`)

This allows autobahn to track which agent/model/harness conducted each review round without spec-driver needing to understand those concepts.

### 3.7 Cross-round finding collection and disposition in-place

Disposition modifies a finding **in-place within its originating round**. A finding raised in round 1 is always stored in `rounds[0].blocking` (or `non_blocking`). When dispositioned, its `disposition` and `status` fields are updated in that same location — no duplication into later rounds.

**Collection function for `can_approve()`**:

```python
def collect_blocking_findings(rounds: list[dict]) -> list[ReviewFinding]:
  """Collect all blocking findings from all rounds.

  Finding IDs are round-scoped (R{round}-{seq}) and unique across
  rounds — no deduplication is needed. Dispositions are applied
  in-place within the originating round (§3.8), so each finding's
  current disposition state is always in its originating entry.
  """
```

The guard sees every blocking finding ever raised, with its current disposition state. A finding from round 1 that was never dispositioned blocks approval in round 3 — the accumulative model is load-bearing.

**Disposition command data flow**: `review finding resolve DE-103 R1-001 --resolved-at abc1234` searches all rounds for finding ID `R1-001`, updates it in-place with the disposition, derives status, and writes back. The finding stays in its originating round.

### 3.8 Review → Audit traceability

Dispositioned review findings create an audit trail:

- `fix` with `resolved_at`: audit can verify the sha is an ancestor of HEAD and the fix persists
- `defer` with `backlog_ref`: audit target — was the backlog item addressed?
- `waive` with rationale: audit scrutiny point — was the waiver justified?

**DE-109 scope**: Ensure the data model supports this traceability (stable IDs, structured dispositions, `resolved_at` sha). Audit-side consumption (extending `_validate_audit_disposition` or equivalent to also check review dispositions) is a follow-up delta.

## 4. CLI UX — disposition and query commands

### 4.1 Finding disposition commands (in scope)

```
spec-driver review finding resolve   <delta> <finding-id> [--resolved-at <sha>]
spec-driver review finding defer     <delta> <finding-id> --rationale "..." [--backlog-ref ISSUE-XXX]
spec-driver review finding waive     <delta> <finding-id> --rationale "..." [--authority user|agent]
spec-driver review finding supersede <delta> <finding-id> --superseded-by <finding-id>
```

Follows existing verb-noun pattern (`review prime`, `review complete`, `review teardown`). Each command locates the finding by ID within its originating round (findings are dispositioned in-place, not duplicated across rounds — see §3.7).

Each command: reads findings, validates constraints (authority, rationale, backlog_ref per validity table in §3.4), updates the finding's disposition in-place, derives status, writes back. Thin CLI orchestration delegating to `review_io.py`.

### 4.2 Query commands (deferred)

```
spec-driver review status   <delta>           # bootstrap + judgment state, finding counts
spec-driver review findings <delta>           # list findings with disposition status
spec-driver review findings <delta> --open    # filter to undispositioned
```

These are the commands DE-108 would add `--output json` to for autobahn consumption.

### 4.3 Existing commands (updated behaviour)

```
spec-driver review prime     <delta>          # sets judgment to in_progress via transition
spec-driver review complete  <delta> -s ...   # enforces can_approve() guard
spec-driver review teardown  <delta>          # lifecycle reset (see below)
```

**Teardown and the judgment state machine**: `review teardown` deletes review-index, review-findings, and review-bootstrap files. This is a **lifecycle reset that operates outside the judgment state machine** — it is not modelled as a transition. After teardown, judgment status is undefined (no review-index exists). The next `review prime` re-initialises to `in_progress` via the cold-start path. This is an intentional escape hatch, not a gap: teardown is for cleanup after `approved` or abandonment, not for mid-review state manipulation.

## 7. Risks & Mitigations

- **Risk**: Schema v2 breaks existing review-findings files — _Likelihood_: medium — _Impact_: low (review findings are transient, not long-lived) — _Mitigation_: v1 read raises clear error; teardown + re-prime is low-cost recovery
- **Risk**: Accumulative round model grows large — _Likelihood_: low (reviews rarely exceed 3-4 rounds) — _Impact_: low — _Mitigation_: No action needed; could add archiving later if proven
- **Risk**: Disposition model is over-designed for current use — _Likelihood_: medium — _Impact_: low — _Mitigation_: Model follows proven audit pattern; CLI commands are thin orchestration to domain logic. Risk accepted given guard depends on it.
- **Risk**: Formalisation reveals inconsistencies in current implementation — _Likelihood_: medium — _Impact_: low — _Mitigation_: Fix obvious bugs, document edge cases, follow-up for ambiguities

## 8. Design Decisions Summary

| ID | Decision | Rationale |
|----|----------|-----------|
| DEC-109-001 | New `review_state_machine.py` module | SRP — parallel to `state_machine.py`, each module owns one state machine |
| DEC-109-002 | Remove `warming` from bootstrap | Never observable — prime is synchronous atomic |
| DEC-109-003 | Remove `blocked` from judgment | Parent workflow owns blocking; review-level blocked creates confusion |
| DEC-109-004 | Accumulative rounds | Destructive overwrite loses review history; rounds are intrinsic to judgment lifecycle |
| DEC-109-005 | Disposition model inspired by audit (DE-079) | Same structural pattern, different domain semantics; bidirectional review↔audit traceability |
| DEC-109-006 | Agents cannot waive blocking findings | `waive` on blocking requires `authority: user` + rationale — human-in-the-loop gate |
| DEC-109-007 | Enums in review_state_machine, metadata derives | Follows existing pattern; satisfies POL-002 |
| DEC-109-008 | Schema v2 for review-findings | Accumulative model changes structure; v1 reads rejected with clear error |
| DEC-109-009 | Round session metadata opaque to spec-driver | Freeform dict extension point — autobahn defines, spec-driver carries |
| DEC-109-010 | Review→audit traceability (data model only) | Stable finding IDs + structured dispositions enable it; audit consumption is follow-up |
| DEC-109-011 | Bootstrap is derived status, not command-driven state machine | Stored value is snapshot, not authority; derivation from observable context is the truth |
| DEC-109-012 | Finding status derived from disposition action | Prevents status/disposition inconsistency; status is a convenience projection |
| DEC-109-013 | `defer` on blocking requires `backlog_ref` | Must create tracked artifact — disposition is not just "later" |
| DEC-109-014 | `defer` conditionally unblocks (authority: user + backlog_ref) | User has made explicit, traceable decision with concrete follow-up |
| DEC-109-015 | FindingDisposition and ReviewFinding as Pydantic BaseModel | Per DE-107 migration direction; `can_approve()` operates on typed domain objects |
| DEC-109-016 | DispositionAuthority StrEnum replaces freeform `by` string | POL-002 compliance, prevents guard bypass via case/whitespace variation |
| DEC-109-017 | `superseded_by` field on FindingDisposition | Supports CLI contract for supersede action |
| DEC-109-018 | Teardown is lifecycle reset outside state machine | Judgment undefined until next prime; intentional escape hatch |
| DEC-109-019 | v1 findings not auto-migrated | Transient files, no external consumers; teardown + re-prime path |
| DEC-109-020 | `review prime` is dual-purpose (bootstrap + judgment) | Priming is the act that makes a review actionable |
| DEC-109-021 | Judgment status stored in review-index, not review-findings | review-index is the coordination point (both prime and complete write judgment transitions); review-findings exclusively owned by complete for round content |
| DEC-109-022 | `can_approve()` evaluates all rounds, dedup by finding ID | Accumulative model is load-bearing for approval safety |
| DEC-109-023 | `fix` on blocking requires `resolved_at` | Creates verifiable claim; closes path-of-least-resistance bypass |
| DEC-109-024 | Disposition commands in scope | Guard is unusable without them — operational deadlock |
| DEC-109-025 | Disposition modifies finding in-place within originating round | No cross-round duplication; dedup collects from all rounds |
| DEC-109-026 | `superseded_by` required when action=supersede | Structural consistency with other actions' required fields |
