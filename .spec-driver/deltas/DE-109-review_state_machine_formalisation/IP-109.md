## 1. Summary

- **Delta**: DE-109 — Review state machine formalisation
- **Design Revision**: DR-109
- **Specs Impacted**: PROD-011
- **Desired Outcome**: Explicit review lifecycle with StrEnums, transition tables, guards, accumulative rounds, structured finding disposition, and CLI disposition commands

## 2. Context & Constraints

- **Current Behaviour**: Review lifecycle is implicit, scattered across staleness.py, review_io.py, and CLI commands. No guards, no disposition model, destructive round overwrites.
- **Target Behaviour**: Per DR-109 — two formalised lifecycles (bootstrap derivation, judgment state machine), Pydantic-based disposition model, accumulative rounds (schema v2), cross-round approval guard, four disposition CLI commands.
- **Dependencies**: None hard. DE-107 (pydantic) is parallel — FindingDisposition uses Pydantic BaseModel independently.
- **Constraints**: Must not break existing interactive CLI usage. v1 findings files rejected with clear error.

## 5. Phase Detail Snapshot

- **Design Revision**: `DE-109/DR-109.md`
- **Notes**: `DE-109/notes.md`
- **Active Phase Sheet**: (created per phase)

### Phase 1 — Domain model (leaf module)

New `workflow/review_state_machine.py` with no internal dependencies:

- `BootstrapStatus(StrEnum)` — 5 values, validity matrix
- `ReviewStatus(StrEnum)` — 4 values
- `FindingStatus(StrEnum)` — 4 values
- `FindingDispositionAction(StrEnum)` — 4 values
- `DispositionAuthority(StrEnum)` — 2 values
- `FindingDisposition(BaseModel)` — action, authority, actor_id, rationale, backlog_ref, resolved_at, superseded_by, timestamp
- `ReviewFinding(BaseModel)` — id, title, summary, status, disposition
- `derive_bootstrap_status()` — pure derivation with validity matrix assertion
- `apply_review_transition()` — judgment transition table
- `derive_finding_status()` — status from disposition action
- `can_approve()` — cross-round guard with disposition constraints
- `collect_blocking_findings()` — collects from all rounds
- Full test suite in `review_state_machine_test.py`

### Phase 2 — I/O layer

Refactor `review_io.py` for accumulative rounds:

- Schema v2 structure: `rounds` array, per-round content
- `build_findings()` → accumulative (append round, preserve prior rounds)
- v1 read → `FindingsVersionError` with clear message
- Add `judgment_status` field to `build_review_index()` and `REVIEW_INDEX_METADATA`
- Finding serialisation: `ReviewFinding` ↔ dict with status re-derivation on read
- Tests updated for accumulative model

### Phase 3 — CLI commands

- `review finding resolve|defer|waive|supersede` — 4 disposition subcommands
- Update `review prime` to set judgment via `apply_review_transition()`
- Update `review complete` to enforce `can_approve()` guard
- `review complete` writes judgment transition to review-index
- Thin CLI orchestration: args → read → validate → update → write

### Phase 4 — Integration and cleanup

- Derive `workflow_metadata.py` validation lists from StrEnums (POL-002)
- Refactor `staleness.py`: import `BootstrapStatus` from review_state_machine, call `derive_bootstrap_status()`
- End-to-end test: multi-round review with disposition and approval
- `just check` green (lint + test)
- Update notes.md
