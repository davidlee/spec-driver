## 1. Summary & Context

- **Product Spec(s)**: PROD-012 (Code Contract Documentation)
- **Implementation Plan**: [IP-110](./IP-110.md)
- **Change Drivers**: autobahn DE-001/DR-001 OQ-002 — autobahn needs canonical workflow YAML fixtures and enum values exported by spec-driver for cross-project contract testing

## 2. Motivation

- autobahn defines provisional pydantic models for spec-driver's 5 workflow YAML schemas (`supekku:workflow.*@v1`)
- These models need validation against authoritative fixtures — not hand-maintained schema comparison
- spec-driver currently has no mechanism to export canonical fixture files or enum constants for external consumers
- Contract tests in autobahn would load these fixtures and validate parsing; drift breaks tests, not runtime
- DE-107 (pydantic spike) may eventually provide typed schema exports, but this delta is independent — fixtures are the immediate need

## 3. Scope & Objectives

- **Primary Outcomes**:
  - Canonical fixture YAML files for all 5 workflow schemas (state.yaml, handoff.current.yaml, review-index.yaml, review-findings.yaml, sessions.yaml)
  - Fixtures are representative, generated/validated by spec-driver's own schema logic
  - Enum value exports: Role, WorkflowStatus, PhaseStatus, and other workflow constants in a machine-readable format
  - Export mechanism: either committed fixture files or a CLI command (`spec-driver export fixtures`)
- **Operational Constraints**: Fixtures must stay in sync with schema changes — prefer generated over hand-maintained
- **Dependencies**:
  - Soft: DE-107 (pydantic) — if pydantic models land, fixtures can be generated from them; otherwise generate from current schema validators
  - Soft: DE-109 (review state machine) — review state enums should be included in export once formalised

## 4. Out of Scope

- Shared Python package between repos (premature — fixtures + contract tests are sufficient)
- Schema versioning infrastructure beyond current `@v1` convention
- autobahn-side contract test implementation (autobahn's responsibility)

## 5. Approach Overview

- **System Touchpoints**: `blocks/workflow_metadata.py` (schema definitions), new fixture generation, possibly CLI export command
- **Key Changes**:
  - Create canonical fixture YAML files covering each workflow schema with realistic data
  - Add fixture generation/validation script or CLI command
  - Export workflow enum values in a machine-readable format (YAML or JSON constants file)
  - Establish convention for fixture location (e.g., `.contracts/workflow/` or `exports/fixtures/`)

## 6. Verification Strategy

- **Acceptance Criteria**:
  - Fixture files exist for all 5 workflow schemas
  - Each fixture passes spec-driver's own schema validation
  - Enum export file contains all canonical workflow constants
  - A downstream consumer (autobahn) can parse fixtures without spec-driver as a dependency

## 7. Risks & Mitigations

- **Risk**: Fixtures drift from schema changes — _Likelihood_: medium — _Impact_: high — _Mitigation_: Generate fixtures from schema validators; include fixture validation in CI
- **Risk**: Fixture format becomes a de facto API — _Likelihood_: medium — _Impact_: medium — _Mitigation_: Version fixtures alongside schema versions; document stability guarantees

## 8. Follow-ups & Tracking

- **Related Deltas**: DE-107 (pydantic — future typed export), DE-108 (CLI contract — output schemas are also export surface), DE-109 (review states — enum values to export)
- **Open Questions**:
  - Committed fixtures vs CLI generation command? (committed is simpler; CLI is always-fresh)
  - Where do fixtures live? `.contracts/workflow/fixtures/` aligns with ADR-007
