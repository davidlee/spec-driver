# DR-110 – Complete schema show coverage for all registered block types

## 1. Executive Summary

- **Delta**: [DE-110](./DE-110.md)
- **Status**: draft
- **Last Updated**: 2026-03-22
- **Synopsis**: Eliminate the hardcoded metadata registry in `schema.py` by attaching `BlockMetadata` directly to `BlockSchema`, then fill the 9 missing metadata definitions (7 workflow + 2 spec) so every registered block type supports `--format=json-schema` and `--format=yaml-example`.

## 2. Problem & Constraints

### Current Behaviour

15 block schemas are registered in `schema_registry.BLOCK_SCHEMAS`. Only 6 support `--format=json-schema` and `--format=yaml-example`:

| Status | Block Types |
|--------|-------------|
| ✓ Working (6) | `delta.relationships`, `plan.overview`, `phase.overview`, `phase.tracking`, `revision.change`, `verification.coverage` |
| ✗ Missing metadata (7) | `workflow.state`, `workflow.handoff`, `workflow.review-index`, `workflow.review-findings`, `workflow.sessions`, `workflow.notes-bridge`, `workflow.phase-bridge` |
| ✗ No BlockMetadata at all (2) | `spec.relationships`, `spec.capabilities` |

The 7 workflow schemas already have full `BlockMetadata` definitions in `workflow_metadata.py` with complete field structures — but `schema.py` doesn't know about them because its `metadata_registry` dict only lists 6 entries.

The 2 spec schemas use the older renderer-only approach and have no `BlockMetadata`.

### Root Cause

`schema.py` maintains a hardcoded dict mapping block type strings to module paths:

```python
metadata_registry = {
    "verification.coverage": "supekku.scripts.lib.blocks.verification_metadata",
    "delta.relationships": "supekku.scripts.lib.blocks.delta_metadata",
    # ... 4 more entries
}
```

This dict is duplicated in both `_render_json_schema()` and `_render_yaml_example()`. Adding a new metadata-backed block requires editing this dict — and nobody did for the workflow schemas.

Additionally, the dict lookup uses a `getattr(module, f"{block_type.upper().replace('.', '_')}_METADATA")` name-munging convention that is structurally broken for hyphenated schema names: `workflow.review-findings` would produce `WORKFLOW.REVIEW-FINDINGS_METADATA` — an invalid Python identifier. Even adding workflow entries to the dict would not work without also fixing the name derivation. This is further evidence that the parallel-dict approach is a dead end.

### Drivers

- **DE-110 / autobahn:DE-001 OQ-002**: autobahn needs to call `spec-driver show schema workflow.* --format=json-schema` to validate its Pydantic models against canonical schemas. Currently returns "not yet available".
- **Completeness**: `show schema` is the self-documentation surface. Gaps erode trust.

### Constraints

- **ADR-007**: Contracts are derived observed corpus. Schema export is a read-only derived view — fits.
- **ADR-009**: Standard registry API convention. `BlockSchema` is a registry record; enriching it with metadata follows the convention of keeping registry records self-describing.
- **POL-001**: Maximise reuse. `BlockMetadata` and `metadata_to_json_schema` already exist — we wire them in, not create parallel infrastructure.
- **POL-002**: Avoid magic strings. Replacing the hardcoded module-path dict eliminates a class of magic strings.
- **mem.fact.autobahn-independence**: spec-driver provides the contract surface; autobahn consumes it. No autobahn awareness needed here.

### Out of Scope

- Pydantic model export (DE-107/DE-109 concern).
- CLI `export` bulk command (autobahn can call `show schema` per-type).
- Workflow schema validation engine wiring (separate from documentation).
- Frontmatter schema gaps (already covered by `FRONTMATTER_METADATA_REGISTRY`).

## 3. Architecture Intent

### Target Outcomes

- Every registered block type (`list schemas`) supports `show schema <type> --format=json-schema` and `--format=yaml-example`.
- No hardcoded metadata lookup dicts in `schema.py`.
- autobahn can call `spec-driver show schema workflow.state --format=json-schema` and receive valid JSON Schema Draft 2020-12.

### Design Principle

**Metadata belongs on the registry record, not in a parallel lookup.**

`BlockSchema` is the canonical registry entry for a block type. It already carries `name`, `marker`, `version`, `renderer`, `description`. Adding `metadata: BlockMetadata | None` makes it self-describing for documentation purposes. The `schema.py` display code just reads `schema.metadata` — no import gymnastics, no dict maintenance.

## 4. Code Impact Summary

### 4.1 `schema_registry.py` — Add `metadata` field

```python
@dataclass
class BlockSchema:
    name: str
    marker: str
    version: int
    renderer: Callable[..., str]
    description: str
    metadata: BlockMetadata | None = None  # NEW — optional, backward-compatible
```

Single-field addition. `None` default means all existing registrations work unchanged.

Import note: `BlockMetadata` is in `supekku.scripts.lib.blocks.metadata.schema`. Use `TYPE_CHECKING` guard to avoid circular imports if needed (metadata modules import from schema_registry).

### 4.2 `schema.py` — Replace hardcoded dict with `schema.metadata`

**Before** (`_render_json_schema`):
```python
metadata_registry = {
    "verification.coverage": "supekku.scripts.lib.blocks.verification_metadata",
    "delta.relationships": "supekku.scripts.lib.blocks.delta_metadata",
    ...
}
if block_type not in metadata_registry:
    console.print("JSON Schema not yet available...")
    return
module = importlib.import_module(metadata_registry[block_type])
metadata = getattr(module, f"{block_type.upper().replace('.', '_')}_METADATA")
```

**After**:
```python
if not schema.metadata:
    console.print(f"[yellow]JSON Schema not yet available for {block_type}[/yellow]")
    return
json_schema = metadata_to_json_schema(schema.metadata)
```

Same change in `_render_yaml_example`. Removes both hardcoded dicts, the `importlib` dance, and the `getattr` name-munging convention.

    "delta.relationships",

## 7. Design Decisions & Trade-offs

### DEC-110-001 — Attach metadata to BlockSchema

**Decision**: Add `metadata: BlockMetadata | None = None` to `BlockSchema` dataclass.

**Rationale**: Eliminates the hardcoded `metadata_registry` dict in `schema.py` (duplicated in two functions). Makes the registry self-describing. Backward-compatible — existing registrations default to `None`.

**Alternative rejected**: Keep separate lookup dict, just add workflow entries. Rejected because the dict is a maintenance trap (missed for 7 schemas already) and involves fragile import-by-name conventions.

**Consequence**: Registration sites must import their metadata objects. This creates a module-level import from `*_metadata.py` into the registering module (e.g., `delta.py` imports from `delta_metadata.py`). These are sibling modules in the same package — no coupling concern.

**Asymmetry note**: Two registration patterns will coexist after this change:
- Domain modules (`delta.py`, `plan.py`, `relationships.py`, `verification.py`, `revision.py`) import from sibling `*_metadata.py` files and pass `metadata=` at their own registration calls.
- `workflow_metadata.py` defines metadata AND registers in the same file — it passes `metadata=` directly in its own loop.

This asymmetry is acceptable. The workflow module co-locates metadata and registration because the schemas are closely related (7 schemas, shared helpers, shared enum constants). Domain modules separate them because their metadata files were added incrementally. A future unification delta (DE-118) may normalise this, but it is not a problem to solve here.

### DEC-110-002 — New `spec_metadata.py` for spec block metadata

**Decision**: Create a new `spec_metadata.py` file following the established `*_metadata.py` convention.

**Rationale**: `relationships.py` is already 300+ lines handling parsing, validation, and rendering. Adding ~120 lines of metadata declarations would dilute its focus. Separate metadata files are the established pattern (6 existing files follow it).

**Alternative rejected**: Inline in `relationships.py`. Rejected for consistency and file-length reasons.

### DEC-110-003 — Author workflow examples from real files

**Decision**: Base workflow examples on real YAML from `.spec-driver/deltas/*/workflow/` where available, author from schema structure where not.

**Rationale**: Real data ensures examples are actually valid. Authoring from schema structure is necessary for schemas without real instances yet (review-index, review-findings, sessions).

## 8. Open Questions

None — all resolved during adversarial review.

### Adversarial Review Findings (resolved)

1. **Circular import risk**: `schema_registry.py` importing `BlockMetadata` — confirmed safe. `metadata/schema.py` has no dependency on `schema_registry`. Direct import works at runtime. `TYPE_CHECKING` guard is optional but preferred to keep `schema_registry` lightweight.

2. **`spec.capabilities` metadata vs existing imperative validator**: `RelationshipsBlockValidator` validates `spec.relationships` imperatively. New `BlockMetadata` provides schema documentation only — not a replacement for runtime validation. Consistent with how workflow schemas work (metadata for docs, separate validation). No conflict.

3. **DE-110 scope vs DR scope**: DE originally scoped to "workflow schema export with fixtures". Revised during design to "complete schema show coverage for all 15 block types via existing CLI". DE-110.md updated to match.

4. **Renderer fallback path**: After this change, all 15 types will have metadata. The `schema.metadata is None` fallback path in `schema.py` still works for any future block type registered without metadata. No dead code concern — it's a safety net.

5. **STD-002 lint compliance**: New `spec_metadata.py` is pure data declarations — low lint risk. Must pass ruff + pylint as part of verification.

## 9. Rollout & Operational Notes

- **Migration**: None. Additive change — new `metadata` field defaults to `None`.
- **Backward compatibility**: `schema.py` fallback path for `metadata is None` produces same "not yet available" message as today. No regression for any block type.
- **Recovery**: Revert single commit.
