## 1. Summary & Context

Follow-on from DE-107 (Pydantic model layer migration). DE-107 converted all mechanical @dataclass → BaseModel models but deferred three items that need separate design thought.

## 2. Motivation

DE-107 successfully converted 16 models across 7 modules. Three items were deferred:

1. **Spec model** (`specs/models.py`): structurally different — `@dataclass(frozen=True)` wrapping `FrontmatterValidationResult`, with `@property` accessors into `frontmatter.data`. Not the from_frontmatter/to_dict boilerplate pattern. Needs design decision: keep wrapper or extract fields.
2. **Frontmatter metadata updates**: `plan.py` frontmatter metadata needs `plan` and `delta` fields (DE-106 gap noted in DR-107 §4).
3. **Schema docs**: update `list schemas frontmatter` output to reflect Pydantic models.

## 3. Scope & Objectives

- **Primary Outcomes**:
  - Spec model converted to Pydantic (or design decision to keep as dataclass with rationale)
  - plan.py frontmatter metadata updated with plan/delta fields
  - Schema documentation reflects current model state
- **Dependencies**: DE-107 complete

## 4. Out of Scope

- Kind-aware frontmatter validation extension (IMPR-024 subsumed by DE-107, partially addressed)
- Further model layer improvements beyond what DE-107 established

## 5. Risks & Mitigations

- **Risk**: Spec model conversion may not yield net benefit given its wrapper pattern — _Likelihood_: medium — _Mitigation_: DR should evaluate keep-as-dataclass as a valid outcome
