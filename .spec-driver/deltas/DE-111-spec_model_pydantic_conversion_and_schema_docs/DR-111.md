# DR-111 – Spec model Pydantic conversion and schema docs

## 1. Executive Summary

- **Delta**: [DE-111](./DE-111.md)
- **Status**: approved
- **Synopsis**: Resolve three deferred items from DE-107. Spec model stays as dataclass (conscious decision). Plan metadata gains plan/delta fields. Schema docs update automatically.

## 2. Problem & Constraints

### Deferred from DE-107

1. **Spec model**: only remaining `@dataclass` in the model layer. Structurally different from the 16 models DE-107 converted.
2. **plan.py metadata**: missing `plan` and `delta` fields that phase frontmatter now carries (DE-106 PhaseSheet).
3. **Schema docs**: `list schemas frontmatter` should reflect current metadata.

### Constraints

- POL-001: maximise reuse, minimise sprawl — don't convert for consistency's sake if it adds no value
- No breaking changes to Spec consumers

## 3. Design Decisions

### DEC-111-001: Keep Spec as @dataclass(frozen=True)

`Spec` is a **facade** over `FrontmatterValidationResult`, not a frontmatter-to-fields mapping:

- Receives an already-validated `FrontmatterValidationResult` (not a raw dict)
- 12 `@property` accessors with defensive coercion (Iterable checks, Mapping filtering)
- 2 external consumers access `.frontmatter` directly (`spec_formatters.py`, `requirements/registry.py`)
- `extra="ignore"` adds nothing — there's no raw frontmatter dict to absorb unknown fields from
- Pydantic field declarations would duplicate data already structured by `FrontmatterValidationResult`

**Decision**: leave as `@dataclass(frozen=True)`. This is a conscious design choice, not an oversight.

### DEC-111-002: Add plan and delta fields to plan.py metadata

Phase frontmatter now carries `plan: IP-XXX` and `delta: DE-XXX` (written by `create_phase` since DE-106). The metadata schema doesn't declare them, so `list schemas frontmatter` doesn't show them and validation can't check them.

Add two optional string fields and update the phase example.

### DEC-111-003: Schema docs collapse into DEC-111-002

`list schemas frontmatter` auto-discovers from `frontmatter_metadata/` modules. Once plan.py is updated, schema output updates automatically. No separate work needed.
