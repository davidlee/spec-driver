# DR-112 – Kind-aware Pydantic validation and remaining dataclass cleanup

## 1. Executive Summary

- **Delta**: [DE-112](./DE-112.md)
- **Status**: approved
- **Synopsis**: Wire existing Pydantic models into the validator for kind-aware frontmatter checking. Convert 2 remaining core dataclasses. Fix historical validation noise.

## 2. Problem & Constraints

### Current state

- Validator does Pydantic validation for phases only (PhaseSheet). Memory, backlog, and drift files are not validated via their Pydantic models despite having them since DE-107.
- `FrontmatterValidationResult` and `Relation` in `frontmatter_schema.py` are the last remaining frozen dataclasses in the model layer.
- `spec-driver validate` emits 11 errors (AUD-012 status vocabulary mismatch) and ~14 warnings (legacy phase files missing kind/status) that obscure real issues.

### Target state

- Validator catches malformed memory, backlog, and drift frontmatter via the Pydantic models DE-107 created.
- Validation functions are method-scoped for batch use now, extractable to shared helpers for future per-invocation hooks.
- `spec-driver validate` output reflects real issues, not historical debt.

### Constraints

- POL-001: reuse existing Pydantic models, don't create parallel validation
- Validation is permissive (warnings, not errors) — consistent with existing PhaseSheet pattern
- Registry behaviour unchanged — registries stay permissive, validator is opt-in

## 3. Design

### Item 1: Convert FrontmatterValidationResult and Relation

Mechanical conversion, same as DE-107 Phase 3:

```python
# Before
@dataclass(frozen=True)
class Relation:
    type: str
    target: str
    attributes: Mapping[str, Any] = field(default_factory=dict)

# After
class Relation(BaseModel, frozen=True):
    type: str
    target: str
    attributes: Mapping[str, Any] = {}
```

Same for `FrontmatterValidationResult`. The `validate_frontmatter()` function constructs it directly — trivial call site update.

**Gotcha**: `Relation.type` shadows `type` builtin. Pydantic handles this fine but warrants a note.

**Gotcha**: `FrontmatterValidationResult.dict()` collides with Pydantic's `.dict()` (deprecated v2 wrapper for `.model_dump()`). The custom method returns `self.data`; Pydantic's version dumps model fields. Verified zero callers exist — the method is dead code. **Drop it during conversion.**

**Gotcha**: `FrontmatterValidationResult.data` and `Relation.attributes` receive `MappingProxyType` instances. Pydantic rejects `MappingProxyType` without `arbitrary_types_allowed=True`. Since `frozen=True` already prevents reference mutation, the `MappingProxyType` wrapping is redundant. **Simplify both to `dict[str, Any]`** — change `validate_frontmatter()` to pass a plain `dict` for `data`, and `_normalize_relations()` to pass `dict(extras)` instead of `MappingProxyType(dict(extras))` for `attributes`.

### Item 2: Kind-aware validator methods

Add three methods to `WorkspaceValidator`, parallel to existing `_validate_phase_frontmatter`:

```python
def _validate_memory_frontmatter(self, fm: dict, artifact: str) -> None:
    """Validate memory frontmatter via MemoryRecord model."""
    try:
        MemoryRecord(**fm)
    except ValidationError as exc:
        self._warning(artifact, f"Memory frontmatter validation: {exc.error_count()} error(s)")

def _validate_backlog_frontmatter(self, fm: dict, artifact: str) -> None:
    """Validate backlog item frontmatter via BacklogItem model."""
    try:
        BacklogItem(**fm)
    except ValidationError as exc:
        self._warning(artifact, f"Backlog frontmatter validation: {exc.error_count()} error(s)")

def _validate_drift_frontmatter(self, fm: dict, artifact: str) -> None:
    """Validate drift ledger frontmatter via DriftLedger model."""
    try:
        DriftLedger(**fm)
    except ValidationError as exc:
        self._warning(artifact, f"Drift frontmatter validation: {exc.error_count()} error(s)")
```

Each is wired into a new top-level traversal method:

```python
def _validate_memory_files(self) -> None:
    """Validate all memory file frontmatter."""
    memory_dir = get_memory_dir(self.workspace.root)
    if not memory_dir.exists():
        return
    for mem_file in sorted(memory_dir.glob("mem.*.md")):
        fm, _ = load_markdown_file(mem_file)
        if fm:
            self._validate_memory_frontmatter(fm, fm.get("id", mem_file.name))
```

Similar for backlog (glob per-kind subdirectory) and drift.

Called from `validate()` alongside existing checks.

### Item 3: Historical noise cleanup

**AUD-012**: 11 findings have `disposition.status: resolved` but `disposition.kind: aligned`. The vocabulary requires `reconciled` for `aligned` findings. Fix: sed/edit the YAML blocks in AUD-012.md.

**Legacy phases**: ~12 phase files from DE-002, DE-004, DE-028, DE-031, DE-051 lack `kind: phase` and/or `status:` in frontmatter. Fix: add missing fields to frontmatter. These are completed/draft phases in closed deltas — safe to backfill.

**IP-109 phases**: `status: in_progress` (underscore) instead of `in-progress` (hyphen). Fix if `--fix` flag handles it, otherwise manual.
