## 1. Summary & Context

- **Technical Spec(s)**: SPEC-110 (supekku/cli), SPEC-122 (requirements), SPEC-114 (blocks/metadata)
- **Change Drivers**: Code quality audit. Five files exceed 1000 lines with low cohesion.
- **Governance**: POL-001 (minimise sprawl)

## 2. Motivation

Five files have grown into god modules with multiple unrelated responsibilities:

| File                          | Lines | Concerns                                                                                              |
| ----------------------------- | ----- | ----------------------------------------------------------------------------------------------------- |
| `cli/list.py`                 | 3171  | 19 CLI commands across 6 domains                                                                      |
| `cli/common.py`               | 1065  | Exit codes, artifact resolution, ID normalization, pager/editor utils                                 |
| `requirements/registry.py`    | 1511  | Registry I/O, sync orchestration, content parsing, relation application, coverage computation, models |
| `changes/creation.py`         | 1056  | 5 unrelated creation domains sharing 15 lines of utility                                              |
| `blocks/workflow_metadata.py` | 1299  | 7 unrelated schema definitions                                                                        |

These files are difficult to navigate, review, and maintain. Splitting them is a prerequisite for focused work on each subsystem.

## 3. Scope & Objectives

- **Primary Outcomes**:
  - **Phase 1 – CLI layer**:
    - `list.py` → `list/` package (`__init__.py` + `specs.py`, `deltas.py`, `changes.py`, `reviews.py`, `governance.py`, `requirements.py`, `backlog.py`, `backlog_items.py`, `misc.py`)
    - `common.py` → slim `common.py` (exit codes, options), `cli/artifacts.py` (resolve/find dispatch), `cli/ids.py` (normalization), `cli/io.py` (pager/editor)
  - **Phase 2 – Lib layer**:
    - `requirements/registry.py` → `models.py`, `parser.py`, `sync.py`, `relations.py`, `coverage.py`, `registry.py` (slim)
    - `changes/creation.py` → `delta_creation.py`, `revision_creation.py`, `audit_creation.py`, `phase_creation.py`, `_creation_utils.py`
    - `workflow_metadata.py` → per-schema files mirroring existing one-file-per-schema pattern
- **Operational Constraints**: Pure mechanical extraction. Zero logic changes. All tests pass unchanged.
- **Dependencies**: DE-113 (dead code purge shrinks the surface first)

## 4. Out of Scope

- Refactoring logic within the split files (that's DE-115 through DE-122)
- Changing public APIs or function signatures
- Adding new tests (existing tests verify behaviour; import paths update mechanically)

## 5. Approach Overview

- **Key Changes**:
  - Move functions/classes to new files
  - Update `__init__.py` re-exports where needed for backward compatibility
  - Update all import sites
  - Verify with `just check`
- **Migration Notes**: Typer app registration for split `list_*.py` files requires wiring sub-apps into `main.py`. The existing pattern (adding commands to a shared `app` object) makes this straightforward.

## 6. Verification Strategy

- **Acceptance Criteria**:
  - `just check` passes
  - No file in the affected set exceeds 500 lines (target: each file has one cohesive responsibility)
  - `git diff --stat` shows only renames/moves plus import updates
  - No new public API surfaces — only re-organization

## 7. Risks & Mitigations

- **Risk**: Typer command registration breaks after list.py split – _Likelihood_: low – _Impact_: medium – _Mitigation_: CLI integration tests cover all 19 list commands.
- **Risk**: Circular imports after extraction – _Likelihood_: medium – _Impact_: low – _Mitigation_: The current deferred-import pattern already handles this; splits reduce rather than increase circular risk.

- **Dependency of**: DE-115 (utilities extracted from now-smaller files), DE-116 (registry work on cleaner registry.py), DE-117 (CLI boilerplate in smaller files)
- **Sequence**: Sprint 2 of quality remediation.
