# DR-114 – God-file structural splits

## 1. Executive Summary

- **Delta**: [DE-114](./DE-114.md)
- **Status**: draft
- **Last Updated**: 2026-03-22
- **Synopsis**: Split 5 god files (8,371 lines total) into ~30 focused modules. Pure structural extraction — zero behaviour change. `RequirementsRegistry` sync internals extracted to standalone functions receiving the mutable `self.records` dict by reference (mutation in place, identical semantics).

## 2. Problem & Constraints

- **Current Behaviour**: Five files exceed 1,000 lines with low cohesion. `list.py` has 19 commands across 6 domains. `common.py` mixes exit codes, artifact resolution, ID normalization, and editor/pager utils. `RequirementsRegistry` is a 1,300-line class doing I/O, sync, parsing, relation application, and coverage computation. `creation.py` has 5 unrelated creation domains. `workflow_metadata.py` has 7 unrelated schema definitions.
- **Drivers**: POL-001 (minimise sprawl), code quality audit
- **Constraints**: Zero behaviour change. All tests pass. No new public APIs.
- **Out of Scope**: Logic refactoring within split files (DE-115+). Adding new tests beyond import path updates.

## 3. Architecture Intent

- **Target Outcomes**:
  - No file in the affected set exceeds ~500 lines
  - Each file has one cohesive responsibility (target: ≤500 lines)
  - Existing import paths work via re-exports during transition
- **Guiding Principles**: Mechanical extraction. Re-export for backward compat. Test files mirror source splits.

## 4. Code Impact Summary

### 4a. `cli/list.py` (3,195 lines → package)

**Decision DEC-114-01**: Convert `cli/list.py` to `cli/list/` package. `__init__.py` creates the aggregate `app` and adds sub-apps.

| New File | Commands | ~Lines |
|----------|----------|--------|
| `list/__init__.py` | aggregate `app`, shared imports, `_parse_relation_filter` | ~100 |
| `list/specs.py` | `list specs` | ~340 |
| `list/deltas.py` | `list deltas` | ~340 |
| `list/changes.py` | `list changes`, `list plans` | ~350 |
| `list/reviews.py` | `list revisions`, `list audits` | ~320 |
| `list/governance.py` | `list adrs`, `list policies`, `list standards` | ~430 |
| `list/requirements.py` | `list requirements` | ~310 |
| `list/backlog.py` | `list backlog` | ~270 |
| `list/backlog_items.py` | `list drift`, `list issues`, `list problems`, `list improvements`, `list risks` | ~540 |
| `list/misc.py` | `list cards`, `list memories`, `list schemas` | ~340 |

**Wiring pattern**: Each sub-module defines its own `app = typer.Typer()` with commands registered via `@app.command()`. The `__init__.py` imports each sub-app and registers commands onto the aggregate `app` that `main.py` mounts.

Actually — simpler: each sub-module just defines bare functions with `@app.command()` using the **shared `app`** from `__init__.py`. Each sub-module imports `app` from the package and registers onto it. This matches the current pattern exactly and avoids sub-app aggregation complexity.

```python
# list/__init__.py
app = typer.Typer(help="List artifacts", no_args_is_help=True)

# Import sub-modules to trigger @app.command() registration
from supekku.cli.list import specs as _specs  # noqa: F401
from supekku.cli.list import changes as _changes  # noqa: F401
# ... etc
```

```python
# list/specs.py
from supekku.cli.list import app

@app.command("specs")
def list_specs(...):
    ...
```

**Test split**: `list_test.py` (2,247 lines) stays as-is for now — test classes don't map 1:1 to source files (e.g. `ListBacklogShortcutsTest` tests multiple commands). Splitting tests is optional follow-up.

**Import compatibility**: `from supekku.cli.list import app` continues to work because `list/` is now a package with `app` in `__init__.py`. The 3 test files that import this need zero changes.

### 4c. `requirements/registry.py` (1,511 lines → 5 files)

**Decision DEC-114-03**: Extract sync internals and parser methods to standalone functions. The public field is `self.records` (public, part of ADR-009 read surface — `collect()`, `find()`, `iter()` all use it directly). Extracted functions receive the mutable `records` dict **by reference** and mutate it in place, exactly as the current methods do. No copy, no return-value threading, no semantic change.

**DE-116 alignment**: DE-116 will define `RegistryProtocol[T]` and expects `RequirementsRegistry` to conform. The public read surface (`find`, `collect`, `iter`, `filter`) and mutation surface (`save`, `set_status`, `search`, `move_requirement`, `find_by_*`) stay as class methods. What we extract is the sync internals — private complexity that DE-116 explicitly excludes. Extracting `RequirementRecord` to `models.py` helps DE-116 by making `T` importable without the full registry. The split is neutral for protocol work (it neither helps nor hinders DE-116's protocol enforcement beyond the `models.py` extraction).

**Mutation semantics**: Extracted functions receive `records: dict[str, RequirementRecord]` by reference and mutate in place — identical to current `self.records` access. No copies created. `sync()` stays as a registry method that imports and calls the standalone functions:

```python
# registry.py (slim) — top-level imports (graph is acyclic, no need for deferred)
from .sync import apply_delta_relations, apply_revision_blocks, ...
from .parser import records_from_frontmatter, records_from_content

class RequirementsRegistry:
    def sync(self, ...):
        # delegates, passing self.records, self.registry_path.parent, etc.
        apply_delta_relations(self.records, ...)
        apply_revision_blocks(self.records, ..., spec_registry=self._specs)
        ...
```

## 7. Design Decisions

- **DEC-114-01**: `list.py` → `list/` package. Sub-modules register commands on shared `app` imported from `__init__.py`. Tests unchanged.
- **DEC-114-02**: `common.py` re-exports all public symbols from new files. Zero importer changes in this delta. **Follow-up**: backlog issue to remove re-exports and update importers after DE-114 lands (applies to all 4 re-export shims: `common.py`, `creation.py`, `registry.py`, `workflow_metadata.py`). Without a removal ticket these calcify per POL-001 competing-systems clause.
- **DEC-114-03**: `RequirementsRegistry` sync internals (all `_apply_*`, parser, coverage methods) extracted to standalone functions receiving the mutable `records` dict by reference — mutation in place, identical to current `self.records` access. Signatures change (explicit params instead of `self`); behaviour doesn't.
- **DEC-114-04**: Test files split to mirror source splits where the test classes map cleanly to the new source files. `list_test.py` stays unified because its test classes span multiple command groups and splitting would require refactoring test fixtures. `registry_test.py` splits because its 17 test classes do map to distinct concerns (models, parsing, sync, coverage, queries). `creation_test.py` gets a partial split (phase tests only). This is pragmatic, not principled — splitting tests that don't map cleanly is a separate task.
- **DEC-114-05**: `workflow_metadata.py` re-exports all symbols for backward compat.

## 8. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Circular imports after `common.py` split | Low | Low | Existing deferred-import pattern handles this; splits reduce circular risk |
| `list/` package conversion breaks `main.py` wiring | Low | Medium | `main.py` imports `list.app` — works identically with package `__init__.py` |
| `RequirementsRegistry` extraction introduces subtle bugs | Medium | Medium | Tests are comprehensive (2,787 lines); pure function signatures make dependencies explicit |
| `creation.py` phase helpers have hidden coupling | Low | Low | Phase helpers form a clear cluster with no cross-domain deps |

## 9. Follow-ups

- **Backlog issue**: Remove 4 re-export shims (`common.py`, `creation.py`, `registry.py`, `workflow_metadata.py`) and update all importers to use canonical paths. Without this, POL-001 competing-systems clause applies.
- **Optional**: Split `list_test.py` (2,247 lines) to mirror `list/` package structure.
