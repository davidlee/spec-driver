## 1. Summary & Context

- **Technical Spec(s)**: SPEC-117 (decisions), SPEC-126 (policies), SPEC-127 (standards)
- **Change Drivers**: Code quality audit identified 8 reimplementations of sequential ID generation, 3 identical `parse_date()` copies, 3 identical title extraction blocks, 3 inlined atomic write patterns, 6 duplicated code-fence regex patterns, 2 identical `_matches_pattern` functions.
- **Governance**: POL-001 (maximise code reuse), ADR-009 (standard registry convention)

## 2. Motivation

Multiple utility functions are copy-pasted across domain modules with zero or trivial variation. This violates POL-001 and creates maintenance traps where a fix to one copy doesn't propagate. The duplications are:

| Utility                  | Copies                                                                                                                                                                                            | Variation                                |
| ------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------- |
| `parse_date()`           | 3 (decisions, policies, standards registries)                                                                                                                                                     | Byte-for-byte identical                  |
| Sequential ID generation | 8 (`_next_identifier`, `determine_next_identifier`, `generate_next_adr_id`, `generate_next_policy_id`, `generate_next_standard_id`, `_next_ledger_id`, `next_identifier`, `CardRegistry.next_id`) | Same algorithm, different prefix strings |
| Title extraction from H1 | 3 (decisions, policies, standards registries)                                                                                                                                                     | Same logic, different prefix             |
| Atomic file write        | 3 (`review_io.py` extracted, `state_io.py` + `handoff_io.py` inlined)                                                                                                                             | Identical pattern                        |
| Code-fence block regex   | 6 (delta.py, relationships.py, verification.py, plan.py ×3)                                                                                                                                       | Same regex template, different marker    |
| `_matches_pattern`       | 2 (common.py, find.py)                                                                                                                                                                            | Byte-for-byte identical                  |
| `create_title_slug`      | 2 (policies, standards creation)                                                                                                                                                                  | 1-line passthrough to `slugify()`        |

## 3. Scope & Objectives

- **Primary Outcomes**:
  - New `core/dates.py`: shared `parse_date(value) -> date | None`
  - New `core/ids.py`: shared `next_sequential_id(names, prefix, separator="-") -> str`
  - Extend `core/spec_utils.py`: shared `extract_title_from_content(content, prefix) -> str`
  - New `core/io.py`: shared `atomic_write(path, content)` (promoted from `review_io._atomic_write`)
  - Extend `blocks/yaml_utils.py`: `make_block_pattern(marker) -> re.Pattern`
  - Consolidate `_matches_pattern` into one location
  - Delete `create_title_slug()` wrappers; callers use `slugify()` directly
- **Operational Constraints**: Pure extract-and-delegate. Each extraction replaces identical code. All callers updated to use the shared utility.
- **Dependencies**: DE-114 (splits make the files smaller and easier to work in)

## 4. Out of Scope

- Changing the behaviour of any extracted utility
- Registry protocol or base class work (DE-116)
- Unifying creation flows (DE-116)

## 5. Approach Overview

- For each utility: create/extend the target module, write tests, replace all call sites, delete the originals.
- Order: `parse_date` first (smallest, most mechanical), then `next_sequential_id` (most copies), then the rest.

## 6. Verification Strategy

- **Acceptance Criteria**:
  - `just check` passes
  - `grep -rn 'def parse_date' supekku/` returns exactly 1 result
  - `grep -rn 'def.*next.*identifier\|def.*generate_next.*id\|def.*next_ledger_id' supekku/` returns exactly 1 result (the shared one)
  - No `_matches_pattern` duplication
  - No `create_title_slug` wrappers remain

## 7. Risks & Mitigations

- **Risk**: Subtle behavioural differences between "identical" copies – _Likelihood_: low – _Impact_: low – _Mitigation_: Diff each pair before consolidation; existing tests catch regressions.
- **Risk**: Some ID generators use registry.collect() while others scan directories – _Likelihood_: medium – _Impact_: low – _Mitigation_: The shared function parameterizes the source (directory scan vs collection iteration).

## 8. Follow-ups & Tracking
