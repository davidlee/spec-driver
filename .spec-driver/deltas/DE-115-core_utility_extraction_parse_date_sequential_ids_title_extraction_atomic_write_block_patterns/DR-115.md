# DR-115 – Core utility extraction: parse_date, sequential IDs, title extraction, atomic write, block patterns

## 1. Executive Summary

- **Delta**: [DE-115](./DE-115.md)
- **Status**: draft
- **Synopsis**: Pure extract-and-delegate refactor — 8 sequential ID generators, 3 `parse_date` copies, 3 `create_title_slug` wrappers, 6 block-pattern definitions, 2 `_matches_pattern` copies, and 3 atomic-write implementations (1 named + 2 inlined) promoted to canonical `core/` and `blocks/` homes. No behaviour changes; all callers updated to delegate.

## 2. Problem & Constraints

- **Current Behaviour**: Identical or near-identical utility logic is copy-pasted across domain modules. A fix or improvement to any copy requires hunting down all duplicates manually.
  - `parse_date()` — 3 byte-for-byte identical instance methods in `decisions/registry.py`, `policies/registry.py`, `standards/registry.py`
  - Sequential ID generation — 8 variants (`_next_identifier`, `determine_next_identifier`, `_next_ledger_id`, `generate_next_adr_id`, `generate_next_policy_id`, `generate_next_standard_id`, `next_identifier`, `CardRegistry.next_id`) all implement the same directory-scan-for-highest-integer algorithm
  - `create_title_slug` — 3 one-line passthrough wrappers around `slugify()` in decisions, policies, standards creation modules
  - Block-pattern compilation — 6 identical `re.compile(r"```(?:yaml|yml)\s+" + re.escape(MARKER) + r"\n(.*?)```", re.DOTALL)` expressions in `blocks/delta.py`, `blocks/relationships.py`, `blocks/verification.py`, `blocks/plan.py` (×3)
  - `_matches_pattern` — 2 byte-for-byte identical functions in `cli/artifacts.py` and `cli/find.py`
  - `_atomic_write` — named private function in `workflow/review_io.py`; identical fd/mkstemp/replace pattern inlined in `state_io.py:write_state` and `handoff_io.py:write_handoff` (3 total)
- **Drivers**: POL-001 (maximise code reuse, minimise sprawl), ADR-009 (standard registry API convention)
- **Constraints**:
  - Pure extract-and-delegate; no behaviour changes to any extracted function
  - All callers must be updated; no orphaned originals
  - `just check` must pass at completion of each extraction batch
- **Out of Scope**: Registry protocol or base-class unification (DE-116), creation-flow consolidation, changing any utility's behaviour

## 3. Architecture Intent

- **Target Outcomes**:
  - `core/dates.py`: single `parse_date(value: Any) -> date | None` function; all three registry instance methods delegate or are removed
  - `core/ids.py`: single `next_sequential_id(names: Iterable[str], prefix: str, separator: str = "-") -> str` function; all 8 ID generators delegate or are deleted
  - `core/io.py`: `atomic_write(path: Path, content: str) -> Path` promoted from `review_io._atomic_write`; all callers updated
  - `core/spec_utils.py`: `extract_h1_title(content: str, prefix: str = "") -> str` added; inline H1-extraction blocks in domain registries delegate to it
  - `blocks/yaml_utils.py`: `make_block_pattern(marker: str) -> re.Pattern` added; all 6 block-pattern `re.compile` calls replaced
  - `core/filters.py`: `matches_pattern(artifact_id: str, pattern: str) -> bool` added (existing module, consistent with its purpose); `cli/artifacts.py` and `cli/find.py` delete their private copies
  - `create_title_slug` wrappers deleted; their call sites use `slugify()` directly
- **Guiding Principles**:
  - Place utilities in the module that best matches their responsibility: date logic → `core/dates.py`, ID logic → `core/ids.py`, file I/O → `core/io.py`, pattern matching → `core/filters.py`, markdown helpers → `core/spec_utils.py`, block parsing helpers → `blocks/yaml_utils.py`
  - Each extraction is a separate, independently checkable commit
  - No new abstractions — just canonical homes for what already exists

## 7. Design Decisions & Trade-offs

- **`next_sequential_id` signature**: Use `(names: Iterable[str], prefix: str, separator: str = "-") -> str`. All 8 variants share the same core algorithm (scan → regex-extract numeric suffix → max+1 → format `:03d`). The variations are input source (directory entries, registry keys, card objects) and separator (`-` for most, none for cards). Callers convert their collection to `Iterable[str]` themselves — one line per site. The `separator` param handles cards (`separator=""`) without a special case. Regex pattern: `rf"{re.escape(prefix)}{re.escape(separator)}(\d+)"`. Note: `specs/creation.py:determine_next_identifier` uses a looser regex that doesn't check prefix — tighten it to match the shared implementation; existing tests will catch any regression.
- **`parse_date` as module function, not method**: Extract as a module-level pure function in `core/dates.py`. The three registries currently define it as an instance method only because there was no better home. No registry state is used.
- **`matches_pattern` placement**: `core/filters.py` already exists for pure CLI filter utilities and is imported by CLI modules. Adding `matches_pattern` there is coherent with its existing purpose.
- **`extract_h1_title` vs `extract_title`**: The domain registries use a narrower H1 extraction (looking for `# PREFIX-` lines). `backlog/registry.py:extract_title` also checks frontmatter `name`. These are distinct; only the H1-extraction block is extracted and shared. Do not conflate.
- **`create_title_slug` deletion**: Wrappers are one-liners with zero logic. Callers that import `create_title_slug` should import `slugify` directly. No new utility is created.
