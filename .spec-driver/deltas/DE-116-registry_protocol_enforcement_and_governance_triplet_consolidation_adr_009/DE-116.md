## 1. Summary & Context

- **Technical Spec(s)**: SPEC-117 (decisions), SPEC-126 (policies), SPEC-127 (standards)
- **Change Drivers**: ADR-009 (standard registry API convention) mandates a canonical surface but enforcement is purely documentary. The Decision/Policy/Standard registries are structural triplets with 80%+ code duplication.
- **Governance**: ADR-009, POL-001

## 2. Motivation

ADR-009 established the canonical registry API (`find/collect/iter/filter`) but deferred Protocol/ABC standardisation. The result: 10 registries independently implement the surface with drift in keyword conventions, method availability, and semantics. The three governance registries (Decision, Policy, Standard) are the worst case — `parse_date()`, `collect()`, `_parse_X_file()`, `write()`, `iter()`, `find()`, `to_dict()`, and creation flows are near-identical, differing only in prefix strings and glob patterns.

Specific drift:

- `iter(status=...)` is keyword-only in some registries, positional in others
- `DriftLedgerRegistry` omits `filter()` entirely
- `CardRegistry` uses `lane` not `status`
- `RequirementsRegistry.sync()` has completely different semantics from `DecisionRegistry.sync()`
- Constructor conventions vary: keyword-only `root` vs positional vs required

> **Scope revised 2026-06-01** (see [DR-116](./DR-116.md)). The original card predated the
> `spec_driver` layered-migration program (DE-125/127/128). Building the base in legacy `core/`
> with registry-owned backlinks would replant the misplacement that program is undoing. Per user
> direction ("do it properly"), DE-116 now also **migrates the records + registries into
> `spec_driver.domain`** and **hoists backlink composition to `Workspace`**. The original §2
> drift analysis is retained as historical motivation.

## 3. Scope & Objectives

- **Primary Outcomes**:
  - Define `RegistryProtocol[T]` (ADR-009 read surface: `find/collect/iter/filter`) in
    `spec_driver/domain/registries/frontmatter.py` — typed + `runtime_checkable`.
  - Extract `FrontmatterFileRegistry[T]` ABC (template-method `_parse_file` + abstract
    `_build_record`/`_artifact_dir` + overridable `_resolve_status`); parameterised by prefix,
    YAML root key, record type, backlink declaration.
  - Collapse Decision/Policy/Standard registries onto the base (~330L → ~40L each); delete
    duplicated bodies and redundant `load()` classmethods.
  - Make registries **pure**: remove `_build_backlinks`; **hoist backlink composition to
    `Workspace`** (`build_backlinks_multi`), driven by each registry's declarative
    `backlink_inputs`. Registries import nothing from `domain.relations`.
  - **Migrate** records → `spec_driver/domain/records/`, registries → `spec_driver/domain/
    registries/`; leave re-export shims at legacy paths (37 consumers stay unedited).
  - Unify creation: one `create_governance_artifact(spec, registry, options)` +
    `GovernanceArtifactSpec`/`Options`/`Result`/`ExistsError` replace the 3 triplets.
  - Keep `iter()` **positional-or-keyword** per ADR-009 §1 (corrects the original "keyword-only").
- **Operational Constraints**: External CLI stdout unchanged. Generated registry YAML byte-identical
  (backlink-hoist safety net). `domain.registries` must not import `domain.relations` (layer contract).
- **Dependencies**: DE-115 (helpers — done); DE-125/127/128 (migration program — done).

## 4. Out of Scope

- Updating the 37 downstream consumers to canonical paths + deleting shims (tracked follow-up delta).
- `DriftLedgerRegistry.filter()` addition (separate registry; deferred to a follow-up).
- `RequirementsRegistry` / `CardRegistry` (different domains; separate deltas).
- Migrating `creation` / `Workspace` into `spec_driver.orchestration` (kept legacy this delta).
- Persistence **naming** unification (`sync`/`write`/`save`) — ADR-009 §5 defers this.

## 5. Approach Overview

- **P0 (spike)**: Prove `RegistryProtocol` conformance under `ty` + `isinstance` for the
  `filter`/`iter` signature variance, before migrating. Records the fallback if strict.
- **P1**: Records → `domain/records/`; `RegistryProtocol` + `FrontmatterFileRegistry` →
  `domain/registries/frontmatter.py`; `DecisionRegistry` onto base; legacy shims; green.
- **P2**: `PolicyRegistry` + `StandardRegistry` onto base; hoist backlinks to `Workspace`; golden
  YAML diff; import-linter green.
- **P3**: Unify creation flow + `cli/create.py` (unified Result/Exception).
- **P4**: Verification sweep — `just check`, `uvx import-linter`, `uv run ty check`; file shim-debt
  follow-up.

## 6. Verification Strategy

- **Acceptance Criteria**:
  - `just check` + `uvx import-linter` + `uv run ty check` all green.
  - `spec_driver/domain/registries/**` imports nothing from `domain.relations` (grep = 0).
  - Decision/Policy/Standard registries share one base; **zero duplicated method bodies** — the 3
    `domain/registries/{decision,policy,standard}.py` define only `_build_record`, `_artifact_dir`,
    `filter`, class attrs (+ ADR symlink methods); none redefine `collect/_parse_file/iter/find/
    write/sync` (grep = 0). No `load()` classmethods remain.
  - Golden registry-YAML diff empty before/after, run over a **fixture corpus with known
    cross-references** (the live corpus has zero backlinks — a live diff is vacuous; see DR AR-2).
  - `RegistryProtocol` conformance verified (`ty` + runtime `isinstance`).
  - Legacy shims forward all public names; the 37 consumers pass unedited.

## 7. Follow-ups & Tracking

  - `DriftLedgerRegistry.filter()` addition (registry-surface completeness).
- **Sequence**: Sprint 4 of quality remediation; first registry migration into `spec_driver.domain`.
