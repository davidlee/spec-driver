## 1. Summary & Context

- **Technical Spec(s)**: SPEC-117 (decisions), SPEC-126 (policies), SPEC-127 (standards)
- **Change Drivers**: ADR-009 (standard registry API convention) mandates a canonical surface but enforcement is purely documentary. The Decision/Policy/Standard registries are structural triplets with 80%+ code duplication.
- **Governance**: ADR-009, POL-001

## 2. Motivation

ADR-009 established the canonical registry API (`find/collect/iter/filter`) but deferred Protocol/ABC standardisation. The result: 10 registries independently implement the surface with drift in keyword conventions, method availability, and semantics. The three governance registries (Decision, Policy, Standard) are the worst case — `parse_date()`, `collect()`, `_parse_X_file()`, `write()`, `iter()`, `find()`, `to_dict()`, and creation flows are near-identical, differing only in prefix strings and glob patterns.

Specific drift:
- `iter(status=...)` is keyword-only in some registries, positional in others
- `DriftLedgerRegistry` omits `filter()` entirely
- `CardRegistry` uses `lane` not `status`
- `RequirementsRegistry.sync()` has completely different semantics from `DecisionRegistry.sync()`
- Constructor conventions vary: keyword-only `root` vs positional vs required

## 3. Scope & Objectives

- **Primary Outcomes**:
  - Define `RegistryProtocol[T]` in `core/registry.py` formalising the ADR-009 read surface
  - Extract `FrontmatterFileRegistry` base class parameterised by: glob pattern, YAML root key, record type, ID prefix
  - Collapse Decision/Policy/Standard registries onto the base class
  - Unify `iter()` as keyword-only across all registries
  - Add `filter()` to `DriftLedgerRegistry`
  - Unify creation flow: `create_governance_artifact(prefix, registry, options, template)` replaces 3 near-identical functions
  - Delete redundant `load()` classmethods
- **Operational Constraints**: External API (CLI-visible behaviour) unchanged. Internal API may change (callers updated).
- **Dependencies**: DE-115 (shared `parse_date`, `next_sequential_id`, `extract_title` must exist first)

## 4. Out of Scope

- Refactoring `RequirementsRegistry` internals (addressed by DE-114 split)
- `CardRegistry` lane/status unification (domain-specific, needs separate consideration)
- Persistence naming unification (`sync` vs `write` vs `save`) — ADR-009 explicitly defers this

## 5. Approach Overview

- **Phase 1**: Define `RegistryProtocol[T]` and verify all registries conform (type-check only, no code changes)
- **Phase 2**: Extract `FrontmatterFileRegistry` base from `DecisionRegistry` (the most complete). Parameterise.
- **Phase 3**: Migrate `PolicyRegistry` and `StandardRegistry` onto the base. Delete duplicated methods.
- **Phase 4**: Unify creation flows. Parameterise template and prefix.
- **Phase 5**: Fix `iter()` keyword consistency, add missing `filter()` to `DriftLedgerRegistry`.

## 6. Verification Strategy

- **Acceptance Criteria**:
  - `just check` passes
  - `mypy` (or equivalent) confirms all registries satisfy `RegistryProtocol`
  - Decision/Policy/Standard registries have zero duplicated method bodies
  - No `load()` classmethods remain (where they add no value over constructor)
  - `grep -rn 'def parse_date' supekku/scripts/lib/{decisions,policies,standards}` returns 0 results

## 7. Risks & Mitigations

- **Risk**: Base class becomes a leaky abstraction forcing domain-specific hacks – _Likelihood_: medium – _Impact_: medium – _Mitigation_: Keep base minimal (only the truly shared surface); domain methods remain on subclasses.
- **Risk**: `FrontmatterFileRegistry` couples three registries that may diverge later – _Likelihood_: low – _Impact_: low – _Mitigation_: The triplet has been stable for 100+ deltas; divergence would be visible.
