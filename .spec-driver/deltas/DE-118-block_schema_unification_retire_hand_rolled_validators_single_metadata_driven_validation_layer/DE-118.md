## 1. Summary & Context

- **Technical Spec(s)**: SPEC-114 (blocks/metadata), SPEC-116 (frontmatter_metadata), SPEC-115 (changes/blocks)
- **Change Drivers**: Code quality audit found two parallel validation systems for the same YAML block schemas, active regex bugs in the hand-rolled JSON Schema, and structural gaps in the metadata system.
- **Governance**: POL-001 (maximise code reuse)

## 2. Motivation

Every YAML-block type has both a hand-rolled validator class (~200+ lines each) **and** a declarative `BlockMetadata` instance describing the same schema. The two layers have diverged:

- Hand-rolled validators enforce unknown-key rejection; `MetadataValidator` does not.
- `REVISION_BLOCK_JSON_SCHEMA` has 4 double-escaped regex patterns (`r"^RE-\\d{3,}$"` matches literal backslashes). Active bug.
- The `BlockSchema` registry has no `metadata` field — cannot answer "what is the schema for this block type?"
- `FieldMetadata` cannot model dynamic-key objects; uses `_entry_shape` sentinel hack.
- 7 workflow schemas registered with `_placeholder_renderer` that produces empty strings.
- `DeltaRelationshipsValidator` error messages print dict repr instead of path names.

The `frontmatter_metadata/` subsystem demonstrates the metadata-driven approach works at scale (14 artifact kinds). The blocks subsystem should complete the same transition.

## 3. Scope & Objectives

- **Primary Outcomes**:
  - **Phase 1 – MetadataValidator enhancements**:
    - Add unknown-key rejection to `MetadataValidator` (closes the strictness gap)
    - Add `additional_properties: FieldMetadata | None` to `FieldMetadata` (fixes dynamic-key hack)
    - Add `metadata: BlockMetadata | None` field to `BlockSchema` (connects registry to validation)
  - **Phase 2 – Per-block migration**:
    - Migrate `RevisionBlockValidator` → `MetadataValidator` + `REVISION_CHANGE_METADATA`
    - Migrate `DeltaRelationshipsValidator`, `VerificationCoverageValidator`, `RelationshipsBlockValidator`
    - Migrate plan validators (`PlanOverviewValidator`, `PhaseOverviewValidator`, `PhaseTrackingValidator`)
  - **Phase 3 – Cleanup**:
    - Delete `REVISION_BLOCK_JSON_SCHEMA` (and its 4 regex bugs)
    - Delete or implement `_placeholder_renderer` registrations
    - Fix `RELATIONSHIPS_MARKER` / `VALID_STATUSES` naming collisions
    - Remove `FieldMetadata.required` on `items` definitions (semantically meaningless)
- **Operational Constraints**: Validation behaviour must be equivalent or stricter post-migration. No silent relaxation of constraints.
- **Dependencies**: None strict, but benefits from DE-114 (workflow_metadata.py split) and DE-115 (shared block-pattern factory).

## 4. Out of Scope

- Changing what is validated (only changing how validation is implemented)
- Adding new block types
- Frontmatter metadata changes (that system is already on the right path)

## 5. Approach Overview

- For each validator: write parallel tests against both old and new validation, confirm identical results, then swap the implementation.
- The `revision.py` extractor (bespoke line-by-line state machine) is retained — it handles nested fences correctly, unlike the regex extractors. Migration is about validation, not extraction.

## 6. Verification Strategy

- **Acceptance Criteria**:
  - `just check` passes
  - Zero hand-rolled `*Validator` classes remain for block validation
  - `REVISION_BLOCK_JSON_SCHEMA` deleted
  - `MetadataValidator` rejects unknown keys
  - All `BlockSchema` entries with metadata have `metadata` field populated

## 7. Risks & Mitigations

- **Risk**: Hand-rolled validators have subtle semantic checks not expressible in `BlockMetadata` – _Likelihood_: medium – _Impact_: medium – _Mitigation_: Audit each validator's checks exhaustively before migration. Where `BlockMetadata` lacks expressiveness, extend it rather than leaving a validator behind.
- **Risk**: `MetadataValidator` unknown-key rejection breaks existing valid documents – _Likelihood_: low – _Impact_: medium – _Mitigation_: Run validation against all existing `.spec-driver/` artifacts before deploying.
