## 1. Summary & Context

- **Technical Spec(s)**: SPEC-114 (blocks/metadata), SPEC-116 (frontmatter_metadata), SPEC-115 (changes/blocks)
- **Change Drivers**: Code quality audit found two parallel validation systems for the same YAML block schemas, active regex bugs in the hand-rolled JSON Schema, and structural gaps in the metadata system.
- **Governance**: POL-001 (maximise code reuse)

## 2. Motivation

Every YAML-block type has both a hand-rolled validator class (~200+ lines each) **and** a declarative `BlockMetadata` instance describing the same schema. The two layers have diverged:

- Hand-rolled validators enforce unknown-key rejection; `MetadataValidator` does not.
- `REVISION_BLOCK_JSON_SCHEMA` has 4 double-escaped regex patterns (`r"^RE-\\d{3,}$"` matches literal backslashes). Active bug.
- The `BlockSchema` registry has no `metadata` field — cannot answer "what is the schema for this block type?"
- `FieldMetadata` cannot model dynamic-key objects; uses `_entry_shape` sentinel hack.
- 7 workflow schemas registered with `_placeholder_renderer` that produces empty strings.
- `DeltaRelationshipsValidator` error messages print dict repr instead of path names.

The `frontmatter_metadata/` subsystem demonstrates the metadata-driven approach works at scale (14 artifact kinds). The blocks subsystem should complete the same transition.

## 3. Scope & Objectives

- **Primary Outcomes** (canonical detail in DR-118; this list is summary-level):
  - **Phase 1 – Foundations**:
    - Add per-instance `strict_unknown_keys: bool = False` to `MetadataValidator` (DR-118 DEC-001). The flag is mechanism only — flipped `True` at retirement call sites in Phase 2; cross-cutting governance is DE-137 (DEC-003).
    - Add `additional_properties: FieldMetadata | None` to `FieldMetadata` (DEC-004); replaces `_entry_shape` sentinel.
    - ~~Add `metadata: BlockMetadata | None` field to `BlockSchema`~~ — already satisfied by DE-110 (commit `f6354c1c`); DR-118 records as already-met.
    - Make `BlockSchema.renderer` Optional; retire `_placeholder_renderer` (DEC-005).
  - **Phase 2 – Per-block migration** (5 swap commits ordered by coupling depth; details in DR-118 §4):
    1. `VerificationCoverageValidator` — zero external call sites, no ID kwarg.
    2. `PlanOverviewValidator` + `PhaseOverviewValidator` + `PhaseTrackingValidator` (single commit).
    3. `DeltaRelationshipsValidator` — requires `validate_delta_relationships(...)` wrapper for `delta_id` ID-equality check; external sites in `changes/artifacts.py:122` and `requirements/registry.py:243`.
    4. `RelationshipsBlockValidator` — requires `validate_spec_relationships` and `validate_spec_capabilities` wrappers for `spec_id` ID-equality checks; external site at `requirements/registry.py:162`.
    5. `RevisionBlockValidator` — last; external sites in `requirements/sync.py:211`, `changes/updater.py:135`; re-export at `changes/blocks/__init__.py:48,57` removed.
  - **Phase 3 – Cleanup**:
    - Delete `REVISION_BLOCK_JSON_SCHEMA` (and its 4 regex bugs).
    - Replace `_entry_shape` sentinel with `additional_properties=_SESSION_ENTRY` at `sessions_schema.py:88`.
    - Disambiguate `RELATIONSHIPS_MARKER` / `VALID_STATUSES` collisions (sites enumerated in IP-118 phase planning).
    - Drop `FieldMetadata.required` from `items` definitions (semantically meaningless).
- **Operational Constraints**: Validation verdict must be equivalent or stricter post-migration. No silent relaxation of constraints. Workflow.* unknown-key strictness is **deferred to DE-137 per DR-118 DEC-006** and tracked as **IMPR-035**.
- **Dependencies**: None strict; benefits from DE-114 (workflow_metadata.py split), DE-115 (block pattern factory). Sequenced before DE-137 per IP-136 Phase 2.

## 4. Out of Scope

- Changing what is validated (only changing how validation is implemented)
- Adding new block types
- Frontmatter metadata changes (that system is already on the right path)

## 5. Approach Overview

- For each validator: write parallel tests against both old and new validation, confirm identical results, then swap the implementation.
- The `revision.py` extractor (bespoke line-by-line state machine) is retained — it handles nested fences correctly, unlike the regex extractors. Migration is about validation, not extraction.

## 7. Risks & Mitigations

- **Risk R1**: Hand-rolled validators have subtle semantic checks not expressible in `BlockMetadata` – _Likelihood_: medium – _Impact_: medium – _Mitigation_: Per-validator inventory of cross-block / external-state checks before each Phase 2 swap (DR-118 §4 pre-Phase-2 inventory tasks).
- **Risk R2**: `MetadataValidator` strict path breaks existing valid documents – _Likelihood_: low – _Impact_: medium – _Mitigation_: snapshot-compare harness + `spec-driver validate` baseline-diff at every commit (DR-118 §5).
- **Risk R3** (surfaced during design): External call sites and ID-kwarg semantics. Five sites confirmed across `requirements/sync.py`, `changes/updater.py`, `changes/blocks/__init__.py`, `requirements/registry.py`, `changes/artifacts.py`. Two retiring validators (`RelationshipsBlockValidator`, `DeltaRelationshipsValidator`) carry parent-context ID-equality checks that `MetadataValidator` cannot express — preserved via thin wrapper helpers (DR-118 §4).
- **Risk R4** (surfaced during design): `additional_properties` with empty data passes silently (DR-118 DEC-004 documents this as preserved-not-regressed; OQ-NON-EMPTY-INVARIANT records the deferral).
- **Risk R5** (surfaced during design): Helper-shape mismatch on swap (`_disallow_extra_keys` Sequence vs. `properties` dict) — covered by VT (strict_unknown_keys) + per-block corpora.
- **Risk R6** (surfaced during design): DR-136 §11.3 cross-reference is best-effort; not under DE-118's authority. **IMPR-035 is the single durable record** for the workflow.* deferral; DR-136 cross-ref is informational.
- **Risk R7** (surfaced during second-pass review): `_entry_shape` migration is *new* enforcement, not equivalence. Real `workflow.sessions` data may break under strict per-entry shape validation. Phase 1 includes a one-shot pre-check against live sessions data; divergence resolved before swap commit lands.
- **Risk R8** (surfaced during second-pass review): Phase 2 partial-stall recovery — partial Phase 2 acceptable for ≤4 weeks; beyond that, complete or revert. DE-118 close-change must observe Phase 2 fully landed.
