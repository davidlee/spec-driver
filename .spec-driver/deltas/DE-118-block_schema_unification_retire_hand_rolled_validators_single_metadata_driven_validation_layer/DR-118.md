# DR-118 – Block schema unification: retire hand-rolled validators, single metadata-driven validation layer

## 1. Executive Summary

- **Delta**: [DE-118](./DE-118.md)
- **Status**: draft
- **Owners / Team**: unassigned
- **Last Updated**: 2026-05-09
- **Synopsis**: Collapse 7 hand-rolled block validators into the existing `MetadataValidator` driven by `BlockMetadata` declarations, fix the `REVISION_BLOCK_JSON_SCHEMA` regex bugs, retire the `_entry_shape` and `_placeholder_renderer` workarounds, and ship a dual-validate snapshot harness as durable regression infrastructure. Stays mechanism-only; cross-cutting validation surface (CLI strict flag, alias autocorrect, migrations) remains DE-137.

## 2. Problem & Constraints

### Current behaviour

- Two parallel validation systems for the same YAML block schemas: hand-rolled `*Validator` classes (~200 lines each) and declarative `BlockMetadata` instances. Hand-rolled validators reject unknown keys; `MetadataValidator` does not.
- `REVISION_BLOCK_JSON_SCHEMA` carries 4 double-escaped regex patterns (e.g. `r"^RE-\\d{3,}$"` matches literal backslashes) — active bug shielded by the duplicate-truth situation.
- `FieldMetadata` cannot model dynamic-key objects; a `_entry_shape` sentinel hack at `sessions_schema.py:88` papers over the gap and disables per-entry shape enforcement.
- 7 workflow.* schemas register with a `_placeholder_renderer` that returns `""` solely so `inspect.signature` succeeds in `BlockSchema.get_parameters()`.
- `DeltaRelationshipsValidator` error messages print dict reprs instead of path-aware diagnostics.

### Drivers / inputs

- DE-118.md §2 — code-quality audit of the parallel validation systems.
- POL-001 — maximise reuse, minimise sprawl.
- DR-136 §11.1 — application validators are current-schema only; cross-cutting validation infrastructure consolidates under DE-137.

### Constraints / guardrails

- **Behaviour preservation.** Each retirement is dual-validated against curated and corpus inputs before the swap commit lands. Verdict-level equivalence; message text differences are acceptable.
- **DE-137 ownership boundary.** DE-118 introduces *no* CLI flags, no alias autocorrect, no `validate --kind` / `validate file` / `schema enums`, no `migrations/` package wiring, no workflow.toml schema additions. Per-instance `strict_unknown_keys` is the only strictness surface.
- **Consumer-repo impact contained.** The 7 retirement targets already reject unknown keys today (hand-rolled), so flipping their loaders to `MetadataValidator(..., strict_unknown_keys=True)` preserves observable behaviour. The 7 workflow.* blocks (today loose under MetadataValidator) stay loose; their strict flip is deferred to DE-137 (DEC-006).
- **DR-136 §11.1 separation.** No imports from / to a (future) `migrations/` package. Validator stays current-schema only.
- **Pre-merge gate.** `spec-driver validate` introduces no NEW errors or warnings against this repo (pre-existing unrelated warnings unchanged) at every Phase 1, Phase 2, and Phase 3 commit boundary.

### Out of scope

- Cross-cutting validation infrastructure (DE-137).
- Workflow.* unknown-key strict flip (DEC-006 → DE-137).
- New block types or new block content.
- Frontmatter metadata changes (`frontmatter_metadata/` is already on the right path).
- Renderer implementation for workflow.* blocks (DR-136 §5.1's metadata-driven template work is DE-137).

## 3. Architecture Intent

### Target outcomes

- Single validation path for the 7 listed block types via `MetadataValidator` against existing `BlockMetadata` declarations (POL-001).
- Per-instance `strict_unknown_keys` mechanism in `MetadataValidator` — the only DE-118-introduced strictness surface.
- `FieldMetadata` expressive enough for dynamic-key objects via `additional_properties`; `_entry_shape` sentinel retired.
- `BlockSchema.renderer` is `Optional`; workflow blocks declared validate-only.
- `REVISION_BLOCK_JSON_SCHEMA` deleted; canonical revision-change validation lives only in `REVISION_CHANGE_METADATA`.
- Snapshot-compare harness available as durable infrastructure for future schema changes (POL-001 — reusable beyond DE-118).

> Note: DE-118.md §3 originally listed three Phase-1 enhancements. The third — "Add `metadata: BlockMetadata | None` field to `BlockSchema`" — was satisfied by DE-110 (commit `f6354c1c`) before DR-118 was drafted. DR-118 records this as already-met; only `additional_properties` and `strict_unknown_keys` mechanism remain.

### Guiding principles

- **Behaviour preservation, then unification.** Each retirement is dual-validated (curated + corpus) before swap.
- **POL-003 boundary respect.** New surface stays inside `blocks/metadata/`. No imports from / to `migrations/` or cross-cutting CLI surface (DE-137 territory).
- **No premature abstraction.** Per-instance flag, not declarative metadata, until DE-137 specifies governance.
- **Pure-function bias.** Validator semantics remain `(metadata, data) → errors`; `strict_unknown_keys` is constructor-time configuration, no runtime statefulness.

### State transitions / lifecycle impact

- No artefact lifecycle changes. Block validation continues at parse-time; raise-on-error semantics preserved.
- Forward compatibility with DE-137: per-instance `strict_unknown_keys=True` call sites map mechanically to whatever per-kind / declarative shape DE-137 chooses. No behavioural drift expected at the seam.

### Phase 2 — Ordering principle

DR-118 commits to the following retirement order. Order chosen by *coupling depth* (lowest-first), not surface size — the simplest swaps are the ones with no external-state checks and no ID-kwarg semantics:

1. `VerificationCoverageValidator` — zero external call sites, no ID kwarg; smallest meaningful coupling. Exercises path-aware errors first.
2. `PlanOverviewValidator`, `PhaseOverviewValidator`, `PhaseTrackingValidator` — three plan-related validators in a single commit (shared file `plan.py`; coupled domain; verify whether any take ID kwargs in pre-Phase-2 inventory).
3. `DeltaRelationshipsValidator` — **requires `validate_delta_relationships(...)` wrapper** (ID-equality check on `delta_id`). External call sites in `changes/artifacts.py:122` and `requirements/registry.py:243` migrated in same commit.
4. `RelationshipsBlockValidator` — **requires two wrappers** (`validate_spec_relationships`, `validate_spec_capabilities`); single class, two metadata wirings. External call site at `requirements/registry.py:162` migrated in same commit.
5. `RevisionBlockValidator` — last; largest blast radius; external call sites at `requirements/sync.py:211`, `changes/updater.py:135`, re-export at `changes/blocks/__init__.py:48,57` migrated in the same commit.

Total: 5 swap commits in Phase 2. Each commit must satisfy the §2 pre-merge gate before merging.

### ID-kwarg wrappers — preserved external-state checks

`DeltaRelationshipsValidator` and `RelationshipsBlockValidator` perform a check that `MetadataValidator` cannot express: the block's declared parent ID must match a kwarg passed by the caller (the artefact ID). This is parent-context state, not block content.

DR-118 retains this behaviour via thin wrapper helpers placed alongside their metadata declarations (e.g. `delta_metadata.py`, `spec_metadata.py`). Sketch:

```python
def validate_delta_relationships(
    block: DeltaRelationshipsBlock, *, delta_id: str | None = None
) -> list[str]:
    errors = [
        str(e) for e in MetadataValidator(
            DELTA_RELATIONSHIPS_METADATA, strict_unknown_keys=True
        ).validate(block.data)
    ]
    if delta_id and block.data.get("delta") != delta_id:
        errors.append(f"delta block id {block.data.get('delta')} != expected {delta_id}")
    return errors
```

Same shape for `validate_spec_relationships` and `validate_spec_capabilities`. The wrapper keeps `MetadataValidator` pure (block-only) and the ID check explicit. No new abstraction; just a function. (Three wrapper functions total; ~5 lines each.)

### Validation baseline mechanism

The "no NEW issues" gate (§2) requires a baseline. Phase 0 of IP-118 captures `spec-driver validate` output verbatim into `.spec-driver/deltas/DE-118-…/validate-baseline.txt`. Each Phase-1, Phase-2, and Phase-3 commit must produce identical-or-shorter output (any new issue is a blocker). This baseline file is delta-scoped, retired at `/close-change`.

### Equivalence semantics

"Equivalence" is at **verdict level** (accept/reject) only. Error message text may differ — `MetadataValidator` errors are path-aware and structured; hand-rolled errors are ad-hoc strings or dict reprs. Tests assert agreement on `len(errors) == 0` versus `len(errors) > 0`, not on message contents.

### Transitional invariants during Phase 2

- The codebase passes through mixed states (some validators hand-rolled, others metadata-driven). Required invariants:
  - Snapshot harness validates each block type against whichever validator is currently wired into the production parse path (reads `BLOCK_SCHEMAS` registry at runtime; not hard-coded).
  - Registry mutations during Phase 2 (e.g. a new `BlockSchema(renderer=None)` registration) are picked up automatically by the harness; new metadata-bearing schemas added in DE-118 dual-validate alongside retiring ones without harness changes.
  - Each parallel-test file imports the hand-rolled class; the import becomes the trigger for retirement (file deletion participates in the swap commit).
  - `spec-driver validate` baseline-diff (no NEW issues) remains the load-bearing acceptance gate at every commit.

## 7. Design Decisions & Trade-offs

### DEC-001 — Per-instance `strict_unknown_keys` parameter

**Decision.** Add `strict_unknown_keys: bool = False` to `MetadataValidator.__init__`. When `True`, validation rejects keys not declared in `properties` and not matching `additional_properties`. The 7 retirement loaders pass `True` at call sites; everything else inherits the default `False`.

**Rationale.** Mechanism only; minimum viable surface for the retirement work. Avoids global flag, BlockMetadata field, and workflow.toml integration — those are DE-137's surface (DEC-003). Per-call-site opt-in preserves hand-rolled rejection behaviour exactly where it lived before.

**Alternatives rejected.** Global flag at validator construction (blurs DE-137 boundary); `BlockMetadata.strict_unknown_keys` declarative (premature; DE-137 may take a different shape); `StrictMetadataValidator` subclass (needless inheritance for one bool).

### DEC-002 — Reuse existing `ConditionalRule`

**Decision.** DE-118 does not modify `schema.py:ConditionalRule` or `validator.py:_validate_conditional_rules`. DE-142's anticipated revision-action conditional rules extend behaviour by adding `ConditionalRule` instances to `REVISION_CHANGE_METADATA`, not by extending validator code.

**Rationale.** Existing surface (`condition_field × condition_value × requires`) supports DE-142's needs as known today. Re-verification when DE-142 enters scope is acknowledged as not blocking (OQ-DE142-CONDITIONAL-FIT).

### DEC-003 — DE-137 ownership boundary

**Decision.** DE-118 ships only the per-instance mechanism, `additional_properties`, the optional renderer, the 7 validator retirements, and the snapshot harness. All cross-cutting validation infrastructure remains DE-137: CLI strict flag (per-kind workflow.toml), alias autocorrect, `tolerated_aliases`, `validate --kind`, `validate file <path>`, `schema enums`, `admin migrate` orchestrator, `migrations/` package wiring, workflow.toml schema additions.

**Rationale.** B1 boundary chosen during preflight. Minimises DE-118's surface and prevents drift with DE-137's eventual flag shape. DE-118 closes without DE-137; DE-137 retroactively governs DE-118's flipped sites by promoting `strict_unknown_keys` from per-instance to declarative or kind-level.

### DEC-004 — `additional_properties` shape and validator semantics

**Decision.** Add `additional_properties: FieldMetadata | None = None` to `FieldMetadata`. Validator integration is at the object-type validation site in `_validate_field` (`validator.py:184`), not at the top-level `_validate_fields` declared-iteration. The unknown-key pass runs **after** the declared-property pass, at every object nesting depth the validator reaches.

Concrete algorithm (replacement for the existing `_validate_field` object-type branch):

```python
elif field_meta.type == "object":
    if not isinstance(value, dict):
        errors.append(ValidationError(...))
        return errors

    declared = set()
    if field_meta.properties:
        # Existing required-field + recursive declared-property validation.
        errors.extend(self._validate_fields(value, field_meta.properties, field_path))
        declared = set(field_meta.properties.keys())

    # NEW: unknown-key handling (runs at every object depth via recursion).
    for key in value:
        if key in declared:
            continue
        sub_path = f"{field_path}.{key}"
        if field_meta.additional_properties is not None:
            errors.extend(
                self._validate_field(value[key], field_meta.additional_properties, sub_path)
            )
        elif self.strict_unknown_keys:
            errors.append(ValidationError(path=sub_path, message="unknown key"))
        # else: silently accept (preserves current loose behaviour)
```

`self.strict_unknown_keys` is an instance attribute, so strictness propagates naturally to every nested object via recursion. Required-field checks for declared `properties` (in `_validate_fields`) are unchanged.

The same logic also runs at the top-level `validate(data)` entry — the data-level loop checks for unknown keys against `metadata.fields` and against (the absence of) declared `additional_properties`. **No `additional_properties` field is added to `BlockMetadata` itself in DE-118** — no current schema needs top-level dynamic keys, and ADR-009 argues against speculative structure. The validator simply rejects any top-level data key not in `metadata.fields` when `strict_unknown_keys=True`. If a future block needs top-level dynamic keys, a follow-up delta can extend `BlockMetadata` symmetrically. The `_entry_shape` sentinel at `sessions_schema.py:88` migrates: drops sentinel, sets `additional_properties=_SESSION_ENTRY`, `properties=None`.

**Empty-data semantics (explicit, not a regression).** When `properties=None, additional_properties=<shape>, data={}`: the validator passes silently. This matches today's behaviour — the sentinel hack also accepted empty maps; the description text at `sessions_schema.py:83` ("Map of role name to session entry (at least one entry required)") was advisory, not programmatically enforced. DE-118 preserves this. A future enhancement could add `min_entries` (or repurpose `min_items`) to FieldMetadata for non-emptiness on dynamic-key maps; that is **not** in scope here. OQ-NON-EMPTY-INVARIANT records the deferral.

**Rationale.** Single mental model unifies "extra-key behaviour" and "dynamic-key shape". Per-entry shape becomes enforceable for dynamic-keyed objects.

**Alternatives rejected.** JSON Schema-style `bool | FieldMetadata` (more surface than DE-118 needs); keep sentinel and document (leaves the gap DE-118 §3 explicitly targets).

### DEC-007 — Equivalence verification methodology

**Decision.** Per retiring validator:

1. Author curated parallel test corpus: positive paths + every negative branch in hand-rolled source + the 4 `REVISION_BLOCK_JSON_SCHEMA` regex-bug cases.
2. Snapshot-compare harness enumerates block instances of the relevant marker across `.spec-driver/`, dual-validates, asserts agreement.
3. Once green: swap loader call site to `MetadataValidator(<META>, strict_unknown_keys=True)`, delete hand-rolled class plus the dual-validation half of the test, retain MetadataValidator-only corpus assertions.

**Snapshot harness.** New module `supekku/scripts/lib/blocks/metadata/snapshot_compare.py`. CLI: `python -m supekku.scripts.lib.blocks.metadata.snapshot_compare --root <path>`. Pointable at any `.spec-driver/` corpus, including external/offline ones the user may run privately before authorising DE-118 Phase 2 close.

**Rationale.** Curated cases prove known semantics; corpus catches BlockMetadata-vs-actual drift (REVISION_BLOCK_JSON_SCHEMA's regex bugs survived undetected because no corpus check existed). Harness is reusable for future schema changes — earns its keep beyond DE-118.

**Limitation acknowledged.** Curated parallel-test corpora are pinned to the *current* hand-rolled validator's semantics. Once swap commits land and the parallel half retires, only the metadata-driven side remains. Future tightenings of `BlockMetadata` (e.g. narrowing an enum) will pass-or-fail against the *new* metadata, not the original hand-rolled intent — that is the desired behaviour, but it does mean DE-118's tests cannot signal "metadata has drifted from original intent" after the swap. Each corpus test file should therefore carry a header comment: `# Captures hand-rolled <Validator> behaviour as of DE-118; tightening or relaxing rules is an *intended* drift event handled in the delta that introduces it.`

## 9. Rollout & Operational Notes

- **Migration.** No data migration. Behaviour-preserving for the 7 retiring blocks (hand-rolled rejection → MetadataValidator rejection). No behaviour change for the 7 workflow.* blocks (DEC-006 defers).
- **Observability.** No metrics/telemetry changes. Validation failure mode (raise) and surface (`spec-driver validate` exit code) unchanged.
- **Recovery / rollback.** Each retirement is a discrete commit. Rollback for any retirement = revert that commit (re-introduces the hand-rolled validator). Phase 1 changes are additive and backwards-compatible (default `strict_unknown_keys=False`); rollback is revert. Phase 3 cleanup deletions can be reverted independently.

### Risks surfaced during design (supplement to DE-118 §7)

- **R3 — External call sites and ID-kwarg semantics.** Five external call sites confirmed: `RevisionBlockValidator` at `requirements/sync.py:211`, `changes/updater.py:135`, re-exported via `changes/blocks/__init__.py:48,57`; `RelationshipsBlockValidator` at `requirements/registry.py:162`; `DeltaRelationshipsValidator` at `changes/artifacts.py:122` and `requirements/registry.py:243`. Two of these (`RelationshipsBlockValidator`, `DeltaRelationshipsValidator`) carry **ID-kwarg external-state checks** that `MetadataValidator` cannot express (parent-context state). *Mitigation*: thin wrapper helpers (`validate_delta_relationships`, `validate_spec_relationships`, `validate_spec_capabilities`) retain the ID equality check while delegating shape validation to MetadataValidator. Pre-Phase-2 inventory task confirms there are no analogous unsurfaced checks in the remaining 4 retiring classes.
- **R4 — Empty dynamic-key map regression risk.** `additional_properties` with empty data passes silently (DEC-004). Description text at `sessions_schema.py:83` claims non-emptiness that is not (and was never) enforced programmatically. *Mitigation*: documented as continuing behaviour; OQ-NON-EMPTY-INVARIANT records the deferral; if a real failure mode surfaces during the corpus snapshot run, escalate.
- **R5 — Helper shape mismatch on swap.** `_disallow_extra_keys` accepts `Sequence[str]` of allowed keys; `MetadataValidator` derives the closed set from `properties: dict[str, FieldMetadata]`. Functionally equivalent (same key set) but different surface — corpus tests catch any divergence. *Mitigation*: covered by VT (strict_unknown_keys) + per-block corpora.
- **R6 — DR-136 cross-reference is best-effort.** DEC-006's "informational records" include a paragraph in DR-136 §11.3 not under DE-118's authority. If declined by DR-136 owners, the backlog IMP remains the only durable record. *Mitigation*: DEC-006's record-collapse acknowledges this explicitly.
- **R7 — `_entry_shape` migration is *new* enforcement, not equivalence.** Today the sentinel `_entry_shape` key is ignored by `MetadataValidator` (the meta-driven loop never finds it in real data). After DE-118, every dynamic session entry validates against `_SESSION_ENTRY` shape — including required-field checks that didn't run before (e.g. `last_seen` semantics in `sessions_schema`). Real `workflow.sessions` data may break. *Mitigation*: Phase 1 includes a one-shot pre-check — run the new `additional_properties` semantics against any existing `.spec-driver/run/sessions/` data (or equivalent live corpus) and resolve any disagreements before the swap commit lands. If real divergence exists, surface as a curated test case AND a drift entry.
- **R8 — Phase 2 partial-stall recovery.** If only some swap commits land (e.g. 1–3 of 5 due to external priorities), the codebase carries `MetadataValidator` for some block types and hand-rolled for others indefinitely — coherent compile-state but a two-system maintenance burden. *Policy*: a partial Phase 2 is acceptable for ≤4 weeks; beyond that, either complete Phase 2 or revert the landed swaps to restore single-system state. DE-118 close-change must observe Phase 2 is fully landed (5/5 commits).
