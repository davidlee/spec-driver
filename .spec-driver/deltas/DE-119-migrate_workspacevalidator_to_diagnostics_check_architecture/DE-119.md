## 1. Summary & Context

- **Technical Spec(s)**: SPEC-125 (validation), SPEC-158 (diagnostics/checks)
- **Change Drivers**: Code quality audit found two parallel validation systems. `WorkspaceValidator` (594-line god class) and `diagnostics/` (clean, extensible architecture) both in active use.
- **Governance**: POL-001

## 2. Motivation

`WorkspaceValidator` is a 594-line accumulator-pattern class with 12 private validation methods all appending to `self.issues`. It predates the `diagnostics/` subsystem which has the correct architecture: pure functions, registry-based composition, isolated checks returning `list[DiagnosticResult]`.

Both are in active use. New validation checks get added to the legacy class because that's where the mass is, preventing the diagnostics system from reaching critical mass. The migration path is clear: port each private method to a `diagnostics/checks/*.py` module.

The `diagnostics/` architecture is exemplary and should be the sole validation system.

## 3. Scope & Objectives

- **Primary Outcomes**:
  - Inventory all 12 `WorkspaceValidator` private validation methods
  - Port each to a corresponding `diagnostics/checks/*.py` pure function
  - Register new checks in `diagnostics/__init__.py`
  - Update any callers of `WorkspaceValidator` to use `diagnostics.runner`
  - Delete `WorkspaceValidator` and `validation/validator.py`
  - Disposition the `validator.py` TODO ("implement grace period check") â€” either implement as a diagnostic check or file as backlog
- **Operational Constraints**: All existing validation coverage must be preserved. No checks silently dropped.
- **Dependencies**: None strict. Independent of other deltas.

## 4. Out of Scope

- Adding new validation rules (only porting existing ones)
- Changing the diagnostics architecture itself
- CLI changes (already wired to `diagnostics.runner`)

## 5. Approach Overview

- **Phase 1**: Inventory and categorize all `WorkspaceValidator` checks. Map each to an existing or new `diagnostics/checks/` category.
- **Phase 2**: Port checks one category at a time. For each: write the diagnostic check function, add tests, verify output parity with the validator.
- **Phase 3**: Remove `WorkspaceValidator` and `validation/` package. Update imports.

## 6. Verification Strategy

- **Acceptance Criteria**:
  - `just check` passes
  - `validation/validator.py` deleted
  - All validation findings previously produced by `WorkspaceValidator` are now produced by `diagnostics/` checks
  - `spec-driver validate` output covers all prior check categories
  - No `WorkspaceValidator` references remain in codebase
