## 1. Summary & Context

- **Technical Spec(s)**: SPEC-110 (supekku/cli)
- **Change Drivers**: Code quality audit found `Workspace` facade systematically bypassed. `cli/common.py` contains duplicate factory functions; `cli/create.py` constructs registries directly.

## 2. Motivation

`Workspace` is declared as a "unified facade over project registries" providing consolidated lifecycle and `sync_all_registries()`. However:

- `cli/common.py` contains duplicate registry factory functions (lines 435–495 and 899–931)
- `cli/create.py` constructs `DecisionRegistry`, `PolicyRegistry`, `StandardRegistry` directly
- `cli/find.py` imports 6 registry classes directly
- The facade's value (ordered sync, consistent root resolution) is unavailable to the callers that need it most

Routing all registry access through `Workspace` makes the dependency explicit, reduces import sprawl, and ensures consistent root resolution.

## 3. Scope & Objectives

- **Primary Outcomes**:
  - Audit all direct registry instantiation in `cli/`
  - Replace with `Workspace` property access (`ws.specs`, `ws.changes`, `ws.decisions`, etc.)
  - Remove duplicate factory functions from `cli/common.py`
  - Construct `Workspace` once per CLI invocation; pass via Typer context or callback
  - Ensure `Workspace` exposes all needed registries (add properties if missing)
- **Operational Constraints**: CLI behaviour unchanged. `Workspace` may need new registry properties.
- **Dependencies**: Loosely DE-116 (consistent registry API makes facade cleaner). Can proceed independently.

## 4. Out of Scope

- Changing `Workspace` internal architecture
- Adding caching or lazy-loading to `Workspace` (future optimisation)
- Non-CLI callers (scripts, tests that instantiate registries directly)

## 5. Approach Overview

- Add missing registry properties to `Workspace` as needed
- In `cli/main.py` callback: construct `Workspace(root)` and store in Typer context
- In each CLI module: retrieve `Workspace` from context instead of constructing registries
- Delete orphaned factory functions from `common.py`

## 6. Verification Strategy

- **Acceptance Criteria**:
  - `just check` passes
  - `grep -rn 'Registry(root' supekku/cli/` returns zero results (no direct construction)
  - All CLI registry access goes through `Workspace`
  - `common.py` factory functions deleted

## 7. Risks & Mitigations

- **Risk**: `Workspace` becomes a god object – _Likelihood_: medium – _Impact_: medium – _Mitigation_: `Workspace` is a facade, not a service. Properties are lazy. Each registry retains its own logic.
- **Risk**: Typer context threading is awkward with sub-apps – _Likelihood_: medium – _Impact_: low – _Mitigation_: Typer supports `Context.obj`; existing pattern in the codebase.

## 8. Follow-ups & Tracking

- **Depends on**: Loosely DE-116 (cleaner registry API)
- **Sequence**: Sprint 6 of quality remediation. Parallel with DE-122.
