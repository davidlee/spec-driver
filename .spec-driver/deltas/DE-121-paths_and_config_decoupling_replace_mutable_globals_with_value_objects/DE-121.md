## 1. Summary & Context

- **Change Drivers**: Code quality audit identified the most dangerous coupling in the codebase: `init_paths()` mutates module-level globals via `globals()[const_name] = value`, affecting 92 importing files.
- **Governance**: POL-002 (avoid magic strings/numbers — mutable "constants" are the extreme case)

## 2. Motivation

`core/paths.py` exports 15+ directory path "constants" that are secretly mutable. `init_paths()` rewrites them at runtime via `globals()` mutation. Every file that imports a path constant silently depends on `init_paths()` having been called first by `cli/main.py`. Library consumers, tests, and any non-CLI entry point see incorrect default paths with no error.

Additionally:
- Directory defaults are duplicated between `paths.py` (module constants) and `config.py` (`DEFAULT_CONFIG["dirs"]`)
- `config.py` has three unrelated responsibilities (config loading, exec detection, TOML template rendering)
- `events.py` uses mutable module-level singletons (`_touched_artifacts`) with implicit coupling
- `load_workflow_config()` is re-parsed from disk per subcommand instead of passed through context

## 3. Scope & Objectives

- **Primary Outcomes**:
  - **Phase 1 – Introduce `Paths` value object**:
    - Define frozen `Paths` dataclass constructed from config
    - Add `Paths` as a property of `Workspace`
    - Keep `init_paths()` and global constants as fallback during migration
  - **Phase 2 – Migrate callers**:
    - Module-by-module, replace `from core.paths import X` with `paths.X` from a `Paths` instance
    - Thread `Paths` through call stacks (via `Workspace`, Typer context, or explicit parameter)
  - **Phase 3 – Remove globals**:
    - Delete `init_paths()`, `reset_paths()`, and module-level mutable constants
    - Single source of truth for defaults: `DEFAULT_CONFIG["dirs"]` only
  - **Phase 4 – Config/events cleanup**:
    - Split `config.py` three-way: `core/config.py` (loading), `core/exec_detect.py`, `install/toml_template.py`
    - Replace `events.py` globals with `CommandContext` dataclass passed explicitly
    - Parse `workflow.toml` once per CLI invocation; pass config through context
- **Operational Constraints**: 92 files affected. Must be phased to avoid a single massive PR. Each phase independently shippable.
- **Dependencies**: Benefits from all prior cleanup (DE-113 through DE-120 reduce the number of files to migrate and make the codebase more navigable). Should be last.

## 4. Out of Scope

- Changing the config file format or schema
- Changing the directory structure of `.spec-driver/`
- Test framework changes (test helpers adapt to `Paths` object naturally)

## 5. Approach Overview

- **Key insight**: The phased approach means `Paths` object and globals coexist temporarily. The `Paths` object is authoritative; globals are a compatibility shim that logs deprecation warnings in dev mode.
- **Migration pattern per file**: Replace `from core.paths import get_X_dir` with accepting `paths: Paths` parameter; update callers to pass it.
- **config.py split**: Mechanical extraction, same as DE-114 approach.

## 6. Verification Strategy

- **Acceptance Criteria**:
  - `just check` passes at each phase boundary
  - After Phase 3: `grep -rn 'init_paths\|reset_paths\|globals()' supekku/scripts/lib/core/paths.py` returns 0 results
  - After Phase 3: no module-level mutable path constants remain
  - After Phase 4: `config.py` < 200 lines; `events.py` has no module-level mutable state
  - Library use (non-CLI) works without calling any init function

## 7. Risks & Mitigations

- **Risk**: 92-file migration is error-prone and conflicts with concurrent work – _Likelihood_: high – _Impact_: medium – _Mitigation_: Phase the migration by package. Each phase is a self-contained PR. Global fallback provides safety net during transition.
- **Risk**: Performance regression from threading `Paths` through deep call stacks – _Likelihood_: low – _Impact_: low – _Mitigation_: `Paths` is a frozen dataclass; passing it is zero-cost.
- **Risk**: Circular imports when `Workspace` carries `Paths` and modules import both – _Likelihood_: medium – _Impact_: medium – _Mitigation_: `Paths` lives in `core/paths.py` (leaf module); `Workspace` in `lib/workspace.py`. Direction is always core ← domain ← workspace.
