## 1. Summary & Context

- **Product Spec(s)**: PROD-010 (CLI UX), PROD-011 (Implementation Execution Workflow)
- **Implementation Plan**: [IP-124](./IP-124.md)
- **Design Revision**: [DR-124](./DR-124.md)
- **Change Drivers**: autobahn DE-004 needs to call spec-driver's workflow domain logic as typed Python imports. The current module structure (`supekku.scripts.lib.workflow.*`) is internal and unstable. A public facade provides a stable import contract without requiring a full codebase restructure.

## 2. Motivation

- autobahn (runtime orchestration layer) is a Python library in the same uv workspace. It needs to call spec-driver's review/workflow domain functions directly — subprocess + JSON envelope is unnecessary overhead for a same-process Python consumer.
- The internal module path `supekku.scripts.lib.workflow.*` is deep, unstable, and reflects historical structure (`scripts/lib/`) not intentional architecture.
- `supekku` is an internal codename; the product is `spec-driver`. A new `spec_driver` top-level package provides the right namespace for public API without requiring an immediate rename of the entire codebase.
- This follows the `_pytest`/`pytest` pattern: internal implementation in one package, curated public surface in another. The facade is the contract; internals are free to change.

## 3. Scope & Objectives

- **Primary Outcomes**:
  - New top-level package `spec_driver/` with a `workflow` subpackage
  - `spec_driver.workflow` re-exports ~30 symbols: review I/O functions, state machine types/enums, staleness evaluation, git helpers
  - Packaging updated so both `supekku` and `spec_driver` ship in the same distribution
  - Basic import tests proving the public surface resolves correctly
- **Operational Constraints**: Zero changes to existing `supekku/` code. Pure additive. No new dependencies.
- **Dependencies**: None — re-exports existing completed work (DE-108, DE-109)

## 4. Out of Scope

- Renaming `supekku` → `spec_driver` internally (future, driven by DE-114–DE-122 refactoring arc)
- Public API for non-workflow subsystems (memories, specs, registries) — add when consumers exist
- Handoff I/O, bridge, or state machine re-exports — add when autobahn needs them
- Versioning/stability guarantees beyond "re-export contract"

## 5. Approach Overview

- **Key Changes**:
  - `spec_driver/__init__.py` — package root, version re-export
  - `spec_driver/workflow/__init__.py` — curated re-exports from `supekku.scripts.lib.workflow.*` and `supekku.scripts.lib.core.git`
  - `pyproject.toml` — include `spec_driver` in packages
  - Tests — import smoke tests verifying all public symbols resolve

## 6. Verification Strategy

- **Acceptance Criteria**:
  - `from spec_driver.workflow import read_state, BootstrapStatus, evaluate_staleness` works
  - All re-exported symbols are importable and are the same objects as their `supekku` counterparts
  - `just check` passes
  - No changes to existing `supekku/` files

## 7. Risks & Mitigations

- **Risk**: Internal module refactoring breaks re-exports — _Likelihood_: medium — _Impact_: low — _Mitigation_: Import tests catch breakage immediately; re-exports are updated in the same PR as internal moves
- **Risk**: Namespace collision if `spec-driver` is installed alongside another `spec_driver` package — _Likelihood_: very low — _Impact_: low — _Mitigation_: We control the distribution; no PyPI publish planned

## 8. Follow-ups & Tracking

- autobahn DE-004 re-scope to import from `spec_driver.workflow` instead of subprocess
- autobahn ADR-002: "CLI for agents/humans, Python API for orchestrators"
- Future: incremental migration of `supekku` internals into `spec_driver` (aligned with DE-114–DE-122 refactoring arc)
- Future: additional public API subpackages as consumers emerge
