# DR-124 – Public Python API facade

## 1. Executive Summary

- **Delta**: [DE-124](./DE-124.md)
- **Status**: draft
- **Last Updated**: 2026-03-23
- **Synopsis**: Extract review orchestration logic from CLI commands into domain functions, expose them through a `spec_driver.workflow` public API package. CLI becomes a thin formatting layer.

## 2. Problem & Constraints

- **Current Behaviour**: Review operations (`prime`, `complete`, `disposition`, `teardown`) are orchestrated in CLI command functions (120–150 lines each). The orchestration includes load-bearing guards, state transitions, staleness cascade logic, and write ordering. The underlying building blocks (`review_io`, `review_state_machine`, `state_io`, `staleness`) are primitives that must be composed correctly.
- **Drivers**:
  - autobahn needs typed Python access to review operations (not building blocks)
  - Exporting only building blocks forces autobahn to replicate orchestration logic, violating POL-001
  - CLI commands currently violate the skinny CLI pattern (mem.pattern.cli.skinny)
- **Constraints**:
  - Existing CLI behaviour must be preserved (exit codes, JSON envelope, human output)
  - All existing CLI tests must pass without modification
  - No changes to workflow file formats or schemas
  - No new external dependencies
- **Out of Scope**: Non-review workflow operations, god-init fix (DE-114), internal rename

## 3. Architecture Intent

- **Target Outcomes**:
  - Orchestration logic lives in domain layer, testable without CLI
  - CLI commands are thin wrappers: parse args → call domain → format output
  - autobahn imports `spec_driver.workflow.prime_review(...)` — a composed operation, not a bag of primitives
  - Guards (approval, state transition) enforced in exactly one place (domain function)
- **Guiding Principles**:
  - Skinny CLI pattern: CLI does parsing and formatting, domain does everything else
  - One canonical orchestration path: CLI and autobahn call the same function
  - Exceptions for errors, not exit codes — CLI maps exceptions to exit codes
- **State Transitions**: No changes to review or workflow state machines. Operations call existing transitions.

## 4. Extraction Design

### 4.1 New module: `supekku/scripts/lib/workflow/operations.py`

Four composed operations extracted from CLI:

#### `prime_review`

```python
@dataclass
class PrimeResult:
    delta_id: str
    action: str              # "created" | "rebuilt" | "refreshed"
    bootstrap_status: str    # always "warm" after successful prime
    judgment_status: str     # "in_progress"
    review_round: int
    index_path: Path
    bootstrap_path: Path

def prime_review(
    delta_dir: Path,
    repo_root: Path,
    *,
    config: dict | None = None,
) -> PrimeResult:
```

Extracted from `review_prime_command` (lines 844–1058). Includes:
- State read + validation
- Git HEAD resolution
- Workflow config loading (if not provided)
- Cache staleness evaluation (cold → invalid → stale → reusable cascade)
- Deleted domain_map file check
- Section carry-forward for reusable caches
- State transition (`begin_review`)
- Index build + write
- Bootstrap markdown generation + write
- Action and round number derivation

Raises: `StateNotFoundError`, `StateValidationError`, `ReviewIndexValidationError`

#### `complete_review`

```python
@dataclass
class CompleteResult:
    delta_id: str
    round_number: int
    outcome: str             # "approved" | "changes_requested"
    previous_state: str
    new_state: str
    findings_path: Path
    teardown_performed: bool

def complete_review(
    delta_dir: Path,
    repo_root: Path,
    *,
    status: str,
    summary: str | None = None,
) -> CompleteResult:
```

Extracted from `review_complete_command` (lines 1241–1448). Includes:
- State read + validation
- Workflow transition validation (`apply_transition`)
- Approval guard: `collect_blocking_findings` + `can_approve` (raises on violation)
- Round build/append (v2 accumulative model)
- Write ordering: findings → review-index → state (per DR-102 §5)
- Auto-teardown on approval (policy-driven)

Raises: `StateNotFoundError`, `StateValidationError`, `ReviewApprovalGuardError` (new), `TransitionError`, `FindingsValidationError`

#### `disposition_finding`

```python
@dataclass
class DispositionResult:
    delta_id: str
    finding_id: str
    action: str
    previous_status: str
    new_status: str

def disposition_finding(
    delta_dir: Path,
    finding_id: str,
    disposition: dict,
) -> DispositionResult:
```

Extracted from `_disposition_finding` (lines 1503–1589). Includes:
- Read findings + version check
- Capture previous status
- Apply disposition
- Write findings with validation

Raises: `FindingsNotFoundError`, `FindingsVersionError`, `FindingsValidationError`, `FindingNotFoundError` (new, cleaner than returning bool)

#### `teardown_review`

```python
@dataclass
class TeardownResult:
    delta_id: str
    removed: list[str]

def teardown_review(
    delta_dir: Path,
) -> TeardownResult:
```

Extracted from `_do_teardown` (lines 1997–2028). Trivial: delete files, return list.

### 4.2 Helper migration

These CLI-private helpers move to `operations.py` (they're domain logic, not CLI logic):

| Helper | Current location | Rationale |
|---|---|---|
| `_prime_action` | CLI L1060 | Maps cache status to action label |
| `_build_domain_map` | CLI L1080 | Assembles domain map from delta bundle |
| `_generate_bootstrap_markdown` | CLI L1148 | Generates bootstrap document |
| `_load_workflow_config` | CLI L114 | Config loading (stays as shared helper, used by non-review commands too) |
| `_resolve_delta_dir` | CLI L54 | Delta directory resolution (stays in CLI — used by all commands, could be extracted separately) |

`_load_workflow_config` and `_resolve_delta_dir` stay in the CLI module for now — they're shared across non-review commands. The operations module accepts `delta_dir: Path` and `config: dict` as parameters, keeping it independent of delta resolution strategy and config loading mechanism.

### 4.4 CLI refactoring

After extraction, each CLI command becomes:

```python
@review_app.command("prime")
def review_prime_command(delta, format_type, root):
    repo_root = find_repo_root(root)
    delta_dir = _resolve_delta_dir(delta, repo_root)
    json_mode = format_type == "json"
    try:
        result = prime_review(delta_dir, repo_root)
    except StateNotFoundError as exc:
        # format error → exit
    except ...:
        # format error → exit
    if json_mode:
        emit_json_and_exit(cli_json_success("review.prime", asdict(result)))
    typer.echo(f"Review primed: {result.delta_id} ({result.action})")
    ...
```

~20–30 lines per command: arg parsing, error formatting, output formatting.

**State I/O** (from `supekku.scripts.lib.workflow.state_io`):

- `ReviewApprovalGuardError`, `FindingNotFoundError`
- `ReviewTransitionError`
- `StateNotFoundError`, `StateValidationError`
- `FindingsNotFoundError`, `FindingsValidationError`, `FindingsVersionError`
- `ReviewIndexNotFoundError`, `ReviewIndexValidationError`

**Explicit `__all__`** — machine-auditable, ~40 symbols.

### 5.3 What is NOT exported

- Write functions (`write_state`, `write_review_index`, `write_findings`) — mutations go through composed operations
- Building block functions (`build_review_index`, `build_findings`, `append_round`, `update_finding_disposition`) — internal to operations
- Path helpers (`index_path`, `findings_path`, `bootstrap_path`) — internal to operations
- Git helpers (`get_head_sha`, `get_changed_files`) — internal to operations
- Config loading — caller's responsibility, or operations load it themselves

This is a deliberate choice: the public surface exposes *operations* (things you want to do) and *types* (things you need to understand the results), not *building blocks* (things you'd need to assemble correctly).

## 6. Design Decisions

**DEC-124-001**: New `spec_driver` package rather than `supekku.public` or `supekku.api`.
- Product name is "spec-driver", not "supekku". Public imports use the product namespace.
- Anticipates eventual full migration without coupling to it.

**DEC-124-002**: Extract orchestration, not just re-export building blocks.
- Building-block-only export forces consumers to replicate load-bearing orchestration (guards, transitions, write ordering).
- Violates POL-001 (the orchestration would exist in two places).
- Adversarial review (DR-124 review F1) identified this as a blocking issue.

**DEC-124-003**: Typed result dataclasses for each operation.
- Callers get structured returns, not dicts or tuples.
- `@dataclass` not `BaseModel` — these are simple value objects, no validation needed.
- Fields mirror the JSON envelope `data` payloads from DE-108 for consistency.

**DEC-124-004**: Flat export from `workflow/__init__.py`.
- ~40 symbols. Below the threshold for sub-module split.
- Split into submodules if/when surface exceeds ~50 symbols.

**DEC-124-005**: Domain functions raise exceptions, not exit codes.
- CLI maps exceptions to exit codes (presentation concern).
- Programmatic callers catch typed exceptions.
- New `ReviewApprovalGuardError` and `FindingNotFoundError` replace exit-code-based control flow.

## 7. Known Limitations

1. **Transitive import cascade**: Importing any `supekku.scripts.lib.*` module loads ~77 modules via the god-init (`supekku/scripts/lib/__init__.py`). Cost: ~100ms. Resolves when DE-114 lands. Does not affect correctness.

2. **`_pytest`/`pytest` analogy is imprecise**: This is a namespace facade with extracted operations, not a separate abstraction layer. The stability guarantee is "the re-export file is the contract" — signature changes in underlying functions do propagate. Consumers should pin to the operations, not reach past them.

3. **`_resolve_delta_dir` stays in CLI**: Operations take `delta_dir: Path`, not `delta_id: str`. Callers (including autobahn) are responsible for resolving delta ID → directory path. This keeps operations independent of spec-driver's directory layout conventions. If this becomes friction, a `resolve_delta_dir(delta_id, root)` helper can be added to the public surface later.

## 8. Verification

- **Extraction correctness**: Existing CLI integration tests (`workflow_review_test.py`, `workflow_integration_test.py`) exercise the full orchestration paths. They must pass without modification after extraction — this proves behavioural equivalence.
- **Domain function unit tests**: New tests in `operations_test.py` for each composed operation, including guard violations and error paths.
- **Import smoke test**: Every symbol in `__all__` is importable and is `is`-identical to its source.
- **`just check`** passes.
