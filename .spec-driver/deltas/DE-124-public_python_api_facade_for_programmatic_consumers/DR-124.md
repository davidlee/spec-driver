# DR-124 – Public Python API facade

## 1. Executive Summary

- **Delta**: [DE-124](./DE-124.md)
- **Status**: draft
- **Last Updated**: 2026-03-23
- **Synopsis**: Extract review orchestration logic from CLI commands into domain functions, expose them through a `spec_driver.workflow` public API package. CLI becomes a thin formatting layer.

## 2. Problem & Constraints

- **Current Behaviour**: Review operations (`prime`, `complete`, `disposition`, `teardown`) are orchestrated in CLI command functions (120–150 lines each). The orchestration includes load-bearing guards, state transitions, staleness cascade logic, and write ordering. The underlying building blocks (`review_io`, `review_state_machine`, `state_io`, `staleness`) are primitives that must be composed correctly.
- **Drivers**:
  - autobahn needs typed Python access to review operations (not building blocks)
  - Exporting only building blocks forces autobahn to replicate orchestration logic, violating POL-001
  - CLI commands currently violate the skinny CLI pattern (mem.pattern.cli.skinny)
- **Constraints**:
  - Existing CLI behaviour must be preserved (exit codes, JSON envelope, human output)
  - All existing CLI tests must pass without modification
  - No changes to workflow file formats or schemas
  - No new external dependencies
- **Out of Scope**: Non-review workflow operations, god-init fix (DE-114), internal rename

## 3. Architecture Intent

- **Target Outcomes**:
  - Orchestration logic lives in domain layer, testable without CLI
  - CLI commands are thin wrappers: parse args → call domain → format output
  - autobahn imports `spec_driver.workflow.prime_review(...)` — a composed operation, not a bag of primitives
  - Guards (approval, state transition, disposition validation) enforced in exactly one place (domain function)
- **Guiding Principles**:
  - Skinny CLI pattern: CLI does parsing and formatting, domain does everything else
  - One canonical orchestration path: CLI and autobahn call the same function
  - Exceptions for errors, not exit codes — CLI maps exceptions to exit codes
  - Typed end-to-end: enum parameters, enum result fields, typed exceptions
- **State Transitions**: No changes to review or workflow state machines. Operations call existing transitions.

## 4. Extraction Design

### 4.1 New module: `supekku/scripts/lib/workflow/operations.py`

Five composed operations extracted from CLI. One new query operation.

#### `resolve_delta_dir`

Extracted from `review_prime_command` (lines 844–1058). Loads its own config from `repo_root` internally (DEC-124-007). Includes:

#### `complete_review`

    new_state: WorkflowState

Extracted from `review_complete_command` (lines 1241–1448). Includes:
- State read + validation
- Workflow transition validation (`apply_transition`)
- Approval guard: `collect_blocking_findings` + `can_approve` (raises on violation)
- Round build/append (v2 accumulative model)

Raises: `StateNotFoundError`, `StateValidationError`, `ReviewApprovalGuardError` (new), `TransitionError`, `FindingsValidationError`

#### `disposition_finding`

```python
@dataclass
class DispositionResult:
    delta_id: str
    finding_id: str
    action: FindingDispositionAction
    previous_status: FindingStatus
    new_status: FindingStatus

class DispositionValidationError(Exception):
    """Raised when disposition parameters violate domain constraints."""

def disposition_finding(
    delta_dir: Path,
    finding_id: str,
    *,
    action: FindingDispositionAction,
    authority: DispositionAuthority = DispositionAuthority.AGENT,
    rationale: str | None = None,
    backlog_ref: str | None = None,
    resolved_at: str | None = None,
    superseded_by: str | None = None,
) -> DispositionResult:
```

Extracted from `_disposition_finding` (lines 1503–1589). Uses typed keyword arguments, not a raw dict (DEC-124-010). Includes:
- **Domain validation before write** (DEC-124-011, per DR-109 §3.4 validity constraints):
  - `waive` requires `rationale`
  - `defer` requires `rationale`; blocking findings additionally require `backlog_ref` to unblock
  - `supersede` requires `superseded_by`
  - `fix` on blocking findings requires `resolved_at`
  - Authority constraints on blocking findings: `waive` and `defer` require `authority: user` to unblock
  - Non-blocking findings: only `rationale` (waive/defer) and `superseded_by` (supersede) enforced
- Read findings + version check
- Capture previous status
- Build disposition dict from typed kwargs
- Apply disposition
- Write findings with validation

Raises: `DispositionValidationError` (new), `FindingsNotFoundError`, `FindingsVersionError`, `FindingsValidationError`, `FindingNotFoundError` (new)

#### `summarize_review`

```python
@dataclass
class ReviewSummary:
    current_round: int
    judgment_status: ReviewStatus
    blocking_total: int
    blocking_dispositioned: int
    non_blocking_total: int
    all_blocking_resolved: bool   # can_approve() would pass
    outcome_ready: bool           # judgment not in {not_started, in_progress}

def summarize_review(
    delta_dir: Path,
) -> ReviewSummary:
```

**New operation** — the read-only query counterpart to the mutation operations (DEC-124-012). No equivalent in the current CLI. Encapsulates:
- Read findings (via `read_findings`)
- Read review-index (via `read_review_index`, for `judgment_status`)
- Count findings by category (blocking/non-blocking) and disposition status
- Call `collect_blocking_findings` + `can_approve` for the approval gate check
- Derive `outcome_ready` from judgment status

Raises: `FindingsNotFoundError`, `FindingsVersionError`, `ReviewIndexNotFoundError`

This eliminates the need for autobahn to import `read_findings`, `read_review_index`, `collect_blocking_findings`, or `can_approve` individually. The query operation answers "what happened in the review?" as a single call.

## 5. Public Surface: `spec_driver.workflow`

### 5.1 Package structure

```
spec_driver/
    __init__.py                 # version only
    workflow/
        __init__.py             # curated re-exports + __all__
```

### 5.2 Design principle: types are public, not just functions

Programmatic consumers need the types, not just the operations. autobahn currently maintains **parallel copies** of spec-driver's enums and models (`autobahn/models/enums.py`, review-related models in `autobahn/models/artifacts.py`) — cited transcriptions kept in sync by contract tests. This was necessary when the integration was CLI subprocess. With a Python import path, these duplicates become pure debt:

- Every new type requires transcription + contract test
- Two `FindingDisposition` models with identical fields violates POL-001
- Type mismatches between spec-driver operations and autobahn models create bridging friction

The public surface exports types as first-class citizens alongside operations. Consumers import one set of types. No duplication, no bridging.

| autobahn StrEnum | spec-driver source |
|---|---|

These are not part of DE-124's scope — they're metadata schema values, not workflow domain types. Promoting them to StrEnums in spec-driver is a small follow-up. Until then, autobahn keeps its own StrEnum definitions for these.

### 5.5 What is NOT exported

- Write functions (`write_state`, `write_review_index`, `write_findings`) — mutations go through composed operations
- Building block functions (`build_review_index`, `build_findings`, `append_round`, `update_finding_disposition`, `find_finding`, `next_round_number`) — internal to operations
- Read functions for review artifacts (`read_findings`, `read_review_index`) — encapsulated by `summarize_review` and the mutation operations
- Path helpers (`index_path`, `findings_path`, `bootstrap_path`) — internal to operations
- Git helpers (`get_head_sha`, `get_changed_files`) — internal to operations
- Config loading — operations load their own config (DEC-124-007)

This is a deliberate choice: the public surface exposes *operations* (things you want to do), *types* (things you need to understand the domain), and *exceptions* (things you need to handle) — not *building blocks* (things you'd need to assemble correctly).

### 5.6 Impact on autobahn

When autobahn DE-004 consumes this surface:

- `autobahn/models/enums.py`: Delete review-related enums (`BootstrapStatus`, `ReviewStatus`, `FindingStatus`, `FindingDispositionAction`, `DispositionAuthority`). Import from `spec_driver.workflow`. Keep autobahn-only enums (`SessionStatus`, `Role`, etc.) until spec-driver promotes them.
- `autobahn/models/artifacts.py`: Delete `FindingDisposition`, `Finding` (use `spec_driver.workflow.FindingDisposition`, `ReviewFinding`). Review-related file models (`ReviewFindingsFile`, `ReviewIndexFile`) may still be needed if autobahn's artifact loader parses these files independently — but with operations handling mutations and `summarize_review` handling queries, the loader's role shrinks to read-only context assembly.
- `summarize_review_outcome`: Wraps `spec_driver.workflow.summarize_review(delta_dir)`. One call, shared types, no building blocks needed.
- `disposition_finding`: Wraps `spec_driver.workflow.disposition_finding(...)`. Domain validation (waive needs rationale, etc.) handled by spec-driver. autobahn doesn't reimplement.
- Contract tests: Review type transcription tests deleted. Remaining artifact model contract tests retained.
- `ToolInvocationError`, `ToolContractError`: Dead code — delete.

## 6. Design Decisions

**DEC-124-001**: New `spec_driver` package rather than `supekku.public` or `supekku.api`.
- Product name is "spec-driver", not "supekku". Public imports use the product namespace.
- Anticipates eventual full migration without coupling to it.

**DEC-124-002**: Extract orchestration, not just re-export building blocks.
- Building-block-only export forces consumers to replicate load-bearing orchestration (guards, transitions, write ordering).
- Violates POL-001 (the orchestration would exist in two places).
- Adversarial review (DR-124 review, initial round) identified this as a blocking issue.

**DEC-124-003**: Typed result dataclasses with enum fields, not raw strings.
- `PrimeResult.bootstrap_status: BootstrapStatus`, not `str`. `CompleteResult.outcome: ReviewStatus`, not `str`.
- Consumers don't do string comparisons against magic values — they pattern-match on enums.
- New `PrimeAction` StrEnum for the 3-value action field. Consistent with POL-002 (no magic strings).
- Adversarial review (cross-review F1/F7) identified raw `str` as a DEC-124-006 contradiction.

**DEC-124-004**: Flat export from `workflow/__init__.py`.
- 52 symbols. At the upper threshold but still manageable in a single module.
- Split into submodules if/when surface exceeds ~60 symbols.

**DEC-124-005**: Domain functions raise exceptions, not exit codes.
- CLI maps exceptions to exit codes (presentation concern).
- Programmatic callers catch typed exceptions.
- New `ReviewApprovalGuardError`, `FindingNotFoundError`, `DispositionValidationError` replace exit-code-based control flow.

**DEC-124-006**: Types are first-class public exports — consumers import, not duplicate.
- Enums (`BootstrapStatus`, `ReviewStatus`, etc.), Pydantic models (`FindingDisposition`, `ReviewFinding`), and exception classes are all exported alongside operations.
- Exception classes are part of the public contract — consumers need them for typed `except` handling. `StateNotFoundError`, `ReviewTransitionError`, `FindingsVersionError` etc. are as important to import as the enums. Without them, callers are forced to catch bare `Exception` or duplicate error hierarchies.
- Consumers (autobahn) delete their parallel type copies and import from `spec_driver.workflow`.
- This eliminates: transcription effort, contract sync tests for duplicated types, type-bridge code at integration boundaries.
- Non-review enums that spec-driver only has as raw value lists (e.g. `Role`, `PhaseStatus`) are out of scope — autobahn keeps its own StrEnums for those until spec-driver promotes them.

**DEC-124-007**: Operations load their own config — no config parameter leakage.
- `prime_review` and `complete_review` need workflow config (e.g. teardown policy, bootstrap config). They load it internally from `repo_root` via `load_workflow_config`.
- No `config: dict | None` parameter on the public surface. Consumers shouldn't need to know about `workflow.toml` structure.
- Adversarial review (cross-review F6) identified the config param as an internal implementation detail leaking through.

**DEC-124-008**: `resolve_delta_dir` exported as a public utility.
- Every consumer that has a `delta_id: str` needs to resolve it to `delta_dir: Path`.
- The resolution logic is non-trivial (case-insensitive match, slug handling, error on ambiguity).
- Keeping it private forces consumers to reimplement — exactly the kind of primitive-assembly DEC-124-002 argues against.
- Adversarial review (cross-review F4) identified this as a gap.

**DEC-124-009**: `complete_review` teardown is opt-in, not implicit.
- `auto_teardown: bool = True` parameter. Default True preserves CLI backward compat.
- Programmatic callers can pass `auto_teardown=False` to avoid surprise file deletion.
- When teardown occurs, `CompleteResult.removed_files` reports what was deleted.
- Adversarial review (cross-review F3) identified implicit teardown as a side-effect surprise.

**DEC-124-010**: `disposition_finding` uses typed keyword arguments, not a raw dict.
- Explicit kwargs: `action: FindingDispositionAction`, `authority: DispositionAuthority`, `rationale: str | None`, etc.
- The internal `update_finding_disposition` takes `dict` because it operates on raw YAML. The public API shouldn't expose that.
- Consumers get IDE completion, type checking, and clear parameter documentation.
- Adversarial review (cross-review F2/F8) identified the untyped dict as a boundary violation.

**DEC-124-011**: Disposition domain validation lives in spec-driver.
- Constraints like "waive requires rationale" and "defer requires backlog_ref" are review domain rules (DR-109), not orchestration policy.
- They belong in `operations.py:disposition_finding()`, not in consumers.
- New `DispositionValidationError` for constraint violations.
- Adversarial review (cross-review F3) identified ownership as ambiguous — resolved to spec-driver.

**DEC-124-012**: `summarize_review` is a read-only query operation.
- Mutation operations: `prime_review`, `complete_review`, `disposition_finding`, `teardown_review`.
- Query operation: `summarize_review` — the read counterpart.
- Without this, consumers must import building blocks (`read_findings`, `collect_blocking_findings`, `can_approve`) to answer "what happened in the review?" — violating the operations-not-primitives principle.
- Adversarial review (cross-review F2/F5/F6) identified this as a blocking gap from both sides.

## 7. Known Limitations

1. **Transitive import cascade**: Importing any `supekku.scripts.lib.*` module loads ~77 modules via the god-init (`supekku/scripts/lib/__init__.py`). Cost: ~100ms. Resolves when DE-114 lands. Does not affect correctness.

2. **Facade is a contract, not an isolation layer**: Signature changes in underlying functions propagate through re-exports. The stability guarantee is "the `__all__` list is the contract and changes are caught by import smoke tests." Consumers should use the operations, not reach past them into `supekku.*`.

## 8. Verification

- **Extraction correctness**: Existing CLI integration tests (`workflow_review_test.py`, `workflow_integration_test.py`) exercise the full orchestration paths. They must pass without modification after extraction — this proves behavioural equivalence.
- **Domain function unit tests**: New tests in `operations_test.py` for each composed operation, including:
  - `prime_review`: cold start, rebuild, refresh scenarios
  - `complete_review`: approve (with/without auto_teardown), changes_requested, approval guard violation
  - `disposition_finding`: each action type, validation failures, finding-not-found
  - `summarize_review`: various finding/round states, empty findings, no findings file
  - `resolve_delta_dir`: found, not found, case-insensitive match
- **Import smoke test**: Every symbol in `__all__` is importable and is `is`-identical to its source.
- **`just check`** passes.
