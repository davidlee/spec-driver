## 1. Summary & Context

- **Design Revision**: [DR-125](./DR-125.md)
- **Change Drivers**: The existence of 10+ refactoring deltas (e.g. DE-114, DE-124) addressing symptoms of structural decay. `supekku/scripts/lib` has become an obfuscatory dumping ground with a tangled internal dependency graph.
- **Prerequisites**: DE-107 through DE-113 (completed). These deltas established the Pydantic models and cleaned up basic technical debt, preparing the ground for the layer re-homing.

## 2. Motivation

The `spec-driver` internal architecture suffers from a lack of explicit coupling constraints. Modules import from one another indiscriminately, leading to:
- The "god-init" problem (importing one file recursively loads the entire application).
- Difficulty exposing a clean public API (DE-124) because domain logic is entangled with CLI/TUI concerns.
- A flat, confusing file hierarchy (`supekku/scripts/lib/*`) that hides the intended system architecture.

Before executing symptomatic refactorings, we must define the structural blueprint and enforce one-way dependency rules.

## 3. Scope & Objectives

- **Primary Outcomes**:
  - Define strict, one-way architectural layers (Core, Models, Domain/Registry, Orchestration, Presentation).
  - Draft and approve POL-003 to formalize coupling constraints (a layer may only import from itself or layers below it).
  - Re-home and rename the obfuscatory `supekku/scripts/lib` paths into an explicit, layer-aligned package structure (e.g., `spec_driver/core/`, `spec_driver/domain/`).
  - Separate fast unit tests (sitting alongside their target code) from slow integration tests (re-homed to `tests/integration/` and invoked via a separate command).
- **Operational Constraints**:
  - The refactoring must be mechanically verifiable (e.g., via `import-linter` or explicit CI checks).
  - Must not change the external CLI behaviour or workflow artifact schemas.
- **Dependencies**:
  - DE-107 through DE-113 are completed. These solidified the `models/` layer and cleaned up orphaned/dead code.

## 4. Out of Scope

- Changes to artifact schemas.
- Changing CLI output formats.
- Rewriting business logic (this is a structural move/import refactor, not a behaviour change).

## 5. Approach Overview

- **Phase 1: Blueprint & Policy**: Thrash out the layer definitions in DR-125 and ratify POL-003.
- **Phase 2: Establish API Shell & Boundaries**: Create the `spec_driver/__init__.py` and `spec_driver/orchestration/__init__.py` public APIs. Re-import necessary types/functions from `supekku` to immediately unblock `autobahn` (DE-124). Establish explicit `__all__` exports in `spec_driver` internal layers.
- **Phase 3: Enforcement**: Configure `import-linter` in `pyproject.toml` to rigidly enforce the layers for `spec_driver/`. Set `pylint` to aggressively fail for `spec_driver/` while remaining lenient on `supekku/`.
- **Phase 4: The Strangler Fig Move**: Move individual files from `supekku/` to `spec_driver/` one by one. Fix their import graphs as they move, utilizing the strict internal boundaries.
- **Phase 5: Cleanup**: Eradicate `supekku/` once empty. This naturally resolves the god-init problem (DE-114).

## 6. Verification Strategy

- **Requirements Coverage**: CI must enforce the layer dependency graph (upward imports break the build).
- **Acceptance Criteria**:
  - `supekku/scripts/lib` no longer exists.
  - The codebase is divided into explicit layers.
  - `pytest` passes with all existing tests.
  - A linter tool enforces the coupling constraints defined in POL-003.

## 7. Risks & Mitigations

- **Risk**: Massive merge conflicts for parallel deltas (e.g. DE-001, DE-107).
  - _Mitigation_: Freeze non-essential feature development. Land DE-107 first. Execute the file moves mechanically and quickly.

## 8. Follow-ups & Tracking

- **Future Deltas**: Unblocks DE-114 (god-init) and DE-124 (Public API Facade).
