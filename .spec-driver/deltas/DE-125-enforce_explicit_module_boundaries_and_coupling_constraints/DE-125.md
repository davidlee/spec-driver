## 1. Summary & Current State

- **Design Revision**: [DR-125](./DR-125.md)
- **Implementation Plan**: [IP-125](./IP-125.md)
- **Phase Sheet**: [phase-01.md](./phases/phase-01.md)
- **Governing Policy**: [POL-003](../../policies/POL-003-module-boundaries.md)

This delta no longer proposes creating `spec_driver/` or introducing `import-linter`
from scratch. Both already exist in the repository. The remaining work is to make
the architecture enforceable and migration-ready by:

- tightening the top-level layering story to match the current repo state
- defining **internal contracts inside `spec_driver.domain`** so "domain" does not
  become a renamed hairball
- sequencing migration away from `supekku/scripts/lib` through narrow pilots
  rather than a big-bang move
- aligning test layout and verification commands with the new package boundaries

## 2. Motivation

The original problem statement still stands: `supekku/scripts/lib/*` hides
responsibility boundaries and allows modules to import laterally and upward with
minimal friction. The repo now has enough new structure for the next problem to
be more specific:

- the **outer** layer contract exists, but only at a coarse level
- the **inner** `domain` structure is still unresolved
- several legacy modules currently combine local authority, cross-artifact
  traversal, and whole-workspace validation in the same import cluster

The critical thesis of this delta is therefore narrower and stronger than the
initial draft:

> top-level layering is necessary but insufficient; the architectural win comes
> from making `domain` a directed set of internal contracts rather than a single
> free-for-all package.

## 3. Scope & Objectives

- **Primary Outcomes**
  - Operationalise `POL-003` against the repo as it exists today.
  - Define the internal `spec_driver.domain` contract as ordered sub-areas:
    `lifecycle/records -> registries -> relations -> validation`.
  - Establish migration rules so sibling registries do not directly import one
    another as a default integration mechanism.
  - Pilot the first migration steps in places where dependency direction is
    already reasonably clear.
  - Align verification so `import-linter`, test topology, and plan artefacts all
    describe the same architecture.
- **Operational Constraints**
  - No deliberate CLI behaviour changes.
  - No workflow artifact schema changes.
  - Migration should proceed incrementally alongside other deltas.
- **Deliberate Non-Linkage**
  - `applies_to.specs` and `applies_to.requirements` remain empty for now because
    this delta is governing structural policy and migration sequencing rather than
    introducing a product-facing requirement. Closure should reassess whether
    architecture specs or standards need explicit linkage.

## 4. Out of Scope

- Completing the entire `supekku/` to `spec_driver/` migration in one delta.
- Redesigning workflow semantics or user-facing CLI output.
- Treating every legacy helper as domain code. Several current modules likely
  belong in orchestration, presentation, or core utility packages instead.

## 5. Approach Overview

- **Phase 1: Reconcile Architecture Story**
  - Rewrite DE/DR/IP so they match repo reality.
  - Define the internal `domain` contracts and migration rules.
  - Identify the first concrete coupling seams to break.
- **Phase 2: Enforce Internal Boundaries**
  - Extend `import-linter` beyond the current top-level layer contract.
  - Add domain-internal contracts and, where needed, independence contracts for
    presentation siblings.
- **Phase 3: Pilot Migration**
  - Move low-risk modules and establish the sanctioned seams:
    local registries, relation/query services, and validation.
  - Break the worst sibling-registry imports before broader re-homing.
- **Phase 4: Broader Migration & Cleanup**
  - Continue strangler-fig moves as modules become quiet.
  - Reconcile test layout, verification commands, and remaining `supekku/`
    surface area.

## 6. Verification Strategy

- **Architecture Contracts**
  - `import-linter` must enforce the top-level package layers.
  - `import-linter` must also enforce the internal `domain` order defined by
    DR-125.
- **Migration Readiness**
  - Phase work must identify which legacy modules are local authorities, which
    are cross-artifact relation services, and which are orchestration concerns.
- **Test Topology**
  - The plan must explicitly reconcile the current `pytest` configuration, which
    still targets `supekku/*`, with the intended `spec_driver/` and
    `tests/integration/` layout.
- **Acceptance Criteria**
  - `DE-125`, `DR-125`, `IP-125`, and the active phase sheet describe the same
    migration thesis and sequence.
  - The repo contains a domain-internal contract design that can be enforced in
    CI.
  - The first migration targets are chosen based on dependency direction rather
    than package rename enthusiasm.

## 7. Risks & Mitigations

- **Risk**: `domain` becomes a catch-all label that preserves lateral coupling.
  - _Mitigation_: treat `domain` as an ordered set of contracts, not a single
    package bucket.
- **Risk**: registry modules continue to act as integration hubs.
  - _Mitigation_: push cross-artifact traversal into `relations` or orchestration
    services; keep registries as local authorities.
- **Risk**: verification prose outruns repo reality again.
  - _Mitigation_: treat changes to `pyproject.toml`, test layout, and phase
    artefacts as part of the same tracked work rather than deferred narrative.

## 8. Follow-ups & Tracking

- **Immediate follow-up**: execute [phase-01.md](./phases/phase-01.md) to lock
  the contract shape before broader module moves.
- **Parallel coordination**: keep DE-124 moving against the orchestration facade
  while DE-125 clarifies and enforces internal boundaries.
