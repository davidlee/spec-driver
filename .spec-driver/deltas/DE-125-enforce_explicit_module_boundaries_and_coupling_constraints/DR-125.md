# DR-125 – Enforce explicit module boundaries and coupling constraints

## 1. Executive Summary

- **Delta**: [DE-125](./DE-125.md)
- **Status**: draft
- **Synopsis**: Reconcile the architecture work already started in `spec_driver/`
  and define the next-order constraint that actually matters: internal domain
  dependency direction.

The first draft of DE-125 over-indexed on creating `spec_driver/` and adding a
top-level layer contract. Both are already present. The unresolved design
problem is smaller and more consequential:

> how do we stop `domain` becoming a renamed copy of the old `lib/` tangle?

This revision answers that by treating the outer layered architecture as a shell
and the internal `domain` contracts as the real anti-coupling mechanism.

## 2. Problem Restatement

Today’s repo has three simultaneous truths:

1. `spec_driver/` already exists.
2. `import-linter` already enforces one top-level `Architectural Layers`
   contract.
3. most real behaviour still lives in `supekku/scripts/lib/*`, where sibling
   packages import one another directly.

That means the remaining risk is not simply "presentation imports domain" or
"models import orchestration". The sharper risk is lateral entanglement inside
what would broadly be called the domain area:

- requirements code reaching directly into backlog and specs registries
- policy/standard code reaching into decision registries
- whole-workspace validation importing across multiple sibling areas

If those move wholesale into `spec_driver.domain`, the repo gets a neater
directory tree but not a more legible dependency graph.

## 3. Architectural Intent

### 3.1 Outer Layers Stay

The top-level ordering remains:

1. `spec_driver.core`
2. `spec_driver.models`
3. `spec_driver.domain`
4. `spec_driver.orchestration`
5. `spec_driver.presentation`

This remains valuable because it forbids upward leaks and preserves a public API
boundary around orchestration.

### 3.2 Domain Is Not One Layer Internally

Inside `spec_driver.domain`, the allowed dependency direction is:

1. `spec_driver.domain.lifecycle`
2. `spec_driver.domain.records`
3. `spec_driver.domain.registries`
4. `spec_driver.domain.relations`
5. `spec_driver.domain.validation`

Read as: lower numbers may not import higher numbers.

#### Lifecycle

- Status sets, transition rules, invariant enums, and local semantic rules.
- Must not import registries or relation/query code.

#### Records

- Artifact record models and narrow parsing/normalisation helpers that are still
  domain-shaped rather than generic infrastructure.
- May depend on lifecycle and shared models/core utilities.

#### Registries

- Local authorities for one artifact family or bounded domain area.
- Responsible for reading, writing, indexing, and querying their own records.
- Must not import sibling registries as a default mechanism for integration.

#### Relations

- Cross-artifact traversal, reference collection, graph walking, and resolution
  helpers that intentionally span bounded areas.
- This is the sanctioned place for "how do these areas connect?" logic.

#### Validation

- Whole-workspace or cross-area consistency checking.
- May depend on registries and relations because it is deliberately at the top of
  the domain-internal ordering.

### 3.3 Registry Rule

This is the key design choice.

Registries are **local authorities**, not integration hubs.

A registry may expose local record access, discovery, mutation, and local
consistency checks. It should not routinely import sibling registries to answer
cross-artifact questions. If a question spans multiple authorities:

- prefer `domain.relations` for traversal/resolution
- prefer orchestration when the work is multi-step, side-effectful, or workflow
  oriented

### 3.4 Cross-Layer Placement Heuristics

The current legacy tree mixes several concerns. Use these heuristics when
deciding whether a module belongs in domain at all:

- **Local authority over one artifact family**: usually `domain.registries`
- **Cross-artifact graph logic**: usually `domain.relations`
- **Whole-workspace checking**: usually `domain.validation`
- **State transitions, syncs, multi-step workflows, side effects**: usually
  `orchestration`
- **Pure shared parsing/formatting/utilities**: usually `core`
- **Terminal/UI formatting**: `presentation`

## 4. Legacy Module Mapping

The mapping below is directional guidance, not a claim that all files should
move immediately.

### 4.1 Likely Domain Foundations

- `supekku/scripts/lib/*/lifecycle.py` -> `domain.lifecycle/*` or shared domain
  lifecycle modules
- artifact record types that are truly domain-local rather than global models ->
  `domain.records/*`

### 4.2 Likely Local Registries

- `supekku/scripts/lib/specs/registry.py`
- `supekku/scripts/lib/backlog/registry.py`
- `supekku/scripts/lib/requirements/registry.py`
- `supekku/scripts/lib/policies/registry.py`
- `supekku/scripts/lib/decisions/registry.py`

These should be moved only after their sibling-registry imports are reduced or
redirected.

### 4.3 Likely Relations Layer

- `supekku/scripts/lib/relations/query.py`
- `supekku/scripts/lib/relations/graph.py`
- `supekku/scripts/lib/relations/manager.py`

These are the natural landing zone for cross-artifact traversal that is
currently smeared across registries and validators.

### 4.4 Likely Validation Layer

- `supekku/scripts/lib/validation/validator.py`

Because it already coordinates across multiple authorities, it belongs above
registries and relations rather than beside them.

### 4.5 Likely Not Domain

- `supekku/scripts/lib/sync/*` -> orchestration
- `supekku/scripts/lib/changes/*` -> orchestration
- terminal-facing formatters -> presentation
- generic pure helpers -> core

## 5. First Couplings To Break

The first implementation targets should be chosen for architectural leverage,
not for rename convenience.

### 5.1 Requirements Registry

Current smell: `requirements.registry` imports both backlog and specs registry
surfaces. That makes one registry responsible for cross-area integration.

Target fix:

- keep local requirement record/index behaviour in the registry
- move cross-artifact traversal into `domain.relations`
- allow orchestration to compose requirement + spec + backlog workflows when
  mutation or sequencing is involved

### 5.2 Policies / Decisions Coupling

Current smell: policy and standard registries reach into decisions registries to
resolve related authority.

Target fix:

- represent local record ownership in each registry
- move cross-authority joins into shared relation/query code

### 5.3 Validation as a Top Consumer

Current smell: validator imports multiple sibling areas directly because it is
effectively acting as a top consumer.

Target fix:

- make that status explicit
- allow `validation` to sit at the top of the domain-internal order rather than
  pretending it is a peer of registries

## 6. Automated Enforcement

The current `pyproject.toml` contract is a good start:

```toml
[[tool.importlinter.contracts]]
name = "Architectural Layers"
type = "layers"
layers = [
    "spec_driver.presentation",
    "spec_driver.orchestration",
    "spec_driver.domain",
    "spec_driver.models",
    "spec_driver.core",
]
```

DE-125 should extend that with the next contract, once enough code has moved:

```toml
[[tool.importlinter.contracts]]
name = "Domain Internal Layers"
type = "layers"
layers = [
    "spec_driver.domain.validation",
    "spec_driver.domain.relations",
    "spec_driver.domain.registries",
    "spec_driver.domain.records",
    "spec_driver.domain.lifecycle",
]
```

Additional independence contracts may be needed for presentation siblings and
possibly for unrelated registry families once the initial DAG exists.

## 7. Rollout Sequence

1. **Reconcile docs and contracts**
   - make DE/IP/phase agree on the domain-internal thesis
   - identify the first migration targets and stop conditions
2. **Harden verification**
   - extend `import-linter`
   - reconcile test topology and commands with the intended package structure
3. **Move low-coupling foundations**
   - lifecycle and record-level code first
4. **Extract sanctioned seams**
   - relation/query helpers before sibling registry moves
5. **Migrate top consumers**
   - validation once supporting seams exist
6. **Broaden migration incrementally**
   - only move registry families when dependency direction is clear

## 8. Consequences

### Positive

- import graphs begin to communicate ownership
- registries become easier to reason about in isolation
- DE-124 can rely on a cleaner orchestration boundary without waiting for a
  whole-repo move

### Negative / Cost

- some files will need partial decomposition before they can move
- migration sequencing becomes more deliberate and slower than a bulk rename
- import-linter contracts will need staged adoption to avoid blocking work

## 9. Readiness Statement

Open implementation questions do remain, but they are now conscious
assumptions rather than hidden ambiguity:

- exact module-by-module placements will be resolved during migration pilots
- some current "domain" code will be reclassified into orchestration/core
- the domain-internal order may need refinement once the first pilots land

That is acceptable. The design is now specific enough to guide implementation
without pretending that package creation alone solves lateral coupling.
