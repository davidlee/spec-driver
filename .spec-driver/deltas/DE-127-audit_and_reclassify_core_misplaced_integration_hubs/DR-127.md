# DR-127 – Audit and reclassify core/ misplaced integration hubs

## 1. Executive Summary

- **Delta**: [DE-127](./DE-127.md)
- **Status**: draft
- **Synopsis**: Classify every module in `supekku/scripts/lib/core/` by its
  actual dependency profile and architectural role, then move the two modules
  that violate the layering out of core.

## 2. Problem Statement

`spec_driver.core` sits at the bottom of the outer layer contract:

```
presentation > orchestration > domain > models > core
```

For this to be enforceable, core must have no upward or lateral dependencies.
Today, two modules in `core/` break this:

| Module | Cross-area imports | Role |
|--------|-------------------|------|
| `artifact_view.py` | 10 (all registry families) | Projection service: normalises registries to uniform `ArtifactEntry` view |
| `enums.py` | 11 (all lifecycle areas) | Aggregation service: builds `ENUM_REGISTRY` for CLI status validation |

The remaining 22 modules have zero cross-area imports and are genuinely core.

### 3.2 Misplaced — reclassify

| Module | Current layer | Correct layer | Rationale |
|--------|-------------|---------------|-----------|
| `artifact_view.py` | core | orchestration | Composes 10 registries; consumers are TUI and CLI |
| `enums.py` | core | orchestration | Aggregates lifecycle constants from 11 areas for CLI introspection |

## 4. Reclassification Design

### 4.1 `artifact_view.py` → orchestration

**Target**: `spec_driver/orchestration/artifact_view.py`

This module defines `ArtifactType`, `ArtifactEntry`, `ArtifactSnapshot`, and
functions that resolve artifact IDs across all registries. It's a read-only
projection service — no mutation, no side effects. But it composes 10 registries,
making it architecturally an orchestration concern.

**Consumers** (7 sites, all lazy or TYPE_CHECKING imports):
- `supekku/tui/browser.py`
- `supekku/tui/search/index.py`
- `supekku/tui/app.py`
- `supekku/tui/widgets/type_selector.py`
- `supekku/tui/widgets/search_overlay.py`
- `supekku/tui/widgets/artifact_list.py`
- `supekku/tui/track.py`

**Migration path**:
1. Copy module to `spec_driver/orchestration/artifact_view.py`.
2. Update registry imports to use canonical paths where available (e.g.
   `spec_driver.domain.relations` for relation types).
3. Leave re-export shim at `supekku/scripts/lib/core/artifact_view.py`.
4. Consumer updates are follow-on (TUI migration is a larger effort).

**Type exports**: `ArtifactType`, `ArtifactGroup`, `ArtifactTypeMeta`,
`ArtifactEntry`, `ArtifactSnapshot` are the key public types. These should be
re-exported from `spec_driver.orchestration.__init__` if they become part of
the public API.

### 4.2 `enums.py` → orchestration

**Target**: `spec_driver/orchestration/enums.py`

This module builds a unified `ENUM_REGISTRY` dict that maps dotted paths
(e.g. `"delta.status"`) to sorted lists of valid values. Each domain area
already exports its own lifecycle constants; `enums.py` aggregates them for
CLI-facing introspection (`spec-driver schema`, `spec-driver edit --status`).

**Consumers** (3 sites):
- `supekku/cli/edit.py` — `validate_status_for_entity`
- `supekku/cli/schema.py` — `get_enum_values`, `list_enum_paths`
- `supekku/scripts/lib/validation/validator.py` — `get_enum_values`

**Migration path**:
1. Copy module to `spec_driver/orchestration/enums.py`.
2. Keep the same import pattern for lifecycle constants (they're leaf imports
   from individual domain areas — the direction is correct even if the paths
   are legacy).
3. Leave re-export shim at `supekku/scripts/lib/core/enums.py`.

**Alternative considered**: decompose into per-domain enum providers. Rejected —
the whole point of this module is centralised aggregation for CLI introspection.
Decomposing it would scatter the registry and lose the single lookup surface.
The aggregation belongs somewhere; orchestration is the right layer for it.

## 5. Verification

- **Cross-area import count**: after reclassification,
  `supekku/scripts/lib/core/` must have zero cross-area imports.
  ```bash
  for f in supekku/scripts/lib/core/*.py; do
    rg "from supekku.scripts.lib." "$f" | grep -v "from supekku.scripts.lib.core" | wc -l
  done
  ```
- **import-linter**: `uvx import-linter lint` — both contracts pass.
- **Tests**: `uv run pytest supekku spec_driver -x` — full pass.
- **Lint**: `uv run ruff check` — clean.

## 6. Consequences

### Positive

- `core/` becomes honestly foundational — zero upward dependencies.
- `spec_driver.core` migration becomes viable without dragging in 10 registries.
- The outer layer contract stops being aspirational.

### Costs

- Two re-export shims added (following DE-125 pattern).
- `spec_driver/orchestration/` gains two modules — acceptable given it already
  contains `operations.py`.

### Risks

- Consumers import `ArtifactType` from core at many TUI sites. Re-export shim
  mitigates; eventual TUI migration will clean up.
- `enums.py` lifecycle imports may break if lifecycle module paths change during
  broader migration. The aggregation pattern is stable; only the import paths
  are fragile.
