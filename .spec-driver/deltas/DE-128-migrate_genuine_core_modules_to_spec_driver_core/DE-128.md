## 1. Summary & Context

- **Design Revision**: [DR-128](./DR-128.md)
- **Implementation Plan**: [IP-128](./IP-128.md)
- **Predecessors**: DE-125 (migration pattern), DE-127 (core audit — confirmed
  22 modules are clean)
- **Governing Policy**: [POL-003](../../policies/POL-003-module-boundaries.md)

DE-127 reclassified the two misplaced integration hubs and confirmed that the
remaining 22 `core/` modules have zero cross-area imports. This delta migrates
them to `spec_driver.core` in dependency order, making the outer layer contract
truthful.

## 2. Motivation

The outer layer contract says `spec_driver.core` is the foundation. Today it
contains one module (`file_ops.py`). The actual core infrastructure — 22 modules
totalling ~2,800 lines — still lives in `supekku/scripts/lib/core/`. Every
module above core that has migrated to `spec_driver.*` (e.g.
`domain/relations/manager.py`) imports core via legacy paths, creating an
architecturally backwards dependency.

Migrating core makes the layer contract honest and eliminates the legacy-core
import debt documented in DE-125.

> **Re-validated 2026-05-31** (drafted 2026-03-24, never executed). Inventory
> grew 22 → 26 genuine modules; `enums`/`artifact_view` are now shims (already
> migrated); `spec_utils`/`config` grew upward deps now decoupled in-scope. See
> DR-128 §2.4–2.5 and `notes.md`.

## 3. Scope & Objectives

- **Primary Outcomes**:
  - All 26 genuine core modules live in `spec_driver/core/` (`slugify` folds into
    the existing `string_utils.py`).
  - `spec_utils` and `config` decoupled from their upward deps (DR-128 §2.4):
    `dump_markdown_file_create` relocated to `orchestration.templates`;
    `load_workflow_config` takes injected `known_kinds`.
  - Re-export shims at legacy locations preserve existing consumers.
  - Migrated domain modules (`spec_driver/domain/relations/`) import from
    `spec_driver.core` instead of legacy paths.
  - `import-linter` contracts pass after every phase; core provably free of
    upward deps; outer layer contract is truthful.
- **Operational Constraints**:
  - No CLI behaviour changes, except the deliberate strict-kind warning scope
    change (DR-128 §2.4: warning fires only at validating entrypoints).
  - Re-export shims at legacy `supekku/scripts/lib/core/` for all moved modules.
  - Migration proceeds in dependency-tier order (leaves first).
- **Dependencies**:
  - DE-127 (completed) — confirmed original modules are clean.

## 4. Out of Scope

- Migrating modules outside `core/` (registries, formatters, etc.).
- `enums.py`, `artifact_view.py` — already pure shims to
  `spec_driver.orchestration` (DE-127); leave in place.
- `frontmatter_metadata/` subpackage — domain knowledge, not core (POL-003);
  belongs to a domain migration delta.
- Updating `spec_driver/orchestration/` legacy-core imports (follow-on).
- Retiring the re-export shims (tracked as separate debt).

> Note: migrating each module's `*_test.py` with it **is** in scope — required by
> POL-003 Scope and matching the proven pattern. (Corrected 2026-05-31; the
> original "tests stay legacy" line was drift. See DR-128 §4.)

## 5. Approach Overview

### Internal dependency graph

```
Tier 0 (zero deps — true leaves) — 16:
  repo, strings(→string_utils), dates, ids, io, filters, relation_types,
  artifact_ids, version, cli_utils, editor, git, go_utils, npm_utils,
  pylint_report, frontmatter_schema

Tier 1 (depends on tier 0) — 4:
  paths (→ repo)
  spec_utils (→ frontmatter_schema) [DECOUPLE: relocate dump_markdown_file_create]
  frontmatter_writer (→ spec_utils)
  events (→ repo, paths — lazy imports)

Tier 2 (depends on tier 0 + 1) — 3:
  config (→ paths) [DECOUPLE: inject known_kinds; drop frontmatter_metadata]
  templates (→ paths, jinja2)
  registry_migration (→ git, paths)

Tier 3 (depends on tier 0 + 1 + 2) — 3:
  agent_docs (→ config, paths, templates)
  preboot (→ config, paths)
  sync_preferences (→ config, paths)
```

### Migration strategy

Each tier migrates as a phase. Within each tier, modules move in parallel
(no internal ordering needed). Each moved module:

1. Copies to `spec_driver/core/`.
2. Updates internal imports to use `spec_driver.core.*` relative imports.
3. Gets a re-export shim at the legacy location.
4. Verification runs after each tier.

### Priority target

The frontmatter cluster (`frontmatter_schema`, `spec_utils`) is the
highest-value move because it unblocks the three legacy-core imports in
`spec_driver/domain/relations/`. Both are tier 0/1, so they land in the
first two phases.

## 6. Verification Strategy

- `uvx import-linter lint` — both contracts pass after each phase
- `uv run pytest supekku spec_driver -x` — full pass
- `uv run ruff check` — clean
- Cross-area import audit: `core/` maintains zero cross-area imports
- After tier 1: `spec_driver/domain/relations/` imports updated to
  `spec_driver.core` — domain legacy-core debt eliminated
- Note: `spec_driver/orchestration/` also has legacy-core imports
  (`operations.py`, `artifact_view.py`, plus ~20 test mock-patch paths).
  Updating those is follow-on work, not gated by this delta.

## 7. Follow-ups & Tracking

- **Shim retirement**: all legacy `supekku/scripts/lib/core/` shims should be
  tracked for eventual removal alongside DE-125 relation shims.
- **Domain import cleanup**: after tier 1, update `spec_driver/domain/relations/`
  to import from `spec_driver.core` instead of legacy paths.
- **Test migration**: tests remain at legacy paths; migrating them alongside
  code is a future concern.
