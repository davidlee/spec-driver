# DR-128 – Migrate genuine core modules to spec_driver.core

## 1. Executive Summary

- **Delta**: [DE-128](./DE-128.md)
- **Status**: draft (re-validated 2026-05-31)
- **Synopsis**: Move 26 confirmed-foundational modules from
  `supekku/scripts/lib/core/` to `spec_driver/core/` in dependency-tier order,
  using the established re-export shim pattern. Two modules (`spec_utils`,
  `config`) grew upward dependencies since the original draft and are decoupled
  as part of the migration (see §2.4).

> **Re-validation note (2026-05-31)**: This DR was drafted 2026-03-24 and never
> executed. Two months of feature work drifted the landscape. The inventory grew
> from 22 to 26 genuine modules; `enums`/`artifact_view` became shims (already
> migrated elsewhere); and `spec_utils`/`config` acquired upward dependencies
> that `import-linter` does not yet catch (legacy `supekku/` is unlinted) but
> which will fail the moment those modules land in `spec_driver.core`. Full
> drift analysis in `notes.md`.

## 2. Design

### 2.1 Dependency tiers

Migration proceeds bottom-up. Each tier can be migrated as a batch; modules
within a tier have no mutual dependencies.

### 2.2 Import update rules

- **Within `spec_driver/core/`**: use relative imports (e.g.
  `from .repo import find_repo_root`).
- **From `spec_driver/domain/`**: use absolute imports (e.g.
  `from spec_driver.core.frontmatter_schema import Relation`).
- **Legacy shims**: re-export everything from canonical location.

### 2.3 Domain import debt cleanup

After tier 0 + 1 land, update three files:

| File | Old import | New import |
|------|-----------|------------|
| `spec_driver/domain/relations/manager.py` | `supekku.scripts.lib.core.frontmatter_schema` | `spec_driver.core.frontmatter_schema` |
| `spec_driver/domain/relations/manager.py` | `supekku.scripts.lib.core.spec_utils` | `spec_driver.core.spec_utils` |
| `spec_driver/domain/relations/graph.py` | `supekku.scripts.lib.core.artifact_ids` | `spec_driver.core.artifact_ids` |

This eliminates the "domain imports legacy" debt from DE-125. Confirmed
2026-05-31: `manager.py` imports only `load_markdown_file` +
`dump_markdown_file_update` (both pure, no orchestration), and `graph.py` only
`normalize_artifact_id` — so the debt clears without depending on the decoupled
`dump_markdown_file_create`.

### 2.4 Decoupling upward dependencies (NEW — re-validation)

Two tier-1/2 targets grew upward deps after the original draft. `import-linter`
(`root_package = spec_driver`) does not flag them today because they sit in
unlinted `supekku/`, but each would fail the layer contract the instant the
module lands in `spec_driver.core`. Mechanism chosen per module by blast radius
(DEC-128-004/005).

#### spec_utils → orchestration.templates — RELOCATE

`spec_utils.dump_markdown_file_create` lazily imports
`render_frontmatter_for_kind` / `UnknownKindError` from
`spec_driver.orchestration.templates` (kind-aware frontmatter render). No other
spec_utils function touches orchestration.

- **Move** `dump_markdown_file_create` into `spec_driver/orchestration/templates.py`
  (it already hosts `render_frontmatter_for_kind` — maximal cohesion).
- **Expose a public write primitive in core** (adversarial finding F1): the
  relocated `create` body uses core-private `_normalise_body` + `_atomic_write`.
  Rather than reach into core privates from orchestration, add
  `write_markdown_file(path, fm_yaml: str, body: str)` to
  `spec_driver/core/spec_utils.py` (wrapping normalise + atomic write). The
  relocated `create` becomes: render kind-aware `fm_yaml`
  (`render_frontmatter_for_kind` / `emit_yaml_block` fallback) → call the core
  primitive. Clean split: **core owns write mechanics, orchestration owns
  kind-aware rendering.** `dump_markdown_file_update` is re-expressed on the same
  primitive.
- **Keep pure** in `spec_driver/core/spec_utils.py`: `load_markdown_file`,
  `dump_markdown_file_update`, `write_markdown_file` (new), `_atomic_write`,
  `_normalise_body`, `_extract_inline_comments`, `_read_existing_frontmatter_text`,
  `ensure_list_entry`, `append_unique`, `extract_h1_title`,
  `load_validated_markdown_file`. These depend only on `core.yaml_emit`.
- **12 call sites across 8 files** (all legacy `supekku/` + `sync_specs`, plus
  one `presentation` site) keep working via the legacy `core/spec_utils.py`
  shim, which re-exports the pure funcs from `core` and `dump_markdown_file_create`
  from its new orchestration home.
- *Why relocate not DI*: injecting the renderer would push the
  `orchestration.templates` import into all 8 caller files — spreading the very
  coupling we are removing.

#### config → frontmatter_metadata — DEPENDENCY INJECTION

`config._warn_unknown_strict_kinds` (called eagerly inside `load_workflow_config`)
uses `FRONTMATTER_METADATA_REGISTRY.keys()` (the per-kind domain registry) to
warn on unknown `[validation.strict]` keys. The "deferred to break cycle"
comment is **stale** — `frontmatter_metadata` no longer imports `config`.

- **Signature**: `load_workflow_config(repo_root, known_kinds: set[str] | None = None)`.
  When `known_kinds` is `None`, skip the strict-kind warning.
- Core `config` no longer imports `frontmatter_metadata`.
- **The single validating entrypoint is `presentation/cli/validate/file.py:263`**
  (`get_strict_map(load_workflow_config(repo_root))`) — adversarial finding F3.
  It *already imports* `FRONTMATTER_METADATA_REGISTRY`, so it passes
  `known_kinds=set(FRONTMATTER_METADATA_REGISTRY.keys())` with **zero new
  imports**. Pure-infra readers (`preboot`, `sync_preferences`, `agent_docs`)
  pass nothing — they should not be warning on user kind config anyway.
- **Behaviour change** (deliberate, tested): the strict-kind warning fires only
  at the validating entrypoint, not on every config read.
  `supekku/scripts/lib/core/config_test.py` currently asserts the eager warning
  and **must be updated**; a VT must pin both the warn-when-injected and
  silent-when-not paths.
- *Why DI not relocate*: a one-line warning does not justify a new orchestration
  wrapper module; one optional param keeps core pure with minimal churn.

### 2.5 Explicitly out of scope (NEW — re-validation)

- `enums.py`, `artifact_view.py` — already pure re-export shims pointing at
  `spec_driver.orchestration` (migrated by DE-127). Not core; leave shims in place.
- `frontmatter_metadata/` subpackage — 18 per-kind metadata modules
  (adr/spec/delta/…). This is **domain** knowledge, not core infrastructure, per
  POL-003 ("core knows nothing of the domain"). Migrating it belongs to a domain
  migration delta, not here.
- `orchestration/` legacy-core imports — unchanged from original DR (follow-on).

## 3. Verification

- `uvx import-linter lint` after **each** phase — it now actively guards the
  decoupling: any residual core→orchestration / core→domain import fails the
  build (this is the point, not a side effect).
- `uv run pytest supekku spec_driver -x` — full pass after each phase.
- `uv run ruff check` — clean after each phase.
- Domain import audit: no `supekku.scripts.lib.core` imports remain in
  `spec_driver/domain/` (after tier 1).
- **Decoupling assertions** (new):
  - migrated `spec_driver/core/spec_utils.py` imports no `orchestration`/`domain`
    symbol; `dump_markdown_file_create` resolves from `orchestration.templates`.
  - migrated `spec_driver/core/config.py` imports no `frontmatter_metadata`;
    a VT pins the strict-kind warning fires with `known_kinds` injected and is
    silent without it.

## 4. Consequences

### Positive

- `spec_driver.core` becomes the real foundation: **26 migrated units** land on
  top of the 3 existing files (`file_ops`, `string_utils`, `yaml_emit`). Because
  `slugify` folds into the existing `string_utils.py` (not a new file), this is
  ~28 core modules / 25 new files.
- The outer layer contract becomes truthful — core actually contains core, and
  is provably free of upward deps (linter-enforced).
- DE-125 legacy-core import debt eliminated.
- Two latent layer violations (`spec_utils`, `config`) are removed before they
  spread further.
- Future domain/orchestration migration can depend on `spec_driver.core` directly.

### Costs

- ~26 re-export shims in `supekku/scripts/lib/core/`.
- Shim maintenance burden until consumers migrate.
- One deliberate behaviour change: strict-kind warnings fire only at validating
  entrypoints (see §2.4).

### Not changed

- Consumer import paths (all go through shims).
- CLI behaviour, except the scoped strict-kind warning change.

> **Correction (2026-05-31, surfaced during planning)**: the original DR claimed
> "tests stay at legacy paths." This conflicts with **POL-003 Scope** ("Unit
> Tests must reside within the `spec_driver/` package alongside the code they
> test, e.g. `spec_driver/core/file_ops_test.py`") and with the **proven
> pattern** — `file_ops`, `string_utils`, `yaml_emit` all migrated their tests
> into `spec_driver/core/`. Therefore **each module's `*_test.py` migrates with
> it** (test import paths repointed to `spec_driver.core.*`; legacy test file
> deleted — tests need no shim). This is now in scope.
