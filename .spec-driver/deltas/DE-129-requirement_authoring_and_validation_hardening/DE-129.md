## 1. Summary & Context

- **Product Spec(s)**: PROD-008 (Requirements Lifecycle Coherence), PROD-009 (Requirement Lifecycle Semantics)
- **Technical Spec(s)**: SPEC-122 (requirements package — parser, registry, sync)
- **Implementation Plan**: [IP-129](./IP-129.md)
- **Change Drivers**: Client-repo postmortem exposing 6 requirement-related failure modes; ISSUE-050, ISSUE-051, IMPR-012

## 2. Motivation

A postmortem from a client repo documented a painful 138-requirement migration caused by cascading failures in requirement authoring and validation:

1. **Silent format divergence**: Frontmatter `requirements:` arrays look authoritative but are completely ignored by the parser. No warning at sync time.
2. **Fragile regex with no collision detection**: Compound IDs like `FR-012-01` are silently misparsed as `FR-012`, producing one aggregate entry instead of N individual requirements.
3. **No stale requirement pruning** (ISSUE-051): Deleted requirements persist in registry forever.
4. **Misleading `show spec` counts**: Registry-sourced counts reflect whatever sync produced — no cross-check against body content.
5. **Bare requirement IDs in `applies_to`**: `FR-013` accepted without `SPEC-XXX.` qualification, producing ambiguous references.
6. **`implements → SPEC-XXX`** accepted at delta creation: semantically wrong (implements targets requirements, not specs), caught only at validate time.

These are not exotic edge cases — they represent the default failure mode when agents author specs without immediate feedback. The current feedback loop (sync → validate) is too far downstream.

## 3. Scope & Objectives

- **Primary Outcomes**:
  - Parser emits diagnostics when frontmatter contains `requirements:` array (maps to §2.1)
  - Parser detects ID collisions from compound-ID misparsing (maps to §2.3, IMPR-012)
  - Sync prunes requirements absent from their owning spec (ISSUE-051)
  - `show spec` cross-checks body requirement-like lines against registry count (maps to §2.4)
  - Validator warns on bare (unqualified) requirement IDs in delta `applies_to` (maps to §2.5)
  - Validator warns when `implements` relation targets a spec ID rather than a requirement (maps to §2.6)
  - Memory record updated with postmortem learnings for agent guidance
- **Operational Constraints**: New checks should target diagnostics architecture (per DE-119 direction) where practical, but may live in existing validator if DE-119 hasn't landed yet
- **Dependencies**: None blocking. DE-119 (validator → diagnostics migration) is parallel work; checks added here will migrate later.

## 4. Out of Scope

- Full DE-119 validator migration (separate delta)
- DE-117 CLI boilerplate reduction
- Compound ID format (`FR-NNN-NNN`) as a first-class supported format — we reject/warn, not adopt
- `validate --errors-only` / exit code differentiation (good idea, separate delta)
- Requirement tagging CLI (ISSUE-050 — related but distinct)
- SPEC-150 stub reconciliation with SPEC-122 (ISSUE-039 territory)

## 5. Approach Overview

- **System Touchpoints**:
  - `supekku/scripts/lib/requirements/parser.py` — frontmatter detection, collision detection, count mismatch warning
  - `supekku/scripts/lib/requirements/sync.py` — stale requirement pruning
  - `supekku/scripts/lib/validation/validator.py` — bare ID warning, implements-target-kind check
  - `supekku/cli/show.py` + formatters — count cross-check diagnostic
  - `.spec-driver/memory/` — updated requirement lifecycle memory
- **Key Changes**:
  - **Parser**: Add `_warn_frontmatter_requirements()` check during extraction; add ID collision detection when regex matches produce duplicate IDs within same spec
  - **Sync**: After extraction, diff registry UIDs against extracted UIDs per spec; prune orphans (or warn + prune with `--force`)
  - **Validator**: Two new checks in `_validate_change_relations`: (a) bare requirement IDs in `applies_to`, (b) `implements` targeting spec-shaped IDs
  - **Show**: Compare `requirement_like_line` count from body against registry count; emit info note on mismatch

## 6. Verification Strategy

- **Requirements Coverage**:
  - SPEC-122.FR-003 (registry load/save): pruning changes registry write behaviour → VT
  - SPEC-122.FR-004 (extraction pattern): frontmatter warning, collision detection → VT
  - SPEC-122.NF-002 (sync idempotency): pruning must not break idempotency → VT
- **Planned Artefacts**: Unit tests for each parser/sync/validator change
- **Acceptance Criteria**:
  - `spec-driver sync` on a spec with frontmatter `requirements:` array emits warning
  - `spec-driver sync` on a spec with compound IDs emits collision warning
  - `spec-driver sync` prunes requirements no longer in spec body
  - `spec-driver validate` warns on bare requirement IDs in delta `applies_to`
  - `spec-driver validate` warns on `implements → SPEC-XXX`
  - All existing tests pass; no regressions

## 7. Risks & Mitigations

- **Risk**: Stale pruning deletes requirements that are legitimately defined elsewhere (revision blocks, backlog items) – _Likelihood_: medium – _Impact_: high – _Mitigation_: Only prune requirements whose `primary_spec` matches the spec being synced AND the requirement is absent from that spec's body. Revision-introduced requirements have `introduced` field set and should be preserved.
- **Risk**: Frontmatter warning fires on legitimate structured blocks that happen to use "requirements" key (e.g., `spec.relationships` block) – _Likelihood_: low – _Impact_: low – _Mitigation_: Check specifically for frontmatter YAML `requirements:` array containing objects with `id`/`description` fields, not the relationships block format.
- **Risk**: Compound-ID collision detection produces false positives on intentionally-numbered specs – _Likelihood_: low – _Impact_: low – _Mitigation_: Warning only, not error. Message includes both the matched ID and the full line for disambiguation.
