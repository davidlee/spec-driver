## 1. Summary & Context

- **Product Spec(s)**: PROD-008 (Requirements Lifecycle Coherence), PROD-009 (Requirement Lifecycle Semantics)

## 2. Motivation

A postmortem from a client repo documented a painful 138-requirement migration caused by cascading failures in requirement authoring and validation:

1. **Silent format divergence**: Frontmatter `requirements:` arrays look authoritative but are completely ignored by the parser. No warning at sync time.
2. **Fragile regex with no collision detection**: Compound IDs like `FR-012-01` are silently misparsed as `FR-012`, producing one aggregate entry instead of N individual requirements.
3. **No stale requirement pruning** (ISSUE-051): Deleted requirements persist in registry forever.
4. **Misleading `show spec` counts**: Registry-sourced counts reflect whatever sync produced — no cross-check against body content.
5. **Bare requirement IDs in `applies_to`**: `FR-013` accepted without `SPEC-XXX.` qualification, producing ambiguous references.
6. **`implements → SPEC-XXX`** accepted at delta creation: semantically wrong (implements targets requirements, not specs), caught only at validate time.

These are not exotic edge cases — they represent the default failure mode when agents author specs without immediate feedback. The current feedback loop (sync → validate) is too far downstream.

## 3. Scope & Objectives

- **Primary Outcomes**:
  - Parser emits diagnostics when frontmatter contains `requirements:` array (DR §1.1)
  - Parser detects ID collisions from compound-ID misparsing (DR §1.2, IMPR-012)
  - Sync prunes stale requirements absent from their owning spec, with post-relation ordering (DR §1.3, ISSUE-051)
  - Validator warns on bare (unqualified) requirement IDs in delta `applies_to` (DR §1.4)
  - Validator warns when `implements` relation targets a spec ID rather than a requirement (DR §1.5)
  - Validator checks that revision-created requirements have `introduced_by` set (DR §1.6)
  - `show spec` hints when registry has 0 entries for a spec (DR §1.7)
  - Sync summary line with log-level discipline (DR §1.8)
  - Memory record updated with postmortem learnings (DR §1.9)
- **Operational Constraints**: New validator checks go in `WorkspaceValidator` (DE-119 hasn't landed); will migrate later
- **Dependencies**: None blocking. DE-119 (validator → diagnostics migration) is parallel work.

## 5. Approach Overview

- **System Touchpoints**:
  - `supekku/scripts/lib/requirements/parser.py` — frontmatter detection, collision detection, mismatch warning
  - `supekku/scripts/lib/requirements/sync.py` — post-relation stale pruning pass, summary line
  - `supekku/scripts/lib/requirements/models.py` — `SyncStats` additions
  - `supekku/scripts/lib/validation/validator.py` — bare ID warning, implements-target-kind check, revision-introduced invariant
  - `supekku/cli/show.py` + formatters — zero-entry hint
  - `.spec-driver/memory/` — updated requirement lifecycle memory
- **Key Changes**:
  - **Parser**: `_has_frontmatter_requirement_definitions()` check; ID collision detection via `seen_uids` tracking; mismatch threshold relaxed from `== 0` to `<`
  - **Sync**: Collect `{spec_id: extracted_uids}` during extraction; prune orphans in separate pass after all relation/revision/coverage steps; summary line with log-level discipline
  - **Validator**: Three new checks: (a) bare requirement IDs in `applies_to`, (b) `implements` targeting spec-shaped IDs, (c) revision-created requirements without `introduced_by`
  - **Show**: Zero-entry hint when registry has 0 requirements for the spec (no body parsing)

- **Planned Artefacts**: Unit tests for each parser/sync/validator change
- **Acceptance Criteria**:

  - `spec-driver sync` on a spec with compound IDs emits collision diagnostic
  - `spec-driver sync` prunes requirements no longer in spec body (preserving revision-introduced)
  - `spec-driver sync` emits summary line with counts
  - `spec-driver validate` warns on bare requirement IDs in delta `applies_to`
  - `spec-driver validate` warns on `implements → SPEC-XXX`
  - `spec-driver validate` warns on revision-created requirements missing `introduced_by`
  - `spec-driver show spec` hints when registry has 0 entries
  - All existing tests pass; no regressions

## 7. Risks & Mitigations

- **Risk**: Stale pruning deletes revision-moved requirements – _Likelihood_: medium – _Impact_: high – _Mitigation_: Pruning runs as post-relation pass (after step 5), so revision moves have already updated `primary_spec`. Requirements with `introduced` set are also exempt. See DR §1.3.
- **Risk**: Revision-created requirements without `introduced_by` are vulnerable to pruning – _Likelihood_: low – _Impact_: medium – _Mitigation_: Validator check enforces the invariant at source (DR §1.6).
- **Risk**: Frontmatter warning fires on legitimate structured blocks – _Likelihood_: low – _Impact_: low – _Mitigation_: Heuristic checks for list-of-dicts with `id`/`description` keys, structurally distinct from relationships block.
- **Risk**: New warnings add to signal-to-noise problem – _Likelihood_: medium – _Impact_: medium – _Mitigation_: Per-item diagnostics at `info` level; summary line at `warning` with counts. See DR §1.8.
