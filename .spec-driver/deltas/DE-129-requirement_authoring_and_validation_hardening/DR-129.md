# DR-129 – Requirement Authoring and Validation Hardening

## 1. Current vs Target Behaviour

### 1.1 Parser — Frontmatter Requirements Detection

**Current**: `_records_from_frontmatter()` passes frontmatter to `_records_from_content()`, which reads body only. A frontmatter `requirements:` array (list of dicts with `id`/`description` keys) is silently ignored. No warning at any point in the pipeline.

**Target**: During extraction, inspect the raw frontmatter mapping for a `requirements:` key whose value is a list containing dicts with `id` or `description` keys. When found, emit `logger.warning()`:

```
Spec SPEC-XXX at SPEC-XXX.md: frontmatter contains a 'requirements:' array
with N entries that will not be indexed. Requirements must be defined as body
bullets: '- **FR-001**: Title'. See mem.concept.spec-driver.requirement-lifecycle.
```

**Discrimination**: The `spec.relationships` block also uses a `requirements:` key, but its value is a dict with `primary`/`collaborators` keys — structurally distinct from definition arrays. The heuristic:

```python
def _has_frontmatter_requirement_definitions(fm: Mapping) -> list[dict]:
    """Return frontmatter entries that look like requirement definitions."""
    raw = fm.get("requirements")
    if not isinstance(raw, list):
        return []
    return [
        entry for entry in raw
        if isinstance(entry, dict) and ("id" in entry or "description" in entry)
    ]
```

### 1.2 Parser — ID Collision Detection

**Current**: `_records_from_content()` yields records via iterator. If two lines parse to the same UID (e.g., `FR-012-01` and `FR-012-02` both captured as `FR-012`), the caller's dict silently overwrites. The diagnostic heuristic only fires when `extracted_count == 0`.

**Target**: Track extracted UIDs within `_records_from_content()`. Two new diagnostics:

1. **Collision**: When a UID is yielded that was already yielded in this spec, emit:

   ```
   Spec SPEC-XXX: duplicate requirement ID FR-012 extracted from lines:
     First:  '- **FR-012-01**: Crate topology...'
     Second: '- **FR-012-02**: CLI binary...'
   Compound IDs (FR-NNN-NNN) are not supported. Use sequential IDs: FR-001, FR-002, ...
   ```

2. **Mismatch**: Extend the existing diagnostic to fire when `extracted_count < len(requirement_like_lines)` (not just `== 0`):

   ```
   Spec SPEC-XXX: Found 19 requirement-like lines but extracted only 1.
   This usually indicates a format mismatch. Expected: '- **FR-001**: Title'
   ```

**Implementation**: Add a `seen_uids: dict[str, str]` mapping UID → first source line inside `_records_from_content()`. Check before yielding each record. The mismatch threshold change is a one-line edit to the existing `if` condition.

### 1.3 Sync — Stale Requirement Pruning

**Current**: `sync` creates and updates records but never removes them. Requirements deleted from spec body persist in `requirements.yaml` indefinitely.

**Target**: After all extraction and relation-application steps complete, identify and remove orphaned registry entries per spec.

**Placement — post-relation pruning pass**: Pruning runs as a separate pass *after* all five sync stages:

1. Spec extraction (per-spec loop) — collects `{spec_id: extracted_uids}`
2. Relationship blocks
3. Delta relations
4. Revision relations + revision blocks (moves update `primary_spec` here)
5. Coverage blocks

Then: **pruning pass** iterates collected `{spec_id: extracted_uids}` and removes orphans.

This ordering is critical. If pruning ran inside step 1, a revision-move that updates `primary_spec` in step 4 would not yet be reflected — the moved requirement would be incorrectly pruned from its old spec, then recreated by the revision block, causing oscillation across sync runs.

**Pruning logic**:

```python
for spec_id, extracted_uids in spec_extractions.items():
    orphans = {
        uid for uid, rec in records.items()
        if rec.primary_spec == spec_id
        and uid not in extracted_uids
        and not rec.introduced  # protect revision-introduced requirements
    }
    for uid in orphans:
        del records[uid]
        stats.pruned += 1
        logger.info("Pruned stale requirement %s (absent from %s body)", uid, spec_id)
```

**SyncStats change**: Add `pruned: int = 0` field to the `SyncStats` dataclass in `models.py`.

**Safety invariants**:

- Requirements with `introduced` set (truthy string) are always preserved. These were created by revision blocks (`_apply_revision_requirement`).
- Requirements whose `primary_spec` was updated by a revision move in step 4 will point to the destination spec, not the source — the source spec's pruning loop won't match them.
- Backlog-sourced requirements have `primary_spec` set to the backlog item ID, not a spec ID — not matched by the spec-iteration pruning loop.

**Idempotency** (SPEC-122.NF-002): Pruning is deterministic — same body content produces same pruning decisions. Re-running sync on an already-pruned registry produces no further changes.

### 1.4 Validator — Bare Requirement ID Warning

**Current**: `_validate_change_relations()` checks `applies_to.requirements` entries for existence in `applies_to_ids`. Bare IDs like `FR-013` either resolve (if unique) or fail with "not found."

**Target**: Before the existence check, inspect each requirement string. If it matches a requirement-shaped pattern (`^(FR|NF)-\d{3}$`) without a spec prefix (no `.`), emit a warning:

```
DE-XXX: applies_to requirement 'FR-013' is not fully qualified.
Use 'SPEC-XXX.FR-013' to avoid ambiguity.
```

This is a warning, not an error — bare IDs still work when unambiguous. The warning guides agents toward the robust form.

The check specifically targets requirement-shaped IDs (`FR-NNN`, `NF-NNN`), not backlog item IDs (`ISSUE-NNN`, `IMPR-NNN`) which also appear in `applies_to` but have a different shape and don't need qualification.

**Implementation**: ~5 lines in `_validate_change_relations()`, gated on `expected_type == "implements"` to avoid false positives on other relation types. A compiled constant `_BARE_REQUIREMENT_PATTERN` for the match.

### 1.5 Validator — `implements` Target Kind Check

**Current**: When `implements → SPEC-012` fails the requirement-ID lookup, the error says "does not match any known requirement" — technically correct but uninformative about *why*.

**Target**: Before the existence check, test whether the target matches a spec-shaped pattern (`^(SPEC|PROD)-\d{3}$`). If so, emit a specific warning:

```
DE-XXX: Relation 'implements -> SPEC-012' targets a spec, not a requirement.
'implements' is for requirements (e.g., SPEC-012.FR-001). Use 'relates_to' for
spec-level relations.
```

This fires *instead of* the generic "not found" error for this case, providing actionable guidance.

**Implementation**: A compiled regex constant `_SPEC_ID_PATTERN` and a 4-line guard clause before the existing lookup.

### 1.6 Validator — Revision-Created Requirements Must Have `introduced_by`

**Current**: `_apply_revision_requirement()` creates placeholder records when the target UID doesn't exist. If the revision block's `lifecycle` payload omits `introduced_by`, the record is created with `introduced = None`. This makes the record indistinguishable from a body-extracted requirement and vulnerable to stale pruning (§1.3).

**Target**: Add a validator check: for each requirement in the registry whose `source_kind` or provenance suggests revision origin but `introduced` is falsy, emit a warning:

```
SPEC-XXX.FR-005: Requirement appears to be revision-created but has no
introduced_by field. This may cause incorrect pruning. Check revision blocks.
```

This enforces the invariant at the source rather than adding defensive guards in the pruning logic.

**Implementation**: ~8 lines in `WorkspaceValidator.validate()`, iterating requirement records and checking the condition.

### 1.7 Show Spec — Zero-Entry Hint

**Current**: `show spec` displays requirement counts sourced from the registry. If sync has never been run, or produced wrong data, the display is confidently wrong with no diagnostic.

**Target**: After computing registry counts, if the spec has 0 requirements in the registry, emit a simple info note:

```
ℹ No requirements found in registry for this spec. Run 'spec-driver sync' if
  the spec defines requirements.
```

No body parsing, no threshold heuristic, no new imports. This covers both the "never synced" case and the "sync produced wrong data" case with a lightweight nudge.

### 1.8 Sync — Summary Line and Log Level Discipline

**Current**: Parser and sync diagnostics emit at `logger.warning()` level. Each diagnostic is a separate warning line. In repos with many specs, sync output becomes a wall of warnings indistinguishable from each other.

**Target**: Demote per-item diagnostics (frontmatter detection, collision, mismatch, pruning) to `logger.info()` level. Add a summary line at the end of sync at `logger.warning()` level when any issues were found:

```
Sync complete: 5 created, 2 updated, 3 pruned, 4 warnings (run with -v for details)
```

When no issues exist, the summary emits at `logger.info()`:

```
Sync complete: 5 created, 2 updated, 0 pruned
```

**Implementation**: Accumulate a warning counter in `SyncStats` (add `warnings: int = 0`). Parser functions increment it via a callback or return diagnostics alongside records. The sync orchestrator emits the summary after all steps.

### 1.9 Memory Update

Update `mem.concept.spec-driver.requirement-lifecycle` with a new "Common Pitfalls" section:

```markdown
## Common Pitfalls

1. **Frontmatter `requirements:` array**: Defining requirements as a YAML array
   in spec frontmatter looks correct but is completely ignored by sync. The
   registry parser reads body bullets only. Sync will warn if it detects this.

2. **Compound IDs (`FR-012-01`)**: The parser captures `FR-012` and discards
   the `-01` suffix. Multiple compound IDs collapse into one registry entry.
   Use sequential IDs: `FR-001`, `FR-002`, etc.

3. **Bare requirement IDs in `applies_to`**: `FR-013` is ambiguous when
   multiple specs define it. Always qualify: `SPEC-012.FR-013`.
```

## 3. Design Decisions

| # | Decision | Rationale | Alternatives Rejected |
|---|----------|-----------|----------------------|
| D1 | Frontmatter detection is a warning, not error | The structure isn't invalid YAML — it's just not consumed. Proportionate response. | Error (too aggressive for existing repos) |
| D2 | Auto-prune on sync; `introduced` field is the exemption guard | Simplest correct behaviour. No flag ceremony. Revision-introduced requirements have `introduced` set by `_apply_revision_requirement()`. | Warn-only (doesn't fix the problem); flag-gated (unnecessary complexity) |
| D3 | Pruning runs after all relation/revision/coverage steps | Prevents oscillation when revision moves update `primary_spec` in step 4. Collected `{spec_id: extracted_uids}` from step 1 is compared against post-step-5 state. | Inside per-spec loop (race with revision moves — ext. review F1) |
| D4 | Collision detection is per-spec | Two specs can legitimately define `FR-001`. The failure mode is *within* a spec. | Global collision check (wrong semantics) |
| D5 | Bare-ID check is warning, not error; targets requirement-shaped IDs only | Bare IDs work when unambiguous. Breaking existing deltas is disproportionate. Backlog IDs (`ISSUE-NNN`) have a different shape and don't need qualification. | Error (breaks existing deltas); any-dotless-string (false positives on backlog IDs — int. review AR-4) |
| D6 | Enforce `introduced_by` invariant via validator check | Fixes the root cause (revision blocks omitting `introduced_by`) rather than adding defensive guards in pruning. | Extra guard in pruning (treats symptom); ignore (leaves pruning vulnerable — ext. review F7) |
| D7 | Show spec: simple zero-entry hint, no body parsing | Eliminates threshold fragility, coupling to parser, and false positives. Covers both "never synced" and "sync failed" cases. | Body-count cross-check with 1.5× threshold (wrong level, fragile — ext. review F2) |
| D8 | Per-item diagnostics at `info` level; summary at `warning` | Prevents wall-of-warnings problem. Users see actionable summary; `-v` reveals detail. | All at `warning` (drowns signal — ext. review F3); all at `info` (too quiet for first-run) |
| D9 | New validator checks go in `WorkspaceValidator` | DE-119 hasn't landed. Small checks aren't worth pioneering the migration. | diagnostics/checks/ (correct long-term home, premature now) |

## 4. Governance Alignment

| Constraint | Compliance |
|-----------|-----------|
| POL-001 (reuse) | No duplicate logic; `count_requirement_like_lines` retained as public API for future callers |
| POL-002 (no magic strings) | `_SPEC_ID_PATTERN`, `_BARE_REQUIREMENT_PATTERN` as compiled constants; frontmatter keys checked by name with clear intent |
| POL-003 (module boundaries) | parser.py stays in domain layer; show.py (presentation) imports from domain — correct direction per layered architecture; zero-entry hint removes the need for parser import in show.py |
| STD-003 (utility placement) | `_has_frontmatter_requirement_definitions` is parser-domain, stays in parser.py |
| ADR-008 (normative vs observed) | Pruning respects the normative/observed split: body content is normative for body-defined requirements; revision-introduced requirements are normative via revision blocks; validator enforces `introduced_by` invariant so the distinction is reliable |
