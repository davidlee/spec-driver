## 1. Summary & Context

- **Implementation Plan**: [IP-130](./IP-130.md)
- **Change Drivers**: User quality requirement — validation errors on status fields must always enumerate allowable values.

## 2. Motivation

Several status-field validation warnings and errors in the codebase either omit the set of allowable values entirely, or hard-code them as prose rather than deriving them from the source constant. This makes it hard for users and agents to self-correct without reading source code.

## 3. Scope & Objectives

- **Primary Outcomes**:
  - Every validation error/warning that rejects a status value must include the set of allowable values in its message.
  - Allowable values must be derived from the authoritative constant, not duplicated as prose.
- **Operational Constraints**: No behaviour changes beyond error message text.
- **Dependencies**: None.

## 4. Out of Scope

- Changing which statuses are valid.
- Restructuring how `ValidationError` is formatted (the `expected` field already renders; message text is sufficient).

## 5. Approach Overview

- **System Touchpoints**:
  - `supekku/scripts/lib/requirements/coverage.py`
  - `supekku/scripts/lib/validation/validator.py`
  - `supekku/scripts/lib/changes/completion.py`
- **Key Changes**:
  - `coverage.py:143-149` — append `f"; valid: {sorted(VALID_COVERAGE_STATUSES)}"` to the WARNING message.
  - `validation/validator.py:552` — append `f"; valid: {sorted(valid_statuses)}"` to the "Non-canonical phase status" warning.
  - `changes/completion.py:364-365` — replace hardcoded prose with `sorted(valid_statuses)` derived from the `valid_statuses` set already in scope.

## 6. Verification Strategy

- **Acceptance Criteria**: All three messages include allowable values; existing tests pass; no new lint warnings.

## 7. Risks & Mitigations

- **Risk**: Test assertions on exact message text will break — _Likelihood_: medium — _Impact_: low — _Mitigation_: update assertions.
