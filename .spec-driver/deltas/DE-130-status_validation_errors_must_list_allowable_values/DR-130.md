# DR-130 – status validation errors must list allowable values

## 1. Executive Summary

- **Delta**: [DE-130](./DE-130.md)
- **Status**: draft
- **Synopsis**: Three status-rejection messages omit the set of allowable values; patch each to enumerate them from the authoritative constant already in scope.

## 2. Problem & Constraints

- **Current Behaviour**: Three validation paths emit messages when a status value is unrecognised but do not tell the caller what values are accepted:
  1. `coverage.py` — `WARNING: Coverage entry … has unknown status 'X' … entry will not influence derived requirement status` (no list)
  2. `validator.py` — `Non-canonical phase status: 'X'` (no list)
  3. `completion.py` — `Expected status: draft or in-progress` (hardcoded prose, not derived from constant)
- **Constraints**: No behaviour change beyond message text. No new imports, no new abstractions.
- **Out of Scope**: Changing which statuses are valid; restructuring `ValidationError` formatting.

## 3. Architecture Intent

- **Target**: Every message that rejects a status value includes `valid: [sorted list]` derived from the authoritative constant in scope at that call site.
- **Guiding Principle**: Values must come from the constant, not be duplicated as prose, so they stay correct if the constant changes.

## 4. Code Impact Summary

| Path | Current message | Target message |
|---|---|---|
| `requirements/coverage.py:148` | `"entry will not influence derived requirement status"` | `"entry will not influence derived requirement status; valid: {sorted(VALID_COVERAGE_STATUSES)}"` |
| `validation/validator.py:552` | `f"Non-canonical phase status: '{status}'"` | `f"Non-canonical phase status: '{status}'; valid: {sorted(valid_statuses)}"` |
| `changes/completion.py:364-365` | `"Expected status: draft or in-progress"` | `f"Expected status: {', '.join(sorted(valid_statuses))}"` |

In each case the constant (`VALID_COVERAGE_STATUSES`, `valid_statuses`) is already imported and in scope at the call site. No new imports required.

## 5. Verification Alignment

| Verification | Impact | Notes |
|---|---|---|
| existing tests | update | Any assertion on exact message text for these three paths must be updated to match new wording |

## 6. Design Decisions & Trade-offs

- Format `sorted(...)` for deterministic ordering in messages (consistent with the pattern used in `requirements/registry.py`, `changes/updater.py`).
- Separator: use `; valid:` prefix (matches existing style in `blocks/verification.py` and `requirements/registry.py`).
