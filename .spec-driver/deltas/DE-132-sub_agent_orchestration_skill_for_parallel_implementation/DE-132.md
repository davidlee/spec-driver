## 1. Summary & Context

- **Product Spec(s)**: PROD-011 – Implementation Execution Workflow; PROD-016 – Agentic SDLC Configurator
- **Implementation Plan**: [IP-132](./IP-132.md)
- **Change Drivers**: User request for sub-agent orchestration; lazyspec build skill as external reference

## 2. Motivation

The current `/implement` skill runs in a single agent context. For multi-phase
implementation plans with independent tasks, this is:

1. **Slow** — tasks that could run concurrently execute sequentially
2. **Context-wasteful** — a single agent accumulates context from all tasks
3. **Model-inefficient** — all tasks use the same model regardless of complexity

Target state: a skill that analyses an IP's phase/task structure, identifies
parallelizable work, assigns appropriate models (sonnet vs opus), and dispatches
sub-agents in isolated worktrees for concurrent execution.

## 3. Scope & Objectives

- **Primary Outcomes**:
  - New skill (`build` or similar) that orchestrates sub-agent implementation
  - Parallelism analysis: identify which tasks/phases can run concurrently
  - Model routing: assign sonnet to straightforward tasks, opus to complex ones
  - Worktree isolation: concurrent agents work in separate git worktrees
  - Two-stage review: spec compliance + code quality checks per task
  - Integration with existing spec-driver workflow (notes, continuation, etc.)
- **Operational Constraints**:
  - Claude-only skill (installed to `.claude/skills/` only)
  - Must integrate with existing `/execute-phase` and `/implement` workflow
  - Must not break single-agent flow — skill should degrade gracefully
- **Dependencies**: None (uses existing Claude Code Agent tool capabilities)

## 4. Out of Scope

- Modifying the existing `/implement` or `/execute-phase` skills
- Auto-merge of worktree branches (human/orchestrator decision)
- CI/CD integration
- Non-Claude agent targets (codex, etc.)

## 5. Approach Overview

- **System Touchpoints**: `.spec-driver/skills/`, `.claude/skills/` (symlink), skills.allowlist
- **Key Changes**:
  - New skill directory: `.spec-driver/skills/build/SKILL.md`
  - Skill added to `.spec-driver/skills.allowlist`
  - Skill synced to `.claude/skills/build/` via existing install pipeline
- **Design Considerations**:
  - Leverage Claude Code's `Agent` tool with `isolation: "worktree"` for parallel tasks
  - Use `model` parameter on Agent tool for sonnet/opus routing
  - Orchestrator reads IP/phase sheets, decomposes into dispatchable units
  - Each sub-agent receives: task description, relevant context (DR, spec excerpts, prior task results), and instructions to follow spec-driver workflow conventions

## 6. Verification Strategy

- **VH-001**: Manual test — invoke skill on a multi-phase IP with independent tasks, verify parallel dispatch occurs
- **VH-002**: Manual test — verify worktree isolation (concurrent agents don't conflict)
- **VH-003**: Manual test — verify model routing (simple tasks get sonnet, complex get opus)
- **VA-001**: Review skill against existing skill format conventions
- **Acceptance Criteria**:
  - Skill file follows project SKILL.md conventions
  - Skill is installable via `spec-driver skills sync`
  - Skill correctly identifies parallelizable vs sequential tasks from an IP

## 7. Risks & Mitigations

- **Risk**: Worktree merge conflicts from parallel agents – _Likelihood_: medium – _Impact_: medium – _Mitigation_: Skill should detect file overlap in task scopes and force sequential execution for overlapping tasks
- **Risk**: Sub-agent context loss (can't access spec-driver memories/doctrine) – _Likelihood_: medium – _Impact_: low – _Mitigation_: Orchestrator injects relevant context into sub-agent prompts; sub-agents in worktrees have full repo access
- **Risk**: Skill complexity exceeds single-file SKILL.md capacity – _Likelihood_: low – _Impact_: low – _Mitigation_: Keep orchestration logic in the prompt; complex heuristics belong in future code, not this delta

## 8. Resolved Decisions

- **Skill name**: `dispatch`
- **Dispatch granularity**: Batched tasks within a phase. A single sub-agent
  handles as many tasks as fit in ~120k token budget, up to a whole phase.
  This avoids per-task boot overhead while allowing phase-splitting when a
  phase exceeds a single agent's capacity.
- **Model routing**: Per-batch. Orchestrator assigns model based on the
  hardest task in the batch (opus if any task requires it, sonnet otherwise).
- **Review strategy**: Batch review at phase boundary by default. Orchestrator
  may tighten to per-batch review when ACs have tight dependencies that are
  easy to get wrong. Sub-agents return AC evidence summaries alongside
  artefact updates.
- **Relationship to /execute-phase**: Peer skill, not wrapper. Sub-agents
  use a dedicated lightweight skill (not full execute-phase ceremony).
- **Parallelism**: Independent phases or independent batches within a phase
  run in parallel worktrees. Sequential dependencies are respected.
