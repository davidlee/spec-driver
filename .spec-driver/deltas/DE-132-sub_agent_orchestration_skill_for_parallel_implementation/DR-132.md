# DR-132 – Sub-agent orchestration skill for parallel implementation

## 1. Executive Summary

- **Delta**: [DE-132](./DE-132.md)
- **Status**: draft
- **Last Updated**: 2026-04-16
- **Synopsis**: New `/dispatch` skill that decomposes an IP into parallelizable
  batches, routes them to appropriate models, and executes via sub-agents in
  isolated worktrees.

## 2. Problem & Constraints

- **Current Behaviour**: `/implement` and `/execute-phase` run in a single
  agent context. Multi-phase IPs execute sequentially, wasting time and context
  on independent work. All tasks use the same model regardless of complexity.
- **Drivers**: User request; lazyspec build skill as prior art.
- **Constraints**:
  - Skill + agent definition files only — no Python code in this delta
  - Claude-only install target
  - Must not modify existing skills
  - Must degrade gracefully to sequential execution when parallelism isn't safe
  - Subagents cannot spawn subagents (platform constraint)
  - Subagents don't receive CLAUDE.md or .claude/rules/ — all context via
    system prompt (agent body) + injected skills
- **Out of Scope**: CI integration, non-Claude agent targets, plugin packaging
  (noted as future direction).

## 3. Architecture Intent

### 3.1 Orchestration Model

The `/dispatch` skill is an **orchestrator** — it reads structured artefacts
(IP, phase sheets, DR) and produces a dispatch plan, then executes it.

```
User invokes /dispatch <delta-id>
  │
  ├─ 1. Read IP, DR, phase sheets
  ├─ 2. Analyse: parallelism graph, model requirements, token budgets
  ├─ 3. Produce dispatch plan (batches, ordering, models, isolation)
  ├─ 4. Present plan to user for approval
  ├─ 5. Execute batches (parallel where safe, sequential where required)
  ├─ 6. Collect results + AC evidence summaries
  ├─ 7. Review at phase boundary (or tighter if warranted)
  └─ 8. Report outcomes, update notes
```

### 3.2 Batching Strategy

**Problem**: Per-task agent dispatch is context-wasteful (boot overhead, lost
shared understanding between related tasks). Per-phase dispatch wastes capacity
when phases are small.

**Solution**: Token-budget batching.

1. Estimate each task's token cost (heuristic: complexity tag from phase sheet,
   file count, description length).
2. Pack tasks into batches targeting ~120k tokens, respecting:
   - Task ordering constraints within a phase
   - File-scope overlap (tasks touching same files must be in same batch or sequential)
   - Model requirements (if any task in batch needs opus, batch gets opus)
3. Maximum batch size = one full phase.

### 3.3 Parallelism Analysis

Tasks/batches can run in parallel when:
- No data dependencies between them (no task references another's output)
- No file-scope overlap (disjoint set of touched files/modules)
- Different phases are independent (phase ordering in IP is respected)

The orchestrator builds a simple dependency graph from:
- Explicit `depends_on` in phase sheet tasks
- File-scope overlap detection (from task descriptions / `touches` fields)
- Phase ordering (phases execute in IP order unless marked independent)

### 3.4 Model Routing

Per-batch model assignment:

| Signal | Model |
|--------|-------|
| Task marked `complexity: high` or `requires: opus` | opus |
| Task involves architecture, design decisions, cross-cutting concerns | opus |
| Task is mechanical (rename, format, boilerplate, test scaffolding) | sonnet |
| Default (no signal) | sonnet |

Orchestrator assigns based on the hardest task in the batch.

### 3.5 Worktree Isolation

Parallel batches use `Agent` tool with `isolation: "worktree"`. This:
- Creates a temporary git worktree per agent
- Gives each agent a full, isolated copy of the repo
- Automatically cleans up worktrees with no changes
- Returns worktree path and branch name for changed worktrees

Sequential batches (or single-batch phases) run in the main worktree without
isolation overhead.

### 3.6 Sub-Agent Context Architecture

This is the hardest design problem in the skill.

**Platform reality** (confirmed from Claude Code docs):
- Subagents receive ONLY their markdown body as system prompt (plus basic
  env details like working directory)
- Subagents do NOT receive CLAUDE.md, .claude/rules/, or parent skills
- Subagents CAN load skills via `skills:` frontmatter — full content is
  injected at startup
- Subagents cannot spawn other subagents
- Subagents can be resumed via SendMessage (preserves full context)

This means ALL project-specific context must flow through two channels:
1. The agent definition body (system prompt)
2. Skills injected via `skills:` frontmatter

#### 3.6.1 Three-file architecture

```
.claude/agents/dispatch-worker.md     ← agent definition (system prompt + config)
  └─ skills: [dispatch-worker]        ← loads tourist skill at startup
.spec-driver/skills/dispatch-worker/  ← tourist skill (spec-driver fluency)
.spec-driver/skills/dispatch/         ← orchestrator skill (user-invocable)
```

**dispatch-worker.md** (agent definition):
- Frontmatter: name, description, tools, model, permissionMode, skills
- Body: task-generic system prompt (how to work, completion protocol)
- Does NOT contain task-specific context — that comes from the orchestrator's
  prompt when spawning

**dispatch-worker skill** (injected via `skills:` frontmatter):
- Spec-driver tourist knowledge: memory commands, artefact update commands,
  governance boundary recognition
- Target: 2-5k tokens of spec-driver overhead
- No routing ceremony, no governance listings, no cascade-triggering directives

**dispatch skill** (orchestrator, runs in main agent):
- Reads IP/DR/phase sheets, builds dispatch plan
- Constructs per-batch prompts with task-specific context
- Spawns `dispatch-worker` agents via `Agent` tool

#### 3.6.2 Context flow

```
Orchestrator (main agent, full spec-driver context)
  │
  │  reads & distills:
  │  ├─ CLAUDE.md (lint/test/commit policy)
  │  ├─ doctrine.md (project conventions)
  │  ├─ Relevant ADRs/policies/standards
  │  ├─ Pre-fetched memories for task file scopes
  │  ├─ DR excerpts, spec requirements
  │  └─ Phase sheet tasks
  │
  ▼
Agent(subagent_type="dispatch-worker", isolation="worktree", model="sonnet")
  prompt = orchestrator-assembled task context
  │
  Agent receives:
  ├─ System prompt: dispatch-worker.md body (task-generic)
  ├─ Injected skill: dispatch-worker (spec-driver tourist fluency)
  └─ User turn: orchestrator's per-batch prompt containing:
     ├─ Task list with descriptions + file scopes
     ├─ Policy digest (lint/test/commit commands)
     ├─ Design context (DR excerpts)
     ├─ Pre-fetched memories
     └─ Governance excerpts (if relevant)
```

The key insight: the agent definition + tourist skill are **static** (same
for every batch). The orchestrator's prompt is **dynamic** (varies per batch).
This separates "how to work" from "what to work on".

#### 3.6.3 Sources of project-specific truth

| Source | Location | Who provides it | Channel |
|--------|----------|----------------|---------|
| Code standards, lint/test/commit policy | `CLAUDE.md` | Orchestrator digests | Per-batch prompt |
| Project conventions, commit policy | `doctrine.md` | Orchestrator digests | Per-batch prompt |
| Governance (ADRs, policies, standards) | `.spec-driver/decisions/` | Orchestrator selects relevant | Per-batch prompt |
| Design intent | DR, IP, phase sheet | Orchestrator extracts | Per-batch prompt |
| Pre-known file memories | `.spec-driver/memory/` | Orchestrator pre-fetches | Per-batch prompt |
| Spec-driver commands & patterns | tourist skill | `skills:` frontmatter | Injected at startup |
| How to work, completion protocol | agent body | Agent definition | System prompt |
| Reactive file memories | `.spec-driver/memory/` | Worker queries during work | Tourist skill teaches how |

#### 3.6.4 What the tourist skill contains

The dispatch-worker skill gives sub-agents spec-driver fluency without
ceremony overhead. Target budget: **2-5k tokens**. Contents:

**Capabilities** (commands the worker can use):
- `uv run spec-driver list memories -p <path>` — query memories for files
- `uv run spec-driver create memory "<name>" --type gotcha` — record findings
- Phase sheet task status updates (read/edit frontmatter)
- Append to delta/phase notes

**Boundaries** (what NOT to do):
- Do NOT create deltas, revisions, audits, or specs
- Do NOT interpret governance artefacts — flag concerns in return summary
- Do NOT invoke `/boot`, `/using-spec-driver`, or other ceremony skills
- If ambiguous: document it and continue. Orchestrator reviews.

**Completion protocol** (structured return summary):
- Tasks completed (with status)
- AC evidence per task
- Files changed
- Test + lint results (must pass before reporting complete)
- Blockers, deferred items, governance concerns
- Memories created or updated

#### 3.6.5 Agent definition frontmatter

```yaml
name: dispatch-worker
description: >
  Implementation worker for /dispatch orchestration. Executes batched
  tasks from a spec-driver implementation plan in an isolated worktree.
  Do not invoke directly — spawned by the dispatch orchestrator.
tools: Read, Edit, Write, Bash, Grep, Glob
model: sonnet
permissionMode: acceptEdits
skills:
  - dispatch-worker
```

Notes:
- `model: sonnet` is the default; orchestrator overrides to opus per-batch
  when needed (per-invocation model takes precedence over definition)
- `permissionMode: acceptEdits` — workers edit files without prompting
- `tools` excludes Agent (workers can't spawn sub-agents) and Skill
  (tourist skill is pre-injected; no other skills needed)
- `isolation` is NOT set here — orchestrator passes it per-invocation
  (parallel batches get worktrees, sequential batches may not)

#### 3.6.6 Token budget

Measured from current repo (2026-04-16), full spec-driver boot costs ~23k
static tokens + 15-30k of triggered cascade actions.

Worker agent budget:

| Source | Tokens | Notes |
|--------|--------|-------|
| Platform (system prompt + tools) | ~13k | Reduced: fewer tools, no Skill/Agent |
| Agent body (system prompt) | ~1k | Task-generic instructions |
| Tourist skill (injected) | 2-5k | Spec-driver fluency |
| Orchestrator prompt (per-batch) | 5-15k | Task context, policy digest, memories |
| **Total overhead** | **~21-34k** | |
| **Available for work** | **~86-99k** | Out of 120k budget |

The tourist skill avoids cascade costs by giving commands to use, not
directives that trigger further ceremony. No routing → no governance
listing → no ADR reads → no cascading token spend.

The orchestrator should warn if its per-batch prompt exceeds ~15k tokens,
as this would squeeze the work budget below ~85k.

#### 3.6.7 Risk: context gap

The biggest residual risk is the orchestrator failing to include a relevant
policy, memory, or governance constraint. Subagents do NOT get CLAUDE.md
or rules auto-loaded, so anything not in the agent definition, tourist
skill, or orchestrator prompt is invisible.

Mitigations:
- Orchestrator reads CLAUDE.md + doctrine.md and distills policy digest
- Orchestrator pre-fetches memories for known file scopes
- Tourist skill instructs workers to query memories for discovered files
- Workers flag governance-looking concerns rather than resolving them
- Phase boundary review catches policy violations before merge
- Workers have full repo file access — they can read governance files
  if the tourist skill instructs them to check specific paths

### 3.7 Review Strategy

**Default**: Batch review at phase boundary.

After all batches in a phase complete, orchestrator:
1. Collects AC evidence summaries from each sub-agent
2. Reviews against phase exit criteria
3. Flags gaps or failures
4. Reports to user

**Tighter pass**: When phase sheet marks ACs as `review: tight` or tasks have
tight cross-dependencies, orchestrator reviews per-batch completion before
dispatching dependent batches.

**AC evidence flow**: Sub-agents record evidence in artefacts (phase notes,
verification blocks) AND return a structured summary to the orchestrator.
The orchestrator aggregates these into phase-level evidence.

## 7. Design Decisions & Trade-offs

### DEC-132-01: Batched dispatch over per-task dispatch

Per-task dispatch (lazyspec approach) wastes ~10-20k tokens on boot/context per
agent. Batching amortises this cost. Trade-off: less isolation between tasks in
a batch, but tasks in the same phase typically share context anyway.

### DEC-132-02: Worktree isolation via Agent tool

Claude Code's native `isolation: "worktree"` handles worktree lifecycle. No
custom git worktree management needed. Trade-off: less control over worktree
naming/location, but simplicity wins for a skill-only deliverable.

### DEC-132-03: Peer to /execute-phase

`/dispatch` is chosen instead of `/execute-phase` for parallelizable work, not
layered on top. Rationale: `/execute-phase` assumes single-agent, sequential
execution with full ceremony. Wrapping it would fight its assumptions.

### DEC-132-04: Lightweight sub-agent instructions

Sub-agents get condensed context, not full `/boot` → `/using-spec-driver` →
`/execute-phase` chains. Full ceremony per sub-agent would consume most of the
token budget on workflow overhead. Trade-off: sub-agents may miss edge-case
governance, but the orchestrator handles governance at the coordination level.

### DEC-132-05: Adaptive review granularity

Default batch review at phase boundary keeps overhead low. Tighter per-batch
review activates when phase sheet signals it. Trade-off: default may miss
inter-batch integration issues, but phase boundary review catches these.

### DEC-132-06: Tourist-mode skill for sub-agents

Sub-agents get a dedicated `dispatch-worker` skill that provides lightweight
spec-driver fluency: memory access/creation, artefact updates, governance
boundary awareness. The skill is injected at agent startup via the `skills:`
frontmatter field in the agent definition — not invoked as a slash command,
not constructed inline by the orchestrator. The key insight is that some
context is reactive (discovered during work) so the sub-agent needs enough
literacy to fish for itself — but not enough rope to make governance decisions.

### DEC-132-07: Sub-agents cannot escalate — non-interactive by design

Sub-agents (via Claude Code `Agent` tool) are non-interactive: they cannot
prompt the user. When a sub-agent hits ambiguity, it documents the blocker in
its return summary. The orchestrator surfaces unresolved blockers to the user
after batch completion. This is a hard constraint of the platform, not a choice.

### DEC-132-08: Auto-merge on green

Completed worktree branches merge back into the orchestrator's branch as soon
as checks pass. The orchestrator attempts the merge; if conflicts arise, it
reports them to the user rather than retrying or forcing.

### DEC-132-09: Agent definition over inline prompt construction

Workers are defined as a `.claude/agents/dispatch-worker.md` file, not
constructed ad-hoc in the orchestrator's prompt. This separates "how to work"
(static, in agent definition + tourist skill) from "what to work on" (dynamic,
in orchestrator's per-batch prompt). Benefits:
- Agent definition is version-controlled and reviewable
- Tourist skill is maintainable independently
- Orchestrator prompt construction is simpler (only task-specific context)
- Agent config (tools, permissionMode, model default) is declarative

### DEC-132-10: Plugin packaging as future distribution path

Claude Code supports distributing agents via plugins (`agents/` directory
in a plugin). A spec-driver Claude plugin could provide dispatch-worker (and
potentially other agents) less invasively than the current installer mechanism.
Deferred — not in scope for this delta, but noted as a natural evolution path.

## 8. Resolved Questions

- **Sub-agent escalation**: Not possible — `Agent` tool is non-interactive.
  Sub-agents document blockers in return summaries; orchestrator surfaces them.
- **Phase sheet fields**: Orchestrator infers from existing structure for now.
  Explicit annotations (`parallelizable`, `complexity`, `touches`, `review`)
  backlogged as IMPR for `/plan-phases` to produce. See IMPR-028.
- **Merge strategy**: Auto-merge into orchestrator's branch when checks pass.
  Conflict resolution escalated to user.
