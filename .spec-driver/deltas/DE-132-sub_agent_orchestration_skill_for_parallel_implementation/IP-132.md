## 1. Summary

- **Delta**: DE-132 — Sub-agent orchestration skill for parallel implementation
- **Specs Impacted**: PROD-011 (Implementation Execution Workflow), PROD-016 (Agentic SDLC Configurator)
- **Desired Outcome**: A `/dispatch` skill that decomposes IPs into parallelizable
  batches, routes to appropriate models, and executes via sub-agents in isolated
  worktrees. Workers get spec-driver fluency through a dedicated agent definition
  and tourist skill.

## 2. Context & Constraints

- **Current Behaviour**: Single-agent sequential execution via `/implement` and
  `/execute-phase`. No parallelism, no model routing, no worktree isolation.
- **Target Behaviour**: Orchestrated multi-agent execution with batching,
  parallelism analysis, model routing, and worktree isolation per DR-132.
- **Dependencies**: None — uses existing Claude Code Agent tool capabilities.
- **Constraints**:
  - Skill + agent definition files only — no Python code
  - Subagents don't receive CLAUDE.md or .claude/rules/
  - Subagents cannot spawn subagents
  - Worker token overhead target: ~6.5k for injected skills
  - Must not modify existing skills

## 5. Phase Detail

### P01 — Worker foundation (sonnet)

The static context layer. Creates the two files that define how workers
operate, independent of any specific task.

**Tasks:**
1. Write `.claude/agents/dispatch-worker.md` — agent definition with
   frontmatter (name, description, tools, model, permissionMode, skills)
   and system prompt body (task-generic working instructions, completion
   protocol)
2. Write `.spec-driver/skills/sub-driver/SKILL.md` — tourist skill:
   exec.md pointer, mini glossary (VT/VA/VH/FR/NF, DE/DR/IP identifiers),
   phase sheet update commands, governance boundaries, what NOT to do
3. Verify token budget: injected skills (sub-driver + retrieving-memory +
   capturing-memory + notes) should total ~6.5k tokens

**Exit criteria:**
- Agent definition parses correctly (valid YAML frontmatter)
- Tourist skill follows project SKILL.md conventions
- Token estimate within budget

### P02 — Orchestrator skill (opus)

The dynamic coordination layer. This is the complex piece — reading
artefacts, building dependency graphs, constructing per-batch prompts,
dispatching agents, collecting results, reviewing.

**Tasks:**
1. Write `.spec-driver/skills/dispatch/SKILL.md` — full orchestrator:
   - Read IP/DR/phase sheets
   - Parallelism analysis (dependency graph from task descriptions)
   - Batching strategy (token budget estimation, file-scope overlap)
   - Model routing (complexity signals → sonnet/opus)
   - Per-batch prompt construction (policy digest, design context,
     pre-fetched memories, task scope)
   - Dispatch execution (Agent tool with subagent_type, isolation, model)
   - Result collection + AC evidence aggregation
   - Phase boundary review
   - Auto-merge on green, conflict escalation
   - Reporting to user
2. Ensure orchestrator prompt construction references dispatch-worker
   capabilities correctly (completion protocol, expected return structure)

**Exit criteria:**
- Skill follows project SKILL.md conventions
- Orchestration flow covers all DR-132 sections (3.1–3.7)
- Prompt construction template is concrete and reviewable

### P03 — Integration & verification (sonnet)

Wire everything together and verify.

**Tasks:**
1. Add `dispatch` and `sub-driver` to `.spec-driver/skills.allowlist`
2. Run `uv run spec-driver skills sync` — verify both skills install to
   `.claude/skills/`
3. Verify dispatch-worker agent definition loads (check via `/agents` or
   `claude agents`)
4. VA-001: Review all three files against project skill/agent conventions
5. VH-001: Invoke `/dispatch` on a real or synthetic multi-phase IP —
   verify dispatch plan is produced and parallel dispatch occurs
6. VH-002: Verify worktree isolation during parallel execution
7. VH-003: Verify model routing (check agent invocations use correct model)
8. Record verification evidence in phase notes

**Exit criteria:**
- Skills sync cleanly
- Agent definition loads
- All VH/VA verification items pass or have documented exceptions

## 6. Testing & Verification Plan

No automated tests (pure skill/agent files). Verification is manual (VH)
and agent-review (VA) per the verification coverage block above.
