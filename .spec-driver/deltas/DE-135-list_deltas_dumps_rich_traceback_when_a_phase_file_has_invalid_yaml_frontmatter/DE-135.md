## 1. Summary & Context

- **Product Spec(s)**: PROD-010 – CLI UX (FR-010 actionable error messages).
- **Technical Spec(s)**: none directly cover `supekku/scripts/lib/changes/{registry,artifacts}` or `core/spec_utils` today; treat this as a defect fix in those modules.
- **Implementation Plan**: to be drafted (`IP-135`).
- **Change Drivers**: ISSUE-054.

## 2. Motivation

A single malformed phase file currently breaks `spec-driver list deltas` by escaping all loader-level error guards and surfacing a Rich traceback with no file/line context. This violates PROD-010.FR-010 (actionable errors) and degrades agent and human ergonomics for any workspace with even one bad frontmatter block.

## 3. Scope & Objectives

- **Primary Outcomes**:
  - Loader-level catches around `load_markdown_file` reliably trap YAML parse failures (in addition to validation/IO errors) and convert them into a friendly stderr diagnostic naming the offending file (and line/column when available).
  - `spec-driver list deltas` continues to list well-formed deltas when one phase or delta file is malformed; the bad file is reported and skipped.
  - Applies the same fix consistently to sibling change-artifact loader paths and any other call sites where `load_markdown_file` is wrapped in too-narrow `except` clauses.
- **Operational Constraints**: low-risk defect fix; no behavioural change for well-formed workspaces.
- **Dependencies**: none.

## 4. Out of Scope

- Authoring-time schema validation UX (PROD-004; PROD-010.FR-015 `validate file`).
- Replacing or restructuring the frontmatter loader API beyond what is needed to surface parse errors cleanly.
- Wider audit of every `load_markdown_file` call site outside change-artifact loading; non-change paths covered only if the same defect is observed during implementation.

## 5. Approach Overview

- **System Touchpoints**: `supekku/scripts/lib/core/spec_utils.py` (only file changed); `supekku/scripts/lib/changes/artifacts.py` and `changes/registry.py` are exercised by the new behaviour but unmodified.
- **Key Changes** (per DR-135):
  - Add `MarkdownLoadError(ValueError)` to `core/spec_utils.py`; have `load_markdown_file` wrap `frontmatter.loads(text)` in `try/except yaml.YAMLError`, raising `MarkdownLoadError` with file path and (when available) line/column, chaining the original via `from exc`.
  - No call-site edits. Existing `except (ValueError, OSError)` / `except ValueError` clauses (notably `changes/artifacts.py:181` and `changes/registry.py:85`) start handling parse failures correctly because `MarkdownLoadError` is a `ValueError`.
  - User sees the existing `Console(stderr=True)` skip-and-continue warning in `ChangeRegistry.collect`, now with the new diagnostic text supplied by the helper.
- **Migration / Rollout Notes**: none; pure defect fix.

## 6. Verification Strategy

- **Requirements Coverage**: PROD-010.FR-010 verified by VT exercising the malformed-file path end-to-end through `list deltas` and asserting (a) non-zero or zero exit code per chosen policy, (b) friendly stderr message identifying the file, (c) absence of Python traceback output, (d) remaining well-formed artefacts still listed.
- **Planned Artefacts**: at least one VT for the change-artifact loader path; targeted unit tests around the chosen helper.
- **Acceptance Criteria**: repro from ISSUE-054 produces a one-line diagnostic naming the file and line; `just test` and `just lint` pass.

## 7. Risks & Mitigations

- **Risk**: Broadening the helper masks unrelated bugs. _Likelihood_: low – _Impact_: medium – _Mitigation_: catch only `yaml.YAMLError` inside `load_markdown_file`; chain the original via `from exc` so debugging detail is preserved.
- **Risk**: Some caller wants to handle parse failures differently from validation failures and silently changes behaviour. _Likelihood_: low – _Impact_: low – _Mitigation_: no such caller exists today; `MarkdownLoadError` is a distinct subclass so any future caller that needs the distinction can match on it.

## 8. Follow-ups & Tracking

- **Future Phases / Deltas**: none planned. Optional follow-up issue: normalise inconsistent `except` clauses across `requirements/sync.py`, `requirements/parser.py`, `drift/registry.py` (already silently fixed by this delta, but stylistically divergent).
- **Backlog Items**: ISSUE-054 (closed by this delta).
- **Open Decisions / Questions**: none — settled in DR-135. Skip-and-warn (existing pattern) is canon.

## 9. Implementation Notes

- The existing `Console(stderr=True)` block in `changes/registry.py:84-96` already prints a multi-line validation error; reuse and extend that pattern rather than introducing a new error sink.
- PyYAML errors (`yaml.YAMLError` subclasses) carry `problem_mark` with `line`/`column`; surface those when present.
