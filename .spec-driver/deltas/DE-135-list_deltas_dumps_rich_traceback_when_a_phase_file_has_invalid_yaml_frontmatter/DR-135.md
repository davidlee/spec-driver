# DR-135 – list deltas dumps Rich traceback when a phase file has invalid YAML frontmatter

## 1. Executive Summary

- **Delta**: [DE-135](./DE-135.md)
- **Status**: draft (update when approved)
- **Owners / Team**: tbd
- **Last Updated**: 2026-05-01
- **Synopsis**: Translate PyYAML parse errors at the source (`load_markdown_file`) into a single `ValueError`-derived exception so existing caller `except` clauses surface the friendly diagnostic that already exists in `ChangeRegistry.collect`.

## 2. Problem & Constraints

- **Current Behaviour**: Any malformed YAML frontmatter in a phase, delta, revision, or sibling artifact file causes `frontmatter.loads(text)` inside `load_markdown_file` to raise `yaml.YAMLError`. That exception is not a subclass of `ValueError` or `OSError`, so it escapes:
  - `changes/artifacts.py:181` — `except (ValueError, OSError)` around the per-phase-file call.
  - `changes/registry.py:85` — `except ValueError` in `ChangeRegistry.collect`.
  - And the same too-narrow pattern in `requirements/sync.py`, `requirements/parser.py`, `drift/registry.py`.

  The result is a Rich traceback ending in `ScannerError: mapping values are not allowed in this context` with no file or line context, violating PROD-010.FR-010.
- **Drivers / Inputs**: ISSUE-054 (user-reported repro on `list deltas`); PROD-010.FR-010 (actionable error messages).
- **Constraints / Guardrails**:
  - Must not silently mask unrelated failures: catch only `yaml.YAMLError`, not bare `Exception`.
  - Must preserve the original PyYAML exception via exception chaining (`raise ... from exc`) so debugging information is not lost.
  - Must not change behaviour for well-formed workspaces.
  - Must respect POL-001 (no parallel implementations) and STD-003 (utilities live in `core/`).
- **Out of Scope**:
  - Normalising the inconsistent `except` clauses across `requirements/`, `drift/`, etc. (they will silently become correct via the new error type and are tracked for an optional follow-up).
  - Authoring-time per-file validation UX (PROD-010.FR-015 / PROD-004 work).
  - Replacing `python-frontmatter` or restructuring the loader API beyond adding the typed exception.

## 3. Architecture Intent

- **Target Outcomes**:
  - `load_markdown_file` raises a single, typed exception (`MarkdownLoadError`) whenever its frontmatter cannot be parsed, regardless of whether the underlying cause is YAML scan, YAML parse, or composer error.
  - The exception is a subclass of `ValueError`, so call sites that catch `ValueError` (directly or as part of a tuple) continue to handle the failure correctly without modification.
  - For top-level malformed delta/revision/audit files: the existing `Console(stderr=True)` block in `changes/registry.py:87-89` already prints the friendly skip warning. Now also handles parse failures (was validation-only).
  - For malformed **phase** files inside a delta bundle: `changes/artifacts.py` previously swallowed the error silently via `continue`. We add a small `Console(stderr=True)` warning before the `continue` so the user-reported repro path (ISSUE-054) actually names the offending file. This is a deliberate, scoped call-site edit — the only one in this delta — and exists because the per-phase-file skip path is not reachable from `registry.py`'s warning block.
- **Auto-fixed call sites (catch `ValueError` directly or via tuple)**:
  - `changes/artifacts.py:181` — `except (ValueError, OSError)`
  - `changes/registry.py:85` — `except ValueError`
  - `drift/registry.py:77` — `except (OSError, ValueError, KeyError)`
  - `requirements/parser.py:132` — `except (OSError, ValueError)`
- **Sites that still leak YAML errors after this delta** (acknowledged, deferred):
  - `requirements/sync.py:77` — `except OSError` only. Not on the ISSUE-054 path. Tracked as a follow-up note in DE-135 §8 rather than expanded into this delta.
  - Sites that already catch `Exception` (`backlog/registry.py`, `validation/validator.py`, `changes/registry.py:discover_plans`) are unaffected — they already swallow YAML errors today, just without the typed message; they could be migrated to `MarkdownLoadError` in a follow-up cleanup but do not regress here.
- **Guiding Principles**:
  - Source-of-truth error translation: PyYAML internals stop at the boundary of `core/spec_utils.py`; no other module imports `yaml.YAMLError`.
  - One concern, one place: a single change-set fixes every observed caller bug.
  - Subclass `ValueError` so the existing taxonomy (`FrontmatterValidationError(ValueError)` for schema failures, new `MarkdownLoadError(ValueError)` for parse failures) stays coherent.
- **State Transitions / Lifecycle Impact**: none.

### Helper signature

```python
class MarkdownLoadError(ValueError):
  """Raised when a markdown file's frontmatter cannot be parsed."""


def load_markdown_file(path: Path | str) -> tuple[dict[str, Any], str]:
  path = Path(path)
  text = path.read_text(encoding="utf-8")
  try:
    post = frontmatter.loads(text)
  except yaml.YAMLError as exc:
    mark = getattr(exc, "problem_mark", None)
    where = f" at line {mark.line + 1}, column {mark.column + 1}" if mark else ""
    msg = f"invalid YAML frontmatter in {path}{where}: {exc.__class__.__name__}"
    raise MarkdownLoadError(msg) from exc
  frontmatter_data: dict[str, Any] = dict(post.metadata or {})
  body = post.content.lstrip("\n")
  if body and text.endswith("\n") and not body.endswith("\n"):
    body = body + "\n"
  return frontmatter_data, body
```

### User-facing output (after fix)

For a malformed phase file (the ISSUE-054 path):

```
WARNING: Skipping phase file: invalid YAML frontmatter in /abs/path/.spec-driver/deltas/DE-XXX/phases/phase-01.md at line 4, column 42: ScannerError
```

For a malformed top-level delta/revision/audit file (already-handled path, now also covers parse errors):

```
WARNING: Skipping .spec-driver/deltas/DE-XXX/DE-XXX.md: invalid YAML frontmatter in /abs/path/.spec-driver/deltas/DE-XXX/DE-XXX.md at line 4, column 42: ScannerError
```

In both cases the new diagnostic content (filename + line/column) comes from `MarkdownLoadError`'s message; the "Skipping" prefix and Rich formatting come from the surrounding `Console(stderr=True)` blocks.

## 6. Supporting Context

They will be unaffected by this change but informed its choice (any helper-level guard must remain compatible with their broader catch).
- **Related Deltas / Specs**: PROD-010 (CLI UX), `core/frontmatter_schema.py` (precedent for `FrontmatterValidationError(ValueError)`).
- **Policies / Standards**: POL-001 (reuse over sprawl) — favours single-source fix; STD-003 (utility placement) — confirms `core/spec_utils.py` is the correct home.

## 7. Design Decisions & Trade-offs

- **DEC-DR135-001**: Wrap the YAML error inside `load_markdown_file`, not at each caller. Rationale: POL-001; one place to change, one place to test, no risk of forgetting a call site. Trade-off: callers cannot distinguish parse failure from validation failure by exception type alone — acceptable because in every observed call site the desired behaviour is identical (skip and warn).
- **DEC-DR135-002**: Make `MarkdownLoadError` derive from `ValueError`. Rationale: existing `except (ValueError, OSError)` and `except ValueError` clauses become correct with zero call-site edits, matching `FrontmatterValidationError(ValueError)` precedent. Trade-off: callers that want to *exclude* parse errors from their handler would need to catch `MarkdownLoadError` first and re-raise; no such caller exists today.
- **DEC-DR135-003**: Format the message as `invalid YAML frontmatter in {path}{where}: {ExcClass}` rather than embedding the full PyYAML message text. Rationale: the original exception is preserved as `__cause__` for debugging; the user-facing one-liner stays compact and predictable for tests. Trade-off: callers wanting the full PyYAML detail must walk `__cause__`.
