## 1. Summary

- **Delta**: DE-135 — list deltas dumps Rich traceback when a phase file has invalid YAML frontmatter.
- **Source**: ISSUE-054.
- **Requirement**: PROD-010.FR-010 (actionable error messages).
- **Design**: DR-135.
- **Desired Outcome**: A malformed YAML frontmatter in any change-artifact file causes `list deltas` (and siblings) to emit a single-line stderr warning with file and line/column, skip the bad file, and continue listing well-formed deltas — never a Python traceback.

## 2. Context & Constraints

- **Current Behaviour**: `load_markdown_file` lets `yaml.YAMLError` propagate. Surrounding `except (ValueError, OSError)` clauses do not catch it, so a Rich traceback reaches the user.
- **Target Behaviour**: `load_markdown_file` raises `MarkdownLoadError(ValueError)` with a friendly message; existing `except ValueError`-style guards swallow and surface it.
- **Dependencies**: none.
- **Constraints**:
  - Single-file source change (`core/spec_utils.py`); no call-site edits.
  - Catch only `yaml.YAMLError` — no broader `except`.
  - Preserve original exception via `from exc` chaining.

## 6. Testing & Verification Plan

- **VT-DR135-001 (unit)**: `core/spec_utils_test.py` (or sibling). Build a tempfile with deliberately malformed YAML frontmatter (e.g. unquoted colon inside a value). Assert `pytest.raises(MarkdownLoadError)`; assert message contains the path and a `line`/`column` reference; assert `isinstance(exc.__cause__, yaml.YAMLError)`.
- **VT-DR135-002 (integration)**: prefer reusing existing change-artifact fixtures (`changes/completion_test.py` patterns) over rolling new ones. Construct a workspace with one good delta and one delta with a malformed phase frontmatter; invoke the `list deltas` code path twice (default and `--format json`/equivalent machine-readable mode); assert exit 0; stderr contains the friendly skip warning naming the file and a YAML location; stdout has no Python traceback in either run; the JSON run produces valid JSON; the good delta still appears in stdout.
