## 1. Summary & Context

- **Product Specs**: PROD-004 (Frontmatter Metadata Validation, primary), PROD-010 (CLI UX), PROD-006 (Phase Management, precedent).
- **Technical Specs**: SPEC-116 (frontmatter_metadata), SPEC-114 (blocks/metadata).
- **Implementation Plan**: [IP-136](./IP-136.md) — coordination plan; bulk implementation lives in child deltas.
- **Design Revision**: [DR-136](./DR-136.md) — per-artefact field-placement analysis applying DR-106 §3a's pattern to spec, delta, revision, audit, plus the cross-cutting block-validator and discoverability concerns.
- **Change Drivers**:
  - ADR-010 (placement heuristic) accepted but applied only to phases (DE-106).
  - PROD-004 (frontmatter metadata validation) program is mid-march — three of four capabilities still partly planned.
  - IMPR-024 (kind-aware Pydantic validation across artefacts) — subsumed, awaiting execution.
  - ISSUE-054 (open) — `list deltas` traceback on invalid YAML.
  - User-observed friction: agents guessing status enums; `list deltas` partial info due to body/frontmatter duplication; substantial unenforced JSON Schema.

## 2. Motivation

The current schema landscape inverts the agent-native promise: each artefact's frontmatter is *both* over-large (many fields with no code consumer, duplicated by body sections) *and* under-enforced (validator hard-fails only on the seven base fields; type-specific fields warn or pass silently). Token-cost is paid for upkeep that delivers neither machine-readable richness nor authoring guidance.

The principle to fix this is already doctrine — **ADR-010**: frontmatter for stable contract fields with code consumers; code-fenced `supekku:schema@vN` blocks for schema-versioned data; prose for volatile detail; never duplicate. **DR-106 §3a** demonstrates the analytical pattern (per-field table: current location, current consumer, orchestration value, rationale) applied to phases. This delta executes the same propagation across the four artefacts that pre-date ADR-010, completes the validator unification (DE-118) that gives the placement model its teeth, and folds in two adjacent agent-friction concerns surfaced during preflight: status-enum discoverability beyond the create-time hint already shipped in DE-077, and faster time-to-detect for invalid YAML (ISSUE-054).

## 3. Scope & Objectives

This delta is an **umbrella / framework**. Its body is coordination; its bulk is delegated to child deltas which it sequences and gates.

- **Primary outcomes (program-level)**:
  1. ADR-010 placement heuristic uniformly applied across **spec, delta, revision, audit** (phase already done). For each: a DR-106-style field-placement analysis exists and the artefact conforms.
  2. **Single validation layer**. DE-118 lands first; hand-rolled block validators retired; `MetadataValidator` covers frontmatter and blocks; `BlockSchema` registry connected to validation; active regex bugs in `REVISION_BLOCK_JSON_SCHEMA` removed.
  3. **Revision gets a metadata class** (currently the only artefact in this group without one).
  4. **`list X` enrichment** for delta, spec, revision, audit — informative columns derived from parsed model fields (frontmatter plus canonical `supekku:*@vN` blocks); formatters never re-parse markdown body, per POL-003. Generalises the DE-131 pattern.
  5. **Status-enum discoverability at edit time** — agents updating an existing artefact see allowed values without retrieving memory. Mechanism TBD in DR (skill reinforcement, template comments, validator hints, or all three).
  6. **Faster invalid-YAML feedback** — ISSUE-054 closed; `list`/`show` paths surface a clear, single-line error rather than a Rich traceback or silent skip.
  7. **PROD-004 capabilities closed**: FR-001, FR-002, FR-003, FR-005, FR-006, FR-007, NF-001 move from `planned` to `verified` (FR-004 already verified).
- **Operational constraints**:
  - **No silent breakage of existing artefacts.** Every cut field needs a read-time tolerance window; deletions only after sweep + observation.
  - **No new ADR.** ADR-010 governs; this program implements.
  - **Bounded blast radius per child delta.** One artefact per delta, plus the cross-cutting deltas (DE-118, discoverability, list enrichment).
  - **Verifiable closure.** Each child delta must demonstrate (a) its target artefact has no *structural* duplication — i.e. no parsed fact appears in both frontmatter and a canonical `supekku:*@vN` block — and (b) `validate` plus `list` plus `show` round-trip without regression on real artefacts. Narrative prose that restates structured facts is acceptable per DR-136 §11.5 and addressed via opt-in `validate --prune-duplicates`, not gated.
- **Dependencies**:
  - DE-118 must land before any per-artefact propagation delta touches block schemas.
  - PROD-004 / SPEC-116 / SPEC-114 spec text reconciliation occurs as the umbrella's audit/closure step, not up front (delta-first per workflow stance).

## 4. Out of Scope

- **Bulk migration of legacy artefacts** — IMPR-023 (legacy phase sheets) and equivalent for other artefacts remain separate. This program ensures *new* artefacts conform and *legacy* artefacts read tolerantly.
- **New artefact kinds** — no taxonomy expansion (drift ledgers, structured handover, orchestration unaffected).
- **Compaction profile authoring beyond what PROD-004 FR-007 already specifies** — DE-036 infrastructure is reused, not redesigned.
- **Spec-revision-first reframing** — workflow stance is delta-first; we do not pre-revise PROD-004/SPEC-116/SPEC-114. Any spec patches happen at audit/close.

## 5. Approach Overview

- **System touchpoints**:
  - `supekku/scripts/lib/core/frontmatter_metadata/` (per-artefact metadata classes; revision class to be created)
  - `supekku/scripts/lib/blocks/metadata/` and `supekku/scripts/lib/blocks/` (block validator unification)
  - `supekku/scripts/lib/validation/validator.py` (single layer post-DE-118)
  - `supekku/scripts/lib/changes/{artifacts,registry,*_creation}.py` (artefact loaders, creation)
  - `supekku/scripts/lib/specs/{models,registry,creation}.py`
  - `supekku/scripts/lib/formatters/{change,spec,decision}_formatters.py` and `column_defs.py` (`list X` enrichment)
  - `supekku/cli/list/*.py`, `supekku/cli/show/*.py`, `supekku/cli/edit/*.py` (edit-time discoverability surface)
  - `.spec-driver/templates/*.md` (template comments / inline enum guidance)
  - `.claude/skills/*` (skill reinforcement for status-enum edit-time discovery)
- **Placement policy (Q1)**: place each field by data nature; enforce single-source authority. **Frontmatter** for identity/lifecycle/simple-list contract fields; **code-fenced `supekku:X@vN` blocks** for structured records, schema-versioned data, lists of objects with constraints — authoritative, not duplicative; **prose** for narrative (motivation, rationale, observations). ADR-010 "never duplicate across locations" is the active constraint.
- **Validator strictness policy (Q5)**: asymmetric — **strict on `validate` and CI**, **tolerant on read**. `validate` hard-fails on (1) unknown frontmatter keys, (2) non-canonical status enums, (3) type-specific field violations, (4) missing required fields beyond the base-7. Reads continue to use `normalize_status()`-style tolerance; `ChangeRegistry` skip-on-error continues. Each per-artefact child delta runs a `validate` sweep and dispositions every legacy non-conformance: (a) `validate --fix` applies, (b) accept as legacy with a tracked drift entry, or (c) extend `tolerated_aliases`.
- **Key changes (program-level)**:
  - **DE-118** — block-validator unification (drafted; first child).
  - **DE-1xx (cross-cutting)** — edit-time discoverability + authoring-time validation surface + skill tuning + migration orchestrator. Single delta sequenced after DE-118. Scope:
    1. **Y** — template-generation pipeline reads metadata classes; generated artefact templates carry inline enum-comment hints (e.g. `status: draft  # one of: draft|pending|in-progress|completed|deferred`). Comment-preserving YAML emitter required.
    2. **Z** — `MetadataValidator` actionable errors with did-you-mean and named alias autocorrect ("status 'complete' is a known alias for 'completed'; run `validate --fix`"); `validate --fix` extended to cover the full canonical-alias table for all enum fields.
    3. **X** — `spec-driver schema enums [<kind>.<field>]` CLI surface.
    4. **`spec-driver validate file <path>`** — authoring-time validation surface deferred from DE-135 (PROD-010.FR-015).
    5. **Skill tuning** — wire `validate` (and `sync` where appropriate) into `execute-phase` (phase closure), `close-change` (delta close), `audit-change`, `notes`, `update-delta-docs`. `sync` runs only at delta close or on explicit request; `validate` runs broadly.
    6. **Migration subsystem skeleton** — `supekku/scripts/lib/migrations/` package, `MigrationStep` base, registry, `admin migrate` orchestrator. Architecturally separated from application code per POL-003. Per-kind `workflow.toml` strict flag + schema_version state. New installs ship strict-on; upgrading installs preserve existing tolerance.
  - **DE-1xx (delta)** — first per-artefact propagation; field-placement table for delta frontmatter; resolve `summary` / `outcome_summary` / `risk_register` / `context_inputs` / `applies_to` body-vs-frontmatter authority; enrich `list deltas` from frontmatter only.
  - **DE-1xx (spec)** — second; largest duplication surface; resolve `concerns`, `responsibilities`, `guiding_principles`, `assumptions`, `hypotheses`, `decisions`, `constraints`, `verification_strategy`, `scope`; per-field choose frontmatter vs block vs prose per Q1 policy; enrich `list specs`. **Burden note**: taxonomy-strict migration may produce 50–80 drift entries across in-tree specs (unit-vs-assembly determination needs human judgement). Spec child delta closes when sweep's structural changes complete; manual taxonomy reconciliation is tracked separately and gated by umbrella close, not spec-delta close.
  - **DE-1xx (requirements-in-spec)** — block-ify requirement records currently parsed by `requirements/parser.py` regex from spec prose. New `supekku:spec.requirements@v1` block; strict-validated requirement-lifecycle enum; interactive `admin migrate-requirements <SPEC-ID>` tool; legacy regex parser retained as tolerant fallback during migration.
  - **DE-1xx (audit)** — fourth; tighten `outcome` / `disposition` enum enforcement (AUD-012's `outcome: pass` becomes a validate-strict finding); enrich `list audits`.
  - **DE-1xx (revision)** — fifth; introduce `REVISION_FRONTMATTER_METADATA` in `core/frontmatter_metadata/revision.py` (POL-003 placement); reconcile `revision_creation.py`; decide custom-field placement (frontmatter vs `supekku:revision.change@vN` block vs body §1).
  - **PROD-004 / SPEC-116 / SPEC-114 reconciliation** — at umbrella close; spec coverage updates; FR statuses to `verified`.
- **Sequence (Q4 confirmed; expanded with requirements-in-spec)**:
  1. **DE-118** — block validator unification.
  2. **Cross-cutting** — discoverability surfaces + authoring-time validate + skill tuning.
  3. **Delta** — addresses user's named `list deltas` pain; pilots the placement-table pattern.
  4. **Spec** — largest surface; benefits from delta's precedent.
  5. **Requirements-in-spec** — retires the regex parser; structured requirement-lifecycle enforcement.
  6. **Audit** — strict enum enforcement matters most here; requirement lifecycle now strictly validated.
  7. **Revision** — last; metadata class invented with four precedents in hand.
  8. **Umbrella close** — spec reconciliation, PROD-004 FR statuses, conformance audit.
- **Migration / rollout**:
  - Reads tolerant via `normalize_status()` pattern, generalised per-field as needed.
  - Each per-artefact child delta runs a `validate` sweep and produces a per-case disposition list.
  - No bulk rewrites in this program — opportunistic via `validate --fix` only.

## 6. Verification Strategy

- **Requirements coverage**: PROD-004 FR-001..FR-007 and NF-001 are verified by the *aggregate* of child deltas. Each child delta's IP must declare which PROD-004 FRs it advances and to what state. The umbrella's closure check confirms every requirement reached `verified`.
- **Planned artefacts**:
  - VT — per-child-delta unit/integration tests ensuring metadata-driven validation matches prior behaviour or is intentionally stricter.
  - VA — agent-run sweep of `.spec-driver/**/*.md` validating against the new metadata after each child delta.
  - VH — user attestation at umbrella close that `list X` outputs are useful and edit-time discoverability is non-trivial improvement.
- **Acceptance criteria (program-level, Q6 medium audit gate)**:
  - DE-118 closed; zero hand-rolled block validators remain; `MetadataValidator` rejects unknown keys for both blocks and frontmatter.
  - All 5 artefact types have a metadata class in `core/frontmatter_metadata/`; revision included.
  - **Cross-artefact sweep**: zero new structural duplications introduced (no parsed fact in both frontmatter and a canonical `supekku:*@vN` block); for sampled artefacts (≥5 per kind) every structured field has exactly one canonical home per the DR-136 placement tables. Legacy narrative prose excluded — see DR-136 §11.5.
  - `list specs|deltas|revisions|audits` each surface useful columns derived from parsed model fields (frontmatter plus canonical `supekku:*@vN` blocks); no markdown body re-parsing in formatters per POL-003.
  - **Strict-on-validate** active: `validate` across `.spec-driver/**` either passes or every failure has an explicit drift-register entry or `tolerated_aliases` extension with sunset criteria.
  - **Discoverability surfaces verified**: Y (template hints) generated correctly for all kinds; Z (actionable errors + did-you-mean + autocorrect) demonstrated for at least one canonical alias per field; X (`schema enums` CLI) returns valid enum values for every controlled-vocab field.
  - **Authoring-time** `spec-driver validate file <path>` returns clear single-line diagnostics for invalid YAML and schema violations.
  - **Skill tuning verified**: `execute-phase`, `close-change`, `audit-change`, `notes`, `update-delta-docs` invoke `validate` at the named gates; diff visible in commits.
  - PROD-004 FR-001..FR-007, NF-001 status: `verified`.
  - `complete delta DE-136` succeeds without `--force`.
