# DR-136 – metadata schema consolidation program: propagate ADR-010 across artefacts and close PROD-004

## 1. Executive Summary

- **Delta**: [DE-136](./DE-136.md)
- **Status**: draft
- **Synopsis**: ADR-010 established a placement heuristic for structured artefact metadata in 2026-03-22, and DE-106 applied it to phase sheets. This DR extends the same principle, with the same per-field analytical pattern (DR-106 §3a), to the four artefact types that pre-date ADR-010: **delta, spec, audit, revision**. It also delivers the cross-cutting infrastructure that makes the placement model self-enforcing — unified block/frontmatter validation (via DE-118), authoring-time discoverability (template enum-comments, did-you-mean errors with alias autocorrect, `schema enums` CLI), authoring-time validation surface (`validate file <path>`), and skill-level integration so `validate` runs at workflow gates rather than depending on agent recall. Validator becomes asymmetric — strict on `validate`/CI, tolerant on read — so legacy artefacts continue to load while drift becomes explicit reconciliation rather than silent precedence (ADR-008). Migrations live in a separate package (POL-003) so version-bridging concerns don't taint hot-path application code. Consumers upgrade at their own pace via `admin migrate`.

## 2. Problem & Constraints

### Current behaviour

- Frontmatter, code-fenced YAML blocks, and prose body sections frequently carry the same fact across delta, spec, audit, revision (`summary`, `outcome_summary`, `risk_register`, `context_inputs`, `applies_to`, plus most spec body fields). Agents must keep parallel representations in sync; they don't.
- `list deltas` (and siblings) renders only ID/Name/Tags/Status because the data they could surface either lives in body sections (formatters can't reach there per POL-003) or is duplicated unsafely in frontmatter.
- Validator is loud on the base 7 fields and silent on type-specific fields. Real artefacts contain invalid values (AUD-012's `outcome: pass`) without surfacing.
- Revision has no metadata class; custom fields written by creation code are read only from body.
- Status enums documented only in a memory record, not where agents author or edit.
- Requirements records inside specs are parsed by regex from prose; lifecycle hides in prose; structurally fragile.
- Hand-rolled block validators co-exist with declarative `BlockMetadata` (DE-118 unfinished), with active regex bugs and strictness gaps.

### Drivers / inputs

- ADR-010 placement heuristic — accepted, applied only to phases.
- ADR-002 forward-only links; ADR-008 normative truth ownership; ADR-009 no speculative structure; ADR-011 Workspace-mediated registries.
- PROD-004 capability program — six of seven verification entries still `planned`.
- IMPR-024 (subsumed) — kind-aware Pydantic validation across artefacts.
- ISSUE-054 (closes via DE-135) — `list deltas` traceback on bad YAML; authoring-time gap closes here.
- DE-118 (drafted) — block validator unification.
- User-observed friction during preflight: agents guessing status enums; `list deltas` partial info; validator silence.

### Constraints / guardrails

- ADR-010 governs placement; this DR implements, does not replace.
- ADR-002 (forward-only frontmatter links) preserved.
- ADR-008 (specs own normative truth) preserved.
- ADR-011 (CLI registry access via Workspace) — `list X` enrichment uses Workspace.
- POL-003 (5-layer architecture) — metadata classes in `core/`; validators in domain/orchestration; formatters never parse body; migrations in their own package.
- POL-001 + STD-003 — DE-118 unification serves directly; utilities go in `core/`; no parallel implementations.
- POL-002 — controlled vocabularies as named enums.
- No bulk rewrite of legacy artefacts; opportunistic via `admin migrate <kind>`.
- Consumers upgrade gradually; strict mode opt-in per kind.

### Out of scope

- Bulk migration of legacy phase sheets (IMPR-023 — adjacent).
- New artefact taxonomies (drift, structured handover, orchestration unaffected).
- Pre-emptive spec revision (delta-first; spec reconciliation at umbrella close).
- Relations-shape unification across governance artefacts (OQ-RELATIONS deferred).
- Moving requirements to separate per-requirement files (a bigger taxonomy question).

## 3. Architecture Intent

### Target outcomes

- Every **structured fact** across delta/spec/audit/revision has exactly one canonical, parsed home, chosen by data nature: identity/lifecycle/simple lists in **frontmatter**; structured records in **`supekku:X@vN` blocks**, authoritative; narrative in **prose**. Narrative prose that restates a structured fact is acceptable while the structured form remains canonical and the prose remains read-only — only structural duplication (the same parsed fact in two parsed locations) is forbidden (resolves external review F2).
- `MetadataValidator` is the single validation layer for both frontmatter and code-fenced blocks (DE-118 closes the gap). Strict on `validate`, tolerant on read.
- Templates carry inline enum hints derived from metadata; agents see allowed values where they edit.
- `validate` runs at named workflow gates (phase-close, delta-close, audit) via skill tuning — discipline doesn't depend on agent memory.
- `list X` derives display from frontmatter and parsed blocks alone; no body parsing in formatters.
- Migration code is sectioned off from application code; consumers upgrade via opt-in `admin migrate <kind>`.

### Guiding principles

- **Single canonical home per *structured* fact.** ADR-010 "never duplicate" applied to structured/parsed data; narrative prose that restates structured facts is permitted (excised opportunistically, not gated).
- **Place by nature, not by ceremony.** Structure where structure helps; prose where prose is genuine.
- **Strict on intentional checks, tolerant on incidental ones.** `validate` is intentional. Loading is incidental.
- **Surface the schema where the agent works.** Templates, validator, CLI all return enum guidance.
- **Promote on need, not speculation.** Block schemas exist when a concrete consumer or constraint is in the same delta.
- **Migrations are a separate subsystem.** Application code stays current-schema; legacy lives in `migrations/`.

### State transitions / lifecycle impact

- No new artefact statuses introduced.
- Revision gets first-class metadata; its lifecycle transitions become validator-governed.
- Validator behaviour shifts from "warn on type-specific fields" to "fail on `validate`, tolerate on read."
- Strict-mode flag in `workflow.toml` is per-kind, allowing partial program states.

## 5. Cross-cutting child delta — detailed design

Single delta sequenced after DE-118. Five concrete deliverables.

### 5.1 Y — Template generation from metadata

**Vendor side (spec-driver maintainer):**
- `BlockMetadata.render_template_frontmatter() -> str` returns YAML with placeholders and inline enum-comment hints (e.g. `status: draft  # one of: draft|pending|in-progress|completed|deferred`).
- `spec-driver admin regenerate-templates` regenerates the YAML frontmatter section in each `supekku/templates/*.md`, leaving body untouched. Idempotent.
- `spec-driver validate templates` (CI gate) — fails build on metadata/template drift.
- **Comment-preserving emitter (resolves adversarial F-7)**: regeneration uses `ruamel.yaml` (already a transitive dep, otherwise add) or a custom round-trip emitter that preserves YAML comments. Standard `pyyaml` strips comments; using it would defeat Y's purpose. The renderer test (VT-CC-001) verifies a regenerated template still contains the enum-comment string for at least one field per kind.

**Install side (downstream user):**
- Single template layer at `.spec-driver/templates/*.md` (real files, copied not symlinked).
- New `copy_templates_split_aware()` in `core/templates.py` — for each template file, splits into (frontmatter, body); frontmatter auto-updated to match vendor (no prompt, installer-owned); body prompted on diff per existing `copy_directory_if_changed` UX (project-owned).
- **Malformed user frontmatter (resolves adversarial F-13)**: if user's `.spec-driver/templates/<kind>.md` has unparseable YAML in its frontmatter, install fails loudly with the file path and parse error. Never silently overwrites; never partially merges. User must fix or delete the local file.
- `admin preboot` runs `validate templates` non-fatally over project templates and surfaces drift hints.

### 5.2 Z — Actionable validation + alias autocorrect

- `FieldMetadata.aliases: Mapping[str, str] | None` — alias → canonical (permanent equivalences, e.g. `complete → completed`).
- `FieldMetadata.tolerated_aliases` — time-bounded escape hatch with sunset trigger; used during migration windows (e.g. `unknown` for spec category).
- `MetadataValidator` enum failures emit:
  ```
  delta.md: status 'live' is not a valid value.
    Allowed: draft, pending, in-progress, completed, deferred.
    Did you mean: 'in-progress'?
    Run `spec-driver validate --fix` to apply.
  ```
- Did-you-mean utility lives in `supekku/scripts/lib/core/string_utils.py` (Levenshtein, top-1 within distance 3) per STD-003.
- `MetadataValidator.validate(strict: bool=False)` — loaders pass `strict=False` (alias silently normalised); the `validate` CLI passes `strict=True` (alias becomes a fixable warning).

## 11. Migration mechanics

### 11.1 Architectural separation (load-bearing)

**Application code** (validators, registries, formatters, creation, models, CLI) operates only on the *current* schema. **Migration code** lives in dedicated `supekku/scripts/lib/migrations/` package and bridges legacy formats to current.

| Subsystem | Knows about | Imported by |
|---|---|---|
| `core/frontmatter_metadata/` | Current schema only; permanent `aliases`; time-bounded `tolerated_aliases` | application |
| `validation/` | Current metadata; canonicalises via `aliases`; never bridges versions | application |
| `migrations/` | Legacy schemas (per version); transforms legacy → current | only `admin migrate` orchestrator |
| `cli/admin/migrate.py` | Orchestrates migration steps | imports `migrations/`; nothing imports it |

**Boundary, restated precisely** (resolves adversarial F-1; reconciled with §11.4 per external review F1):

Migration steps **may** import from `core/` (file ops, parsing utilities like `load_markdown_file`, raw YAML/markdown parsers) and `blocks/` (extractors). Migration steps **must not** import from `validation/`, registries, or `cli/`. Each step encodes its legacy-schema knowledge directly in its transform code; "tolerant reading" of legacy data means parsing raw YAML/markdown without going through the schema validator at all. Re-validation of post-migration output runs through the application's strict path via the `admin migrate` orchestrator after the step completes — not from inside the step.

Rationale: the validator is current-schema-only by §11.1's main claim. Letting migration steps invoke it (even in tolerant mode) couples migrations to whichever current schema happens to be loaded, defeating the version-bridging purpose. Steps own their legacy shape; the validator owns the destination shape; the orchestrator joins them.

**`aliases` vs migration steps:**
- `FieldMetadata.aliases` — values equivalent to canonical; long-lived, not coupled to a specific migration window. In metadata.
- `FieldMetadata.tolerated_aliases` — time-bounded escape hatch with sunset; e.g. `unknown` for spec category until manual reconciliation.
- `migrations/v0X_*.py` — structural change between schema versions. Versioned. In migrations package.

`validate --fix` is reserved for current-schema canonicalisation (apply current `aliases`). Version-bridging happens via `admin migrate` dispatching `MigrationStep` modules.

### 11.2 In-repo migration (DE-136 program-internal)

Each child delta's sweep runs `spec-driver admin migrate <kind>` against this repo's `.spec-driver/`:
- Drift register entries land in this repo's `.spec-driver/drift/DL-*.md`.
- Per-kind strict flip in this repo's `workflow.toml`.
- `tolerated_aliases` additions ship to consumers; sunset triggers reference delta IDs only this repo's program tracks.
- Each child delta cannot close until sweep produces clean strict validate or every error has a drift entry.

Standard sweep procedure per kind (corrected order — pre-migration data must be read tolerantly, resolves adversarial F-2; CLI shape reconciled with current `validate` surface plus `--kind` flag landed in §5.4a, resolves external review F4):
```bash
spec-driver validate --kind <kind>                                  # 1. read state in tolerant mode (kind hasn't migrated yet)
spec-driver admin migrate <kind> --dry-run                          # 2. preview
spec-driver admin migrate <kind>                                    # 3. apply
spec-driver validate --kind <kind> --strict                         # 4. re-validate strict (post-migration)
# 5. for residual errors: manual fix + drift entries
spec-driver validate --kind <kind> --strict --no-tolerated-aliases  # 6. final gate before strict flip
```

### 11.3 Consumer-repo migration

Consumer runs `spec-driver install` with newer package; post-install detects schema-version drift and surfaces migration hint.

Design constraints:
1. Strict mode OFF by default for **upgrading** consumer repos. Per-kind strict flip is opt-in.
2. **Strict mode ON by default for fresh installs** post-DE-136 (resolves adversarial F-19). New projects have no legacy data; shipping tolerant-by-default would only delay the discipline. Detection: install creates a fresh `workflow.toml` ⇒ strict-on; install detects existing `workflow.toml` ⇒ preserves whatever flags are present (defaults tolerant only on first creation pre-DE-136).
3. `spec-driver install` never auto-mutates consumer artefacts; templates handled via split-aware copy (§5.1).
4. Post-install message surfaces migration path without performing it.
5. Consumer's drift register is theirs; entries created by their migration go in their `.spec-driver/drift/`.

**`workflow.toml` schema (resolves adversarial F-10)**:
```toml
[validation.strict]
delta = false      # per-kind strict-validate flag; default false for upgrading repos, true for fresh installs
spec = false
audit = false
revision = false
plan = false       # phase shares plan metadata

[schema_version]
delta = 1          # current schema version per kind; bumped by migration steps
spec = 1
audit = 1
revision = 1
plan = 1
```

`spec-driver admin migrate` (§5.6) orchestrates. Each migration step is keyed by schema version declared in metadata; consumer's `workflow.toml` records applied migrations. Re-running idempotent. Skipping versions walks intermediate steps in order.

PROD-004 FR-005, FR-006 (gradual migration) honoured: consumers may stay tolerant indefinitely; tightening is opt-in.

Migration log per run in `.spec-driver/run/migrations/<timestamp>-<kind>.md` — files touched, drift entries created, manual reconciliation needed, tolerated_alias additions.

```

├── __init__.py              # registry of migration steps
├── _registry.py             # ordered list keyed by (kind, from_version, to_version)
├── _step.py                 # MigrationStep base class

├── v06_revision_metadata.py # revision metadata + change-block enrichment

```

`MigrationStep` API:
```python
class MigrationStep:
    kind: str
    from_version: int
    to_version: int
    description: str

    def applies_to(self, file_path: Path) -> bool: ...
    def preview(self, file_path: Path) -> StepPreview: ...
    def apply(self, file_path: Path) -> StepResult: ...
```

Steps:
- Read files via `core/spec_utils.load_markdown_file`.
- Write atomically (temp + rename).
- Emit step log + drift entries.
- Independently runnable and testable.
- Never import from `validation/` or registries — work at file level.

### 11.5 Body-prose excision (narrative duplication)

Scope: **narrative duplication only** — prose body sections (delta §7, spec §§3–5, audit Findings, revision §§1+3) that restate structured data living in frontmatter or canonical blocks. Structural duplication (the same parsed fact in two parsed locations — e.g. frontmatter `applies_to` alongside an authoritative `supekku:delta.relationships@v1` block) is **not** narrative duplication and is hard-gated by strict validation, not by this excision flow.

- `spec-driver validate --prune-duplicates [--target <path>]` — interactive per-file flow over narrative duplications only.
- Never silent; never under `--fix`; never in CI.

Per-artefact deltas don't block on narrative body cleanup — strict-mode validation looks at structured data only. Narrative cleanup is opportunistic; umbrella close audit asks for "no new structural duplications introduced," not "all legacy narrative duplications excised."
