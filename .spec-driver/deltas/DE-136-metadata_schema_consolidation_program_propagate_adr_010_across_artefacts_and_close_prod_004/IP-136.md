## 1. Summary

- **Delta**: DE-136 — metadata schema consolidation program (umbrella).
- **Specs Impacted**: PROD-004 (primary, closing), SPEC-116, SPEC-114; collaborators PROD-006, PROD-010.
- **Problems / Issues**: ISSUE-054 (closes via DE-135; authoring-time gap closes via cross-cutting child delta), IMPR-024 (subsumed), IMPR-023 (adjacent — out of scope).
- **Desired Outcome**: ADR-010 placement heuristic propagated across delta/spec/audit/revision; single validation layer; revision metadata class; `list X` enrichment; edit-time discoverability; PROD-004 FR-001..FR-007 + NF-001 verified.

## 2. Context & Constraints

- **Current Behaviour**: Per DR-136 §2 — frontmatter/block/prose duplications across all four target kinds; validator silent on type-specific fields; revision has no metadata class; requirements-in-spec parsed by regex; hand-rolled block validators co-exist with `BlockMetadata`.
- **Target Behaviour**: Per DR-136 §3 — every structured fact has one canonical parsed home; `MetadataValidator` is the single layer (strict on validate, tolerant on read); templates carry inline enum hints; skills invoke `validate` at workflow gates; `list X` derives from parsed model fields only; migrations live in a separate package per POL-003.
- **Dependencies**:
  - DE-118 (block validator unification) — Phase 2 entrance gate.
  - DE-036 (frontmatter compaction infrastructure) — already landed; reused for delta `applies_to` derivation.
  - DE-135 (read-time YAML resilience) — already landed; closes ISSUE-054. Cross-cutting child delta covers authoring-time `validate file <path>`.
- **Constraints**:
  - No bulk rewrite of legacy artefacts; opportunistic via `admin migrate <kind>`.
  - No new ADR; ADR-010 governs.
  - Bounded blast radius per child delta (one artefact kind per delta, plus the two cross-cutting deltas).
  - Workflow stance is delta-first; spec reconciliation happens at umbrella close (Phase 4), not up front.
