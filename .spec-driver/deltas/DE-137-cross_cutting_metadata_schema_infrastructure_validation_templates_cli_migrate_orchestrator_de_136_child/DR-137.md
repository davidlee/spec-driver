# DR-137 – Cross-cutting metadata schema infrastructure: validation+templates+CLI+migrate orchestrator (DE-136 child)

## 1. Executive Summary

- **Delta**: [DE-137](./DE-137.md) — cross-cutting metadata schema infrastructure (DE-136 program child).
- **Canonical design reference**: DR-136 §5 (cross-cutting deliverables), DR-136 §11 (migration mechanics). DR-137 refines and locks the residual implementation decisions and supersedes specific DR-136 wording where noted in §10 (Supersedes).
- **Status**: draft.
- **Synopsis**: Land the mechanism layer of the DE-136 program — strict-mode + alias-aware `MetadataValidator`, metadata-as-source-of-truth for enum values + aliases, comment-preserving template-regenerator with CI gate, `validate` Typer group (`workspace`/`file`/`templates`), `schema enums` CLI, `admin migrate` orchestrator with self-contained semver-linearised migrations, skill-level validation gates, and `workflow.toml` schema additions. Zero artefact data is touched in this delta; per-artefact sweeps land in DE-138..142.

## 2. Problem & Constraints

### 2.1 Current behaviour (verified against codebase)

- `MetadataValidator` (`blocks/metadata/validator.py:60`) supports `strict_unknown_keys` but no broader strict/alias mode; loaders cannot uniformly request strict validation.
- `normalize_status()` (`changes/lifecycle.py:43`) is a single-purpose global helper covering change-status only; no per-field alias surface; no did-you-mean diagnostics.
- `ENUM_REGISTRY` (`spec_driver/orchestration/enums.py`) and `FieldMetadata.enum_values` are two competing sources of enum truth (POL-001 friction).
- `validate` is a single workspace-wide Typer command; no `file` or `templates` subcommand; no `--kind` filter.
- Templates are a mix of body-only-with-Jinja (delta/spec/revision/design_revision/phase) and frontmatter-bearing example (audit only); no metadata-driven regeneration; no CI drift gate.
- No migration framework exists; schema changes either break old artefacts silently or require ad-hoc one-off sweeps.
- `workflow.toml` has no `[migrations]` or `[validation.strict]` sections.
- Five skills (`execute-phase`, `close-change`, `audit-change`, `notes`, `update-delta-docs`) lack validation gates at named lifecycle moments.

### 2.2 Drivers

- DE-136 program (DR-136 §5/§11 — canonical).
- ADR-010 placement heuristic (frontmatter / blocks / prose; never duplicate).
- POL-001 (reuse), POL-002 (no magic strings/numbers), POL-003 (one-way layering).
- STD-001 (Typer/Rich CLI), STD-003 (utility module placement).
- ISSUE-054 residual scope (authoring-time YAML feedback) — re-verified at delta start (VA-CC-002).

### 2.3 Constraints / guardrails

- DE-118 must be closed (verified at DR-137 sign-off; `MetadataValidator.strict_unknown_keys` flag confirmed present).
- POL-003 layering: new modules land in `spec_driver.<layer>` (core/orchestration/presentation) or `spec_driver/migrations/` (new peer); legacy `supekku/scripts/lib/` extensions are additive on existing modules only.
- POL-001 single source of truth: metadata is canonical for enum_values + aliases; lifecycle constants become transition-window re-exports.
- **POL-003 boundary on migrations**: migration step modules import **nothing** from `spec_driver.*` or `supekku.scripts.lib.*` except `spec_driver/migrations/_protocol.py` (frozen forever). Enforced by an import-linter contract entry (VT-CC-021).
- Tolerant-on-read for unmigrated kinds is preserved until per-kind strict-mode is flipped in `workflow.toml`.
- Acceptance honours PROD-004 FR-005/FR-006 (gradual migration; consumers may stay tolerant indefinitely).

### 2.4 Out of scope

- Per-artefact placement work (DR-136 §§6–10) — sibling child deltas DE-138..142.
- Migration **step** implementations — owned by per-artefact deltas. DE-137 ships zero migrations.
- `validate workspace --prune-duplicates` interactive narrative cleanup — deferred (DEC-137-10).
- Legacy-to-layered extraction of existing subsystems (`blocks/`, `validation/`, `core/frontmatter_metadata/`) — separate concern; DE-137 extends in place.
- `validate templates` runtime UX beyond CI gate semantics — minimal surface, not a full diff tool.
- Relations-shape unification (DR-136 DEC-005) — post-umbrella cleanup delta.

## 3. Architecture Intent

### 3.1 Target outcomes

1. **Single validation API surface**: `MetadataValidator.validate(data, *, strict, accept_tolerated)` covers strict-unknown-keys, strict-enum, strict-alias modes in one call. Loaders pass `strict=False` (tolerant; aliases silently normalised); `validate workspace/file --strict` passes `strict=True` (aliases become fixable warnings).
2. **Metadata as single source of truth** for enum_values, aliases, tolerated_aliases. `ENUM_REGISTRY` is a derived view; lifecycle constants become transition-window re-exports; `normalize_status()` retired in favour of `normalize_field(kind, field_name, value)`.
3. **Template regenerator + CI gate**: `spec-driver admin regenerate-templates` and `spec-driver validate templates` keep `supekku/templates/*.md` frontmatter aligned with metadata; created artefacts also carry inline enum-comment hints via a comment-preserving emitter at dump time. Two emit paths kept aligned by the CI gate.
4. **`validate` Typer group** with `workspace` / `file` / `templates` named peers; bare `validate` dispatches to `workspace` via callback.
5. **`spec-driver schema enums [<kind>[.<field>]]`** CLI surfaces enum_values + aliases + tolerated_aliases from metadata.
6. **`spec-driver admin migrate`** orchestrator discovers self-contained migrations in `spec_driver/migrations/<spec-driver-version>_<NNN>_<slug>/`, applies them in semver order, records `[migrations] last_applied` watermark in `workflow.toml`, re-validates via the **current** `MetadataValidator` after each step.
7. **Skill gates**: five named skills gain explicit `validate workspace`/`validate file` insertion points per DR-136 §5.5 (text shipped verbatim).
8. **`workflow.toml` schema extensions**: `[migrations] last_applied`, `[validation.strict]` per-kind toggle, optional `[schema_version]` derived cache.

### 3.2 Guiding principles

- **Strict capture-of-the-day for migrations**: each migration is a frozen, self-contained script. Only `_protocol.py` is shared. Migration steps work at the bytes-and-stdlib-yaml level — never importing schema, validation, or registry code. This is the framework's load-bearing invariant (DEC-137-11).
- **Mechanism vs sweep**: DE-137 ships the framework. Per-artefact deltas (DE-138..142) author the migrations and exercise them. DE-137 acceptance does NOT include applying any migration to this repo.
- **Tolerant on read; strict on validate/CI**: per kind, per opt-in `[validation.strict]`. Asymmetric per DR-136 §5.2 Q5.
- **Single source of truth wherever possible**: metadata for enum_values + aliases; `last_applied` watermark for migration state; templates for reference frontmatter (regenerated from metadata).
- **POL-003 boundary discipline**: enforced at architectural test layer (import-linter contract); new code lands in `spec_driver.<layer>`; legacy extensions are additive only.

## 5. Per-deliverable detail

### 5.1 Deliverable Y — Template regeneration + comment-preserving emit

#### Comment-preserving YAML emitter

New module `spec_driver/core/yaml_emit.py`:

```python
def emit_yaml_block(
  data: Mapping[str, Any],
  comments: Mapping[str, str] | None = None,
) -> str:
  """Emit YAML for *data* with optional inline `# comment` per top-level key.

  - Top-level keys with simple-type values (str/int/bool/None/[]) accept
    inline trailing comments when *comments* maps that key to a string.
  - Container values (dict/list-of-dicts) emit normally; comment, if any,
    follows the key line.
  - Nested structures emit via stdlib yaml.safe_dump (no comment support).
  - Output is stable: keys preserve dict insertion order.
  """
```

~60 LOC pure-Python + stdlib `yaml`. **Not ruamel.yaml** — one-way emit only, narrow surface, no round-trip required; zero new direct deps. Tracked as OQ-137-01: if implementation surface exceeds ~120 LOC or hits stdlib-yaml edge cases during VT-CC-005 implementation, reverse to ruamel.yaml.

#### Template regenerator + validator

```python
TEMPLATE_PLACEHOLDERS: Mapping[str, Mapping[str, str]] = {

def render_template_frontmatter(kind: str) -> str:
  """Render reference frontmatter for *kind* from BlockMetadata.

  - Required fields with placeholder mapping ⇒ emit `{{ placeholder }}`.
  - Required fields without ⇒ emit metadata's first example value or sensible default.
  - Optional fields ⇒ emit field's default_value.
  - Enum fields ⇒ canonical first value + inline comment `# one of: a | b | c`.
  - Aliases NOT shown in template (validator-side concern only).
  - Tolerated aliases ⇒ trailing comment `# tolerated: x (sunsets at ...)` if present.

def regenerate_template(kind: str, template_path: Path) -> bool:

  Returns True if file changed. Atomic (temp + rename). Body preserved verbatim.
  If template lacks leading `---` block, insert one at top.
  """

def validate_templates(repo_root: Path) -> list[TemplateDrift]:
```

### 5.2 Deliverable Z — Strict validation, alias autocorrect, did-you-mean

#### FieldMetadata extensions

In `supekku/scripts/lib/blocks/metadata/schema.py` (additive):

```python
@dataclass
class FieldMetadata:
  # ... existing fields ...

  aliases: Mapping[str, str] | None = None
  """Permanent alias → canonical map. Equivalent values, not deprecations.
  Example: `{"annotation": "nature"}` (F-B). `validate --fix` rewrites
  aliases to canonical at the file level. Loaders apply aliases silently
  at read time when strict=False."""

  tolerated_aliases: Mapping[str, ToleratedAlias] | None = None
  """Time-bounded alias → metadata about sunset. Migration-window only.
  `validate --no-tolerated-aliases` rejects these; `validate --fix` does NOT
  rewrite them (migration concern)."""

@dataclass(frozen=True)
class ToleratedAlias:
  canonical: str
  sunset_after: str           # delta ID or semver (e.g. "DE-140" or "0.11.0")
  rationale: str
```

Both fields default `None`; opt-in per FieldMetadata.

#### Two-axis strict matrix

`MetadataValidator.validate` gains a `strict: bool` flag + `accept_tolerated: bool`:

| `strict` | `accept_tolerated` | Tolerated aliases | Permanent aliases | Unknown keys | Enum violations |
|---|---|---|---|---|---|
| `False` (loader default) | (n/a; tolerated silently normalised) | silent normalise | silent normalise | silent accept | silent accept |
| `True` (CLI `--strict`) | `True` (default) | warning + sunset note | warning + `--fix` hint | **error** | **error** + did-you-mean |
| `True` | `False` (`--no-tolerated-aliases`) | **error** | warning + `--fix` hint | **error** | **error** + did-you-mean |

```python
class MetadataValidator:
  def __init__(self, metadata: BlockMetadata):
    self.metadata = metadata

  def validate(
    self,
    data: dict[str, Any],
    *,
    strict: bool = False,
    accept_tolerated: bool = True,
  ) -> list[ValidationError]: ...
```

Existing `__init__(metadata, strict_unknown_keys=False)` parameter retired (DEC-137-14); callers in `validation/validator.py` and `blocks/metadata/validator.py:60` consumers migrate to passing `strict=` to `.validate()`. Ripple is small (validator is mostly called from `validation/validator.py`).

#### Error message format

```
delta.md: status 'live' is not a valid value.
  Allowed: completed, deferred, draft, in-progress, pending
  Did you mean: in-progress
  Run `spec-driver validate workspace --fix` to apply
```

For alias normalisation under strict:
```
delta.md: relations[0].annotation is a known alias for 'nature'
  Run `spec-driver validate workspace --fix` to canonicalise
```

For tolerated alias under `--no-tolerated-aliases`:
```
delta.md: status 'unknown' is a tolerated alias for 'pending' (sunsets at DE-140)
  --no-tolerated-aliases mode rejects tolerated aliases
```

`ValidationError` gains optional `fix_hint: str | None` carrying the canonical replacement; `validate workspace --fix` consumes this.

#### Did-you-mean utility

New `spec_driver/core/string_utils.py`:

```python
def closest_match(value: str, candidates: Iterable[str]) -> str | None:
  """Closest match to *value* from *candidates*, or None.

  Stdlib difflib.get_close_matches(n=1, cutoff=0.6). Empty/single-char
  inputs return None.
  """
```

~5 LOC. Used by `MetadataValidator` for enum violations; reusable for any future enum diagnostic surface.

#### `validate workspace --fix` behaviour

Existing `--fix` auto-repairs non-canonical phase statuses (`validator.py:548`). Extended to apply any alias canonicalisation surfaced via `ValidationError.fix_hint`. Per-kind, per-field, idempotent. Atomic per-file write (temp + rename).

- Permanent alias mismatch ⇒ `--fix` rewrites file.
- Tolerated alias ⇒ `--fix` leaves file untouched (migration concern; rides through `admin migrate`).
- Unknown key ⇒ never auto-fixed (no canonical target).
- Enum violation with did-you-mean ⇒ never auto-fixed (high false-positive risk; explicit hint only).

#### Source-of-truth migration: ENUM_REGISTRY becomes derived

Transition shape:

1. **DE-137 ships**: each per-kind metadata file declares enum_values + aliases + tolerated_aliases on its enum FieldMetadata entries.
2. **Lifecycle constants** (`CHANGE_STATUSES`, `REQUIREMENT_STATUSES`, etc.) become thin re-exports:
   ```python

   from supekku.scripts.lib.core.frontmatter_metadata.delta import DELTA_FRONTMATTER_METADATA
   CHANGE_STATUSES: frozenset[str] = frozenset(
     DELTA_FRONTMATTER_METADATA.fields["status"].enum_values
   )
   ```
3. **`ENUM_REGISTRY`** becomes a registry walk:
   ```python
   def _kind_status(kind: str) -> Callable[[], list[str]]:
     def _provider() -> list[str]:
       meta = FRONTMATTER_METADATA_REGISTRY.get(kind)
       if meta is None:
         return []
       return sorted(meta.fields["status"].enum_values or [])
     return _provider
   ```
4. **`normalize_status()`** retired in favour of:
   ```python
   # supekku/scripts/lib/blocks/metadata/aliases.py — NEW

     """Apply FieldMetadata.aliases (+ tolerated_aliases) to value at read time.
     Returns value unchanged if no metadata available."""
   ```
5. **`CANONICAL_STATUS_MAP`** (`changes/lifecycle.py:30`) data migrates to `FieldMetadata.aliases` on each kind's status field. Phase-status aliases (`active`, `done`, `in_progress`) become aliases on plan-status FieldMetadata.

### 5.6 Deliverable 7 + 8 — `admin migrate` orchestrator + `workflow.toml`

#### `_protocol.py` — frozen forever

`spec_driver/migrations/_protocol.py`:

```python
from dataclasses import dataclass, field
from pathlib import Path
from typing import Protocol

@dataclass(frozen=True)
class StepPreview:
  touched: list[Path]
  skipped: list[Path]
  drift: list[str]                          # human-readable drift descriptions

@dataclass(frozen=True)
class StepResult:
  touched: list[Path]
  skipped: list[Path]
  drift_entries: list[Path] = field(default_factory=list)  # paths to written drift files

class MigrationStep(Protocol):
  applies_to_kinds: tuple[str, ...]
  description: str
  def applies_to(self, file_path: Path) -> bool: ...
  def preview(self, file_path: Path) -> StepPreview: ...
  def apply(self, file_path: Path) -> StepResult: ...
```

~30 LOC. Treat any change here as a framework v2 break. Each migration imports from `_protocol` only; nothing else from inside the spec-driver package (DEC-137-11; VT-CC-021 enforces via import-linter).

#### Migration folder structure

```
spec_driver/migrations/
├── _protocol.py                            # FROZEN
├── <spec-driver-version>_<NNN>_<slug>/     # e.g. 0.10.0_001_delta_blocks/
│   ├── __init__.py                         # exposes `step: MigrationStep` instance
│   ├── migration.py                        # apply/preview impls + vendored helpers inline
│   ├── fixtures/
│   │   ├── before/                         # representative pre-state markdown files
│   │   └── after/                          # expected post-state
│   └── test_migration.py                   # snapshot tests vs fixtures
└── (DE-137 ships zero migrations; first land in DE-138..142)
```

Folder name parsed by orchestrator into `(semver, ordinal, slug)`. Sorting via `packaging.version.Version` (already a transitive dep). Each migration vendors its own ~15-line `_split_frontmatter` helper, calls `yaml.safe_load`/`safe_dump` directly, writes atomically via `Path.write_text` to a sibling tempfile + `Path.replace`.

#### `admin migrate` orchestrator

`spec_driver/presentation/cli/admin/migrate.py`:

```
spec-driver admin migrate --check                 # preview pending migrations (no writes)
spec-driver admin migrate --list                  # show pending + applied migrations
spec-driver admin migrate <kind>                  # sweep one kind, interactive
spec-driver admin migrate <kind> --dry-run        # preview the diff, no writes
spec-driver admin migrate <kind> --strict         # flip strict mode after sweep passes
spec-driver admin migrate all                     # walk all kinds
```

Behaviour:
- Walks `spec_driver/migrations/` directory; sorts by semver+ordinal.
- For each step after `[migrations] last_applied`:
  1. Calls `step.preview(path)` for each candidate file (dry-run hint).
  2. Calls `step.apply(path)`.
  3. Re-runs the **current** `MetadataValidator` on each touched file (in tolerant mode, since other kinds may be unmigrated).
  4. Writes per-run log to `.spec-driver/run/migrations/<timestamp>-<step>.md`.
  5. Updates `[migrations] last_applied` atomically (workflow.toml tempfile + rename).
- On error mid-walk: log preserves diagnostic; `last_applied` is updated only for fully-completed steps.
- `<kind>` filter: only invokes steps whose `applies_to_kinds` tuple contains `<kind>`; skips others.
- `--strict` flag flips `[validation.strict] <kind> = true` after the kind's sweep completes cleanly.

Idempotency: each step's `apply()` returns no-op on already-applied files (per-step responsibility; VT-CC-019).

Atomicity: per-file (temp + rename), and `last_applied` watermark advances only after the step's full pass completes (VT-CC-020).

#### `workflow.toml` schema additions

In `supekku/scripts/lib/core/config.py` `DEFAULT_CONFIG`:

```python
"migrations": {
  "last_applied": None,          # None ⇒ pre-first-migration
},
"validation": {
  "strict": {
    # Per-kind strict-mode toggle. None per kind ⇒ default (tolerant for upgrade,
    # strict-on for fresh install — install resolves the default at workflow.toml

  },
},
```

Upgrading install: keys absent ⇒ orchestrator treats as fully tolerant, pre-first-migration. Consumer opts in on their schedule via `admin migrate <kind> --strict` (DEC-137-13).

Default resolution logic in `install.py`:
- `workflow.toml` does not exist ⇒ `_generate_default_workflow_toml(strict_on=True)`.
- `workflow.toml` exists ⇒ preserve existing keys; do not auto-insert strict toggles.

## 7. Design Decisions & Trade-offs

See frontmatter `design_decisions` for the canonical list. Decisions DEC-137-01 through DEC-137-14 each carry user-accepted rationale captured in the DR-137 drafting transcript (Q1-Q10).

Sub-decisions worth elaborating:

- **DEC-137-08 (custom yaml_emit vs ruamel.yaml)**: chose custom because (a) one-way emit only — no round-trip parsing — and (b) zero new direct deps. Tracked as OQ-137-01: revisit if implementation exceeds ~120 LOC or hits stdlib-yaml edge cases.
- **DEC-137-09 (TEMPLATE_PLACEHOLDERS placement)**: orchestration layer, not on `BlockMetadata`. Reason: template-shaping is presentation concern; metadata stays as pure data shape. Minor DR-136 §5.1 wording correction recorded in §10.

- **DEC-137-12 (single watermark, not per-kind integer)**: simpler state shape; per-kind status remains accessible via the orchestrator's `applies_to_kinds` walk; per-kind strict-mode toggle is the user-facing per-kind concern (kept orthogonal).

- Upgrading install ⇒ all keys absent; tolerant-on-read preserved; `admin migrate --check` surfaces hint.

### 9.2 Observability / alerts

Per-run migration log: `.spec-driver/run/migrations/<timestamp>-<step>.md`. Contains:
- Step ID (folder name).
- Files touched / skipped / drift entries.
- `MetadataValidator` results before vs after.
- Total duration.

`admin migrate --list` surfaces all log files chronologically with summary headers.

### 9.3 Recovery / rollback

- Per-step atomicity: `last_applied` watermark updates only on full completion. Mid-step abort ⇒ partial file changes (idempotency means re-running is safe).
- `[validation.strict] <kind>` per-kind toggle is the rollback for strict-mode regret: flip to `false` to restore tolerance for that kind. Migration step state is not affected.
- No "downgrade" migration semantics. Migrations are forward-only (capture-of-the-day discipline).

### 9.4 Performance

- `validate workspace`: existing baseline (8 audit-gate warnings + 2 install-skew lines per DE-118 baseline). `--kind` filter reduces working set proportionally. No new full-corpus passes added.
- `validate file <path>`: O(1) — single file load + frontmatter + block validation.
- `validate templates`: O(kinds) — 7-9 templates, sub-second.
- `admin migrate <kind>`: O(files-of-kind) × O(steps); per-step idempotency check is a frontmatter read.
- `schema enums`: O(1) — registry walk, no file IO.

## 10. Supersedes (DR-136 reconciliations)

DR-137 supersedes specific DR-136 wording. Each reconciliation noted here and surfaces at audit/closure for explicit acceptance:

1. **DR-136 §11.3** — `[schema_version]` per-kind integer table ⇒ `[migrations] last_applied` watermark + per-kind `[validation.strict]` toggle. `[schema_version]` becomes an optional derived cache, not authoritative state.
2. **DR-136 §11.4** — `supekku/scripts/lib/migrations/` ⇒ `spec_driver/migrations/` as top-level peer of `orchestration/`. `MigrationStep` base class with `from_version`/`to_version` ints ⇒ Protocol; folder name encodes version+ordinal.
3. **DR-136 §11.1 boundary table** — "Migration steps may import from core/ and blocks/" line ⇒ migration steps may import **only** `spec_driver/migrations/_protocol`. They vendor splitters/emitters per-step. Stronger isolation; enforced by import-linter.
4. **DR-136 §5.2** — `core/string_utils.py` ⇒ `spec_driver/core/string_utils.py`. Levenshtein top-1 (distance 3) ⇒ stdlib `difflib.get_close_matches(n=1, cutoff=0.6)`. Behaviour difference negligible for the practical cases ("status 'live' → in-progress").
5. **DR-136 §5.1** — `BlockMetadata.render_template_frontmatter()` ⇒ `spec_driver/orchestration/templates.py:render_template_frontmatter(kind)`. POL-003: presentation/orchestration logic doesn't live on core-layer data classes.
6. **DR-136 §5.4a** — `validate --kind` flag form ⇒ `validate workspace --kind` (under the new Typer group). `validate` (bare) dispatches to `workspace`; `validate file` and `validate templates` are named peers.
7. **DR-136 §5.6** — `admin migrate <kind>` orchestrator now keyed by semver+ordinal folder names, not per-kind from_version/to_version. Single `last_applied` watermark; per-kind step dispatch via `step.applies_to_kinds`.
8. **DR-136 §11.1** — `aliases` vs `tolerated_aliases` distinction: DR-137 elevates `tolerated_aliases` from prose mention to a typed `ToleratedAlias` dataclass with `canonical`/`sunset_after`/`rationale`. Cleaner serialisation; explicit sunset reference.
