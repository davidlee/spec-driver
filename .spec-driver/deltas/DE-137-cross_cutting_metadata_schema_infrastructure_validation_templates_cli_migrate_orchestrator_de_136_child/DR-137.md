# DR-137 – Cross-cutting metadata schema infrastructure: validation+templates+CLI+migrate orchestrator (DE-136 child)

## 1. Executive Summary

- **Delta**: [DE-137](./DE-137.md) — cross-cutting metadata schema infrastructure (DE-136 program child).
- **Canonical design reference**: DR-136 §5 (cross-cutting deliverables), DR-136 §11 (migration mechanics). DR-137 refines and locks the residual implementation decisions and supersedes specific DR-136 wording where noted in §10 (Supersedes).
- **Status**: draft.
- **Synopsis**: Land the mechanism layer of the DE-136 program — strict-mode + alias-aware `MetadataValidator`, metadata-as-source-of-truth for enum values + aliases, comment-preserving template-regenerator with CI gate, `validate` Typer group (`workspace`/`file`/`templates`), `schema enums` CLI, `admin migrate` orchestrator with self-contained semver-linearised migrations, skill-level validation gates, and `workflow.toml` schema additions. Zero artefact data is touched in this delta; per-artefact sweeps land in DE-138..142.

## 2. Problem & Constraints

### 2.1 Current behaviour (verified against codebase)

- `MetadataValidator` (`blocks/metadata/validator.py:60`) supports `strict_unknown_keys` but no broader strict/alias mode; loaders cannot uniformly request strict validation.
- `normalize_status()` (`changes/lifecycle.py:43`) is a single-purpose global helper covering change-status only; no per-field alias surface; no did-you-mean diagnostics.
- `ENUM_REGISTRY` (`spec_driver/orchestration/enums.py`) and `FieldMetadata.enum_values` are two competing sources of enum truth (POL-001 friction).
- `validate` is a single workspace-wide Typer command; no `file` or `templates` subcommand; no `--kind` filter.
- Templates are a mix of body-only-with-Jinja (delta/spec/revision/design_revision/phase) and frontmatter-bearing example (audit only); no metadata-driven regeneration; no CI drift gate.
- No migration framework exists; schema changes either break old artefacts silently or require ad-hoc one-off sweeps.
- `workflow.toml` has no `[migrations]` or `[validation.strict]` sections.
- Five skills (`execute-phase`, `close-change`, `audit-change`, `notes`, `update-delta-docs`) lack validation gates at named lifecycle moments.

### 2.2 Drivers

- DE-136 program (DR-136 §5/§11 — canonical).
- ADR-010 placement heuristic (frontmatter / blocks / prose; never duplicate).
- POL-001 (reuse), POL-002 (no magic strings/numbers), POL-003 (one-way layering).
- STD-001 (Typer/Rich CLI), STD-003 (utility module placement).
- ISSUE-054 residual scope (authoring-time YAML feedback) — re-verified at delta start via VT-CC-026 (F-10 converts v1's VA into a regression test).

### 2.3 Constraints / guardrails

- DE-118 must be closed (verified at DR-137 sign-off; `MetadataValidator.strict_unknown_keys` flag confirmed present).
- POL-003 layering: new modules land in `spec_driver.<layer>` (core/orchestration/presentation) or `spec_driver/migrations/` (new peer); legacy `supekku/scripts/lib/` extensions are additive on existing modules only.
- POL-001 single source of truth: metadata is canonical for enum_values + aliases; lifecycle constants become transition-window re-exports.
- **POL-003 boundary on migrations**: migration step modules import **nothing** from `spec_driver.*` or `supekku.scripts.lib.*` except `spec_driver.migrations.{_protocol,_helpers,_folder}` (frozen forever). Stdlib + pinned pyyaml are allowed. Enforced by an import-linter contract entry (VT-CC-021).
- Tolerant-on-read for unmigrated kinds is preserved until per-kind strict-mode is flipped in `workflow.toml`.
- Acceptance honours PROD-004 FR-005/FR-006 (gradual migration; consumers may stay tolerant indefinitely).

### 2.4 Out of scope

- Per-artefact placement work (DR-136 §§6–10) — sibling child deltas DE-138..142.
- Migration **step** implementations — owned by per-artefact deltas. DE-137 ships zero migrations.
- `validate workspace --prune-duplicates` interactive narrative cleanup — deferred (DEC-137-10).
- Legacy-to-layered extraction of existing subsystems (`blocks/`, `validation/`, `core/frontmatter_metadata/`) — separate concern; DE-137 extends in place.
- `validate templates` runtime UX beyond CI gate semantics — minimal surface, not a full diff tool.
- Relations-shape unification (DR-136 DEC-005) — post-umbrella cleanup delta.

## 3. Architecture Intent

### 3.1 Target outcomes

1. **Single validation API surface**: `MetadataValidator.validate(data, *, strict, accept_tolerated)` covers strict-unknown-keys, strict-enum, strict-alias modes in one call. Loaders pass `strict=False` (tolerant; aliases silently normalised); `validate workspace/file --strict` passes `strict=True` (aliases become fixable warnings).
2. **Metadata as single source of truth** for enum_values, aliases, tolerated_aliases. `ENUM_REGISTRY` is a derived view; lifecycle constants become transition-window re-exports; `normalize_status()` retired in favour of `normalize_field(kind, field_name, value)`.
3. **Template regenerator + CI gate**: `spec-driver admin regenerate-templates` and `spec-driver validate templates` keep `supekku/templates/*.md` frontmatter aligned with metadata; created artefacts also carry inline enum-comment hints via a comment-preserving emitter at dump time. Two emit paths kept aligned by the CI gate.
4. **`validate` Typer group** with `workspace` / `file` / `templates` named peers; bare `spec-driver validate` prints help (no default dispatch — DEC-137-17).
5. **`spec-driver schema enums [<kind>[.<field>]]`** CLI surfaces enum_values + aliases + tolerated_aliases from metadata.
6. **`spec-driver admin migrate`** orchestrator discovers self-contained migrations in `spec_driver/migrations/v<M>_<m>_<p>_<NNN>_<slug>/`, applies them in semver order, records `[migrations] last_applied` watermark in `workflow.toml`, re-validates via the **current** `MetadataValidator` after each step.
7. **Skill gates**: five named skills gain explicit `validate workspace`/`validate file` insertion points per DR-136 §5.5 (text shipped verbatim).
8. **`workflow.toml` schema extensions**: `[migrations] last_applied`, `[validation.strict]` per-kind toggle, optional `[schema_version]` derived cache.

### 3.2 Guiding principles

- **Strict capture-of-the-day for migrations**: each migration is a frozen, self-contained script. Migration steps work at the bytes-and-pyyaml level — never importing schema, validation, or registry code. Sole shared files are `spec_driver/migrations/{_protocol,_helpers,_folder}.py` (frozen-forever, narrow surface). This is the framework's load-bearing invariant (DEC-137-11). Rationale: a migration targets a *historical* schema version — pulling current-schema knowledge into the step makes it brittle to evolution of the current code.
- **Mechanism vs sweep**: DE-137 ships the framework. Per-artefact deltas (DE-138..142) author the migrations and exercise them. DE-137 acceptance does NOT include applying any migration to this repo.
- **Tolerant on read; strict on validate/CI**: per kind, per opt-in `[validation.strict]`. Asymmetric per DR-136 §5.2 Q5.
- **Single source of truth wherever possible**: metadata for enum_values + aliases; `last_applied` watermark for migration state; templates for reference frontmatter (regenerated from metadata).
- **POL-003 boundary discipline**: enforced at architectural test layer (import-linter contract); new code lands in `spec_driver.<layer>`; legacy extensions are additive only.

def emit_yaml_block(
  data: Mapping[str, Any],
  comments: Mapping[str, str] | None = None,

### 5.2 Deliverable Z — Strict validation, alias autocorrect, did-you-mean

#### Schema extensions — two distinct alias mechanisms (F-30)

The adversarial review flagged that "alias" covers two structurally different concerns that v2 conflated:

- **Field-name aliasing** (F-B): the YAML key `annotation` should be treated as the field named `nature`. This is a key-rename concern that lives on the *container* (the block schema) — not on any one FieldMetadata. The validator applies it at parse-time, before per-field dispatch.
- **Field-value aliasing** (status normalisation): the value `complete` for a delta's `status` field should be treated as `completed`. This is a value-transformation concern that lives on the FieldMetadata for the field whose values it normalises. The validator applies it after per-field dispatch.

DR-137 v3 separates them:

```python
# spec_driver.blocks (or supekku/scripts/lib/blocks/metadata/schema.py) — additive

@dataclass
class BlockMetadata:
  # ... existing fields ...

  field_aliases: Mapping[str, str] | None = None
  """Field-NAME alias map: {alias_key: canonical_key}.
  Applied at parse time: if the parsed YAML contains `alias_key`, the
  validator renames it to `canonical_key` before per-field dispatch.
  Example (on the relations item BlockMetadata):
    {"annotation": "nature"}
  Loaders apply silently at read time when strict=False.
  Under strict, the rename emits a warning whose --fix rewrites the key
  in the source file."""

@dataclass
class FieldMetadata:
  # ... existing fields ...

  aliases: Mapping[str, str] | None = None
  """Field-VALUE alias map: {alias_value: canonical_value}.
  Applied to THIS field's value after per-field dispatch.
  Example (on the delta status FieldMetadata):
    {"complete": "completed"}
  Same strict/--fix semantics as field_aliases, scoped to value rewrites."""

  tolerated_aliases: Mapping[str, ToleratedAlias] | None = None
  """Time-bounded field-VALUE alias: {alias_value: ToleratedAlias}.
  Migration-window only.
  `validate --no-tolerated-aliases` rejects these; `validate --fix` does NOT
  rewrite them (migration concern; rides through admin migrate)."""

@dataclass(frozen=True)
class ToleratedAlias:
  canonical: str
  sunset_after: str           # delta ID or semver (e.g. "DE-140" or "0.11.0")
  rationale: str
```

All three fields default `None`; opt-in per schema.

Validator pseudo-code:

```python
def validate(self, data, *, strict, accept_tolerated):
  errors = []
  # Pass 1: field-name aliasing (BlockMetadata.field_aliases)
  for alias_key, canonical_key in (self.metadata.field_aliases or {}).items():
    if alias_key in data:
      data[canonical_key] = data.pop(alias_key)
      if strict:
        errors.append(ValidationError(
          path=f"{alias_key}",
          severity="warning",
          message=f"field name {alias_key!r} is an alias for {canonical_key!r}",
          fix_hint=canonical_key,
          fix_kind="rename_key",
        ))
  # Pass 2: per-field validation, including FieldMetadata.aliases / tolerated_aliases
  for field_name, field_meta in self.metadata.fields.items():
    ...
  return errors
```

`ValidationError.fix_kind` ∈ {`rewrite_value`, `rename_key`} disambiguates how `--fix` applies the canonical replacement.

#### Did-you-mean utility (DEC-137-20)

New `spec_driver/core/string_utils.py`:

```python
def closest_match(value: str, candidates: Iterable[str]) -> str | None:
  """Closest match to *value* from *candidates*, or None.

  Stdlib difflib.get_close_matches(n=1, cutoff=0.6). Empty/single-char
  inputs return None.
  """
```

~5 LOC. Used by `MetadataValidator` for enum violations; reusable for any future enum diagnostic surface.

**Did-you-mean is for typos; aliases are for semantic alternatives — they are orthogonal.** Spike (recorded against F-15 disposition) verified:

- `cutoff=0.6` reliably catches typos: `'complete'→'completed'@0.94`, `'pendng'→'pending'@0.92`, `'in_progres'→'in-progress'@0.86`, `'defered'→'deferred'@0.93`.
- Semantic alternatives (`'live'`, `'active'`, `'done'`, `'wip'` for status `'in-progress'`) share too few characters with their target — no difflib cutoff catches them without sweeping in unrelated false positives (`'asdf'→'draft'`, `'open'→'pending'`). These belong in `FieldMetadata.aliases` (permanent) and get silent normalisation at read time, not did-you-mean.
- VT-CC-010 enumerates the canonical typos and asserts each surfaces the expected match; VT-CC-008 covers the alias path independently.

#### `validate workspace --fix` behaviour

Existing `--fix` auto-repairs non-canonical phase statuses (`validator.py:548`). Extended to apply any alias canonicalisation surfaced via `ValidationError.fix_hint`. Per-kind, per-field, idempotent. Atomic per-file write (temp + rename).

- Permanent alias mismatch ⇒ `--fix` rewrites file.
- Tolerated alias ⇒ `--fix` leaves file untouched (migration concern; rides through `admin migrate`).
- Unknown key ⇒ never auto-fixed (no canonical target).
- Enum violation with did-you-mean ⇒ never auto-fixed (high false-positive risk; explicit hint only).

#### `--kind <kind>` semantics (F-8)

`validate workspace --kind <kind>` defines a precise sweep procedure for the per-artefact migration pattern (DR-136 §11.2):

1. **Loading phase**: only artefacts whose frontmatter `kind: <kind>` matches are *loaded with full per-kind frontmatter validation*. Non-matching kinds are skipped (not loaded). This is the load-time filter.
2. **Relation-traversal phase**: the full corpus is still loaded *minimally* (registry-level only, no frontmatter validation) because relation graphs cross kinds. Warnings emitted during traversal are filtered to those whose subject artefact matches `<kind>`.
3. **Exit**: non-zero iff any error/warning targeting `<kind>` artefacts surfaced.

This makes the sweep procedure reproducible: post-migration, `validate workspace --kind delta --strict` should report zero warnings even while other kinds remain unmigrated (VT-CC-025 covers).

Composes with `--strict`, `--no-tolerated-aliases`, `--fix`.

**Two-layer dispatch (F-33)**. The Protocol exposes both `applies_to_kind: str` (class attribute) and `applies_to(file_path) -> bool` (method). The orchestrator uses them in sequence:

1. **Kind filter (cheap)**: orchestrator reads `<file>`'s frontmatter `kind:` field; if it doesn't equal `step.applies_to_kind`, skip — `applies_to(...)` is never called.
2. **File-level predicate (per-step)**: orchestrator calls `step.applies_to(file_path)` to ask "is this *specific* file in the target state already, or does it need work?" — the canonical use is `return not <has the new block>`.

The default `_protocol.MigrationStep` implementation (or a base class) provides a `applies_to` that returns `True` (every file of the right kind needs work). Step authors override only when they need finer file-level discrimination (e.g. "only files lacking the new `relations:` block — the migration already ran on this file in a prior pass that crashed midway").

Step authors who override `applies_to` MUST NOT re-check the kind there — the orchestrator already filtered by kind. Mixing the two would be a bug. The default base class in `_protocol.py` makes the right thing easy.

~30 LOC. Treat any change here as a framework v2 break.

**Multi-kind steps are forbidden (DEC-137-16, F-4)**. Each migration step targets exactly one artefact kind. Reasons:

- Single watermark + multi-kind step is ill-defined under `--kind` filter (does the watermark advance on a partial pass?).
- Cross-kind atomicity is harder to reason about and harder to roll forward.
- Cross-kind concerns naturally factor into sequential per-kind steps (e.g. "introduce relations block on delta" then "introduce relations block on spec" — two steps, two ordinals).
- The per-kind strict-flip story (`admin migrate <kind> --strict`) becomes coherent.

This is a stronger constraint than DR-136 §11.4 (which used `applies_to_kinds: tuple[str, ...]`); recorded in §10.

**Kind validation at discovery (F-26)**. The orchestrator validates each loaded step's `applies_to_kind` against the set of registered kinds at startup. Mistyped values (`'speec'`, `'deltas'`) fail-fast with a clear error:

```
admin migrate: migration step v0_10_0_001_delta_blocks declares
applies_to_kind='speec' which is not a registered kind.
Registered kinds: adr, audit, delta, design_revision, issue, memory,
phase, plan, policy, problem, prod, requirement, revision, risk, spec,
standard, task, verification.
```

Validation runs before any step is dispatched, even under `--check`. VT-CC-031 covers.

**Plan/phase/task shared-schema dispatch (F-27)**. `FRONTMATTER_METADATA_REGISTRY` aliases three kinds (`plan`, `phase`, `task`) onto one shared `PLAN_FRONTMATTER_METADATA`. `applies_to_kind` matches by the literal frontmatter `kind:` value of the artefact, not by schema-equivalence class. A migration that conceptually affects all three:

- Lives as three sibling folders (`v0_10_0_001_x_plan/`, `v0_10_0_002_x_phase/`, `v0_10_0_003_x_task/`) with identical `migration.py` payload (or imported helper at the per-step level) and distinct `applies_to_kind`. Folder ordinals capture the intended sequence.
- OR: lives as one folder with `applies_to_kind = "plan"` and an explicit `applies_to(file_path)` check that *also* matches `phase`/`task` files — *but this is forbidden* by DEC-137-16 (singular kind). The discipline says: three folders.

The cost (three folders instead of one) is acceptable because (a) the three kinds are semantically distinct lifecycle objects despite shared schema, (b) being able to migrate plan files without sweeping phase files is a real operator concern (you might want to gate strict-mode on plan before phase), (c) repetition is mechanical and the shared-payload helper can live inline.

#### Shared `_helpers.py` (F-20)

`spec_driver/migrations/_helpers.py` is the **sole exception** to the per-step vendoring rule. It ships ~3 frozen-forever bytes-level helpers that every migration would otherwise re-copy verbatim:

  """Split markdown text into (yaml_block, body). Returns ('', text) when
  no leading `---` delimiter pair exists. Bytes-only; does NOT parse YAML."""

  """Write *content* to *path* via sibling tempfile + os.replace. Preserves

**Reuse rationale (clarification — F-40)**: `_helpers.py` is a deliberately minimal bytes-only API the migration subsystem needs to avoid schema-version coupling. Naming it "POL-001 exemption" was misleading — POL-001 favours reuse, and `_helpers.py` *enables* reuse of bytes-level utilities. The actual discipline is schema-version stability: nothing in `_helpers.py` may depend on or expose any schema concept, and the surface stays narrow so future migrations don't accidentally gain schema-aware helpers. **Frozen-forever ≠ bug-frozen (F-35)**. The "frozen forever" discipline applies to *behaviour and signature*, not to fixing implementation bugs. If `atomic_write` is found to mis-handle Windows file modes, that's a bug fix — required, allowed, and lands like any other fix (with VT coverage). What is forbidden:

- Changing the function *signature* (parameter names, types, return type) — that would break past migration steps snapshotted against the old API.
- Adding new public functions — those should go inline in the step that needs them.
- Adding schema awareness — any inbound dependency on current-schema concepts.

If a behavioural change is genuinely required (e.g. a deprecated stdlib path was removed), the change goes through `/scope-delta` like any other behavioural change and explicitly enumerates the migration steps that need re-verifying against fixtures. The frozen-forever invariant raises the bar for those changes, it doesn't prohibit them.

Behaviour:
- Acquires `.spec-driver/run/migrations/.lock` (PID file). On detected stale/fresh lock, exit non-zero with: `migrate: concurrent invocation detected (lock held by PID <n>); aborting`. Lock released on exit (success or failure). DEC-137-22.
- Walks `spec_driver/migrations/` directory; non-parseable folder names ignored; sorts by `(version, ordinal)`.
- For each step after `[migrations] last_applied`:
  1. Calls `step.preview(path)` for each candidate file (dry-run hint).
  2. Calls `step.apply(path)`.
  3. Re-runs the **current** `MetadataValidator` on each touched file (in tolerant mode, since other kinds may be unmigrated).
  4. Writes per-run log to `.spec-driver/run/migrations/<timestamp>-<step>.md`.
  5. Updates `[migrations] last_applied` atomically (workflow.toml tempfile + rename) **only after** the step finishes its full pass without raising.
- On error mid-walk: `last_applied` is NOT advanced for the failed step. Per-step idempotency (VT-CC-019) means re-running the migrate command after fixing the cause walks the same step again and resumes cleanly. VT-CC-023 covers: fixture step raises mid-corpus on file N of M; assert (a) files 1..N-1 are in the post-state, (b) file N is unchanged or fully rolled back (atomic temp+rename invariant), (c) files N+1..M are unchanged, (d) re-running the same step (with the raise removed) reaches all M files and final state matches "fully applied".
- `<kind>` filter: only invokes steps whose `applies_to_kind` equals `<kind>`; skips others.
- `--strict` flag flips `[validation.strict] <kind> = true` after the kind's sweep completes cleanly.

Idempotency: each step's `apply()` returns no-op on already-applied files (per-step responsibility; VT-CC-019).

Atomicity: per-file (temp + rename), and `last_applied` watermark advances only after the step's full pass completes (VT-CC-020).

## 7. Design Decisions & Trade-offs

See frontmatter `design_decisions` for the canonical list. Decisions DEC-137-01 through DEC-137-22 each carry user-accepted rationale captured in the DR-137 drafting transcript (Q1-Q10) plus adversarial-review revisions (F-1..F-23 dispositions).

Sub-decisions worth elaborating:

- **DEC-137-08 (custom yaml_emit vs ruamel.yaml)**: chose custom because (a) one-way emit only — no round-trip parsing — and (b) zero new direct deps. Tracked as OQ-137-01: revisit if implementation exceeds ~120 LOC or hits stdlib-yaml edge cases.
- **DEC-137-09 (TEMPLATE_PLACEHOLDERS placement)**: orchestration layer, not on `BlockMetadata`. Reason: template-shaping is presentation concern; metadata stays as pure data shape. Minor DR-136 §5.1 wording correction recorded in §10.
- **DEC-137-11 (migration isolation)**: stronger than DR-136 §11.1's "may import core + blocks" line. Migrations import only `_protocol`, `_helpers`, `_folder`, stdlib, and pinned pyyaml. **Rationale**: a migration step targets a *historical* schema version. Pulling current-schema knowledge — directly or transitively through helpers — makes the step fragile against subsequent evolution of the current code. The frozen `_helpers.py` is acceptable because (a) the surface is tiny and bytes-level (split/atomic-write/replace-scalar), (b) it ships *with* the protocol and is treated as part of the protocol's stable contract, (c) any pressure to grow it would itself be a signal that "capture-of-the-day" is being eroded. Pyyaml is third-party (not stdlib) and gets explicit lock-in via a version pin in pyproject.toml — accepted because hand-rolling a YAML loader is the worse trade.
- **DEC-137-12 (single watermark, not per-kind integer)**: simpler state shape; per-kind status is implicit via the orchestrator's `applies_to_kind` walk; per-kind strict-mode toggle is the user-facing per-kind concern (kept orthogonal).
- **DEC-137-16 (singular-kind steps)**: forbidding multi-kind steps loses theoretical flexibility but gains operational clarity. The single-watermark-plus-`--kind`-filter combination is incoherent under multi-kind steps (does the watermark advance on a partial sweep?). Migrations that conceptually span kinds split into ordered per-kind steps; their ordinals capture the dependency.
- **DEC-137-19 (sole shared `_helpers.py`)**: explicit POL-001 trade-off. POL-001 says "extract on third copy." Here the extraction happens *before* the third copy, in a single frozen-forever module. Acceptable because the alternative — each migration step re-implementing `split_frontmatter` / `atomic_write` — is worse for both readability and reliability of bytes-level invariants we can't afford to get wrong (e.g. atomic rename semantics). Trade-off is bounded by the frozen-forever discipline: `_helpers.py` does NOT grow over time; new helpers go inline in the migration step that needs them, even if a second step would benefit.
- **DEC-137-21 (drop :line:col for semantic errors)**: F-7 disposition. `ValidationError` carries no positional info for post-parse semantic failures, so claiming `path:line:col:` is dishonest. Parser-mark errors retain line/col because they actually carry it. **Trade-off acknowledged (F-38)**: editor quickfix linkifiers and CI grep regexes (`<path>:\d+:\d+:`) won't catch semantic diagnostics. We mitigate as follows:
  - The dotted field path lets a user navigate quickly inside a frontmatter block, and the editor's "go to symbol" finds the field by name.
  - For CI integration: a separate machine-readable output mode (out of DE-137 scope; flagged in OQ-137-03 below) can produce JSON diagnostics for CI parsers that need precise location info.
  - We considered emitting `:1:1:` placeholder line/col for semantic errors to retain linkifier behaviour — rejected as dishonest in the other direction (every diagnostic links to line 1, swamping the gutter). Leaving the choice to a future JSON output mode.

## 9. Rollout & Operational Notes

### 9.1 Migration / backfill

DE-137 itself does **not** migrate any artefact data; first migrations land in DE-138..142.

**One-time template regeneration (F-42)**. Today's templates are body-only Jinja (no leading `---` frontmatter). The first commit that lands DE-137's template regenerator infrastructure MUST also run `spec-driver admin regenerate-templates` and commit the resulting templates with their new frontmatter blocks — otherwise the `just validate-templates` CI gate detects drift on every subsequent PR until somebody does this. IP-137 phase planning makes this a named task.

Phase-2 entrance check (per IP-136 phase-02): re-verify DE-118 closed and `MetadataValidator` strict_unknown_keys flag remained opt-in (not unconditional). Confirmed satisfied at session start — `supekku/scripts/lib/blocks/metadata/validator.py:60` carries `strict_unknown_keys: bool = False` (DE-136 notes.md F-E entry).

Consumer rollout (DEC-137-18 trigger):
- **Fresh install** (no `.spec-driver/` present) post-DE-137 ⇒ strict-on for all per-kind frontmatter; `admin migrate` watermark set to current shipping version's marker; install message printed.
- **Upgrading install** (`.spec-driver/` exists) ⇒ keys absent ⇒ tolerant-on-read preserved; `admin migrate --check` surfaces hint about pending migrations.

### 9.2 Observability / alerts

Per-run migration log: `.spec-driver/run/migrations/<timestamp>-<step>.md`. Contains:
- Step ID (folder name).
- Files touched / skipped / drift entries.
- `MetadataValidator` results before vs after.
- Total duration.

`admin migrate --list` surfaces all log files chronologically with summary headers.

### 9.3 Recovery / rollback

- Per-step atomicity: `last_applied` watermark updates only on full completion. Mid-step abort ⇒ partial file changes; idempotency (VT-CC-019) + atomic per-file write (temp+rename) means re-running is safe. VT-CC-023 covers the mid-walk-error recovery path.
- `[validation.strict] <kind>` per-kind toggle is the rollback for strict-mode regret: flip to `false` to restore tolerance for that kind. Migration step state is not affected.
- No "downgrade" migration semantics. Migrations are forward-only (capture-of-the-day discipline).
- **Single-process invariant (DEC-137-22, F-21 / F-36)**: `admin migrate` acquires `.spec-driver/run/migrations/.lock` at entry; lock content is `<pid>\n<iso-timestamp>\n<short-uuid>\n` (the uuid disambiguates against PID reuse). On detected lock:
  - **POSIX** (Linux, Darwin): orchestrator probes liveness via `os.kill(pid, 0)`; raises `ProcessLookupError` if dead. Distinguishes "lock held by live PID" from "stale lock (PID dead)" in the error message.
  - **Windows** (best-effort): liveness probing is unreliable across PID reuse; the orchestrator reports the lock as held without claiming staleness — operator must inspect and clear manually.
  - In either case, **no automatic stale-lock takeover**: the orchestrator exits with a clear error mentioning held-PID + start-timestamp + uuid; the operator confirms staleness and deletes `.lock` to unblock.
  - Lock released on normal exit AND on unhandled-exception path (via `atexit` registration). Crash without atexit cleanup (SIGKILL) leaves the lock; the staleness check then catches it on next invocation.

### 9.4 Performance

- `validate workspace`: existing baseline (8 audit-gate warnings + 2 install-skew lines per DE-118 baseline). `--kind` filter reduces working set proportionally. No new full-corpus passes added.
- `validate file <path>`: O(1) — single file load + frontmatter + block validation.
- `validate templates`: O(kinds) — 7-9 templates, sub-second.
- `admin migrate <kind>`: O(files-of-kind) × O(steps); per-step idempotency check is a frontmatter read.
- `schema enums`: O(1) — registry walk, no file IO.

## 10. Supersedes (DR-136 reconciliations)

DR-137 supersedes specific DR-136 wording. Each reconciliation noted here and surfaces at audit/closure for explicit acceptance:

1. **DR-136 §11.3** — `[schema_version]` per-kind integer table ⇒ `[migrations] last_applied` watermark + per-kind `[validation.strict]` toggle. `[schema_version]` becomes an optional derived cache, not authoritative state. Strict-on-default trigger keyed to `.spec-driver/` workspace absence, not `workflow.toml` presence (DEC-137-18; F-5).
2. **DR-136 §11.4** — `supekku/scripts/lib/migrations/` ⇒ `spec_driver/migrations/` as top-level peer of `orchestration/`. `MigrationStep` base class with `from_version`/`to_version` ints ⇒ Protocol; folder name encodes version+ordinal as `v<M>_<m>_<p>_<NNN>_<slug>/` (DEC-137-13; F-13).
3. **DR-136 §11.1 boundary table** — "Migration steps may import from core/ and blocks/" line ⇒ migration steps may import **only** `spec_driver.migrations.{_protocol,_helpers,_folder}`, stdlib, and pinned pyyaml (third-party, acknowledged). They vendor anything else inline. Stronger isolation; enforced by import-linter (F-2 verbatim pyproject diff in §5.6; VT-CC-021).
4. **DR-136 §5.2** — `core/string_utils.py` ⇒ `spec_driver/core/string_utils.py`. Levenshtein top-1 (distance 3) ⇒ stdlib `difflib.get_close_matches(n=1, cutoff=0.6)`. Spike confirmed cutoff=0.6 catches typos and that semantic alternatives belong in aliases, not did-you-mean (DEC-137-20; F-15).
5. **DR-136 §5.1** — `BlockMetadata.render_template_frontmatter()` ⇒ `spec_driver/orchestration/templates.py:render_frontmatter_for_kind(kind, data=None)`. POL-003: presentation/orchestration logic doesn't live on core-layer data classes. Same function powers both regenerator and creation-time emit (POL-001; DEC-137-15 / F-6).
6. **DR-136 §5.4a** — `validate --kind` flag form ⇒ `validate workspace --kind` (under the new Typer group). `validate file`, `validate workspace`, `validate templates` are named peers; bare `spec-driver validate` prints help (DEC-137-17; F-11).
7. **DR-136 §5.6** — `admin migrate <kind>` orchestrator now keyed by semver+ordinal folder names, not per-kind from_version/to_version. Single `last_applied` watermark; per-kind step dispatch via `step.applies_to_kind` (singular — DEC-137-16; F-4). Multi-kind steps forbidden.
8. **DR-136 §11.1** — `aliases` vs `tolerated_aliases` distinction: DR-137 elevates `tolerated_aliases` from prose mention to a typed `ToleratedAlias` dataclass with `canonical`/`sunset_after`/`rationale`. Cleaner serialisation; explicit sunset reference.
9. **DR-136 §5.3** (new — F-1) — `dump_markdown_file(...)` with optional `kind=` ⇒ split into `dump_markdown_file_create(..., kind=)` (emit comments) and `dump_markdown_file_update(...)` (preserve existing comments). Every caller migrates explicitly. Avoids silent comment-stripping on edit paths (DEC-137-15).
10. **DR-136 §5.2** (new — F-17) — `ENUM_REGISTRY` split into Category A (artefact-frontmatter-derived, becomes derived view) and Category B (verification.*, backlog.*, drift.*, command.format, requirement.kind — stays hardcoded or relocates per-block). DR-136 conflated both into one migration path.
