## 1. Summary

- **Delta**: DE-137 — Cross-cutting metadata schema infrastructure (validation + templates + CLI + migrate orchestrator).
- **Specs Impacted**: PROD-004 (FR-001/-002/-003/-006), SPEC-114 (blocks/metadata), SPEC-116 (frontmatter_metadata), SPEC-125 (validation). Touches SPEC-134 (cli) surfaces; templates and skills.
- **Problems / Issues**: ISSUE-054 (authoring-time YAML feedback gap — re-verified via VT-CC-026), IMPR-024 (subsumed by DE-136).
- **Desired Outcome**: Mechanism layer of the DE-136 program lands — strict-mode + alias-aware `MetadataValidator`, metadata-as-source-of-truth for enums + aliases, comment-preserving template regenerator with CI gate, `validate` Typer group (`workspace`/`file`/`templates`), `schema enums` CLI, `admin migrate` orchestrator with self-contained semver-linearised migrations, skill-level validation gates, and `workflow.toml` schema additions. **Zero artefact data is touched** — per-artefact sweeps land in DE-138..142.

## 2. Context & Constraints

- **Current Behaviour**: `MetadataValidator` has opt-in `strict_unknown_keys` only (DE-118 baseline); `normalize_status` is change-status-only; `ENUM_REGISTRY` competes with `FieldMetadata.enum_values` (POL-001 friction); templates are body-only Jinja with no frontmatter or CI gate; no migration framework; five skills lack validation gates.
- **Target Behaviour**: Per DR-137 §3 — single validation API, metadata as single source of truth for enums + aliases, template regenerator + CI gate, `validate` Typer group, `schema enums` CLI, `admin migrate` orchestrator, skill gates, workflow.toml schema.
- **Dependencies**:
  - **Hard**: DE-118 closed (provides `MetadataValidator(strict_unknown_keys=False)` opt-in baseline — verified at `supekku/scripts/lib/blocks/metadata/validator.py:60`).
  - **Soft**: DE-036 (frontmatter compaction infra), DE-135 (read-time YAML resilience).
  - **Sibling**: DE-138..142 consume this delta's infrastructure but their DRs are NOT pre-drafted.
- **Constraints**:
  - DE-137 ships **zero migrations** — only the framework and orchestrator.
  - Tolerant-on-read preserved for unmigrated kinds (PROD-004 FR-005/-006 gradual migration commitment).
  - POL-003 boundary on migrations: steps import only `_protocol`/`_helpers`/`_folder`/stdlib/pyyaml — enforced by import-linter (DEC-137-11 / VT-CC-021).
  - All ~33 `dump_markdown_file` callers migrate explicitly to `_create(..., kind=)` or `_update(...)` (no shim — DEC-137-15 / F-1).
