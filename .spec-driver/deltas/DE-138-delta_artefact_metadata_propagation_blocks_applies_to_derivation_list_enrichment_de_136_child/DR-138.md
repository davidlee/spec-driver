# DR-138 – Delta artefact metadata propagation: blocks + applies_to derivation + list enrichment (DE-136 child)

## 1. Executive Summary

- **Delta**: [DE-138](./DE-138.md) — propagate ADR-010 placement to delta artefact kind.
- **Canonical design reference**: DR-136 §6 (delta placement table + body reshape + list enrichment + applies_to derivation + migration step naming). DR-138 refines and locks residual implementation decisions for the delta kind.
- **Status**: draft.
- **Synopsis**: First per-artefact propagation of the DE-136 program (DEC-004 sequence position 3 — after DE-118 and DE-137). `applies_to` becomes a derived load-time property sourced from `supekku:delta.relationships@v1`; `context_inputs` and `risk_register` move from FM to strict-schema blocks (`supekku:delta.context_inputs@v1`, `supekku:delta.risk_register@v1`); `outcome_summary` moves from FM to body `## Outcome`; body `## 7. Risks` deletes; `list deltas` columns enrich; `[validation.strict] delta = true` flips inside this delta with post-flip verification gate.

## 2. Problem & Constraints

### 2.1 Current behaviour

Delta artefacts duplicate structured data across three surfaces:

| Field | FM | Block | Body |
|---|---|---|---|
| `applies_to` | ✓ | partially synthesised in `artifacts.py:111-149` (block-first, FM-fallback at load) | – |
| `context_inputs` | ✓ (heterogeneous: `id`/`ref`, `summary`/`note`/`annotation`, sometimes plain strings) | – | – |
| `risk_register` | ✓ (drift: `title` vs `description`, optional `exposure`/`status`) | – | sometimes mirrored in §7 |
| `outcome_summary` | ✓ | – | sometimes mirrored in §1/§9 |
| §7 Risks (prose) | – | – | ✓ (parallel maintenance with FM `risk_register`) |

Triple-entry pattern per ADR-010 §"Conflict resolution" rule 4 (never duplicate). Cost: silent drift between FM and prose, FM bloat (PROD-004.FR-007 motivation), inconsistent block schemas across artefact kinds (PROD-004.FR-001 strict-on-validate impossible to deliver for delta kind).

### 2.2 Drivers / inputs

- DE-136 program — propagate ADR-010 across artefact kinds; DE-138 is delta-kind propagation (first per-artefact per DEC-004).
- DR-136 §6 — canonical placement table for delta kind.
- DE-118 + DE-137 closed (AUD-026 conformance evidence) — `MetadataValidator` strict-mode, alias machinery, `admin migrate` orchestrator, `validate workspace --kind`/`file`, skill validate-gates all operational.
- PROD-004.FR-001 (single validation layer), FR-002 (strict-on-validate), FR-007 (compaction infra exercised) are the umbrella requirements this delta verifies.

### 2.3 Constraints / guardrails

- ADR-010 placement is doctrinal default; deviation requires DR amendment.
- POL-001 (reuse): extend existing `BlockMetadata` / `MetadataValidator` infra; no parallel validator.
- POL-003 (module boundaries) + DR-136 §11.1: `migrations/v0_10_0_001_delta_blocks/migration.py` may not import `validation/` or `supekku.scripts.lib.blocks.metadata.*` directly (uses `_helpers` only). Enforced by import-linter `Migrations isolation` contract.
- DR-136 §6.5 migration is forward-only, idempotent; drift entries logged, durable cases promoted to DL-* on human review (DEC-138-05).
- Tolerant-on-read holds across in-repo corpus until per-kind strict-flip; strict-flip is verification-bearing operation (DEC-138-04).

### 2.4 Out of scope

- Relations-shape changes (DR-136 DEC-005 deferred to sibling DE-142).
- Body-prose excision of legacy `## Risks` sections beyond the migration step's own handling (DR-136 §11.5 `validate --prune-duplicates` — opportunistic, not blocking DE-138).
- `audit_gate` semantics (governed by DE-074/DE-079).
- Per-kind sweep for non-delta artefacts (siblings DE-139..DE-142).
- Removal of transition-window FM-fallback in `_derive_applies_to` (follow-up cleanup delta post consumer-migration window).

## 3. Architecture Intent

### 3.1 Target placement (DR-136 §6.1)

| Field | Target | Rationale |
|---|---|---|
| `applies_to` | **Derived** at load from `supekku:delta.relationships@v1` (FM key cut; transition-window fallback until strict-flip) | ADR-010 rule 4 (never duplicate); single source of truth in relationships block |
| `context_inputs` | **Block** `supekku:delta.context_inputs@v1` (strict canonical schema + field aliases) | ADR-010 §2 (schema-versioned, self-contained list-of-objects) |
| `risk_register` | **Block** `supekku:delta.risk_register@v1` (strict canonical schema + field aliases) | Same |
| `outcome_summary` | **Prose body** `## Outcome` section (before `## Implementation Notes` or EOB) | ADR-010 §3 (volatile execution detail; human-authored, narrative) |
| `audit_gate` / `audit_gate_rationale` | FM (unchanged) | ADR-010 §1 (stable contract, tooling-consumed) |
| Body §7 Risks | **Deleted** | Block-canonical for risks; body version was parallel-truth |

### 3.2 Lifecycle impact

- **Before strict-flip** (sweep ongoing): loaders read block-first then FM-fallback. Unmigrated deltas with FM `applies_to` only continue to satisfy coverage gates. `validate file` strict reports FM `applies_to` as a fixable warning.
- **At strict-flip** (end of DE-138 P04 — DEC-138-04): `[validation.strict] delta = true`. FM `applies_to`/`context_inputs`/`risk_register`/`outcome_summary` become strict-mode errors. `validate --fix` removes leftover FM keys.
- **After strict-flip + consumer-migration window**: transition-window FM-fallback in `_derive_applies_to` becomes dead code; removed in follow-up cleanup delta (tracked in §15).

### 3.3 Guiding principles

- **Single source of truth per field**: ADR-010 conflict-resolution rule 4. Migration cuts FM where block becomes canonical; deletes body §7 where block holds risk data.
- **Tolerant-on-read, strict-on-validate**: per DR-136 §5.2. Loaders pass `strict=False` (alias silently normalised); CLI `validate` passes `strict=True` (alias becomes fixable warning).
- **Forward-only migration**: DR-137 frozen-forever Protocol (DEC-137-26). Defects in `v0_10_0_001_delta_blocks` are fixed via new ordinal step (`v0_10_0_002_*`), never by editing the existing step.
- **POL-001 reuse**: existing `BlockMetadata`, `FieldMetadata.aliases`, `MetadataValidator`, `admin migrate` orchestrator, `validate workspace --kind`, skill gates — all consumed unchanged. DE-138 declares schemas and lands the step folder; does not modify infra.
- **POL-003 module boundary** (DR-136 §11.1): migration step is in scope for the `Migrations isolation` import-linter contract.

## 6. applies_to derivation + coverage gate

### 6.1 Load-time synthesis (DEC-138-01)

`ChangeArtifact.applies_to: dict[str, Any]` stays a stored field on the frozen dataclass. `load_change_artifact` in `supekku/scripts/lib/changes/artifacts.py` is the single synthesis point. New helper:

```python
def _derive_applies_to(
    block: DeltaRelationshipsBlock | None,
    frontmatter: dict[str, Any],
    *,
    strict: bool,
) -> dict[str, Any]:
    """Synthesise applies_to from the relationships block.

    Block-first; FM-second only while strict=False (transition window).
    Post-strict-flip, FM applies_to is silently ignored (validate file
    surfaces the strict error separately).
    """
    if block is not None:
        specs = sorted({str(s) for s in (block.data.get("specs", {}).get("primary") or [])})
        reqs = sorted({
            str(r)
            for key in ("implements", "updates", "verifies")
            for r in (block.data.get("requirements", {}).get(key) or [])
        })
        if specs or reqs:
            return {"specs": specs, "requirements": reqs}
    if strict:
        return {}
    fm = frontmatter.get("applies_to") or {}
    return dict(fm) if isinstance(fm, dict) else {}
```

`load_change_artifact` calls `_derive_applies_to(block, frontmatter, strict=strict_map.get("delta", False))`. The current lines 111-149 dual-merge logic is **deleted**; replaced by single-source synthesis above.

### 6.2 Transition-window semantics

- `strict=False`: block-first, FM-fallback. Unmigrated deltas with FM `applies_to` only continue to satisfy coverage gates.
- `strict=True`: block is sole source. FM `applies_to` silently ignored at load. `validate file` separately reports the FM key as a strict-mode error.

### 6.3 Coverage gate (`changes/coverage_check.py:120`)

**Unchanged**. Reads `delta.applies_to.get("requirements", [])` — same surface, different source. Single-source synthesis behind the field is invisible to coverage logic.

DE-138 itself self-bootstraps: `complete delta DE-138` uses this coverage gate. The first delta to close under the new synthesis is DE-138; its `applies_to` derives from its own `supekku:delta.relationships@v1` block (DE-138.md lines 40-60). VT-DE138-COV-001 covers.

### 6.4 Removal trigger for FM-fallback (follow-up delta)

Per DR-136 §6.4: fallback removed after consumer-migration windows close. Not in DE-138 scope. Trigger conditions for the cleanup delta:

1. All in-repo deltas pass `validate workspace --kind delta --strict --no-tolerated-aliases` clean.
2. Consumer-repo migration window elapsed (≥1 minor version cycle past DE-138 release).
3. No tolerated-alias entries remain on `delta.context_inputs.type`.

When all three hold, cleanup delta deletes the `strict=False` branch in `_derive_applies_to` and removes the FM `applies_to` key from any leftover artefacts.

### 6.5 `ChangeArtifact.to_dict()`

Lines 63-64 emit `applies_to` if non-empty. **Unchanged** — same registry payload. Consumers see the same shape; the difference is upstream (where it came from), not downstream (what they get).

### 7.2 Step class

```python
class DeltaBlocksStep(BaseMigrationStep):
    """v0_10_0_001 — propagate ADR-010 placement to delta artefact kind.

    Forward-only. Idempotent. Cuts §4 universals + FM keys (applies_to,
    context_inputs, risk_register, outcome_summary); synthesises blocks;
    moves outcome_summary to body §Outcome.
    """

    applies_to_kind = "delta"
    description = "ADR-010 placement: applies_to derived; context_inputs/risk_register → blocks; outcome_summary → body §Outcome"

    def applies_to(self, file_path: Path) -> bool:
        # Orchestrator already kind-filters; descend further only to skip
        # files that already match the post-migration shape (idempotence
        # short-circuit — avoids rewriting clean files).
        fm, body = load_markdown_file(file_path)
        legacy_keys = {"applies_to", "context_inputs", "risk_register", "outcome_summary"}
        return bool(legacy_keys & set(fm))

    def preview(self, file_path: Path) -> StepPreview: ...
    def apply(self, file_path: Path) -> StepResult: ...
```

### 7.3 `apply()` mechanics

Forward-only transform per file:

1. **Load**: `load_markdown_file(path)` → `(frontmatter, body)`.
2. **Cut §4 universals** (per DR-136 §4): re-runs cleanly if already cut.
3. **Synthesise `supekku:delta.context_inputs@v1` block** from FM `context_inputs` if block missing:
   - Normalise per §5.3 rules.
   - Render block via new `render_delta_context_inputs_block` helper.
4. **Synthesise `supekku:delta.risk_register@v1` block** from FM `risk_register` if block missing:
   - Normalise (`description→title`, default `status=open`, drift on missing `likelihood`/`impact`).
5. **Move `outcome_summary` FM → body `## Outcome`** (DEC-138-07):
   - Insertion point: before `## Implementation Notes` if present; else EOB.
   - Idempotence: if body already has `## Outcome`, no-op (skip + log).
6. **Remove FM keys**: `applies_to`, `context_inputs`, `risk_register`, `outcome_summary`. Cut universals.
7. **Body §7 Risks handling**: if body contains `## 7. Risks & Mitigations` (case-insensitive on `risks`/`mitigations`), delete section header + content until next `## ` heading. Captured drift entry `kind: body_risk_narrative` carries the deleted prose verbatim for VA-DE138-RISK-RECON-001.
8. **Renumber body** §§8-9 → §§7-8 if numbered prefix present. Unnumbered headings (e.g. `## Implementation Notes`) left as-is. Drift entry per affected file (cross-references may break — operator audits manually post-sweep).
9. **Serialise**: rewrite file via `dump_markdown_file` (comment-preserving emitter from DE-137).

### 7.4 Idempotence

Re-running step on a migrated file:
- §7.2 `applies_to(path)`: returns `False` (legacy keys all gone) → orchestrator skips.
- If user manually re-adds an FM key mid-sweep: step re-runs the cut on next sweep; drift entry if block synthesis would produce different shape than existing block.

### 7.5 Drift handling (DEC-138-05)

Drift entry sources:
- `context_inputs[]` plain string with unmappable `type` → log.
- `context_inputs[]` block already exists AND FM `context_inputs` disagrees with block shape → log.
- `risk_register[]` missing `likelihood`/`impact` → log.
- FM `applies_to` disagrees with derived block → log.
- Body `## Outcome` exists AND FM `outcome_summary` is present and differs → log (no auto-overwrite).
- Body §7 Risks deletion → log `kind: body_risk_narrative` with full deleted content for VA reconciliation.
- Body renumbering applied → log `kind: body_renumber` listing shifted heading numbers.

Log destination (orchestrator, DR-137 §5.6): `.spec-driver/run/migrations/<timestamp>-delta.md`. Format: markdown table per file × drift type × resolution-needed annotation.

Promotion to DL-*: out of step scope. Operator + agent review (VA-DE138-DRIFT-001 + VA-DE138-RISK-RECON-001) post-sweep; manual DL-* filing for durable drift.

## 11. Strict-flip + post-flip gate

### 11.1 Phase boundary placement

DE-138's IP-138 (drafted in next step) carries:

```
P01 — Schema + load-time synthesis
P02 — Migration step (code, no sweep)
P03 — Sweep + reconciliation
P04 — Strict-flip + post-flip gate
```

Strict-flip lives **inside P04**, not at delta close. Reason: flip is the verification-bearing operation; closing the delta without the flip having held under workspace validation leaves the strict-on-validate assertion unverified.

### 11.2 Pre-flip checklist (P04 entrance)

- [ ] P03 sweep applied to all in-repo deltas.
- [ ] VA-DE138-RISK-RECON-001 closed.
- [ ] VA-DE138-DRIFT-001 closed.
- [ ] `validate workspace --kind delta` clean (tolerant).
- [ ] `validate workspace --kind delta --strict` clean (strict).

### 11.3 Flip mechanics

`workflow.toml`:

```toml
[validation.strict]
delta = true

[schema_version]
delta = "0.10.0+001"
```

Edit method: `uv run spec-driver admin migrate delta --strict` (DR-137 §5.6 — orchestrator writes flag after sweep passes) OR manual edit.

### 11.4 Post-flip gate (P04 exit)

- [ ] `validate workspace --kind delta --strict --no-tolerated-aliases` returns 0 errors.
- [ ] `validate workspace --strict` whole-corpus does not regress (only sibling-draft audit-gate warnings remain).
- [ ] `complete delta DE-138` (dry-run) passes coverage gate via derived `applies_to`.
- [ ] `list deltas` smoke-check: every row renders correctly under new column set.

### 11.5 Rollback

If post-flip baseline regresses:

1. Triage: identify which artefact(s) failed strict validation.
2. Tactical: revert `[validation.strict] delta = true` in `workflow.toml` (one-line).
3. Resolution: fix offending artefact(s) or amend migration step via new ordinal step (`v0_10_0_002_<slug>`) — forward-only, never edit existing.
4. Re-flip: P04 re-runs after fix lands.

### 11.6 Consumer-repo migration (out of DE-138 scope)

DR-136 §11.3. Consumer-repo upgrade path:

- Consumer pulls new spec-driver version.
- Runs `spec-driver admin migrate --check` → sees pending `v0_10_0_001_delta_blocks`.
- Runs `spec-driver admin migrate delta` → sweep applies on consumer corpus.
- Optionally runs `spec-driver admin migrate delta --strict` → flips their workflow.toml.

## 12. Design decisions & trade-offs

See frontmatter `design_decisions` for canonical list. Key trade-offs:

- **DEC-138-01** (stored field vs `@property`): chose stored field per Q1. Trade-off — "derived" semantics held only at load, not at access. Acceptable because `frozen=True` precludes mutation and `load_change_artifact` is the single load path. Coverage gate untouched.
- **DEC-138-02** (strict block schemas): chose canonical schemas + aliases per Q2. Trade-off — locks corpus shape now; new artefact types must extend the enum (DR amendment if not absorbable by alias). Accepted because (i) field aliases absorb observed dialect drift, (ii) DR-136 §6.5 already signals strict intent, (iii) ADR-010 §2 requires schema-versioned blocks.
- **DEC-138-04** (flip inside delta): chose A per Q4. Trade-off — risk surfaces post-flip mean reopening or follow-up. Mitigated by pre/post-flip gates (§11.2, §11.4) and rollback path (§11.5).
- **DEC-138-05** (drift handling): chose C per Q5. Trade-off — operator review burden during sweep. Accepted because flat DL-* auto-filing would explode noise; light-touch matches ADR-008 (observed evidence reconciled selectively).
- **DEC-138-06** (FR-007 acceptance): chose C per Q6. Trade-off — VT fixture scaffolding higher upfront cost than VA manifest review. Amortised across siblings DE-139..DE-142.
- **DEC-138-07** (body §7 → drift log → VA reconciliation): trade-off — multi-step process introduces a gate. Accepted because automated narrative migration is unsafe (body prose is richer than FM `mitigation:` one-liner; agent judgement required).
- **DEC-138-08** (verification asymmetry deferred): trade-off — accepting a real defensible asymmetry without immediate fix. Reasoning — structure exists on IP (correct location); patching practitioner habits requires multi-vector improvement, not flat code change.
