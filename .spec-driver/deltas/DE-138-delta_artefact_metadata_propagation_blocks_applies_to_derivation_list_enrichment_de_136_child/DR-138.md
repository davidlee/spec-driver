# DR-138 – Delta artefact metadata propagation: blocks + applies_to derivation + list enrichment (DE-136 child)

## 1. Executive Summary

- **Delta**: [DE-138](./DE-138.md) — propagate ADR-010 placement to delta artefact kind.
- **Canonical design reference**: DR-136 §6 (delta placement table + body reshape + list enrichment + applies_to derivation + migration step naming). DR-138 refines and locks residual implementation decisions for the delta kind.
- **Status**: draft.
- **Synopsis**: First per-artefact propagation of the DE-136 program (DEC-004 sequence position 3 — after DE-118 and DE-137). `applies_to` becomes a derived load-time property sourced from `supekku:delta.relationships@v1`; `context_inputs` and `risk_register` move from FM to strict-schema blocks (`supekku:delta.context_inputs@v1`, `supekku:delta.risk_register@v1`); `outcome_summary` moves from FM to body `## Outcome`; body `## 7. Risks` deletes; `list deltas` columns enrich; `[validation.strict] delta = true` flips inside this delta with post-flip verification gate.

## 2. Problem & Constraints

### 2.1 Current behaviour

Delta artefacts duplicate structured data across three surfaces:

| Field | FM | Block | Body |
|---|---|---|---|
| `applies_to` | ✓ | partially synthesised in `artifacts.py:111-149` (block-first, FM-fallback at load) | – |
| `context_inputs` | ✓ (heterogeneous: `id`/`ref`, `summary`/`note`/`annotation`, sometimes plain strings) | – | – |
| `risk_register` | ✓ (drift: `title` vs `description`, optional `exposure`/`status`) | – | sometimes mirrored in §7 |
| `outcome_summary` | ✓ | – | sometimes mirrored in §1/§9 |
| §7 Risks (prose) | – | – | ✓ (parallel maintenance with FM `risk_register`) |

Triple-entry pattern per ADR-010 §"Conflict resolution" rule 4 (never duplicate). Cost: silent drift between FM and prose, FM bloat (PROD-004.FR-007 motivation), inconsistent block schemas across artefact kinds (PROD-004.FR-001 strict-on-validate impossible to deliver for delta kind).

### 2.2 Drivers / inputs

- DE-136 program — propagate ADR-010 across artefact kinds; DE-138 is delta-kind propagation (first per-artefact per DEC-004).
- DR-136 §6 — canonical placement table for delta kind.
- DE-118 + DE-137 closed (AUD-026 conformance evidence) — `MetadataValidator` strict-mode, alias machinery, `admin migrate` orchestrator, `validate workspace --kind`/`file`, skill validate-gates all operational.
- PROD-004.FR-001 (single validation layer), FR-002 (strict-on-validate), FR-007 (compaction infra exercised) are the umbrella requirements this delta verifies.

### 2.3 Constraints / guardrails

- ADR-010 placement is doctrinal default; deviation requires DR amendment.
- POL-001 (reuse): extend existing `BlockMetadata` / `MetadataValidator` infra; no parallel validator.
- POL-003 (module boundaries) + DR-136 §11.1: `migrations/v0_10_0_001_delta_blocks/migration.py` may not import `validation/` or `supekku.scripts.lib.blocks.metadata.*` directly (uses `_helpers` only). Enforced by import-linter `Migrations isolation` contract.
- DR-136 §6.5 migration is forward-only, idempotent; drift entries logged, durable cases promoted to DL-* on human review (DEC-138-05).
- Tolerant-on-read holds across in-repo corpus until per-kind strict-flip; strict-flip is verification-bearing operation (DEC-138-04).

### 2.4 Out of scope

- Relations-shape changes (DR-136 DEC-005 deferred to sibling DE-142).
- Body-prose excision of legacy `## Risks` sections beyond the migration step's own handling (DR-136 §11.5 `validate --prune-duplicates` — opportunistic, not blocking DE-138).
- `audit_gate` semantics (governed by DE-074/DE-079).
- Per-kind sweep for non-delta artefacts (siblings DE-139..DE-142).
- Removal of transition-window FM-fallback in `_derive_applies_to` (follow-up cleanup delta post consumer-migration window).

## 3. Architecture Intent

### 3.1 Target placement (DR-136 §6.1)

| Field | Target | Rationale |
|---|---|---|
| `applies_to` | **Derived** at load from `supekku:delta.relationships@v1` (FM key cut; transition-window fallback until strict-flip) | ADR-010 rule 4 (never duplicate); single source of truth in relationships block |
| `context_inputs` | **Block** `supekku:delta.context_inputs@v1` (strict canonical schema + field aliases) | ADR-010 §2 (schema-versioned, self-contained list-of-objects) |
| `risk_register` | **Block** `supekku:delta.risk_register@v1` (strict canonical schema + field aliases) | Same |
| `outcome_summary` | **Prose body** `## Outcome` section (before `## Implementation Notes` or EOB) | ADR-010 §3 (volatile execution detail; human-authored, narrative) |
| `audit_gate` / `audit_gate_rationale` | FM (unchanged) | ADR-010 §1 (stable contract, tooling-consumed) |
| Body §7 Risks | **Deleted** | Block-canonical for risks; body version was parallel-truth |

### 3.2 Lifecycle impact

- **Before strict-flip** (sweep ongoing): loaders are unconditionally tolerant — block-first, FM-fallback when block absent (DEC-138-10). Unmigrated deltas with FM `applies_to` only continue to satisfy coverage gates. `validate file` strict reports FM `applies_to` as a fixable warning.
- **At strict-flip** (end of DE-138 P04 — DEC-138-04): `[validation.strict] delta = true` + `--no-tolerated-aliases` wired through workspace validator (DEC-138-14). FM `applies_to`/`context_inputs`/`risk_register`/`outcome_summary` become validator-layer errors. Loader behaviour unchanged: when block present it remains the sole source; FM keys silently shadowed at load (validator surfaces them separately). `validate --fix` removes leftover FM keys.
- **After strict-flip + consumer-migration window**: transition-window FM-fallback in `_derive_applies_to` becomes dead code; removed in follow-up cleanup delta (tracked in §15).

### 3.3 Guiding principles

- **Single source of truth per field**: ADR-010 conflict-resolution rule 4. Migration cuts FM where block becomes canonical; deletes body §7 where block holds risk data.
- **Tolerant-on-read, strict-on-validate** (DEC-138-10): per DR-136 §5.2. Loaders are unconditionally tolerant — no `strict` parameter threaded through load path. CLI `validate` owns strict mode; alias becomes fixable warning at validator layer.
- **Forward-only migration**: DR-137 frozen-forever Protocol (DEC-137-26). Defects in `v0_10_0_001_delta_blocks` are fixed via new ordinal step (`v0_10_0_002_*`), never by editing the existing step.
- **POL-001 reuse**: existing `BlockMetadata`, `FieldMetadata.aliases`, `MetadataValidator`, `admin migrate` orchestrator, `validate workspace --kind`, skill gates — all consumed unchanged. DE-138 declares schemas and lands the step folder; does not modify infra.
- **POL-003 module boundary** (DR-136 §11.1): migration step is in scope for the `Migrations isolation` import-linter contract.

`BlockMetadata` fields (declared in `supekku/scripts/lib/blocks/delta_metadata.py`):

| Field | Type | Required | Enum / aliases |
|---|---|---|---|
| `schema` | str | yes | constant `supekku.delta.context_inputs` |
| `version` | int | yes | constant `1` |
| `entries[]` | list[object] | yes | (entry shape below) |
| `entries[].type` | str (enum) | yes | enum with normalisation aliases (below); `tolerated_aliases` reserve for legacy long-tail during sweep |
| `entries[].id` | str | yes | free-form; `field_aliases: {ref: id}` |
| `entries[].summary` | str | no | free-form; `field_aliases: {note: summary, annotation: summary}`. Schema is non-nullable (str, not `str | None`); the migration must **omit** the key when no value is available — never emit `summary: null` (F-138-31). |

**`entries[].type` enum (canonical set observed in corpus)**:
`delta`, `revision`, `audit`, `phase`, `plan`, `spec`, `prod`, `adr`, `issue`, `problem`, `improvement`, `risk`, `memory`, `decision`, `policy`, `standard`, `document` (external/non-spec-driver reference), `code` (repo file/module), `research` (investigation note or RC-*).

**`type` aliases** (permanent, in `FieldMetadata.aliases`):
- `reference` → `document`
- `brief` → `document`
- `external` → `document`
- `investigation` → `research`

**`type` tolerated_aliases** (F-138-B): `{unknown: document}` with sunset trigger = delta-sweep close. Migration step writes `unknown` for any FM `type` that fails alias map AND canonical enum; tolerated_alias mechanism normalises to `document` at load while strict-mode + `--no-tolerated-aliases` flags the artefact for human re-classification. Post-sweep, operator audit promotes specific entries to canonical types; tolerated alias sunsets when zero entries remain.

### 5.3 Migration normalisation rules

Migration step (§7) applies these on FM → block conversion:

- `context_inputs[]` plain string → `{type: "document", id: <string>}` (summary key omitted, not emitted as `null` — schema is non-nullable str; F-138-31) + drift entry (likely needs human assignment of canonical `type`).
- `context_inputs[].ref` → `.id`.
- `context_inputs[].note` → `.summary`.
- `context_inputs[].annotation` → `.summary`.
- `context_inputs[].type` matched against alias map then enum; unknown → tolerated_alias `unknown` + drift entry.
- `risk_register[].description` → `.title`.
- `risk_register[]` missing `likelihood`/`impact` → drift entry (no auto-default; risk metadata is intentional).
- `risk_register[]` missing `status` → defaults to `open`.

### 5.4 Validation surface

- Loader path: `MetadataValidator.validate(data, strict=False)` — aliases silently normalised, truly unrecognised values warn.
- `validate file` / `validate workspace --kind delta --strict`: permanent aliases become fixable warnings (`--fix` applies). The literal tolerated alias `unknown` (mapped to `document` per §5.1) is **accepted** under default strict and warned-for-future-review; truly unrecognised values (not in canonical enum, not in `aliases`, not in `tolerated_aliases`) error out (F-138-32).
- `validate workspace --strict --no-tolerated-aliases`: rejects the tolerated alias `unknown` as an error; truly unrecognised values also error. This is the gate that surfaces remaining `unknown` entries for operator re-classification before tolerated_alias sunset.
- `validate templates`: `.spec-driver/templates/delta.md` carries inline enum-comment hints (`# one of: …`) per DR-136 §5.1 (Y).

## 6. applies_to derivation + coverage gate

### 6.1 Load-time synthesis (DEC-138-01, DEC-138-10, DEC-138-11)

`ChangeArtifact.applies_to: dict[str, Any]` stays a stored field on the frozen dataclass. `load_change_artifact` in `supekku/scripts/lib/changes/artifacts.py` is the single synthesis point. Loader is unconditionally tolerant — strict mode lives at the validator layer, not the load path (DEC-138-10). Two discrete helpers replace the current merge region.

```python
def _derive_applies_to(
    block: DeltaRelationshipsBlock | None,
    frontmatter: dict[str, Any],
) -> dict[str, Any]:
    """Synthesise applies_to from the relationships block.

    Block-first when present (sole source — FM silently shadowed even if
    populated). FM-fallback only when block absent (transition window for
    unmigrated artefacts). Validator layer separately reports FM applies_to
    as a strict-mode error post-flip.
    """
    if block is not None:
        spec_groups = block.data.get("specs") or {}
        specs = sorted({
            str(s)
            for key in ("primary", "collaborators")
            for s in (spec_groups.get(key) or [])
        })
        reqs = sorted({
            str(r)
            for key in ("implements", "updates", "verifies")
            for r in (block.data.get("requirements", {}).get(key) or [])
        })
        if specs or reqs:
            return {"specs": specs, "requirements": reqs}
    fm = frontmatter.get("applies_to") or {}
    return dict(fm) if isinstance(fm, dict) else {}


def _derive_revision_link_relations(
    block: DeltaRelationshipsBlock | None,
) -> list[dict[str, str]]:
    """Project revision_links.{introduces,supersedes} into relations entries.

    Preserves existing artifacts.py:151-154 behaviour as a discrete helper so
    deleting the applies_to merge region does not collateral-drop the
    RE-* relation projection (F-138-16). ADR-002 backlinks rule is not
    violated — projection derives from authored block content, not from
    foreign-artefact backlinks.
    """
    if block is None:
        return []
    revision_links = block.data.get("revision_links") or {}
    out: list[dict[str, str]] = []
    for rel_type in ("introduces", "supersedes"):
        for target in revision_links.get(rel_type) or []:
            out.append({"type": rel_type, "target": str(target)})
    return out
```

`load_change_artifact` calls both helpers; appends revision-link relations to the relations list it already builds from FM. The current lines 111-149 dual-merge logic is **deleted**; the revision_links projection at 151-154 is replaced by `_derive_revision_link_relations`.

**Collaborators rationale (DEC-138-11)**: 16 in-repo deltas carry collaborator spec IDs inside FM `applies_to.specs` (e.g. DE-075, DE-079, DE-136). Current loader merges them through FM. ### 7.3 `apply()` mechanics

Forward-only transform per file. **All I/O routes through `spec_driver.migrations._helpers`** (`split_frontmatter`, `atomic_write`, `replace_in_frontmatter_block`) + `yaml` stdlib parser — the legacy `dump_markdown_file` entrypoint was removed in IP-137-P02 and `load_markdown_file`/`dump_markdown_file_*` live in `supekku/scripts/lib/core/spec_utils.py` which the `Migrations isolation` contract forbids (F-138-26, DEC-138-12).

1. **Read + split**: `text = file_path.read_text(encoding="utf-8")`; `yaml_text, body = split_frontmatter(text)`; `frontmatter = yaml.safe_load(yaml_text) or {}`.
2. **Cut §4 universals** (DR-136 §4 — `lifecycle`, `aliases`, `auditers`, `source`; **not** `owners` or `summary`): re-runs cleanly if already cut. Applied via in-memory mutation of `frontmatter` dict.
3. **Ensure `supekku:delta.relationships@v1` block exists** (F-138-27): if absent AND FM `applies_to` non-empty, synthesise minimal block from FM payload before any FM cut. Mapping: `fm.applies_to.specs[]` → `specs.primary[]`; `fm.applies_to.requirements[]` → `requirements.implements[]` (default — most conservative; revision/verifies splits cannot be inferred from FM and would need operator review). Emit drift entry `kind: relationships_block_synthesised_from_fm` listing the synthesised IDs so operators can refine `primary` vs `collaborators` and `implements` vs `updates`/`verifies` post-sweep. **Block-present case**: no-op. **Block-absent + FM-empty case**: no-op (no data to preserve; coverage gate already returns empty). Without this step, consumer repos with FM `applies_to` but no block would lose linkages at FM cut.
4. **Synthesise `supekku:delta.context_inputs@v1` block** from FM `context_inputs` if block missing:
   - Normalise per §5.3 rules.
   - Render block via migration-local `_render_context_inputs_block` (frozen-local emitter per §7.7, DEC-138-12).
5. **Synthesise `supekku:delta.risk_register@v1` block** from FM `risk_register` if block missing:
   - Normalise (`description→title`, default `status=open`, drift on missing `likelihood`/`impact`).
   - Render via migration-local `_render_risk_register_block`.
6. **Move `outcome_summary` FM → body `## Outcome`** (DEC-138-07):
   - Insertion point: before `## Implementation Notes` if present; else EOB.
   - Idempotence: if body already has `## Outcome`, no-op (skip + log).
   - **Multi-line handling (F-138-H)**: single-line scalar → `## Outcome\n\n<one-liner>`; YAML folded scalar (`>`) → paragraphs preserved; literal scalar (`|`) → verbatim with markdown intact. VT-DE138-BODY-001 includes single-line + folded + literal fixtures.
7. **Reconcile FM `applies_to.specs` ↔ block `specs.primary ∪ specs.collaborators`** (DEC-138-11, F-138-15):
   - Compute `unmatched = set(fm.applies_to.specs) - (block.specs.primary ∪ block.specs.collaborators)`.
   - If non-empty: emit drift entry `kind: fm_specs_unmatched` listing the unmatched IDs. Do **not** auto-promote to block — author may have meant a renamed/removed spec. Operator review via VA-DE138-DRIFT-001.
   - Same reconciliation for `fm.applies_to.requirements ↔ block.requirements.{implements,updates,verifies}`.
8. **Remove FM keys**: `applies_to`, `context_inputs`, `risk_register`, `outcome_summary` + §4 universals already mutated in step 2.
9. **Body §7 Risks handling**: if body contains `## 7. Risks & Mitigations` (case-insensitive on `risks`/`mitigations`), delete section header + content until next `## ` heading. Captured drift entry `kind: body_risk_narrative` carries the deleted prose verbatim for VA-DE138-RISK-RECON-001.
10. **Renumber body** §§8-9 → §§7-8 if numbered prefix present (F-138-D: top-level `## N.` only; sub-headings `### N.M` left as-is — cross-references typically point at top-level sections). Unnumbered headings (e.g. `## Implementation Notes`) left as-is. Drift entry `kind: body_renumber` per affected file lists shifted heading numbers; **cross-refs in other artefacts (DR/IP/phase/notes) are not rewritten** — operator audits affected files manually post-sweep using the drift log as the change manifest.
11. **Serialise**: reassemble `text = "---\n" + yaml.safe_dump(frontmatter, sort_keys=False) + "---\n" + body`; write via `atomic_write(file_path, text)`. Atomic write semantics (rename-into-place) come from `_helpers.atomic_write`; no supekku/spec_utils call. **Note**: this serialiser is a frozen migration-internal emitter — it does not preserve YAML comments, since legacy DE-* FM should not carry comments and any present would already be flagged for human review at strict-flip.

### 7.4 Idempotence + partial-shape handling

Re-running step on a migrated file:
- §7.2 `applies_to(path)`: returns `False` (legacy keys all gone) → orchestrator skips.
- If user manually re-adds an FM key mid-sweep: step re-runs the cut on next sweep; drift entry if block synthesis would produce different shape than existing block.

**Partial-shape handling (F-138-K, F-138-27)** — file has both FM key and corresponding block:

| FM vs block | Resolution |
|---|---|
| Block matches FM exactly (normalised) | Cut FM silently; no drift entry; idempotent |
| Block disagrees with FM (different content) | **Keep block** (canonical); discard FM with drift entry `kind: fm_block_disagreement` logging the dropped FM payload verbatim |
| Block missing, FM present (context_inputs / risk_register) | Synthesise block from FM per §5.3 normalisation |
| Block present, FM absent | Step skips file (already migrated) |
| `delta.relationships@v1` block missing, FM `applies_to` non-empty | Synthesise minimal relationships block (§7.3 step 3); drift entry `kind: relationships_block_synthesised_from_fm` so operator refines primary/collaborators + implements/updates/verifies splits post-sweep |
| `delta.relationships@v1` block missing, FM `applies_to` empty/absent | No-op (no data to preserve) |

### 11.5 Rollback

Rollback has two distinct scopes — they are not interchangeable (F-138-22):

**A. Validation-mode rollback** (covers borderline strict-mode regressions on already-migrated artefacts):

1. Triage: identify which artefact(s) failed strict validation.
2. Revert `[validation.strict] delta = true` in `workflow.toml` (one-line). Validator returns to tolerant; FM-cut artefacts continue to work because loader is already unconditionally tolerant.
3. Resolution: fix offending artefact(s) by hand OR amend migration step via new ordinal step (`v0_10_0_002_<slug>`) — forward-only, never edit existing v0_10_0_001.
4. Re-flip: P04 re-runs after fix lands.

**B. Data recovery** (covers undesired data-shape mutation from the sweep itself — deleted body §7 prose, removed FM keys, mutated blocks):

1. Migration is forward-only by DR-137 DEC-137-26 — the sweep does not have an inverse step. Toggling `[validation.strict] delta = false` does **not** restore deleted content.
2. Recovery path is `git`-mediated: P03 sweep is committed as a discrete checkpoint commit; `git revert <sweep-commit>` restores the pre-sweep file shapes wholesale. Drift log + reconciliation patch (VA-DE138-RISK-RECON-001) commits separately so revert grain is clean.
3. Selective recovery (single file): `git checkout <pre-sweep-sha> -- .spec-driver/deltas/<DE-NNN>/<file>.md`, then re-run sweep against that file once the fix lands in a `v0_10_0_002_<slug>` step.
4. Rehearsal: P03 entrance gate (§11.2) includes a tag/commit checkpoint immediately before sweep so the revert anchor is unambiguous.

Validation-mode rollback (A) is one-line and trivial. Data recovery (B) requires git operations; treat the sweep commit as a transactional boundary.
