# DR-139 – Spec artefact metadata propagation: PROD+SPEC blocks, taxonomy strict, list enrichment (DE-136 child)

## 1. Executive Summary

- **Delta**: [DE-139](./DE-139.md) — propagate ADR-010 placement to spec and prod artefact kinds.
- **Canonical design reference**: DR-136 §7 (spec placement table); F-A scope-note (PROD placement sub-table, this DR's responsibility).
- **Status**: draft.
- **Synopsis**: Second per-artefact propagation of the DE-136 program (DEC-004 sequence position 4). `concerns`, `hypotheses`, `decisions` move from FM to strict-schema blocks; `scope` moves to prose; `verification_strategy` and `packages` cut from FM; `category` promoted to strict enum (SPEC only; PROD stays freeform); `c4_level` tolerates `unknown` with drift tracking; `list specs` columns enriched; `[validation.strict] spec = true` flips with post-flip verification gate. PROD coverage (F-A) addressed via shared block schemas and kind-aware migration.

## 2. Problem & Constraints

### 2.1 Current behaviour

Spec FM carries structured list-of-objects fields (`concerns`, `hypotheses`, `decisions`) and scalar fields (`scope`, `verification_strategy`) that no code path reads from FM. These fields exist only as schema definitions — validated on write but never consumed programmatically. This is inert weight per ADR-010 rule 4.

`packages` (deprecated) has 50 in-tree occurrences and active model/index code despite being superseded by `sources`. `category`/`c4_level` are defined but only warn-level validated — specs can carry invalid values or omit them without error.

PROD frontmatter carries `hypotheses` and `decisions` with identical shapes to SPEC but no shared block schema infrastructure.

### 2.2 Key numbers (migration burden)

| Field | SPEC files | PROD files | Code readers |
|---|---|---|---|
| `concerns` | 0 | n/a | 0 |
| `hypotheses` | 0 | 0 | 0 |
| `decisions` | 0 | 0 | 0 |
| `scope` | 1 | 0 | 0 |
| `verification_strategy` | 0 | 0 | 0 |
| `packages` | ~50 | 0 | model property, index builder, CLI flag |
| `category` | 53 | n/a | model property, index, CLI filter |
| `c4_level` | 53 | n/a | model property, index, CLI filter |

Migration burden is light for FM→block moves (0 files to migrate). Main effort is `packages` removal (50 files, auto-cleaned) and taxonomy strict promotion (drift entries for ~8 specs missing values).

### 2.3 Drivers / inputs

- DR-136 §7 — canonical placement table for spec kind.
- F-A scope-note (DE-136 notes.md) — PROD placement sub-table is this delta's responsibility.
- DE-138 precedent — sweep procedure, migration step shape, strict-flip gate, list enrichment pattern.
- ADR-003 — unit/assembly category distinction; strict enforcement here closes the gap.
- PROD-004.FR-001 (single validation layer), FR-002 (strict-on-validate), FR-003 (discoverability).

### 2.4 Constraints / guardrails

- ADR-010 placement is doctrinal default.
- POL-001 (reuse): shared block schemas for SPEC/PROD where shapes match (DEC-139-01).
- POL-003 (module boundaries): migration step in `spec_driver/migrations/`, may not import `validation/`.
- DR-136 §7.4 burden realism: manual taxonomy reconciliation tracked but not gating delta close.
- Tolerant-on-read holds until strict-flip (DEC-139-11, consistent with DEC-138-10).

### 2.5 Out of scope

- Requirements-in-spec block-ification (DE-140 scope).
- PROD `product_requirements` restructuring (lightweight FM field; DE-140 may subsume).
- Body-prose excision (`validate --prune-duplicates`) — DR-136 §11.5 interactive flow.
- Taxonomy manual reconciliation across all specs (tracked as drift; follow-up batch before umbrella close).

### 3.3 Lifecycle impact

- **Before strict-flip**: loaders unconditionally tolerant. Cut FM fields silently ignored at load. `validate file` strict reports them as fixable warnings.
- **At strict-flip**: `[validation.strict] spec = true` in `workflow.toml`. FM `concerns`/`hypotheses`/`decisions`/`scope`/`verification_strategy`/`packages` become validator-layer errors. `category` rejects non-enum values (except tolerated `unknown`). `validate --fix` removes cut keys.
- **Taxonomy reconciliation**: specs with `category: unknown` or `c4_level: unknown` carry drift entries. Manual reconciliation is tracked separately and gated at umbrella close, not delta close.

### 3.4 Guiding principles

- **Single source per field** (ADR-010): block-canonical for structured objects; FM for scalars and simple lists; prose for narrative.
- **Tolerant-on-read, strict-on-validate** (DEC-139-11, DEC-138-10 precedent).
- **Shared schemas where shapes match** (DEC-139-01, POL-001).
- **Forward-only migration** (DEC-137-26): defects fixed via new step, not editing existing.

## 4. Block schemas

### 4.1 `supekku:spec.concerns@v1`

```yaml supekku:spec.concerns@v1
schema: supekku.spec.concerns
version: 1
spec: SPEC-XXX
concerns:
  - name: concern-name
    description: |
      What this concern covers and why it matters.
```

Fields:
- `spec`: string (required) — parent spec/prod ID
- `concerns`: array (required, min_items: 0) of objects:
  - `name`: string (required) — kebab-case identifier
  - `description`: string (required) — prose explanation

### 4.2 `supekku:spec.hypotheses@v1`

Shared by SPEC and PROD kinds (DEC-139-01).

```yaml supekku:spec.hypotheses@v1
schema: supekku.spec.hypotheses
version: 1
spec: PROD-004
hypotheses:
  - id: HYP-01
    statement: Users prefer inline validation feedback over post-hoc reports
    status: proposed   # enum: proposed | validated | invalid
```

Fields:
- `spec`: string (required) — parent spec/prod ID
- `hypotheses`: array (required, min_items: 0) of objects:
  - `id`: string (required) — hypothesis identifier
  - `statement`: string (required) — testable claim
  - `status`: enum (required) — `proposed` | `validated` | `invalid`

### 4.3 `supekku:spec.decisions@v1`

Shared by SPEC and PROD kinds (DEC-139-01). `rationale` optional (PROD omits it).

```yaml supekku:spec.decisions@v1
schema: supekku.spec.decisions
version: 1
spec: SPEC-116
decisions:
  - id: DEC-01
    summary: Single metadata class per artefact kind
    rationale: |
      Centralises schema definition; enables template generation.
```

Fields:
- `spec`: string (required) — parent spec/prod ID
- `decisions`: array (required, min_items: 0) of objects:
  - `id`: string (required) — decision identifier
  - `summary`: string (required) — what was decided
  - `rationale`: string (optional) — why

## 9. Migration mechanics

### 9.1 Sweep procedure (DR-136 §11.2 pattern, DE-138 precedent)

```bash
spec-driver validate --kind spec                                  # 1. tolerant baseline
spec-driver admin migrate spec --dry-run                          # 2. preview
spec-driver admin migrate spec                                    # 3. apply
spec-driver validate --kind spec --strict                         # 4. strict gate
# 5. manual fix + drift for residuals
spec-driver validate --kind spec --strict --no-tolerated-aliases  # 6. final (pre-reconciliation)
```

Same procedure for `--kind prod`.

### 9.2 Migration step actions

For each spec/prod file:
1. **Cut deprecated FM keys**: `packages`, `scope`, `verification_strategy`, universal cuts (if not already done).
2. **Scope → prose**: if `scope` value exists in FM and body lacks `## Scope` section, append `## Scope\n\n{value}` before first numbered section or at body start. If body already has scope section, drop FM key only.
3. **Concerns/hypotheses/decisions FM → blocks**: if FM has populated arrays, synthesise the corresponding `supekku:spec.{field}@v1` block and insert after existing blocks. FM key removed. 0 files affected today, but step handles future edge cases.
4. **Taxonomy tolerance**: if SPEC-kind file missing `category` or `c4_level`, set to `unknown` + emit drift entry with sunset trigger.
5. **PROD-kind**: scope → prose, hypotheses/decisions FM → blocks (same logic), verification_strategy cut. category/product_requirements untouched.

Idempotent. Each action checks before applying.

### 9.3 Strict-flip

At delta close: `[validation.strict] spec = true` in `workflow.toml`. Post-flip `validate --kind spec --strict` must pass (all errors resolved or drift-tracked).
