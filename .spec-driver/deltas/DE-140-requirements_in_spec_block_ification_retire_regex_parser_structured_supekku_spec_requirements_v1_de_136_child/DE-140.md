## 3. Scope & Objectives

- **Primary Outcomes** (DR-136 §8):
  1. **Block schema** (§8.1): `supekku:spec.requirements@v1` — id, title, lifecycle (`pending|in-progress|active|baseline|verified|retired` enum), description, acceptance_criteria, tags. Strict-validated; requirement-lifecycle enum becomes first-class enforced.
  2. **Reading pipeline** (§8.2): `requirements/parser.py` reads block first; falls back to regex on legacy specs. Strict-on-validate per-spec only after migration.
  3. **Interactive migration** (§8.3): `spec-driver admin migrate-requirements <SPEC-ID>` — per-record `[a]ccept / [e]dit / [s]kip / [A]ccept-all-remaining / [q]uit`; `$EDITOR` opens YAML buffer per-record; writes block above prose body; legacy prose preserved (excision deferred to `validate --prune-duplicates`).
  4. **Migration step** (§8.3): `migrations/v04_spec_requirements.py` wraps interactive flow as MigrationStep; `--dry-run` shows proposed blocks; drift entry where parsing fails or lifecycle can't be inferred.
  5. **Code impact** (§8.4): NEW `blocks/spec_requirements.py`; `requirements/parser.py` block-first + regex fallback; `requirements/registry.py` wires block source; `specs/creation.py` emits empty block; `templates/spec.md` block placeholder; NEW `cli/admin/migrate_requirements.py`.
- **Operational Constraints**:
  - DE-137 closed first.
  - Per-spec migration is interactive (human-in-the-loop) — sweep cadence is human-paced, not batch.
  - Strict-on-validate flips per-kind only after all specs migrated (per-spec tracking is overkill; spec stays tolerant for unmigrated specs).
  - Legacy prose preserved by default — excision is opportunistic via `validate --prune-duplicates` (DR-136 §11.5).
- **Dependencies**:
  - **Hard**: DE-118, DE-137.
  - **Soft**: DE-129 (recent parser hardening — informs shape and lifecycle semantics).
