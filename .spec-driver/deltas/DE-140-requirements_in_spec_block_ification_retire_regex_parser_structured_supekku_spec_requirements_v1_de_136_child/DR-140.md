# DR-140 — Requirements-in-spec block-ification

## 1. Executive Summary

- **Delta**: [DE-140](./DE-140.md)
- **Status**: draft
- **Last Updated**: 2026-05-23
- **Synopsis**: Replace regex-based requirement parsing with structured `supekku:spec.requirements@v1` YAML blocks. Block-first reading with regex fallback for unmigrated specs. Interactive per-spec migration. Strict enforcement via schema_version flip.

## 2. Problem & Constraints

### Current Behaviour

Requirements are parsed from markdown prose via two regex grammars:
- **Bullet format** (`- **FR-001**: Title`) in spec/product files
- **Heading format** (`### FR-016.001: Title`) in backlog issues

Parser extracts `id`, `title`, `kind`, `category`, `tags`. Supplemental breakout metadata (`ext_id`, `ext_url`, additional tags) is merged from `spec/requirements/*.md` files. No structured `description` or `acceptance_criteria` at source.

Lifecycle status defaults to `pending`. Coverage-derived updates apply to non-terminal states only — terminal states (`retired`, `deprecated`, `superseded`) are sticky. Parser is warning-based and lossy: formatting variations produce warnings but can still drop requirements.

### Drivers

- DE-136 program (ADR-010 block-ification across artifact kinds)
- DE-129 exposed parser fragility and lifecycle enum gaps
- ISSUE-051 demonstrated regex brittleness in sync

### Constraints

- **Block pattern**: must follow established marker → dataclass → extract/render → metadata → schema registry → WorkspaceValidator wiring
- **Migration isolation** (DEC-138-12): migration steps import only stdlib + `_helpers` + `_protocol` + pyyaml
- **POL-001**: block-first with regex fallback. When block is present, prose requirements are ignored deterministically — no merge, no error, no warning. Regex fallback only for specs with no block.
- **POL-003**: block module in `blocks/`, metadata in `blocks/`, validation wiring in `validation/validator.py`. Domain imports infrastructure, not vice versa.
- **Lifecycle semantics**: requirement lifecycle is normative requirement state (`pending`, `in-progress`, `active`, `retired`, `deprecated`, `superseded`). Coverage status is observational evidence in verification.coverage blocks. Coverage may derive non-terminal lifecycle updates during sync; terminal states are never overwritten by coverage.
- **Breakout metadata files** (`spec/requirements/*.md`) remain a supplemental merge source — not replaced by blocks.
- **Coexistence**: migrated files may temporarily contain both block and legacy prose. Prose is ignored once block exists; prose excision is out of scope.

### Out of Scope

- Per-requirement files (DR-136 §8.5)
- Cross-spec requirement aggregation
- Prose excision (`--prune-duplicates`, deferred)
- Sync pruning changes (DE-129)
- Heading-form backlog requirements (DEC-140-08 — different artifact kind, not spec requirements)

## 3. Block Schema

### Marker & Constants

- **Marker**: `supekku:spec.requirements@v1`
- **Schema ID**: `supekku.spec.requirements`
- **Version**: `1`

### Example

```yaml supekku:spec.requirements@v1
schema: supekku.spec.requirements
version: 1
spec: PROD-004
requirements:
  - id: FR-001
    title: Metadata-driven validation
    lifecycle: active
    kind: functional
    category: core
    description: |
      The system must validate frontmatter against declarative metadata definitions.
    acceptance_criteria:
      - All registered kinds validate via metadata
      - Errors are clear and actionable
    tags: [validation, metadata]
  - id: NF-001
    title: Validation performance
    lifecycle: pending
    kind: non-functional
    description: |
      Validation of a single artifact must complete within 500ms.
    acceptance_criteria:
      - p99 latency under 500ms for single-artifact validation
    tags: [performance]
```

### Top-Level Fields

| Field | Type | Required | Constraints |
|-------|------|----------|-------------|
| `schema` | const | yes | `supekku.spec.requirements` |
| `version` | const | yes | `1` |
| `spec` | string | yes | Pattern: `^(SPEC\|PROD)-\d{3,}$`. Cross-validated against containing artifact ID by WorkspaceValidator. |
| `requirements` | array | yes | `min_items: 0`. Empty valid for new/zero-requirement specs. Post-strict-flip: every spec/prod artifact must carry this block. |

### Per-Requirement Entry Fields

| Field | Type | Required | Constraints |
|-------|------|----------|-------------|
| `id` | string | yes | Pattern: `^(FR\|NF)-\d{3}$`. Qualified UID derived as `{spec}.{id}` at parse time. |
| `title` | string | yes | Short human-readable title. |
| `lifecycle` | enum | yes | `pending`, `in-progress`, `active`, `retired`, `deprecated`, `superseded`. Default for new: `pending`. Maps to `RequirementRecord.status`. |
| `kind` | enum | yes | Canonical: `functional`, `non-functional`. Tolerated aliases via standard `ToleratedAlias`: `FR` → `functional`, `NF`/`NFR` → `non-functional`, `sunset_after: "DE-140"`. |
| `category` | string | no | Classification category. Replaces inline `(category)` prose syntax for block-bearing files. |
| `description` | string | yes | Requirement description. Empty `""` permitted pre-strict-flip only (migration/bootstrap). Post-flip: trimmed-empty rejected. |
| `acceptance_criteria` | array[string] | yes | Testable criteria. Empty `[]` permitted pre-strict-flip only. Post-flip: empty list and blank-only items rejected. |
| `tags` | array[string] | no | Classification tags. Default: `[]`. |

### Cross-Field Invariant

ID prefix must match kind after alias canonicalization:
- `FR-*` ↔ `functional`
- `NF-*` ↔ `non-functional`

Enforced in custom validation wrapper `validate_spec_requirements()`, not via `ConditionalRule` (DEC-140-10 — current metadata engine doesn't support regex-conditioned cross-field rules).

`validate_spec_requirements()` is THE canonical validation entry point (DEC-140-16). It composes: `SPEC_REQUIREMENTS_VALIDATOR.validate()` (schema) + cross-field invariant + duplicate ID check. Callers must not use the raw metadata validator alone.

### Dataclass

```python
@dataclass(frozen=True)
class SpecRequirementsBlock:
    raw_yaml: str
    data: dict[str, Any]
```

Cached `.data` shape per block-class taxonomy. `.data` is post-validation normalized: tolerated aliases canonicalized before downstream access.

### Design Notes

- `description` and `acceptance_criteria` are block-only — they do NOT map to `RequirementRecord`. Registry is a lean index; rich content stays at source. Consumers needing rich text read the source block directly.
- `category` in the block replaces inline `(category)` prose syntax. When block is present, prose is ignored entirely — no same-document prose parsing.
- Breakout metadata files remain a supplemental merge source at sync time for `ext_id`, `ext_url`, and additional tags. Orthogonal to the block.

## 5. Reading Pipeline

### Parser API

New public entrypoint in `requirements/parser.py`:

```python
def records_from_spec(
    spec_id: str,
    frontmatter: Mapping[str, Any],
    body: str,
    spec_path: Path,
    repo_root: Path,
    stats: SyncStats | None = None,
) -> Iterator[RequirementRecord]:
```

### Block-First Logic

1. Attempt `extract_spec_requirements(body)`:
   - Returns `None` → no block present, fall through to regex
   - Returns `SpecRequirementsBlock` → proceed with block path
   - Raises `ValueError` → malformed block (extraction failure)
     - Pre-strict-flip: log warning, fall back to regex
     - Post-strict-flip: log error, yield zero records (no fallback)

2. If block found, validate via `validate_spec_requirements()` (canonical entry point):
   - **Extraction failure** (ValueError): pre-flip → regex fallback; post-flip → error, zero records
   - **Schema-invalid but extractable** (e.g. missing required field, bad enum): block remains authoritative. Validation errors reported. Records built from valid entries only; invalid entries skipped with error.
   - **Warnings only** (tolerated aliases, etc.): block authoritative. Warnings reported. All records built normally.
   - Rule: once extraction succeeds, the block is authoritative regardless of validation outcome. Fallback only on extraction failure.

3. Build `RequirementRecord` per entry:
   - `uid` = `f"{spec_id}.{entry['id']}"`
   - `status` = `entry["lifecycle"]` (field name mapping)
   - `kind` = canonicalized from `entry["kind"]`
   - `category` = `entry.get("category")`
   - `source_kind` = `"block"`, `source_type` = `"spec"`

4. If no block: call existing `_records_from_content()` with `source_kind` = `"prose"`

### Invariants

- **Mutual exclusion**: block path and regex path never both execute for the same spec
- **Source tracking**: `source_kind` on RequirementRecord is diagnostic only — enables auditing, does not drive enforcement
- **Breakout merge**: `_load_breakout_metadata()` runs regardless of source
- **No new RequirementRecord fields**: `description`/`acceptance_criteria` are block-only

### Registry Integration

No changes to `sync()` orchestration logic. Registry calls `records_from_spec()` (new API). Parser handles source selection internally. Registry is source-agnostic. Both spec-ingestion paths in sync (via `spec_registry.all_specs()` and raw `spec_dirs`) must use the same `records_from_spec()` entrypoint and breakout merge behavior.

### Stale Pruning

Block-sourced extraction produces the same UID set format. Pruning compares UIDs source-agnostically. Requirements removed from blocks are pruned on next sync. Orphaned breakout files are explicitly tolerated — cleanup is out of scope.

## 6. Migration

### Location

`spec_driver/migrations/v0N_spec_requirements/migration.py`

Discovered by migration orchestrator via folder convention. Standalone interactive command, not a batch sweep. No watermark advancement.

### Isolation (DEC-138-12)

Imports: stdlib + `spec_driver.migrations._helpers` + `spec_driver.migrations._protocol` + pyyaml only.

Frozen-local copies of: block marker, schema/version constants, requirement regex patterns, lifecycle enum values, kind canonical values and alias map. Test verifies frozen patterns match runtime parser patterns.

### Command

```
spec-driver admin migrate-requirements <SPEC-ID>
spec-driver admin migrate-requirements <SPEC-ID> --dry-run
```

### Interactive Flow

1. **Guard**: refuse if spec already has a `spec.requirements` block. User must remove existing block manually to re-migrate.
2. **Parse**: extract requirements via frozen-local regex (same patterns as `parser.py`).
3. **Enrich**: for each parsed requirement, populate `description` and `acceptance_criteria`:
   - LLM-assisted: use surrounding prose context to generate content
   - Placeholder: `description: ""`, `acceptance_criteria: []`
4. **Present**: display each candidate requirement record as proposed block YAML.
5. **Review**: per-record interactive prompt:
   - `[a]ccept` — accept this record as-is
   - `[e]dit` — open YAML buffer in `$EDITOR`
   - `[s]kip` — abort migration for entire spec (atomic rule)
   - `[A]ccept-all-remaining` — accept remaining without review
   - `[q]uit` — abort migration entirely
6. **Write**: if all records accepted/edited:
   - Render block via frozen-local renderer
   - Insert above prose body (after frontmatter, before first heading)
   - Preserve existing prose (no excision)
   - Atomic write via `_helpers.atomic_write()`
7. **Validate**: post-write validation of emitted block. Failure = failed migration, atomic revert.
8. **Drift**: unparseable requirements → `DL-*` ledger entry in `.spec-driver/drift/`

### Atomicity

Per-spec, not per-record. ALL requirements must be in the emitted block or the file is not modified. `[s]kip` at any record aborts the entire spec.

### Dry-Run

Shows proposed block with placeholders for unenriched fields. Drift indicators inline. Preview only, no writes, no prompts.

### Drift Ledger

One `DL-*` ledger entry per spec migration in `.spec-driver/drift/`. Each entry contains:
- `spec_id`: migrated spec
- `kind`: drift kind (see table)
- `detail`: human-readable description
- `created`: ISO timestamp

ID allocated sequentially from existing drift ledger.

### Drift Kinds

| Kind | Description |
|------|-------------|
| `DRIFT_REQUIREMENT_UNPARSEABLE` | Regex couldn't extract requirement from prose line |
| `DRIFT_DESCRIPTION_PLACEHOLDER` | Description field is empty placeholder |
| `DRIFT_ACCEPTANCE_PLACEHOLDER` | Acceptance criteria field is empty placeholder |

### What Migration Does NOT Do

- Excise legacy prose
- Modify breakout metadata files
- Flip strict mode
- Migrate heading-form backlog requirements
- Batch-sweep all specs

## 7. Validation & Strict Flip

### WorkspaceValidator Wiring

New method `_validate_spec_requirements_blocks()`:

1. Iterate all specs via `self.workspace.specs.all_specs()`
2. Extract block via `extract_spec_requirements(body)`
3. If extraction raises `ValueError`: error
4. If block is `None`:
   - Pre-flip: no action
   - Post-flip: error ("missing spec.requirements block")
5. If block present:
   - Schema validation via `SPEC_REQUIREMENTS_VALIDATOR.validate(block.data, strict=..., accept_tolerated=...)`
   - Cross-validate `spec` field against artifact ID
   - Dispatch each `ValidationError` via `_block_issue()`

Validation sequence: extract → schema validate → semantic checks (spec mismatch, cross-field invariant).

### Strict Flip

Consistent with DE-139 pattern via schema_version bump:

| Aspect | Pre-Flip | Post-Flip |
|--------|----------|-----------|
| Missing block | No action | Error |
| Invalid block (extraction failure) | Warning, parser falls back to regex | Error, no fallback |
| Validation warnings | Reported | Reported |
| Empty `description` / `acceptance_criteria` | Permitted | Error (trimmed-empty, blank items) |
| Empty `requirements: []` | Valid | Valid (zero-requirement specs) |

Every spec/prod artifact must carry a block post-flip. Empty `requirements: []` is valid for zero-requirement specs.

### Strict Flip Operational Guard

The schema_version bump command must verify all spec/prod artifacts have a `spec.requirements` block before allowing the bump. If any spec/prod file lacks a block, the bump is refused with a list of unmigrated files. This prevents the post-flip parser (zero records, no fallback) from silently pruning requirements from unmigrated specs during sync.

### Duplicate Detection

- **Duplicate `id` entries** within one `requirements` array: hard validation error.
- **Multiple `spec.requirements@v1` blocks** in one file: hard validation error at extraction (return first, error on second).

### What Validation Does NOT Check

- Prose/block consistency (prose ignored when block present)
- Breakout metadata alignment (supplemental, orthogonal)
- Requirement coverage completeness (verification.coverage block's responsibility)
- Orphaned breakout files (explicitly tolerated, deliberately silent — no diagnostic emitted)

## 9. Design Decisions

### DEC-140-01: Use existing lifecycle enum

Existing requirement lifecycle values (`pending`, `in-progress`, `active`, `retired`, `deprecated`, `superseded`). DR-136 §8.1 proposed adding `baseline`/`verified` — rejected because it blurs the boundary between requirement lifecycle (normative state) and coverage status (observational evidence). Neither value is load-bearing anywhere in the codebase or DR-136 program.

### DEC-140-02: Full block schema

Block includes `description` and `acceptance_criteria` as required fields. Richer machine-readable representation than regex could extract. Migration handles via LLM assist or placeholders.

### DEC-140-03: Block-first, regex fallback, never both

When block is present, prose is ignored deterministically. No merge, no warning, no error on legacy prose coexisting with block. Regex fallback only for specs without blocks. Prose excision deferred.

### DEC-140-04: Strict flip via schema_version

Consistent with DE-139 pattern. Pre-flip: blocks optional, fallback allowed. Post-flip: blocks required, no fallback. Empty `requirements: []` valid for zero-requirement specs.

### DEC-140-05: Interactive migration in migration folder

Per-spec interactive flow with accept/edit/skip/accept-all/quit. Lives in `spec_driver/migrations/` per folder convention. Not a batch sweep — standalone interactive command. No watermark.

### DEC-140-06: kind field — canonical values with tolerated aliases

Canonical: `functional`, `non-functional`. Tolerated: `FR`, `NF`, `NFR` via standard `ToleratedAlias` machinery with `sunset_after: "DE-140"`.

### DEC-140-07: category as optional block field

Category added to block schema as optional string. Replaces inline `(category)` prose syntax for block-bearing files. Block must be self-contained — no same-document prose parsing when block present.

### DEC-140-08: Heading-form backlog requirements excluded

`### FR-016.001: Title` format lives in `.spec-driver/backlog/issues/`, not spec/product files. Different artifact kind. Regex parser continues handling them. Separate block type if needed in future delta.

### DEC-140-09: Breakout metadata files remain supplemental

`spec/requirements/*.md` files continue providing `ext_id`, `ext_url`, additional tags via sync-time merge. Not replaced by blocks. Orphaned breakout files explicitly tolerated.

### DEC-140-10: Cross-field invariant in custom wrapper

ID prefix ↔ kind enforced in `validate_spec_requirements()` wrapper, not via `ConditionalRule`. Current metadata engine doesn't support regex-conditioned cross-field validation.

### DEC-140-11: Migration atomic per-spec

All requirements in block or file untouched. No partial blocks. `[s]kip` aborts entire spec.

### DEC-140-12: Post-write validation mandatory

Migration validates emitted block immediately after write. Failure = failed migration, atomic revert. Prevents pre-flip regex fallback from masking migration defects.

### DEC-140-13: Strict flip operationally gated

Schema_version bump must verify all spec/prod artifacts have blocks before allowing the bump. Prevents post-flip parser from silently pruning unmigrated requirements.

### DEC-140-14: Template emits empty requirements block

New specs get `requirements: []`, not a sample `FR-001`. Sample canonical requirements would become real registry entries if left unedited — an implementer trap. Empty block is safe post-flip (zero-requirement specs allowed).

### DEC-140-15: Duplicate IDs and duplicate blocks are hard errors

Duplicate `id` entries within one `requirements` array: validation error. Multiple `spec.requirements@v1` blocks in one file: extraction error. Both prevent ambiguous UID ownership.

### DEC-140-16: validate_spec_requirements() is canonical validation surface

The wrapper composes schema validation + cross-field invariant + duplicate ID check. Callers must use the wrapper, not the raw metadata validator, to get complete validation.
