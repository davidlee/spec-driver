# DR-141 – Audit artefact metadata propagation

## 1. Executive Summary

- **Delta**: [DE-141](./DE-141.md)
- **Status**: draft
- **Last Updated**: 2026-05-28
- **Synopsis**: Move audit `findings` from frontmatter array to `supekku:audit.findings@v1` block, tighten strict enforcement, enrich `list audits` columns, and provide migration step — following DE-138 per-artefact precedent.

## 2. Problem & Constraints

- **Current Behaviour**: Audit findings live as a rich FM array (`findings: [{id, description, outcome, disposition, ...}]`). `list audits` shows generic change columns (ID, Name, Tags, Status) with no audit-specific detail. Outcome/disposition validation is warn-level, not error.
- **Drivers**: DR-136 §9 specifies exact placement table, strict enforcement rules, list enrichment columns, and migration shape. F-C (audit_window optional) and F-D (per-finding outcome only) inherited from DE-136 notes.
- **Constraints**:
  - DEC-138-12: migration modules import only stdlib + `spec_driver.migrations._helpers` + `_protocol` + yaml. No `supekku.*` imports.
  - POL-003: formatters must not contain business logic; domain functions compute summaries.
  - ADR-008: lifecycle-truth doctrine — audit findings represent observed evidence, not normative state.
- **Out of Scope**: Disposition matrix redesign. New audit modes. Body-prose excision of legacy Findings sections. Audit lifecycle changes.

## 3. Block Reshape — findings FM → block

### 3.1 Block schema: `supekku:audit.findings@v1`

```yaml
schema: supekku.audit.findings
version: 1
audit: AUD-027
findings:
  - id: F-001
    description: "Delta frontmatter uses deprecated 'lifecycle' key"
    outcome: drift
    linked_issue: ISSUE-055
    linked_delta: ""
    disposition:
      status: reconciled
      kind: spec_patch
      refs:
        - kind: revision
          ref: RE-042
      rationale: "Resolved via RE-042 spec patch"
```

Schema shape identical to current FM array. Fields per finding:

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `id` | string | yes | Finding ID (e.g. `F-001`) |
| `description` | string | yes | |
| `outcome` | enum | yes | `aligned \| drift \| risk` — strict |
| `linked_issue` | string | no | |
| `linked_delta` | string | no | |
| `disposition` | object | no | Required for completed audits under strict |

Disposition sub-schema (unchanged from current `audit.py`):

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `status` | enum | yes | `reconciled \| accepted \| pending` |
| `kind` | enum | yes | `aligned \| spec_patch \| revision \| follow_up_delta \| follow_up_backlog \| tolerated_drift` |
| `refs` | array | no | `[{kind, ref}]` |
| `drift_refs` | array | no | `[{kind, ref}]` |
| `rationale` | string | no | |
| `closure_override` | object | no | `{effect: warn\|none, rationale: string}` — both required when present |

### 3.2 Block rendering and parsing

New module: `supekku/scripts/lib/blocks/audit_findings.py`

- `render_audit_findings_block(audit_id, findings) -> str` — renders code-fenced block
- `extract_audit_findings(body) -> dict | None` — parses block from body text
- Register `supekku:audit.findings@v1` in block schema registry (follows `spec.requirements` and `delta.relationships` pattern)
- Follows existing pattern from `blocks/spec_requirements.py` and `blocks/delta_relationships.py`

### 3.3 FM changes

Remove `findings` from `AUDIT_FRONTMATTER_METADATA` field list. The metadata class retains `mode`, `delta_ref`, `spec_refs`, `prod_refs`, `code_scope`, `audit_window` as FM fields.

**Phase ordering constraint**: FM `findings` field cut MUST happen after migration sweep. During the transition window (block parser deployed, migration not yet applied to all audits), callers use dual-path reads: check block first, fall back to FM. This applies to `collect_gating_findings` and `audit_findings_summary`.

**DEC-141-03**: `audit_window` stays optional (F-C). No change from current metadata — confirming intent.

**DEC-141-04**: Per-finding `outcome` is the only outcome surface (F-D). No artefact-level `outcome` field. AUD-012's `outcome: pass` is a per-finding value that becomes a strict-mode error after migration.

## 4. Strict Enforcement

Tightening already-coded validation in `audit.py` / `validator.py`:

| Rule | Current | Target |
|------|---------|--------|
| Invalid `outcome` enum (per-finding) | warn | **error** under strict |
| Disposition `status × kind` invalid pair | warn | **error** under strict |
| Disposition `outcome × kind` invalid pair | warn | **error** under strict |
| Completed audit + undisposed finding | pass | **error** under strict |
| `closure_override.effect` ≥ derived gate | unchecked | **error** under strict |
| `closure_override` present without `rationale` | unchecked | **error** under strict |

Implementation: gate these checks on `self.strict` in `WorkspaceValidator._validate_audit_blocks()` (new method, or extend existing audit validation path).

### 5.2 Findings summary — domain layer (DEC-141-01)

New in `changes/audit_check.py`:

```python
@dataclass(frozen=True)
class AuditFindingsSummary:
  total: int
  aligned: int
  drift: int
  risk: int
  disposed: int

  def findings_cell(self) -> str:
    """DR-136 §9.4: '7 (3a/2d/2r)' or '–'."""
    if not self.total:
      return "–"
    return f"{self.total} ({self.aligned}a/{self.drift}d/{self.risk}r)"

  def disposed_cell(self) -> str:
    """DR-136 §9.4: '5/7' or '–'."""
    if not self.total:
      return "–"
    return f"{self.disposed}/{self.total}"
```

### 5.4 Formatter

```python
def format_audit_list_row(
  artifact: ChangeArtifact,
  summary: AuditFindingsSummary,
) -> dict[str, str]:
  return {
    "id": artifact.id,
    "name": _format_display_name(artifact),
    "status": artifact.status,
    "mode": _format_mode_glyph(artifact),    # C or D
    "delta_ref": _format_delta_ref(artifact), # truncated

    "disposed": summary.disposed_cell(),
  }
```

## 6. Migration Step

### 6.1 Module: `spec_driver/migrations/v0_10_0_004_audit_findings/`

**DEC-141-05**: Follows `v0_10_0_003_prod_blocks`. Version key: `v0_10_0_004_audit_findings`.

Structure (DE-138 precedent):
```
v0_10_0_004_audit_findings/
├── __init__.py          # AuditFindingsStep registration
├── _emitters.py         # render_audit_findings_block (frozen-local copy)
├── migration.py         # AuditFindingsStep(BaseMigrationStep)
└── migration_test.py
```

### 6.2 Migration logic

1. **§4 universal cuts**: Remove `lifecycle`, `aliases`, `auditers`, `source` from FM (if present)
2. **Extract findings**: Read `findings` array from FM YAML
3. **Render block**: `render_audit_findings_block(audit_id, findings)` — code-fenced `supekku:audit.findings@v1`
4. **Insert block**: After frontmatter, before first heading (same insertion pattern as DE-138)
5. **Remove FM key**: Delete `findings` from frontmatter
6. **Drift entries**: Invalid `outcome` values (e.g. `pass`) → drift entry, NOT auto-fixed
7. **Post-write validation**: Re-parse written file, verify block present and parseable

### 6.3 Strict flip sequence

Per DR-136 §11.3:
1. `admin migrate --kind audit --dry-run` — preview
2. `admin migrate --kind audit` — apply
3. `validate workspace --kind audit --strict` — verify
4. Manual disposition of AUD-012 etc (drift entries guide)
5. Flip `[validation.strict].audit = true` in `workflow.toml`

## 7. Body Section Reshape

Per DR-136 §9.2: Keep Observations, Recommendations, optional Evidence sections. Delete prose Findings section (block is canonical). Migration step does NOT auto-delete body prose — `validate --prune-duplicates` (future, out of scope) handles that. Migration only inserts block and cuts FM key.

## 10. Design Decisions

- **DEC-141-01**: Findings summary computation lives in domain (`audit_check.py`). Formatter receives pre-computed `AuditFindingsSummary`; CLI does thin wiring only.
- **DEC-141-02**: `_collect_audited_delta_ids` relocated from `cli/list/deltas.py` to `changes/audit_check.py`. Same logic, proper layer.
- **DEC-141-03**: `audit_window` stays optional (F-C resolution). Current corpus has audits without it (AUD-024). No strictness change.
- **DEC-141-04**: Per-finding `outcome` is the only outcome surface (F-D resolution). No artefact-level outcome rollup. `outcome: pass` on AUD-012 is a per-finding error.
- **DEC-141-05**: Migration step numbered `v0_10_0_004` — follows `v0_10_0_003_prod_blocks` in sequence.
