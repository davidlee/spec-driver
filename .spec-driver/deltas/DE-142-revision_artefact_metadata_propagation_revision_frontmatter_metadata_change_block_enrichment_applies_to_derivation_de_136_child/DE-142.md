## 3. Scope & Objectives

- **Primary Outcomes** (DR-136 §10):
  1. **Field placement** (§10.1): complete the DE-137 `REVISION_FRONTMATTER_METADATA` stub (Base 7 + relations + tags + ext_id/ext_url); §4 universals cut; `source_specs`/`destination_specs`/`requirements` → derived `applies_to` from `supekku:revision.change@v1` block.
  2. **Block schema enrichment** (§10.2, additive per F-F): the `action` enum (`introduce | modify | move | retire`) **already ships** with DE-118; DE-142 adds only the conditional `action × field-presence` rules (extend `FieldMetadata.conditional_rules`, applied in `_validate_object`). Intent vs observed (ADR-008): block represents revision *intent*; reconciliation against actual spec state happens at revision close.
  3. **Body reshape** (§10.3): delete §§1 (Source/Destination Specs) and 3 (Requirements list); keep Rationale, Migration/Patch.
  4. **`list revisions` enrichment** (§10.4): default columns ID, Name, Status, Source (count + first), Destination (count + first), Requirements (count).
  5. **Migration** (§10.5): `migrations/v06_revision_metadata.py` — §4 cuts; if FM has source_specs/destination_specs/requirements but no block, synthesise block with `action: move` + `from_lifecycle`/`to_lifecycle: unknown` (tolerated_alias) + drift entry per record; if both FM and block present, block wins, FM keys removed; body §§1, 3 cleanup deferred to `--prune-duplicates`.
- **Operational Constraints**:
  - DE-118 must have closed (provides `REVISION_CHANGE_METADATA` foundation; this delta extends with action enum).
  - DE-137 closed (orchestrator + state).
  - All four prior per-artefact deltas (DE-138/139/140/141) closed — full inherited pattern.
  - Strict-mode flip happens after sweep produces clean validate or all errors are drift-tracked.
- **Dependencies**:
  - **Hard**: DE-118, DE-137, DE-138, DE-139, DE-140, DE-141.

## 5. Approach Overview

_Refined by DR-142 (authored 2026-05-29). Block shape is **additive only** over
DE-118 — DR-136 §10.2's flat sketch was reconciled to the shipped rich schema
via in-place erratum (see DE-136 phase-03 §9)._

- **System Touchpoints** (DR-142 §10 code-impact table):
  - `core/frontmatter_metadata/revision.py` — **complete** the DE-137 stub (Base 7 + relations + tags + ext_id/ext_url); not "new from scratch".
  - `blocks/metadata/schema.py` + `blocks/metadata/validator.py` — additive engine extension: `conditional_rules` on `FieldMetadata`, applied in `_validate_object`.
  - `blocks/revision_metadata.py` — declare `conditional_rules` on `requirements[]` item (action × field-presence). Block payload **unchanged**.
  - `changes/artifacts.py` — revision-block `applies_to` deriver (DEC-138-10 pattern); cut FM scope keys.
  - `changes/revision_check.py` (NEW) — `RevisionChangeSummary` domain.
  - `formatters/{column_defs,change_formatters}.py` — `REVISION_COLUMNS` + formatter.
  - `cli/list/reviews.py` — `list revisions` wiring (same file as audits).
  - `spec_driver/migrations/v0_10_0_005_revision_metadata/` (NEW) — not "v06".
  - `templates/revision.md`.
- **Key Changes**: per the five outcomes in §3, with the F-F correction that the
  block `action` enum already exists (DE-118); DE-142's residual block work is
  the *conditional rules* only.
- **Migration / Rollout Notes**: standard sweep per DR-136 §11.2; revision corpus
  is auto-generated via `complete delta` — most legacy revisions follow a
  predictable shape; expect low drift. New conditional rules may surface legacy
  `action: move` entries missing `origin` → drift-track at sweep (DR-142 §13.4).
