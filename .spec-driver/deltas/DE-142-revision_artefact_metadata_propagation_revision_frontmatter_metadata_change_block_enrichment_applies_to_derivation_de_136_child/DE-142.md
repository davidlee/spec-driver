## 3. Scope & Objectives

- **Primary Outcomes** (DR-136 §10):
  1. **Field placement** (§10.1): hand-rolled FM → new `REVISION_FRONTMATTER_METADATA` (Base 7 + relations + tags + ext_id/ext_url); §4 universals cut; `source_specs`/`destination_specs`/`requirements` → derived `applies_to` from `supekku:revision.change@v1` block.
  2. **Block schema enrichment** (§10.2): `supekku:revision.change@v1` enriched with strict `action` enum (`introduce | retire | move | modify`) and conditional `action × field-presence` rules via `MetadataValidator`. Intent vs observed (ADR-008): block represents revision *intent*; reconciliation against actual spec state happens at revision close (audit step).
  3. **Body reshape** (§10.3): delete §§1 (Source/Destination Specs) and 3 (Requirements list); keep Rationale, Migration/Patch.
  4. **`list revisions` enrichment** (§10.4): default columns ID, Name, Status, Source (count + first), Destination (count + first), Requirements (count).
  5. **Migration** (§10.5): `migrations/v06_revision_metadata.py` — §4 cuts; if FM has source_specs/destination_specs/requirements but no block, synthesise block with `action: move` + `from_lifecycle`/`to_lifecycle: unknown` (tolerated_alias) + drift entry per record; if both FM and block present, block wins, FM keys removed; body §§1, 3 cleanup deferred to `--prune-duplicates`.
- **Operational Constraints**:
  - DE-118 must have closed (provides `REVISION_CHANGE_METADATA` foundation; this delta extends with action enum).
  - DE-137 closed (orchestrator + state).
  - All four prior per-artefact deltas (DE-138/139/140/141) closed — full inherited pattern.
  - Strict-mode flip happens after sweep produces clean validate or all errors are drift-tracked.
- **Dependencies**:
  - **Hard**: DE-118, DE-137, DE-138, DE-139, DE-140, DE-141.

## 7. Risks & Mitigations

- **Risk**: `action × field-presence` conditional rules subtle to express in `MetadataValidator` — _Likelihood_: medium – _Impact_: medium – _Mitigation_: DR-136 §10.2 specifies the conditional matrix; DE-118 already lands the `MetadataValidator` foundation; extend with conditional rule type if not already present.
- **Risk**: Synthesising legacy revision blocks with `unknown` lifecycles produces opaque change history — _Likelihood_: medium – _Impact_: low – _Mitigation_: drift entry per synthesised record; tolerated_alias `unknown`; DR-136 §10.5 explicitly accepts this for legacy revisions.
- **Risk**: Intent vs observed (ADR-008) confusion — strict-validating `from_lifecycle`/`to_lifecycle` at write time may suggest agreement with actual state — _Likelihood_: low – _Impact_: medium – _Mitigation_: DR-136 §10.2 explicitly distinguishes intent-validation from spec-state reconciliation (revision close audit step).
- **Risk**: Five hard dependencies (DE-118/137/138/139/140/141) push this delta to the end of program — schedule risk — _Likelihood_: medium – _Impact_: low – _Mitigation_: by design (DEC-004); benefits from accumulated precedent; if blockers persist, revision delta can be reduced in scope (defer enrichment to follow-up) without breaking umbrella close as long as `REVISION_FRONTMATTER_METADATA` lands.
