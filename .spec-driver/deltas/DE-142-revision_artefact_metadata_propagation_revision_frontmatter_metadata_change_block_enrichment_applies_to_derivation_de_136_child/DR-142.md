# DR-142 тАУ Revision artefact metadata propagation

## 1. Executive Summary

- **Delta**: [DE-142](./DE-142.md)
- **Status**: draft
- **Last Updated**: 2026-05-29
- **Synopsis**: Complete the revision frontmatter metadata class, enforce the
  `supekku:revision.change@v1` block's `action ├Ч field-presence` rules
  declaratively, derive `applies_to` from the block (cutting hand-rolled FM
  keys), enrich `list revisions`, and ship the `v0_10_0_005` migration. Last
  per-artefact child under DE-136; inherits the DE-138/DE-141 precedent. The
  block schema is **not** redefined тАФ DE-142 is additive over DE-118 (F-F).

## 2. Problem & Constraints

- **Current Behaviour**:
  - `REVISION_FRONTMATTER_METADATA` is a DE-137 stub (Base + status enum only;
    docstring: "full schema is DE-142 scope"). Real revisions still carry
    hand-rolled FM keys (`source_specs`, `destination_specs`, `requirements`,
    `aliases`).
  - `supekku:revision.change@v1` (shipped by DE-118) already has the
    requirement-level `action` enum (`introduce | modify | move | retire`) and
    rich `specs`/`requirements` payloads, validated by `MetadataValidator`. But
    the `action ├Ч field-presence` conditionality is **prose-only** in field
    descriptions тАФ not enforced. `ConditionalRule` exists but only traverses
    top-level dot-paths; it cannot reach `requirements[].action`.
  - `list revisions` shows generic change columns; no source/destination/
    requirement detail.
- **Drivers**: DR-136 ┬з10 (reconciled placement table); F-F scope-note
  (DE-136 `notes.md`): "additive enrichment, not redefinition" of the DE-118
  block. PROD-004 FR-001/FR-002.
- **Constraints / Guardrails**:
  - **F-F (load-bearing)**: extend, do not redefine, `REVISION_CHANGE_METADATA`.
    DR-136 ┬з10.2's earlier *flat* `changes:` sketch was a presentation defect;
    corrected in place via erratum (2026-05-29). The shipped rich schema is
    canonical. **No `@v2`, no corpus block re-shape.**
  - **DEC-138-10 (no competing truths)**: `applies_to` is derived from the
    block at load, never persisted in FM. Storing it alongside the block is a
    strict-mode error.
  - **DE-118 ┬з11.1-equivalent boundary** (DR-136 ┬з11.1): the metadata engine is
    application code (current-schema only); migration code is legacy-aware and
    lives in `spec_driver/migrations/`, importing only stdlib + migration
    helpers. The engine extension here (`FieldMetadata.conditional_rules`) is
    additive application code, used by all blocks.
  - POL-003: formatters carry no business logic; domain computes summaries.
  - ADR-008: the block is revision *intent*; reconciliation against observed
    spec state is a revision-close concern, not this delta's.
- **Out of Scope**:
  - Block top-level reshape (no `@v2`).
  - Spec-state reconciliation at revision close.
  - Body-prose excision (`validate --prune-duplicates`).
  - Migrating the `blocks/revision.py` line-by-line extractor (DE-118 retains it
    for nested fences).

## 3. Architecture Intent

- **Target Outcomes**:
  1. Revision FM is governed by a complete `REVISION_FRONTMATTER_METADATA`
     equal in rigour to delta/audit (FR-001/FR-002).
  2. The change block's `action ├Ч field-presence` invariant is **declared**, not
     prose, and enforced under strict.
  3. `applies_to` is the single derived view of spec/requirement scope; FM stops
     carrying competing copies.
  4. `list revisions` surfaces scope at a glance.
  5. Legacy revisions migrate cleanly or are drift-tracked; strict flips.
- **Guiding Principles**: additive over DE-118; declarative metadata over
  hand-rolled validation; derive-don't-duplicate; reuse the DE-141 per-artefact
  shape (block тЖТ domain summary тЖТ formatter column тЖТ migration тЖТ strict flip).
- **Lifecycle Impact**: `[validation.strict].revision = true` at close. No
  change to revision status enum (inherited from change-status via DE-137).

## 4. Block conditional-rule enforcement (additive тАФ DEC-142-02)

### 4.1 Engine extension

`ConditionalRule` (top-level only today) cannot express per-item rules because
`_validate_conditional_rules` walks `data` via dot-paths and never iterates
arrays. Extend the schema so any **object** can carry its own rules:

```python
# schema.py тАФ FieldMetadata gains (additive, default empty):
conditional_rules: list[ConditionalRule] = field(default_factory=list)
```

```python
# validator.py тАФ _validate_object, after property validation:
if field_meta.conditional_rules:
  errors.extend(self._apply_conditional_rules(value, field_meta.conditional_rules, field_path))
```

`_apply_conditional_rules(obj, rules, path)` is the existing
`_validate_conditional_rules` body, generalised to take an explicit object +
rule list + path prefix (so item errors read `requirements[2].origin`). The
top-level `validate()` call delegates to the same helper with
`self.metadata.conditional_rules` and `path=""` тАФ **no behaviour change** for
blocks that declare none (VT-142-ENGINE-003).

Because array items dispatch `_validate_field тЖТ _validate_object`, declaring
rules on the `requirements[]` *item* `FieldMetadata` makes them fire per element
with `action` resolved as a direct key of each item (`_get_nested_value(item,
"action")`).

### 4.2 Declared rules (revision_metadata.py)

On the `requirements[]` item object `FieldMetadata`:

```python
conditional_rules=[
  ConditionalRule("action", "move",      ["origin", "destination"],
    description="move requires both origin and destination"),
  ConditionalRule("action", "introduce", ["destination"],
    description="introduce requires destination"),
  ConditionalRule("action", "modify",    ["destination"],
    description="modify requires destination"),
]
```

`retire` adds no requirement (DEC-142-03). `specs[]` `action` stays
unconstrained тАФ ┬з10.2 intent is requirement-level.

**Intent vs observed (ADR-008)**: these enforce well-formed *intent*
(structurally complete change descriptions), not agreement with current spec
state.

## 6. `applies_to` derivation (DEC-142-05)

Mirror `_derive_applies_to` (DR-138 ┬з6.1 / DEC-138-10). Add a revision-block
deriver in `artifacts.py`, hooked into `load_change_artifact` for
`kind == "revision"`:

- `specs` тЖР unique `block.specs[].spec_id`
- `requirements` тЖР unique `block.requirements[].requirement_id`
- destination/source distinction (for the list columns) read from
  `requirements[].destination.spec` vs `requirements[].origin[].ref` тАФ computed
  in the domain summary (┬з7), not stored.

`applies_to` is **runtime-only**. If FM contains `applies_to` (or the cut
`source_specs`/etc.) alongside a block тЖТ strict-mode error (VT-142-DERIVE-002),
matching DEC-138-10.

## 8. Migration тАФ `v0_10_0_005_revision_metadata` (DEC-142-07)

Structure follows `v0_10_0_004_audit_findings` (DE-141 ┬з6). Logic:

1. **┬з4 universal cuts**: remove `lifecycle`, `aliases`, `auditers`, `source`.
2. **Hand-rolled cuts**: remove `source_specs`, `destination_specs`,
   `requirements` from FM.
3. **Idempotence**: block present + FM keys already cut тЖТ skip.
4. **Block synthesis (FM-only)**: FM has `source_specs`/`destination_specs`/
   `requirements` but no block тЖТ synthesise the **rich** block: one
   `requirements[]` entry per FM requirement with `action: move`, `origin` from
   `source_specs`, `destination.spec` from `destination_specs`,
   `lifecycle.status: unknown` (tolerated_alias); `specs[]` from the union.
   Emit a drift entry per synthesised record
   (`.spec-driver/drift/DL-NNN-de_142_migrate_<id>.md`).
5. **Both present**: block wins; cut FM keys; no drift.
6. **Post-write validation**: re-parse, confirm block parses and `metadata.revision`
   matches artifact id.
7. Body ┬з┬з1/3 prose cleanup deferred to `validate --prune-duplicates`.

**`unknown` tolerated_alias**: added to `REQUIREMENT_STATUSES` lifecycle enum
metadata as a time-bounded `tolerated_alias` sunset-referenced to DE-142, so
synthesised records load until manually reconciled.

### 8.1 Strict flip sequence (DR-136 ┬з11.2)

```
spec-driver validate --kind revision                          # tolerant baseline
spec-driver admin migrate revision --dry-run                  # preview
spec-driver admin migrate revision                            # apply
spec-driver validate --kind revision --strict                 # re-validate
# manual disposition of residual drift
spec-driver validate --kind revision --strict --no-tolerated-aliases  # final gate
# then: [validation.strict].revision = true in workflow.toml
```

## 13. Adversarial review residuals & risks

Surfaced by self-review; resolved at IP/phase granularity, recorded here so they
are not lost in polish.

1. **Engine refactor must be behaviour-preserving.** No production block declares
   top-level `conditional_rules` today (only `test_engine.py`). Generalising
   `_validate_conditional_rules` into `_apply_conditional_rules(obj, rules,
   path)` must keep the top-level call (`path=""`) and ensure the error path for
   `path=""` yields `origin`, not `.origin` (no leading dot). VT-142-ENGINE-003
   guards "no behaviour change". Engine tests required.
2. **`applies_to.specs` composition.** Derived `applies_to.specs` = union of
   `specs[].spec_id` тИк `requirements[].destination.spec` тИк
   `requirements[].origin[].ref` (kind=spec). The list-column source/destination
   *split* is recomputed in the domain summary (┬з7.1), not read back from
   `applies_to`. IP to finalise dedup/order.
3. **`unknown` tolerated_alias blast radius.** It is added to the *shared*
   `REQUIREMENT_STATUSES` lifecycle enum, so during the window any
   requirement-lifecycle field accepts `unknown`, not just synthesised revision
   blocks. Acceptable per DR-136 ┬з10.5, but the alias MUST carry a sunset
   referencing DE-142 and be removed once synthesised records are reconciled.
4. **Legacy `action: move` without `origin`.** The new conditional rules turn
   any existing hand-authored `move` entry lacking `origin`/`destination` into a
   strict error at flip. Expected and desirable (drift), but the ┬з8.1 sweep must
   surface and drift-track these before the flip тАФ do not assume the corpus is
   clean. VA-142-CORPUS-001 is the gate.
5. **FM-beside-block strict check is new for revision.** Rejecting FM
   `applies_to`/`source_specs`/etc. alongside a block (VT-142-DERIVE-002) reuses
   the delta check path (DEC-138-10); confirm it generalises to `kind:revision`
   rather than being delta-keyed.
