## 1. Summary & Context

- **Product Spec**: PROD-004 — metadata schema consolidation
- **Technical Spec**: SPEC-114 — validation subsystem
- **Implementation Plan**: [IP-143](./IP-143.md)
- **Change Drivers**: DE-140 completed without migrating existing corpus; DE-136 P03 task 3.5 (`[validation.strict].requirement`) blocked until migration complete

## 2. Motivation

DE-140 built `admin migrate-requirements` (per-spec) and `admin strict-flip-requirements` (operational guard). The guard blocks the strict flip until all specs have `supekku:spec.requirements@v1` blocks. 70 specs/prods lack them. This delta runs the migration at scale and flips the switch.

## 3. Scope & Objectives

- **Primary Outcomes**:
  - All 70 unmigrated specs/prods have `supekku:spec.requirements@v1` blocks (PROD-004.FR-002)
  - `admin strict-flip-requirements` passes and writes config
  - DE-136 P03 task 3.5 unblocked
- **Dependencies**: DE-140 (completed — tooling exists)

## 4. Out of Scope

- Changes to the migration tool itself (that's DE-140's domain)
- Spec content changes beyond adding the requirements block
- Any new code — this is a corpus migration delta

## 5. Approach Overview

- **Key Changes**:
  - Batch `admin migrate-requirements <ID>` across all 70 unmigrated specs
  - Validate results per spec
  - Run `admin strict-flip-requirements` to write `[validation.strict_requirements]` config
- **Migration Notes**: `--dry-run` first pass for review, then apply

## 6. Verification Strategy

- **Acceptance Criteria**:
  - `admin strict-flip-requirements --dry-run` exits 0
  - `validate workspace --strict` shows no new error class vs pre-migration baseline
  - All specs parse requirements via block path (no regex fallback)

## 7. Follow-ups & Tracking

- DE-136 P03 task 3.5 can be marked done after this delta closes
