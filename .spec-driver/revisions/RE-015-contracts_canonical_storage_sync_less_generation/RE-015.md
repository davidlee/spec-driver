## 1. Context

-- **Why**: PROD-014 delivered a mirror-of-source contracts index quickly, but it intentionally left several major product needs as follow-ups:

- Reduce diff churn by inverting the relationship (store canonical contract files under `.contracts/`, with other views as symlinks).
- Enable generating contracts for a legacy repo without requiring pre-existing specs / prior sync runs.
- Improve robustness where index builders depend on adapter filename conventions (warn when conventions drift).
- **Clarification**: `.contracts/` is canonical storage, but it is still _derived_. Contract generation is deterministic and can be trivially regenerated from source code at any time.

## 3. Summary of Changes

- **Specs Updated**:
  - PROD-014: add v2 requirements to make `.contracts/` the canonical storage location for generated contract files and to support sync-less generation.

- **Requirements** (proposed additions to PROD-014):
  - **Canonical storage**:
    - System MUST generate contract artefacts directly into `.contracts/` as real files (not symlinks) as the canonical storage location.
    - System MUST keep the `.contracts/<view>/...` paths stable across evolutions of generation tooling.
    - System MUST treat `.contracts/` as derived output: it MUST be safe to delete and regenerate without losing unique information.
  - **Sync-less generation**:
    - System MUST support generating contracts for explicit targets without requiring pre-existing spec bundles or registry entries (e.g. “generate contracts for these 50 files”).
    - System SHOULD support generating contracts for “all files” of a language without first stubbing or syncing specs (bootstrapping for legacy codebases).
  - **Convention drift warnings**:
    - If a spec has a non-empty `contracts/` directory but the mirror builder produces zero entries for it, the system MUST warn (indicates filename convention mismatch or mapper bug).
    - If multiple identifiers map to the same SPEC (Python) the builder MUST deduplicate to avoid self-conflicts (documented as a regression risk).

- **Responsibilities / Relations**:
  - Keep `.contracts/` explicitly “generated / overwriteable” (aspirational/manual contracts are tracked elsewhere).

## 4. Consequences

- **Positive**:
  - Enables using `.contracts/` as the stable “contracts corpus” for agents without depending on SPEC directory structure.
  - Makes adoption in legacy repos far less messy: generate contracts first, then decide what deserves a spec.
- **Negative / Trade-offs**:
  - Stronger coupling to `.contracts/` as an interface; requires careful migration and compatibility shims.
  - If `.contracts/` is gitignored, the “contracts corpus” becomes ephemeral; workflows must assume regeneration is always available.
  - Sync-less generation needs clear semantics (what is the registry role, if any?) unless explicitly “no-spec” mode.
- **Impact on verification**:
  - Add/extend VT coverage to include convention-drift warnings and sync-less generation.

  - `.contracts/` rebuild can be “nuke-and-rebuild” for simplicity; selective cleanup is the future optimization path if NF-001 ever becomes tight.
