## 1. Context

- **Why**: Auto-stubbing unit specs on first `sync` creates sprawl in large legacy repos and blurs the line between unit specs and assembly specs (ADR-003). We want `sync` to be contracts-first by default and require explicit opt-in to spec auto-creation.
- **Source Specs**: PROD-012
- **Destination Specs**: PROD-012
- **Requirements Affected**: Updates/additions to PROD-012 FRs related to sync defaults, opt-in persistence, and explicit contract generation flags.

## 3. Summary of Changes

- **Specs Updated**:
  - PROD-012: revise sync UX and defaults to separate contract generation from spec creation.

- **Requirements** (proposed PROD-012 additions/changes):
  - **Defaults**:
    - `sync` MUST default to **not** auto-creating specs on a repo that has not explicitly opted in.
    - `sync` SHOULD default to generating contracts (unless explicitly disabled) since contracts are the primary discovery artefact.
  - **Persistence**:
    - Once a user opts into spec auto-creation (e.g. by running `sync --specs`), the tool MUST persist that choice in-repo and subsequent `sync` runs MUST default to the persisted behavior.
  - **Explicit contract control**:
    - `sync` MUST support `--contracts/--no-contracts` independent of spec creation flags.
  - **Marker/config strategy** (implementation direction; not user-facing by default):
    - Store persisted sync preferences under `.spec-driver/` (committed) to keep behavior stable across machines/agents.
    - v1 persistence SHOULD use a tiny presence file marker, e.g. `.spec-driver/enable_spec_autocreate`.
    - If/when more preferences are needed, migrate to a structured config file (e.g. `.spec-driver/config.yaml`).

This revision primarily revises intent for:

- `PROD-012.FR-005` (spec auto-creation becomes opt-in + persisted)
- `PROD-012.FR-006` (canonical storage moves to `.contracts/**` per PROD-014; legacy SPEC bundle storage becomes compatibility-only)

## 4. Consequences

- **Positive**:
  - Makes legacy adoption smoother: you can generate a contracts corpus without immediately creating hundreds/thousands of unit specs.
  - Aligns with unit vs assembly spec taxonomy: spec creation becomes an explicit act, not a side effect of “generate contracts”.
- **Negative / Trade-offs**:
  - Changes default behavior; needs clear CLI messaging (first-run hint: “run with --specs to stub specs and persist default”).
  - Requires a small amount of state/config in-repo.
- **Impact on verification**:
  - Add tests for first-run defaults, persistence marker behavior, and `--contracts/--no-contracts` interactions.
