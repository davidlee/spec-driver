## 1. Context

- **Why**: [`mem.brief.md`](./mem.brief.md) defines a strong memory model, but it does not fit spec-driver's current artifact frontmatter contract or verb-noun CLI taxonomy without normalization.
- **Source Specs**: none (new capability proposal)
- **Destination Specs**: PROD/SPEC to be created in DE-033
- **Requirements Affected**: to be introduced in DE-033 (memory schema, retrieval, lifecycle, CLI)

- **Input Brief**: [`mem.brief.md`](./mem.brief.md) (JAMMS v0.1 proposal)

  - ADR-003 (contracts and specs truth boundaries; avoid competing truths)

## 3. Summary of Changes

- **Fit Assessment (frontmatter)**:
  - Current base frontmatter expects `id`, `name`, `slug`, `kind`, `status`, `created`, `updated`.
  - JAMMS uses `title`, `type`, nested `time.created/updated`, `owner.contact`, and other fields not represented in current frontmatter metadata.
  - JAMMS status vocabulary (`active|draft|deprecated|superseded|obsolete`) diverges from common spec-driver lifecycle vocab.
- **Fit Assessment (CLI taxonomy)**:
  - JAMMS command examples are noun-first (`mem list`, `mem show`, `mem lint`, `mem verify`).
  - Spec-driver standard is verb-noun (`create spec`, `list specs`, `show delta`, `find revision`).
- **Normalization Direction**:
  - Introduce memory as a first-class artifact family (`memory`) with dedicated metadata and registry behavior.
  - Keep base frontmatter canonical; move JAMMS-specific semantics into dedicated fields and/or typed YAML blocks.
  - Translate commands to verb-noun equivalents such as `create memory`, `list memories`, `show memory`, `find memory`.

## 4. Consequences

- **Positive**:
  - Preserves consistency with existing CLI UX and parser architecture.
  - Enables deterministic memory retrieval without introducing a second incompatible metadata universe.
  - Keeps memory as pointers/constraints instead of competing canonical truths.
- **Trade-offs**:
  - Requires explicit schema and registry work before memory artifacts are production-safe.
  - Some JAMMS field names must be renamed or mapped (for example `type` to `memory_type`).
- **Verification Impact**:
  - New tests required for metadata validation, command naming consistency, and deterministic ranking/filter behavior.
