## 1. Context

- **Why**: ADR-002 (accepted) prohibits storing backlinks in frontmatter. PROD-003
  was authored before ADR-002 and describes stored backlinks throughout: data model,
  registry YAML, capabilities, flows, tests, and glossary. Reverse references
  should be computed at runtime by scanning forward references, consistent with
  ADR-002 and DE-045 precedent.
- **Source**: ADR-002 (Do not store backlinks in frontmatter)
- **Destination**: PROD-003 (Policy and Standard Management)
- **Requirements Affected**: FR-007 (bidirectional cross-references), FR-008 (artifact references)
- **Drift Ledger**: DL-047.005, DL-047.015

## 2. Related Artefacts

- **Decisions / ADRs**: ADR-002
- **Delta**: DE-047 (spec corpus reconciliation)

## 3. Summary of Changes

- **Data model**: Remove `backlinks: dict[str, list[str]]` field from PolicyRecord
- **Registry YAML example**: Remove backlinks section
- **Capabilities**: "Maintain backlinks automatically" → "Compute reverse references at runtime"
- **FR-007 verification**: "backlink generation" → "computed reverse reference resolution"
- **FR-008 verification**: "backlink validation" → "forward reference validation"
- **Flows**: Replace "sync to update backlinks" with "query to resolve reverse references"
- **Tests**: "backlinks" → "computed reverse references" in test descriptions
- **Glossary**: Update "Backlink" definition to clarify runtime computation
- **Display example**: Remove stored backlinks from show output example

## 4. Consequences

- Positive: PROD-003 aligns with ADR-002 and the computed-reference model.
- Positive: No competing stored truth — forward references are canonical, reverse
  references are derived views.
- Neutral: No code changes required if backlinks aren't implemented yet (they aren't).
