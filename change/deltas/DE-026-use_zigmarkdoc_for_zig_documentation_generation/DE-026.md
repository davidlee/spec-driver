## 1. Summary & Context
- **Technical Spec(s)**: SPEC-124 – Sync command and contract generation framework
- **Implementation Plan**: [IP-026](./IP-026.md) – draft
- **Design Revision**: [DR-026](./DR-026.md) – draft
- **Change Drivers**: IMPR-004 - Use zigmarkdoc for Zig documentation generation

## 2. Motivation
- Current ZigAdapter uses manual regex-based parsing that misses complex Zig patterns (comptime, inline fn, advanced doc comments)
- zigmarkdoc is a purpose-built, deterministic Markdown documentation generator for Zig
- GoAdapter successfully uses gomarkdoc via subprocess - same pattern should apply to Zig
- Manual parsing duplicates effort and produces inconsistent output compared to official tooling

## 3. Scope & Objectives
- **Primary Outcomes**:
  - ZigAdapter calls zigmarkdoc via subprocess instead of manual parsing
  - Support both public-only and all (including private) documentation variants
  - Maintain `--check` mode support for CI verification
  - Follow GoAdapter pattern for consistency across language adapters
- **Operational Constraints**: None - purely internal refactoring
- **Dependencies**: zigmarkdoc must be available at ~/.local/bin/zigmarkdoc

## 4. Out of Scope
- Changes to contract storage locations or frontmatter schema
- Zig LSP integration improvements
- Other language adapter refactoring
- zigmarkdoc installation/distribution automation

## 5. Approach Overview
- **System Touchpoints**: `supekku/scripts/lib/sync/adapters/zig.py`
- **Key Changes**:
  - Add `is_zigmarkdoc_available()` and `ZigmarkdocNotAvailableError` (following GoAdapter)
  - Replace `generate()` implementation with subprocess calls to zigmarkdoc
  - Remove manual parser methods: `_generate_zig_docs`, `_parse_zig_file`, `_extract_declaration`
  - Update `describe()` to return both "public" and "all" variants
  - Support zigmarkdoc's `--check`, `--include-private`, and `--output` flags
- **Migration / Rollout Notes**: Regenerate all existing Zig contracts after implementation

## 6. Verification Strategy
- **Requirements Coverage**: No new requirements - internal quality improvement

## 7. Risks & Mitigations
- **Risk**: zigmarkdoc output format differs significantly from current parser – *Likelihood*: medium – *Impact*: low – *Mitigation*: Accept as improvement; update contracts and verify no hard dependencies
- **Risk**: zigmarkdoc not available in all environments – *Likelihood*: low – *Impact*: medium – *Mitigation*: Raise clear error with installation instructions; document dependency
- **Risk**: Zig contract structure changes break downstream tooling – *Likelihood*: low – *Impact*: low – *Mitigation*: Verify contract consumption patterns before rollout
