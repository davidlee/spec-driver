## 1. Summary & Context
- **Product Spec**: PROD-012 – Code Contract Documentation
- **Implementation Plan**: IP-028 – draft
- **Design Revision**: DR-028 – draft
- **Change Drivers**: ADR-003 (contracts-first sync) + RE-016 (spec revision)

## 2. Motivation
- `sync` on legacy repos creates spec sprawl
- ADR-003: unit vs assembly specs are different artefacts
- Contracts are the primary discovery artefact and should be the default
- Currently `--specs` defaults `True`, no `--contracts` flag

## 3. Scope & Objectives
- **Primary Outcomes**:
  - `sync` defaults to contracts-only (FR-005)
  - `--specs` opts in + persists via `.spec-driver/enable_spec_autocreate` (FR-005)
  - Subsequent syncs respect persisted preference (FR-005)
  - `--contracts` / `--no-contracts` independent of spec creation (RE-016)
  - First-run hint message
  - Help text updated
- **Operational Constraints**: backward compat for existing repos
- **Dependencies**: None

## 4. Out of Scope
- Canonical `.contracts/` real-file storage (DE-029/RE-015)
- Sync-less contract generation (DE-029/RE-015)
- Unit vs assembly category metadata (ADR-003 follow-up)
- Structured config file (`.spec-driver/config.yaml`)

### Known Limitation: contracts require an existing spec bundle

Contract generation writes into `specify/tech/SPEC-*/contracts/`. With spec auto-creation off, units without an existing spec have nowhere to write. Fresh repos: sync is effectively a no-op. Partial coverage repos: existing specs get contracts. This is deliberate — DE-029 decouples from spec bundles. DE-028 must not introduce coupling that constrains that path:

- Preference module must not encode "specs required for contracts"
- `--contracts` flag must be wired so DE-029 can generate into `.contracts/`
- Hint message frames as "spec creation is off" not "contracts require specs"

## 5. Approach Overview
- **System Touchpoints**: `supekku/cli/sync.py`, new `core/sync_preferences.py`
- **Key Changes**:
  1. **Sync preferences module**: read/write marker, expose `spec_autocreate_enabled()` and `persist_spec_autocreate()`
  2. **CLI default change**: `--specs` becomes `None` (tri-state). Resolution: explicit flag > marker > `False`
  3. **`--contracts` / `--no-contracts`**: independent, default `True`. Gates intent — current constraint (contracts need spec dir) is in `process_source_unit`, not flag logic. DE-029 relaxes capability without touching flag semantics.
  4. **First-run messaging**: hint when specs skipped. Do NOT frame as "contracts require specs".
  5. **Backward compat**: existing registry entries = implicit opt-in
- **Migration**: existing repos unchanged. New repos contracts-first. Fresh repo sync is no-op until `--specs` or DE-029.

## 9. Implementation Notes
- `--specs` is `bool=True` at `sync.py:76`, change to `bool | None = None`
- Preference resolution: explicit flag > marker > `False`

- Contract generation is inside `process_source_unit`. `--no-contracts` needs `generate_contracts` param or skip at call site.

## 10. Design Direction: Toward Spec-Independent Contract Generation (DE-029 Seam)

### Current Architecture

- `adapter.describe(unit)` → `SourceDescriptor`
- `adapter.generate(unit, spec_dir=Path)` → takes `spec_dir`, internally hardcodes `contracts_dir = spec_dir / "contracts"` (Go/Zig/TS) or `output_root = spec_dir / "contracts"` (Python)
- `process_source_unit`: couples "where" to SPEC bundles, hardcoded path strategy, adapter appends `/contracts/` internally
- `MirrorEntry`: backward-looking post-hoc indexing

Coupling in two places:

1. `process_source_unit` bundles output path resolution with spec creation
2. Each adapter hardcodes `spec_dir / "contracts"`. Adapter decoupled from which spec, but not from `contracts/` subdir.

### Adapter Multiplicity: Python vs Fixed-Filename Languages

- **Go/Zig/TS**: fixed predictable filenames, resolvable before generation from `SourceDescriptor.variants`
- **Python**: many per-module files discovered during generation. Cannot resolve all output paths pre-generation.

### What DE-029 Needs

Planner → generator → writer pipeline:

1. **Planner** decides what to generate
2. **Generator** produces content (or staging/temp paths)
3. **Writer** places artefacts at final destinations using resolver

`ContractTarget` dataclass: `unit`, `view` (canonical), `variant_name` (language-specific), `dest_path`, `provenance`

`ContractOutputResolver` protocol with:
- `resolve_planned(unit, variant)` → pre-generation for fixed filenames
- `resolve_produced(unit, produced_path, metadata)` → post-generation for Python

Two implementations: `SpecBundleResolver` (current) and `MirrorTreeResolver` (DE-029).

Alternative: adapters accept `contracts_root` directly.

### What DE-028 Should Do

1. Add `generate_contracts: bool` to `process_source_unit` — gates `adapter.generate()`
2. Do not change `adapter.generate()` signature — DE-029's scope
3. Keep `--contracts` at CLI orchestration level — controls intent
4. Ensure `_sync_specs` callable with `generate_contracts=False`

### What DE-028 Should NOT Do

- "if no spec, skip contracts" logic outside `process_source_unit`
- Wire `--contracts` through sync preferences
- Make `MirrorEntry.spec_id` non-optional
- Assume output paths always resolvable before generation
