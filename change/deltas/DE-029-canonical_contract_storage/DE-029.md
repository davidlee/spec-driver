## 2. Motivation

Contracts are the primary discovery artefact (ADR-003). Today they live as second-class citizens inside spec bundles (`SPEC-*/contracts/`), and `.contracts/` is a post-hoc symlink index pointing back in. This creates three problems:

1. **Spec-dependent generation**: contracts cannot be generated without a spec directory to write into. On fresh repos with spec auto-creation off (DE-028), `sync` is a no-op — no contracts produced at all.
2. **Fragile indirection**: `.contracts/` entries are symlinks whose targets depend on spec bundle layout. Deleting or restructuring a spec bundle breaks the mirror silently.
3. **Wrong ownership model**: the canonical files live under a spec-centric namespace, but contracts are derived from source code, not from specs. The ownership should be source-path-based (`.contracts/`), with spec bundles as optional consumers.

**Target state**: `.contracts/` contains real files as the canonical storage location. Adapters write directly to `.contracts/<view>/<mirror-path>`. Spec bundles that want contracts get them via compat symlinks pointing *into* `.contracts/`. The mirror builder inverts from "index of spec bundles" to "canonical writer + compat provider".

## 3. Scope & Objectives

- **Primary Outcomes**:
  - Contract artefacts written as real files under `.contracts/<view>/<mirror-path>` (FR-008)
  - Backward-compat symlinks from `SPEC-*/contracts/` → `.contracts/` entries for existing workflows (FR-011)
  - Convention drift warnings when a SPEC has contracts but produces zero mirror entries (FR-012)
- **Operational Constraints**:
  - Must not break existing `sync --specs` workflows — spec creation, registry updates, frontmatter updates all continue unchanged
  - `.contracts/` remains `.gitignore`-safe (derived, deterministic, regenerable)
  - DE-028's `--contracts/--no-contracts` flag continues to gate intent; this delta changes *where*, not *whether*
- **Dependencies**:
  - DE-028 (complete): `generate_contracts` param exists on `process_source_unit`; `--contracts` flag wired

## 4. Out of Scope

- **Spec-independent generation** (DE-030/future): generating contracts without any spec/registry flow. This delta changes the *destination* but still uses the existing sync discovery pipeline.
- **Removing `SPEC-*/contracts/` directories**: existing files remain; compat symlinks point into `.contracts/`. Cleanup is a future migration concern.
- **Removing `by-*` indices**: existing symlink trees under `specify/tech/by-*/` are untouched.
- **Structured config** (`.spec-driver/config.yaml`): no new configuration needed.

## 5. Approach Overview

### System Touchpoints

| Component | Current | Target |
| --- | --- | --- |
| `adapter.generate()` (all 4) | `spec_dir: Path` → writes to `spec_dir/contracts/` | `variant_outputs: dict[str, Path]` → writes each variant to exact output file path |
| `process_source_unit` | Passes `spec_dir` to adapter | Computes per-variant canonical output file paths using mirror mapping; passes to adapter |
| `ContractMirrorTreeBuilder` | Post-hoc: scans `SPEC-*/contracts/`, creates symlinks under `.contracts/` | Creates compat symlinks from `SPEC-*/contracts/` → `.contracts/` entries; emits drift warnings |
| `_sync_specs` (post-sync) | Calls `mirror_builder.rebuild()` | Calls compat symlink builder (or integrated into sync flow) |

### Key Changes

1. **Adapter interface: `spec_dir` → `variant_outputs`**

   Change `adapter.generate(unit, spec_dir=...)` to `adapter.generate(unit, variant_outputs=...)`. The caller provides a `dict[str, Path]` mapping each variant name to its exact canonical output file path. This replaces the current pattern where adapters internally compute `contracts_dir = spec_dir / "contracts"` and choose filenames.

   **Why not `contracts_dir`**: Canonical mirror paths from DE-027 are per-source-unit leaf files, not directories with fixed filenames. For Zig and TS, the canonical path is `{identifier}.md` (e.g. `.contracts/public/src/agent.zig.md`), but adapters write `interfaces.md`/`api.md` internally. A `contracts_dir` parameter can't reconcile this — the adapter's internal filename would be wrong for the canonical path. Per-variant output file paths eliminate the mismatch.

   **Canonical mirror path shapes** (from DE-027 `mirror.py`):
   - Go: `.contracts/<view>/<identifier>/<contract_name>` → e.g. `.contracts/public/internal/foo/bar/interfaces.md` (original filename preserved)
   - Zig: `.contracts/<view>/<identifier>.md` → e.g. `.contracts/public/src/domain/agent.zig.md` (adapter filename discarded)
   - TS: `.contracts/<view>/<identifier>.md` → e.g. `.contracts/public/src/api.ts.md` (adapter filename discarded)
   - Python: `.contracts/<view>/<module-path>.py.md` → e.g. `.contracts/public/supekku/scripts/lib/specs/index.py.md` (post-generation distribution)

   **Return value contract**: `DocVariant.path` MUST match the provided output path. Centrally validated after `adapter.generate()`. Python may return empty/placeholder paths (filesystem scan of staging dir used instead); Go/Zig/TS must return the paths they were given. One unit test per adapter locks this contract.

   - Go: receives e.g. `{"public": Path(".contracts/public/foo/bar/interfaces.md"), "internal": Path(".contracts/internal/foo/bar/internals.md")}`. Adapter writes to those exact paths.
   - Zig: receives e.g. `{"public": Path(".contracts/public/src/agent.zig.md"), "internal": Path(".contracts/internal/src/agent.zig.md")}`. Adapter writes content from `interfaces.md`/`internals.md` generation to the provided paths.
   - TS: same pattern as Zig.
   - Python: uses staging dir (see §5.2 below).

2. **Output path resolution in `process_source_unit`**

   Before calling `adapter.generate()`, compute per-variant canonical output file paths using the same mapping logic from `mirror.py`, applied *before* generation rather than after.

   - Go: deterministic from `(identifier, variant)`. E.g. `{"public": .contracts/public/internal/foo/bar/interfaces.md, "internal": .contracts/internal/internal/foo/bar/internals.md}`.
   - Zig: deterministic from `(identifier, variant)`. E.g. `{"public": .contracts/public/src/agent.zig.md, "internal": .contracts/internal/src/agent.zig.md}`.
   - TS: deterministic from `(identifier, variant)`. E.g. `{"public": .contracts/public/src/api.ts.md, "internal": .contracts/internal/src/api.ts.md}`.
   - Python: uses staging dir (see below).

   **Python multiplicity handling**: Stage + distribute (decided).
   - Adapter writes to `.contracts/.staging/<language>/<identifier-slug>/`. Staging key uses `unit.language` + identifier slug (not spec-id) — works even when `--no-specs` is used and no spec exists.
   - Post-generation, a distribute step parses file headers (reusing `read_python_module_name()` from `mirror.py`) and moves files to `.contracts/<view>/<module-path>.py.md`.
   - The distribute step is the sole consumer of mirror mapping logic for Python's N outputs. Staging dir is cleaned up after distribution.
   - Adapter-aware views rejected: would push `.contracts/**` mapping logic into adapters, harder to keep consistent across languages.

3. **Compat symlinks (SPEC-*/contracts/ → .contracts/)**

   Invert the mirror builder. For each entry that would have been a symlink *from* `.contracts/` → `SPEC-*/contracts/`, instead create a symlink *from* `SPEC-*/contracts/<filename>` → `.contracts/<view>/<path>`. This preserves navigation for anything that still browses spec bundles.

   - Only created for source units that have a registered spec (i.e., units in `registry_v2.json`)
   - Compat symlinks are also derived — rebuilt on every sync, safe to delete
   - Builder must `unlink()` existing files before creating symlinks; warn when replacing a non-symlink file (first migration run will replace real files with symlinks)

4. **Convention drift warnings (FR-012)**

   During compat symlink creation, if a spec has a non-empty `contracts/` directory (contains ≥1 `.md` file) but the canonical generation produced zero `.contracts/**` entries for that spec's source unit, emit a warning. This catches mapper mismatches or adapter naming convention changes. Empty directories do not trigger warnings.

### Migration / Rollout

- **Existing repos**: First `sync` after this delta will write real files to `.contracts/` and replace the old symlinks there. Old real files under `SPEC-*/contracts/` remain (not deleted). Compat symlinks overlay them on next sync.
- **Fresh repos**: `.contracts/` populated directly once discovery is decoupled from specs (DE-030). This delta still uses the existing sync discovery pipeline, so fresh repos require at least one `sync` run to enumerate source units.
- **Recovery**: `rm -rf .contracts/` + `sync` regenerates everything. No data loss risk.

## 7. Risks & Mitigations

- **Risk**: Python stage+distribute adds latency – *Likelihood*: low – *Impact*: low – *Mitigation*: Python contracts are small text files; I/O is negligible vs AST parsing. Profile if needed.
- **Risk**: Compat symlinks break on platforms without symlink support (Windows) – *Likelihood*: low – *Impact*: low – *Mitigation*: Same constraint as existing mirror builder. Document Unix-only.
- **Risk**: Old real files under `SPEC-*/contracts/` coexist with new compat symlinks – *Likelihood*: high – *Impact*: low – *Mitigation*: Compat builder `unlink()`s existing files before symlinking, and warns when replacing a non-symlink. Document that first sync after upgrade replaces real files with symlinks in spec bundles. The real files now live under `.contracts/`.
- **Risk**: Adapter signature change (`spec_dir` → `variant_outputs`) breaks external consumers – *Likelihood*: low – *Impact*: medium – *Mitigation*: `process_source_unit` is the only caller of `adapter.generate()` via `_sync_specs`. Verify with `find_referencing_symbols`.

## 9. Implementation Notes

### Adapter interface change

All 4 adapters share the same signature pattern:
```python
def generate(self, unit, *, spec_dir: Path, check: bool = False) -> list[DocVariant]:
    contracts_dir = spec_dir / "contracts"  # hardcoded
```

Change to:
```python
def generate(self, unit, *, variant_outputs: dict[str, Path], check: bool = False) -> list[DocVariant]:
    # caller provides exact output file path for each variant
    # adapter writes to those paths directly
```

This is a single-callsite change — `process_source_unit` is the only consumer. `LanguageAdapter` (ABC) should be updated to match.

**Why per-variant file paths, not `contracts_dir`**: The canonical mirror paths from DE-027 differ by language. For Zig/TS, the canonical leaf file is `{identifier}.md` (e.g. `agent.zig.md`), but the adapter internally generates `interfaces.md`/`api.md`. A `contracts_dir` parameter would produce the wrong filename. Per-variant output paths let the caller dictate the exact canonical path, and the adapter just writes content there.

**Python exception**: Python discovers N module files during AST walk, so per-variant output file paths can't be pre-computed. Python's `generate()` receives a staging directory via a separate code path (see §5.2). The staging key is `{language}/{identifier-slug}`, not spec-id.

### Path mapping reuse

`mirror.py` contains per-language functions that compute mirror paths from contract artefacts:
- `zig_mirror_entries()`: `mirror_path = f"{identifier}.md"` — single leaf file per view
- `go_mirror_entries()`: `mirror_path = f"{identifier}/{contract_name}"` — dir + original filename
- `ts_mirror_entries()`: `mirror_path = f"{identifier}.md"` — single leaf file per view
- `python_mirror_entries()`: reads contract header to get module identity, then `module_path.py.md`

For Go/Zig/TS, extract pre-generation path resolvers from these post-hoc functions. The mapping is deterministic from `(language, identifier, variant)`.

For Python, mirror paths are only computable *after* generation (module names discovered during AST walk). Use the existing header-parsing logic from `read_python_module_name()` in a post-generation distribution step.

## 10. Design Constraint: Python Multiplicity

Python adapters produce many per-module files discovered during generation. Output paths cannot all be resolved pre-generation. This delta handles it via post-generation distribution, not by requiring pre-generation path resolution. This preserves the existing `generate_docs()` interface and avoids planner→generator→writer pipeline complexity.

The key invariant: adapters write files, then the caller knows where those files ended up (via return value or filesystem scan). The caller distributes to canonical locations. This is the simplest path that unblocks spec-independent generation (DE-030) without over-engineering the adapter interface.
