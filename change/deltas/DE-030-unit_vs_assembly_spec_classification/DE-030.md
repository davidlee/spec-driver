## 1. Summary & Context
- **Product Spec(s)**: `specify/product/PROD-015/PROD-015.md` – taxonomy + navigation intent.
- **Decision(s)**: `specify/decisions/ADR-003-separate_unit_and_assembly_specs.md` – unit vs assembly split and truth model.
- **Related Product Intent**:
  - `specify/product/PROD-012/PROD-012.md` – contract sync + spec stubbing (now constrained by ADR-003 + RE-016).
  - `specify/product/PROD-014/PROD-014.md` – contracts corpus ergonomics.
- **Change Drivers**: Legacy adoption ergonomics; avoid unit-stub sprawl drowning out assembly specs.

## 2. Motivation
Spec-driver currently treats “tech specs” as a single bucket. In practice, there are two different artefact types:
1) **Unit specs** (auto-generated / 1:1 with a language unit)
2) **Assembly specs** (human-authored / cross-unit)

Without explicit classification and navigation support, unit-spec sprawl makes subsystem/integration spec work feel unworkably messy, especially in legacy repos.

This delta introduces a non-breaking, metadata-first implementation of ADR-003:
- standardize taxonomy fields (`category`, `c4_level`)
- add navigation views and CLI filters
- add validator warnings (warn-only) to guide adoption

No filesystem/prefix migration is attempted in this delta.

## 3. Scope & Objectives
**Primary Outcomes** (mapped to PROD-015):
- Taxonomy is explicit and consistent for tech specs (`category`, `c4_level`). (`FR-001`)
- Sync-created unit specs are clearly labeled (`category: unit`, `c4_level: code`). (`FR-002`)
- Humans/agents can list and browse by taxonomy without manual grep. (`FR-003`, `FR-004`)
- Adoption is guided via warnings, not blocking validation failures. (`FR-005`)

**Operational Constraints**
- Non-breaking: existing spec IDs/paths must remain valid. (`NF-002`)
- Keep performance acceptable for interactive use. (`NF-001`)

**Dependencies**
- Sync defaults and spec auto-creation opt-in (DE-028/RE-016 direction) should already be in place; this delta builds on that intent.

## 4. Out of Scope
- Introducing new top-level roots / prefixes (e.g. `specify/unit/UNIT-###`, `specify/assembly/ASM-###`) and migrating existing specs.
- Introducing an ownership/claims schema for overlapping interface constraints (ADR-003 future hook).
- Spec-independent contract generation / discovery (separate contracts deltas).

## 5. Approach Overview
**System Touchpoints**
- Frontmatter: `supekku/scripts/lib/core/frontmatter_metadata/spec.py`
- Spec indexing/views: `supekku/scripts/lib/specs/index.py` (or adjacent index builder)
- CLI listing: `supekku/cli/list.py`
- Validation: `supekku/scripts/lib/validation/validator.py`
- Sync stub creation path (wherever spec frontmatter is synthesized during auto-create)

**Key Changes**
- Standardize reserved values:
  - `category`: `unit` | `assembly` (others allowed but these are reserved and first-class)
  - `c4_level`: existing enum; recommend `code` for `unit`
- Treat missing values as `unknown` for navigation and filtering.
- Add deterministic index views under `specify/tech/`:
  - `by-category/unit/…`, `by-category/assembly/…`
  - `by-category/unknown/…`
  - `by-c4-level/code/…`, etc.
- Extend by-level views with `by-c4-level/unknown/…`.
- Extend `list specs` with `--category` and `--c4-level` filters.
- Validation emits warnings (not errors) when:
  - a tech spec is missing taxonomy fields
  - inconsistent combinations are present (`unit` + non-`code` `c4_level`)

**Migration / Rollout Notes**
- No bulk backfill is required for correctness; missing fields simply produce warnings.
- Sync-created specs should always have defaults set, so new content becomes coherent over time.

## 6. Verification Strategy
- **Requirements Coverage**: See `specify/product/PROD-015/PROD-015.md` coverage block.
- **Acceptance Criteria**:
  - `spec-driver list specs --category assembly` yields a strict subset of specs.
  - `specify/tech/by-category/assembly` exists after sync/index build and contains only assembly specs.
  - Validation warnings are actionable and do not block normal workflows.

## 7. Risks & Mitigations
- **Risk**: “category is already in schema and used inconsistently” – *Likelihood*: medium – *Impact*: low – *Mitigation*: treat `unit`/`assembly` as reserved; warn-only; avoid schema tightening in v1.
- **Risk**: View/index build increases runtime – *Likelihood*: low – *Impact*: medium – *Mitigation*: reuse existing index patterns; measure (NF-001).
