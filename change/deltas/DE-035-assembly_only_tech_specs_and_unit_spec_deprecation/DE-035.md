## 1. Summary & Context
- **Product Spec(s)**:
  - `specify/product/PROD-015/PROD-015.md` – taxonomy and navigation intent.
  - `specify/product/PROD-012/PROD-012.md` – sync and contract-generation behavior.
- **Decision(s)**:
  - `specify/decisions/ADR-003-separate_unit_and_assembly_specs.md` – accepted baseline to be superseded/amended.
- **Implementation Plan**: [IP-035](./IP-035.md) – draft.
- **Change Drivers**:
  - unit-spec sprawl remains low-signal for humans;
  - contracts-first workflows are now the practical source of code-facing truth;
  - assembly specs are the high-value authored layer.

## 2. Motivation
- Current taxonomy support solved discoverability pain, but it preserved a concept that is not paying for itself: unit tech specs as a first-class artifact family.
- The target state is simpler:
  - generated contracts are code-facing truth (`.contracts/**`);
  - tech specs are authored assembly-level intent and constraints;
  - legacy `category: unit` remains readable but is not generated or encouraged.

## 3. Scope & Objectives
- **Primary Outcomes**:
  - Stop creating new unit tech specs in sync and creation paths. (`PROD-015.FR-002`, `PROD-012.FR-005`)
  - Keep assembly/unknown navigation and remove unit as a first-class UX surface. (`PROD-015.FR-003`, `PROD-015.FR-004`)
  - Shift validation from "unit consistency" to "unit deprecation + assembly taxonomy quality". (`PROD-015.FR-005`)
- **Operational Constraints**:
  - Non-breaking transition for existing `SPEC-*` IDs and files.
  - Existing `category: unit` specs must not fail validation as errors in v1.
- **Dependencies**:
  - RE-018 alignment of PROD requirements.
  - ADR follow-up (supersede/amend ADR-003 wording).

## 4. Out of Scope
- Hard migration/rewriting of all historical specs to new IDs or directories.
- Removal of existing `category: unit` metadata from old specs in one shot.
- Advanced ownership/claims schema for cross-spec interface governance.

## 5. Approach Overview
- **System Touchpoints**:
  - `supekku/scripts/sync_specs.py`
  - `supekku/scripts/lib/specs/creation.py`
  - `supekku/cli/list.py`
  - `supekku/scripts/lib/specs/index.py`
  - `supekku/scripts/lib/validation/validator.py`
  - related tests under `supekku/**/_test.py`
- **Key Changes**:
  - Sync-created specs no longer set `category: unit`; default to assembly-oriented metadata or skip creation unless explicitly requested.
  - `list specs` category behavior no longer centers `unit`; keep compatibility for explicit `--category unit` against legacy records only.
  - Index views remove `units` alias and demote/remove `by-category/unit` from first-class guarantees.
  - Validator warns that `category: unit` is deprecated and suggests assembly-oriented alternatives.
- **Migration / Rollout Notes**:
  - First pass is compatibility-preserving: legacy unit specs continue to resolve.
  - Follow-up cleanup can backfill legacy metadata after behavior stabilizes.

## 6. Verification Strategy
- **Requirements Coverage**: tracked in [IP-035](./IP-035.md) with VT-035-* entries.
- **Planned Artefacts**:
  - VT-035-001: sync/creation stop generating new unit specs.
  - VT-035-002: CLI/index show assembly-first behavior with legacy compatibility.
  - VT-035-003: validator deprecation warnings for legacy `unit`.
  - VT-035-004: non-breaking compatibility for old SPEC IDs and existing metadata.
- **Acceptance Criteria**:
  - No new generated specs contain `category: unit`.
  - Assembly workflows operate without unit-focused flags/views.
  - Existing repositories with unit-tagged specs pass validation without errors.

## 7. Risks & Mitigations
- **Risk**: Existing docs and tests rely on `unit` semantics – *Likelihood*: high – *Impact*: medium – *Mitigation*: staged update of tests/docs in same delta.
- **Risk**: Users depend on `--category unit` workflows – *Likelihood*: medium – *Impact*: low – *Mitigation*: keep legacy filter behavior temporarily; document deprecation.
- **Risk**: ADR inconsistency causes decision drift – *Likelihood*: medium – *Impact*: medium – *Mitigation*: create follow-up ADR update in this rollout.
