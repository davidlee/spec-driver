# DR-036 – frontmatter metadata compaction and canonicalization controls

## 1. Executive Summary
- **Delta**: [DE-036](./DE-036.md)
- **Status**: draft
- **Last Updated**: 2026-03-03
- **Synopsis**: Extend FieldMetadata with persistence classification, enabling compaction profiles that keep persisted frontmatter lean while preserving semantic fidelity.

## 2. Problem & Constraints
- **Current Behaviour**: `resolve links` persists full resolved payloads (`id`, `path`, `kind`, `label`) for every `[[...]]` link in memory frontmatter. Empty-default fields (`aliases: []`, `relations: []`) persist even when they carry no information. This inflates files by 22–62% on average.
- **Drivers / Inputs**: DE-036 motivation; PROD-004.FR-007 (added via RE-019).
- **Constraints / Guardrails**:
  - No retrieval/ranking regressions for memories.
  - Backward-compatible parsing of existing files (compaction is write-side only).
  - Deterministic serialization for stable diffs.
  - Agent guidance must account for compacted frontmatter (JSON Schema query path).
- **Out of Scope**:
  - Replacing `[[...]]` link syntax.
  - Hard migration of all historical artifacts in one pass.
  - Redesigning taxonomy or lifecycle semantics.

## 3. Architecture Intent
- **Target Outcomes**:
  - `FieldMetadata` gains `persistence` and `default_value` attributes.
  - Memory and delta metadata definitions annotated with persistence classifications.
  - `links_to_frontmatter()` supports mode-controlled output (`none`, `missing`, `compact`, `full`).
  - Sync/resolve commands default to lean output; verbose persistence is explicit opt-in.
- **Guiding Principles**:
  - Metadata as single source of truth (PROD-004 principle).
  - Write-side compaction, read-side tolerance — never break existing files.
  - Minimal schema engine changes — two new optional fields on `FieldMetadata`.
- **State Transitions / Lifecycle Impact**: None — compaction is a serialization concern, not a lifecycle change.

## 7. Design Decisions & Trade-offs

### DEC-036-001: Compact-by-default on sync commands only
- **Rationale**: Sync is the automation layer that causes inflation. Manual authoring should remain unconstrained — authors can include whatever fields they want.
- **Consequence**: Compaction logic lives in sync/resolve write paths, not in general file-write utilities.

### DEC-036-002: Link persistence default is missing-only
- **Rationale**: `links.out` is fully reconstructible from body `[[...]]` tokens via `resolve links`. `links.missing` is a quality signal (unresolved references) worth persisting.
- **Consequence**: Default mode is `missing`; `full` mode available for workflows that need persisted `out` data.

### DEC-036-003: Pilot non-memory family is delta frontmatter
- **Rationale**: Deltas have clear empty-default fields (`aliases: []`, `relations: []`, `applies_to: {specs: [], requirements: []}`) that inflate without information. Memory compaction is more specialized (links); delta compaction tests the general framework.
- **Consequence**: P2 pilot targets delta frontmatter metadata with `default-omit` classifications.

### DEC-036-004: Extend FieldMetadata with persistence + default_value
- **Rationale**: The metadata definitions are already the source of truth for validation and JSON Schema. Adding persistence semantics here keeps the single-source-of-truth principle intact rather than creating separate compaction rule tables.
- **Fields added**:
  - `persistence: str = "canonical"` — classification: `canonical`, `derived`, `optional`, `default-omit`
  - `default_value: Any = None` — value that signals "omit during compaction"
- **Consequence**: Existing FieldMetadata instances unchanged (default is `canonical`, which means "always persist"). Opt-in per field.
