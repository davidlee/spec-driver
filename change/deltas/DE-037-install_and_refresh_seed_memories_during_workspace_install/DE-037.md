## 1. Summary & Context
- **Product Spec(s)**:
  - `specify/product/PROD-015/PROD-015.md` (agent workflow ergonomics and discoverability)
- **Technical Spec(s)**:
  - Memory system baseline from DE-033 (`memory` records, retrieval, and resolution)
  - Install workflow in `supekku/scripts/install.py`
- **Implementation Plan**: [IP-037](./IP-037.md) – draft
- **Change Drivers**:
  - Install currently bootstraps templates/about/agents/skills, but not memory packs.
  - We need two distinct memory ownership models:
    - `seed`: starter memories users may customize (install-if-missing; do not overwrite).
    - `spec-driver`: spec-driver-owned capability/concept memories (replace/sync as a managed set).
  - Current `seed` tagging appears conflated with spec-driver-owned records and must be sifted.

## 2. Motivation
- A single “seed = replace” rule is wrong for user-customizable starter memories.
- We need explicit ownership semantics so installer behavior matches intent:
  - user-owned starter content must persist local edits;
  - spec-driver-owned knowledge must stay current and consistent across installs.
- Desired behavior:
  - `seed` memories are installed once (or added when missing), then left alone.
  - `spec-driver` managed memories are refreshed from package source on reinstall.
  - unmanaged user memories remain untouched.

## 3. Scope & Objectives
- **Primary Outcomes**:
  - Introduce two install buckets and enforce different policies:
    - `seed` (starter/user-customizable): install-or-leave.
    - `spec-driver` (managed): replace/sync as package-owned set.
  - Define deterministic classification and source discovery for both buckets.
  - Add explicit sifting step to reclassify existing memory corpus into the two buckets.
  - Preserve local unmanaged memory IDs not in either managed pack.
  - Integrate dry-run and interactive category prompts consistent with existing install UX.
- **Operational Constraints**:
  - Must respect `--dry-run` and `--yes` behavior.
  - Must not overwrite unrelated user files.
  - Must remain idempotent across repeated installs.
- **Dependencies**:
  - Existing install category-copy workflow and test harness in `supekku/scripts/lib/install_test.py`.
  - Seed memory source corpus packaging decision (path in package artifact).

## 4. Out of Scope
- Auto-generating non-seed memories during install.
- Repository-wide memory deduplication beyond bucket/classification migration.
- Memory compaction/serialization policy changes from DE-036.
- Post-install memory ranking or scope tuning.

## 5. Approach Overview
- **System Touchpoints**:
  - `supekku/scripts/install.py`
  - `supekku/scripts/lib/install_test.py`
  - packaged seed memory source directory (to be finalized in DR-037)
- **Key Changes**:
  - Define canonical memory pack metadata in package source:
    - bucket = `seed` or `spec-driver`
    - stable memory ID is the replacement key
  - Installer behavior:
    - `seed` bucket: create when missing; do not overwrite existing IDs.
    - `spec-driver` bucket: upsert/replace managed IDs from package source.
    - optional managed-set prune decision for removed spec-driver IDs (tracked as open decision).
  - Sift existing corpus:
    - move currently conflated `seed` records into correct buckets.
    - ensure tags/metadata reflect ownership semantics rather than mixed meaning.
  - Add separate install category reporting for `seed` and `spec-driver` memory updates.
  - Ensure behavior is no-op in dry-run except reporting.
- **Migration / Rollout Notes**:
  - Existing workspaces receive:
    - missing `seed` installs;
    - spec-driver managed memory refresh on reinstall.
  - Sifting migration must happen before enabling replacement behavior broadly.
  - If source path is absent in package artifact, installer should fail gracefully or skip with explicit warning (decision in DR-037).

- **Planned Artefacts**:

- **Acceptance Criteria**:

  - Re-running install:
    - leaves customized `seed` records intact;
    - refreshes `spec-driver` managed records;
    - does not modify unmanaged memory IDs.
  - Test suite additions pass with no regressions in existing install behavior.

## 7. Risks & Mitigations
- **Risk**: Bucket classification is ambiguous in source metadata – *Likelihood*: medium – *Impact*: high – *Mitigation*: explicit classifier field/manifest and migration checks.
- **Risk**: Managed replacement accidentally targets user-customized starter memories – *Likelihood*: medium – *Impact*: high – *Mitigation*: strict bucket isolation and tests for “seed never overwritten”.
- **Risk**: Packaging misses memory source files – *Likelihood*: medium – *Impact*: high – *Mitigation*: add source-discovery tests and explicit warning path.
- **Risk**: Prompt fatigue from extra category – *Likelihood*: low – *Impact*: low – *Mitigation*: keep category-level prompt consistent with existing install UX and `--yes` bypass.

### Classification Decision

**Resolved**: all 19 memories from the initial corpus fall into two buckets based
on what they represent, not on their `memory_type`:

**spec-driver bucket** (19 memories — platform-owned, replaced on update):
All concept, signpost, and pattern memories that define what spec-driver's
primitives and workflows *are*. These are platform reference material — if
spec-driver's model evolves, every project should get the update.

Includes: `philosophy`, `posture`, `core-loop`, `ceremony` (signpost + 3 modes),
`backlog`, `spec`, `revision`, `delta`, `design-revision`, `plan`, `audit`,
`contract`, `truth-model`, `relations`, `verification`, `delta-completion`.

**seed bucket** (new — project-owned starters, install-if-missing):
Thin project-workflow stubs that reference spec-driver platform memories as
sensible defaults but give the project a place to customise. These must serve
two audiences simultaneously:

- **Humans**: “edit this to define your project's workflow”
- **Agents**: “follow this sensible default if it hasn't been edited”

Initial seed starters:

- `mem.pattern.project.workflow` — project's customised development loop,
  initially referencing `[[mem.pattern.spec-driver.core-loop]]` as the default.
  Projects add local shortcuts, required steps, or ceremony overrides here.
- `mem.pattern.project.completion` — project's completion checklist,
  initially referencing `[[mem.pattern.spec-driver.delta-completion]]`.
  Projects add deployment steps, sign-off requirements, etc.

**Design rationale**: Platform reference memories evolve with spec-driver
releases. Seed starters are installed once and then owned by the project —
a new release can update the platform reference without overwriting the
project's customised workflow. The seed stubs work as-is (agent follows the
referenced platform default) until a human edits them.

**Classification mechanism**: The `seed` tag is retired as a bucket classifier.
Bucket identity is determined by ID namespace:
- `mem.*.spec-driver.*` → spec-driver bucket (managed)
- `mem.*.project.*` → seed bucket (install-if-missing)
- Everything else → unmanaged (never touched by installer)
