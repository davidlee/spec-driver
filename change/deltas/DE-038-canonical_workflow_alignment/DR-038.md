# DR-038 – Canonical workflow alignment

## 1. Executive Summary
- **Delta**: [DE-038](./DE-038.md)
- **Status**: draft
- **Synopsis**: Align memories and workflow research to promote delta-first as canonical, correct stale claims, and define the switchable lock-in surface area for a future implementation delta.

## 2. Problem & Constraints
- **Current Behaviour**: Memories and docs present competing workflow narratives. Core-loop leads with revision-first. Status enums in memories cite values that don't exist in code. Ceremony mode described with false enforcement implications.
- **Drivers / Inputs**: [workflow-research.md](./workflow-research.md) section E (drift findings) and section F (future switch surface area)
- **Constraints**: Memory/skill/doc changes only — no code changes
- **Out of Scope**: Rewriting source docs (lifecycle.md, README.md, etc.)

## 3. Architecture Intent

### Target Narrative

An agent encountering spec-driver for the first time should absorb this sequence:

1. Create a delta to scope the change
2. (Optional) Write a DR and IP with phases
3. Implement
4. Audit findings and produce revision where needed
5. Patch specs from the audit/revision outcome
6. Complete and reconcile owning records

Revision-first appears only in Town Planner guidance, explicitly framed as a higher-ceremony concession.

### Ceremony as Guidance Strength

Memories should say: "ceremony mode tells agents what to expect, not what to forbid." No memory should imply runtime enforcement that doesn't exist.

### Future `strict_mode` Contract (Documentation Only)

This delta defines the target behavior for a `workflow.toml` setting:

- **Setting**: `strict_mode` (boolean) in `.spec-driver/workflow.toml`
- **Default (`false`, today)**: permissive sequencing remains available
- **Enabled (`true`, future)**: enforce canonical ordering and hard-fail non-canonical prerequisites/flag paths
- **Status in this delta**: contract documented only; no code paths branch on the setting yet
- **Exception policy**: none in baseline strict contract (no escape hatches specified in this delta)
- **Evolution note**: this starts as one setting and may split into finer controls later

### Enforcement Touchpoints (Future Implementation Map)

Follow-up implementation should evaluate gates in these surfaces:

- artifact creation flow (`create delta`, `create phase`, revision/audit creation surfaces)
- closure flow (`complete delta` ordering and prerequisite checks)
- workspace validation (`validate` checks for sequence coherence where possible)
- agent guidance generation (`agents/workflow.md` rendering based on switch)
- flag interaction policy with hard-fail handling for non-canonical paths (`--allow-missing-plan`, `--force`, `--skip-update-requirements`, `--skip-sync`)
- shared status-policy checks used by both write-time mutation commands and read-time sync/validation paths

### Coverage Aggregation Precedence (Recommended v1)

To reduce ambiguity when/if plan coverage is included in default sync:

- Keep conservative lifecycle aggregation:
  - any `failed` or `blocked` => `in-progress`
  - all `verified` => `active`
  - all `planned` => `pending`
  - any mixed set (for example `verified` + `planned`) => `in-progress`
- Validation should always warn (not error) on mixed statuses across artifact sources, including source paths in diagnostics.
- Warning severity remains warning in both permissive mode and strict mode.

### 4.7 NEW: `mem.fact.spec-driver.coverage-gate` (CREATE)

Sharp-edge memory documenting the coverage block prerequisite:
- What: `complete delta` requires all `applies_to.requirements` to have `status: verified` in parent spec `supekku:verification.coverage@v1` blocks
- Env var: `SPEC_DRIVER_ENFORCE_COVERAGE` (default: enabled)
- Bypass: `--force` flag
- Common failure: agent doesn't know to add coverage blocks, hits wall at completion
- Valid coverage statuses: `planned | in-progress | verified | failed | blocked`
- Tags: `spec-driver, coverage, verification, delta, completion, sharp-edge`

### 4.8 NEW: `mem.fact.spec-driver.status-enums` (CREATE)

Authoritative status enum reference:
- Requirement lifecycle canonical set: `pending | in-progress | active | retired` (source: `lifecycle.py:7-10`)
- Requirement lifecycle ingestion caveat: revision lifecycle payloads are currently tolerant of non-canonical status strings; follow-up strict-mode implementation should validate both read/write through one shared code path (source: `requirements/registry.py`)
- Change artifact canonical set: `draft | pending | in-progress | completed | deferred` plus legacy alias `complete` normalized to `completed` (source: `changes/lifecycle.py:11-37`)
- Coverage verification: `planned | in-progress | verified | failed | blocked` (source: `blocks/verification.py:24`)
- WARNING: `implemented`, `verified`, `live`, and `archive` are not canonical requirement lifecycle statuses despite appearing in some docs
- Tags: `spec-driver, lifecycle, status, sharp-edge`

## 5. Gaps Document

A new file `gaps-to-adoption.md` in the delta bundle documenting critical barriers to mass adoption. This delta is responsible for specifying the gaps and evidence, not for delivering remediation plans. Structure:

1. **Hard blockers** — things that will cause agent/user failure
2. **Confusion vectors** — things that will waste time or create wrong mental models
3. **Missing automation** — things that require manual work where commands should exist
4. **Documentation debt** — specific files with specific wrong claims

Each entry: gap description, severity, affected user journey, recommended fix, evidence.
