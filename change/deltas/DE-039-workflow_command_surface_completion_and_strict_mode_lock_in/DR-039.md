# DR-039 – workflow command-surface completion and strict-mode lock-in

## 1. Executive Summary
- **Delta**: [DE-039](./DE-039.md)
- **Status**: draft
- **Owners / Team**: Dev
- **Last Updated**: 2026-03-04
- **Synopsis**: Complete workflow command surface by adding strict-mode runtime controls and first-class revision/audit lifecycle commands.

## 2. Problem & Constraints
- **Current Behaviour**:
  - strict-mode contract is documented but not enforced at runtime.
  - audit/revision lifecycle steps are partially manual due missing command coverage.
  - completion UX historically depended on interactive prompt choreography in automation.
- **Drivers / Inputs**:
  - DE-038 gaps register and workflow alignment research.
  - PROD-016 command-surface requirements (FR-002/FR-009/FR-010).
- **Constraints / Guardrails**:
  - Keep default permissive mode unchanged unless `strict_mode=true`.
  - Preserve skinny CLI layering (CLI orchestration, domain logic in scripts/lib).
- **Out of Scope**:
  - new lifecycle status taxonomy.
  - non-workflow command redesign.

## 3. Architecture Intent
- **Target Outcomes**:
  - `strict_mode` is loaded from workflow config and gates non-canonical command paths.
  - revision/audit flows are discoverable and executable via first-class CLI commands.
  - completion flow remains deterministic in both interactive and non-interactive contexts.
- **Guiding Principles**:
  - Prefer explicit flags and deterministic defaults over implicit prompt behavior.
  - Reuse existing lifecycle/status utilities instead of duplicating command-specific logic.
- **State Transitions / Lifecycle Impact**:
  - No new statuses introduced; command behavior should use existing canonical change status semantics.

## 4. Code Impact Summary
| Path | Current State | Target State |
| --- | --- | --- |
| `supekku/scripts/lib/core/config.py` | No `strict_mode` runtime knob | Load and expose `strict_mode` with safe default |
| `supekku/scripts/complete_delta.py` | Prompt + gating behavior partially implicit | Deterministic non-interactive behavior plus strict-mode gate hooks |
| `supekku/cli/create.py` | No first-class `create audit` | Add audit creation command path |
| `supekku/cli/complete.py` | Only `complete delta` | Add revision completion command path |
| `supekku/scripts/lib/changes/coverage_check.py` | Legacy docs pointer text | Canonical docs pointer and guidance |

## 7. Design Decisions & Trade-offs
- DEC-039-01 – Keep strict-mode as a single boolean in this delta; defer split-policy model.
- DEC-039-02 – Prefer first-class command verbs for revision/audit over documenting manual file scaffolding.
- DEC-039-03 – Non-interactive command defaults must be deterministic and test-backed.
- DEC-039-04 – In strict mode, universally block force-style bypass paths rather than allowing command-specific exceptions.

## 9. Rollout & Operational Notes
- **Migration / Backfill**: none; command and config behavior only.
- **Recovery / Rollback**: disable strict-mode (`strict_mode=false`) to restore permissive behavior if rollout issues occur.
