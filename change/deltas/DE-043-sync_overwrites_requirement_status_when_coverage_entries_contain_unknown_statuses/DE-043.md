  work" on status validation

## 2. Motivation

Coverage entries with statuses not in `VALID_STATUSES` (e.g. `deferred`) flow
unchecked through `_apply_coverage_blocks()` into `_compute_status_from_coverage()`,
silently corrupting derived requirement state on every sync. This violates
PROD-009's guiding principle: "Later evidence overlays earlier declarations but
never silently discards them."

The deeper problem: three distinct `VALID_STATUSES` sets (requirement lifecycle,
change artifact lifecycle, verification coverage) are easily conflated by agents
and users. No discoverable guidance explains the differences, common corner cases,
or how requirements transit between lifecycle phases. Agents make plausible but
wrong choices because the system fails silently.

## 3. Scope & Objectives

- **Primary Outcomes**:
  1. **Validation at sync boundary**: `_apply_coverage_blocks()` validates coverage
     entries against `VALID_STATUSES` from `verification.py`, warns on invalid entries,
     and excludes them from status derivation. (Addresses PROD-008.FR-001: spec
     coverage block is authoritative; invalid data must not corrupt it.)
  2. **Defensive derivation**: `_compute_status_from_coverage()` references the
     canonical `VALID_STATUSES` set instead of hardcoded strings. Unknown statuses
     cannot influence derived state. (Addresses PROD-009.FR-003: validation must
     emit corrective warnings.)
  3. **Discoverable lifecycle guidance**: authoritative, agent-accessible documentation
     answering common requirement lifecycle questions — including:
     - What statuses are valid in which context (requirement vs coverage vs change)?
     - What happens when a user wants to change an established requirement?
     - How do requirements transit from backlog/revision into a spec?
     - At what workflow points are these transitions expected/encouraged?
     (Addresses PROD-009.FR-001: explicit lifecycle statuses with clear semantics.)
  4. **Memory corrections**: fix `passed` vs `verified` drift in
     `mem.concept.spec-driver.verification`.
- **Operational Constraints**: Minimal code change surface. No schema changes.
  No new CLI commands.
- **Dependencies**: None.

## 4. Out of Scope

- **Shared status-policy validator** across all write/read paths (flagged in
  `mem.fact.spec-driver.status-enums` as future work — larger refactor).
- **`deferred` as a new valid coverage status** — the user's original `deferred`
  usage was a reasonable mistake given poor discoverability, not a signal to
  expand the enum.
- **Provenance/timestamp-based overlay** (PROD-009 §4 Solution Outline) — that's
  a larger feature, not a bug fix.
- **Tech spec for `blocks/verification.py`** — useful but separate work.

## 5. Approach Overview

- **System Touchpoints**:
  - `supekku/scripts/lib/requirements/registry.py` — `_apply_coverage_blocks()`,
    `_compute_status_from_coverage()`
  - `supekku/scripts/lib/blocks/verification.py` — `VALID_STATUSES` (import target)
  - Memory records — `mem.concept.spec-driver.verification`,
    `mem.fact.spec-driver.status-enums`
  - Agent guidance — new or updated memory record(s) covering lifecycle corner cases
- **Key Changes**:
  1. Import `VALID_STATUSES` from `verification.py` in registry; filter/warn in
     `_apply_coverage_blocks()` before entries reach status derivation
  2. Replace hardcoded status strings in `_compute_status_from_coverage()` with
     reference to canonical set
  3. Author lifecycle guidance as a memory record (or equivalent discoverable
     artifact) that agents can retrieve
  4. Correct memory drift

## 6. Verification Strategy

- **Requirements Coverage**:
  - VT: unit tests for invalid-status filtering and warning in `_apply_coverage_blocks()`
  - VT: unit tests for `_compute_status_from_coverage()` with unknown statuses
  - VA: agent review confirming lifecycle guidance is discoverable and answers
    the corner cases listed in §3
- **Acceptance Criteria**:
  1. Coverage entry with status `deferred` (or any unknown string) does NOT change
     derived requirement status
  2. Warning emitted to stderr identifying the invalid entry
  3. All existing tests pass (`just check`)
  4. Lifecycle guidance is retrievable via `spec-driver list memories` with
     relevant scope queries

## 7. Risks & Mitigations

- **Risk**: Existing coverage blocks in the wild use non-canonical statuses that
  will now trigger warnings – *Likelihood*: low – *Impact*: low –
  *Mitigation*: warning is non-blocking; document in release notes
- **Risk**: Guidance document becomes stale as lifecycle evolves –
  *Likelihood*: medium – *Impact*: medium –
  *Mitigation*: link guidance to canonical code; include provenance references
