# DR-047 – Contradiction discovery and resolution framework

## 1. Executive Summary

- **Delta**: [DE-047](./DE-047.md)
- **Status**: draft
- **Owners**: david
- **Last Updated**: 2026-03-05
- **Synopsis**: Systematic process to surface, resolve, and codify contradictions across the PROD spec corpus before any spec can be promoted to `active`.

## 2. Problem & Constraints

- **Current Behaviour**: 16 PROD specs (15 draft, 1 active) contain evolved and
  sometimes contradictory design intent. Terms are used inconsistently. Some
  specs reflect earlier architectural thinking that has since been superseded by
  ADRs or informal decisions. Agents encounter conflicting guidance and cannot
  treat any single document as authoritative.

- **Drivers / Inputs**:
  - ISSUE-039 — umbrella for spec corpus reconciliation
  - ADR-003 (unit vs assembly specs) — established after several PROD specs were written
  - Informal evolution of concepts like "ceremony modes", "contracts vs specs",
    "unit vs assembly" that may not be consistently reflected

- **Constraints / Guardrails**:
  - Contradiction resolution requires human design decisions — cannot be fully automated
  - Must not change design intent; only surface and resolve conflicts
  - Resolutions must be durable (ADRs or spec revisions), not ad-hoc edits

- **Out of Scope**:
  - Promoting specs to `active` (subsequent delta)
  - TECH spec work (Phase C/D)
  - Code changes

## 3. Architecture Intent

- **Target Outcomes**:
  - A **contradiction register** documenting every cross-document conflict
  - Each contradiction **resolved** via ADR, spec revision, or editorial fix
  - All PROD specs internally consistent and aligned with accepted ADRs

- **Guiding Principles**:
  - Surface contradictions systematically — don't rely on stumbling across them
  - Batch related contradictions so one design decision resolves several
  - Prefer ADRs for decisions that apply across multiple specs
  - Prefer spec revisions for conflicts local to one or two specs

## 4. Contradiction Register

The register is the primary output artifact. It lives in this delta directory
as `contradiction-register.md`.

### Register entry schema

Each entry captures:

```
### CR-NNN: <short title>

**Severity**: blocking | significant | cosmetic
**Topic area**: <e.g. "spec granularity", "lifecycle semantics", "contract role">

**Conflict**:
- Document A (§section): "<quoted or paraphrased statement>"
- Document B (§section): "<quoted or paraphrased statement>"

**Analysis**: <why these conflict; what the underlying design question is>

**Resolution path**: ADR | spec-revision | editorial
**Resolution status**: open | resolved
**Resolution**: <decision made, ADR/RE reference if applicable>
```

### Severity definitions

- **blocking**: Specs cannot be promoted until this is resolved; fundamental
  design question
- **significant**: Causes confusion or incorrect agent behaviour; should be
  resolved before promotion but won't break anything
- **cosmetic**: Inconsistent wording or outdated terminology; can be fixed
  editorially during promotion

### Topic areas (expected)

Based on initial survey, likely topic areas include:

- **Spec granularity & taxonomy**: unit vs assembly, file vs package, category semantics
- **Contract role**: what contracts are for, where they live, canonical vs derived
- **Lifecycle semantics**: requirement statuses, spec statuses, promotion criteria
- **Ceremony & governance**: ceremony modes, when deltas are required, revision-first vs delta-first
- **Agent workflow**: what agents should/shouldn't do, autonomy boundaries
- **CLI design**: flag conventions, output modes, progressive disclosure

## 5. Discovery process

### Phase 1: Agent survey (VA-047-001)

An agent reads the full corpus systematically:

1. **Document inventory**: All 16 PROD specs, 3 ADRs, 2 policies, CLAUDE.md,
   glossary, doctrine, workflow docs
2. **Concept extraction**: For each document, extract key claims about:
   - Definitions of terms/concepts
   - Lifecycle rules and transitions
   - Responsibility boundaries (what belongs where)
   - Process requirements (what must happen, in what order)
   - Assumptions and constraints
3. **Cross-reference**: For each concept, compare all documents that mention it.
   Flag where they disagree, are ambiguous, or make incompatible assumptions.
4. **Register population**: Write each conflict as a register entry with
   severity and topic classification.

### Phase 2: Human resolution (VH-047-001)

The register is presented to the maintainer grouped by topic area:

1. **Review batch**: Read all contradictions in a topic area together
2. **Design decision**: Decide which position is correct (or formulate a new one)
3. **Codify**: Write an ADR if the decision is architectural, or note the
   spec revision needed if local
4. **Mark resolved**: Update register entry with resolution and reference

### Phase 3: Application

For each resolved contradiction:
1. If ADR: create the ADR via `spec-driver create adr`
2. If spec revision: create RE-xxx documenting the change
3. Update affected PROD specs to align with resolution
4. Editorial fixes applied directly (no RE needed)

### Phase 4: Consistency check (VA-047-002)

Agent re-scans the updated corpus to verify:
- No remaining contradictions in the register's topic areas
- New ADRs are reflected in all relevant PROD specs
- No new contradictions introduced by the resolution edits

## 7. Design Decisions & Trade-offs

- **Batch by topic, not by document**: Resolving all "lifecycle semantics"
  contradictions together is more efficient than going doc-by-doc, because
  one design decision often resolves multiple contradictions.

- **ADR threshold**: Use ADRs for decisions that (a) affect 3+ specs, or (b)
  resolve a genuinely architectural question. Everything else is a spec
  revision or editorial fix.

- **Register as living document**: The register persists in the delta directory
  as a record of what was found and how it was resolved. Useful for future
  agents to understand why specs say what they say.
