# DR-049 – Consolidate workspace directories under .spec-driver with backward-compat symlinks

## 1. Executive Summary
- **Delta**: [DE-049](./DE-049.md)
- **Status**: draft
- **Last Updated**: 2026-03-06
- **Synopsis**: Flatten four top-level workspace dirs into `.spec-driver/`, remove intermediate grouping, add backward-compat symlinks.

## 2. Problem & Constraints

- **Current Behaviour**: Workspace scatters content across four top-level directories
  (`specify/`, `change/`, `backlog/`, `memory/`) with intermediate grouping dirs
  (`specify/tech/`, `change/deltas/`) that add nesting without meaning. This feels
  invasive for public adoption and complicates git operations.

- **Drivers / Inputs**:
  - IMPR-008 spike — thorough surface analysis (462 non-Python references, 309 registry entries, ~90 actionable files)
  - DE-048 — established `init_paths(config)` / `reset_paths()` mechanism and `[dirs]` config
  - ADR-004 (accepted) — prerequisite for layout changes

- **Constraints / Guardrails**:
  - Content cross-reference cleanup (~357 files) deferred — symlinks keep old paths working
  - `kanban/` stays at repo root (intentionally external)
  - All registries auto-regenerate — no manual registry migration needed
  - This repo migrates first via `git mv` as proof-of-concept before any automation

- **Out of Scope**:
  - `config/` subdirectory for user-serviceable files (follow-up delta)
  - Content cross-reference mass-update
  - Automated migration tooling (deferred; manual `git mv` for now)

## 3. Architecture Intent

- **Target Outcomes**:
  - New installs use flat `.spec-driver/` layout with no intermediate grouping dirs
  - Existing cross-references, agent instructions, and memories work via backward-compat symlinks
  - `git add .spec-driver` captures all spec-driver changes atomically
  - Path helpers resolve directly under `.spec-driver/` without intermediate grouping

- **Guiding Principles**:
  - Eliminate intermediate constants (`SPECS_DIR`, `CHANGES_DIR`) that exist only as path composition intermediates
  - Leaf helpers (`get_tech_specs_dir()`, `get_deltas_dir()`, etc.) resolve directly against `get_spec_driver_root()`
  - Backward-compat symlinks are cosmetic — spec-driver internally resolves everything through `.spec-driver/`
  - Installer is the single owner of symlink lifecycle

- **State Transitions / Lifecycle Impact**: No lifecycle model changes. This is a structural refactoring of the workspace layout.

### Path Model Before/After

**Before** (two-level grouping):
```
repo_root/
  specify/            ← SPECS_DIR
    tech/             ← TECH_SPECS_SUBDIR
    product/
    decisions/
    policies/
    standards/
  change/             ← CHANGES_DIR
    deltas/           ← DELTAS_SUBDIR
    revisions/
    audits/
  backlog/            ← BACKLOG_DIR
    issues/
    problems/
    improvements/
    risks/
  memory/             ← MEMORY_DIR
  .spec-driver/       ← SPEC_DRIVER_DIR (internal only)
```

**After** (flat under `.spec-driver/`):
```
repo_root/
  .spec-driver/       ← SPEC_DRIVER_DIR (everything lives here)
    tech/             ← TECH_SPECS_SUBDIR (direct child)
    product/
    decisions/
    policies/
    standards/
    deltas/
    revisions/
    audits/
    backlog/          ← BACKLOG_DIR (contains issues/, problems/, etc.)
    memory/
    registry/
    agents/
    hooks/
    templates/
    about/
  specify/            ← real dir containing targeted symlinks (see §7 DEC-049-03)
    tech/ → ../.spec-driver/tech/
    product/ → ../.spec-driver/product/
    decisions/ → ../.spec-driver/decisions/
    policies/ → ../.spec-driver/policies/
    standards/ → ../.spec-driver/standards/
  change/             ← real dir containing targeted symlinks
    deltas/ → ../.spec-driver/deltas/
    revisions/ → ../.spec-driver/revisions/
    audits/ → ../.spec-driver/audits/
  backlog/ → .spec-driver/backlog/
  memory/ → .spec-driver/memory/
```

### Constant Model Change

The current model has three tiers: root dirs → subdirs → helpers compose them.
The new model has two tiers: `SPEC_DRIVER_DIR` → leaf subdirs.

**Eliminated constants**: `SPECS_DIR`, `CHANGES_DIR` (intermediate grouping).

**Retained/unchanged constants**: All `*_SUBDIR` constants keep their values.
`BACKLOG_DIR` and `MEMORY_DIR` become subdirs of `SPEC_DRIVER_DIR` rather than
repo-root dirs, but their string values stay the same (`"backlog"`, `"memory"`).

**Eliminated helpers**: `get_specs_dir()`, `get_changes_dir()`. One production
caller (`changes/registry.py`) is refactored to use leaf helpers directly.
The other (`deletion/executor.py`) is a dead assignment — removed outright.

**Changed helpers**: All leaf helpers (`get_tech_specs_dir()`, `get_deltas_dir()`,
`get_decisions_dir()`, etc.) resolve against `get_spec_driver_root()` instead of
the former grouping helpers.

`get_backlog_dir()` and `get_memory_dir()` resolve as
`get_spec_driver_root() / BACKLOG_DIR` and `get_spec_driver_root() / MEMORY_DIR`.

### `_CONFIG_KEY_TO_CONSTANT` Mapping Change

The config key mapping changes in two ways:

1. **Keys removed**: `specs` and `changes` (no corresponding constants after
   grouping elimination). Existing `[dirs]` overrides for these keys become
   no-ops. This is acceptable — the keys had no independent meaning; they were
   composition intermediates.

2. **Subdir keys reparented**: Keys like `tech_specs`, `decisions`, `deltas`
   retain their constant names (`TECH_SPECS_SUBDIR`, `DECISIONS_SUBDIR`, etc.)
   and default values (`"tech"`, `"decisions"`, `"deltas"`). But their *semantic
   meaning* changes: the value was previously a subdir name within a grouping
   dir, now it's a direct child name under `.spec-driver/`. An override like
   `tech_specs = "specifications"` now resolves as `.spec-driver/specifications/`
   instead of `specify/specifications/`.

The mapping still powers `init_paths()` / `reset_paths()`. The `_ORIGINAL_DEFAULTS`
snapshot captures the new compiled defaults.

**Migration for existing overrides**: Any workspace with `[dirs]` overrides for
`specs` or `changes` will see those keys silently ignored. This is unlikely in
practice (DE-048 only shipped days ago) but should be documented in release notes.

### Backward-Compat Symlink Strategy

Symlinks preserve old path resolution for existing content cross-references
using **targeted symlinks inside real directories**, not broad dir-to-dir
symlinks. This prevents leaking unrelated `.spec-driver/` contents through
the compat paths.

**`specify/`**: A real directory containing symlinks to the subset of
`.spec-driver/` children that previously lived under `specify/`:
```
specify/
  tech/ → ../.spec-driver/tech/
  product/ → ../.spec-driver/product/
  decisions/ → ../.spec-driver/decisions/
  policies/ → ../.spec-driver/policies/
  standards/ → ../.spec-driver/standards/
```

**`change/`**: Same pattern:
```
change/
  deltas/ → ../.spec-driver/deltas/
  revisions/ → ../.spec-driver/revisions/
  audits/ → ../.spec-driver/audits/
```

**`backlog/`** and **`memory/`**: Direct symlinks (no leakiness concern since
they map 1:1):
```
backlog/ → .spec-driver/backlog/
memory/ → .spec-driver/memory/
```

This means `specify/tech/SPEC-001/` resolves correctly, but `specify/registry/`
does not (as intended — `registry/` was never under `specify/`).

**Lifecycle**: These compat structures are temporary. Once content cross-references
are cleaned up (deferred), the `specify/` and `change/` directories and their
symlinks can be removed entirely.

## 7. Design Decisions & Trade-offs

### DEC-049-01: Flatten path model — eliminate intermediate grouping dirs

**Decision**: Remove `SPECS_DIR` (`specify/`) and `CHANGES_DIR` (`change/`) as
intermediate grouping concepts. All artifact directories become direct children
of `.spec-driver/`.

**Rationale**: The grouping dirs (`specify/`, `change/`) add a nesting level
without semantic value. `decisions/` sits under `specify/` but isn't a
"specification". Flattening makes the hierarchy honest and reduces path depth.

**Consequences**: 1 production caller of `get_changes_dir()` needs refactoring
(`changes/registry.py`). The other (`deletion/executor.py`) is a dead assignment.
~52 test files reference path constants but import them from `core.paths`, so
value changes propagate automatically. Parent-child structural tests (e.g.
`test_tech_specs_is_child_of_specs`) must be rewritten — they encode the old
grouping invariant.

### DEC-049-02: Leaf helpers resolve directly against spec-driver root

**Decision**: `get_tech_specs_dir()`, `get_deltas_dir()`, `get_decisions_dir()`,
etc. all compute `get_spec_driver_root(repo_root) / SUBDIR_CONSTANT` instead
of chaining through intermediate grouping helpers.

**Rationale**: Simplifies the helper model from three tiers to two. Each helper
is self-contained — no implicit dependency on grouping constants.

**Consequences**: Slightly more repetition in helper implementations (each calls
`get_spec_driver_root()` instead of a grouping helper), but the repetition is
trivial and the indirection removal improves readability.

### DEC-049-03: Targeted symlinks inside compat dirs

**Decision**: Backward-compat uses **real directories** (`specify/`, `change/`)
containing **targeted symlinks** to specific `.spec-driver/` children, not
broad dir-to-dir symlinks.

**Rationale**: A broad `specify/ → .spec-driver/` symlink leaks unrelated
content — `specify/registry/`, `specify/agents/`, `specify/memory/` would all
resolve, which is nonsensical. Targeted symlinks preserve exactly the old
`specify/` structure (tech, product, decisions, policies, standards) without
exposing internals.

**Trade-offs**: More symlinks to create and manage (8 targeted vs 2 broad for
`specify/` + `change/`). Acceptable because the installer already manages
directory creation, and these structures are explicitly temporary.

**Future**: Once content cross-references are cleaned up, both the compat
directories and their symlinks can be removed entirely.

### DEC-049-04: config/ grouping deferred

**Decision**: The `config/` subdirectory for user-serviceable files
(`workflow.toml`, `doctrine.md`, templates) is deferred to a follow-up delta.

**Rationale**: Separating user-owned config from managed internals is valuable
but orthogonal to the directory consolidation. Combining them increases blast
radius unnecessarily. The consolidation stands alone as a coherent change.

### DEC-049-05: Migration via git mv, this repo first

**Decision**: Existing workspaces migrate via `git mv` of content into
`.spec-driver/`, then creation of backward-compat symlinks at old locations.
This repo migrates first as proof-of-concept before considering automation.

**Rationale**: The DR originally stated "no migration tooling" but didn't
address how this repo — which has real content in `specify/`, `change/`,
`backlog/`, `memory/` — transitions. Changing path defaults without moving
content would break everything: helpers would resolve to empty `.spec-driver/`
subdirs while content sits at old locations.

**Migration sequence**:
0. Delete derived symlinks before moving content — spec index symlinks
   (`by-slug/`, `by-package/`, `by-language/`, `by-category/`, `by-c4-level/`,
   alias symlinks), contract mirror symlinks (`SPEC-*/contracts/`), and
   decision status symlinks (`decisions/accepted/`, `decisions/draft/`).
   These are ~400 entries rebuilt automatically by `SpecIndexBuilder`,
   `ContractMirrorTreeBuilder`, and `rebuild_status_symlinks()`. Deleting
   before `git mv` avoids ~400 rename entries in the diff.
1. `git mv` content directories into `.spec-driver/` (preserves git history)
2. Create compat symlink structures at old locations (see DEC-049-03)
3. Update path defaults in code
4. Re-run installer / sync to regenerate registries, agent docs, derived
   symlinks, etc.
5. Verify full test suite passes

**Trade-offs**: `git mv` of large directory trees produces a large diff even
after derived symlink cleanup. Mitigated by doing the structural move in a
dedicated commit before code changes.

**Future**: If migration needs to be automated for other workspaces, the
manual experience here informs what a `spec-driver migrate` command would do.

### DEC-049-06: Config override model — grouping keys removed, subdir keys reparented

**Decision**: The `[dirs]` config section loses `specs` and `changes` keys
(no longer meaningful). Subdir keys (`tech_specs`, `decisions`, `deltas`, etc.)
retain their names but their resolution context changes from "child of grouping
dir" to "child of `.spec-driver/`".

**Rationale**: The grouping constants they mapped to (`SPECS_DIR`, `CHANGES_DIR`)
are eliminated. Keeping dangling config keys would be confusing. The subdir keys
remain useful for renaming individual artifact directories.

**Consequences**:
- `init_paths()` emits `warnings.warn()` for any `[dirs]` key not present in
  `_CONFIG_KEY_TO_CONSTANT`. This catches removed keys (`specs`, `changes`),
  typos, and alerts users to the reparenting. Low implementation cost.
- `[dirs] tech_specs = "my-specs"` previously resolved as `specify/my-specs/`,
  now resolves as `.spec-driver/my-specs/`. The user intent ("put tech specs
  here") is preserved; only the parent changes.
- `_CONFIG_KEY_TO_CONSTANT` mapping updated to remove the two grouping entries.
- `_ORIGINAL_DEFAULTS` snapshot recaptured at import time with new values.

**Known interaction with DEC-049-04**: A future delta adding `config/` grouping
would introduce another resolution context change for `workflow.toml` paths.
This is acceptable but flagged as a known follow-up interaction.

## 9. Rollout & Operational Notes

- **Migration (this repo)**: `git mv` content dirs into `.spec-driver/`, create
  compat symlinks, update code, re-sync. This is the first migration and serves
  as proof-of-concept. Done as a dedicated phase before code changes.
- **Migration (new installs)**: Installer creates the flat layout directly. No
  old dirs to migrate.
- **Migration (other existing workspaces)**: Manual `git mv` following the
  pattern established here. Automated tooling deferred until needed.
- **Recovery / Rollback**: Symlinks are easy to remove. Content can be
  `git mv`'d back. If the change causes problems, revert the structural commit
  and the code changes independently.
