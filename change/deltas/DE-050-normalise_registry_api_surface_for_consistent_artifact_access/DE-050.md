## 1. Summary & Context

- **Change Drivers**: ISSUE-019 (registry API drift), IMPR-009 (TUI dashboard
  needs consistent registry access)
- **Implementation Plan**: [IP-050](./IP-050.md)
- **Research**: [research.md](./research.md) — full comparative audit of all 9
  registries

## 2. Motivation

Registry APIs have drifted into 5+ incompatible patterns for basic operations
(ID lookup, collection, iteration, construction). This makes it impossible to
build generic artifact access without per-type adapter logic — a direct blocker
for IMPR-009 (TUI dashboard) and a maintenance burden in `cli/common.py` which
already contains workarounds for each inconsistency.

Four registries (Decision, Policy, Standard, Memory) have converged on a
consistent convention. The remaining five need alignment.

## 3. Scope & Objectives

- **Primary Outcomes**:
  - Every registry exposes `find(id) → Record | None` for consistent ID lookup
  - Every class-based registry exposes `collect() → dict[str, Record]`
  - Constructor signatures normalised to `(*, root=None)` keyword-only
  - Contracts generated at registry boundaries to codify the normalised API
- **Operational Constraints**:
  - Additive changes only in Tier 1 — no breaking removals
  - Deprecation aliases for renamed methods (`get()` → `find()`)
  - All existing tests must continue to pass
  - Both linters must pass (`just lint` + `just pylint`)
- **Dependencies**: None (this delta unblocks IMPR-009, not the reverse)

## 4. Out of Scope

- Class-ifying BacklogRegistry (Tier 3 — higher risk, separate delta)
- Defining a formal Registry Protocol/ABC (premature until API is stable)
- Backfilling stub specs SPEC-115, SPEC-117, SPEC-123 (informed by the ADR, but separate work)
- Refactoring RequirementsRegistry internals (complex, different concern)
- Changing persistence semantics (`sync` vs `save` naming — low value)

## 5. Approach Overview

### Tiered delivery

**Tier 0 (prerequisite): ADR for coherent registry design**

Draft and accept an ADR establishing the canonical registry API convention:
- Required methods (`find`, `collect`, `iter`, `filter`)
- Constructor contract (`*, root=None`, auto-discovery)
- Return type conventions (None vs raise, dict vs list)
- Model location and mutability guidance
- What remains domain-specific vs what must be uniform

This codifies the Group A convention (Decision, Policy, Standard, Memory) as
the accepted pattern before applying it to the remaining registries. Without it,
normalisation is ad hoc and future registries will drift again.

**Tier 1 (P0): Consistent `find(id)` everywhere**

| Registry | Change |
|---|---|
| ChangeRegistry | Add `find(id) → ChangeArtifact \| None` |
| RequirementsRegistry | Add `find(uid) → RequirementRecord \| None` |
| SpecRegistry | Rename `get()` → `find()` with deprecation alias |
| CardRegistry | Add `find(id) → Card \| None` (non-raising) alongside `resolve_card()` |
| BacklogRegistry | Add `find_item(id, root) → BacklogItem \| None` function |

**Tier 2 (P1): Collection and iteration consistency**

| Registry | Change |
|---|---|
| SpecRegistry | Add `collect() → dict[str, Spec]` wrapping `_specs` |
| SpecRegistry | Add `iter(status=None)` |
| ChangeRegistry | Add `iter(status=None)` |
| RequirementsRegistry | Add `iter(status=None)` |
| SpecRegistry | Constructor → `(*, root=None)` keyword-only |
| CardRegistry | Constructor → `(*, root=None)` with auto-discovery |

**Tier 3 (deferred):** Class-ify Backlog, formal Protocol, model location
consistency.

### System touchpoints

- `supekku/scripts/lib/specs/registry.py`
- `supekku/scripts/lib/changes/registry.py`
- `supekku/scripts/lib/requirements/registry.py`
- `supekku/scripts/lib/backlog/registry.py`
- `supekku/scripts/lib/cards/registry.py`
- `supekku/cli/common.py` — simplify workarounds post-normalisation
- Tests for each modified registry

## 6. Verification Strategy

- **VT**: Unit tests for every added/renamed method on every registry
- **VT**: Existing test suites pass unchanged (no regressions)
- **VA**: `cli/common.py` resolver functions simplified to use normalised API
- **VA**: Contracts regenerated and reviewed for consistency
- **Acceptance Criteria**:
  - `find(id)` returns correct record or None for every artifact type
  - `collect()` returns complete dict for every class-based registry
  - `just` passes (tests + both linters)

## 7. Risks & Mitigations

- **Risk**: Renaming `get()` → `find()` on SpecRegistry breaks callers —
  *Likelihood*: medium — *Impact*: low — *Mitigation*: Deprecation alias,
  grep all callers before removing
- **Risk**: CardRegistry `find()` vs `resolve_card()` semantic confusion —
  *Likelihood*: low — *Impact*: low — *Mitigation*: Clear docstrings
  distinguishing the two; `resolve_card()` kept for backwards compat
- **Risk**: Constructor signature changes break positional callers —
  *Likelihood*: medium — *Impact*: medium — *Mitigation*: Grep for
  positional usage before changing; phase in Tier 2
