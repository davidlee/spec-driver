parity between written plans / design documents and implementation is an important goal.

Don't follow them slavishly. Do:
- always read and refer to the design doc, implementation plan, and task card (when they exist), and treat them as canonical references to guide implementation.
- where conflict arises between the design doc and implementation plan: consider the implementation plan heretical
- where the implementation drifts from either: consider this to be heresy
- raise any major risks, inconsistencies, or suggested deviations with the user for discussion & agreement
- record these along with any minor decisions, omissions, deviations, or other potential sources of drift whenever you finish a task and record progress
- keep design docs updated whenever significant changes during implementation would make them inaccurate or incorrect
   - only edit existing parts of plans or design docs when the user has explicitly agreed to an alternate approach.
   - Prefer additive changes preserving history except where clarity would suffer.

After each assigned task on a kanban card, you must:
1. /parity check the implementation
2. update the implementation notes with any potential drift
3. summarize concisely for the user.
